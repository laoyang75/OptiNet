# LAC 数据质量分析汇总报告 (2025-12-09)

本日志汇总了针对 LAC 库（统计库与全量回填库）的四轮深度分析结论。

## 1. 核心结论摘要

1.  **LAC 编码不可靠**：LAC 的 16 进制前缀不能作为区域判断依据。**唯一的判断标准是 GPS 实际分布**。
2.  **共享 LAC (Shared LAC) 是重灾区**：
    -   **96% 的数据量**（7700万条）落在“多运营商共享 ID”的 LAC 上。
    -   如果不区分运营商（Network Group），这些数据将混淆在一起，导致严重的位置错误。
3.  **"2G+5G" 异常组是数据黑洞**：
    -   仅 **514 个** 异常 LAC ID（占总数 <1%）。
    -   却包含了 **4288 万条** 记录（占全库 **53.6%**）。
    -   这些 LAC 必须**全网拉黑**。

---

## 2. 详细分析记录

### 2.1 第一轮：GPS 分布验证 (Phase 1)
-   **对象**：11 个疑似异常的前缀 LAC。
-   **发现**：
    -   绝大多数（8/11）虽然前缀奇怪，但在 GPS 上**高度集中于北京**，是有效的本地 LAC。
    -   个别（如 `10016`）确认为外地（云南），需剔除。
    -   部分（如 `12368`）虽然主Camp在北京，但有严重的长尾飘移。
-   **结论**：**建立“GPS 为王”的验证标准，不再纠结前缀规则。**

### 2.2 第二轮：共享 LAC 问题 (Phase 2 & 3)
-   **现象**：发现大量 LAC ID 同时出现在 Mobile / Unicom / Telecom 下。
-   **统计**：
    -   在活跃度 >14 天的稳定 LAC 中，**78%** 是共享 ID。
    -   在全量回填库（8000万行）中，**96%** 的记录属于共享 ID 组。
-   **对策**：
    -   **必须弃用单纯的 `lac_id` 作为主键**。
    -   **新主键必须是 `(network_group + lac_id)`**。只有这样才能把物理上不同、但 ID 撞车的基站区分开。

### 2.3 第三轮：制式与异常值深度挖掘 (Phase 4)
-   **现象**：全库有一半以上的数据标记为奇怪的 "2G+5G" 混合制式。
-   **挖掘**：
    -   这 4200 万条数据仅源自 **514 个** LAC ID。
    -   典型 ID：`2336002`, `2097287`, `2097179` 等（多为 209/222 开头）。
-   **定性**：这些极有可能是核心网虚拟 ID 或默认值，不具备地理意义。
-   **对策**：**直接拉黑这 514 个 ID**。
