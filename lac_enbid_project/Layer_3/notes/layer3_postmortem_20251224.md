# Layer_3 Postmortem（20251224）

## 结论（为什么 Step30 曾经卡一周、这次 30 分钟跑完）

根因不是“数据规模太大”，而是旧 Step30 的执行形态在 PG 上触发了两个工程级问题：

1) **查询形态导致并行退化**  
   Step30 大量 `GROUP BY + WINDOW + Finalize/Gather` 组合，容易出现 workers 长期 `MessageQueueSend`、leader 单核汇总成为天花板（典型 `LEADER_SINGLE_THREAD`）。

2) **“分片并发”的实现把底座扫描放大了 N 倍**  
   旧分片方案里，分片过滤点位过晚（底座表扫描/聚合不分片），导致每个 shard 会话重复扫 Step02/Step05，变成并发抢 IO/CPU，整体更慢。

本次 v4 的关键变化是：

- **先落桶（semi-join）**：只保留能落入 bucket_universe 的点，避免对无效点做任何重算。
- **把底座物化一次**：Step05 唯一映射、Step02 点归一落桶在 PREPARE 一次完成，shard 不再重复扫大表。
- **每桶止损（最近 N=1000 点）**：重链路按桶运行且有上限，最坏情况可控。
- **外层分片并行 + 禁止查询内并行**：避免“会话并行 × 查询并行”互相打架。

## 经验规则（确保后续层级越来越快）

1) **任何“分片并行”都要保证分片过滤在最早的输入扫描处生效**  
   否则就是把底座工作量放大 N 倍（最常见的性能陷阱）。

2) **优先把“不会进入最终输出”的数据在最早阶段剔除**  
   例如：bucket_universe 的 semi-join、无效 GPS、lac 无法归一、越界坐标等。

3) **对重算链路强制止损**  
   对单桶/单 key 的点数做硬上限（本轮 N=1000），否则长尾桶会把整体拖死。

4) **每步都要能解释成本来源**  
   - “扫描多少行/过滤掉多少/剩多少进入重链路”
   - “重链路在多少桶上执行、每桶最大样本数多少”
   - 这样才能在下一层出现慢时快速定位是“底座重复扫”还是“长尾桶爆炸”。

## 补充：本期发现的占位脏值（不回写历史）

本期在 `Step31` 明细里发现少量占位值：

- `bs_id=0`
- `cell_id_dec=0`

这些值在规范上应视为非法/占位（至少在本项目口径下不应进入画像与映射），否则会：

- 把 `bs_id=0` 的“伪基站”聚出一批桶，污染 BS 画像统计
- 让 `cell_id_dec=0` 形成“伪 cell”映射，影响下一阶段按 cell 补齐/过滤

本期处理策略：

- **不回写/不重跑历史**（避免扩大影响面）
- 在交付层 `Final` 表里做**硬过滤**，确保下游使用的“交付库有效”
- 同时把“上游也应过滤”的改动记录在 SQL 文件中，供后续治理时启用
