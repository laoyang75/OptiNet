## 3) 反哺补齐/纠偏：补了多少？

以 `public."Y_codex_Layer2_Step06_L0_Lac_Filtered"` 为准：

| lac_enrich_status | 行数 |
|---|---:|
| KEEP_TRUSTED_LAC | 21,657,743 |
| REPLACE_UNTRUSTED_LAC | 96,623 |
| BACKFILL_NULL_LAC | 62,153 |

因此本次“发生补齐/纠偏”的总行数为：

- `BACKFILL_NULL_LAC + REPLACE_UNTRUSTED_LAC = 158,776`

### 3.1 纠偏部分：原始 LAC 的“可信/不可信/缺失”拆解（回答：如果用 lac+cell 纠偏还能多多少？）

纠偏子集口径：`lac_enrich_status in ('BACKFILL_NULL_LAC','REPLACE_UNTRUSTED_LAC')`。

| 指标 | 值 |
|---|---:|
| 纠偏总行数（corrected_total_rows） | 158,776 |
| 原始 LAC 为空（裸 cell_id；corrected_lac_null_rows） | 62,153 |
| 原始 LAC 有值且在可信 LAC 库中（corrected_lac_trusted_rows） | 0 |
| 原始 LAC 有值但不在可信 LAC 库中（corrected_lac_untrusted_rows） | 96,623 |
| 纠偏涉及去重 cell 数（corrected_distinct_cells） | 26,204 |

进一步拆分（按纠偏类型）：

| lac_enrich_status | row_cnt | distinct_cells | distinct_final_lac |
|---|---:|---:|---:|
| BACKFILL_NULL_LAC | 62,153 | 12,758 | 287 |
| REPLACE_UNTRUSTED_LAC | 96,623 | 16,396 | 445 |

结论（针对“lac+cell 纠偏还能多多少”）：

- 在当前 Step06 约束下（**最终必须落在 Step04 可信 LAC 白名单**），纠偏子集里的原始 LAC **没有任何一行**是“原始 LAC 已可信但仍需要纠偏”（=0），因此仅增加“lac+cell 作为纠偏依据”并不会带来额外纠偏增量。
- 若你希望进一步提升纠偏覆盖，方向不在“lac+cell”，而在：扩大可用映射证据（Step05 的可用 cell 覆盖）、或引入二阶段策略（例如处理无运营商/处理碰撞的机制），或重新评估 Step04 白名单强度。

## 5) 最终明细库 vs 最初基线：怎么看变化？

建议用两条基线对照理解：

1) **全量 LAC 基线（LAC_RAW）**：包含所有运营商/制式/缺失情况。  
2) **可反哺候选基线（LAC_ELIGIBLE_5PLMN_4G5G_HAS_CELL）**：限定到 5PLMN + 4G/5G + cell 不为空，是 Step06 能“尝试处理”的范围。

对照结论（直观版）：

- 从 `LAC_RAW` 到 `LAC_ELIGIBLE`：把分析范围收敛到“可按 cell 反哺”的主战场（占 LAC_RAW 的 32.03%）。
- 从 `LAC_ELIGIBLE` 到最终明细库 `LAC_SUPPLEMENTED_TRUSTED`：进一步只保留 `lac_dec_final` 命中可信白名单的记录（候选中留存 57.46%）。
- 最终明细库中，真实发生补齐/纠偏的占比只有 0.73%（`158,776` 行），其余主要是“原始 lac 已可信，直接保留”。
