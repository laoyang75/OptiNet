本文件对应你长期路线里的 **阶段二 + 阶段三 + 最终主库**。本轮 Layer_2（Step00~06）只负责“统计建库”；以下三大步骤仅做 **接口说明 / 字段结构 / 验收标准**，不实现具体算法与回填逻辑。

## 大步骤一：更严的补数/纠偏（Cleaned_Augmented_DB）

### 目标（做什么）

- 针对“抽取到了但无值/或值不可信”的记录，进入逐条处理：验证、纠偏、补齐或剔除。
- 解决你特别强调的质量问题：**统计阶段看起来有了，但实际上仍然无值/不可信**。

### 输入（来自 Layer_2/Layer_0）

- 记录级明细（GPS 路、LAC 路、可能还有其它来源）
- 可信库（来自阶段一）：
  - 可信 LAC（Step04）
  - 可信映射（Step05）

### 输出接口（字段结构，中文（English））

建议对象名（示例）：`public.cleaned_augmented_db`

关键字段（必须能回溯）：

- 来源seq_id（src_seq_id）/来源记录ID（src_record_id）
- 处理状态（status）：Verified / Corrected_LAC / New_CellID_Candidate / Unverified_Omitted
- 原始LAC（original_lac）/修正LAC（corrected_lac）
- GPS状态（gps_status）：Verified / Drift / Missing
- 数据来源（data_source）：APP / SS1 / ThirdParty / ...
- GPS来源（gps_source）：Original / Augmented_from_BS（为最终回填预留）
- 关键键：运营商id_细粒度（operator_id_raw）/制式_标准化（tech_norm）/Cell十进制（cell_id_dec）
- 时间口径：报文时间（ts_std）/上报日期（report_date）
- 坐标：经度（lon）/纬度（lat）

### 验收标准（至少 5 条）

1. 每条记录必须能通过 `src_seq_id/src_record_id` 回溯到原始明细。
2. `status` 枚举值受控且分布可统计（必须能做 status 报表）。
3. `corrected_lac` 仅在 `status=Corrected_LAC` 时出现（其余状态为空或等于 original）。
4. `gps_status` 与 `gps_source` 可组合解释（例如 Drift 的回填来源）。
5. `data_source` 字段必须可扩展（支持未来多数据源接入）。

## 大步骤二：enbid/基站库生成与基站 GPS 计算（Master_BS_Library）

### 目标（做什么）

- 把 CellID 归并为基站/扇区维度（bs_id/sector_id），并用 Verified GPS 聚合计算基站中心点。
- 产出可复用的“基站主库”，用于后续回填与纠偏。

### 输入

- `Cleaned_Augmented_DB`（只取 `status != Unverified_Omitted`，并优先使用 `gps_status=Verified`）

### 输出接口（字段结构，中文（English））

建议对象名（示例）：`public.master_bs_library`

关键字段：

- 基站ID（bs_id）/扇区ID（sector_id）
- 基站中心经度/纬度（bs_gps_center_lon/bs_gps_center_lat）
- 关联LAC（associated_lac_list / associated_lac_count）
- 小区数（cellid_count）
- 覆盖时间（first_seen_ts / last_seen_ts / active_days）
- （可选）按运营商/制式分层字段：operator_id_raw / tech_norm（便于切片）

### 验收标准（至少 5 条）

1. `bs_id/sector_id` 生成规则明确且可替换（4G/5G 不同规则允许版本化）。
2. 只用 `gps_status=Verified` 的点参与中心点聚合（避免污染）。
3. 基站中心点经纬度必须落在合法范围，且不为 (0,0)。
4. `cellid_count/associated_lac_count` 可统计且可解释（不能全为 0）。
5. 同一 `bs_id/sector_id` 不应出现多条主记录（主键必须唯一）。

## 大步骤三：利用 enbid 做最终修正（Final_Master_DB）

### 目标（做什么）

- 用基站中心点回填缺失/漂移 GPS，并完成最终主库输出。
- 提供明确的 GPS 回填来源标记（gps_source），支持追溯与质量评估。

### 输入

- `Cleaned_Augmented_DB`
- `Master_BS_Library`

### 输出接口（字段结构，中文（English））

建议对象名（示例）：`public.final_master_db`

关键字段：

- 最终LAC（lac_final）
- 最终经纬度（lon_final/lat_final）
- GPS回填标记（gps_source）：Original_Verified / Augmented_from_BS / ...
- GPS状态（gps_status）：Verified / Drift / Missing
- 处理状态（status）
- 时间口径：ts_std / report_date
- 回溯字段：src_seq_id/src_record_id

### 验收标准（至少 5 条）

1. `gps_source` 必须能区分“原始可信”与“基站回填”，且分布可统计。
2. 回填后的坐标必须落在合法范围，不允许 (0,0)。
3. `gps_status` 与 `gps_source` 的组合关系合理（例如 Augmented_from_BS 不应标为 Missing）。
4. 最终主库必须可回溯到原始记录（src_seq_id/src_record_id）。
5. 提供 GPS 回填效果报表（例如回填前后 has_gps 的变化），作为上线验收项。
