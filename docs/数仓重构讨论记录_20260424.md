# 本地数仓完整重构 — 讨论记录

> 日期：2026-04-24  
> 背景：基于对线上 MaxCompute 数仓的诊断，决定在本地 PostgreSQL + Citus 集群上完整重构数仓，作为所有项目的统一数据底座。

---

## 一、问题背景

### 为什么要重构？

用户已经构建了 3 套系统，但每次都需要从底层原始数据重新抽取。根因在于线上 MaxCompute 数仓存在以下结构性问题：

| 问题 | 具体表现 |
|---|---|
| ODS 层不完整 | 线上只有 `_1h` 小时流，无全量快照；每次研究都要回溯原始流 |
| DWD 层已被裁剪 | `dwd_*_legal_d_external_cold_storage` 加了法务合规过滤，原始字段不完整 |
| 无字段管理 | 字段含义散落在代码注释里，无统一字典；换项目就全部重来 |
| 无表血缘 | 中间 tmp 表几百张（按小时切片），无法追溯来源 |
| 多项目各自为战 | IP库、rebuild5、CDID 各自建表，相同字段有多种写法 |

### 线上 MaxCompute 实际表结构（已确认）

通过 MCP 扫描，线上数仓分布在 3 个 Project：

- **`p4_219685084919881265`**：核心数仓（ODS/DIM/DWD/DWS/ADS/IMP 层）
- **`1568935375600958`**：运算中间层（大量 `tmp_android_mdna_*`、`tmp_dna_daa_*` 按小时切片）
- **`p4_209926680848017092`**：另一组研究 tmp 表

## 三、架构认知与重构范围

### 核心认知（2026-04-24 更新）

**现有的 DNA/DAA 两张宽表本身就相当于 ODS 层**，不需要重新设计这一层。  
只需要将其迁入 Citus 集群，补充 `ip_long` 分布键即可。

**真正需要从头构建的是 CDM 层和 ADS 层。**

### 分层结构

```
【已有 — ODS 层】（迁入 Citus 即可）
  PG17 原始宽表 → Citus ods schema
  ods.ods_dna_raw  822列 × 1000万行  分布键: ip_long
  ods.ods_daa_raw  524列 × 1000万行  co-located with dna
         │
         │ ← 以上是起点，不需要重新设计
         ▼
【需要重构 — CDM 层】（Common Data Model）
┌─────────────────────────────────────────────┐
│  DIM 层（schema: dim）                       │
│  dim.dim_carrier     运营商编号映射           │
│  dim.dim_device      设备型号/品牌            │
│  dim.dim_ip_geo      IP 地理库（多源）        │
└──────────────┬──────────────────────────────┘
               ↓
┌─────────────────────────────────────────────┐
│  DWD 层（schema: dwd）                       │
│  dwd.dwd_device_report                      │
│  从 ODS 抽 ~200 列 + 类型化 + DNA/DAA 合并   │
│  分布键: ip_long                             │
└──────────────┬──────────────────────────────┘
               ↓
┌─────────────────────────────────────────────┐
│  DWS 层（schema: dws）                       │
│  dws.dws_ip_stats       IP 级聚合统计        │
│  dws.dws_ip_network_tier  网段统计           │
│  分布键: ip_long                             │
└──────────────┬──────────────────────────────┘
               │
【需要重构 — ADS 层】（按项目隔离）
┌─────────────────────────────────────────────┐
│  ads_ip.*   → IP库项目（H/E/F 分流体系）     │
│  ads_rb5.*  → rebuild5（候选种子/富化记录）  │
│  ads_cdid.* → CDID 项目（待确认）            │
└─────────────────────────────────────────────┘
               +
┌─────────────────────────────────────────────┐
│  GOV 治理层（schema: gov）                   │
│  字段字典 + 表注册 + 血缘图                   │
└─────────────────────────────────────────────┘
```

### 重构范围总结

| 层级 | 状态 | 说明 |
|---|---|---|
| ODS | **已有，待迁移** | DNA/DAA 宽表即 ODS，迁入 Citus 加 ip_long 即可 |
| DIM | **需新建** | 运营商、设备、IP 地理等维度表 |
| DWD | **需新建** | 从 ODS 抽 ~200 列，类型化，合并两表 |
| DWS | **需新建** | IP 级聚合，供各项目共用 |
| ADS | **需新建** | 各项目业务逻辑层 |
| GOV | **需新建** | 字段字典 + 血缘管理 |

### 关键设计决策

1. **ODS/DWD/DWS 三层所有项目共享，ADS 层按项目隔离**  
   → DWS 层加一个字段，三个项目同时受益

2. **分布键统一用 `ip_long bigint`**  
   → 由 IP 字符串计算而来，保证按 IP JOIN 全部本地化执行  
   → 计算公式：`(split_part(ip,'.',1)::bigint << 24) + ... + split_part(ip,'.',4)::bigint`

3. **DNA / DAA co-located**  
   → 两表按同一 `ip_long` 分片，未来 JOIN 不跨节点

4. **入库顺序：DAA 先于 DNA**  
   → DAA 只有 38GB，先跑通链路；DNA 121GB 再追

5. **目标数据库：新建独立库**（名称待定，不用已有的 `yangca`）

## 四、治理层设计（GOV）

核心四张表：

### `gov.field_registry` — 字段字典
每个字段只定义一次，跨表字段在此统一。

| 列名 | 说明 |
|---|---|
| `field_name` | DWD 层标准字段名（全局唯一） |
| `field_alias` | 原始别名数组（如 `android__id` 和 `android_id`） |
| `category` | 分类：ip/device_id/carrier/wifi/cell/fraud/gps/time/meta |
| `source_dna` / `source_daa` | 各原始表中的列名 |
| `null_rate_dna` / `null_rate_daa` | 覆盖率（已实测） |
| `is_selected` | 是否进入 DWD 层 |

### `gov.table_registry` — 表注册
每张表的层级、项目归属、分布键等元信息。

### `gov.lineage` — 表血缘
记录哪张表从哪张表 ETL 而来，替代线上 tmp 表的混乱状态。

### `gov.field_mapping` — 跨表字段映射
记录 DNA 的 `android__id` ↔ DAA 的 `android_id` 这类等价关系。

## 五、实施策略

### 入库策略
- 使用 `postgres_fdw` 从 PG17（5433）跨服务器查询，实时计算 `ip_long` 入库
- 分批入库（每批 50 万行），避免单次大事务
- 先入 100 万行验证链路，再全量

### 研究驱动的字段选择
- 不在 PG17 上预先裁字段
- 先把全量原始数据入 ODS 层（全列保留）
- 在 Citus 集群上研究哪些字段有价值（覆盖率 + 业务语义）
- 目标：选出 ~200 列进入 DWD 层
- 2000 万行（DNA + DAA 各 1000 万）确保低概率事件有足够样本

## 八、工具选型讨论（2026-04-24）

### 背景与需求

核心问题：技术人员不懂业务，业务人员看不懂 DataWorks 这类工具。  
需要一个能让业务人员理解数据流转的 UI，同时方便本地验证，并且将来工程人员移植到线上也要容易。

### 候选方案对比

| 工具 | 类型 | 本地部署 | 适合现阶段 |
|---|---|---|---|
| OpenMetadata | 数据目录 + 血缘 UI | ✅ Docker（但需 8+ 容器，较重） | ⚠️ 偏重，是"观察者"不是"建设者" |
| DataHub | 数据目录 + 血缘 UI | ✅ | ⚠️ 同上，功能强但复杂 |
| **dbt Core** | 转换管理 + 文档化 | ✅ 极轻量 | ✅ 最适合 |
| dbt + 自定义 UI | 转换管理 + 定制展示 | ✅ | ✅✅ 最推荐 |

### 选定方案：dbt Core + 自定义 UI

**结论：完全可行，推荐采用。**

#### dbt 负责的部分

- 管理 CDM 层（ODS → DWD → DWS）的所有 SQL 转换逻辑
- 自动推断并记录表血缘（不需要手动配置）
- 自动生成字段文档（在 yaml 文件里写业务说明）
- 数据质量测试（行数校验、字段非空校验等）
- 跑完后生成 `manifest.json` + `catalog.json`，包含所有元数据

#### 自定义 UI 负责的部分（复用 rebuild5 技术栈：Vue + FastAPI）

- **FastAPI 后端**：读取 dbt 生成的 `manifest.json`，封装成 REST API
- **Vue 前端**：按业务视角展示
  - 血缘图（业务人员看数据怎么流转的）
  - 字段字典（每个字段的业务含义）
  - 数据质量仪表盘（行数、覆盖率、更新时间）
  - 项目视图（IP库 / rebuild5 / CDID 分项展示）

#### 整体架构示意

```
数据层（Citus 集群）
  └── ODS 原始表（DNA/DAA 入库后）

转换层（dbt Core，免费开源）
  ├── models/dim/       维度表 SQL
  ├── models/dwd/       DWD 层 SQL（字段裁剪 + 类型化）
  ├── models/dws/       DWS 层 SQL（IP 级聚合）
  ├── models/ads_ip/    IP库 ADS SQL
  └── models/ads_rb5/   rebuild5 ADS SQL
  
元数据层（自动生成）
  └── dbt docs generate → manifest.json + catalog.json

展示层（自定义 UI）
  ├── FastAPI：读 manifest.json，查 Citus 运行状态
  └── Vue：业务友好的数据流转可视化界面
```

### 移植到线上的难度

| 目标环境 | 迁移工作量 | 说明 |
|---|---|---|
| 本地 Citus → 线上 Citus | **极低** | 只改 `profiles.yml` 连接配置，SQL 零改动 |
| 本地 Citus → 线上 PostgreSQL | **极低** | 同上 |

**核心优势**：dbt 的每一个转换都是独立 `.sql` 文件 + `.yml` 配置，工程人员移植时只需改连接配置，业务逻辑原封不动交接。

|---|---|---|

## 九、DataWorks 痛点与 dbt 替代方案（2026-04-24）

### 背景

线上技术栈在阿里云 DataWorks + MaxCompute。DataWorks 的核心问题是：**浏览器操作 + SQL 存在云端**，导致 Agent 辅助开发非常困难——Agent 没有本地文件可以直接读写，无法进行自动化。

### DataWorks 的根本局限

- SQL 逻辑存在云端 Web IDE，无法本地 git 管理
- Agent 无法直接读写 DataWorks 里的 SQL
- 调试依赖页面操作，不适合自动化和脚本化
- 无法在本地预跑验证，每次改动都要推到云端才能看结果

### 现有的 CLI 替代方案对比

| 方案 | 说明 | Agent 可编辑 | 本地验证 | 切到 MaxCompute |
|---|---|---|---|---|
| DataWorks Web IDE | 当前现状 | ❌ | ❌ | 本来就在上面 |
| `odpscmd` + 本地 SQL | MaxCompute 官方 CLI，执行本地文件 | ✅ | ❌ | 手动改参数 |
| `PyODPS` | Python SDK，适合写批量脚本 | ✅ | ❌ | 需手动切连接 |
| **dbt + dbt-maxcompute** | **推荐方案** | ✅ | ✅（接 Citus） | **只改 target** |

### dbt 同时解决两个问题

**问题 1：DataWorks 不适合 Agent 开发**  
→ dbt 的所有 SQL 逻辑都在本地 `.sql` 文件里，Agent 可以直接读写、修改、测试

**问题 2：本地验证和线上部署割裂**  
→ `profiles.yml` 配置两个 target，切换一个参数即可：

```yaml
# profiles.yml
my_dw:
  outputs:
    dev:          # 本地 Citus 验证
      type: postgres
      host: 192.168.200.217
      port: 5488
      dbname: ip_warehouse
      schema: dwd

    prod:         # 线上 MaxCompute
      type: maxcompute
      endpoint: http://service.cn-beijing.maxcompute.aliyun-inc.com/api
      project: p4_219685084919881265
      access_id: "..."
      secret_access_key: "..."
```

```bash
dbt run --target dev    # 在本地 Citus 验证
dbt run --target prod   # 推到线上 MaxCompute
```

### 工作流变化

**现在（DataWorks）：**
```
想法 → 打开浏览器 → DataWorks 写 SQL → 在云端跑 → 看结果 → 改 SQL → 重复
（Agent 全程无法参与）
```

**改用 dbt 后：**
```
想法 → Agent 生成/修改本地 .sql 文件 → dbt run --target dev → 本地 Citus 验证
     → 验证通过 → dbt run --target prod → 推到 MaxCompute 线上
（Agent 可以全程参与，SQL 有版本控制，本地可预跑）
```

### 结论

dbt 不只是"本地建数仓"的工具，它同时解决了 DataWorks 的 Agent 开发痛点：
- SQL 逻辑全部本地化，纳入 git 版本控制
- Agent 可以直接编辑、测试、重构 SQL
- 本地 Citus 验证通过后，一条命令部署线上 MaxCompute
- 工程人员移植时只需改 `profiles.yml`，业务逻辑零改动
