# 需求理解与成功标准

## 1. 你的核心需求（还原）
1. 不是只要“结果表”，而是要“可视化研究过程”。
2. 要按 Layer 分层解释每一步：为什么存在、做了什么、输入输出如何变化。
3. 要能看见问题数据在哪里出现、如何被补丁处理、处理后是否真的变好。
4. 要形成“Agent协作闭环”：发现问题 -> 让Agent修复 -> 记录逻辑 -> 再验证。
5. 要把分析师脑中的隐性想法显性化，避免后续团队无法复现或无法审查。
6. 最终沉淀成下一阶段大规模执行的基础工具与方法。

## 2. 你当前痛点（问题本质）
1. 旧方案偏“分析师个人经验”，过程不可见、不可审计。
2. 复杂判断散落在多处 SQL/文档中，跨层逻辑易丢失。
3. 出问题时只能看结果，很难快速定位到具体层、具体规则、具体补丁。
4. 修复动作和决策理由没有标准化记录，后续难复用。

## 3. 成功标准（验收口径）
1. 任意一层都能在页面上回答：
- 输入量是什么
- 过滤/补齐规则是什么
- 输出量变成多少
- 为什么变化
2. 任意异常都能在 15 分钟内定位到：层、规则、数据样本、补丁记录。
3. 任意补丁都能追溯：触发原因、修改内容、影响范围、前后对比、回滚方式。
4. 同一问题交给不同人，能得到一致的排查路径与结论表达。
5. 新成员不读全部 SQL，也能理解链路并做有效审查。

## 4. 非目标（避免跑偏）
1. 不是先追求界面“好看”，而是先确保“可解释、可追溯、可审计”。
2. 不是一次性定死全部规则，而是支持可控迭代和补丁演进。
3. 不是只做静态报告，而是要支持持续研究与反复验证。
