# rebuild2：数据驱动的网优数仓重构

> 创建日期：2026-03-30
> 核心理念：**先看数据，再建结构；每建一层，可视化确认一层**

---

## 1. 为什么要 rebuild2

rebuild 的问题不在技术栈，而在构建顺序：
- 先做了 22 张 workbench 元数据表和完整的工作台 UI
- 但 pipeline 表基本是 legacy 数据的直接复制，字段没有经过重新审视
- 快照和指标在查"还没真正跑过"的表

rebuild2 的核心改变：**从数据本身出发，一层一层构建数仓，每层都有可视化验证**。

---

## 2. 业务原则（继承不变）

1. **有效 cell_id = 有效记录** — 不因 GPS 漂移丢弃记录
2. **修正优于丢弃** — 优先修正、补齐、标记
3. **层层收敛、互相印证** — 正向 LAC→Cell→BS→GPS，反向 BS→Cell
4. **基线驱动日更** — 冷启动建基线，后续撞基线

## 3. 数据现状

### PG17 (192.168.200.217:5433/ip_loc2)

**原始数据（legacy schema，28 张表）：**

| 表 | 行数 | 说明 |
|----|------|------|
| Y_codex_Layer0_Gps_base | 1.33 亿 | GPS 定位明细 |
| Y_codex_Layer0_Lac | 1.19 亿 | LAC 定位明细 |
| Layer2~5 各步骤产出 | 数百~3000 万 | 上一轮治理的中间和最终结果 |

**两张 Layer0 表共有 42 个字段（结构完全一致）：**

| 分类 | 字段 | 说明 |
|------|------|------|
| 标识 | seq_id, 记录id, did, ip, oaid | 记录和设备标识 |
| 时间 | cell_ts, cell_ts_std, ts, ts_std | 基站时间戳和上报时间戳 |
| 网络 | tech, 运营商id, 原始lac, cell_id, lac_dec, lac_hex, cell_id_dec, cell_id_hex, bs_id, sector_id | 基站网络标识 |
| 位置 | gps_raw, lon_raw, lat_raw, gps_final, lon, lat, gps_info_type | GPS 定位信息 |
| 信号 | sig_rsrp, sig_rsrq, sig_sinr, sig_rssi, sig_dbm, sig_asu_level, sig_level, sig_ss | 信号强度指标 |
| 元数据 | sdk_ver, brand, model, 数据来源, 北京来源, parsed_from, match_status, is_connected | 设备和解析信息 |

## 4. 构建路径（6 层，每层 = SQL + UI）

| 层 | 名称 | SQL 产出 | UI 产出 | 验收标准 |
|----|------|---------|---------|---------|
| **L0** | 字段审计 | 无新表，只查 legacy | 字段审计页：42 个字段的取值分布、空值率、样本值，人工标注保留/丢弃/转换 | 产出字段契约文档 |
| **L1** | ODS 建表 | `ods.cell_records` — 两表合一、字段标准化 | 数据概览页：总行数、字段质量热力图、运营商/制式分布 | 合并后数据量 = GPS + LAC |
| **L2** | 可信库 | `dim.lac_trusted`, `dim.cell_stats`, `dwd.fact_filtered` | 可信库页：LAC 筛选漏斗、Cell 聚合统计、过滤前后对比 | 可信 LAC ~881, Cell ~50 万 |
| **L3** | BS+GPS修正 | `dim.bs_trusted`, `dwd.fact_gps_corrected`, `dwd.fact_signal_filled` | BS 页 + GPS 修正页 + 信号补齐页 | GPS 修正覆盖率 > 80% |
| **L4** | 完整回归 | `dwd.fact_final` | 回归对比页：回归前后差异 | 最终明细 ~3000 万行 |
| **L5** | 画像基线 | `dws.profile_lac/bs/cell` | 画像页：三级画像 + 异常统计 | 画像产出完整 |

## 5. 技术栈（复用 rebuild 验证过的选型）

| 组件 | 技术 | 说明 |
|------|------|------|
| 数据库 | PostgreSQL 17 | 本地，已有数据 |
| 后端 | FastAPI + asyncpg + SQLAlchemy | 复用 rebuild 的 config/database/main 模式 |
| 前端 | 原生 HTML + CSS + ES Modules | 复用 rebuild 的样式系统和 api.js |
| 构建 | 无 | 不引入 npm/vite/webpack |

---

## 6. Schema 规划

| Schema | 用途 | 说明 |
|--------|------|------|
| `legacy` | 原始数据（只读） | 28 张 Y_codex 表，不修改 |
| `ods` | 标准化层 | Layer0 两表合一后的标准化表 |
| `dim` | 维度层 | lac_trusted, cell_stats, bs_trusted |
| `dwd` | 明细层 | fact_filtered, fact_gps_corrected, fact_signal_filled, fact_final |
| `dws` | 汇总层 | profile_lac, profile_bs, profile_cell |
| `meta` | 元数据 | 字段注册表、字段契约、变更日志 |

---

## 7. 与 rebuild 的关系

- **继承**：业务原则、PG17 数据、技术栈、CSS 样式系统
- **不继承**：22 张 workbench 表、快照/Gate 机制、版本管理体系
- **改变**：从"先 UI 后数据"变为"每层数据 + 每层 UI"
