# 控制操作：管道运行与重跑

本文件定义 `rebuild5` 的管道运行控制。核心功能是：触发全流程运行、查看运行状态和历史。

当前版本采用“单活数据集模式”：

- 当前运行的数据集由 `config/dataset.yaml` 决定
- 页面不支持在线切换数据集
- 如需更换数据集，需要清理共享结果后重跑

当前控制侧工作的核心目标仍然是：

- 对当前指定数据库稳定跑通
- 支撑调试、回归、重基线和分支重跑

因此：

- 冷启动、继续执行、分支重跑都先按脚本运维能力收口
- 暂不把这些能力视为通用在线开放功能

## 模块定位

| 项目 | 说明 |
|------|------|
| 模块角色 | 管道运行的触发、监控和历史记录 |
| 核心操作 | 数据准备 → Step 1-5 全流程运行 |
| 运行记录 | 每次运行写入 `run_log`，可追溯 |
| 当前入口 | 标准日批脚本 `scripts/run_daily_increment_batch_loop.py` / API `POST /api/pipeline/run` |

## 运行模式

### 生产日批（标准语义）

`rebuild5` 的产品级标准语义是**按天增量处理**：

```text
Day 1:
  当天原始数据 → Step 1 → Step 2 → Step 3 → Step 4 → Step 5
  产出 v1

Day N (N>1):
  新导入的当天数据 → Step 1
  然后只读取上一版已发布库 / 冻结快照
  以“当天新增证据”为当前批输入继续 Step 2 → Step 5
  产出 vN
```

关键点：

- `batch_id` 的产品语义应与“按天新导入的一批数据”对齐
- 不应把同一份固定全量数据反复重放出来的 `batch2/batch3/...` 当成产品日批
- 生产日批的目标是：每天只处理新增数据，依赖上一版正式库继续流式累计
- 如果历史库里已经存在“验证性重放 batch”，进入产品语义重基线前必须先清空 Step 2-5 状态，再从 `batch1` 重建

### 验证性重放（仅用于验证链路）

为了验证 Step 2-5 的多轮链路是否稳定，可以在固定数据集上做“验证性重放”：

```text
同一份固定数据集
  → 重复执行 Step 2 → Step 5
  → 观察 snapshot / library / carry-forward / Step 4 donor / Step 5 维护链路
```

这个模式只用于：

- 冷启动后的多轮机制验证
- 实现修复后的 clean rerun 验证
- 参数或物理执行策略变更后的回归

它**不是**产品日批语义。

### 全量初始化（冷启动）

首次导入新数据集时的全量处理。没有上一轮冻结快照和可信库。

```text
Step 0: 数据准备（合并去重 → raw_gps）
  → Step 1: ETL（解析 → 清洗 → 补齐）
  → Step 2: 路由分流 + 基础画像（首轮无 Path A，全走 Path B/C）
  → Step 3: 流式评估（产出首版冻结快照）
  → Step 4: 知识补数（首轮无可信库，补数为 0，但仍执行创建 enriched_records）
  → Step 5: 画像维护（发布首版可信库 + collision_id_list）
```

首轮完成后，推荐立即跑二轮（Step 2-5），让 Path A 命中、补数和维护生效。
注意：这里的“二轮”是冷启动后的验证性重放，不应混同为产品日批的定义。

### 参数调整后重跑

修改 `profile_params.yaml` 或 `antitoxin_params.yaml` 后，从 Step 2 开始重跑 Step 2-5。Step 1 的 ETL 产出（`etl_cleaned`）可复用，不需要重跑。

```text
修改参数 yaml
  → Step 2-5 全量重跑
  → 产出新版快照和可信库
```

### 更换数据集（当前实现）

更换数据集不是页面内切换，而是运维动作：

```text
修改 config/dataset.yaml
  → 清理共享 Step 0-5 结果表
  → 重新执行全流程
```

当前版本不保证多数据集结果并存展示；切换能力留待后续开发。

### 规则变更后全量回归

解析规则（Step 1.2）或清洗规则（Step 1.3）变更时，必须从 Step 1 开始全量重跑，保证口径统一。

## 批次编排

```text
Step 1 → Step 2 → Step 3 → Step 4 → Step 5
```

| 步骤 | 前置条件 | 说明 |
|------|---------|------|
| Step 0 | 无 | 数据准备，合并 legacy 表到 raw_gps |
| Step 1 | raw_gps 存在 | ETL 三步（解析/清洗/补齐） |
| Step 2 | etl_cleaned 存在 | 路由分流 + 基础画像 |
| Step 3 | profile_base 存在 | 流式评估，与 Step 2 合并执行 |
| Step 4 | path_a_records 存在 | 知识补数（无 Path A 时仍执行，结果为空） |
| Step 5 | enriched_records 存在 | 画像维护、发布可信库 |

冻结原则始终生效：当前批次只读取上一轮已发布版本。

## 运行记录

每次运行写入 `rebuild5_meta.run_log`：

| 字段 | 说明 |
|------|------|
| `run_id` | 运行标识（如 `prepare_20260409_143000`） |
| `run_type` | bootstrap / daily / rerun |
| `dataset_key` | 所属数据集 |
| `snapshot_version` | 本次发布的快照版本 |
| `status` | running / completed / failed |
| `started_at` / `finished_at` | 运行时间 |
| `step_chain` | 执行的步骤链（如 `step1-step2-step3-step4-step5`） |
| `result_summary` | 各步骤的关键指标（JSONB） |
| `error` | 失败时的错误信息 |

各步骤也有独立统计表（`step1_run_stats` / `step2_run_stats` 等），记录每步的详细指标。

## API 端点

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/system/prepare-current-dataset` | 数据准备（按当前 dataset.yaml 合并去重） |
| POST | `/api/pipeline/run` | 全流程运行（Step 1-5） |
| GET | `/api/system/config` | 当前数据集配置和参数 |
| GET | `/api/system/run-log` | 运行历史列表 |

## 脚本入口

| 脚本 | 用途 |
|------|------|
| `scripts/run_step1_to_step5_daily_loop.py` | 当前推荐的全链自动化重跑入口：按 `raw_gps.ts` 逐天切原始数据，逐天跑 `Step 1 -> Step 5`，最后自动收口正式 `etl_cleaned` |
| `scripts/run_daily_increment_batch_loop.py` | 当前按天增量 / 历史按天重基线标准入口：按 `event_time_std` 日桶物化当天输入，并支持 `--start-batch-id` / `--resume-phase` 分支续跑 |
| `scripts/build_daily_sample_etl_input.py` | 快速测试样本构建脚本：按天从 `etl_cleaned` 抽样，默认每天最多 100 万行 |
| `scripts/reset_step2_to_step5_for_daily_rebaseline.sql` | 当前产品语义重基线前的标准 reset：保留 Step 1，仅清空 Step 2-5 状态 |
| `scripts/reset_step1_to_step5_for_full_rerun_v3.sql` | V3 全量重跑 reset：清空 Step 1-5 派生数据与运行统计，供完整自动化重跑使用 |
| `scripts/runbook_beijing_7d_daily_standard_v3.md` | 当前唯一推荐的 V3 重跑基线 runbook |
| `scripts/archive/pre_v3/README.md` | 历史 runbook / 历史 rerun 脚本归档索引 |

## UI 展示

控制操作不需要独立页面，收敛到"运行历史"页面展示：

| 展示项 | 说明 |
|--------|------|
| 运行类型 | 标注 bootstrap / daily / rerun |
| 执行步骤 | 显示本次运行了哪些步骤 |
| 运行状态 | running / completed / failed |
| 结果摘要 | 各步骤的关键数字 |
| 时间 | 开始/结束时间 |

## 待扩展

以下功能当前不实现，作为后续方向记录：

| 功能 | 说明 |
|------|------|
| 局部重算 | 按 LAC 或对象集合只重跑受影响的部分 |
| 对比验证 | 不同运行版本之间的结果对比 |
| 数据分支 | 不同参数产出独立的结果版本并行对比 |
| UI 触发 | 在页面上点击按钮触发运行（当前用脚本） |
| 定时调度 | 自动化的定时运行 |

## 成功标准

| 能力 | 判定标准 |
|------|----------|
| 跑得通 | 新数据集能从零稳定构建第一版可信库 |
| 看得到 | 每次运行有完整记录，可追溯参数和结果 |
| 重跑得了 | 参数调整后能重跑 Step 2-5，规则变更后能全量回归 |
