# 控制操作：初始化、重算与回归
本文件定义 `rebuild5` 的控制侧链。运行主链负责日常稳态处理，控制侧链负责非日常操作：当数据集、规则或验证目标发生变化时，决定“从哪一步开始、用什么范围、按什么类型重跑”。

来源：
- `rebuild5/README.md` 的“控制侧链”段落
- `rebuild5/docs/00_全局约定.md` 的冻结快照原则、实体模型、命名规范

## 与运行主链的关系
| 模块 | 范围 | 目标 |
|------|------|------|
| 运行主链 | Step 1-5 | 日常稳态生产可信结果 |
| 控制侧链 | 初始化、局部重算、完整回归、对比验证 | 应对非日常场景，保证系统可控 |
控制侧链不改 Step 1-5 的算法和阈值，只改运行边界、起跑步骤、运行类型和结果切换方式；所有重跑继续遵守“只读上一轮已发布版本”的冻结原则，先验证后切换。

## 冷启动初始化
| 维度 | 日常运行 | 冷启动初始化 |
|------|----------|--------------|
| 上一轮冻结快照（`trusted_snapshot_t-1`） | 已存在 | 不存在 |
| 上一轮可信小区正式库（`trusted_cell_library`） | 已存在，可命中 Path A | 不存在，因此不会有 Path A 命中 |
| Step 4 知识补数 | 可用上一轮正式库 | 不生效 |
| 冲突小区编号清单（`collision_id_list`） | 可复用上一轮结果 | 不存在，需在首次维护后产出 |
触发场景：首次导入新数据集，或当前数据集没有历史快照。
```text
Step 1 导入全量数据
    -> Step 2 因无上一轮正式库，只会出现 Path B / Path C
    -> Step 3 从空状态开始全量评估，产出首版冻结快照
    -> Step 5 做首次完整维护，发布首版正式库和冲突小区编号清单
    -> 下一轮开始切入日常运行模式
```

## 局部重算
| 场景 | 执行方式 |
|------|----------|
| 阈值调整 | 从 Step 2 或 Step 3 开始，只重算受影响对象 |
| 区域修复 | 指定数据集标识（`dataset_key`）、LAC 或对象集合做局部修正 |
| 异常回补 | 生成新的快照版本号（`snapshot_version`），先和旧版对比，再决定是否切换 |
局部重算是“精准修复”，不是全量回炉；如果变更已经影响 Step 1 解析或清洗口径，就不能用局部重算，应升级为完整回归。

## 完整回归
| 场景 | 执行方式 |
|------|----------|
| 解析规则变更 | 从 Step 1 开始全量重跑当前数据集 |
| 清洗规则变更 | 产出完整新快照，并与旧版本做全量对比 |
| 口径统一要求 | 确认无明显回归后，才允许切换为当前版本 |
为什么不能省略：
| 风险 | 后果 |
|------|------|
| 只从变更点往后跑 | 历史数据仍保留旧口径 |
| 新旧口径混用 | 同一数据集内结果不可比 |
| 缺少全量验证 | 质量退化无法被系统性发现 |

## 对比验证
| 场景 | 说明 |
|------|------|
| 大规模数据扩展 | 新数据集接入后，验证结果是否稳定 |
| 重要规则变更后 | 判断新旧版本是否保持预期一致性 |
| 收敛性检查 | 验证不同起跑方式是否收敛到一致结果 |
```text
同一份数据
    -> 方案 A: 从较短窗口起跑，逐批滚动到目标窗口
    -> 方案 B: 直接目标窗口全量运行
    -> 对比最终冻结快照和关键指标
```
| 指标 | 用途 |
|------|------|
| Cell 覆盖率 | 看可信 Cell 数量是否一致 |
| 质心偏差（中位数 / P90） | 看空间结果是否稳定 |
| 生命周期一致率 | 看状态流转是否一致 |
| 锚点资格一致率 | 看 Step 4 可用锚点集合是否一致 |

## 批次编排
`Step 1 -> Step 2 -> Step 3 -> Step 4 -> Step 5`。Step 1-4 可跟随数据批次连续执行，Step 5 建议按日定时执行一次，作为独立维护批。
| 步骤 | 触发条件 | 频率 |
|------|---------|------|
| Step 1 | 新数据到达 | 按数据到达频率 |
| Step 2 | Step 1 完成 | 跟随 Step 1 |
| Step 3 | Step 2 完成 | 跟随 Step 2 |
| Step 4 | Step 2 完成且存在 Path A 数据 | 跟随 Step 2 |
| Step 5 | 定时触发 | 建议每天一次 |
编排器前期使用 `scripts/run_pipeline.py` 手动触发，后续可升级为定时任务或工作流引擎。编排器属于基础设施，不属于 Step 定义本身；运行类型应写入运行历史，便于区分日常运行和控制操作运行。

## UI 页面
控制侧链不需要独立页面，统一收敛到“运行历史”中展示。
| 展示项 | 说明 |
|--------|------|
| 运行类型（`run_type`） | 标注日常运行、初始化、局部重算、完整回归、对比验证 |
| 影响范围 | 显示数据集、LAC 范围、参数版本 |
| 对比入口 | 展示新旧版本差异摘要 |
| 发布状态 | 标注“验证中 / 已发布 / 已回退” |

## 成功标准
| 能力 | 判定标准 |
|------|----------|
| 起得来 | 新数据集能从零稳定构建第一版正式可信库 |
| 修得准 | 局部问题能局部重算，不误伤全量结果 |
| 回得全 | 规则变更后能全量回归，保证口径统一 |
| 验得清 | 每次控制操作都有运行类型、对比结果和发布记录 |
