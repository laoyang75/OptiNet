# Step 1: 数据源接入 — 功能要求

> 更新声明(2026-04-28)
> - 已按当前生产环境对齐到 PG 18.3 / Citus 14.0-1 / PostGIS 3.6.3 / `5488` / `yangca`
> - 本轮只更新环境信息、schema/SQL 示例、量化基线和文档路径;业务规则、状态机和术语定义保持不变

Step 1 是 `rebuild5` 的独立 ETL（Extract / Transform / Load）入口，负责把多种原始报文转换成统一的结构化数据，为 Step 2 基础画像提供唯一输入。

本文只定义 Step 1 的模块职责、输入输出、字段一致性、页面要求和数据表边界；具体字段提取、ODS 规则、补齐矩阵见 `01b_数据源接入_处理规则.md`。

## 模块定位

### 这是一个独立通用模块

| 定位 | 要求 |
|------|------|
| 独立模块 | Step 1 只负责“原始数据 -> 结构化清洗后数据”，不参与画像、流式评估、知识补数 |
| 可配置多数据源 | 不硬编码原始表名，数据源通过注册表接入 |
| 与后续步骤解耦 | Step 2 以后只读 Step 1 输出，不反向要求 ETL 了解画像逻辑 |
| 统计跟随数据源 | ETL 统计绑定数据源和运行批次，不进入 `trusted_snapshot` 体系 |
| 字段前后一致 | 解析产出的字段，在清洗和字段对齐中必须有明确去向 |

### 不受冻结快照原则约束

`trusted_snapshot_t-1` 只约束 Step 3 的状态判定，以及 Step 4 / Step 5 的版本边界，不约束 Step 1 的原始数据接入。

```text
原始数据源注册/导入
    -> 字段审计
    -> 解析
    -> 清洗
    -> 同报文字段对齐
    -> 产出 etl_cleaned
```

说明：
- Step 1 的运行统计属于“数据源处理统计（source processing stats）”。
- Step 1 的统计写入 `step1_run_stats`；若页面需要按数据源维度展示，可结合 `source_registry` / `dataset_registry` 补充来源元信息。
- Step 1 不读取 `trusted_snapshot_t-1`，也不产生 `snapshot diff`。

### 与 Step 4 的边界

```text
Step 1 字段对齐 = 同一条原始报文内、同一 record_id 的结构性互补
Step 4 知识补数 = 用 trusted Cell 库补新报文的缺失字段
```

| 项目 | Step 1 字段对齐 | Step 4 知识补数 |
|------|----------------|----------------|
| 数据范围 | 同一 `record_id` 内 | 任意后续新报文 |
| 依赖对象库 | 不依赖 | 依赖上一轮已发布的 `trusted_cell_library`（其版本对应 `trusted_snapshot_t-1`） |
| 触发条件 | 同报文内存在互补字段 | 可信 Cell 已成熟可用 |
| 主要用途 | 结构化对齐 | 历史知识回灌 |
| 结果归属 | Step 1 输出 | Step 4 输出 |

## 5 层处理逻辑

### 处理链路

```text
原始数据源表（可配置，1 张或多张）
    |
    +-- 1.1 数据源注册
    |
    +-- 1.2 字段审计
    |
    +-- 1.3 解析（炸开）
    |
    +-- 1.4 清洗（ODS）
    |
    +-- 1.5 字段对齐（同报文内互补）
    |
    +-- 输出：etl_cleaned（70 列，Step 1 最终产出）
```

### 1.1 数据源注册

| 项目 | 说明 |
|------|------|
| 做什么 | 注册可接入的原始数据表，保存表名、来源类型、字段映射和导入规则 |
| 为什么 | Step 1 必须支持 GPS 报文表、LAC 报文表，以及未来其他同结构或近似结构的数据源 |
| 输入 | 原始数据表元信息、字段映射、源表行数、时间范围 |
| 输出 | 数据源注册记录（source registry） |

必需登记项：

| 配置项 | 说明 |
|--------|------|
| `source_id` | 数据源唯一标识 |
| `source_name` | 数据源名称 |
| `source_table` | 原始表名 |
| `source_type` | 如 GPS 报文源、LAC 报文源、未来扩展源 |
| `schema_name` | 数据库 schema |
| `record_id_field` | 原始唯一记录字段 |
| `ts_field` | 上报时间字段 |
| `cell_infos_field` | JSON 基站字段 |
| `ss1_field` | 文本信号字段 |
| `gps_raw_field` | 原始 GPS 字段 |
| `gps_type_field` | GPS 来源字段 |
| `status` | 待接入 / 已接入 / 已停用 |
| `imported_at` | 导入时间 |
| `row_count` | 记录数 |
| `time_range` | 数据时间范围 |

### 1.2 字段审计

| 项目 | 说明 |
|------|------|
| 做什么 | 盘点注册数据源的字段、类型、覆盖率、样本值和处理决策 |
| 为什么 | 在解析前先确认“哪些原始字段保留、哪些要解析、哪些直接丢弃” |
| 输入 | 已注册原始数据源表 |
| 输出 | 字段审计报告（field audit report） |

字段审计至少回答 4 个问题：
- 原始表有哪些字段，是否与注册配置一致。
- 字段的数据类型是否稳定。
- 字段的覆盖率是否满足解析前提。
- 字段决策是什么：`keep / parse / drop`。

样本基线（rebuild4 已验证）：

| 项目 | 数值 |
|------|------|
| 原始字段总数 | `27` |
| `keep` | `17` |
| `parse` | `3` |
| `drop` | `7` |

### 1.3 解析（炸开）

| 项目 | 说明 |
|------|------|
| 做什么 | 一条原始报文拆成多条结构化记录 |
| 为什么 | `cell_infos` 和 `ss1` 都是复合字段，必须展开后才能做逐行清洗和对齐 |
| 输入 | 原始数据表（27 列） |
| 输出 | `etl_parsed` |

解析要求：
- 支持 `cell_infos`（JSON）和 `ss1`（文本）两种输入结构。
- `cell_infos`：一条报文内每个主服务小区展开成 1 行。
- `ss1`：一条报文内每个信号组展开成 1 行。
- 解析结果必须保留 `record_id`，以便后续做同报文对齐。

样本基线：

| 指标 | 值 |
|------|----|
| 原始记录数 | `249,201` |
| 解析后总行数 | `689,361` |
| 扩展比 | `2.77x` |

### 1.4 清洗（过滤）

| 项目 | 说明 |
|------|------|
| 做什么 | 对解析后逐行执行 ODS 清洗，去掉明显无效值 |
| 为什么 | 不合法主键、越界信号、无效坐标不能直接进入后续空间计算 |
| 输入 | `etl_parsed` |
| 输出 | 清洗后的 `etl_cleaned` 基础列 + 派生列 |

清洗要求：
- 固定使用 `ODS-001 ~ ODS-018` 共 19 条规则。
- 两种动作：`nullify`（置空保留行）和 `delete row`（删除行）。
- 清洗阶段负责派生 `bs_id`、`sector_id`、`operator_cn`、标准时间字段。
- 信号值采用比全局约定更严格的 ODS 边界。

样本基线：

| 指标 | 值 |
|------|----|
| 输入（`etl_parsed`） | `689,361` |
| 删除行数 | `1,573` |
| 输出（清洗后） | `687,788` |
| 通过率 | `99.77%` |

### 1.5 字段对齐（同报文内补齐）

| 项目 | 说明 |
|------|------|
| 做什么 | 同一 `record_id` 内不同来源行互补缺失字段 |
| 为什么 | 一次手机上报会同时携带 `cell_infos` 与 `ss1`，同一 `cell_id` 的信息天然互补 |
| 输入 | 清洗后的 `etl_cleaned` 基础列 |
| 输出 | `etl_cleaned` 最终列（保留基础字段 + 新增 `_filled` 与 `*_fill_source` 字段） |

字段对齐要求：
- 只允许同一 `record_id` + 同一 `cell_id` 内互补。
- `cell_infos` 行之间可全字段补齐。
- `ss1` 行按时间差区分：`<= 1 分钟` 全补，`> 1 分钟` 只补运营商和 LAC。
- 对齐后不覆盖原始清洗列，新增 `operator_filled`、`lac_filled`、`lon_filled` 等结果列。
- 明确不做知识补数。

样本基线：

| 指标 | 补齐前 | 补齐后 | 提升 |
|------|--------|--------|------|
| GPS 覆盖率 | `84.8%` | `97.5%` | `+12.7%` |
| RSRP 覆盖率 | `86.9%` | `94.6%` | `+7.7%` |
| 行数 | `687,788` | `687,788` | `0` |

## 前后一致性要求

### 一致性总原则

1. 解析出的字段不能在清洗或字段对齐中“无声丢失”。
2. 清洗只改变值是否合法，不改变字段语义。
3. 同报文字段对齐只写入对齐结果列，不覆盖原字段真相。
4. Step 1 最终输出必须同时保留：
   - 清洗后的基础真相字段
   - 同报文对齐后的结果字段
   - 对齐来源标记字段

### 字段分层

| 层级 | 列数 | 说明 |
|------|------|------|
| 基础结构化列 | `47` | 解析直接产出（含 `dataset_key`、`source_table` 两个上下文字段） |
| 清洗派生列 | `9` | 清洗阶段补加 |
| 字段对齐结果列 | `14` | 同报文对齐阶段补加 |
| 最终输出总列数 | `70` | Step 1 最终 `etl_cleaned` |

## 数据表

### 数据表链路

```text
原始源表（可配置）
    -> etl_parsed
    -> etl_cleaned（清洗基础列）
    -> etl_cleaned（追加字段对齐结果列，Step 1 最终表）
```

### 表定义

| 表 | 类型 | 说明 | 典型规模（样本） |
|----|------|------|------------------|
| 原始源表 | 输入表 | 原始报文源，按数据源注册接入 | `249,201` 条总记录 |
| `etl_parsed` | 中间表 | 解析后结构化结果 | `689,361` 行 |
| `etl_cleaned` | 中间 / 输出表 | 清洗后基础结构；Step 1 最终产出也落在该表 | `687,788` 行 |

## 与 rebuild4 的变化

| 项目 | rebuild4 | rebuild5 |
|------|----------|----------|
| Step 1 最终输出表 | `etl_filled` | `etl_cleaned`（吸收对齐结果后作为最终输出） |
| 同报文补齐边界 | 实现上已是同报文内补齐 | 文档中明确冻结：只做同报文内字段对齐 |
| 知识补数 | 容易与 ETL 混淆 | 明确移到 Step 4，不属于 Step 1 |
| 统计归属 | ETL 页面统计 | 继续保留，但明确不受数据版本和冻结快照约束 |
| 页面结构 | 注册 / 审计 / 解析 / 清洗 / 补齐 | 沿用 5 页结构，文档中明确写成 Step 1 正式组成部分 |
| 规则本体 | 解析、ODS、同报文补齐 | 原则上保持不变 |
