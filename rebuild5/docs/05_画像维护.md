# Step 5: 画像维护

Step 5 是 `rebuild5` 运行主链中负责“可信赖库深度维护”的版本化步骤。它不再回答“对象能不能入库”，而是回答“已经入库的 Cell / BS / LAC 现在是否仍然可信、是否发生碰撞或迁移、是否需要暂缓刷新、是否应该退出可信链路”。

这一步是可信赖库的治理引擎：Step 3 负责准入并生成冻结快照，Step 4 负责可信补数，Step 5 负责把当前批准入结果维护并发布为正式库。只有经过 Step 5 维护后的 `trusted_cell_library / trusted_bs_library / trusted_lac_library`，才是可供 Step 2、Step 4 和 Step 6 持续消费的完整画像版本。

## 模块定位

### Step 5 的角色

| 项目 | 说明 |
|------|------|
| 模块角色 | Step 5 负责“可信赖库重算 + 异常治理 + 标签完善 + 退出管理”，不是准入判定 |
| 直接输入 | `trusted_snapshot_t`（来自 Step 3 的当前批冻结快照） |
| 参考输入 | 上一版正式库 `trusted_cell_library` / `trusted_bs_library` / `trusted_lac_library`、`enriched_records`（来自 Step 4 的治理后事实层，已发布对象当前批的有效观测）、`gps_anomaly_log`（来自 Step 4）、`cell_sliding_window` |
| 维护对象 | 仅维护已进入可信赖库的对象；不处理尚未入库的 `waiting / observing` 对象 |
| 调度方式 | 建议每天一次定时全量维护；也允许对异常对象做按需补跑 |
| 输出 | 更新后的 `trusted_cell_library`、`trusted_bs_library`、`trusted_lac_library`、`collision_id_list`、`step5_maintenance_log` |
| 本步不做 | Step 3 准入判定、Step 4 donor 补数、Step 6 统计/BI 汇总 |

说明：
- Step 5 在重算 Cell 维度指标时，需同时计算 `pressure_avg`（从 `enriched_records` 聚合），并写入 `trusted_cell_library`，供下一批 Step 4 气压补数使用。这与 `center_lon`/`center_lat` 质心重算属同一链路。
- Step 5 处理的是”当前批准入对象的持续治理与正式发布”，不是”重新做一遍 Step 3 准入判断”。
- 可信对象进入 Step 5 后，仍可能被标记为 `dormant` / `retired`、被防毒化阻断，或被确认存在碰撞 / 迁移 / 多质心。
- Step 5 的输出应作为下一轮只读版本发布，不能边维护边让 Step 2 / Step 4 立即读取尚未稳定的中间结果。
- `trusted_snapshot_t` 是当前批冻结视图，不是 Step 5 全层级的最终聚合底表：Cell 发布直接消费当前批 snapshot，BS 必须基于维护后的 Cell 结果上卷，LAC 必须基于维护后的 BS 结果上卷，不能把 BS/LAC 直接从 snapshot 派生后视作正式库。

### Step 5 的四大职责

| 职责 | 含义 |
|------|------|
| 维护进入 | 对可信赖库做全局碰撞检测、多质心分析、面积异常治理 |
| 异常处理 | 汇总 Step 4 的 `gps_anomaly_log`，做迁移 / 漂移 / 时段异常 / 碰撞的时序判定 |
| 特征标签 | 生成 `drift_pattern`、`is_collision`、`is_dynamic`、`is_multi_centroid` 等完整标签 |
| 维护退出 | 维护滑动窗口、静默检测、`dormant -> retired` 退出链路与重新激活语义 |

### 治理视角，不是 BI 视角

Step 5 页面和产出属于“治理视角”：

- 关注规则是否命中、对象是否异常、当前可信状态是否应该变更。
- 关注本次维护和上次维护之间的差异、阻断原因、退出预警和规则版本。
- 不承载跨月覆盖率、全局分布、经营分析类报表；这些属于 Step 6。

### 三层各自维护，不混成一套逻辑

| 层级 | 维护重点 | 主要输入窗口 |
|------|---------|-------------|
| Cell | 质心精度、碰撞、多质心、GPS 异常时序、漂移分类、动态识别、退出 | 最近 N 天或最少 M 条原始观测 |
| BS | 质心精确化、面积异常、多质心、下属 Cell 健康度、退出 | 下属 Cell 窗口聚合 |
| LAC | 区域完整性、异常 BS 比例、边界稳定性、退出 | BS 级单条聚合记录 |

约束：
- Cell / BS / LAC 的维护结果都要独立落表和展示。
- 上层对象可以复用下层聚合结果，但不能把三层异常含义混成一个字段。
- BS / LAC 的退出由下层联动触发，但仍保留自身的生命周期字段和统计口径。

### 与冻结快照原则的关系

Step 5 虽然是维护步骤，但仍要遵守“读旧、算新、批末发布”的冻结思想：

```text
一次 Step 5 维护运行
    读取上一版已发布可信库 / 滑动窗口 / gps_anomaly_log
        -> 试算本次维护结果
        -> 做防毒化对比和退出判断
        -> 批末一次性发布新的维护结果

不能：
    维护到一半就把半成品 collision_id_list 提供给 Step 2
    维护到一半就把部分新标签暴露给 Step 4 当 donor
```

## 5.1 碰撞检测（两层体系）

碰撞检测分为两层，解决不同的问题：

### A 类：绝对碰撞（真碰撞）

**定义**：同一 `(operator_code, tech_norm, lac, cell_id)` 在不同物理位置（不同 BS）出现，且质心间距 ≥ 阈值。这代表同一运营商在同一 LAC 下真正的基站编号复用。

| 项目 | 规则 |
|------|------|
| 扫描对象 | `trusted_cell_library` 中同一 `(operator_code, tech_norm, lac, cell_id)` 关联到不同 `bs_id` 的记录 |
| 命中条件 | 两个实例的质心距离 ≥ `absolute_collision_min_distance_m`（当前 20km，可配置） |
| 影响 | `is_collision=true`, `drift_pattern=collision`, `baseline_eligible=false`, `antitoxin_hit=true` |
| 阈值来源 | `antitoxin_params.yaml` → `collision.absolute_collision_min_distance_m` |

说明：
- A 类碰撞是治理标签，直接影响 `baseline_eligible`，阻断基线刷新。
- 20km 是初始值，需要根据实际数据调整，不硬编码。
- 不同运营商之间的 cell_id 重复不是碰撞（各运营商独立编号体系）。

### B 类：cell_id 映射表（`collision_id_list`）

**定义**：同一 `cell_id` 被多个不同 `(operator_code, lac)` 组合使用。这是补数优化映射表，不是异常标签。

| 项目 | 规则 |
|------|------|
| 扫描对象 | `trusted_cell_library` 全量 Cell |
| 命中条件 | 同一 `cell_id` 对应多个不同 `(operator_code, lac)` 组合 |
| 影响 | 不影响 `is_collision` / `baseline_eligible`；仅告知 Step 2 补数策略 |

Step 2 消费规则：
- cell_id 不在映射表中 → 补数时不需要附加运营商/LAC 验证，直接补齐
- cell_id 在映射表中 → 补数时必须验证运营商匹配，避免跨运营商错误补齐

说明：
- B 类是加速映射表，目的是增加可补数的量、减少错误补数。
- 对于 cell_id 无碰撞的大多数 Cell，Step 2 可以跳过运营商/LAC 比对直接补齐（因为原始上报中运营商和 LAC 可能为空值）。
- B 类映射表的正式产出步骤固定为 Step 5.1。

### 输出表：`collision_id_list`

`collision_id_list` 至少包含以下字段：

| 字段 | 说明 |
|------|------|
| `cell_id` | 全局冲突主键 |
| `is_collision_id` | 是否命中全局碰撞扫描 |
| `combo_count` | 涉及的 `(operator_code, lac)` 组合数 |
| `combo_list_json` | 各组合列表 |
| `centroid_list_json` | 各组合对应的维护后质心、`p90_radius_m`、设备数、观测量 |
| `dominant_combo` | 主组合；可按设备数、活跃天数、观测量投票产生 |
| `last_maintained_at` | 本次碰撞表发布时间 |

说明：
- `dominant_combo` 不是 Step 2 路由必需字段，但对治理页面、人工审核和碰撞确认很有价值。
- 根据 `rebuild4/docs/02_profile/04_cell_research.md` 的研究，碰撞远端位置常呈现“仅 1 台设备、且距离主位置数十公里”的特征，主组合可优先采用“设备数 + 活跃天数 + 独立观测量”的多维投票。

### 提供给 Step 2 的局部防护接口

Step 2 对 `collision_id_list` 的消费规则保持不变：

| 场景 | Step 2 行为 |
|------|-------------|
| `is_collision_id=false` | 按普通可信库命中规则处理 |
| `is_collision_id=true` 且 `(operator_code, lac)` 匹配，GPS 与该组合质心接近 | 允许命中 Path A |
| `is_collision_id=true` 但 GPS 明显不接近 | 视为误命中，丢弃 |

阈值说明：
- Step 2 当前文档已冻结 `collision_match_gps_threshold_m` 默认建议为 `2200m`。
- 该阈值是可配置业务阈值，当前建议与漂移分类里的 `drift.collision_min_spread_m=2200` 保持同口径，避免“碰撞判定”和“碰撞防护”使用两套不一致尺度。

### 碰撞统计

`step5_maintenance_log` 至少记录以下全局碰撞统计：

| 指标 | 含义 |
|------|------|
| `collision_cell_id_count` | 本次命中的碰撞 `cell_id` 数 |
| `collision_combo_count` | 碰撞涉及的 `(operator_code, lac)` 组合总数 |
| `new_collision_cell_id_count` | 相比上一版新发现的碰撞 `cell_id` 数 |
| `collision_operator_dist` | 按运营商统计的碰撞分布 |
| `collision_lac_dist` | 按 LAC 统计的碰撞分布 |

## 5.2 Cell 维护

Cell 是 Step 5 的核心维护单元。Cell 维护必须基于滑动窗口中的原始观测重算，而不是只对 Step 3 的冻结指标做补丁式修改。

**事实层闭环**：已发布对象在当前批的有效观测来自 Step 4 的 `enriched_records`（Path A 记录经 Step 4 治理后落地）。Step 5 必须消费 `enriched_records` 作为已发布对象当前批的窗口更新输入，否则已发布对象将拿不到当前批新观测，维护链路断裂。`enriched_records` 不得回灌本批 Step 2 / Step 3。

```text
trusted_cell_library
    + cell_sliding_window
    + enriched_records (来自 Step 4，已发布对象当前批有效观测)
    + gps_anomaly_log
        -> 窗口更新（enriched_records 纳入滑动窗口）
        -> 面积异常筛选
        -> 多质心分析
        -> GPS 异常时序判定
        -> 防毒化
        -> 漂移分类 / 动态识别 / 碰撞确认
        -> 窗口保鲜 / 退出管理
        -> 更新 trusted_cell_library
```

### 5.2.1 面积异常 Cell 的多质心分析

多质心分析不是 Cell 全量计算，只处理“面积异常或异常证据明显”的子集。

#### 触发范围

| 触发条件 | 性质 | 阈值来源 |
|----------|------|----------|
| `p90_radius_m` 过大 | 可配置触发线 | 当前建议先参考 `lifecycle.active.max_p90_m=1500`，后续独立外化 |
| `drift_pattern` 非 `stable` 且 `max_spread_m` 很大 | 可配置触发线 | 漂移阈值来自 `profile_params.yaml` |
| 同 Cell 持续命中 `gps_anomaly_log` | 规则触发 | Step 4 异常候选输入 |
| 命中 `collision_id_list` 或人工关注清单 | 规则触发 | Step 5 维护策略 |

说明：
- 本步骤的目标是找出“一个 Cell 在窗口内是否其实对应多个物理位置”。
- 不应对全部 Cell 做重聚类，否则会把治理步骤退化成高成本离线分析。

#### 分析方法

推荐口径：

1. 取 Cell 滑动窗口中的原始观测点或日质心点。
2. 做空间聚类，找出候选质心簇。
3. 比较簇间距离、各簇观测占比、设备数、活跃天数。
4. 若存在两个及以上稳定簇，则标记 `is_multi_centroid=true`。

建议保留的聚类证据：

| 字段 | 说明 |
|------|------|
| `cluster_count` | 窗口内质心簇数量 |
| `cluster_centroids_json` | 各簇中心点 |
| `cluster_obs_ratio_json` | 各簇观测占比 |
| `cluster_dev_count_json` | 各簇设备数 |
| `cluster_span_days_json` | 各簇活跃天数 |
| `dominant_cluster_id` | 主簇编号 |

补充规则：
- 当某个远端簇只有 `1-2` 台设备，而主簇有多台设备时，可优先将远端簇标记为“远端单设备可疑簇”。
- 这与研究文档中的碰撞特征一致：远端位置常为极少设备、但与主簇共享相同 `PCI / freq` 组合。

#### 维护结果

| 结果 | 行为 |
|------|------|
| 未发现稳定多簇 | `is_multi_centroid=false`，按单中心 Cell 继续维护 |
| 发现多簇，但主簇占绝对优势 | 标记 `is_multi_centroid=true`，各簇独立记录到 `cell_centroid_detail`，主簇作为可信质心用于路由和服务查询 |
| 发现多簇，且多簇长期交替出现 | 标记 `is_multi_centroid=true`，各簇独立记录，同时转入碰撞 / 迁移进一步确认流程 |

#### 多质心独立记录策略

确认 `is_multi_centroid=true` 的 Cell，各质心簇必须独立记录到 `cell_centroid_detail` 表，而不是仅存为 JSON 字段。独立记录的好处：

1. **流式数据进入时**，新到达的观测可以按距离直接归属到最近的已知簇，避免每次都重新聚类。
2. **服务查询时**，优先返回主簇质心，但可按簇分别提供定位参考。
3. **治理展示时**，各簇的独立指标（设备数、活跃度、面积）可直接查询，不需要解析 JSON。

`cell_centroid_detail` 表：

| 字段 | 说明 |
|------|------|
| `operator_code` / `lac` / `cell_id` | 业务键 |
| `cluster_id` | 簇编号 |
| `is_primary` | 是否为主簇 |
| `center_lon` / `center_lat` | 簇中心 |
| `obs_count` | 簇内观测数 |
| `dev_count` | 簇内独立设备数 |
| `active_days` | 簇活跃天数 |
| `obs_ratio` | 该簇观测占 Cell 总观测的比例 |
| `area_km2` | 簇覆盖面积估算 |
| `last_maintained_at` | 最近维护时间 |

显示规则：
- UI 上优先显示主簇（`is_primary=true`）的质心和指标。
- 次簇以折叠或展开方式展示，标注为辅助参考。
- 当主簇切换时（如迁移确认后新位置成为主簇），更新 `is_primary` 标记。

### 5.2.2 GPS 异常时序判定

Step 4 只对单条记录做 `gps_anomaly=true/false` 的初筛；Step 5 必须把这些记录沿时间轴汇总成 Cell 级结论。

#### 输入与目标

| 项目 | 说明 |
|------|------|
| 输入表 | `gps_anomaly_log` |
| 输入粒度 | 记录级 |
| 输出粒度 | Cell 级主判定 + 记录级矫正动作 |
| 目标 | 区分 `migration_suspect`、`drift`、`time_cluster`，并为碰撞 / 迁移确认提供证据 |

#### 判定流程（ASCII）

```text
gps_anomaly_log
    -> 按 (operator_code, lac, cell_id, day) 聚合
    -> 计算:
         anomaly_days
         max_consecutive_days
         anomaly_hour_concentration
         anomaly_distance_trend
         anomaly_device_count
    -> 是否连续多批异常?
         | no
         v
       是否集中在固定时段?
         | yes -> gps_anomaly_type = time_cluster
         | no  -> gps_anomaly_type = drift
         v
    yes
         -> 是否存在单向持续位移?
              | yes -> gps_anomaly_type = migration_suspect
              | no
              v
           是否伴随多质心/往返跳变?
              | yes -> 进入 collision_confirm 流程
              | no  -> 保留为持续异常待观察
```

#### 判定口径

| 类型 | 含义 | 典型特征 |
|------|------|----------|
| `drift` | 偶发 GPS 漂移 | 异常不连续、距离波动但无明确方向、整体质心稳定 |
| `time_cluster` | 时段性异常 | 异常集中在固定小时段 / 特定场景窗口 |
| `migration_suspect` | 迁移嫌疑 | 连续多批异常，且异常方向接近单向位移 |

说明：
- “连续多批”的具体天数阈值尚未冻结，需外化到维护配置；本文只冻结语义，不新增硬编码天数。
- “固定时段聚集”的判定阈值也属于可配置业务阈值，应与观测模式异常口径共用配置。

#### 异常处理动作

| 类型 | 行为 |
|------|------|
| `drift` | 认定为 GPS 漂移；信号参数仍可信，可用 Cell 可信质心替代异常 GPS |
| `time_cluster` | 保留信号证据，但对该时段 GPS 观测降权或隔离 |
| `migration_suspect` | 不立即更新主质心；进入迁移确认 / 多质心确认 / 防毒化联动 |

重要边界：
- `drift` 只是“GPS 不可信”，不代表 Cell 不可信；信号均值、设备数和存在性仍可保留。
- `migration_suspect` 只是嫌疑，不应在未确认前直接改写历史主质心。
- 若 Step 5 最终确认是碰撞，则需要给 `is_collision=true`，并把主质心限定在主簇。

### 5.2.3 防毒化（独立子步骤）

防毒化不是“只检查 GPS 异常”，而是防止任何异常新数据把已有可信画像污染掉。

#### 执行时机

```text
Cell 滑动窗口加入新数据
    -> 试算新画像
    -> 与上一版维护画像对比
    -> 在容忍范围内?
         | yes -> 发布更新画像
         | no  -> 标记 antitoxin_hit，暂不刷新画像
```

#### 检测维度

以下阈值已外化到 `antitoxin_params.yaml → antitoxin_compare`：

| 维度 | 触发规则 | 默认阈值 | 处理意图 |
|------|----------|----------|----------|
| 质心漂移 | 新旧质心距离 `> max_centroid_shift_m` | `500m` | 防止错误数据把质心拉走 |
| P90 膨胀 | `new_p90 / old_p90 > max_p90_ratio` | `2.0` | 防止错误数据把覆盖半径撑大 |
| 设备突增 | `new_distinct_dev_id / old_distinct_dev_id > max_dev_ratio` | `3.0` | 防止 `cell_id` 复用或异常灌数 |
| 观测模式异常 | 单时段观测占比 `> max_single_window_ratio` | 待冻结 | 防止短时异常流量主导画像（未实现，待数据增长后补充） |

#### 命中后的处理

| 字段 / 行为 | 规则 |
|-------------|------|
| `baseline_eligible` | 暂时设为 `false` |
| `lifecycle_state` | 保持不变，不因为防毒化命中就直接降级 |
| `is_registered` | 保持不变 |
| 画像刷新 | 暂不发布新画像，保留上一版维护结果 |
| 记录处置 | 新数据进入隔离队列或异常证据表，等待人工或后续批次验证 |

说明：
- 防毒化阻断的是“是否允许当前批次新画像生效”，不是“是否承认这个 Cell 仍存在”。
- 一个 Cell 可以仍是 `qualified` / `excellent`，但因为 `antitoxin_hit=true` 而 `baseline_eligible=false`。

### 5.2.4 漂移分类（完整标签）

漂移分类使用“多日质心轨迹”，不是单条 GPS 记录。阈值来自全局约定和现有 `profile_params.yaml`。

#### 核心指标

| 指标 | 含义 |
|------|------|
| `max_spread_m` | 窗口内每日质心之间的最大跨度 |
| `net_drift_m` | 首日质心到末日质心的距离 |
| `ratio` | `net_drift_m / max_spread_m` |
| `drift_days` | 参与漂移计算的有效天数 |

#### 分类规则

| 分类 | 判定规则 | 来源 |
|------|----------|------|
| `insufficient` | `drift_days < 2` | `drift.min_days=2` |
| `stable` | `max_spread_m < 500` | `drift.stable_max_spread_m=500` |
| `collision` | `max_spread_m >= 2200` 且 `ratio < 0.3` | `drift.collision_min_spread_m=2200`、`drift.collision_max_ratio=0.3` |
| `migration` | `max_spread_m >= 2200` 且 `ratio >= 0.7` | `drift.collision_min_spread_m=2200`、`drift.migration_min_ratio=0.7` |
| `large_coverage` | `500 <= max_spread_m < 2200` | `drift.stable_max_spread_m=500`、`drift.large_coverage_max_spread_m=2200` |
| `moderate_drift` | 通常为 `max_spread_m >= 2200` 且 `0.3 <= ratio < 0.7` | 由以上规则反推 |

说明：
- 这些阈值已冻结并外化到 `antitoxin_params.yaml`：
  - `drift_classification.collision_max_ratio=0.3`、`drift_classification.migration_min_ratio=0.7`、`drift_classification.large_coverage_max_spread_m=2200`
  - `drift.stable_max_spread_m=500`、`drift.insufficient_min_days=2`、`collision.min_spread_m=2200`
- `collision / migration / large_coverage / moderate_drift` 是 Cell 画像标签，不是 Step 3 的准入状态。

### 5.2.5 动态 Cell、碰撞确认与迁移确认

Step 5 不只产出 `drift_pattern`，还要把时序异常、多质心和设备投票融合为最终标签。

| 标签 | 触发条件 | 说明 |
|------|----------|------|
| `is_collision=true` | 命中全局 `collision_id_list`，或多质心 + 往返跳变 + 低 `ratio` | 表示 Cell 复用或稳定双位置 |
| `is_dynamic=true` | `drift_max_spread_m > 1500` 且 `drift_pattern in ('migration','large_coverage')` | 来自 `is_dynamic.min_spread_m=1500` |
| `migration_confirmed=true` | `migration_suspect` 持续成立，且新质心单向收敛 | 表示基站搬迁更可能成立 |
| `is_multi_centroid=true` | 多质心分析确认存在多个稳定簇 | 与 `is_collision` 不完全等价 |

补充规则：
- `is_multi_centroid=true` 说明“窗口内看到多个位置簇”；`is_collision=true` 说明“这些多簇已被判定为复用 / 碰撞模式”。
- `is_dynamic=true` 更偏向移动基站 / 临时站点，不等于一定碰撞。
- 当远端簇始终只有 `1-2` 台设备且主簇稳定时，优先标记碰撞证据，而不是贸然认定迁移。

### 5.2.6 滑动窗口数据保鲜

可信 Cell 不能无限累积历史观测，否则画像会越来越像“长期平均”，而不是“当前状态”。

#### 窗口规则

| 项目 | 规则 |
|------|------|
| 维护单位 | 每个 Cell 独立维护一个窗口 |
| 核心原则 | **数量优先** — 精算需要最低样本量和密度保证，数量不够则时间窗口无意义 |
| 密度指标 | `active_days_30d`（过去 30 天活跃天数）、`consecutive_inactive_days`（最近连续不活跃天数） |
| 窗口输入 | Step 4 的 `enriched_records`（用 `lon_final`/`lat_final` 治理后可用值，不是 `lon_raw`） |
| 重算内容 | 质心 / P50 / P90 / rsrp_avg / rsrq_avg / sinr_avg / pressure_avg / 日质心轨迹 |

示例：
- 当前建议默认：保留最近 `14` 天，或至少 `1000` 条观测，取较大范围。
- 若最近 `14` 天内已有 `3000` 条观测，则只保留最近 `14` 天。
- 若最近 `14` 天只有 `120` 条观测，则继续向前回溯，直到至少凑够 `1000` 条。

#### 窗口内重算内容

| 内容 | 说明 |
|------|------|
| `center_lon` / `center_lat` | 用窗口内观测重算 |
| `p50_radius_m` / `p90_radius_m` | 用窗口内观测重算 |
| `daily_centroid_json` | 漂移分类和多质心分析的日轨迹输入 |
| `window_obs_count` / `window_dev_count` | 当前窗口样本量 |
| `last_observed_at` | 退出链路输入 |

#### 归档与删除

| 项目 | 行为 |
|------|------|
| 超出窗口的原始明细 | 从活跃窗口移出 |
| 历史保留 | 仅保留聚合统计或归档表，不继续参与当前画像重算 |
| 页面展示 | 需显示窗口内观测量、窗口起止时间、归档量 |

### 5.2.7 退出管理

退出管理只作用于已经进入可信赖库的 Cell，不作用于 Step 3 中从未入库的观察对象。

#### 退出流程（ASCII）

```text
qualified / excellent
        |
        | 连续静默达到 silent_days_to_dormant
        v
      dormant
        |
        | 持续 dormant 达到 dormant_days_to_retired
        v
      retired
        |
        | 后续再次出现新数据
        v
  回到 Step 2 重新积累
```

#### 判定规则

退出管理基于两个密度感知指标：
- `active_days_30d`：过去 30 天内有观测的天数（密度代理）
- `consecutive_inactive_days`：从最近一次观测到现在的连续无观测天数

| 密度等级 | 条件 | dormant 阈值 | 参数来源 |
|----------|------|-------------|----------|
| 高密度 | `active_days_30d >= 20` | 连续 `3` 天无数据 | `exit.dormant_inactive_days_high_density` |
| 中密度 | `active_days_30d >= 10` | 连续 `7` 天无数据 | `exit.dormant_inactive_days_mid_density` |
| 低密度 | `active_days_30d < 10` | 连续 `14` 天无数据 | `exit.dormant_inactive_days_low_density` |
| retired | dormant 后持续无数据 | `30` 天 | `exit.retired_after_dormant_days` |

所有阈值已外化到 `antitoxin_params.yaml → exit`。

#### 状态动作

| 状态 | 行为 |
|------|------|
| `dormant` | 暂停活跃更新，但保留在可信库和治理页面 |
| `retired` | 从活跃可信库移除，画像转归档 |
| 重新激活 | 不恢复旧状态，回到 Step 2 重建证据链 |

## 5.3 BS 维护

BS 维护完全建立在下属 Cell 的维护结果之上。Step 5 的 BS 维护重点不是“再看一遍原始报文”，而是“用已经清洗过的 Cell 画像重算更稳的 BS 画像”。

### 5.3.1 BS 质心和面积精确化

#### 维护原则

| 项目 | 规则 |
|------|------|
| 质心输入 | 仅使用维护后的可信 Cell 质心 |
| 默认排除 | 已确认碰撞的 Cell、动态 Cell、当前被防毒化阻断的 Cell 可降权或排除 |
| 重算方式 | 以 Cell 质心中位数为主，避免个别异常 Cell 拉偏 BS 中心 |

#### 异常 BS 子集

BS 的多质心检测同样只做异常子集，不做全量聚类。

| 触发条件 | 性质 | 来源 |
|----------|------|------|
| `max(cell_to_bs_distance_m) > 2500` | 已冻结阈值 | `bs_classification.large_spread_min_dist_m=2500` |
| `gps_p90_dist_m` 明显过大 | 可配置触发线 | 维护策略，后续可外化 |
| 下属 Cell 中 `collision / dynamic / multi_centroid` 占比过高 | 规则触发 | Step 5 聚合判断 |

#### 产出

| 字段 | 说明 |
|------|------|
| `center_lon` / `center_lat` | 精确化后的 BS 质心 |
| `gps_p50_dist_m` / `gps_p90_dist_m` | 下属 Cell 到 BS 的离散度 |
| `is_multi_centroid` | BS 是否存在多个稳定簇 |
| `anomaly_cell_ratio` | 异常 Cell 占比 |

确认 `is_multi_centroid=true` 的 BS，各质心簇同样独立记录到 `bs_centroid_detail` 表（延续 rebuild4 `etl_dim_bs_centroid` 的实践）：

| 字段 | 说明 |
|------|------|
| `operator_code` / `lac` / `bs_id` | 业务键 |
| `cluster_id` | 簇编号 |
| `is_primary` | 是否为主簇 |
| `center_lon` / `center_lat` | 簇中心 |
| `cell_count` | 簇内 Cell 数 |
| `cell_ids` | 簇内 Cell 列表 |
| `dev_count` | 簇内独立设备数 |
| `active_days` | 簇活跃天数 |
| `area_km2` | 簇覆盖面积 |
| `obs_count` | 簇内总观测数 |

显示规则同 Cell：UI 优先显示主簇，次簇可展开查看。

### 5.3.2 BS 分类标签

| 分类 | 规则 | 说明 |
|------|------|------|
| `collision_bs` | 下属 Cell 有碰撞确认 | 反映 BS 受碰撞 Cell 污染 |
| `dynamic_bs` | 下属 Cell 有动态 Cell | 反映可能存在移动站点 |
| `large_spread` | `max(cell_to_bs_distance_m) > 2500` | 已冻结阈值 |
| `multi_centroid` | BS 异常子集聚类后确认多簇 | 深度维护标签 |
| `normal_spread` | 不命中以上异常 | 默认正常标签 |

说明：
- `classification` 是 BS 的治理标签，不等价于生命周期状态。
- 一个 BS 可以同时是 `qualified` 且 `classification='collision_bs'`，表示其“存在异常但仍可识别为一个有效 BS 对象”。

### 5.3.3 BS 数据窗口

BS 不维护和 Cell 一样的原始报文窗口，而是维护“下属 Cell 窗口的聚合视图”。

| 项目 | 规则 |
|------|------|
| 窗口来源 | 下属 Cell 的活跃窗口聚合 |
| 样本要求 | 需要足够多的活跃 Cell 和稳定观测量 |
| 展示字段 | `window_active_cell_count`、`window_obs_count`、`last_cell_activity_at` |

说明：
- 若所有下属 Cell 都已进入 `dormant`，BS 也应进入静默评估。
- BS 页面应能直接看到“当前 BS 由哪些 Cell 窗口贡献、异常来自哪些 Cell”。

### 5.3.4 BS 退出管理

| 场景 | 规则 |
|------|------|
| 联动退出 | 所有下属 Cell 都 `dormant` 或 `retired`，BS 进入对应退出链路 |
| 独立超时 | 长时间没有任何下属 Cell 活动，BS 可直接标记 `dormant` |
| `retired` | 达到 BS 级静默确认阈值后退出可信 BS 库 |

说明：
- BS 的退出阈值同样应挂到 `retention_params.yaml`，不在本文硬编码。
- BS 退出应保留最后一次有效质心和下属 Cell 构成，用于历史追溯。

## 5.4 LAC 维护

LAC 是区域级实体，维护重点是“区域完整性和稳定性”，而不是微观的单点异常。

### 5.4.1 LAC 区域完整性

| 项目 | 规则 |
|------|------|
| 输入窗口 | BS 级单条聚合记录 |
| 维护重点 | 区域面积变化、边界稳定性、异常 BS 比例 |
| 变化特征 | LAC 物理上变化频率应远低于 Cell / BS |

建议维护的指标：

| 字段 | 含义 |
|------|------|
| `area_km2` | 区域面积估算 |
| `boundary_stability_score` | 边界稳定性分数 |
| `anomaly_bs_ratio` | 异常 BS 占比 |
| `active_bs_count` | 当前活跃 BS 数 |
| `retired_bs_count` | 已退出 BS 数 |

说明：
- LAC 页面更像区域治理看板，重点是“区域整体是否仍稳定”。
- 如果某个 LAC 的 `anomaly_bs_ratio` 或面积变化突然上升，说明底层 BS / Cell 维护结果需要重点关注。

### 5.4.2 LAC 退出管理

| 场景 | 规则 |
|------|------|
| 联动退出 | 足够多的下属 BS 退出或持续静默，LAC 进入 `dormant` |
| 极端退出 | 整个位置区长期无有效 BS 活动，才进入 `retired` |
| 参数位置 | 待外化到 `retention_params.yaml` |

说明：
- LAC 退出是极端情况，应显著少于 Cell / BS。
- LAC 一旦进入退出链路，治理页面应高亮提示，因为这通常意味着整片区域网络实体发生较大变化。

## 5.5 进阶标签（只处理异常子集）

Step 5 的“进阶标签”强调两件事：

1. 不是所有高级分析都全量计算。
2. 只有真正需要深挖的对象才进入高成本分析。

### 异常子集范围

| 能力 | 是否全量 | 说明 |
|------|----------|------|
| 全局 `cell_id` 碰撞扫描 | 是 | 对 `trusted_cell_library` 全库做键级扫描 |
| Cell 多质心检测 | 否 | 只处理 `p90` 过大、异常漂移、持续 GPS 异常的 Cell |
| BS 多质心检测 | 否 | 只处理 `large_spread` 或面积异常的 BS |
| 碰撞确认 | 否 | 只对已命中异常证据的对象做确认 |
| 迁移确认 | 否 | 只对 `migration_suspect` 且方向性明显的对象做确认 |
| 动态 Cell 识别 | 否 | 只对跨度较大或迁移 / 大覆盖对象做确认 |

### 碰撞确认

碰撞确认建议综合以下证据：

| 证据 | 说明 |
|------|------|
| `collision_id_list` 命中 | 同 `cell_id` 跨 `(operator_code, lac)` 复用 |
| 多质心 | 窗口内存在多个稳定位置簇 |
| 时序往返跳变 | `ratio < 0.3` 的来回振荡 |
| 设备投票 | 主簇多设备、远端簇单设备 |
| `PCI / freq` 一致 | 与研究文档中的复用特征一致 |

### 迁移确认

迁移确认建议综合以下证据：

| 证据 | 说明 |
|------|------|
| `migration_suspect` 连续成立 | Step 5 时序层已判为持续异常 |
| 单向位移 | `ratio >= 0.7`，且新质心持续朝同方向移动 |
| 新位置收敛 | 新位置簇逐渐成为主簇 |
| 非单设备假象 | 不能只由 `1-2` 台设备推动 |

### 动态 Cell 识别

动态 Cell 识别用于抓出移动基站、车载站或临时站点。

| 证据 | 说明 |
|------|------|
| `drift_max_spread_m > 1500` | 已冻结阈值 |
| `drift_pattern in ('migration','large_coverage')` | 已冻结规则 |
| 设备数偏少或位置跨度大 | 辅助证据 |
| 时段性出现 / 消失 | 辅助证据 |

### 未来场景

### 物理地址（省市区）显示

Cell / BS / LAC 维护页和服务层页面需要展示物理位置（省、市、区县）。

**数据来源**：`rebuild4_meta.lac_location_snapshot`（LAC→省市区映射表，4,552 条，100% 有省市区）。

**关联方式**：通过 `(operator_code, lac)` 关联，不需要逆地理编码。

| 层级 | 关联方式 | 覆盖率 |
|------|---------|--------|
| Cell | 通过所属 LAC 关联 `lac_location_snapshot` | 92.6% |
| BS | 通过所属 LAC 关联 `lac_location_snapshot` | 88.6% |
| LAC | 直接关联 `lac_location_snapshot` | 31.3%（受 rebuild4 数据集范围限制） |

**查询示例**：
```sql
SELECT t.*, loc.province_name, loc.city_name, loc.district_name
FROM rebuild5.trusted_cell_library t
LEFT JOIN rebuild4_meta.lac_location_snapshot loc
    ON t.operator_code = loc.operator_code AND t.lac = loc.lac::bigint
WHERE t.batch_id = (SELECT MAX(batch_id) FROM rebuild5.trusted_cell_library)
```

**注意**：
- `lac_location_snapshot.lac` 是 text 类型，JOIN 时需要 `loc.lac::bigint`。
- 未覆盖区域的地址显示为 `-`，属正常行为。
- 后续如需完整覆盖，应在 Step 5 发布流程中基于质心坐标做逆地理编码，写入 `trusted_cell_library` 自身的地址列。

## 小结

Step 5 的本质不是“给可信库补几个标签”，而是把可信赖库从静态准入名单升级成动态治理系统：

- Step 3 负责把对象送进可信链路。
- Step 4 负责消费可信知识做补数。
- Step 5 负责证明这些知识仍然可信、何时该阻断、何时该退出、何时该确认碰撞或迁移。

因此，Step 5 之后的 `trusted_*_library` 才是 `rebuild5` 中真正可长期复用、可解释、可治理的完整画像资产。
