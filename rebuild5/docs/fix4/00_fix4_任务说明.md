# Fix4 任务说明

## 目标

Fix4 不是正式全库重跑任务，而是一轮基于共享样例数据的研究与实现任务。

本研究线围绕同一个主目标展开：

1. 先解决当前逻辑问题，尤其是位置计算失真、无效 GPS 无法有效过滤、Cell 覆盖面积过大、BS 覆盖面积被连带放大。
2. 在问题解决后，再做速度优化。

## 当前约束

- 不跑正式全库。
- 只使用样例数据开展研究、修复和验证。
- 必须完成完整的样例链路验证，而不是只改局部 SQL。
- 需要覆盖 `7` 次 ETL / 日增量批次。

### 7 批的固定时间范围

这里的 `7` 批不是抽象概念，也不是让你自由模拟循环。

固定就是：

- `2025-12-01`
- `2025-12-02`
- `2025-12-03`
- `2025-12-04`
- `2025-12-05`
- `2025-12-06`
- `2025-12-07`

所有 runbook、速度对比、面积对比，都必须使用这同一组日期。

## 样例数据原则

两个独立研究必须使用**同一份原始样例数据集**。

原因很简单：

- 只有原始输入一致，速度变化和面积变化才可比
- 两位 agent 可以给出不同修复方案，但不能给出不同原始样例集

### 责任边界

- 原始样例数据集由主 agent 最终确认
- 当前研究线必须基于同一份冻结后的原始样例数据工作
- 研究线只能基于这份共享样例数据做派生表、修复和验证

### 允许的研究自由度

这里的“共享原始样例数据固定”不等于“研究过程中不能构造自己的工作集”。

允许自由发挥的部分：

1. 基于共享样例表创建派生表 / 视图 / 物化中间表
2. 为了验证逻辑，按不同目的切出子集：
   - 重点 Cell 子集
   - 全量 7 日样例子集
   - 位置失真专用分析子集
   - 性能压测子集
3. 自主设计参数组、过滤策略、对比口径

不允许自由改动的部分：

1. 共享原始样例表本身
2. 共享重点 Cell 清单本身
3. 固定的 7 天时间范围

### 可用原始来源

共享原始样例数据的来源范围应来自：

1. 当前已准备好的样例数据
2. 当前已跑通数据中筛选出的重点 Cell：
   - 覆盖面积较大的 Cell
   - 流量较多的 Cell
   - 明显存在位置失真的 Cell
   - 多质心争议 Cell

但无论最终选哪些对象，都必须冻结成**一份共享样例数据集**后，再交给两个 agent 使用。

## 当前质量基线

当前没有统一的“黄金标签”来定义绝对正确面积。

因此质量判断采用**同一份共享样例数据上的前后对比**，而不是预设一个绝对面积阈值。

必须至少对比：

1. 共享重点 Cell 的覆盖面积 / 半径是否收敛
2. BS 层的面积或 `gps_p90_dist_m` 是否收敛
3. 面积缩小是否有明确的过滤逻辑支撑，而不是简单裁剪
4. 是否引入新的明显误判

## 当前速度基线

共享样例链在当前问题版本上，已观察到的 Step5 参考耗时是：

- `batch3 Step5 ≈ 124s`
- `batch4 Step5 ≈ 344s`
- `batch5 Step5 ≈ 689s`

这只是参考基线，不是硬 SLA。

你的优化结果必须至少和这条基线做同口径对比。

## 交付要求

每个 agent 最终都必须独立交付：

1. 修复后的代码
2. 基于样例数据跑通的 runbook
3. 说明“修复后如何解决覆盖面积过大”的分析
4. 速度变化研究
5. 面积变化研究
6. 共享样例数据使用与派生工作集说明
7. 失败点与风险说明

## 输出目录约定

当前研究线的输出目录由对应 prompt 指定。

建议每个 agent 至少输出：

- `report.md`
- `runbook.md`
- `dataset_plan.md`
- `metrics_summary.json`

### `dataset_plan.md` 应写什么

这里的 `dataset_plan.md` 不是让你重新定义原始共享样例集，而是要求你说明：

1. 你使用了哪张共享原始表
2. 你在共享原始表之上构建了哪些派生工作集
3. 每个派生工作集服务于什么目标：
   - 逻辑修复验证
   - 覆盖面积分析
   - 性能测试
   - 重点 Cell 深挖
4. 对应 SQL / 脚本如何复现
5. 如果你认为共享样例基底仍然不足，需要怎样补样

## 当前优先级

第一优先级永远是：

- 位置计算问题
- 无效 GPS 过滤问题
- 覆盖面积过大问题

速度优化只能排在这些逻辑问题之后。

### 关于 large_coverage

`large_coverage` 需要分析，但不是当前第一优先级的在线分类目标。

要求是：

- 必须在报告中分析它
- 必须说明它与“面积过大问题”的关系
- 但不强制要求在本轮把它固化成最终主链标签
