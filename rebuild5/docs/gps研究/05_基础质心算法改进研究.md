# 基础质心算法改进研究

> 研究日期：2026-04-14
> 数据基础：rebuild5.research_gps_cells（720 cell）+ research_gps_records（ss1 污染已剔除）
> 状态：**推荐方案 V7（MAD 过滤 + raw_p90 双信号）**

## 1. 问题诊断

### 1.1 现象

Step 5 基础质心算法（非 DBSCAN）产出的 p50/p90 **对所有 cell 都偏大**：

| 标签（样例 720 cell） | 当前 p90 中位数 | 理想期望 |
|---------------------|---------------|---------|
| standard | 179m | ~100m |
| multi_centroid | 499m | 应明显 > 标准 |
| large_coverage | 443m | 应明显 > 标准 |
| migration | 192m | 应有大覆盖信号 |

### 1.2 根因

当前 `cell_core_gps_stats` + `cell_radius_stats` 的核心过滤器参数：

```yaml
core_position_filter:
  snap_grid_m: 200        # 种子网格（200m）
  keep_quantile: 0.8      # 保留 80% 分位
  keep_min_radius_m: 800  # 最小保留半径 800m  ← 问题1
  keep_max_radius_m: 3000 # 最大保留半径 3000m ← 问题2
```

**问题**：
1. `keep_min_radius_m = 800` 即使实际点都在 100m 内，也强制保留 800m 半径内的所有点
2. `keep_max_radius_m = 3000` 对于多质心或有极端噪声的 cell，始终保留 3km 范围
3. 这是**固定绝对阈值**，不适应 cell 本身的离散程度

结果：过滤器对普通 cell"过于宽松"，对异常 cell 又"不够激进"。

## 2. 策略探索

在 720 个 cell × 25.1 万 GPS 点上逐一测试 6 个候选策略，统计 **p90 中位数**（越小表示越精确）：

| 策略 | 原理 | standard | multi_centroid | large_coverage | migration |
|------|------|----------|----------------|----------------|-----------|
| **当前 (library)** | seed+keep_quantile | 179 | 499 | 443 | 192 |
| V1：每设备封顶 20 | 每 dev 最近 20 点 | 236 ↑ | 643 ↑ | 902 ↑ | 3542 |
| V2：两层聚合（dev→cell） | 设备中位 → cell 中位 | 187 | 622 | 958 | 6665 |
| V3：IQR 1.5× 过滤 | Q3 + 1.5·IQR 截断 | 146 | 556 | 637 | 1064 |
| V4：MAD 3× 过滤 | med + 3·MAD 截断 | **100** | 379 | 330 | 147 |
| V6：DBSCAN 主簇 | 取最大簇 | 145 | 172 ↓ | 158 ↓ | 114 ↓ |
| **V7：MAD + raw_p90** | MAD 过滤 + 额外 raw 信号 | **100** | 379 | 330 | 147 |

### 关键观察

- **V1/V2（设备权重类）失败**：对 standard cell 反而负向（standard cell 本来就是多设备少重复观测，封顶后反而丢信息）
- **V3（IQR）改进有限**：standard 从 179→146，但 multi_centroid 保持 556，区分度增加
- **V4/V7（MAD）最佳**：
  - standard 从 179→100m（**44% 降幅**）
  - multi_centroid 保持 379m（仍显著大于 standard）
  - 区分比 **3.79x**（当前为 2.79x）
- **V6（DBSCAN 主簇）不适合作"基础"算法**：所有类别都被压到 100-200m，丢失"大覆盖"信号

## 3. 推荐方案：V7（MAD 过滤 + 双 p90 信号）

### 3.1 算法流程

```
给定 cell 的所有 gps_valid=true 点：

1. center0 = 边际中位数（median(lon), median(lat)）
2. 对每点计算 d_i = distance(point_i, center0)
3. med_d = median(d_i)
4. mad    = median(|d_i - med_d|)
5. 保留 = {i | d_i ≤ med_d + 3·mad}
6. center = median of 保留点
7. p50    = P50 of distance(保留点 → center)     ← 核心半径（precision）
8. p90    = P90 of distance(保留点 → center)     ← 核心半径
9. raw_p90 = P90 of distance(所有点 → center)    ← 总覆盖半径（trigger 信号）
10. outlier_ratio = (total - kept) / total       ← 离群比率（辅助信号）
```

### 3.2 在样例上的效果（V7）

| 标签 | p90 (core) | raw_p90 | outlier_ratio | vs 当前 p90 |
|------|-----------|---------|---------------|-------------|
| standard | **100m** | 206m | 16.7% | -79m (-44%) |
| migration | 147m | 6663m | 29.9% | -45m |
| multi_centroid | 379m | 685m | 18.8% | -120m |
| large_coverage | 330m | 990m | 17.9% | -113m |
| dynamic | 357m | 743m | 28.6% | -204m |
| collision_substitute | 354m | 3043m | 30.5% | -2002m |

### 3.3 多信号组合下的判定能力

| 判定 | 触发阈值 | standard 命中率 | multi_centroid 命中率 | migration 命中率 |
|------|---------|----------------|----------------------|------------------|
| 精度过粗 | p90 ≥ 300m | 7% | **63%** | 14% |
| 大覆盖 | raw_p90 ≥ 800m | 8% | **36%** | **80%** |
| 多簇嫌疑 | outlier_ratio ≥ 0.2 | 41% | 46% | 75% |

**推荐组合触发条件**（用于 Step 3/Step 5 多质心深度分析触发）：
```
(p90 ≥ 300m) OR (raw_p90 ≥ 800m AND outlier_ratio ≥ 0.15)
```

对比当前触发（`p90_radius_m >= 800` OR `raw_p90_radius_m >= 800`）：
- 当前方案中 standard cell 的 p90 本身有 25% 超 800m，误触发很多
- 新方案 standard cell 的 p90 仅 3% 超 500m，精准度大幅提升

## 4. 与当前算法的关键差异

| 维度 | 当前 (cell_core_gps_stats) | 推荐 (V7 MAD) |
|------|---------------------------|---------------|
| 起点 | 200m 网格 snap 最密种子 | 全量点边际中位数 |
| 过滤规则 | 距种子 ≤ max(800, min(P80, 3000)) | 距中位 ≤ med + 3·MAD |
| 阈值类型 | 固定绝对（800/3000） | 自适应（MAD 反映数据离散度） |
| 对稀疏 cell | 强制保留 800m 半径（失真） | 按实际离散度收紧 |
| 对多簇 cell | keep_max=3000m 仍保留噪声 | MAD 天然剔除主簇外点 |
| 输出 | p50, p90, raw_p90 | p50, p90, raw_p90, outlier_ratio（新） |

## 5. 生产代码修改说明

### 5.1 需要修改的函数

**位置**：`rebuild5/backend/app/maintenance/window.py`

**替换 `build_cell_core_gps_stats` 和相关级联**：
- 原函数依赖：`cell_core_seed_grid` → `cell_core_primary_seed` → `cell_core_seed_distance` → `cell_core_cutoff` → `cell_core_points`
- 这些中间表全部可以删除，改为以下更简单的流程

### 5.2 新流程 SQL 草图

```sql
-- Step A: 每 cell 初始中位数
DROP TABLE IF EXISTS rebuild5.cell_center_initial;
CREATE UNLOGGED TABLE rebuild5.cell_center_initial AS
SELECT
    {batch_id}::int AS batch_id,
    operator_code, lac, bs_id, cell_id, tech_norm,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY lon_final) AS clon0,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY lat_final) AS clat0
FROM rebuild5.cell_sliding_window
WHERE lon_final IS NOT NULL AND gps_valid = true
GROUP BY operator_code, lac, bs_id, cell_id, tech_norm;
CREATE INDEX ... ON cell_center_initial (batch_id, operator_code, lac, bs_id, cell_id, tech_norm);

-- Step B: 每点到初始中位的距离
CREATE UNLOGGED TABLE rebuild5.cell_point_dist AS
SELECT w.operator_code, w.lac, w.bs_id, w.cell_id, w.tech_norm,
       w.dev_id, w.lon_final, w.lat_final, w.event_time_std, w.gps_valid,
       SQRT(POWER((w.lon_final - c.clon0) * 85300, 2)
          + POWER((w.lat_final - c.clat0) * 111000, 2)) AS d0
FROM rebuild5.cell_sliding_window w
JOIN rebuild5.cell_center_initial c USING(operator_code, lac, bs_id, cell_id, tech_norm)
WHERE w.lon_final IS NOT NULL AND w.gps_valid = true;
CREATE INDEX ...;

-- Step C: 每 cell 的 MAD
CREATE UNLOGGED TABLE rebuild5.cell_mad AS
SELECT d.operator_code, d.lac, d.bs_id, d.cell_id, d.tech_norm,
       m.med_d,
       PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY ABS(d.d0 - m.med_d)) AS mad
FROM rebuild5.cell_point_dist d
JOIN (
    SELECT operator_code, lac, bs_id, cell_id, tech_norm,
           PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY d0) AS med_d
    FROM rebuild5.cell_point_dist
    GROUP BY operator_code, lac, bs_id, cell_id, tech_norm
) m USING(operator_code, lac, bs_id, cell_id, tech_norm)
GROUP BY d.operator_code, d.lac, d.bs_id, d.cell_id, d.tech_norm, m.med_d;

-- Step D: 标记每点是否为 core（MAD 阈值内）
CREATE UNLOGGED TABLE rebuild5.cell_core_points AS
SELECT d.*, 
       (d.d0 <= s.med_d + 3 * s.mad) AS is_core
FROM rebuild5.cell_point_dist d
JOIN rebuild5.cell_mad s USING(operator_code, lac, bs_id, cell_id, tech_norm);
CREATE INDEX ...;

-- Step E: 新 center（从 core 点）
CREATE UNLOGGED TABLE rebuild5.cell_core_gps_stats AS
SELECT
    {batch_id}::int AS batch_id,
    operator_code, lac, bs_id, cell_id, tech_norm,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY lon_final) AS center_lon,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY lat_final) AS center_lat,
    COUNT(*) AS core_gps_valid_count,
    COUNT(DISTINCT DATE(event_time_std)) AS core_active_days,
    (EXTRACT(EPOCH FROM MAX(event_time_std) - MIN(event_time_std))/3600.0)::double precision
        AS core_observed_span_hours
FROM rebuild5.cell_core_points
WHERE is_core = true
GROUP BY operator_code, lac, bs_id, cell_id, tech_norm;

-- Step F: 计算各点到新 center 的距离，汇总 p50/p90/raw_p90/outlier_ratio
-- 合并到 cell_radius_stats
CREATE UNLOGGED TABLE rebuild5.cell_radius_stats AS
WITH with_dist AS (
    SELECT p.operator_code, p.lac, p.bs_id, p.cell_id, p.tech_norm, p.is_core,
           SQRT(POWER((p.lon_final - c.center_lon) * 85300, 2)
              + POWER((p.lat_final - c.center_lat) * 111000, 2)) AS dist_m
    FROM rebuild5.cell_core_points p
    JOIN rebuild5.cell_core_gps_stats c USING(operator_code, lac, bs_id, cell_id, tech_norm)
)
SELECT operator_code, lac, bs_id, cell_id, tech_norm,
       PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY dist_m) FILTER (WHERE is_core)
           AS p50_radius_m,
       PERCENTILE_CONT(0.9) WITHIN GROUP (ORDER BY dist_m) FILTER (WHERE is_core)
           AS p90_radius_m,
       PERCENTILE_CONT(0.9) WITHIN GROUP (ORDER BY dist_m)
           AS raw_p90_radius_m,
       COUNT(*) FILTER (WHERE is_core) AS core_gps_valid_count,
       COUNT(*) AS raw_gps_valid_count,
       CASE WHEN COUNT(*) > 0
            THEN 1.0 - COUNT(*) FILTER (WHERE is_core)::double precision / COUNT(*)
            ELSE 0
       END AS core_outlier_ratio
FROM with_dist
GROUP BY operator_code, lac, bs_id, cell_id, tech_norm;
```

## 7. 风险与注意事项

1. **Step 4（补数）下游影响**：补数用的是发布 center（会更精确），p90 更紧意味着异常检测阈值（当前 2200m）可能需要适度调整，但方向是好的（触发更准确）
2. **历史数据回溯**：重跑需要同时用新算法重算历史 batch，否则新旧混用
3. **性能**：V7 流程比当前更简洁（少 2 个中间表），整体性能应持平或更快
4. **特殊 cell 处理**：
   - 点数 < 10 的 cell：MAD 可能不稳定，建议退化为简单中位数
   - 全部 gps_valid=false 的 cell：直接返回 NULL，行为不变

## 9. 结论

**推荐 V7（MAD 3× + raw_p90 + outlier_ratio）作为新的基础质心算法标准。**

核心优势：
1. ✅ standard cell p90 从 179m → 100m（**-44%**）
2. ✅ 参数自适应（MAD 反映数据离散度，无需固定阈值）
3. ✅ 通过 raw_p90 和 outlier_ratio 保留"大覆盖/多簇"触发信号
4. ✅ SQL 比当前流程更简洁（少 2 个中间表）
5. ✅ 对所有 6 个标签类别都有验证结果

**需要你确认**：
1. 是否接受这个算法作为新标准
2. `k_mad = 3.0` 是否合适，或想更激进（2.5）/更保守（4.0）
3. Step 5 触发阈值调整方案（p90 ≥ 300 OR raw_p90 ≥ 800 + outlier_ratio ≥ 0.15）

确认后将进入代码修改阶段，按 §5 的说明落地到 `window.py` 和 `publish_bs_lac.py`。
