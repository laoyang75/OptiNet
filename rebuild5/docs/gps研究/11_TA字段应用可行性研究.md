# TA（TimingAdvance）字段应用可行性研究

> **创建日期**：2026-04-22
> **上下文**：在设计下一轮 Step 5 精确画像改进时，评估 LTE 协议层的 `TimingAdvance` 字段能否用于"精确定位基站真实位置"或"辅助判定基站覆盖类型"
> **结论摘要**：
> - **精确定位（trilateration）不可行** —— 量化粒度太粗 + 多径噪声 + 远点 GPS 误差放大
> - **覆盖类型分类 / 异常半径甄别可行** —— TA 统计量（p50/p90/n_obs）可替代当前"拍脑袋"的固定半径阈值
> - **当前 pipeline 丢失了 TA** —— `etl_parsed / etl_cleaned` 之后所有下游表都没有 `timing_advance`；需要和 `cell_origin` 透传一样做一次 pipeline 改造
> - **建议执行时机**：下一轮 `DEDUP-V2` 类全库重跑同时加进来，避免多做一次大跑

## 1. 字段定义和物理含义

### 1.1 cell_infos 里的 TimingAdvance（唯一来源）

```json
"1": {
  "isConnected": 1,
  "signal_strength": {
    "rssi": -51, "rsrp": -80, "rsrq": -7,
    "TimingAdvance": 22,    // ← 这个字段
    "Level": 4
  },
  "cell_identity": { "Ci": 20752955, "Tac": 4277, ... }
}
```

- LTE 协议层字段，单位 **Ts ≈ 32.55 ns**
- 代表 **UE↔基站 RTT 的时序测量**，含 UE 处理延迟（略偏大，不是纯物理传播时间）
- 业界换算：**`distance_m ≈ TA × 78.125`**
- 取值范围：0 ~ 1282（理论最大 ≈ 100km，实测 cell 多在 0~30）

**关键点：TA = 0 不代表距离是 0**，而是"≤78m（量化下限）"。密集城市小 cell 部署里绝大多数采样点都 TA=0，此时 TA 无距离区分度。

### 1.2 ss1 里没有 TA

ss1 sig_block 格式 `l,-100,-100,-12,46+` 只有 5 个字段（tech/ss/rsrp/rsrq/sinr），**没有 TA**。所以 TA 的可用来源仅限 cell_infos 且 `isConnected=1` 的对象。

### 1.3 etl_cleaned 覆盖率实测（batch 7 当前数据）

```
cell_infos 4G 总行数        1,301,137
其中 TA 非空                 463,683    (35.6%)
TA 有效（0 ≤ TA < 1000）     463,645    (35.6%)
TA + GPS 都有效              373,115    (28.7%)
TA 中位数                    2          (~156m)
TA p90                       23         (~1797m)
TA 上限哨兵                  INT_MAX    (需过滤)
```

## 2. Trilateration 可行性实验（失败）

### 2.1 方法

每条带 TA 的上报 = `(gps_lon, gps_lat, distance=TA×78)` → 以 GPS 为圆心、distance 为半径画圆。N 条上报得 N 个圆，理论上交集 ≈ 基站真实位置。

数学形式：`argmin_{x*, y*} Σ |haversine((x*, y*), (gps_i, lat_i)) - TA_i × 78|²`

实现：`scipy.optimize.least_squares`（Levenberg-Marquardt）。

### 2.2 6 个候选 cell 的 TA 分布

| cell_id | n_pts | TA 范围 | 适合 trilat？ |
|---|---|---|---|
| 22158786 | 31 | 0~0 全零 | ✗ 无几何信息 |
| 19234867 | 4 | 1~1 全同 | ✗ 无梯度 |
| 20945465 | 14 | 0~5 | 部分梯度 |
| 28436497 | 17 | 2~7 | 最佳候选 |
| 20151070 | 8 | 7~24 | 梯度最大，但样本少 |
| 107892531 | 4 | 2~3 | 样本少 |

### 2.3 实测结果（距 trusted_cell_library 中心的偏差）

| cell_id | n | TA 范围 | median→trusted | **trilat→trusted** | 结论 |
|---|---|---|---|---|---|
| 22158786 | 31 | 全 0 | 32m | n/a（TA 退化）| TA 无法做约束 |
| 19234867 | 4 | 全 1 | 135m | n/a（TA 退化）| TA 无法做约束 |
| 28436497 | 17 | 2~7 | **91m** | **308m** | ✗ trilat 差 3.4x |
| 20945465 | 14 | 0~5 | **32m** | **51m** | ✗ trilat 略差 |
| 20151070 | 8 | 7~24 | **114m** | **1391m** | ✗✗ trilat 灾难 12x |
| 107892531 | 4 | 2~3 | 361m | 199m | ≈ 略好但样本太少不可信 |

### 2.4 失败根因

1. **量化粒度 78m 太粗**：cell 28436497 TA 范围 2~7 对应 156~547m，78m quantization 误差占比过大
2. **城市多径效应**：信号反射 → TA 估算的距离系统性偏大 → trilat 把基站"推"到错位置
3. **远点 GPS 误差被放大**：cell 20151070 TA=7~24 的点在 500m~1.8km 远处，GPS 本身误差几十米，trilat 对这些点的噪声敏感，中心被拉偏 1.4km
4. **稀疏点数**：多数 cell 的 TA+GPS 样本只有 4~30 个，最小二乘在小样本下不稳定

### 2.5 可能反转的场景（不在当前数据集）

只在以下非典型场景 trilateration 可能比 GPS median 好：
- **农村 / 远郊大覆盖 cell**（TA 范围 10~80，距离 1~6km，相对量化误差小）
- **GPS 极稀疏但 TA 多**（罕见）
- **多基站联合定位 OTDOA**（同一 UE 同一时刻多个 cell 的 TA + 几何关系联立）—— 工程量远大于单 cell trilat

### 异常桶（TA=16-31）的内部结构

9,923 个 cell（占该桶 90%）呈现"卡死分布"：
- `ta_p50=21, ta_p90=22, ta_max=23`
- 非零率 0.99（几乎无 TA=0 的样本）
- 分布极度集中 21~23 的窄带

**初期怀疑是设备/网络 bug**，后续深挖（2026-04-22 补加分析）证实：

#### 真相：LTE TDD 频段特征 + 中国移动网络 + 占位值机制

按运营商分：
- **中国移动（46000）**：24.9% cell 中尖峰 ⚠️
- 联通（46001）：0.3%
- 电信（46011）：0.4%

按 earfcn 频段分：
| earfcn 范围 | 频段 | spike% (TA 21-23) | avg ta_p50 |
|---|---|---|---|
| 1300/1350/3590（FDD Band 3/7/8）| FDD | **< 1%** | **1-3** |
| 36000-43589（TDD Band 38/39/40/41）| TDD | **52-66%** | **17-19** |

**关键对比**：FDD 和 TDD cell 的真实 lib_p90 **几乎相同**（134m vs 111m）—— 说明物理覆盖范围一致，但 TA 值差 **13 倍**：

| duplex | avg_ta_p50 | TA×78 估算 | 实测 avg_lib_p90 | lib/ta_dist 比率 |
|---|---|---|---|---|
| **FDD_Band3** | 1.6 | **122m** | **134m** | **1.09** ✓ 完美对齐 |
| **TDD** | 18.2 | **1419m** | **111m** | **0.08** ✗ 差 13 倍 |

#### TDD 全样本 TA 分布 —— 双峰 + 中间空白

```
TA 0      (3.9%)   ← 极少
TA 1-5    (14.5%)  ← 低峰：真实 RRC CONNECTED 测量
TA 6-10   (4.3%)
TA 11-15  (0.7%)   ← 空白！
TA 16-20  (23.2%)  ← 高峰开始：占位值 ↓
TA 21     (21.8%)
TA 22     (17.3%)
TA 23     (7.4%)
TA 24-30  (6.2%)
```

**双峰 + 中间 11-15 几乎空白** —— 这是两种不同的测量状态混合：
- 低峰 TA 0-10（22.8%）：真实距离信号
- **高峰 TA 16-23（69.7%）：占位 / 锁定值**（非实时测量）

#### 技术根因推测

TDD 特有的 TA "不实时更新"场景：
1. **NSA 5G 锚点**：LTE TDD 是中国移动 5G NSA 锚点（Band 41 为主），UE 实际连 NR 后 LTE 侧 TA 不再更新
2. **CA 聚合载波**：UE 通信走主载波，TDD 辅载波的 TA 是上次测量值
3. **TDD 帧结构 Guard Period**：可能加了固定偏移
4. **中国移动特有的 TDD 参数配置**

#### 新结论：TA 的适用性

| 场景 | TA 是否可用 |
|---|---|
| **FDD 频段（earfcn < 36000 或 > 43589）**| ✓ TA × 78 公式准确（实测误差 < 10%）|
| **TDD 频段（earfcn 36000-43589），TA 0-10** | ✓ 少量真实测量，可用 |
| **TDD 频段，TA 11+** | ✗ 多数是占位值，不能作距离信号 |

#### 更新处理建议（替代之前的 "ta_quality_flag=suspicious_saturation"）

- **给每条带 TA 的记录打 `ta_duplex ∈ {fdd, tdd}`**（按 earfcn 判断，不需要额外字段）
- **coverage_class 分档按 duplex 分开**：
  - FDD：直接用 `ta_p90 × 78` 分档
  - TDD：用 `lib_p90 + earfcn 特征` 分档，**不用 TA 算距离**
- **反向异常检测（lib >> TA×78）仅对 FDD 适用**
- 之前发现的 "TA=16-31 病态群" 不再视为 bug，而是 TDD 正常表达

### 反向异常：lib_p90 >> TA×78（有价值的漏判信号）

这是 TA 最有价值的用法 —— 反向校验 lib_p90 是否被异常数据拉大。

| over_ratio | n | stable | 已标识（dual/collision/large_coverage/uncertain）|
|---|---|---|---|
| < 1 | 17,198 | 17,163 | 22 (0.1%) |
| 1-2 | 3,040 | 2,998 | 37 (1.2%) |

- 剩下 **27 个 stable 嫌疑漏判**，lib_p90 > TA×78 的 5~10 倍 → 现系统未抓到的异常候选

### 结论（相关性分析）

- **TA 和 lib_p90 整体相关性差**（Pearson 0.08）—— 但这不是 lib_p90 有问题，而是 TA 本身在不同桶的 overestimate 程度不一致
- **TA 小值桶（0-7）有可用物理规律**，可以作为 small/medium/large 分类基础
- **反向异常（lib >> TA）是稀有但高价值信号**，可以作为 drift_pattern 漏判的交叉校验
- **TA=21~23 的病态分布群**需要识别并从 coverage 计算中排除

## 3. Pivot：TA 统计量用于"基站类型分类 / 异常半径甄别"

Trilateration 失败但 TA 本身**作为统计信号是准的**。

### 3.1 核心洞察

单点 TA 不准（量化 + 多径 + 处理延迟），但 **per-cell 的 TA 分布（p50/p90/直方图）反映的是"用户群距基站的典型距离"**，是**协议层直接测量**，不依赖重建几何。

| TA 统计特征 | 基站类型推断 | 合理 p90 范围 |
|---|---|---|
| `ta_p50 ≤ 1, ta_p90 ≤ 3` | 密集小覆盖（城市核心）| < 200m |
| `ta_p50 ≈ 3, ta_p90 ≈ 10` | 中等城市基站 | 200m~800m |
| `ta_p50 ≥ 5, ta_p90 ≥ 20` | 大覆盖 / 郊区 / 县城 | 800m~5km |
| `ta_p50 ≥ 20, ta_p90 ≥ 40` | 农村 / 远程基站 | 5km+ |

### 3.2 能解决的"拍脑袋"问题

当前 `config/antitoxin_params.yaml::label_rules`：
```yaml
stable_max_p90_m: 1200
large_coverage_max_p90_m: 10000
```

这些阈值一刀切，**没有区分基站类型**。结果：
- **小 cell 漏判**：某个城市 cell 真实 p90 应 < 200m，但因为观测稀疏/漂移实际 p90=800m → 仍被判 stable（漏了异常）
- **大 cell 误判**：郊区 cell 合理 p90=1500m，直接卡在 `stable_max=1200` 被归入 `large_coverage` → 被下游"二等处理"

引入 TA 分类后，可做 **per-cell 动态半径阈值**：
```python
expected_p90 = ta_p90 × 78.125 × 1.3   # 1.3 留给 GPS 误差容忍
if measured_p90 < expected_p90 * 1.5:
    → stable
elif measured_p90 > expected_p90 * 3:
    → 异常（多质心 / 碰撞 / 漂移）
else:
    → 观察（处于合理波动区间）
```

### 3.3 辅助定位其他异常

- **覆盖范围异常**：`ta_p90 × 78m << measured_p90` → 信号实际能到的范围远小于 GPS 半径，可能是碰撞（两个同 cell_id 在不同物理位置）
- **弱 GPS 覆盖 cell**：TA 样本多但 GPS 样本少的 cell → 可以用 TA 分布作为"覆盖范围"的补充证据
- **基站迁移识别**：TA 分布在时间窗口里突变（比如从 p90=5 变成 p90=15）→ 基站位置可能改变

### 3.4 其他用途

- **trusted_cell_library 加一个新字段 `coverage_class ∈ {small, medium, large, special}`**，下游路由/补数/异常判定都可引用
- **Step 4 的 donor 选择**：同 cell_id 候选时优先选 `coverage_class` 一致的 donor
- **UI 展示**：cell 详情页展示 TA 直方图，直观判断该基站类型

## 4. Pipeline 改造方案（和 cell_origin 透传类似）

### 4.1 现状：TA 仅存在于 Step 1 输出

```
raw_gps
  ↓  parse.py 提取 signal_strength.TimingAdvance
etl_parsed          → timing_advance ✓
  ↓
etl_cleaned         → timing_advance ✓（丢失即将发生）
  ↓
path_a_records      → 没有（SELECT e.* 会自动带，但后续表不保留）
_profile_path_b_records → 没有
enriched_records    → 没有
candidate_seed_history → 没有
snapshot_seed_records → 没有
cell_sliding_window → 没有
trusted_cell_library → 没有
```

### 4.2 改造范围（建议一次跟 cell_origin 类似的全链路改动）

- 给 7 张表加 `timing_advance INT`（允许 NULL，`cell_origin='ss1'` 的行天然为 NULL）
- 各 INSERT SELECT 显式透传
- Step 5 聚合：在 `build_cell_metrics_base` 或 trusted_cell_library 填充时加：
  - `ta_p50, ta_p90, ta_max, ta_mean, ta_n_obs, ta_nonzero_rate`
  - `coverage_class` 按上表规则打标
- `antitoxin_params.yaml` 加 `coverage_class` 分档的动态 `max_p90_m`，替换现固定阈值

### 4.3 路径不推荐：单独做一次 TA 跑

- 需要单独建聚合表，维护代价高
- 下游判定若引用 TA 需要做 JOIN，每批都跑
- 综合不如一次改到底

### 4.4 建议执行时机

**下一次全库重跑时同步引入**（类似本次 `DEDUP-V2` 的做法）。不要为了单独引入 TA 再跑一次全库（耗时大）。当前 `DEDUP-V2` 重跑已经在路上，下次有全库跑的需求时（例如发现新异常模式后），把 TA 透传打包进去。

### 6.2 ODS-023 规则细节（详见 `01b_数据源接入_处理规则.md`）

**名称**：LTE TDD 占位 TA 剔除
**作用层**：Step 1 clean.py（ODS_RULES 列表）
**触发条件**：
- `cell_origin = 'cell_infos'`（ss1 行本就 TA NULL，天然无影响）
- `tech_norm = '4G'`
- `freq_channel BETWEEN 36000 AND 43589`（LTE TDD Band 38/39/40/41 等）
- `timing_advance >= 16`

**动作**：`timing_advance = NULL`

**依据**：
- 2026-04-22 研究发现 LTE TDD 样本 TA 呈双峰：低峰 0-10（22.8%，真实测量）+ 高峰 16-23（69.7%，占位值）
- 实测 TDD cell 的物理 p90 和 FDD 几乎相同（111m vs 134m），但 TA 差 13 倍 → TDD 高峰 TA **不是距离信号**
- 中国移动 TDD 基站中 46% 全 cell 都是占位（基站侧 cell-level 禁用 TA），23% 近似纯占位，合计 69%

**预期影响**：
- FDD 样本完全不受影响
- TDD 样本剔除 76.6%，保留低峰 23.4%（真实 TA）
- `etl_cleaned` 里 `timing_advance IS NOT NULL` 的行数减少约 20-25%（因为 TDD 占 4G 的一半）

### 6.3 Step 5 聚合产出的字段（写入 trusted_cell_library）

| 字段 | 类型 | 含义 |
|---|---|---|
| `ta_n_obs` | INT | 有效 TA 样本数（timing_advance IS NOT NULL 的行数）|
| `ta_p50` | INT | 有效 TA 中位数（单位 1 TA ≈ 78.125m，仅 FDD 可按此换算）|
| `ta_p90` | INT | 有效 TA p90 |
| `freq_band` | TEXT | 'fdd' / 'tdd' / 'mixed' / 'unknown'，从 `MODE(freq_channel)` 判断 |
| `coverage_class` | TEXT | 'small' / 'medium' / 'large' / 'ta_invalid' |

#### freq_band 判断规则

```sql
CASE
  WHEN earfcn_mode BETWEEN 36000 AND 43589 THEN 'tdd'
  WHEN earfcn_mode < 36000 OR earfcn_mode > 43589 THEN 'fdd'
  ELSE 'unknown'
END
```

#### coverage_class 分档规则（草案，待最终讨论）

```sql
CASE
  WHEN ta_n_obs < 5 THEN 'ta_invalid'     -- 样本太少
  WHEN freq_band = 'fdd' THEN
    CASE WHEN ta_p90 <= 3 THEN 'small'    -- ≤ 234m
         WHEN ta_p90 <= 10 THEN 'medium'  -- ≤ 781m
         ELSE 'large' END
  WHEN freq_band = 'tdd' THEN
    -- TDD 剔除占位后剩 0-15，分档参考值略放宽
    CASE WHEN ta_p90 <= 3 THEN 'small'
         WHEN ta_p90 <= 10 THEN 'medium'
         ELSE 'large' END
  ELSE 'unknown'
END
```
