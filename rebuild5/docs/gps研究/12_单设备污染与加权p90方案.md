# 单设备 GPS 污染 与 加权 p90 方案

> **创建日期**：2026-04-22
> **上下文**：TA 反向异常分析发现 cell 19450676 被 **单个 GPS 漂移设备 GGUC53ROTN4NIFCUH** 污染 p90 到 6478m。追到 raw_gps 层发现 cell_infos 被 ODS-019 正确过滤（0 行进 etl_cleaned），但 ss1 通道因缺少 age 字段 + forward-fill 机制放大污染，让 58 个陈旧 cell 条目进了 cell_sliding_window。
> **讨论范围**：DEDUP-V2.1（ss1 record_id 去重）+ 方案 B（逆频加权 p90）的决议和效果
> **研究结论**：组合方案能降低 p90 30%，但对"单设备全远点"的极端病例（如 GGUC53）不能根治。方案 D（设备一致性过滤）**暂不做**。

---

## 1. 案例：cell 19450676

### 1.1 污染特征

- **业务键**：`operator=46000, lac=4427, cell_id=19450676, tech=4G`
- **batch 7 trusted 画像**：`drift_pattern=large_coverage, p50=165m, p90=6478m`
- **真实物理位置**：约 `(116.705, 40.075)`（主簇）
- **TA 说这是小 cell**：ta_p90=1（即 78m 覆盖）

### 1.2 谁在污染？

7 设备 105 点，分布如下：

| 设备 | 上报点数 | 距主簇距离 | 备注 |
|---|---|---|---|
| **GGUC53ROTN4NIFCUH** | **57** | 205~9327m | **全部远点，无一在主簇** |
| H561EK:0KGZ:2APN:NY000 | 19 | 11~27m | 主簇内稳定 |
| WIQQV9C1VA6C0SXDX | 10 | 233~253m | 主簇边缘 |
| HCD1EK / GTU1EK | 各 8 | 近 / 240m | 合理 |
| GOV2W5 / ARVRU5 | 2 / 1 | 近 / 344m | 少量 |

### 1.3 raw_gps 追源：为什么 GGUC53 的上报能漏过 ETL

追到 raw_gps 发现：

**cell_infos 通道**：该设备 38 条 raw_gps 中，cell_infos 里 `Ci=19450676` 的 `timeStamp=4786201430` **从 12-03 开始就卡住不动**（表明是 SDK 缓存标识）。`age` 累积到 7.4~10.8 小时。

**ODS-019 正确过滤了** —— etl_cleaned 里 cell_origin='cell_infos' 0 行 ✓

**ss1 通道**：38 条 raw_gps 里 30 条的 ss1 含 `l,19450676,4427,46000`，parse 后产生 58 条 ss1 子记录。原因有两层：

1. **ss1 没有对象级 timeStamp**，每条子记录只有 `ts_block`（测量时刻），**无法判断 cell_id 是否卡住**
2. **forward-fill 继承** + ODS-021 sig-tech 匹配机制：一条 raw_gps 里 sig=l 的子记录即使 cell_block='1'（空），会继承前一条含 cache cell_id 的 cell_block，再和 l_sig 配对放行

→ 结果：**ss1 通道没有任何机制检测 cell_id 卡住**，cache 源源不断进入 cell_sliding_window。

### 1.4 ss1 采集机制（澄清）

> **关键背景**（用户补充）：ss1 不是和 cell_infos 同一时间采集的信号快照。它是**上次上报后到本次上报前累积的历史信号报告**，本次报文上报时一起带回。以 ss1 内部最大 `ts_block` 为锚点，往回推 1 小时以内的子记录判定有效（= 现在 ODS-020 的语义）。

因此：
- **不能用 cell_infos 的 age 信号跨通道过滤 ss1**（两者采样时间基准不同）
- ODS-020 1h 锚点只过滤"信号测量本身是否陈旧"，管不住"cell_id 作为标识符被卡住"
- TA 只属于 cell_infos 的当前连接，**ss1 结构上就不带 TA**（parse.py 把 ss1 的 timing_advance 强制设 NULL）

### 1.5 这是一种新的污染模式

不同于之前记录的 "一次性 did"（dev 只上报 1 次）或 "cache stale"（cell_infos 陈旧）：

**"单设备 GPS 漂移持续污染"**：
- 该设备上报次数多（本案 38 次）
- 所有上报的 GPS 位置全在远处
- cell_infos 被 ODS-019 拦住 ✓
- ss1 绕过所有现有规则进入 sliding_window ✗
- 用户群里这种设备占比可能不小（需量化）

## 2. 讨论的方案

评估了 4 个方向：

| 方案 | 描述 | 本案 p90 估计 | 复杂度 |
|---|---|---|---|
| A | 每设备取 median 为代表点 | 仍 ~5km（没救）| 低 |
| **B** | **逆频加权 p90（每设备权重 1）** | **~4.7km（-30%）** | **中** |
| C | Huber-type M-estimator 迭代 | ~200m | 高 |
| **D** | **DBSCAN + 主簇占比设备过滤** | **~200m** | 中 |

方案 A 因为 GGUC53 代表点也在远处，p90 仍被污染，不可用。

方案 C 效果最好但工程复杂（迭代 + 参数调校）。

方案 D 最彻底但需要引入 DBSCAN 主簇检测。

## 3. 本次决议：DEDUP-V2.1 = record_id 去重 + 方案 B 加权（不做 D）

### 3.1 决策要点

**做**：
- **ss1 专用 record_id 去重**：每 `(operator, lac, cell_id, tech, dev, 'ss1', record_id)` 保留 1 条
  - 消除同一条 raw_gps 内 forward-fill 产生的重复 ss1 子记录
- **ODS-020 阈值从 1h → 3h**：补偿 record_id 去重造成的合法样本损失
- **方案 B 加权 p90**：每点权重 `1 / n_device_pts`，用累积权重算加权百分位
  - 限制单设备总贡献为 1，防止刷屏设备主导

**不做（本期）**：
- 方案 D 设备一致性过滤：工程上需要 DBSCAN 前置，本期先看方案 B + record_id 去重的综合收益再决定
- 方案 C M-estimator 迭代

**保留**：
- cell_infos 这一路继续用 `(..., dev, cell_origin='cell_infos', 5min_bucket)` 的 DEDUP-V2 去重
- 其他阈值（`gps_anomaly_filter.max_anomaly_ratio=10%` 等）不动

### 3.2 对 cell 19450676 的实测效果

| 策略 | n_pts | p50 | p90 | 改善 |
|---|---|---|---|---|
| 现状（无权 + 无 rid 去重）| 105 | 281 | **6750** | baseline |
| 方案 B 单独 | 105 | 154 | **4739** | -30% |
| **DEDUP-V2.1（rid 去重 + 方案 B）** | **68** | **154** | **4718** | **-30%** |

**关键发现**：rid 去重和方案 B **不叠加对加权 p90 的效果**。
- rid 去重：点数维度压缩（57→26）
- 方案 B：每设备总权重归一化到 1（与点数无关）
- 叠加后设备权重依然 14.3%（1/7），超过 10% p90 污染线

但**综合收益**仍在：

| 收益点 | 是否受益 |
|---|---|
| 加权 p90（本案关心）| ✗ 组合不叠加 |
| 无权 p90（老算法 / 兼容 UI）| ✓ 6750 → 5723（-15%）|
| 存储 / compute | ✓ 点数压 35% |
| anomaly_filter 小样本 ratio 失效 | ✓ 分母扩大，10% 阈值更难失效 |

### 3.3 方案 B 根治的条件

方案 B 能让 p90 回归真实值的条件：**污染设备数 / 总设备数 ≤ 10%**。

本案 1/7=14.3% > 10%，p90 救不回。反例：若有 ≥ 10 个正常设备 + 1 个 GGUC53 污染，p90 就能回到 ~190m（1/11=9% < 10%）。对样本多的 cell 方案 B 足够。

### 3.4 GGUC53 这类极端病例的兜底（本次不实施）

对"设备数少 + 单设备全部远点"：

- 方案 D 可以完全剔除 GGUC53（它主簇内占比 = 0%）
- 但工程复杂度高，本期不做
- **后续如果发现这类极端 cell 数量多**，再考虑引入方案 D

## 4. 待实施（下次全库重跑时同步）

### 4.1 代码改动范围

- `parse.py::_parse_ss1`：在 ss1 sub-record 层加 `DISTINCT ON (record_id, cell_id, lac, operator_code, tech_norm)` 去重
- `antitoxin_params.yaml::etl_ss1.max_age_from_anchor_sec`: `3600 → 10800`
- `maintenance/window.py::build_cell_core_gps_stats`：把 p50/p90 算法改成加权百分位
  - 每点权重 = `1 / COUNT(*) OVER (PARTITION BY dev_id)`
  - p50/p90 用累积权重 > `0.5 / 0.9 * total_weight` 的第一个 dist_m
- 同步改 `docs/01b_数据源接入_处理规则.md § 下游去重策略`：DEDUP-V2 → DEDUP-V2.1，说明 ss1 的 record_id 去重 + 方案 B 加权 p90

### 4.2 验收指标

- cell 19450676：p90 (加权) 从 6478m 降到 ≤ 5000m
- 正常 cell 的 p90 (加权) 应比 p90 (无权) 略小（因为加权抑制了异常点的比例）
- 存储层面 cell_sliding_window 行数减少 ~35%（ss1 通道的 forward-fill 冗余被压掉）
- anomaly_filter 触发率上升（因为分母扩大）

### 4.3 不处理的遗留问题

- GGUC53 这类"单设备全远点"的极端 cell → 方案 D 后续做
- Lac 7106/7108/7110 片区的异常集群 → 独立研究
