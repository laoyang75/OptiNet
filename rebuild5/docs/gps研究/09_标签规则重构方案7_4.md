# 09 标签规则重构 · 方案 7.4

> **创建日期**：2026-04-19
> **对应任务**：rebuild5/prompts/24_ui_label_alignment_and_refinement.md §1.2
> **数据基线**：batch 7 (6 LAC 样本)，总 cell 数 418,834
> **评估方法**：3,000 cell 分层抽样 + PostGIS ST_ClusterDBSCAN 离线模拟

---

## 1. 背景问题

Batch 7 的标签分布暴露了三个结构性错误：

| 标签 | 当前占比 | 问题 |
|---|---|---|
| insufficient | 35.1%（146,926）| 大量有数据的 cell 被误判 |
| dual_cluster | 0.5%（2,159）| **98% 是假阳性**（p90<1300m 却标多簇）|
| uncertain | 0.02%（76）| **98.7% 是假阳性** |
| migration | 0（1）| **100% 是假阳性** |

### 1.1 insufficient 35% 的根因（4-9 gps 段诊断）

4-9 gps_valid 段里 insufficient 与 stable 的数据形态几乎相同（dev=5, days=4-5, p90=47-51m），但命运截然不同：

| 命运 | 数量 | 路径 |
|---|---|---|
| **insufficient** | 106,729 | candidate_hit=true + k_eff=0（DBSCAN 跑了但没聚出 ≥10 dev-day 的簇） |
| **stable（非候选短路径）** | 35,693 | candidate_hit=false，根本没跑 DBSCAN，按 p90<1200m 直接判 stable |
| **stable（聚簇成功）** | 15,921 | candidate_hit=true + k_eff=1 |

**规则不对称**：数据稍多触发了候选的 cell 反而更容易掉 insufficient。

### 1.2 多质心假阳性的根因

全量 dual_cluster/uncertain/migration 里 **98%** 的 cell 存储 p90<1300m —— 整体紧凑，不该被判多簇。label_engine 当前对所有候选 cell 无条件跑多质心分析，没有"整体紧凑就直接判 stable"的短路规则。

## 2. 抽样评估方案

### 2.1 抽样

3,000 cell 分 6 层抽样，按数据量进一步分 low / mid / high：

| 层 | 数量 | 说明 |
|---|---|---|
| stable × 3 bucket | 1,800 (600 ×3) | 60% 占比参考组 |
| insufficient × 3 k_raw | 900 (300 ×3) | 核心研究对象 |
| dual_cluster | 100 | 现有多簇样本 |
| large_coverage | 100 | 现有大覆盖 |
| uncertain | 50 | |
| oversize_single | 10 | |
| migration | 1 | 样本内全量 |
| **合计** | **2,961** | |

### 2.2 评估方法

- 对每个 cell 从 `cell_sliding_window` + `enriched/snapshot_seed_records` 拉 `gps_fill_source_final='raw_gps'` 的点
- 按 `(dev_id, date)` 去重
- 用 PostGIS `ST_ClusterDBSCAN(eps=250m, minpoints=4)` 本地重聚类
- 在不同 `min_cluster_dev_day_pts` 门槛下计算 `k_eff`

## 3. 规则演进历程

### 3.1 方案 7（初版）

核心改动：
1. `min_cluster_dev_day_pts`: 10 → 4
2. 取消"非候选短路径"

**评估结果**：insufficient 0%（过度激进），真稀疏 cell 被错判 stable。

### 3.2 方案 7.2（加真稀疏保护）

追加：
- 规则 0：`dedup_pts<4 OR dev<2 OR day<2 → insufficient`

**评估结果**：insufficient 降到 13.6%，但**多质心命中 151 → 654 有 398 个假阳性**（p90<1300m）。

### 3.3 方案 7.3（加 p90<1300 短路）

追加：
- 规则 1：`p90<1300m → stable`（在规则 0 之后）

**评估结果**：多质心假阳性消除。但"规则 0 对 stable 强制稀疏检查"与 Step 3 状态机冲突 —— Step 3 晋级已经保证数据量充足。

### 3.4 方案 7.4（规则顺序修正 · 最终版）

关键认识：**状态机原则**，Step 5 不能质疑 Step 3 晋级决策。稀疏保护只在多质心层才有意义（因为多质心需要更多证据）。

---

## 4. 方案 7.4 最终规则

按顺序命中即停：

```
规则 1：p90 < 1300m → stable
  （Step 3 已验证数据量足以支撑单簇；整体紧凑不做多质心分析）

规则 2：p90 ≥ 1300m → 进入多质心分析

  2a 稀疏门槛（只在多质心层检查）:
     dedup点 < 8 OR 设备 < 3 OR 天数 < 3 → insufficient
     （多簇判定对数据量有更高要求）

  2b 按 k_eff 判:
     k_eff=0 → insufficient（数据够但聚不出簇）
     k_eff=1 + p90 ≤ 10km → large_coverage
     k_eff=1 + p90 > 10km → oversize_single
     k_eff=2 + 距离 ≥ 100km → collision
     k_eff=2 + 近 + 重叠 ≥ 0.5 → dual_cluster
     k_eff=2 + 近 + 重叠 = 0 → migration
     k_eff=2 其他 → uncertain
     k_eff ≥ 3 → uncertain/dynamic（按原有 k≥3 规则）
```

### 4.1 参数落地

| 配置项 | 旧值 | 新值 |
|---|---|---|
| `multi_centroid_v2.min_cluster_dev_day_pts` | 10 | 4 |
| `multi_centroid_v2.multi_centroid_entry_p90_m` | - | 1300 |
| `multi_centroid_v2.min_total_dedup_pts` | - | 8 |
| `multi_centroid_v2.min_total_devs` | - | 3 |
| `multi_centroid_v2.min_total_active_days` | - | 3 |

### 4.2 代码改动点

- `config/antitoxin_params.yaml`：新增 4 个参数 + 修改 min_cluster_dev_day_pts
- `backend/app/profile/logic.py`：`load_multi_centroid_v2_params` 暴露新参数
- `backend/app/maintenance/label_engine.py`：重构 CASE WHEN；在候选阶段加入 `dedup_pts/dedup_dev/dedup_day` 统计

## 6. 遗留问题（不在本轮范围）

### 6.1 Step 3 晋级逻辑审查

样本内发现 **241 个 stable cell** 的 raw_gps dedup 数 <4 或设备<2 或天数<2（真稀疏形态），中位 gps_valid=3 / dev=1。这些 cell 通过什么路径进入 Step 5 并被判 stable 需查：
- Step 3 晋级规则是否允许过稀疏 cell 晋升？
- 是"非候选短路径"的副作用（candidate_hit=false → 按 p90 直接判 stable）？

按状态机原则，Step 5 信任 Step 3 输出，不重查稀疏。此问题作为 **Step 3 审计**独立工单处理。

### 6.2 候选触发器简化

方案 7.4 下所有 cell 按 p90 直接分流（<1300 → stable；≥1300 → 多质心分析）。`_label_candidates` 的复杂触发条件可简化为：

```
p90_radius_m >= 1300  -- 唯一候选条件
```

可在后续清理迭代中落实。
