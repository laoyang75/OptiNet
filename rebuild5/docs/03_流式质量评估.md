# Step 3: 流式质量评估

Step 3 是 `rebuild5` 运行主链中负责”准入判定 + 三层级联 + 冻结快照”的版本化步骤。它接收 Step 2 输出的 `profile_base`，把当前批次新增或继续积累的 Cell 证据与 `trusted_snapshot_t-1` 合并，对**候选域**（尚未进入正式发布库的对象）做全量重评估，最终产出本批冻结快照与准入结果，供 Step 5 维护并发布正式库。

**职责边界**：Step 3 只负责候选域的准入评估。对于上一版已发布到正式库的对象，Step 3 仅在快照中保留其继承展示状态（carry-forward），不对其做当前批维护更新——当前批已发布对象的维护由 Step 5 负责。这一分工与 Step 2 的 Path A 绕过逻辑一致：Path A 命中正式库的记录不进入 Step 3，因此 Step 3 天然不具备对已发布对象做当前批维护的输入条件。

这一步不是“给 Cell 打标签”这么简单。它决定哪些对象能进入可信链路，哪些对象只能继续观察，哪些对象已经具备被 Step 4 用作知识补数锚点的资格。

## 模块定位

### Step 3 的角色

| 项目 | 说明 |
|------|------|
| 模块角色 | Step 3 负责”候选域全量评估 + 可信准入 + 三层 snapshot”，不是完整画像维护终点 |
| 直接输入 | `profile_base`（来自 Step 2 Path B，未命中可信库且具备 GPS 证据的 Cell 基础指标） |
| 参考输入 | `trusted_snapshot_t-1`（上一版冻结快照，用于 diff / 展示继承）、上一版正式库 `trusted_cell_library`（用于识别哪些 Cell 属于已发布域）、`collision_id_list`（来自 Step 5.1） |
| 评估对象 | 候选域 Cell（尚未进入正式发布库的对象）；已发布对象仅做继承展示；BS / LAC 完全由下属对象聚合派生 |
| 输出 | `trusted_snapshot_t`、`snapshot_diff`、`step3_run_stats` |
| 本步不做 | 漂移分析、碰撞确认、多质心检测、防毒化、退出确认 |

说明：
- Step 2 的 `profile_base` 是当前批次的直接增量输入，Step 3 的判定对象是”候选域中所有已注册 Cell 的累计状态”。
- Step 3 需要把本批 `profile_base` 与历史评估池 / 上一版快照中的候选对象状态合并，再统一重评估，不能只对新增对象做局部打分。
- 对于上一版已发布到正式库的对象，Step 3 仅在 Cell 层快照中继承其上一版发布状态（carry-forward rows），不重新评估——这些对象的当前批维护由 Step 5 负责。
- 这里的“继承展示”是指：`trusted_snapshot_t` 作为完整冻结视图，必须先得到“候选域当前评估结果 + 已发布域 Cell 继承展示”的最终 Cell 视图，再由该最终 Cell 视图继续上卷出 BS/LAC snapshot；不能只生成候选域 current-only 结果，再把已发布域留给 Step 5 补洞。
- `profile_base.independent_devs` 在 Step 3 中沿用旧实现口径映射为 `distinct_dev_id`，便于和 `profile_params.yaml` 中的阈值表达保持一致。

### Step 3 的核心目标

| 目标 | 含义 |
|------|------|
| 产出当前批准入结果 | 把 Cell 的空间质量和观测成熟度沉淀为当前批可被维护的准入对象 |
| 维持三层级联关系 | 由 `Cell -> BS -> LAC` 自下而上派生状态与资格 |
| 形成冻结快照 | 批末产出 `trusted_snapshot_t`，供 Step 5 维护发布，供下一批评估参考 |
| 为后续步骤供给维护输入 | Step 5 读取当前批冻结快照做深度维护并发布正式库 |

### 候选域全量评估而不是只评估新对象

每批新数据进入后，Step 3 对候选域中所有已注册 Cell 做全量重评估，原因如下：

- 单批样本通常不足以形成稳定的生命周期判断，需要累计多批证据。
- 新数据会同时改变已有 Cell 的 `independent_obs`、`distinct_dev_id`、`observed_span_hours` 和空间半径。
- BS / LAC 完全由下层聚合而来，只要任一 Cell 状态变化，上层对象都要重新上卷计算。
- 只有候选域全量重评估 + 已发布域继承展示，才能保证 `trusted_snapshot_t` 是当前批次下的完整一致视图。

注意：已发布到正式库的对象不在此"全量重评估"范围内。它们在快照中以 carry-forward 方式展示上一版发布状态，当前批的维护更新由 Step 5 负责。

### Cell 先行、逐级上卷

```text
LAC (operator_code, lac)
  └── BS (operator_code, lac, bs_id)
        └── Cell (operator_code, tech_norm, lac, cell_id)
```

约束：
- Step 3 直接接收的业务输入只有 Cell。
- BS 不直接读取原始报文，只读取下属 Cell 的聚合结果。
- LAC 不直接读取原始报文，也不跳过 BS 直接从 Cell 重判。
- 上层对象的 `lifecycle_state`、`anchor_eligible`、`baseline_eligible` 都来自保守上卷，而不是重新定义一套独立原始指标体系。

### 流式评估可以替代批量初始化

`rebuild4/docs/02_profile/06_流式评估.md` 已验证“逐天累积的流式画像”和全量批量计算在数学上等价，这也是 Step 3 采用流式主链的依据。

| 指标 | Day 7 流式 vs ETL 批量 |
|------|------------------------|
| 质心偏差 | `0.00m` |
| 生命周期一致率 | `98.9%` |
| P90 差异 | `0.00m` |

关键结论：
- Day 3 已达到可用水平。
- Day 5 已达到生产水平。
- Day 7 与同算法 ETL 批量结果等价。
- 剩余差异主要来自碰撞边界，不是流式评估本身的系统性偏差。

补充说明：
- 当前主线中，Step 2 的 `profile_base.center_lon / center_lat / p50_radius_m / p90_radius_m` 已改为”主热点 seed + 核心点过滤”口径，不再直接用全部 GPS 点计算。
- Step 3 读取的是经过这层位置稳健化之后的 `profile_base`，而不是全量原始 GPS 的粗暴半径。
- **2026-04-22 起**，Step 2 的 `_profile_gps_day_dedup` 去重键从 `(cell, dev, date)` 改为 `(cell, dev, cell_origin, 5min_bucket)`（规则 `DEDUP-V2`，详见 `docs/01b_数据源接入_处理规则.md`）。`profile_base.p50/p90/center` 口径与 Step 5 `cell_core_gps_stats` 对齐，样本密度恢复 1.5~4.4 倍但不影响 `independent_obs`（per-minute 口径不变），因此**不影响 Step 3 准入判定**。

### Step 3 明确不做的事情

| 不做项 | 归属步骤 | 原因 |
|--------|----------|------|
| 漂移分析（`stable / migration / collision / large_coverage`） | Step 5 | 需要多日轨迹和异常分桶，不属于准入判定 |
| 碰撞确认 | Step 5.1 / 5.5 | Step 5.1 产出冲突编号清单，Step 5.5 做异常子集深度确认 |
| 多质心检测 | Step 5.5 | 只对异常子集做深度计算 |
| 防毒化判断 | Step 5 | 属于可信库维护与阻断逻辑 |
| 退出确认（`dormant -> retired`） | Step 5.4 | 属于长期维护链路 |

结论：Step 3 只回答一件事——“这个对象现在是否足以进入可信链路，以及它的冻结状态是什么”。

## 入库前置过滤（可配置）

前置过滤控制“哪些对象允许进入评估池或继续参与本批准入”。过滤本身不是质量分级，但它决定 Step 3 的输入边界和统计口径。

### 制式过滤

| 配置项 | 默认口径 | 说明 |
|--------|----------|------|
| 制式白名单 | `4G / 5G` | 当前主链默认只处理 4G / 5G |
| 制式黑名单 | `2G / 3G` | 旧制式可配置为过滤或放行 |
| 过滤统计 | 必须记录 | 过滤后的对象不进入后续评估，但要写入运行统计 |

说明：
- 是否处理 2G / 3G 属于业务配置，不是算法常量。
- 被过滤对象不进入 `trusted_*_library`，也不参与三层级联统计。

### 地区过滤

| 配置项 | 说明 |
|--------|------|
| LAC 白名单 | 允许指定只评估部分 LAC（如 `sample_6lac`） |
| 运营商白名单 | 可按运营商范围限制评估 |
| 过滤结果 | 超出范围的对象不进入本批可信链路，但仍保留原始 ETL 数据 |

适用场景：
- 开发阶段只处理样本 LAC。
- 控制侧链做局部重算时，只放开受影响区域。
- 多数据集共存时，把 Step 3 评估范围和当前 `dataset_key` 强绑定。

### GPS 有效性（硬约束）

**设计原则**：Step 3 的每个独立观察点（`profile_obs` 的一行）必须 GPS 有效。无 GPS 的分钟级记录不参与 `independent_obs` 计数，也不参与空间评估。

| 项目 | 规则 |
|------|------|
| `profile_obs` 构建 | `WHERE lon_raw IS NOT NULL AND lat_raw IS NOT NULL AND gps_valid`（硬过滤，不是 FILTER 聚合修饰） |
| `independent_obs` | 等于 `profile_obs` 的行数，天然只数 GPS 有效的分钟级观察点 |
| 空间判级前提 | 无有效 GPS 的对象不能计算质心和半径，也不能通过空间门槛 |
| 注册语义 | `is_registered` 仅表示对象存在，不等于它已具备空间评估条件 |
| 页面展示 | 无有效 GPS 的对象应在观察态 / 等待态中体现原因 |

说明：
- 现阶段主链中，缺少有效 GPS 的未命中记录会在 Step 2 直接丢弃（Path C），不会成为 Step 3 的输入。
- Path B cell 层过滤只要求 cell 至少有过一条 GPS（`has_raw_gps=true`）；observation 层必须再次筛除 cell 内无 GPS 的分钟级记录，否则 `independent_obs` 会被无位置的信号报文虚高。
- 此为**基本设计原则**，下游所有计数、晋级判定、空间评估均基于"每观察点必有 GPS"这个前提。

## 3.1 Cell 质量评估

### 评估流程

Step 3 对 Cell 的处理分成“合并累计证据”和“按规则重判”两层：

```text
profile_base (本批增量)
    + trusted_snapshot_t-1 / 历史评估池
    + collision_id_list
        |
        +--> 按 (operator_code, tech_norm, lac, cell_id) 合并累计指标
        |
        +--> 统一计算 lifecycle_state
        |
        +--> 统一计算 is_registered / anchor_eligible / baseline_eligible
        |
        +--> 生成 Cell snapshot + diff
```

判级使用的核心累计指标：
- `independent_obs`
- `distinct_dev_id`（兼容 Step 2 的 `independent_devs`）
- `gps_valid_count`
- `observed_span_hours`
- `p50_radius_m`
- `p90_radius_m`
- `center_lon` / `center_lat`
- `active_days`
- `rsrp_avg` / `rsrq_avg` / `sinr_avg`

**当前实现口径说明**：Step 3 仍按批次运行，不存在“分钟级评估调度”。这里的“分钟级”仅指 Step 2 对批内独立观察点的去重口径：同一 Cell 在同一分钟内的重复上报压成 `1` 个独立观察点，用来避免重复上报把观测量虚高。Step 3 当前采用“当前批 `profile_base` + 历史候选池累计证据”的批次重算模式；是否进一步保留更细粒度原始证据，属于后续精度增强问题，不影响当前按批次评估的主链设计。

### 生命周期判定规则（可配置阈值）

Cell 晋级以 `independent_obs`（独立观察点数）为**单一且充分**的判定维度：

| 状态 | 判定条件 | 参数来源 |
|------|----------|----------|
| `waiting` | `independent_obs < 3`（证据明显不足） | `cell.waiting.min_independent_obs=3` |
| `observing` | `3 <= independent_obs < 10` | 由 `waiting` 和 `qualified` 反推 |
| `qualified` | `independent_obs >= 10` | `lifecycle.qualified.min_obs=10` |
| `excellent` | `independent_obs >= 30` | `lifecycle.excellent.min_obs=30` |
| `dormant` | 已入库对象长期无新数据，暂停活跃维护 | 数值阈值待挂到 `retention_params.yaml` |
| `retired` | `dormant` 后持续静默，确认退出可信赖库 | 数值阈值待挂到 `retention_params.yaml` |

补充说明：
- `independent_obs` = GPS 有效的分钟级去重观察点（见 §GPS 有效性）。这个口径已经蕴含了"每观察点独立 + 每分钟去重"的多维约束，所以它是**充分指标**，不需要再叠加 `distinct_dev_id` / `p90_radius_m` / `observed_span_hours` 作为独立门槛：
  - `distinct_dev_id >= N` 冗余：观察点本身已经分钟+天去重，观察点达标时设备数自然 ≥2；
  - `p90_radius_m < N` 不适合放在晋级阶段：空间质量需要 Step 5 的质心重算 / 外点剔除后才可信，Step 3 的 p90 只是 profile_base 的初步值，强加门槛会误伤大覆盖合法 cell；
  - `observed_span_hours >= 24` 不是门槛：它是**累积观察时间窗**的计算粒度（每 24 小时累积一次），当 1 天观察点不足就用 2 天、2 天不够用 3 天…… 累积直到够晋级门槛；此前曾误实现为"只用最近 24 小时的滑动窗口"，已修复为累积口径。
- Step 3 直接消费的碰撞阻断信号是 `is_collision_id`，它来自 Step 5.1 产出的冲突小区编号清单（`collision_id_list`）。
- `dormant` / `retired` 的语义已冻结，但数值阈值尚未从 `retention_params.yaml` 外化，Step 3 不得硬编码新的天数门槛。

### 候选池合并约束

候选池合并"当前批 profile_base + 历史评估池"时，必须遵守以下约束：

| 指标 | 合并方式 | 错误方式（已修复） | 修复依据 |
|------|---------|------------------|---------|
| `p50_radius_m` / `p90_radius_m` | 按 `gps_valid_count` 加权平均 | ~~GREATEST（取最大值）~~ | 脏 GPS 单点不应永久污染半径 |
| `observed_span_hours` | 用 `first_obs_at` 和 `last_obs_at` 的首尾差值 | ~~批次内跨度直接求和~~ | 跨天 Cell 的真实跨度应是首尾时间差 |
| `independent_devs` | 两批取 GREATEST（保守估计） | ~~直接相加~~ | 同一设备跨批次出现不应重复计数 |
| `independent_obs` | 直接相加 | （无变化） | 观测点跨批次天然不重复 |
| `center_lon` / `center_lat` | 按 `gps_valid_count` 加权平均 | （无变化） | 保持现有逻辑 |

设计哲学：每批新数据的加入，Cell 的质量指标**只应变好或保持**，不应因单次脏数据永久恶化。候选池是"积累到及格就放行"的机制。

### 三层资格判定

生命周期状态回答“当前处于哪个阶段”，资格字段回答“能不能被后续步骤使用”。两者独立维护，不混成一个字段。

| 资格 | 判定规则 | 阈值来源 |
|------|----------|----------|
| `is_registered` | 首次出现可解析的 `(operator_code, tech_norm, lac, cell_id)` 即注册 | 结构规则 |
| `anchor_eligible` | `independent_obs >= 10`（GPS 有效观察点达标）且无碰撞阻断；空间质量门槛（如 p90 范围）由 Step 4 在实际取 donor 时再做判定 | `anchorable.*` |
| `baseline_eligible` | 已 `anchor_eligible=true`，且无防毒化异常，且满足成熟条件 | 逻辑冻结；数值门槛待拆到 `antitoxin_params.yaml` / `retention_params.yaml` |

推荐解释：
- `is_registered=true` 只表示对象建档成功。
- `anchor_eligible=true` 才表示它能被 Step 4 用作可信锚点。
- `baseline_eligible=true` 表示它不仅能补数，还已经成熟到可以影响后续画像维护；该字段保留在 Step 3 snapshot 中，但成熟阈值不在本步骤新增硬编码。

### 准入判定

| 对象 | 进入条件 | 去向 |
|------|----------|------|
| Cell / BS / LAC 准入结果 | `lifecycle_state` 达到 `qualified / excellent` | 写入本批冻结快照，供 Step 5 维护 |
| 观察池 | `waiting / observing` | 保留在 Step 3 评估池中，等待后续批次累计 |
| 休眠链路 | `qualified+` 但超时静默 | 标记 `dormant`，交由 Step 5.4 接管 |

说明：
- Step 3 负责“判断谁通过准入”，但不直接发布 `trusted_cell_library` 这类正式库。
- Step 4 实际消费的是下一轮由 Step 5 发布的可信小区正式库；当前主线不再把 `anchor_eligible=true` 作为 donor 的二次准入门槛。
- 这样可以同时保留“当前批判断结果”和“下一轮可稳定读取的正式结果”这两个边界。
- 若业务要求“当前批首次晋级对象在本批 Step 5 就拿到当前批记录级维护事实”，应通过 Step 4 增加一条仅供 Step 5 使用的 seed bridge（如 `snapshot_seed_records`），而不是把 Step 3 候选域事实层直接改写成 Step 5 维护事实层。

### Cell 晋升进度展示

对仍处于 `waiting / observing` 的 Cell，页面需要给出离下一级还差什么的解释，至少包括：

| 维度 | 展示示例 |
|------|----------|
| 观察点差距 | "还需 `2` 个独立观察点"（`independent_obs` 与 `qualified` / `excellent` 门槛的差值） |
| 异常阻断 | "当前命中碰撞阻断（`is_collision_id=true`），不能进入 `qualified`" |

说明：晋级只看 `independent_obs`（GPS 有效观察点），所以只展示观察点差距 + 异常阻断即可。不再展示设备数、跨度、P90 差距 —— 这些不是晋级门槛（见"生命周期判定规则"补充说明）。

## 3.2 BS 级联评估

### 级联原则

BS 不直接读取原始报文，而是完全由下属 Cell 的质量和空间结果上卷生成。

```text
下属 Cell snapshot
    -> 统计 Cell 状态分布与资格分布
    -> 计算 BS 质心与离散度
    -> 计算 BS lifecycle_state
    -> 计算 BS anchor_eligible / baseline_eligible
    -> 产出 BS 层 snapshot
```

### BS 生命周期规则（可配置阈值）

| 状态 | 判定条件 | 参数来源 |
|------|----------|----------|
| `waiting` | 不满足 `observing`，且不满足 `qualified` | 由下两档反推 |
| `observing` | 至少 `1` 个下属 Cell 有 GPS 证据 | `bs_lifecycle.observing.min_cells_with_gps=1` |
| `qualified` | 下属合格及以上 Cell `>= 2` 个 | `bs_lifecycle.qualified.min_qualified_cells=2` |
| `excellent` | 下属优秀 Cell `>= 1` 个 | `bs_lifecycle.excellent.min_excellent_cells=1` |

说明：
- BS excellent 优先于 qualified 判定：有 1 个优秀 Cell 即为优秀 BS，否则需要 2 个合格+ Cell 才达标。
- BS 不直接读取原始报文，完全由下属 Cell 状态上卷。

### BS 属性聚合

| 字段 / 指标 | 口径 |
|-------------|------|
| BS 质心 | 下属 Cell 质心的经纬度中位数 |
| `gps_p50_dist_m` / `gps_p90_dist_m` | 下属 Cell 到 BS 质心距离的分位数 |
| `cell_count` | 下属 Cell 总数 |
| `active_cell_count` | 下属 `qualified+` Cell 数 |
| `good_cell_count` | 下属 `position_grade in ('qualified','good','excellent')` 的 Cell 数 |
| `classification` | `collision_bs / dynamic_bs / large_spread / normal_spread`，分类规则沿用现有实现 |
| `anchor_eligible` | 下属 Cell 的 `BOOL_OR(anchor_eligible)` |
| `baseline_eligible` | 下属 Cell 的 `BOOL_OR(baseline_eligible)`，并保留后续防毒化阻断接口 |

说明：
- BS 分类本身不在 Step 3 做深度异常分析，但可以复用已有分类字段作为聚合展示结果。
- 若分类结果需要重算异常子集，应由 Step 5 负责，不在 Step 3 内新增复杂计算。

### BS 入库判定

| 对象 | 条件 | 去向 |
|------|------|------|
| BS 准入结果 | `lifecycle_state in ('qualified', 'excellent')` | 写入本批冻结快照，供 Step 5 维护并发布正式可信基站库 |
| 观察态 BS | `waiting / observing` | 保留在 snapshot 中，用于流转展示 |

页面展示至少支持：
- 查看每个 BS 下属 Cell 的状态构成。
- 查看“已有 `2/3` 个 qualified+ Cell”这类晋升进度。
- 查看 BS 当前聚合规则和阈值来源。

## 3.3 LAC 级联评估

### 级联原则

LAC 只由下属 BS 聚合，不跳过 BS 直接从 Cell 重判。

```text
下属 BS snapshot
    -> 统计 BS 状态分布与资格分布
    -> 计算 LAC 质心与面积
    -> 计算 LAC lifecycle_state
    -> 计算 LAC anchor_eligible / baseline_eligible
    -> 产出 LAC snapshot + 当前批 LAC 准入结果
```

### LAC 生命周期规则（可配置阈值）

| 状态 | 判定条件 | 参数来源 |
|------|----------|----------|
| `waiting` | 不满足 `observing`，且不满足 `qualified` | 由下两档反推 |
| `observing` | 至少 `1` 个下属 BS 为非 `waiting` | `lac_lifecycle.observing.min_non_waiting_bs=1` |
| `qualified` | 下属优秀 BS `>= 10` 个 | `lac_lifecycle.qualified.min_excellent_bs=10` |
| `excellent` | 下属优秀 BS `>= 30` 个 | `lac_lifecycle.excellent.min_excellent_bs=30` |

说明：
- LAC 的晋级完全基于下属优秀 BS 的数量，反映区域高质量覆盖的充分程度。
- LAC 级联只反映区域整体质量，不新增一套独立的 Cell 级原始空间门槛。

### LAC 属性聚合

| 字段 / 指标 | 口径 |
|-------------|------|
| LAC 质心 | 下属 BS 质心的经纬度中位数 |
| `bs_count` | 下属 BS 总数 |
| `active_bs_count` | 下属 `qualified+` BS 数 |
| `cell_count` | 下属 Cell 总数 |
| `area_km2` | 基于下属 BS / Cell 空间包络的区域面积估算 |
| `anomaly_bs_ratio` | 异常 BS 占比，用于区域整体质量提示 |
| `anchor_eligible` | 下属 BS 的 `BOOL_OR(anchor_eligible)` |
| `baseline_eligible` | 下属 BS 的 `BOOL_OR(baseline_eligible)`，并保留后续防毒化阻断接口 |

### LAC 入库判定

| 对象 | 条件 | 去向 |
|------|------|------|
| LAC 准入结果 | `lifecycle_state in ('qualified', 'excellent')` | 写入本批冻结快照，供 Step 5 维护并发布正式可信位置区库 |
| 观察态 LAC | `waiting / observing` | 保留在 snapshot 中，用于流转展示 |

页面展示至少支持：
- 查看 LAC 下属 BS 的状态分布。
- 查看区域整体质量概览，例如 `qualified BS` 占比、异常 BS 占比。
- 查看“还差多少个 qualified BS 才能晋升”的进度说明。

## 3.4 定期清理

Step 3 需要维护“等待对象清理”和“已入库对象静默转休眠”的两条保守清理机制，避免评估池无限膨胀。

### 等待项清理（当前简化规则）

| 项目 | 规则 |
|------|------|
| 目标对象 | 候选池中的 `waiting / observing` Cell |
| 清理条件 | 当前按日批次口径，`first_seen_batch_id` 起连续 `45` 批仍未晋级 |
| 处理方式 | 直接从 Step 3 评估池删除，不进入 `dormant / retired` 链路 |
| 阈值位置 | 当前为简化实现，后续可再挂到 `retention_params.yaml` |
| 当前值 | `45` 批（按当前日批次近似为 `45` 天） |

说明：
- 这些对象从未进入可信赖库，因此不需要走退出流程。
- 删除只移除评估池记录，不影响 `etl_cleaned` 等原始事实表。
- 已在 Step 3 晋级为 `qualified / excellent` 的对象，不再保留在候选池中，而是交由 Step 5 / 正式库链路继续接管。

### 已入库对象超时

| 项目 | 规则 |
|------|------|
| 目标对象 | 已进入 `trusted_*_library` 的 `qualified+` 对象 |
| 清理条件 | 连续 `M` 天无新数据 |
| 处理方式 | 当前暂未在 Step 3 启用自动标记，后续交给 Step 5.4 的退出链路继续处理 |
| 阈值位置 | `retention_params.yaml` |

说明：
- `M` 的具体数值目前未冻结，Step 3 当前保留统计位，但不主动执行 `dormant` 标记。
- `dormant` 不是删除，而是“暂停活跃更新、等待退出判断”。

### 地区范围收敛

地区限制本质上属于前置过滤，但在长期运行中也承担“只保留当前数据集上下文对象”的清理作用：

- 当前数据集不包含的 LAC 不应误入本批可信库。
- 局部重算时，只保留白名单区域的对象进入当前批 snapshot。
- 超出评估范围的对象保留在历史归档，不纳入当前批运行统计。

## 3.5 Snapshot 冻结

### 批末统一更新

冻结快照原则是 Step 3 最重要的约束：

```text
当前批次的所有判断
    只能读取 trusted_snapshot_t-1
    不能边判边写本批 trusted_snapshot_t
    不能让本批刚晋升对象立刻反哺本批自己
```

执行方式：
- 批次内所有 Cell / BS / LAC 状态先在本批上下文中统一计算。
- 批末一次性冻结为 `trusted_snapshot_t`。
- 下一批 Step 2 / Step 3 / Step 4 才允许读取本批冻结结果。

### Snapshot 内容

#### Cell snapshot

| 字段 | 说明 |
|------|------|
| `operator_code` / `lac` / `bs_id` / `cell_id` | 业务标识 |
| `lifecycle_state` | 当前生命周期状态 |
| `is_registered` / `anchor_eligible` / `baseline_eligible` | 三层资格 |
| `center_lon` / `center_lat` | 冻结质心 |
| `p50_radius_m` / `p90_radius_m` | 冻结空间半径 |
| `position_grade` | 当前定位质量等级 |
| `independent_obs` / `distinct_dev_id` | 累计观测证据 |
| `gps_valid_count` / `active_days` / `observed_span_hours` | 补充判级证据 |
| `rsrp_avg` / `rsrq_avg` / `sinr_avg` | 信号统计 |
| `is_collision_id` | 外部碰撞阻断标记 |

#### BS snapshot

| 字段 | 说明 |
|------|------|
| `operator_code` / `lac` / `bs_id` | 业务标识 |
| `lifecycle_state` | 当前生命周期状态 |
| `is_registered` / `anchor_eligible` / `baseline_eligible` | 三层资格 |
| `cell_count` / `active_cell_count` / `good_cell_count` | 下属 Cell 构成 |
| `classification` | BS 聚合分类 |
| `center_lon` / `center_lat` | 冻结质心 |
| `gps_p50_dist_m` / `gps_p90_dist_m` | Cell -> BS 离散度 |
| `position_grade` | 聚合后的定位质量提示 |

#### LAC snapshot

| 字段 | 说明 |
|------|------|
| `operator_code` / `lac` | 业务标识 |
| `lifecycle_state` | 当前生命周期状态 |
| `is_registered` / `anchor_eligible` / `baseline_eligible` | 三层资格 |
| `bs_count` / `active_bs_count` / `cell_count` | 区域对象构成 |
| `center_lon` / `center_lat` | 冻结质心 |
| `area_km2` | 区域面积估算 |
| `anomaly_bs_ratio` | 异常 BS 占比 |
| `position_grade` | 区域整体质量提示 |

### Diff 计算

`snapshot_diff` 需要记录本批与 `trusted_snapshot_t-1` 的差异，支持流转页面和回溯分析。

| 层级 | 最少记录项 |
|------|------------|
| Cell diff | 新增、删除、晋升、降级、`anchor_eligible` 变化、`baseline_eligible` 变化、质心偏移、`p90_radius_m` 变化 |
| BS diff | 新增、删除、晋升、降级、下属 Cell 数变化、聚合质心偏移 |
| LAC diff | 新增、删除、晋升、降级、下属 BS 数变化、区域面积变化 |

推荐的 diff 分类：
- `new`
- `promoted`
- `demoted`
- `removed`
- `eligibility_changed`
- `geometry_changed`

## 3.6 产出冻结快照与准入结果

### 输出表

| 表 | 粒度 | 说明 |
|----|------|------|
| `trusted_snapshot_t` | Cell / BS / LAC | 本批冻结快照，记录三层状态、资格和基础画像 |
| `snapshot_diff` | Cell / BS / LAC | 本批与上一批的变化记录 |
| `step3_run_stats` | 运行级 | 本批 Step 3 的评估、晋升、清理和收敛统计 |

### `trusted_snapshot_t` 中的 Cell 层核心字段

| 分组 | 字段 | 说明 |
|------|------|------|
| 标识 | `operator_code` / `lac` / `bs_id` / `cell_id` / `tech_norm` | 业务标识和制式维度 |
| 状态 | `lifecycle_state` / `is_registered` / `anchor_eligible` / `baseline_eligible` | 生命周期与资格 |
| 异常阻断 | `is_collision_id` | 来自 Step 5.1 的冲突编号清单；首轮运行默认可全为 `false` |
| 空间 | `center_lon` / `center_lat` / `p50_radius_m` / `p90_radius_m` | 冻结空间画像 |
| 信号 | `rsrp_avg` / `rsrq_avg` / `sinr_avg` | 信号均值 |
| 统计 | `independent_obs` / `distinct_dev_id` / `active_days` / `observed_span_hours` | 观测成熟度 |
| 质量 | `position_grade` / `gps_confidence` / `signal_confidence` | 画像质量补充字段 |

说明：
- Step 3 的核心产物是“冻结快照”，不是“正式库发布结果”。
- Step 5 读取 `trusted_snapshot_t` 做维护后，才发布 `trusted_cell_library` / `trusted_bs_library` / `trusted_lac_library`。
- `baseline_eligible` 字段必须保留在快照中，即使当前成熟阈值仍由 Step 5 维护。

## 3.7 处理流程统计

### 统计链路

```text
profile_base + trusted_snapshot_t-1 + collision_id_list
    -> 前置过滤统计
    -> Cell 全量重评估统计
    -> BS / LAC 级联统计
    -> snapshot + diff 统计
    -> 清理统计
    -> step3_run_stats
```

### `step3_run_stats` 最少指标

| 指标 | 含义 |
|------|------|
| `dataset_key` | 当前数据集 |
| `run_id` / `run_at` | 本次 Step 3 运行标识和时间 |
| `trusted_snapshot_version_prev` | 读取的上一版冻结快照 |
| `profile_base_cell_count` | 来自 Step 2 Path B 的 Cell 数 |
| `mode_filtered_count` | 制式过滤量 |
| `region_filtered_count` | 地区过滤量 |
| `gps_filtered_count` | 因 GPS 证据不足未参与空间评估的对象数 |
| `evaluated_cell_count` | 参与全量重评估的 Cell 数 |
| `waiting_cell_count` / `observing_cell_count` / `qualified_cell_count` / `excellent_cell_count` | Cell 状态分布 |
| `new_qualified_cell_count` / `new_excellent_cell_count` | 本批新晋升 Cell 数 |
| `anchor_eligible_cell_count` | 可供 Step 4 使用的锚点 Cell 数 |
| `bs_waiting_count` / `bs_observing_count` / `bs_qualified_count` | BS 状态分布 |
| `lac_waiting_count` / `lac_observing_count` / `lac_qualified_count` | LAC 状态分布 |
| `waiting_pruned_cell_count` | 等待超时删除数 |
| `dormant_marked_count` | 本批静默转休眠数（当前实现保留统计位，待后续启用） |
| `snapshot_new_count` / `snapshot_promoted_count` / `snapshot_demoted_count` | 快照变化概览 |

### 收敛趋势

为了支撑“流式 = 批量”的持续验证，Step 3 页面和统计表都应保留跨批次趋势：

| 趋势项 | 说明 |
|--------|------|
| 各状态数量变化 | `waiting / observing / qualified / excellent` 随批次的变化曲线 |
| 晋升数量变化 | 每批新晋升到 `qualified / excellent` 的对象数 |
| 锚点数量变化 | `anchor_eligible` Cell 的增长曲线 |
| BS / LAC 成熟度变化 | 区域整体质量是否收敛 |
| 清理数量变化 | 等待删除、静默转休眠数量趋势 |

## 与其他步骤的接口

| 方向 | 接口 | 说明 |
|------|------|------|
| 从 Step 2 接收 | `profile_base` | Path B 的 Cell 基础指标，是 Step 3 当前批次直接输入 |
| 从全局接收 | `trusted_snapshot_t-1` | 上一版冻结快照，只读 |
| 从全局接收 | `collision_id_list` | 来自 Step 5.1 的冲突小区编号清单，只读 |
| 输出到 Step 5 | `trusted_snapshot_t` + `snapshot_diff` | 供完整画像维护、正式库发布和退出链路使用 |
| 输出到流转页面 | `trusted_snapshot_t` + `snapshot_diff` | 供流转总览、流转快照、观察工作台使用 |

## 数据表

### 数据表链路

```text
profile_base
    + trusted_snapshot_t-1
    + trusted_cell_library(t-1)   # 仅用于识别已发布域 Cell 并做 carry-forward 展示
    + collision_id_list
        -> Cell 全量重评估（候选域）
        -> published-domain Cell carry-forward merge
        -> 由最终 Cell 视图上卷 BS
        -> 由最终 BS 视图上卷 LAC
        -> trusted_snapshot_t
        -> snapshot_diff
        -> step3_run_stats
```

### 表定义

| 表 | 类型 | 粒度 | 说明 |
|----|------|------|------|
| `profile_base` | 输入表 | Cell 级 | 来自 Step 2 Path B 的基础指标 |
| `trusted_snapshot_t-1` | 参考表 | Cell / BS / LAC | 上一版冻结快照，只读，用于 diff 与展示继承 |
| `trusted_cell_library(t-1)` | 参考表 | Cell 级 | 上一版正式库，只读；用于限定 carry-forward 仅覆盖已发布域 Cell |
| `collision_id_list` | 参考表 | `cell_id` 级 | 来自 Step 5.1 的冲突小区编号清单，只读 |
| `trusted_snapshot_t` | 输出表 | Cell / BS / LAC | 本批冻结快照 |
| `snapshot_diff` | 输出表 | Cell / BS / LAC | 与上一批的差分 |
| `step3_run_stats` | 统计表 | 运行级 | 本批 Step 3 的评估、晋升、清理和收敛统计 |

结论：Step 3 固定采用“候选域 Cell 全量重评估 + 已发布域 Cell carry-forward merge + 由最终 Cell 视图上卷 BS/LAC + 批末统一冻结”的工作方式，把 Step 2 的基础画像转化为可信准入结果，同时严格遵守冻结快照原则，确保当前批次不能自我强化，下一批才能消费本批新晋升对象。
