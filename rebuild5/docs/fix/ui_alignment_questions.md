# UI 语义对齐待决策问题

> 本文件只记录当前确实无法由文档基线单独拍板的问题。  
> 其余问题已在 `rebuild5/docs/fix/ui_global_alignment.md` 中给出建议，可直接作为后续修订基线。

## Q1 数据集页到底叫“数据集选择”还是“当前数据集”？

- 当前分歧：
  - 导航、UI 文档、系统设计文档使用`数据集选择`
  - 实际页面标题使用`当前数据集`
  - 当前系统又是单活只读模式，不支持在线切换
- 方案 A：继续叫`数据集选择`，但页头和导航统一标注`单活 / 只读`
  - 影响：保留文档与未来多数据集切换的延续性，但不会再暗示“现在就能切换”
- 方案 B：统一改成`当前数据集`
  - 影响：最贴合当前实现，误导最少；但未来支持在线切换时需要再次改名
- 方案 C：统一改成`数据集上下文`
  - 影响：最准确，但对非技术用户不如前两种直观
- 推荐方案：方案 A
- 影响：一旦确定，需同步导航、页面标题、UI 文档、全局设计文档中的页面命名

## Q2 运行历史页应该按“文档口径”还是“后端 run_type 口径”展示运行类型？

- 当前分歧：
  - 文档口径是：`日常运行 / 初始化 / 局部重算 / 完整回归 / 对比验证`
  - 页面当前映射里还有：`完整运行 / 参数重跑` 等实现色彩更强的说法
  - 目前无法仅凭现有代码判断 `pipeline`、`rerun` 是否与文档定义一一对应
- 方案 A：页面主表达统一使用文档口径，run_type 原值仅做辅注
  - 影响：用户理解最稳定，和控制侧链文档完全对齐；但需要额外维护“代码值 → 业务运行类型”的映射
- 方案 B：页面按当前后端 run_type 直接翻译，保留`参数重跑`等说法
  - 影响：实现最省事，但和文档、human guide 会继续错位
- 推荐方案：方案 A
- 影响：需要补齐一张正式映射表，避免不同页面再次各自翻译 run_type

## Q3 服务层默认要走“业务友好模式”还是“专家模式”？

- 当前分歧：
  - 服务层定位文档强调面向业务用户、弱化治理术语
  - 但当前页面直接展示`Cell / BS / LAC`、`锚点资格`、`基线资格`、`qualified+`、`excellent` 等治理表达
  - 页面标题还叫`基站查询`，实际又支持三层搜索
- 方案 A：默认走业务友好模式
  - 内容：主页面改成`站点查询`或`位置查询（Cell/BS/LAC）`；默认隐藏锚点 / 基线等治理字段；用`达标 / 高精度 / 风险提示`替代内部状态术语；如有需要再提供“高级字段”开关
  - 影响：更符合 Step 6 的角色定义，业务用户更容易上手
- 方案 B：继续走专家模式
  - 内容：保留`基站查询`与当前治理字段暴露方式
  - 影响：治理分析师更方便，但业务用户会继续接触大量内部术语
- 推荐方案：方案 A
- 影响：需要同步调整服务层三页文案、搜索表头、详情卡、报表指标名，以及 Step 6 文档措辞

## 当前结论

除以上 3 项外，其余术语漂移问题已经可以按文档基线直接收敛，不必继续等待额外决策。
