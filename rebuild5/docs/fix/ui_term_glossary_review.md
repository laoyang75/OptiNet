> **使用方式**：先看"总结：发现的问题"，再逐节逐表对照原稿审阅；你只需要审定最终版是否可以合并回主文件。

## 一、总结：发现的主要问题

| # | 位置 | 问题性质 | 严重度 |
|---|------|----------|--------|
| 1 | §2 `source cell` 定义 | **语义不准确**：原文说"在 Step 2 中成功撞击到的上一轮正式库小区"，**实际上 Step 2 Path A 的命中对象是可信库小区，不是"source cell"这个词**。`source cell` 是 Step 4 的专用概念，指 Step 2 已确认并随 `path_a_records` 传递下来的补数来源 Cell。两者在文档中描述的步骤和场景不同。 | 🔴 高 |
| 2 | §2 `Path B` 描述 | **说明有偏差**：原文"没命中正式库，但自带 GPS 且信息饱满"——按 `02_基础画像.md`，Path B 的条件是"未命中，且具备**可用于空间计算的有效 GPS 证据**"，**不要求"信息饱满"**，缺少信号也可入 Path B，只要有 GPS 即可。 | 🔴 高 |
| 3 | §2 `routing` 翻译 | **条目可删/合并**：`routing` 在 `02_基础画像.md` 中是步骤定位术语（Step 2 整体叫做"分流路由节点"），UI 层面页面名已经建议用"分流"。该词不作为独立 UI 字段出现，只是导航/模块标题的风格约定。当前解释可以移进脚注，避免干扰主表。 | 🟡 中 |
| 4 | §3 `observing` 服务侧翻译 | **"建议不展示"说法值得商榷**：原稿写"（建议不展示 / 持续积累中）"，`06_服务层.md` 中 Step 6 实际上并没有明确排除 `observing` 的展示，建议改为**"积累中（数据仍不足）"**，让业务侧有一致的感知，而不是什么都不显示。 | 🟡 中 |
| 5 | §3 `dormant` 服务侧翻译 | **细节不准**：原文写"（或暂停活跃更新）"，按 `05_画像维护.md §5.2.7`，进入 `dormant` 的对象**仍然保留在可信库和治理页面**，不是整个对象被暂停；暂停的只是"画像刷新"和"窗口数据更新"。服务侧翻译建议改为："**休眠**（已暂停更新，数据仍保留）"。 | 🟡 中 |
| 6 | §4 `anchor_eligible` 业务解释 | **解释角度有误**：原文说"能不能拿去补别人"，实际上 `anchor_eligible` 的含义是"**该 Cell 能否作为 donor 进入 Step 4 知识补数链路**"——即"它有没有资格贡献给别人"。"拿去补别人"方向是对的，但描述不够精确，容易与"被别人补"混淆。建议改为"**能否作为补数来源（donor）输出位置和信号知识给其他匮乏记录**"。 | 🟡 中 |
| 7 | §4 `baseline_eligible` 业务解释 | **表述有遗漏**：原文说"能不能用来升级大盘"，按 `00_全局约定.md`，`baseline_eligible=true` 的实际条件是 `anchor_eligible=true` **且** `lifecycle_state='excellent'`（Cell 侧）；**防毒化阻断也会让 `baseline_eligible` 置为 false**（`05_画像维护.md §5.2.3`）。建议在业务解释里补充"防毒化阻断后即使是 excellent 也会临时失去此资格"的说明，避免研发误解。 | 🟡 中 |
| 8 | §5 `snapshot_version` 表达 | **用词分叉**：原稿已经说"Step 3/4 读之前的叫'引用快照'，Step 5 冻结出来的叫'当前快照'"，但技术文档 `00_全局约定.md` 中的正式命名是 `trusted_snapshot_t-1`（引用）和 `trusted_snapshot_t`（当前批产出）。两者对应关系合理，但表格里应补一列说明与代码字段的对应，防止歧义。 | 🟡 中 |
| 9 | §6 `collision` (Step 2) 的表达 | **混淆了 Step 2 和 Step 5 的职责**：原稿在 Step 2 列了"ID 碰撞防护"，实际上按 `02_基础画像.md §2.1`，Step 2 **本身不做碰撞检测**，只是**消费 Step 5.1 产出的 `collision_id_list`** 做路由防护。UI 上 Step 2 页面显示的应叫"**碰撞 ID 防护拦截**（引用 Step 5 冲突表）"，不是"碰撞检测"。 | 🔴 高 |
| 10 | §6 `large_spread` 的独立列 | **原稿合并了两个不同概念**：把 `large_coverage` 和 `large_spread` 合为一行。按技术文档：`large_coverage` 是 Cell 层面的漂移分类标签（`drift_pattern`），而 `large_spread` / `BS.large_spread` 是 BS 层面的分类标签（`max(cell_to_bs_distance_m) > 2500`），二者完全不同层级。应拆开单独列出。 | 🔴 高 |
| 11 | §6 `dynamic` / `dynamic_bs` | **说法不完整**：原文"天然的动态信源，像车船基站，基线不要它"。按技术文档，`is_dynamic=true` 的判定条件是 `drift_max_spread_m > 1500` **且** `drift_pattern in ('migration','large_coverage')`，不是"天然动态信源"——可能是固定但发生迁移/大覆盖的基站。UI 侧的解释建议改为"**位置跨度大或位置不稳定**（不建议作为基线参考）"。 | 🟡 中 |
| 12 | §6 `normal_spread` | **原文中文表达模糊**："正常或稳定"——应明确对应的是**BS 层面的分类标签**（`BS.classification = 'normal_spread'`），而非 Cell 层面的 `drift_pattern='stable'`。两者都有"稳定"含义但层级不同。建议补充"（BS级：下属小区分布正常）"。 | 🟡 中 |
| 13 | §7 替换清单 `excellent donor` 翻译 | **略有语义损失**：原文改为"高置信来源小区"，但按 Step 4 文档的说法，donor `excellent` 的含义更接近"**高精度补数来源小区**"（因为 `excellent` 对应 `position_grade=excellent`，即精度圈半径 < 500m）。"高置信"偏语言层，"高精度"更准确地指向系统的质量成因。 | 🟢 低 |
| 14 | §7 替换清单 `qualified BS` 翻译 | **表达需细区分**："`qualified BS`（统计类）→ 达标基站数"。但在治理页面，`qualified BS` 也是生命周期状态词，不只是统计标签。UI 统计卡片上写"**达标基站数**"可以，但治理详单页面在字段值里应写"**合格**（基站状态）"，避免将统计用语混入状态列。 | 🟢 低 |

### § 3. 核心分域：生命周期状态表（修改）

> 核心区分：治理页面看真实生命流转；服务交付面看最终的可用性转化。

| 内部状态值 | 治理页/详单页（精准表达） | 服务侧/交付页（业务理解） | 是什么意思？ |
|------------|--------------------------|--------------------------|--------------| 
| `waiting` | **等待** | （建议不展示 / 证据极度不足） | 证据过少（`independent_obs < 3` 或 `distinct_dev_id < 2`），不构成哪怕起码的位置聚集认知。 |
| `observing`| **观察**（或观察中） | **积累中**（数据仍不足）【**修正**：服务侧改为有实质含义的词，而不是"建议不展示"】 | 有少量有效观测但达不到正式放行的厚度（未达 `qualified` 线）。 |
| `qualified`| **合格** | **达标** / **可用** | 已跨越数据门槛，开始属于"靠谱资产"，可参与 Step 4 补数（需 `anchor_eligible=true`）。 |
| `excellent`| **优秀** | **高精度** / **优质** | 各项聚类半径（p90 < 500m）、设备聚集数都非常好，可用于严苛级寻址；也是 `baseline_eligible` 的必要条件之一。 |
| `dormant` | **休眠** | **休眠**（已暂停更新，数据仍保留）【**修正**：补充"数据仍保留"，避免用户误以为数据被删除】 | 连续多批无新观测，系统暂停画像刷新和窗口更新，但 Cell 仍保留在治理库中，可被查询。 |
| `retired` | **退出** | **已退出**（或作废） | 持续 `dormant` 后被系统确认退出活跃可信库，画像转归档；如再次出现新数据，需回到 Step 2 重新积累证据。 |

### § 6. 治理标签与诊断分类表（重大修改）

这些标签是 Step 2 拦截、Step 5 质检的核心指标枚举，不可互相错位混淆。

| 内部特征词 | 对应推荐中文 | 适用面与含义区别 |
|------------|--------------|------------------|
| `collision` (Step 5.1 产出，Step 2 消费) | **碰撞 ID 防护拦截** / **碰撞小区** | 【**重大修正**】Step 2 **不自己做碰撞检测**，只消费 Step 5.1 产出的 `collision_id_list`，对碰撞 Cell ID 的记录升级为三字段 + GPS 距离防护策略；UI 上 Step 2 页面显示词应为"**碰撞 ID 防护拦截**（依据 Step 5 碰撞表）"，而非"碰撞防护"。 |
| `collision` / `is_collision=true`（Step 5 维护标签） | **空间碰撞**（碰撞小区） | （评估，Step 5 产出）同一 `cell_id` 在不同物理位置出现，或漂移模式呈往返振荡（ratio < 0.3）。影响：`baseline_eligible=false`。 |
| `migration` | **迁移** | 时间线上明确发现基站换了物理部署位置（`max_spread_m ≥ 2200 且 ratio ≥ 0.7`）。 |
| `moderate_drift` | **中度漂移** | 经纬度变了点范围但未达迁移标准（max_spread_m ≥ 2200 且 0.3 ≤ ratio < 0.7）。 |
| `multi_centroid` / `is_multi_centroid=true` | **多质心** | 聚类算出了两个及以上稳定位置簇。服务页需弱化，治理分析页作为重点盯防点。 |
| `large_coverage` （Cell 层漂移分类） | **大覆盖**（Cell 级） | Cell 层漂移分类：`500m ≤ max_spread_m < 2200m`。常见于农村宏站或山顶站点，覆盖半径天然较大。 |
| `large_spread` / `BS.large_spread`（BS 层分类）| **下属小区离散过大**（BS 级）【**重大修正**：从合并行拆出独立行】 | BS 层分类标签：`max(cell_to_bs_distance_m) > 2500m`，表示该基站下属多个小区地理位置相互间距过大，已超过正常基站覆盖模型。 |
| `dynamic` / `is_dynamic=true` | **位置不稳定**（动态站） | 触发条件：`drift_max_spread_m > 1500` **且** `drift_pattern in ('migration', 'large_coverage')`，不特指"天然动态信源"，也可能是固定站发生大跨度漂移或迁移。【**修正**：原文"天然动态信源"说法不准】服务页叫"**位置不稳定**"；不建议参与基线刷新。 |
| `dynamic_bs` | **含动态小区的基站** | BS 层分类标签，表示该 BS 下至少含一个 `is_dynamic=true` 的 Cell，可能存在移动站点污染。 |
| `normal_spread` | **分布正常**（BS 级）【**修正**：补充"BS 级"说明，区别于 Cell 的 `stable`】 | BS 层分类标签（不命中 `collision_bs`、`dynamic_bs`、`large_spread`、`multi_centroid` 的默认正常状态）。 |
| `stable` | **稳定**（Cell 级漂移） | Cell 层漂移分类：`max_spread_m < 500m`。空间聚合紧凑，位置质量好。 |
| `drift_pattern` | **漂移模式** | 特指 Cell 层面的空间随时间演变类别（`stable / collision / migration / large_coverage / moderate_drift / insufficient`），来自 Step 5 多日质心轨迹分析。 |
| `position_grade` | **位置质量** | 特指单点静态聚类出来的精度圈等级（`excellent / good / qualified / unqualified`），不等同于 `lifecycle_state`。 |

### § 7. 页面常见英文/混合句直接替换清单（小修）

| 目前正在用的老旧写法 | 一律快速改成规范说法 |
