# 核心约束与设计原则

> 本文解释 rebuild5 最重要的几个设计决策：**为什么这样做，不这样做会发生什么**。理解这些原则，是正确开发和运维系统的前提。

---

## 原则一：冻结快照原则（最高优先级）

### 什么是冻结快照原则

```mermaid
graph TD
    RULE["当前批次的状态判定与知识补数\n只能读取上一轮已冻结的版本\n不能边算边读本批正在产出的结果"]

    S3_RULE["Step 3 质量评估\n→ 只读 trusted_snapshot_t-1\n（上一批冻结快照）"]
    S4_RULE["Step 4 知识补数\n→ 只用上一轮已发布的 trusted_cell_library"]
    BATCH_RULE["批次内不改状态\n→ 批末统一写入 trusted_snapshot_t"]

    RULE --> S3_RULE & S4_RULE & BATCH_RULE
```

### 违反后会发生什么（错误链路）

```mermaid
sequenceDiagram
    participant CellA as Cell A（本批刚晋升）
    participant Report as 本批缺GPS的报文
    participant Profile as Cell A 的画像指标

    Note over CellA: ❌ 错误：本批刚晋升立刻允许参与补数

    CellA->>Report: 用 A 的质心补充报文的 GPS

    Report->>Profile: 这条被补数的报文\n现在算作 A 的新观测

    Profile->>CellA: A 的指标再次增强\n（观测数 + 1，GPS 证据 + 1）

    Note over CellA: 😱 自我强化闭环\n数据质量不可控
```

### 正确链路

```mermaid
sequenceDiagram
    participant CellA as Cell A（第T批晋升）
    participant SnapT as trusted_snapshot_t（T批冻结）
    participant NextBatch as 第T+1批运行

    CellA->>SnapT: T批末冻结
    Note over SnapT: 不能被本批 Step 4 使用

    NextBatch->>SnapT: ✅ T+1批才读取
    Note over NextBatch: T+1批才允许用 Cell A 补数
```

## 原则二：Cell 先行，自下而上级联

### 为什么必须从 Cell 开始

```mermaid
graph BT
    CELL["Cell（最小可观测单元）\n直接用原始 GPS 观测打分\n→ 计算质心、半径、生命周期"]

    BS["BS（基站）\n由下属 Cell 聚合派生\n→ 不重新看原始报文"]

    LAC["LAC（位置区）\n由下属 BS 聚合派生\n→ 不跳过 BS 直接从 Cell 判"]

    CELL -->|"聚合"| BS
    BS -->|"聚合"| LAC

    REASON["原因：\nCell 是最细粒度的观测单元\nBS 的空间质量依赖下层 Cell 的分布\nLAC 的成熟度只能建立在 BS 稳定的前提上"]
    CELL -.- REASON
```

### 违反后会发生什么

如果试图跳过 Cell 直接对 BS 做独立评分：
- BS 的定位误差会缺少 Cell 级的空间质量背书
- 同一 BS 下，优质 Cell 和碰撞 Cell 的贡献会被混为一谈
- LAC 的成熟度判断失去层级依据，无法确定"因为哪些 BS 达标"

## 原则三：修正优于丢弃

```mermaid
flowchart TD
    DATA["一条有问题的数据"]

    C1{"问题类型"}

    FIX["✅ 修正：\n置空或标记来源\n保留整行"]
    DROP["❌ 删除：\n整行删除"]

    EX1["信号值越界（RSRP/RSRQ/SINR）\n→ 置空，保留行"]
    EX2["坐标越出国界\n→ 标记无效 gps_valid=false，保留行"]
    EX3["运营商编码异常\n→ 置空 operator_code，保留行"]
    EX4["cell_id 不可解析 / 为 0 / 溢出\n→ 删除整行（主键级错误，无法修复）"]

    DATA --> C1
    C1 -->|"字段值异常"| FIX
    C1 -->|"主键级错误"| DROP
    FIX --> EX1 & EX2 & EX3
    DROP --> EX4
```

**背后的逻辑**：一条报文即使 GPS 无效，其信号值、设备 ID、时间戳依然有价值。整行删除会损失这些有效信息。

## 原则四：生命周期状态 vs 三层资格（两套独立系统）

这是系统中最容易引起混淆的设计：

```mermaid
graph LR
    subgraph 生命周期状态
        LS["lifecycle_state\n\n回答：对象处于哪个阶段？\nwaiting → observing → qualified → excellent\n→ dormant → retired"]
    end

    subgraph 三层资格
        R1["is_registered\n系统承认该对象存在"]
        R2["anchor_eligible\n能被 Step 4 用作补数锚点"]
        R3["baseline_eligible\n能参与 Step 5 画像刷新"]
    end

    NOTE["两者独立维护\n不共用一个字段\n不能相互替代"]

    LS -.- NOTE
    R1 -.- NOTE
```

**典型的合法组合**：

| 生命周期 | anchor_eligible | 含义 |
|----------|----------------|------|
| `qualified` | `false` | 观测量够了，但空间质量不达锚点标准（P90 太大） |
| `qualified` | `true` | 可以用于补数，但尚未成熟到刷新基线 |
| `excellent` | `true` | `baseline_eligible` 为 `true` 时可参与画像维护 |

## 原则五：只读上一轮，本批产出留到下批

所有步骤对"已发布库"的读写关系：

```mermaid
flowchart LR
    subgraph 本批运行
        S2["Step 2"]
        S3["Step 3"]
        S4["Step 4"]
        S5["Step 5"]
    end

    subgraph 只读（t-1 版本）
        LIB_OLD["trusted_cell_library\n（上一批发布）"]
        SNAP_OLD["trusted_snapshot_t-1\n（上一批冻结）"]
        COLL_OLD["collision_id_list\n（上一批产出）"]
    end

    subgraph 本批产出（下批才能读）
        SNAP_NEW["trusted_snapshot_t"]
        LIB_NEW["新版 trusted_cell_library"]
        COLL_NEW["新版 collision_id_list"]
    end

    S2 -->|"只读"| LIB_OLD & COLL_OLD
    S3 -->|"只读"| SNAP_OLD & COLL_OLD
    S4 -->|"只读"| LIB_OLD

    S3 -->|"产出"| SNAP_NEW
    S5 -->|"产出"| LIB_NEW & COLL_NEW

    style LIB_NEW fill:#c8e6c9,stroke:#388e3c
    style SNAP_NEW fill:#c8e6c9,stroke:#388e3c
    style COLL_NEW fill:#c8e6c9,stroke:#388e3c
```

## 原则六：治理视角 vs BI 视角（UI 设计边界）

系统 UI 分两种视角，不能混用：

```mermaid
graph LR
    subgraph 治理视角（Step 1-5 页面）
        G1["关注：规则是否命中？"]
        G2["关注：对象是否异常？"]
        G3["关注：本次 vs 上次的 diff"]
        G4["关注：阻断原因 / 退出预警"]
        G5["受众：系统管理员 / 数据工程师"]
    end

    subgraph BI 视角（Step 6 页面）
        B1["关注：基站覆盖率分布"]
        B2["关注：区域信号质量"]
        B3["关注：趋势分析（月度变化）"]
        B4["关注：运营商对比"]
        B5["受众：业务用户 / 分析师"]
    end
```

Step 5 的维护页面绝对不承载"经营分析报表"类内容，这类内容属于 Step 6 的服务层。

## 常见误解清单

| 误解 | 正确理解 |
|------|---------|
| "Step 3 发布可信库" | Step 3 只产出冻结快照，Step 5 才发布正式库 |
| "qualified 就能用于补数" | 补数资格是 `anchor_eligible`，与 `lifecycle_state` 独立 |
| "Step 4 补数结果可以立刻给 Step 3 用" | Step 4 产出不能回灌本批 Step 2/3，下一批才生效 |
| "BS 的质量是独立看原始报文判断的" | BS 完全由下属 Cell 聚合派生，不重新看报文 |
| "collision 标记来自 Step 2" | collision_id_list 由 Step 5.1 产出，Step 2 只是消费者 |
| "防毒化命中意味着 Cell 不可信" | 防毒化只阻断"本批画像是否生效"，不代表 Cell 不存在 |

## 控制侧链：非日常操作

除了日常稳态运行，系统还有四种非日常操作场景：

```mermaid
flowchart LR
    subgraph 控制侧链
        C1["冷启动初始化\n首次导入数据集\n无历史快照可参考\nStep 4 不生效（无正式库）\n先产出第一版快照再发布库"]

        C2["局部重算\n阈值参数调整后\n只对受影响范围重算\n不需要重跑 Step 1\n产出新 snapshot 做对比"]

        C3["完整回归\n解析或清洗规则变更\n必须从 Step 1 全量重跑\n产出新 snapshot 做全量对比\n确认无回归后切换版本"]

        C4["对比验证\n验证收敛一致性\n3天起跑滚到7天\nvs 直接7天全量\n对比质心偏差 / 生命周期一致率"]
    end

    NOTE["为什么需要完整回归？\n如果不支持，规则变更后只有变更点往后\n的数据被重新处理，历史数据口径不一致"]
    C3 -.- NOTE
```

## 统一事件时间（避免不同步骤各自解释时间）

所有步骤必须使用同一个时间字段：

```mermaid
graph LR
    ET["event_time_std\n= COALESCE(cell_ts_std, ts_std, gps_ts)\n\nStep 1 产出\n后续所有步骤统一引用"]

    S1["Step 1 生成 event_time_std"]
    S2_USE["Step 2 分钟级去重\n使用 event_time_std"]
    S3_USE["Step 3 observed_span_hours\nactive_days\n使用 event_time_std"]
    S5_USE["Step 5 滑动窗口\nlast_observed_at\nGPS 异常按日聚合\n使用 event_time_std"]

    S1 --> ET
    ET --> S2_USE & S3_USE & S5_USE

    WARNING["⚠️ 禁止：\n各步骤自行从原始时间字段重新推导\n必须引用 Step 1 产出的 event_time_std"]
    ET -.- WARNING
```

## 物理常量（不可配置，写入代码）

以下是算法常量，不放入业务配置文件：

| 常量 | 值 | 说明 |
|------|-----|------|
| 经度有效范围 | 73° ~ 135° | 中国边界 |
| 纬度有效范围 | 3° ~ 54° | 中国边界 |
| 经度→米系数 | 85,300 m/° | 北京纬度近似 |
| 纬度→米系数 | 111,000 m/° | 全球近似 |
| RSRP 有效范围 | -156 ~ 0 dBm | 物理约束 |
| RSRQ 有效范围 | -50 ~ 0 dB | 物理约束 |
| SINR 有效范围 | -30 ~ 50 dB | 物理约束 |

可配置的业务阈值（放入 `profile_params.yaml` 等配置文件）：晋级门槛、锚点门槛、漂移分桶、退出静默天数等。
