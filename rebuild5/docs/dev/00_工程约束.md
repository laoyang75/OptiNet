# 工程约束

本文件冻结 rebuild5 的所有工程级硬约束。开发过程中不允许绕过这些约束，修改需经过讨论确认。

## 前端强制约束（最高优先级）

**前端开发必须基于已有的 UI 设计工程，禁止另起炉灶。**

已完成的 UI 设计位于：

| 资产 | 位置 | 说明 |
|------|------|------|
| UI 设计总纲 | `rebuild5/docs/08_UI设计.md` | 22 个页面清单、路由、导航结构、用户分层 |
| 页面分册 | `rebuild5/ui/*.md`（8 个文件） | 每组页面的详细框架、模块、字段、交互说明 |
| 设计实现 | `rebuild5/frontend/design/` | 可运行的 Vue 3 + TS 工程，含 22 个页面组件 |
| 通用组件 | `rebuild5/frontend/design/src/components/common/` | 6 个通用组件（StatusTag / SummaryCard / StateDistribution / PercentBar / RuleCard / PageHeader） |
| 设计系统 | `rebuild5/frontend/design/src/style.css` | CSS 变量、颜色冻结、布局规范 |
| Mock 数据 | `rebuild5/frontend/design/src/mock/data.ts` | 静态 Mock 数据层 |
| 类型定义 | `rebuild5/frontend/design/src/types/index.ts` | 实体模型和状态枚举 |
| 截图参考 | `rebuild5/frontend/design/screenshots/` | 10 张设计验收截图 |

### 必须遵守的规则

1. **不准自己从零设计页面**。所有前端页面必须从 `rebuild5/frontend/design/src/views/` 中对应的 `.vue` 文件出发，在其基础上对接真实 API
2. **不准替换通用组件**。`StatusTag`、`SummaryCard`、`StateDistribution` 等组件已冻结，不得用自定义组件替代
3. **不准修改状态颜色**。6 种生命周期状态颜色（`--c-excellent` 到 `--c-retired`）来自 `00_全局约定.md`，不可更改
4. **不准修改路由结构**。22 个页面的路由路径已在 `08_UI设计.md` 中冻结，不可重命名或合并
5. **不准修改导航结构**。三段式侧边栏（全局管理 / 治理控制台 / 服务控制台）已冻结
6. **对接 API 时只替换数据源**。将 `import { mockXxx } from '../mock/data'` 替换为 API 调用，页面结构和组件不变
7. **任何布局或交互调整必须先和用户确认**，不得自行判断"这样更好"后擅自修改

### 验证方式

开发完成每个页面后，必须用 Playwright 截图并与 `rebuild5/frontend/design/screenshots/` 中的设计截图做视觉对比，确认布局、颜色、信息结构一致。

### 为什么有这条约束

此前多次出现开发 agent 忽略已有 UI 设计，自行实现一套完全不同的前端界面的情况。这导致大量返工和不一致。本约束的目的是确保设计资产被正确消费。

## 环境与依赖约束

| 约束 | 说明 |
|------|------|
| Python 版本 | 3.12+（与 rebuild4 一致） |
| 包管理 | `requirements.txt`，不用 poetry / pipenv |
| 后端框架 | FastAPI，Pydantic v2 |
| 数据库驱动 | `psycopg`（psycopg 3，同步模式），不用 asyncpg，不用 SQLAlchemy / ORM |
| 数据库交互 | 直接写 SQL + `execute()` / `fetchone()` / `fetchall()` 工具函数，参考 rebuild4 的 `core/database.py` |
| 几何计算 | 不用 PostGIS，使用工程近似距离公式（`00_全局约定.md` 中已冻结的经纬度转米系数） |
| 前端框架 | Vue 3 + TypeScript + Vite |
| 前端包管理 | npm |
| 前端 UI 库 | 不引入第三方 UI 库（Element Plus / Ant Design 等），使用已有的 CSS 设计系统 |
| Node 版本 | 18+（LTS） |

不允许引入的依赖：
- ORM（SQLAlchemy / Tortoise / Peewee）
- 异步数据库驱动（asyncpg / databases）
- PostGIS 扩展
- 第三方前端 UI 组件库

## SQL 约束

| 约束 | 说明 |
|------|------|
| 单一简单 SQL | 每条 SQL 只做一件事，不用复杂嵌套 CTE / 多层子查询 |
| Python 做编排 | SQL 只做数据操作，流程控制、统计汇总、条件分支由 Python 完成 |
| 可迁移性 | SQL 必须易翻译到其他数据库（如 MaxCompute），避免 PostgreSQL 专有语法（`jsonb_each`、`LATERAL` 等在 Step 1 解析中允许，后续步骤避免） |
| 不回写上游表 | Step 2/3/4/5 绝对不能回写 `etl_cleaned`，数据单向流动 |
| 表操作用 DROP + CREATE | 不用 TRUNCATE + INSERT 或 UPSERT，保持幂等性和简单性 |

## 代码约束

| 约束 | 说明 |
|------|------|
| 单文件上限 | 一个 `.py` 文件不超过 300-400 行，超了按职责拆分 |
| Router 与逻辑分离 | API router 只做参数校验和响应组装，业务逻辑放在独立模块 |
| 前后端分离 | 后端 FastAPI 只提供 JSON API，前端 Vue 3 + TypeScript 独立开发 |
| 配置外化 | 所有业务阈值从 `rebuild5/config/*.yaml` 读取，代码里不硬编码 |
| 统一响应格式 | 所有 API 走 `{ data, meta, error }` 信封格式 |

## 架构约束

| 约束 | 说明 |
|------|------|
| Step 间解耦 | 每个 Step 独立，Step 之间只通过数据库表交换数据，不共享内存状态 |
| Schema 隔离 | 所有 rebuild5 的表放在 `rebuild5` schema 下，不污染 rebuild4 和 legacy |
| 统一启动器 | 参考 rebuild4 的 `launcher/launcher.py`，统一拉起前后端 + 服务状态检查 |
| 前端热更新 | 开发阶段使用 Vite dev server，前后端异步启动 |

## 运行约束

| 约束 | 说明 |
|------|------|
| 每步必有统计表 | 每个 Step 运行后必须产出 `step{N}_run_stats`，记录输入量、输出量、过滤量、运行时间 |
| 每步必有验证 | Step 完成后做行数校验、空值检查、主键唯一性验证，不是跑完就完 |
| 日志存数据库 | 运行日志写入 `rebuild5_meta` schema 的日志表，不只靠 stdout |
| ETL 触发方式 | 封装为 API 端点（方便 agent 调用和未来自动化），同时保留命令行入口 |
| API 查询为主 | API 的主要用途是查询和展示，触发执行是辅助功能 |

## 验证约束

| 约束 | 说明 |
|------|------|
| Playwright 验证 UI | 每个页面开发完成后用 Playwright MCP 截图检查布局和数据展示 |
| PG17 MCP 验证数据 | 每个 Step 执行后用 PG17 MCP 查询表数据，验证正确性 |
| 不靠人眼 | 关键验证点必须有可重复的检查手段，不能只靠手动查看 |
