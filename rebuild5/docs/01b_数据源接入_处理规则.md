## 解析规则

### 处理链路

```text
原始报文（27 列）
    |
    +-- 读取 cell_infos JSON
    |      -> 每个主服务小区 1 行
    |
    +-- 读取 ss1 文本
    |      -> 每个信号组 1 行
    |
    +-- 保留 record_id / cell_id / 时间 / GPS / 元数据
    |
    +-- 合并为 etl_parsed
```

| 规则 | 说明 |

## `ss1` 解析

### 文本输入格式

`ss1` 是分号分组、每组 4 段、段内再按逗号切分的文本串：

```text
信号段&时间段&GPS段&基站段;信号段&时间段&GPS段&基站段;...
```

### 每段说明

| 段 | 格式 | 示例 | 说明 |
|----|------|------|------|
| 信号段 | 逗号分隔，首字母=制式 | `l,-85,-10,15` | 一组信号值 |
| 时间段 | unix 秒 | `1765011706` | 本组扫描时间 |
| GPS 段 | `经度,纬度` 或空 | `116.397,39.908` | 本组自带 GPS |
| 基站段 | `制式,cell_id,lac,plmn` 或 `1` | `l,12345,4176,46000` | 本组基站信息或继承标记 |

### 基站继承（forward-fill）逻辑

```text
组1: ... & l,12345,4176,46000    <- 本组自带基站
组2: ... & 1                      <- 继承组1基站
组3: ... & n,67890,73733,46001   <- 新基站
组4: ... & 1                      <- 继承组3基站
```

规则：
- `cell_block='1'` 表示继承本 `record_id` 内前一组最近一次出现的有效基站段。
- 如果当前组没有可继承基站，则该组不能生成结构化行。
- 同组内若存在多个候选基站或信号条目，可继续按 `+` 拆分，再按制式匹配。

### `ss1` 解析规则

| 规则 | 说明 |
|------|------|
| 先按 `;` 拆组 | 一条 `ss1` 可拆成多个组 |
| 再按 `&` 拆 4 段 | `sig_block`、`ts_block`、`gps_block`、`cell_block` |
| `cell_block` 支持 forward-fill | `1` 表示继承前一组基站 |
| `sig_block` / `cell_block` 可再按 `+` 拆分 | 用于处理同组多条候选 |
| 信号与基站按制式匹配 | `l` 对 `4G`，`n` 对 `5G`，`g` 对 `2G` |
| `gps_filled_from='ss1_own'` | 本组自带 GPS 的行保留该来源标记 |

### `ss1` 特有字段

| 字段 | 说明 |
|------|------|
| `sig_ss` | `ss1` 原始信号值（信号段第 2 个值） |
| `cell_ts_raw` | 组级时间戳（unix 秒） |

### `ss1` 没有的字段（置 NULL）

```text
pci, freq_channel, bandwidth,
rssi, dbm, asu_level, sig_level,
timing_advance,
csi_rsrp, csi_rsrq, csi_sinr, cqi
```

**CellID 派生关系**（clean.py line 84-91）：
```
5G:  bs_id = cell_id / 4096    sector_id = cell_id % 4096
4G:  bs_id = cell_id / 256     sector_id = cell_id % 256
```
因此 CellID 过小会直接派生出 bs_id=0 的非法基站，必须在 ETL 源头过滤。

### CellInfos 解析清洗（`ODS-019`）

不同于前面按字段值过滤的规则，本规则作用在 `cell_infos` JSON 的**对象层**：同一条 `raw_gps` 报文的 `cell_infos` 里包含多个 cell 对象（`{"1": {...}, "2": {...}, ...}`），其中部分对象虽然标记为 `isConnected = 1`，但实际是**设备启动后的历史缓存，从未被 SDK 清理**。这些缓存对象会被关联到当次上报的真实 GPS，产生跨城远点，污染下游质心与 drift 判定。

| ID | 规则 | 条件 | 动作 | 说明 |
|----|------|------|------|------|
| `ODS-019` | `cell_infos` 陈旧缓存 | `age_sec > max_age_sec`（默认 300s） | 丢弃该对象 | `age_sec` 见下；对象字段缺失/非数字时按 KEEP（保守兜底） |

**字段语义与单位**：

| 字段 | 单位 | 含义 | 基准 |
|------|------|------|------|
| `timeStamp` | **毫秒 或 纳秒**（按字符串长度自动识别）| 设备启动到该 cell 测量的时间 | 设备内部单调时钟 |
| `past_time` | 秒（允许小数）| 当次上报时设备启动已经过的秒数（同报文内各对象相同） | 设备内部单调时钟 |
| `age_sec` | 秒 | 见下单位识别后的公式 | 派生字段 |

**timeStamp 单位自动识别（2026-04-21 新增）**：

全库实测约 **9% 的设备使用纳秒级 timeStamp**（14-16 位大数），剩下 91% 是毫秒（10-13 位）。按字符串长度自动切换除数：

```sql
age_sec =
  floor(past_time) -
  CASE WHEN length(timeStamp) <= 13 THEN timeStamp / 1000          -- 毫秒
       ELSE                              timeStamp / 1000000000     -- 纳秒
  END
```

历史上（本次修复之前）只按毫秒处理，纳秒设备的 `age_sec` 会得到荒谬负数而被误判为合法 → 漏过大量陈旧缓存（案例见 `docs/gps研究/cell漂移问题分析.md §9 cell 21330987`）。

- `timeStamp` / `past_time` 都基于设备**单调计数器**，不依赖用户壁钟时间，方案**不受设备系统时间错误影响**
- `max_age_sec` 配置项：`config/antitoxin_params.yaml` 的 `etl_cell_infos.max_age_sec`（默认 `300`）
- 阈值外的边界处理（字段缺/非数字/age 为负）一律 KEEP，避免因个别畸形数据误伤

**案例参考**：cell `5751357441` 被设备 `RALR2TC3G4910Q53S` 的 5 条"5 小时~4.9 天前缓存"对象持续拖累，batch 7 被误判为 `collision`。详见 `rebuild5/docs/gps研究/cell漂移问题分析.md`。

**不影响 `ss1`**：`ss1` 每条子记录自带独立 unix 毫秒时间戳，走另一套过滤规则 `ODS-020 ~ ODS-022`。

### 关联规则 `ODS-024`（Step 5 label 层，非 Step 1 ETL 清洗）

`ODS-024` 不在 ETL 清洗阶段，属于 Step 5 `label_engine.py` 的簇有效性治理：

| ID | 规则 | 条件 | 动作 | 说明 |
|----|------|------|------|------|
| `ODS-024` | 簇最小设备数门槛 | 簇的 `dev_count < min_cluster_dev_count`（默认 2） | 簇判为无效（不计入 `k_eff`） | 防止"单设备跨多天反复刷成假簇"。单设备 N 天贡献 N 个 dev-day 会通过 `dev_day_pts` 门槛但多样性为 1，不应等同多设备独立确认。案例见 `docs/gps研究/cell漂移问题分析.md §9` |

配置位置：`config/antitoxin_params.yaml` 的 `multi_centroid_v2.min_cluster_dev_count`。

### SS1 解析清洗（`ODS-020 ~ ODS-022`）

`ss1` 是"过去一段时间的信号时序"，一条 `raw_gps` 里包含多个 `ss1` 子记录（批）。SDK 在该批里携带的子记录原则上来自最近一段采集窗口（约 1 小时），但实测有部分子记录是设备启动后几小时甚至几天前的残留；另外单个子记录的 `cell_block` 可能同时出现"当前连接 cell"和"SDK 缓存的过期 cell_id"（例：设备在重庆，但 `cell_block` 里仍带着几天前在北京连过的 LTE cell）。这三条规则治理 `ss1` 上这三种典型污染。

| ID | 规则 | 条件 | 动作 | 说明 |
|----|------|------|------|------|
| `ODS-020` | SS1 批内锚点陈旧过滤 | `batch_max_ts_sec - ts_sec > max_age_from_anchor_sec`（默认 3600s） | 丢弃该子记录 | 以批内最新子记录 `ts_block`（unix 秒）为锚点，锚点前 1 小时外视为 SDK 历史残留；字段缺失按 KEEP |
| `ODS-021` | SS1 无信号 Cell 过滤（tech 匹配）| `sigs JOIN cells ON sig_tech = cell_tech` 改为 `INNER JOIN` | 丢弃无配套信号的 cell | 单条 ss1 子记录里没有配套 `sig_block` 的 cell 大概率是 SDK 缓存 cell_id；实测双卡场景 99.3% 只有 1 个 sig，真双连接仅占 0.3% |
| `ODS-022` | SS1 全 -1 Sig 条目过滤 | `sig_entry` 形如 `^[lng],-1,-1,-1,-1` | 丢弃整条 sig | 四个信号值全 -1 的 sig 无任何信息；配套的 cell 如果还有另一张卡的有效 sig 会保留，否则随 ODS-021 一起丢 |

**关键字段**：

| 字段 | 单位 | 含义 |
|------|------|------|
| `ts_block` | unix 秒/毫秒 | 子记录测量时间（SDK 上报时壁钟时间）|
| `batch_max_ts_sec` | 秒 | 批内（同 `record_id`）所有子记录的 `ts_sec` 最大值，即"这批最新一次测量"|
| `sig_tech` | `l/n/g` | 本条 `sig` 的制式（LTE/NR/GSM）|
| `cell_tech` | `l/n/g` | 本条 `cell` 的制式 |

**配置项**：`config/antitoxin_params.yaml` 的 `etl_ss1` section。

**案例参考**：cell `1353583` 在 batch 7 的 `insufficient` 判定，根因是设备 `A3JS3I8PTCORK0GOB` 在重庆但 `ss1` 的 `cell_block` 里始终混入了北京 LTE 1353583（SDK 过期缓存 cell_id）。ODS-021 直接清掉这类污染。详见 `rebuild5/docs/gps研究/cell漂移问题分析.md`。

**历史遗留**：`ss1` 解析之前还有两个隐性 bug（forward-fill 的 `MAX` 字典序错误、`sig` 识别 tech 范围 `[lng]` 与 `cell` 识别范围 `[lnge]` 不对齐），已在 ODS-020 同期修复，不单独编号。

### TA 占位值剔除（`ODS-023`，2026-04-23 新增）

**背景**：LTE `TimingAdvance`（TA）是 UE↔基站 RTT 时序测量字段，理论换算 `distance_m ≈ TA × 78.125`。仅存在于 `cell_infos.signal_strength.TimingAdvance`；`ss1` 没有这个字段（parse 强制 NULL）。

**发现**：2026-04-22 研究（详见 `docs/gps研究/11_TA字段应用可行性研究.md`）表明：
- **LTE FDD 频段**（earfcn < 36000 或 > 43589）：TA × 78m 公式准确，实测 `TA_p50 × 78 ≈ lib_p90`（比率 1.09）
- **LTE TDD 频段**（earfcn 36000-43589，中国移动 Band 38/39/40/41）：**TA 呈双峰分布**
  - 低峰 0-10（22.8%）：真实 RRC 测量
  - 高峰 16-23（69.7%）：**占位/锁定值，非距离信号**
  - 1490 个 TDD 高峰 cell 里 46% 全 cell 所有设备都报占位（cell-level 禁用 TA），23% 几乎全占位
- TDD cell 的物理 p90 和 FDD 几乎相同（111m vs 134m），说明物理覆盖一致，TA 差 13 倍是协议/基站层面的占位机制

**规则**：

| ID | 规则 | 条件 | 动作 | 说明 |
|----|------|------|------|------|
| `ODS-023` | LTE TDD 占位 TA 剔除 | `cell_origin='cell_infos' AND tech_norm='4G' AND freq_channel BETWEEN 36000 AND 43589 AND timing_advance >= 16` | `timing_advance` 置 NULL | TDD 高峰 TA 是占位值（真实 TA 几乎全部 < 16），保留会误导下游 coverage 统计；低峰 0-15 视为真实测量保留 |

**保留的 TA 样本**：
- FDD 行：完全不受影响（TA 分布天然 < 16）
- TDD 行：保留 `0 ≤ TA ≤ 15` 的真实值（约 23.4% TDD 样本），剔除 `TA ≥ 16` 的占位值（约 76.6%）
- `ss1` 行：本来 timing_advance 就是 NULL，不受影响

**为什么阈值选 16**：TDD 样本分布里 TA 11-15 的占比极低（0.7%），是真实分布末尾；从 TA=16 开始占位值占主导，做为切割点副作用最小。

**下游影响**：
- `etl_cleaned.timing_advance` 非空率下降约 20-25%（因 TDD 占 4G 约一半）
- Step 5 聚合时按 `timing_advance IS NOT NULL` 过滤统计，FDD 统计精度不变，TDD 统计自动收敛到真实低峰
- `cell_infos` 和 `ss1` 的其他字段（GPS、信号强度等）都不受本规则影响

**配置项**：无需在 `antitoxin_params.yaml` 添加（阈值 16 和 earfcn 区间作为硬编码规则在 clean.py 的 ODS_RULES 列表里即可；如果后续需要支持不同频段或运营商策略再外化）。

## 补齐规则（同报文字段对齐）

### 补齐范围声明

```text
只做同报文（同 record_id）内的字段互补
不做跨报文补数
不做基于 trusted Cell 库的知识补数
```

| 边界 | 是否允许 |
|------|----------|
| 同一 `record_id` + 同一 `cell_id` 互补 | 允许 |
| 不同 `record_id` 之间互补 | 禁止 |
| 基于历史可信 Cell 补 GPS / 信号 | 禁止（属于 Step 4） |

### 核心思路

同一次手机上报炸开后，同一个 `cell_id` 可能同时出现在 `cell_infos` 与 `ss1`：

```text
record_id = abc123
├── cell_infos -> cell 12345，有 GPS，有信号，有 WiFi
├── ss1 组1   -> cell 12345，有信号，可能有 GPS
└── ss1 组2   -> cell 12345，有信号，无 GPS
```

为什么可以用 `cell_id` 匹配：
- 匹配范围被限制在同一个 `record_id` 内。
- 同一次手机上报中，同一个 `cell_id` 指向同一物理基站的概率极高。
- 因此 Step 1 允许在同一报文内按 `record_id + cell_id` 做结构性互补。

### `cell_infos` 行之间的补齐

同一 `record_id` + 同一 `cell_id` 的 `cell_infos` 行，时间完全一致，允许全字段补齐。

可补齐字段：

```text
operator_code, lac,
rsrp, rsrq, sinr, dbm,
lon/lat (GPS),
wifi_name, wifi_mac
```

### `ss1` 行的时间约束

`ss1` 每组有自己的 `cell_ts_raw`。对 `ss1 <- donor` 的补齐必须先判断时间差。

| 时间差 | 允许补齐的字段 | 原因 |
|--------|----------------|------|
| `<= 1 分钟` | 全字段可补 | 1 分钟内位置和射频环境通常仍连续 |
| `> 1 分钟` | 只补 `operator_code`、`lac` | 超过 1 分钟后，GPS / 信号 / WiFi 可能已失真 |

### 完整字段补齐矩阵

| 字段 | `cell_infos <- cell_infos` | `ss1 <- cell_infos` | `ss1 <- ss1`（`<=1分钟`） | `ss1 <- ss1`（`>1分钟`） | 说明 |
|------|----------------------------|---------------------|-----------------------------|----------------------------|------|
| `operator_code` | 可补 | 可补 | 可补 | 可补 | 运营商短时间内稳定 |
| `lac` | 可补 | 可补 | 可补 | 可补 | LAC 短时间内稳定 |
| `rsrp` | 可补 | 可补 | 可补 | 不补 | 信号随位置快速变化 |
| `rsrq` | 可补 | 可补 | 可补 | 不补 | 同上 |
| `sinr` | 可补 | 可补 | 可补 | 不补 | 同上 |
| `dbm` | 可补 | 可补 | 可补 | 不补 | 同上 |
| `lon/lat` | 可补 | 可补 | 可补 | 不补 | GPS 随移动变化 |
| `wifi_name` | 可补 | 可补 | 可补 | 不补 | WiFi 环境随位置变化 |
| `wifi_mac` | 可补 | 可补 | 可补 | 不补 | 同上 |

### 补齐优先级

#### GPS 优先级

| 优先级 | 来源 | 说明 |
|--------|------|------|
| 1 | `raw_gps` | `cell_infos` 原始 GPS，优先级最高 |
| 2 | `ss1_own` | `ss1` 本组自带 GPS |
| 3 | `same_cell` | 同报文内 donor 补入 |
| 4 | `none` | 无可用 GPS |

#### 信号优先级

| 优先级 | 规则 |
|--------|------|
| 1 | 优先保留当前行已有信号 |
| 2 | 当前行为空时，从同报文同 `cell_id` donor 补 |
| 3 | 多个候选 donor 时，取最强值（`RSRP` 最大） |

#### 运营商 / LAC 优先级

| 优先级 | 规则 |
|--------|------|
| 1 | 优先保留当前行已有值 |
| 2 | 当前行为空时，从 `cell_infos` donor 补 |
| 3 | 若只有 `ss1` donor，则从最近可用 `ss1` donor 补 |

### 补齐来源标记

#### 通用来源标记

| 标记 | 含义 |
|------|------|
| `original` | 当前行原始就有值 |
| `same_cell` | 来自同报文、同 `cell_id` donor |
| `none` | 没有补到 |

#### GPS 特殊来源标记

GPS 需要区分原始来源，所以比通用标记更细：

| 标记 | 含义 |
|------|------|
| `raw_gps` | `cell_infos` 原始 GPS |
| `ss1_own` | `ss1` 本组自带 GPS |
| `same_cell` | 同报文 donor GPS |
| `none` | 没有补到 |

|------|--------|--------|------|
| GPS 覆盖率 | `84.8%` | `97.5%` | `+12.7%` |
| RSRP 覆盖率 | `86.9%` | `94.6%` | `+7.7%` |

### 实现约束

| 约束 | 说明 |
|------|------|
| 不增减行数 | 字段对齐只填空，不新建、不删除记录 |
| 不覆盖基础真相列 | 原始 / 清洗后的基础列保留，对齐结果写 `_filled` 列 |
| 强制记录来源 | 所有关键补齐结果都必须有来源标记 |

| 不进入知识补数 | 可信赖库补数在 Step 4 单独实现 |

## 下游去重策略（`DEDUP-V2.1`）

Step 1 产出的 `etl_cleaned` 每条子记录带 `cell_origin ∈ {cell_infos, ss1}` 字段，跨 Step 2/4/5 透传至 `cell_sliding_window`，作为 Step 5 `cell_core_gps_day_dedup` 和 Step 2 `_profile_gps_day_dedup` 的去重维度之一。

### 规则说明

**规则名**：`DEDUP-V2.1`（覆盖 Step 2 和 Step 5，2026-04-23 起生效；是对 2026-04-22 `DEDUP-V2` 的修订）

**V1**（已废弃）：每 `(cell, dev, date)` 保留 1 个观察点 —— 天粒度 dev 去重。

**V2**（过渡版）：每 `(cell, dev, cell_origin, 5min_bucket)` 保留 1 个观察点（所有 origin 统一 5 分钟桶）。

**V2.1（当前版）**：按 `cell_origin` 分两套去重键：

| cell_origin | 去重键 | 意图 |
|---|---|---|
| `cell_infos` | `(cell, dev, 'cell_infos', 5min_bucket)` | 每设备每 5 分钟每 origin 1 点（同 V2）|
| `ss1` | `(cell, dev, 'ss1', record_id)` | **每 raw_gps 报文 1 点**（消除 forward-fill 同报文内重复）|

**时间桶定义**（UTC 对齐）：
```sql
date_trunc('minute', event_time_std)
  - ((EXTRACT(MINUTE FROM event_time_std)::int % 5) * INTERVAL '1 minute')
```

### 为什么 V2.1 修订 V2

V2 的 5min 桶对 cell_infos 合适（有 age 过滤保证陈旧被清），但对 ss1 不够 —— ss1 缺少对象级 age 字段，同一条 raw_gps 里 forward-fill 机制会把 1 个 cache cell_id 放大成多个子记录（各自 ts_block 不同但属于同一报文）。5min 桶拦不住这类放大。按 `record_id` 去重（ss1 专用）相当于"每报文 1 票"，消除 forward-fill 的同报文内重复。实测压缩 ss1 样本约 50%，主要压掉的是冗余而非合法切换。

**配套调整**：`etl_ss1.max_age_from_anchor_sec` 从 `3600`（1h）→ `10800`（3h），补偿 record_id 去重可能的合法样本损失。见 `config/antitoxin_params.yaml`。

### 为什么替换 V1（天去重）

原 day 去重有两个目的：
1. 反漂移（防单设备同日刷屏主导半径）
2. 反单点脏数据（防设备偶发异常上报污染）

目的 2 已被 `ODS-019 ~ ODS-023` 在 Step 1 parse / clean 层解决。保留 day 去重只剩目的 1，代价却是：
- 单设备同日合法观察被压成 1 点（见 cell 20752955 案例，19 点 → 1 点）
- 异常占比在 dedup 后被人为放大（raw 层 5% → dedup 后 50%），导致 `gps_anomaly_filter` 的 10% 兜底在小样本 cell 上**失效**

**V2/V2.1 的直接收益**（20 cell 评估样本）：
- 样本密度恢复 1.5~4.4x（正常 cell）/ 3~10x（异常 cell）
- cell 20752955 / 17539075 / 4855663 等"单远点污染"异常的 p90 从百万米级降到百米级
- 正常 cell p90 波动 ±10~15%，多数微收敛，无误伤
- 设备数 `distinct_dev_id` 完全不变

### 方案 B：设备逆频加权 p50/p90（2026-04-23 新增）

DEDUP-V2.1 不能解决"单设备大量上报全部远点"的极端情形（如 GPS 漂移设备贡献 57 个远点占 54%，加权后仍占 1/7=14.3% 超过 10% 阈值）。为缓解此类，Step 5 `build_cell_radius_stats` 引入**方案 B 逆频加权**：

- 每 `is_core` 点权重 `w = 1 / (该设备在该 cell 的 is_core 点数)`
- 每个设备对 cell 的**总权重归一化为 1**（不管它贡献 1 点还是 100 点）
- p50/p90 用**累积权重百分位**（`cum_w / total_w`）而不是排序百分位
- `raw_p90_radius_m` 字段保持无权口径（兼容 UI 对照展示）

**根治条件**：污染设备数 / 正常设备数 ≤ 1/9 时，p90 能回归真实值。对样本多的 cell 普适有效；对极端小样本 cell（如 7 个设备 1 个污染）p90 仍受影响但降低 30%。

### 影响的表和字段

`cell_origin`、`timing_advance`、`freq_channel` 三字段透传链：

```
etl_cleaned
    ↓
path_a_records / _profile_path_b_records / profile_obs / profile_base / candidate_seed_history  (Step 2)
    ↓
enriched_records / snapshot_seed_records  (Step 4)
    ↓
cell_sliding_window → cell_core_gps_day_dedup → cell_radius_stats (加权)  (Step 5)
    ↓
trusted_cell_library（新加 ta_n_obs / ta_p50 / ta_p90 / ta_dist_p90_m / freq_band / ta_verification 6 字段）
```

### 与 `gps_anomaly_filter` 的语义关系

`gps_anomaly_filter.max_anomaly_ratio = 0.10` 和 `max_anomaly_count = 2` 保持不动。DEDUP-V2.1 让分母扩大，10% 阈值在真实密度下生效。Step 5 `build_cell_core_gps_stats` 中的异常过滤逻辑不变。

### 其它注意

- `cell_core_gps_day_dedup` / `_profile_gps_day_dedup` 表名沿用仅为向后兼容，**语义已不再是 day 去重**。
- Step 3 `profile_base.p50/p90/center` 自动承接新口径，和 Step 5 `cell_radius_stats` 统计对齐。
- Step 3 的准入判定只看 `independent_obs`（per-minute 口径，不受本规则影响）。
- 详细研究过程见 `docs/gps研究/12_单设备污染与加权p90方案.md`。
