# 核心表说明

时间：2026-04-14  
适用范围：`rebuild5` 当前主链（Step0/1 -> Step5）以及本轮临时流水线方案  
目标：回答“每张核心表是什么、在哪一步生成、谁会读它、是不是最终口径”

## 1. 阅读约定

本文件只覆盖当前主链真正重要的表，不覆盖：

1. `scripts/archive/` 里的历史脚本表
2. benchmark / research 专用表
3. 第三方库或旧版本残留但当前主链不再消费的对象

表类型约定：

| 类型 | 含义 |
|---|---|
| `源表` | 当前阶段的主输入 |
| `中间表` | 批次内可重建、可覆盖的临时结果 |
| `冻结表` | Step3 产出的当批快照结果 |
| `发布表` | Step5 发布后的正式结果，UI / 服务层最终读取 |
| `元数据表` | 运行记录、统计、数据集配置 |
| `视图` | 只读投影，不单独存储 |

命名约定：

1. `rebuild5._*` 基本都是批次内中间表
2. `rebuild5_meta.*` 基本都是元数据 / 运行统计
3. `trusted_*` 基本都是冻结结果或发布结果
4. `rebuild5_tmp.*` 是本轮临时流水线新增的 staging 对象，不属于长期产品表

## 2. 哪些表是最终口径

先给最重要的结论。

### 2.1 服务层 / UI 最终读取

服务层查询当前直接读取：

| 表 | 作用 |
|---|---|
| `rebuild5.trusted_cell_library` | Cell 级正式发布结果 |
| `rebuild5.trusted_bs_library` | BS 级正式发布结果 |
| `rebuild5.trusted_lac_library` | LAC 级正式发布结果 |
| `rebuild5_meta.step5_run_stats` | 当前最新版本信息与发布摘要 |

说明：服务层默认读这些表的 `MAX(batch_id)`，也就是“最新已发布版本”。

### 2.2 治理侧关键读取

治理侧通常还会读：

| 表 | 作用 |
|---|---|
| `rebuild5.trusted_snapshot_cell / bs / lac` | Step3 冻结快照 |
| `rebuild5.snapshot_diff_cell / bs / lac` | 相邻版本 diff |

## 8. 服务层直读关系

当前服务查询默认读最新 `batch_id`。

| 查询层 | 直接读取 |
|---|---|
| Cell 查询 | `trusted_cell_library` |
| BS 查询 | `trusted_bs_library` + `trusted_cell_library` |
| LAC 查询 | `trusted_lac_library` + `trusted_bs_library` |
| 全局版本条 | `step5_run_stats` 最新记录 |

结论：如果要和 UI 对上，最关键的是：

1. `trusted_cell_library`
2. `trusted_bs_library`
3. `trusted_lac_library`
4. `step5_run_stats`

## 10. 最后一句话

如果只记住最关键的主线：

```text
raw_gps
  -> etl_cleaned
  -> path_a_records / profile_base
  -> trusted_snapshot_*
  -> enriched_records / gps_anomaly_log
  -> trusted_*_library + collision_id_list + *_centroid_detail
```

而 UI / 服务层最终对外口径，永远以最新 `batch_id` 的 `trusted_*_library` 为准。
