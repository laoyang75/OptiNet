# Step 2: 基础画像 + 分流

Step 2 是 `rebuild5` 从 ETL 进入业务主链后的第一个版本化步骤。它不是单纯的“指标计算器”，而是一个先做可信库碰撞、再做路径分流、最后只对未命中且有 GPS 的数据计算基础画像的路由节点。

Step 1 输出的是通用结构化事实；从 Step 2 开始，所有输入、输出、统计和页面展示都绑定到当前数据集（如 `sample_6lac`、`beijing_7d`）与当前运行批次。

## 模块定位

### 这是一个版本化分流节点

| 项目 | 说明 |
|------|------|
| 模块角色 | Step 2 负责“可信库碰撞 + 路由分流 + 基础指标”，不是完整画像终点 |
| 所属上下文 | 从这一步开始进入数据版本语境，所有产出都带当前数据集标记 |
| 上游输入 | `etl_cleaned`（来自 Step 1） |
| 参考输入 | 上一轮可信小区正式库（`trusted_cell_library`，发布版本对应 `trusted_snapshot_t-1`）、冲突小区编号清单（`collision_id_list`，来自 Step 5.1） |
| 下游输出 | `path_a_records` -> Step 4，`profile_base` -> Step 3，Path C -> 丢弃 |
| 统计输出 | `step2_run_stats`，记录本次分流和基础画像统计 |

### 当前批次只读上一轮已发布正式库

Step 2 必须遵守冻结快照原则：

```text
当前数据集的本批 Step 2
    只能读取上一轮已发布的可信小区正式库
    其发布版本必须对应 trusted_snapshot_t-1
    不能读取本批刚算出的 Step 3 / Step 5 结果
```

这意味着：
- Path A 的“命中可信库”只以上一轮已发布的 `trusted_cell_library` 为准，不看本批新增对象。
- 本批在 Step 5 新发布进入正式库的 Cell，只能在下一批次开始参与 Path A 命中。
- Step 2 统计属于当前数据集当前批次，不会回写上一版正式库或快照。

### 数据版本从这里开始生效

| 项目 | Step 1 | Step 2-5 |
|------|--------|-----------|
| 是否绑定数据集 | 否，ETL 是通用工具 | 是，全部绑定当前数据集 |
| 统计口径 | 按数据源 / 导入批次 | 按数据集 / 运行批次 / 快照版本 |
| 关键标识 | `source_id`、`run_at` | 数据集标识（`dataset_key`）、运行批次编号（`run_id`）、引用快照版本（`trusted_snapshot_version`） |

### 本步明确不做的事情

| 不做项 | 放在哪一步 | 原因 |
|--------|------------|------|
| 全局 `cell_id` 碰撞扫描 | Step 5.1 | 需要全库、跨日、跨对象观察，不属于 Step 2 的局部路由职责 |
| 漂移分析（日质心、`max_spread_m`、`ratio`） | Step 5 | 需要多日轨迹与异常标签，不属于基础指标 |
| 多质心检测 | Step 5.5 | 只处理异常子集，避免 Step 2 过重 |
| 碰撞 / 迁移 / 大覆盖分类标签 | Step 5 | 属于完整画像维护，不属于路由前置指标 |
| 置信度分级、规模分级 | Step 5 | Step 2 只保留 Step 3 必需的基础统计子集 |

## 2.1 可信库碰撞

### 匹配目标

Step 2 的命中目标不是“所有历史出现过的 Cell”，而是上一轮已经发布、可供命中的可信小区正式库（`trusted_cell_library`）。冷启动时因为正式库还不存在，所以不会有 Path A 命中。

| 项目 | 规则 |
|------|------|
| 匹配来源 | 上一轮已发布的 `trusted_cell_library` |
| 匹配粒度 | Cell 粒度 |
| 业务识别口径 | `(operator_code, tech_norm, lac, cell_id)` |
| 辅助属性 | `bs_id`、`operator_cn` 作为画像维度保留，不作为 Cell 识别键 |
| 结果语义 | 命中表示该 Cell 已被上一版系统验证过，本批新记录不再重复进入 Step 3 重评估 |

### 基础匹配规则

| 场景 | 命中条件 | 结果 | 属性 |
|------|----------|------|------|
| 正常 `cell_id` | `cell_id` 相同，且至少一个 `operator_code` 或 `lac` 相同 | 命中可信库，进入 Path A | 固定规则 |
| 冷启动（无上一轮 `trusted_cell_library`） | 无可命中对象 | 全部按未命中处理 | 固定规则 |
| 同一 `cell_id` 无法满足运营商 / LAC 约束 | 不算命中 | 继续走未命中分流 | 固定规则 |

为什么不能只用 `cell_id`：
- 不同运营商可能复用同一个 `cell_id`。
- 同一运营商不同 `LAC` 下也可能出现同号复用。
- 只按 `cell_id` 命中会把错误对象送进 Step 4，破坏可信库边界。

### 三层 Path A 匹配与碰撞防护

Path A 匹配采用三层逐级放宽的策略，目的是最大化可信库命中率，同时对碰撞 cell_id 保持安全防护。

`collision_id_list` 由 Step 5.1 产出（B 类映射表），记录同一 `cell_id` 被多个 `(operator_code, lac)` 组合使用的情况。Step 2 只消费这个结果，不负责产生。

#### Layer 1: 精确匹配

| 条件 | 说明 |
|------|------|
| `cell_id + tech_norm + operator_code + lac` 全部非空且匹配 | 最可靠的匹配，无论是否在碰撞表中 |
| 结果 | 直接 Path A |

#### Layer 2: 宽松匹配（仅限非碰撞 cell_id）

| 条件 | 说明 |
|------|------|
| `cell_id` 匹配，但 `operator_code` 或 `lac` 为空 | 原始上报数据经常缺运营商/LAC |
| `cell_id` 不在 `collision_id_list` 中 | cell_id 全局唯一，不存在跨运营商复用 |
| 结果 | Path A（安全，因为 cell_id 无碰撞） |

这一层的核心价值：**拯救大量缺运营商/LAC 信息的有效记录**。对于 cell_id 唯一的大多数 Cell，即使原始数据缺运营商也能安全匹配和补数。

#### Layer 3: 碰撞 cell_id + GPS 距离回退

| 条件 | 说明 |
|------|------|
| `cell_id` 在 `collision_id_list` 中，且 `operator_code` 为空 | 无法通过运营商确认是哪个 combo |
| 有 GPS 且距离可信质心 `< 2200m` | GPS 接近度作为回退依据 |
| 结果 | `collision_matched` → Path A |
| 无 GPS 或距离 `>= 2200m` | 无法确认 → Path B/C |

GPS 接近判定使用工程近似距离公式：

```text
dx_m = (lon_in - lon_ref) * 85300
dy_m = (lat_in - lat_ref) * 111000
dist_m ≈ sqrt(dx_m^2 + dy_m^2)
```

#### 脏数据自愈机制

Layer 2 放宽匹配可能引入少量错误数据。自愈链路：
1. Step 4 补数时，GPS 距离 > 2200m 的记录进入 `gps_anomaly_log`
2. Step 5 汇总 `gps_anomaly_log` 做时序判定，标记异常 Cell
3. 下一轮 Step 5 维护时过滤异常数据，可信库保持收敛

其中：
- `lon_ref` / `lat_ref` 来自上一轮 `trusted_cell_library` 中该可信 Cell 的维护后中心点。
- `85300 m/°` 与 `111000 m/°` 是全局固定物理常量，不在 UI 中调整。

### 匹配判定顺序

```text
1. 先查上一轮 `trusted_cell_library` 中是否存在同 cell_id 候选
2. 候选未被标记碰撞 -> 用正常规则判定
3. 候选已被标记碰撞 -> 升级为三字段 + GPS 接近判定
4. 命中 -> Path A
5. 未命中 -> 继续做 Path B / Path C 分流
6. 碰撞命中但 GPS 明显不接近 -> 丢弃
```

### 碰撞相关统计

`step2_run_stats` 至少记录以下碰撞命中统计：

| 指标 | 含义 | 粒度 |
|------|------|------|
| `collision_candidate_count` | 本批遇到的已标记碰撞 `cell_id` 记录数 | 记录级 |
| `collision_path_a_match_count` | 已标记碰撞 `cell_id` 中最终命中 Path A 的记录数 | 记录级 |
| `collision_pending_count` | 已标记碰撞 `cell_id` 中因缺少可比 GPS 未认定命中的记录数 | 记录级 |
| `collision_drop_count` | 已标记碰撞 `cell_id` 中因 GPS 不接近被丢弃的记录数 | 记录级 |
| `collision_match_rate` | `collision_path_a_match_count / collision_candidate_count` | 运行级 |
| `collision_drop_rate` | `collision_drop_count / collision_candidate_count` | 运行级 |

## 2.2 分流路由

### 三条路径

| 路径 | 进入条件 | 输出 | 去向 | 说明 |
|------|----------|------|------|------|
| Path A | 命中 `trusted_cell_library` | `path_a_records` | Step 4 知识补数 | 已评估过的可信 Cell，不再重算基础画像 |
| Path B | 未命中，且具备可用于空间计算的 GPS 证据 | `profile_base` | Step 3 流式评估 | 新 Cell 或尚未进入可信库的 Cell，需要计算基础指标 |
| Path C | 未命中，且缺少可用于空间计算的 GPS 证据 | 丢弃 | 丢弃 | 数据不全，可信库未建时无法补数，没有构建价值，直接丢弃 |

说明：
- Path A 保持记录级输出，方便下游继续补数。Path C 直接丢弃，不存储。
- Path B 输出的是 Cell 粒度的聚合结果 `profile_base`，供 Step 3 直接评估。
- Step 2 的“有 GPS”是指该 Cell 在本批次未命中记录中至少存在可用于空间统计的有效 GPS 观测；用于质心和半径计算时，只使用原始有效 GPS，不把 Step 1 的结构性补齐 GPS 当作真实定位证据。

### 分流判定流程图（ASCII）

```text
etl_cleaned
   |
   +--> 读取当前数据集上下文 + trusted_cell_library + collision_id_list
          |
          +--> 按 cell_id 查可信候选
                 |
                 +--> 命中正常 cell_id 规则 ------------------> Path A -> path_a_records -> Step 4
                 |
                 +--> 命中碰撞 cell_id 规则
                 |        |
                 |        +--> 三字段全匹配 + GPS 距离 < 2200m --> Path A -> path_a_records -> Step 4
                 |        |
                 |        +--> 三字段全匹配但 GPS 距离 >= 2200m -> drop + collision stats
                 |        |
                 |        +--> 缺少可比 GPS -------------------> 继续按未命中处理
                 |
                 +--> 未命中
                          |
                          +--> 有有效 GPS ---------------------> Path B -> profile_base -> Step 3
                          |
                          +--> 无有效 GPS ---------------------> Path C -> 丢弃（数据不全）
```

### 路由判定顺序

| 顺序 | 判定 | 结果 |
|------|------|------|
| 1 | 先判定是否命中可信库 | 命中直接进入 Path A |
| 2 | 对命中候选先做碰撞局部防护 | 防止错误 Path A 命中 |
| 3 | 对未命中记录判断是否具备有效 GPS | 有则去 Path B，无则去 Path C |
| 4 | 对碰撞误命中单独记丢弃 | 不进入 Path B / Path C |

### 分流统计

`step2_run_stats` 至少记录以下分流统计：

| 指标 | 含义 | 粒度 |
|------|------|------|
| `input_record_count` | Step 2 输入总记录数（来自 `etl_cleaned`） | 记录级 |
| `path_a_record_count` | Path A 记录数 | 记录级 |
| `path_b_record_count` | 进入 Path B 的记录数 | 记录级 |
| `path_b_cell_count` | Path B 最终产出的 Cell 数 | Cell 级 |
| `path_c_drop_count` | Path C 丢弃记录数 | 记录级 |
| `path_a_ratio` | `path_a_record_count / input_record_count` | 运行级 |
| `path_b_ratio` | `path_b_record_count / input_record_count` | 运行级 |
| `path_c_drop_ratio` | `path_c_drop_count / input_record_count` | 运行级 |
| `collision_drop_count` | 碰撞误命中被丢弃的记录数 | 记录级 |

Path B 还要单独统计数据完整度：

| 指标 | 含义 |
|------|------|
| `path_b_complete_cell_count` | `gps_original_ratio = 1` 且 `signal_original_ratio = 1` 的 Cell 数 |
| `path_b_partial_cell_count` | 其余进入 Path B 的 Cell 数 |
| `path_b_avg_gps_original_ratio` | Path B Cell 的平均 GPS 原始覆盖率 |
| `path_b_avg_signal_original_ratio` | Path B Cell 的平均信号原始覆盖率 |

## 2.3 基础指标计算（仅 Path B）

Step 2 的基础画像只对 Path B 做，而且只保留 Step 3 判级必需的基础字段。所有漂移、分类、异常标签、规模分级都留给 Step 5 的完整维护阶段。

### 计算粒度

| 项目 | 口径 |
|------|------|
| 输入范围 | Path B 中未命中可信库且存在有效 GPS 证据的记录 |
| 聚合粒度 | 当前数据集内的 Cell 粒度 |
| 主业务识别口径 | `(operator_code, tech_norm, lac, cell_id)` |
| 保留维度 | `operator_cn`、`bs_id` 作为维度字段同步输出 |

### 独立观测点去重

独立观测点的核心口径采用已经验证过的分钟级方案：分钟级去重，而不是小时级去重。

| 项目 | 规则 | 依据 |
|------|------|------|
| 去重窗口 | `1` 分钟 | 可配置，默认 `1` |
| 业务口径 | 同一 Cell 在同一分钟内的多条上报压成 `1` 个独立观测点 | 已验证方案 |
| 为什么不是 `1` 小时 | 小时级会把真实重复观测压掉过多，导致样本量明显不足 | 研究结论 |
| 为什么不是更大窗口 | 设备上报呈脉冲式，`98%` 间隔 `< 5` 分钟，放大窗口收益很小 | 研究结论 |
| 1 分钟方案收益 | 相比小时级方案，额外捕获 `21.6%` 观测点，且新增碰撞检出经验证为真发现 | 研究结论 |

去重后的单观测点聚合方式：

| 字段 | 聚合方式 |
|------|----------|
| `obs_minute` | `date_trunc('minute', event_time_std)` |
| `lon` / `lat` | 仅对原始有效 GPS 求平均，得到该分钟观测点坐标 |
| `rsrp` / `rsrq` / `sinr` | 仅对物理有效范围内的值求平均 |
| `raw_records` | 该分钟原始记录数 |
| `gps_original_records` | 该分钟带原始 GPS 的原始记录数 |
| `signal_original_records` | 该分钟带原始信号的原始记录数 |

### 质心计算

当前主线中，Step 2 的位置计算已经从“直接对全部点取中位数”升级为“主热点 seed + 核心点过滤 + 中位数重算”。

流程如下：

1. 对分钟级独立观测点做 `SnapToGrid`
2. 取观测最密的网格点作为 seed
3. 计算每个点到 seed 的距离
4. 根据 `core_position_filter` 配置得到 `keep_radius_m`
5. 仅用核心点对经度、纬度做 `PERCENTILE_CONT(0.5)` 计算质心
6. 基于核心点到质心的距离计算 `p50_radius_m / p90_radius_m`

也就是说：

- 中位数仍然是主质心算法
- 但输入点集不再是“全部原始有效 GPS”，而是“核心点过滤后的点集”

| 算法 | 偏差中位数 | 偏差均值 | 结论 |
|------|-----------|---------|------|
| AVG | `61.4m` | `296.2m` | 容易被极端值拉偏 |
| RSRP 加权 | `61.8m` | `300.4m` | 对定位精度没有帮助 |
| 中位数 | `8.6m` | `34.0m` | 默认采用，天然抗碰撞、抗噪声 |

按散布类型看，中位数优势更明显：

| Cell 散布类型 | AVG 偏差 | 中位数偏差 | 提升倍数 |
|---------------|---------|-----------|----------|
| 紧凑（`<200m`） | `25.6m` | `6.2m` | `4x` |
| 中等（`200-500m`） | `86.8m` | `12.8m` | `7x` |
| 散布（`500-1500m`） | `271.8m` | `11.4m` | `24x` |
| 极散（`>1500m`） | `1133.4m` | `16.1m` | `70x` |

结论：
- Step 2 默认主质心算法仍然是“中位数”。
- UI 可以提供 `中位数 / AVG / 加权` 的算法切换入口，但默认值固定为中位数，其他选项只作为实验比较配置，不改变系统默认推荐口径。

### 空间半径

当前主线中，P50 / P90 半径基于“核心点到质心的距离分位数”计算，而不是直接对全部原始有效 GPS 点计算。

| 指标 | 定义 | 属性 |
|------|------|------|
| `p50_radius_m` | 观测点到质心距离的中位数 | 固定算法 |
| `p90_radius_m` | 观测点到质心距离的 `90%` 分位 | 固定算法 |
| 经度转米系数 | `85300 m/°` | 固定物理常量 |
| 纬度转米系数 | `111000 m/°` | 固定物理常量 |

距离公式：

```text
dx_m = (obs_lon - center_lon) * 85300
dy_m = (obs_lat - center_lat) * 111000
dist_m ≈ sqrt(dx_m^2 + dy_m^2)
```

说明：
- `85300 m/°` 与 `111000 m/°` 继承全局约定，是算法常量，不放入 `profile_params.yaml`。
- Path B 先对有原始有效 GPS 的独立观测点计算 seed / cutoff / 核心点，再基于核心点计算 P50 / P90。
- 如果某个 Cell 在 Path B 中没有足够的有效 GPS 点，则 `center_lon` / `center_lat` / `p50_radius_m` / `p90_radius_m` 可为空，但该 Cell 不应被误送入 Path A。

### 基础统计字段（`profile_base`）

`profile_base` 必须与历史 Cell 画像表的基础字段子集保持一致，只保留 Step 3 真正需要的基础画像，不携带漂移、分类、置信度和规模字段。

| 分组 | 字段 | 说明 |
|------|------|------|
| 维度 | `operator_code` | 运营商编码 |
| 维度 | `operator_cn` | 运营商中文名 |
| 维度 | `lac` | 位置区编码 |
| 维度 | `bs_id` | 基站编码 |
| 维度 | `cell_id` | Cell 编码 |
| 维度 | `tech_norm` | 标准制式，参与 Cell 识别，同时作为输出维度保留 |
| 观测统计 | `independent_obs` | 分钟级独立观测点数 |
| 观测统计 | `independent_devs` | 原始记录去重后的独立设备数 |
| 观测统计 | `independent_days` | 出现独立观测的天数 |
| 观测统计 | `record_count` | 参与聚合的原始记录总数 |
| 时间 | `observed_span_hours` | 首末观测点跨度（小时） |
| 时间 | `active_days` | 活跃天数；当前口径与 `independent_days` 一致，保留是为了与下游页面和旧表字段兼容 |
| 空间 | `center_lon` | 核心点过滤后中位数质心经度 |
| 空间 | `center_lat` | 核心点过滤后中位数质心纬度 |
| 空间 | `p50_radius_m` | 核心点到质心的中位距离 |
| 空间 | `p90_radius_m` | 核心点到质心的 `90%` 分位距离 |
| 信号 | `rsrp_avg` | 原始有效 RSRP 的平均值 |
| 信号 | `rsrq_avg` | 原始有效 RSRQ 的平均值 |
| 信号 | `sinr_avg` | 原始有效 SINR 的平均值 |
| 数据质量 | `gps_valid_count` | 有效 GPS 的独立观测点数 |
| 数据质量 | `gps_original_count` | 原始记录中带 GPS 的记录数 |
| 数据质量 | `gps_original_ratio` | `gps_original_count / record_count` |
| 数据质量 | `gps_valid_ratio` | `gps_valid_count / independent_obs` |
| 数据质量 | `signal_original_ratio` | `signal_original_count / record_count` |

补充说明：
- `signal_original_count` 已作为 `profile_base` 的输出字段，供 Step 3 计算 `signal_confidence` 使用。
- `dist_to_bs_m`、`drift_pattern`、`position_grade`、`gps_confidence`、`signal_confidence`、`cell_scale`、`is_collision` 等字段全部不进入 `profile_base`。
- `profile_base` 的目标是给 Step 3 提供“可判级的基础证据”，而不是把完整画像提前到 Step 2。

### Step 2 不做的扩展指标

| 项目 | 是否在 Step 2 计算 | 放在哪一步 |
|------|--------------------|------------|
| 漂移分析 | 否 | Step 5 |
| 碰撞分类（`collision` / `stable` / `migration` 等） | 否 | Step 5 |
| 多质心检测 | 否 | Step 5.5 |
| 置信度分级 | 否 | Step 5 |
| Cell 规模分级 | 否 | Step 5 |

## 2.4 分流统计 + UI 页面

### 页面目标

Step 2 页面不是纯算法页，而是”当前数据集在可信命中 / 待评估 / 丢弃三个方向上如何流动”的路由总览页。

| 页面模块 | 作用 |
|----------|------|
| 数据版本头部 | 显示当前数据集、运行批次、引用的正式库版本（对应 `trusted_snapshot_t-1`） |
| 分流总览卡片 | 展示输入总量、Path A/B/C 数量与比例、碰撞丢弃量 |
| 分流可视化 | 用饼图或条形图展示三条路径的占比 |
| 规则配置区 | 调整可配置阈值与算法选项 |
| 基础指标统计表 | 展示 Path B 的 Cell 基础统计与覆盖率概览 |

### 统计指标

页面最少展示以下统计卡片：

| 指标 | 展示内容 |
|------|----------|
| 当前数据集 | 如 `sample_6lac` |
| 当前运行批次 | `run_id` / `run_at` |
| 引用正式库版本 | `trusted_cell_library` 发布版本（对应 `trusted_snapshot_t-1`） |
| 总输入量 | 来自 Step 1 的 `input_record_count` |
| Path A | 记录数 + 占比 |
| Path B | 记录数 + 占比，同时补充 Path B Cell 数 |
| Path C | 数量 + 占比 |
| 碰撞丢弃 | 数量 + 占比 |

Path B 的数据完整度需要单独展示：

| 指标 | 说明 |
|------|------|
| `path_b_complete_cell_count` | GPS 与信号原始覆盖都完整的 Cell 数 |
| `path_b_partial_cell_count` | GPS 或信号存在缺口的 Cell 数 |
| `path_b_avg_gps_original_ratio` | Path B 平均 GPS 原始覆盖率 |
| `path_b_avg_signal_original_ratio` | Path B 平均信号原始覆盖率 |

碰撞防护结果需要单独展示：

| 指标 | 说明 |
|------|------|
| `collision_candidate_count` | 已标记碰撞 `cell_id` 进入命中检查的记录数 |
| `collision_path_a_match_count` | 通过局部防护后成功进入 Path A 的记录数 |
| `collision_pending_count` | 缺少可比 GPS、未直接命中的记录数 |
| `collision_drop_count` | 因 GPS 不接近而被丢弃的记录数 |

### 规则配置区

规则配置区只暴露业务阈值和实验算法，不暴露物理常量。

| 配置项 | 默认值 | 用途 | 属性 |
|--------|--------|------|------|
| `collision_match_gps_threshold_m` | `2200` | 碰撞 `cell_id` 的命中 GPS 接近阈值 | 可配置 |
| `obs_dedup_window_minutes` | `1` | 独立观测点的时间窗口 | 可配置 |
| `centroid_algorithm` | `median` | 质心算法，支持 `median / avg / weighted` | 可配置，默认 `median` |
| 中国边界范围 | `73~135 / 3~54` | GPS 有效性判定 | 固定 |
| 经度 / 纬度转米系数 | `85300 / 111000` | 距离计算 | 固定 |
| 信号有效范围 | `RSRP -156~-1`、`RSRQ -34~10`、`SINR -23~40` | 信号均值统计（与 ODS 清洗边界一致） | 固定 |

### 基础指标统计表

基础指标统计表按当前数据集展示 Path B 的聚合概况，至少包含：

| 指标 | 含义 |
|------|------|
| `path_b_cell_count` | 当前批次需要进入 Step 3 的 Cell 数 |
| `avg_independent_obs` | 平均独立观测点数 |
| `avg_independent_devs` | 平均独立设备数 |
| `avg_observed_span_hours` | 平均观测跨度 |
| `avg_p50_radius_m` / `avg_p90_radius_m` | 平均空间离散度 |
| `avg_gps_original_ratio` | 平均 GPS 原始覆盖率 |
| `avg_signal_original_ratio` | 平均信号原始覆盖率 |

## 数据表

### 数据表链路

```text
etl_cleaned
    -> 可信库碰撞命中 -> path_a_records -> Step 4
    -> 未命中且有 GPS -> profile_base   -> Step 3
    -> 未命中且无 GPS -> 丢弃

trusted_cell_library + collision_id_list
    -> 只读参考，不在 Step 2 内回写

step2_run_stats
    -> 记录本次 Step 2 的运行统计与页面展示指标
```

### 表定义

| 表 | 类型 | 粒度 | 说明 |
|----|------|------|------|
| `etl_cleaned` | 输入表 | 记录级 | Step 1 最终输出，Step 2 唯一业务输入 |
| `trusted_cell_library` | 参考表 | Cell 级 | 上一轮已发布的可信小区正式库，只读 |
| `collision_id_list` | 参考表 | `cell_id` 级 | 来自 Step 5.1 的冲突小区编号清单，只读 |
| `profile_base` | 输出表 | Cell 级 | 仅 Path B 的基础画像结果，供 Step 3 使用 |
| `path_a_records` | 输出表 | 记录级 | 命中可信库的原始记录，供 Step 4 使用 |
| Path C | 丢弃 | 记录级 | 未命中且无 GPS，直接丢弃不存储 |
| `step2_run_stats` | 统计表 | 运行级 | 当前数据集当前批次的分流与基础画像统计 |

### `step2_run_stats` 最少记录项

| 字段 | 说明 |
|------|------|
| `dataset_key` | 当前数据集标记 |
| `run_id` | 本次 Step 2 运行标识 |
| `run_at` | 运行时间 |
| `trusted_snapshot_version` | 引用的上一版冻结快照 |
| `input_record_count` | Step 2 输入记录数 |
| `path_a_record_count` / `path_a_ratio` | Path A 数量与占比 |
| `path_b_record_count` / `path_b_ratio` | Path B 记录数量与占比 |
| `path_b_cell_count` | Path B 输出 Cell 数 |
| `path_c_drop_count` / `path_c_drop_ratio` | Path C 丢弃数量与占比 |
| `collision_candidate_count` | 进入碰撞防护的记录数 |
| `collision_path_a_match_count` | 通过碰撞防护并命中的记录数 |
| `collision_pending_count` | 因缺少可比 GPS 未直接命中的记录数 |
| `collision_drop_count` | 被碰撞防护丢弃的记录数 |
| `path_b_complete_cell_count` / `path_b_partial_cell_count` | Path B 完整 / 部分缺失分布 |
| `path_b_avg_gps_original_ratio` | Path B 平均 GPS 覆盖率 |
| `path_b_avg_signal_original_ratio` | Path B 平均信号覆盖率 |

## 与其他步骤的接口

| 方向 | 接口 | 说明 |
|------|------|------|
| 从 Step 1 接收 | `etl_cleaned` | 已完成解析、清洗、同报文字段对齐的结构化输入 |
| 从全局接收 | `trusted_cell_library` | 上一轮已发布的可信小区正式库，只读 |
| 从全局接收 | `collision_id_list` | 来自 Step 5.1 的冲突小区编号清单，只读 |
| 输出到 Step 3 | `profile_base` | 未命中可信库且有 GPS 的 Cell 基础指标 |
| 输出到 Step 4 | `path_a_records` | 命中可信库的原始记录，由 Step 4 继续补数 |
| 丢弃 | Path C | 未命中且无 GPS 的记录，数据不全直接丢弃 |
| 不直接输出到 Step 5 | 无 | Step 5 自行做全库碰撞确认、漂移和高级标签计算 |

结论：Step 2 固定采用“分钟级去重 + 中位数质心 + P50/P90 半径”的基础画像算法，同时新增“可信库命中 + 三路分流”职责，把完整画像拆成“基础画像前置 + 深度维护后置”的两段式链路。
