# UI设计总纲

本文件定义 `rebuild5` 的 UI 设计框架，并记录已完成的设计实现。

上半部分是设计总纲——系统要有哪些页面、每个页面给谁用、用户进入页面后要完成什么事。下半部分是设计实现——22 个页面已落地为可运行的 Vue 3 + TypeScript 前端工程，位于 `rebuild5/frontend/design/`，执行 `npm run dev` 即可启动。

| 快速入口 | 说明 |
|----------|------|
| 设计总纲 | 本文件上半部分 |
| 页面分册 | `rebuild5/ui/` 目录下的 8 个 Markdown 文件 |
| 设计实现 | `rebuild5/frontend/design/`（Vue 3 + TS，22 页 + 6 个通用组件） |
| 运行方式 | `cd rebuild5/frontend/design && npm install && npm run dev` |
| 截图参考 | `rebuild5/frontend/design/screenshots/` |
| 设计实现详情 | [跳到”设计实现”章节](#设计实现) |

- `rebuild5/docs/01a_数据源接入_功能要求.md`、`rebuild5/docs/02_基础画像.md`、`rebuild5/docs/03_流式质量评估.md`、`rebuild5/docs/04_知识补数.md`、`rebuild5/docs/05_画像维护.md`、`rebuild5/docs/06_服务层_运营商数据库与分析服务.md`、`rebuild5/docs/07_数据集选择与运行管理.md`、`rebuild5/docs/09_控制操作_初始化重算与回归.md`

## 设计目标

| 目标 | 解释 |
|------|------|
| 先讲用户问题，再讲技术步骤 | 页面不是把 Step 1-6 平铺出来，而是让用户知道“我来这里要解决什么问题” |
| 统一版本上下文 | 管道层和服务层页面都必须能回答“当前看的是哪一个数据集、哪次运行、哪个版本” |
| 治理视角与业务视角分开 | 管道层页面面向管理员和治理分析师；服务层页面面向业务用户 |
| 从总览到明细 | 先看概况，再看异常，再看对象明细，再看规则来源 |
| 方便后续逐页设计 | 本轮先搭框架，后续每个页面都可以在 `rebuild5/ui/` 下单独细化 |

## 用户分层

| 用户 | 主要目标 | 关注页面 |
|------|----------|----------|
| 系统管理员 | 选择数据集、运行主链、核对版本、回看运行历史 | 数据集选择、运行历史、ETL 页面、系统配置 |
| 治理分析师 | 找异常、看流转、判断为什么没晋升、为什么被拦截、为什么退出 | 基础画像与分流、流转总览、观察工作台、评估页、知识补数、维护页 |
| 业务用户 | 查询基站位置、看覆盖、导出报表 | 基站查询、覆盖分析、统计报表 |

## 产品分区

`rebuild5` 的 UI 应拆成三层：

```text
全局层
  -> 数据集选择与运行管理
  -> 运行历史与控制操作记录

治理控制台
  -> ETL 数据接入
  -> 基础画像与分流
  -> 流转评估
  -> 知识补数
  -> 画像维护
  -> 系统配置

服务控制台
  -> 基站查询
  -> 覆盖分析
  -> 统计报表
```

### 为什么这样拆

| 分区 | 为什么单独成区 |
|------|----------------|
| 全局层 | 回答“系统现在跑哪批数据、当前版本是哪次运行产生的” |
| 治理控制台 | 回答“数据怎么进来、怎么流转、为什么通过或失败、为什么被维护成这样” |
| 服务控制台 | 回答“最终能给业务用户提供什么查询和分析能力” |

- 服务控制台必须与治理控制台明确分区，避免业务用户误入治理页面。

## 通用页面骨架

所有页面都应遵守同一套信息骨架，便于后续视觉设计统一。

### 通用头部

| 字段 | 作用 |
|------|------|
| 数据集标识（`dataset_key`） | 说明当前页面对应哪批数据 |
| 运行批次编号（`run_id`） | 说明当前页面对应哪次运行 |
| 快照版本号（`snapshot_version`）或引用版本 | 说明页面读的是哪个结果版本 |
| 页面状态 | 运行中 / 已完成 / 验证中 / 已发布 / 已回退 |
| 最近更新时间 | 说明页面内容新鲜度 |

### 通用布局分区

| 区域 | 作用 |
|------|------|
| 顶部版本条 | 固定展示数据版本、运行批次和页面状态 |
| 左侧筛选区 | 控制数据范围，如运营商、LAC、状态、时间 |
| 概览卡片区 | 先给总量、比例、异常概览 |
| 主内容区 | 图表、表格、列表、拓扑或流程图 |
| 详情抽屉 / 右侧明细 | 看单个对象、单次运行、单条规则命中原因 |
| 跳转区 | 明确告诉用户下一步该去哪个页面 |

### 页面行为原则

| 原则 | 说明 |
|------|------|
| 先看结果，再看原因 | 默认先展示概况，再允许展开规则和原始证据 |
| 先看当前，再看历史 | 默认展示当前版本状态，再提供 diff / 趋势 |
| 从卡片跳明细 | 每个概览块都应能跳到对应列表或明细 |
| 规则要可解释 | 页面中的异常或状态变化，都要能看到命中规则 |
| 空状态可解释 | 不能只写“暂无数据”，要解释是冷启动、过滤、无变化还是未运行 |

## 典型使用路径

### 路径一：新数据集接入后看主链是否跑通

```text
数据集选择
  -> 运行历史
  -> 数据源注册 / 字段审计 / 解析 / 清洗 / 补齐
  -> 基础画像与分流
  -> 流转总览
  -> Cell / BS / LAC 评估
```

### 路径二：定位为什么某个对象没有进入正式库

```text
流转总览
  -> 观察工作台
  -> Cell 评估
  -> 知识补数（如果命中过正式库）
  -> Cell 维护
```

### 路径三：业务查询最终结果

```text
基站查询
  -> 覆盖分析
  -> 统计报表
```

## 设计实现

以上总纲和 `rebuild5/ui/` 分册已落地为可运行的 Vue 3 + TypeScript 前端工程，位于 `rebuild5/frontend/design/`。

### 技术栈

| 项目 | 选型 |
|------|------|
| 框架 | Vue 3 + TypeScript |
| 构建 | Vite |
| 路由 | vue-router 4 |
| 样式 | CSS 变量设计系统，无第三方 UI 库 |

```bash
cd rebuild5/frontend/design
npm install      # 首次安装依赖
npm run dev      # 启动开发服务器（默认 http://localhost:5173）
npm run build    # 生产构建，输出到 dist/
```

### 对接 API 的方式

当前所有页面使用 `src/mock/data.ts` 中的静态 Mock 数据。对接后端 API 时：

1. 在 `src/composables/` 或 `src/api/` 下创建 API 调用模块
2. 将页面中的 `import { mockXxx } from '../mock/data'` 替换为 API 调用
3. Mock 数据中的类型定义（`src/types/index.ts`）直接复用，无需修改
