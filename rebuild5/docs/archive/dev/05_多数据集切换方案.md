# 多数据集切换方案（规划）

本文件记录 `rebuild5` 未来实现“多数据集在线切换”的开发方案。当前版本尚未支持该能力，系统仍按单活数据集模式运行。

## 目标

支持在同一套系统中同时保留多个数据集的 Step 2-5 结果，并在 UI 中安全切换查看：

- `sample_6lac`
- `beijing_7d`
- `beijing_30d`
- `national_30d`

切换后应满足：

- 页面展示与当前数据集一致
- 一个数据集的 donor / 快照 / 可信库不会串到另一个数据集
- 运行历史按 `dataset_key` 可追溯
- 切换失败不会污染当前已发布版本

## 当前阻塞点

当前代码虽然在结果表中记录了 `dataset_key`，但很多关键逻辑仍使用“全表最新批次”：

- 取最新 `batch_id` 时未按 `dataset_key` 过滤
- Step 4 donor 库默认取 `trusted_cell_library` 的全局最新批次
- 查询页大量使用 `MAX(batch_id)` 读取“最新结果”
- Step 1 表仍是共享表名，切换数据集时会覆盖

因此当前系统只能保证“单个活跃数据集”的一致性。

## 方案总览

### 方案原则

1. `dataset_key` 从展示字段升级为一级隔离键
2. Step 2-5 结果默认按 `dataset_key + batch_id` 查询
3. UI 切换只切“查看上下文”，不隐式修改数据
4. 运行动作与查看动作分离

### 建议路线

分三阶段实施：

#### 阶段 1：结果层隔离

目标：让 Step 2-5 的结果可以按 `dataset_key` 并存。

需要处理的表：

- `rebuild5.trusted_snapshot_cell`
- `rebuild5.trusted_snapshot_bs`
- `rebuild5.trusted_snapshot_lac`
- `rebuild5.trusted_cell_library`
- `rebuild5.trusted_bs_library`
- `rebuild5.trusted_lac_library`
- `rebuild5.enriched_records`
- `rebuild5.gps_anomaly_log`
- `rebuild5.collision_id_list`
- `rebuild5_meta.step2_run_stats`
- `rebuild5_meta.step3_run_stats`
- `rebuild5_meta.step4_run_stats`
- `rebuild5_meta.step5_run_stats`

主要改动：

- 所有“取上一版 / 最新版”的逻辑按 `dataset_key` 过滤
- 主键与索引增加 `dataset_key`
- 统一封装 `get_latest_batch_id(dataset_key, table_name)` 之类的帮助函数

#### 阶段 2：Step 1 可复用层

目标：让不同数据集的原始准备层也能并存，避免换数据集时必须覆盖 Step 1 产物。

候选方案：

- 共享表 + `dataset_key` 字段
- 按 `dataset_key` 分区
- 按数据集分表 / 分 schema

推荐：

- 使用共享表 + `dataset_key` 或按 `dataset_key` 分区

涉及对象：

- `raw_gps`
- `raw_lac`
- `etl_parsed`
- `etl_cleaned`
- 以及中间 `etl_*` 临时表

#### 阶段 3：UI / API 切换能力

目标：支持真正的“切换查看”与“切换并重跑”。

API 建议：

- `GET /api/system/config`：返回当前 dataset、候选列表、模式信息
- `POST /api/system/switch-dataset`：只切查看上下文
- `POST /api/pipeline/run?dataset_key=...`：明确指定运行哪个数据集
- 所有查询接口都支持 `dataset_key`

UI 建议：

- 全局 dataset selector
- 所有页面顶部统一展示 `dataset_key / run_id / snapshot_version`
- 切换查看不触发重跑
- 运行动作单独确认

## 关键代码改动点

### Pipeline

- `rebuild5/backend/app/profile/pipeline.py`
- `rebuild5/backend/app/evaluation/pipeline.py`
- `rebuild5/backend/app/enrichment/pipeline.py`
- `rebuild5/backend/app/maintenance/pipeline.py`

改动方向：

- 去掉全局 `MAX(batch_id)` 假设
- 每条主链显式传入 `dataset_key`
- 不再依赖模块加载时冻结的 `DATASET_KEY`

### Query Layer

- `rebuild5/backend/app/profile/queries.py`
- `rebuild5/backend/app/enrichment/queries.py`
- `rebuild5/backend/app/maintenance/queries.py`
- `rebuild5/backend/app/service_query/queries.py`

改动方向：

- 所有“最新版本”查询必须带 `dataset_key`
- 分页、详情、统计都从当前 dataset 读取

### Frontend

- `rebuild5/frontend/design/src/views/global/DatasetSelect.vue`
- `rebuild5/frontend/design/src/views/global/RunHistory.vue`
- 各步骤页面顶部版本信息组件

改动方向：

- 数据集切换成为真实功能，而不是元数据展示
- 历史页支持按数据集筛选

## 数据迁移建议

建议采用“重跑优先”而不是“历史结果硬迁移”：

1. 先完成代码与表结构隔离
2. 保留元数据与历史运行记录
3. 对需要长期保留的数据集重新跑 Step 2-5

原因：

- 当前旧结果混有全局最新批次假设
- 直接补字段不一定能修复 donor/上一版引用关系
- 重跑更可控，也更容易验证

## 验收标准

实现切换后，至少验证：

1. `sample_6lac` 与 `beijing_7d` 能并存
2. 切换 UI 后，统计页、详情页、搜索页全部切换到目标数据集
3. 一个数据集运行 Step 4 时不会读到另一个数据集的 donor 库
4. 一个数据集发布新批次时不会污染另一个数据集的“当前版本”
5. 历史运行页能按数据集筛选与回看
