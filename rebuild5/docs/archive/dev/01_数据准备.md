# 数据准备

本文件定义 rebuild5 的原始数据来源、样本构建逻辑和 schema 初始化步骤。

## 两套原始数据

| 数据集 | dataset_key | 来源 | 量级 | 优先级 |
|--------|------------|------|------|--------|
| 6 LAC 样本 | `sample_6lac` | 从 legacy 原始表筛选 | ~25 万条 | 先跑 |
| 北京 7 天全量 | `beijing_7d` | legacy 原始表全量 | ~4000 万条 | 等 6 LAC 跑通后 |

## 6 LAC 样本

### 目标 LAC

| LAC | Cell 数 |
|-----|---------|
| `4176` | 640 |
| `6402` | 855 |
| `6411` | 723 |
| `73733` | 457 |
| `405512` | 673 |
| `2097233` | 477 |

共约 3825 个 Cell。

### 数据来源

| 表 | schema | 行数 |
|----|--------|------|
| `网优项目_lac定位北京明细数据_20251201_20251207` | `legacy` | 17,970,911 |
| `网优项目_gps定位北京明细数据_20251201_20251207` | `legacy` | 22,173,675 |

两张表结构完全相同（27 列），区别在最后一列：`lac定位北京来源ss1或daa` vs `gps定位北京来源ss1或daa`。

### 构建逻辑

**不能直接复用 rebuild4 的 `sample_raw_*`**——rebuild5 需要从头跑，避免继承脏数据。

构建步骤：

1. 从 `legacy.网优项目_lac定位北京明细数据_20251201_20251207` 中解析 `cell_infos`，提取所有 `isConnected=1` 的 cell 记录的 `lac` 字段
2. 筛选 `lac IN ('4176','6402','6411','73733','405512','2097233')` 的原始行（整行复制，不做任何变换）
3. 收集这些行中出现的所有 `cell_id` 集合
4. 从 `legacy.网优项目_gps定位北京明细数据_20251201_20251207` 中，提取 `cell_infos` 中包含这些 `cell_id` 的原始行
5. 分别写入 `rebuild5.raw_lac` 和 `rebuild5.raw_gps`

**为什么要这样做**：lac 表中找到的 cell 在 gps 表中可能出现 lac 缺失的情况，如果只按 lac 过滤 gps 表会丢数据。按 cell_id 命中是最完整的策略。

### 产出表

| 表 | schema | 说明 |
|----|--------|------|
| `rebuild5.raw_lac` | rebuild5 | 6 LAC 样本的 lac 定位原始数据 |
| `rebuild5.raw_gps` | rebuild5 | 命中 6 LAC cell_id 的 gps 定位原始数据 |

表结构与 legacy 原始表完全一致（27 列），不做任何变换。

## Schema 初始化

```sql
-- 创建 rebuild5 schema
CREATE SCHEMA IF NOT EXISTS rebuild5;
CREATE SCHEMA IF NOT EXISTS rebuild5_meta;

-- rebuild5: 业务数据表
-- rebuild5_meta: 运行日志、统计表、配置快照
```

## 执行顺序

1. 创建 schema
2. 构建 `rebuild5.raw_lac` 和 `rebuild5.raw_gps`
3. 验证：行数、cell_id 分布、lac 覆盖
4. 然后开始 Step 1 ETL
