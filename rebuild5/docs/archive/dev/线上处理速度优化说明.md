## 概述

rebuild5 的数据处理流程在设计上需要平衡两个目标：**执行性能**和**数据质量可观察性**。不同步骤根据其运行频率和业务需求，做了不同的取舍。

## Step 1 ETL：优先可观察性

Step 1 是一次性数据准备步骤（后续不需要重跑），因此在设计时优先选择了**可观察性**而非执行速度。

### clean 阶段：18 条 ODS 规则逐条执行

当前实现对 45M 行表执行了 **40+ 次全表扫描/更新**：

```python
for rule in ODS_RULES:
    count_row = fetchone(f"SELECT COUNT(*) ... WHERE {rule['where']}")  # 全表扫描
    if violations > 0:
        execute(f"UPDATE ... SET {rule['field']} = NULL WHERE {rule['where']}")  # 全表更新
    rule_stats.append({'violations': violations, 'pass_rate': ...})
```

**这是有意的设计决策**，不是偷懒：
- 每条规则独立上报 `violations` 计数和 `pass_rate`，用于数据质量审计
- 规则以数据驱动（dict 列表）定义，方便增删不改逻辑代码
- `action` 字段区分 `nullify` / `nullify_pair` / `flag_gps` 三种行为
- 带 `id` 和 `name` 字段，规则统计会出现在报告和 UI 中供业务核查

**代价**：18 条规则 × 2 次全表访问（COUNT + UPDATE）= 36 次全表扫描。

### clean 阶段：派生字段分步 UPDATE

时间戳计算分了 5 次 UPDATE，其中 `event_time_std = COALESCE(cell_ts_std, report_ts, gps_ts)` 必须在前三个时间戳 UPDATE 之后执行——这是 **PG 事务语义约束**（UPDATE SET 读的是行的旧值快照，不能在同一条 UPDATE 中引用刚修改的列）。

### fill 阶段：3 CTE + 3 LEFT JOIN

三池补齐模型（stable_pool / ci_pool / ss1_pool）对 45M 行表做了 3 次 GROUP BY + array_agg，然后 3 路 LEFT JOIN 回原表。这保证了字段级补齐的审计清晰度，但性能代价较高。

### Step 1 当前耗时

全量 113 分钟（beijing_7d 数据集，25M 原始 → 45M 解析后）。

## Step 4 / Step 5：优先执行性能

Step 4 和 Step 5 每轮都要跑，且数据量随轮次增长（尤其 Step 5 的连续窗口）。因此做了性能优先的优化：

- **多进程并行 INSERT**：通过 `core/parallel.py` 按 `cell_id % N` 分片，12-28 个进程并行写入
- **CTAS 利用 PG 并行 worker**：聚合类操作（PERCENTILE_CONT）改用 CTAS/INSERT INTO SELECT，让 PG 自动分配 16 个并行 worker
- **碰撞检测短路**：先查 `multi_bs` 结果集为空则直接 return，避免无意义的全表自 JOIN
- **索引策略**：每个 CTAS/INSERT 完成后立即建索引，确保下游 JOIN 不退化为全表扫描

这些优化在 Step 1 中没有做，因为 Step 1 是一次性的，113 分钟的代价可以接受。

## 生产环境优化方向

如果 Step 1 需要在生产环境频繁重跑，建议以下优化（详见 `rebuild5/prompts/17_step1_optimization_analysis.md`）：

| 优化方向 | 预期效果 | 是否影响可观察性 |
|---------|---------|----------------|
| 合并 18 条 ODS 规则为 1 次 UPDATE + 1 次 COUNT FILTER 统计 | 减少 34 次全表扫描 | 不影响——统计改为一次性采集 |
| 合并 5 次派生字段 UPDATE 为 2 次 | 减少 3 次全表扫描 | 不影响 |
| 中间表使用 UNLOGGED TABLE | 写入速度提升 2-5 倍 | 不影响 |
| 跳过 etl_parsed 中间表 | 减少 1 次 45M 行全表复制 | 不影响 |
| fill 三池 CTE 合并为单次扫描 | 减少 2 次 GROUP BY 扫描 | 需评估 |

**核心原则**：可观察性通过 `COUNT(*) FILTER (WHERE ...)` 在一条 SQL 中采集，不需要为每条规则单独做全表扫描。统计结果不变，只是采集方式从"逐条执行"变成"一次性批采"。
