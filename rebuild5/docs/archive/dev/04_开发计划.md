# 开发计划

本文件定义 rebuild5 的分阶段实现计划。每个阶段有明确的产出和验收标准。

## 阶段总览

| 阶段 | 内容 | 前置条件 | 验收方式 |
|------|------|----------|----------|
| P0 | 数据准备 + 基础骨架 | 文档完备 | PG17 MCP 验证数据 |
| P1 | Step 1 ETL | P0 完成 | 行数校验 + 字段覆盖率 |
| P2 | Step 2 路由 + Step 3 评估 | P1 完成 | snapshot 产出 + UI 页面 |
| P3 | Step 4 补数 + Step 5 维护 | P2 完成至少一批 | 正式库发布 + UI 页面 |
| P4 | Day 0 → Day N 全流程验证 | P3 完成 | 多批连续运行 + 收敛验证 |
| P5 | Step 6 服务查询 + 整体打磨 | P4 完成 | 全页面 Playwright 验证 |

## P0: 数据准备 + 基础骨架

### 任务

1. 创建 `rebuild5` / `rebuild5_meta` schema
2. 从 legacy 原始表构建 `rebuild5.raw_lac` 和 `rebuild5.raw_gps`
3. 搭建后端骨架：`main.py`、`database.py`、`envelope.py`、健康检查端点
4. **前端直接复用设计工程**：将 `rebuild5/frontend/design/` 作为前端项目基础，不从零搭建。22 个页面、6 个通用组件、路由、CSS 设计系统已就绪，只需对接真实 API 替换 Mock 数据
5. 搭建启动器骨架
6. 复制 `profile_params.yaml` 到 `rebuild5/config/`

### 验收（全部用 PG17 MCP / Playwright / curl 可重复验证）

- [ ] `SELECT COUNT(*) FROM rebuild5.raw_lac` 在 10 万~20 万之间
- [ ] `SELECT COUNT(*) FROM rebuild5.raw_gps` > 0，且 `SELECT COUNT(DISTINCT cell_id) ...` 命中的 cell_id 覆盖 6 个 LAC
- [ ] `curl http://localhost:{port}/api/system/health` 返回 `{"data":{"status":"ok"},...}`
- [ ] Playwright 访问前端首页，截图确认侧边栏三段式导航、22 个路由可切换、Mock 数据正常渲染
- [ ] 启动器一键拉起后，后端和前端端口均可访问

## P1: Step 1 ETL

### 任务

1. 实现 `etl/parse.py`：cell_infos + ss1 解析，产出 `etl_parsed`
2. 实现 `etl/clean.py`：清洗规则，产出 `etl_cleaned`（含 `event_time_std`）
3. 实现 `etl/fill.py`：同报文补齐，产出 `etl_filled`
4. 实现 `etl/pipeline.py`：编排 + 统计 + 写入 `step1_run_stats`
5. API 端点：`POST /api/etl/run`、`GET /api/etl/stats`、`GET /api/etl/coverage`
6. 前端：对接 `views/etl/` 下的 5 个页面（DataSource / FieldAudit / Parse / Clean / Fill），替换 Mock 数据为真实 API

### 验收

- [ ] `SELECT COUNT(*) FROM rebuild5.etl_parsed` > `SELECT COUNT(*) FROM rebuild5.raw_lac` + `SELECT COUNT(*) FROM rebuild5.raw_gps`（解析会展开 cell_infos）
- [ ] `SELECT COUNT(*) FROM rebuild5.etl_cleaned WHERE event_time_std IS NULL` = 0
- [ ] `SELECT COUNT(*) FROM rebuild5.etl_cleaned WHERE event_time_source NOT IN ('cell_ts','report_ts','gps_ts')` = 0
- [ ] `etl_filled` 的 GPS 补齐率：`SELECT COUNT(*) FILTER (WHERE lon_filled IS NOT NULL)::float / COUNT(*) FROM rebuild5.etl_filled` > 上游 `etl_cleaned` 同指标
- [ ] `SELECT COUNT(*) FROM rebuild5_meta.step1_run_stats` >= 1
- [ ] Playwright 截图：ETL 5 个子页面（数据源注册/字段审计/解析/清洗/补齐）均显示统计数据

## P2: Step 2 路由 + Step 3 评估

### 任务

**Step 2:**
1. 实现路由分流（Day 0 时全走 Path B，无 Path A）
2. 实现 Path B 基础画像计算（分钟级去重、中位数质心、P50/P90）
3. 统计 Path C 计数（直接丢弃）
4. 前端：对接 `views/profile/Routing.vue`，替换 Mock 数据

**Step 3:**
1. 实现 Cell 质量评估（lifecycle_state + 三层资格）
2. 实现 BS / LAC 级联上卷
3. 实现 snapshot 冻结 + diff 计算
4. 前端：对接 `views/evaluation/` 下 6 个页面（FlowOverview / Snapshot / Watchlist / CellEval / BSEval / LACEval），替换 Mock 数据

### 验收

- [ ] `SELECT COUNT(*) FROM rebuild5.path_a_records` = 0（Day 0 无可信库）
- [ ] `SELECT COUNT(*) FROM rebuild5.profile_base` > 0
- [ ] `SELECT COUNT(*) FROM rebuild5_meta.step2_run_stats WHERE path_c_drop_count > 0` >= 1（Path C 有计数）
- [ ] `SELECT COUNT(*) FROM rebuild5.trusted_snapshot WHERE batch_id = 1` > 0
- [ ] `SELECT lifecycle_state, COUNT(*) FROM rebuild5.trusted_snapshot WHERE batch_id = 1 AND entity_type = 'cell' GROUP BY 1` 至少包含 `waiting` 和 `qualified`
- [ ] `SELECT COUNT(*) FROM rebuild5.trusted_snapshot WHERE batch_id = 1 AND entity_type = 'bs' AND lifecycle_state = 'qualified'` > 0（BS 由 Cell 正确上卷）
- [ ] `SELECT DISTINCT diff_type FROM rebuild5.snapshot_diff WHERE batch_id = 1` 只含 `new`
- [ ] Playwright 截图：Cell / BS / LAC 评估页显示状态分布图和列表数据

## P3: Step 4 补数 + Step 5 维护

### 任务

**Step 5（先做，因为 Day 0 没有 Step 4 输入）:**

6. 实现 LAC 维护

**Step 4（Day 1+ 才有输入）:**
1. 实现 donor 匹配 + 知识补数
2. 实现 GPS 异常初筛
3. `enriched_records` → Step 5 消费
4. 前端：对接 `views/governance/KnowledgeFill.vue`，替换 Mock 数据

### 验收

- [ ] `SELECT COUNT(*) FROM rebuild5.trusted_cell_library WHERE lifecycle_state IN ('qualified','excellent')` > 0
- [ ] `SELECT COUNT(*) FROM rebuild5.collision_id_list` >= 0（即使无碰撞也应有表）
- [ ] `SELECT COUNT(*) FROM rebuild5.cell_centroid_detail WHERE cluster_id > 0` >= 0（有多质心 Cell 时 > 0）
- [ ] Day 1 重跑 Step 2 后：`SELECT COUNT(*) FROM rebuild5.path_a_records` > 0（可信库已有内容，Path A 命中）
- [ ] `SELECT COUNT(*) FROM rebuild5.enriched_records` > 0（Step 4 产出）
- [ ] `SELECT COUNT(*) FROM rebuild5.gps_anomaly_log` >= 0（异常表存在）
- [ ] Playwright 截图：Cell / BS / LAC 维护页 + 知识补数页均有数据

## Day 0 → Day N 的数据流

```
Day 0:
  Step 1: raw → etl_parsed → etl_cleaned → etl_filled
  Step 2: etl_filled → 全走 Path B (无可信库) → profile_base
                        Path C 计数丢弃
  Step 3: profile_base → trusted_snapshot (batch_id=1)
  Step 4: 无输入（Path A = 0），跳过
  Step 5: snapshot → trusted_*_library (首版)
                   → collision_id_list

Day 1:
  Step 1: 同上（如有新数据；或复用 Day 0 的 etl_filled）
  Step 2: etl_filled + trusted_cell_library + collision_id_list
          → Path A (命中) → path_a_records
          → Path B (未命中有 GPS) → profile_base
          → Path C 计数丢弃
  Step 3: profile_base + snapshot(batch_id=1) → snapshot(batch_id=2)
  Step 4: path_a_records + trusted_cell_library → enriched_records + gps_anomaly_log
  Step 5: snapshot(batch_id=2) + enriched_records + gps_anomaly_log
          → 更新 trusted_*_library
          → 更新 collision_id_list

Day 2+: 同 Day 1
```

## 注意事项

- Day 0 的 6 LAC 样本是 7 天数据一次灌入，不是逐天灌入
- 如需模拟多批次，可以按天切分 `event_time_std` 来模拟
- 首轮不实现 antitoxin / retention 的正式生效，只输出诊断字段
