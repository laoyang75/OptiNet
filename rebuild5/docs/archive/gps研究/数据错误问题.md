# 数据错误问题：enriched_records 不完整 + trusted_cell_library 历史快照与源数据脱节

> **创建日期**：2026-04-18
> **本文用途**：交给负责跑数据的 agent，根据以下现象 + 已有开发文档（ETL/Enrichment/Step3/Step5 管道）**定位问题根因**
> **发现者**：Cell 画像维护联合排查（用户与 Claude）
> **严重性**：高 — 影响 Step 5 标签判定（大量 cell 被误归 insufficient）与 Step 3 lifecycle 一致性
> **相关文档**：
> - `rebuild5/docs/gps研究/标准质心修复.md`（§6.5 已提到此怀疑，本文是精确确认）
> - `rebuild5/docs/gps研究/08_标签定义总览.md`（标签规则，受本问题影响）

## 1. 现象总摘要（一句话）

**trusted_cell_library 的历史 batch 行（如 batch 2）显示某 cell 曾有足量观测（如 obs=18），但 enriched_records / cell_sliding_window 里**根本找不到那个 batch 该 cell 的原始数据行**。这导致：

- Step 3 晋升的"账面"数据保留在 trusted_cell_library（历史快照）
- 但 Step 5 累加窗口里只能看到之后 batch 的少量数据（1-4 行）
- 运维视角：cell 是 qualified，但 UI 上看到"只有 3 个点"，新标签规则归 insufficient

---

## 2. 案例 A — cell 608244101 的完整数据链路证据

### 2.1 trusted_cell_library 各 batch 行（该 cell 的历史快照）

```
batch 2: lifecycle=qualified, independent_obs=18, distinct_dev_id=4, active_days=2, drift=insufficient
batch 3: lifecycle=qualified, independent_obs= 1, distinct_dev_id=1, active_days=1
batch 4: lifecycle=qualified, independent_obs= 1, distinct_dev_id=1, active_days=1
batch 5: lifecycle=qualified, independent_obs= 3, distinct_dev_id=3, active_days=2
batch 6: lifecycle=qualified, independent_obs= 3, distinct_dev_id=3, active_days=2
batch 7: lifecycle=qualified, independent_obs= 3, distinct_dev_id=3, active_days=2
```

batch 2 时 obs=18 ≥ Step 3 qualified 门槛 10 → 合法晋升。之后 obs 跌到 1-3，lifecycle 按设计不降级。

### 2.2 enriched_records 里该 cell 的原始数据（跨所有 batch）

```
batch 3: 1 行  (gps_fill_source_final='ss1_own',   gps_valid=true)
batch 5: 2 行  (1 行 raw_gps + 1 行 ss1_own)
batch 7: 1 行  (gps_fill_source_final='raw_gps')

batch 2 / 4 / 6: 0 行  ← ★ 关键现象
```

**batch 2 / 4 / 6 该 cell 在 enriched_records 里完全没有数据**。但 trusted_cell_library batch 2 却显示 obs=18。

### 2.3 cell_sliding_window 里该 cell 的累积数据

```
batch 3: 1 行 (2025-12-03 01:30:59)
batch 5: 2 行 (2025-12-05 13:11 + 14:02)
batch 7: 1 行 (2025-12-07 20:23:29)

总计 4 行，与 enriched_records 完全对应
```

### 2.4 技术事实：这些数据不应该被 trim 掉

`cell_sliding_window` trim 规则（`rebuild5/backend/app/maintenance/window.py:91-124`）：

```
WINDOW_RETENTION_DAYS = 14
保留条件：event_time_std >= latest_event_time - 14 days
      OR obs_rank <= 1000   -- 每 cell 最近 1000 条
```

608244101 的 `latest_event_time = 2025-12-07 20:23:29`，14 天前 = 2025-11-23。
**batch 2 数据时间若在 2025-12-02 左右，完全在 14 天窗口内，按规则不应被 trim。**

**结论**：cell_sliding_window 的 trim 机制没问题。问题在于 **batch 2 该 cell 的数据从未被正确写入（或后来被其他流程擦除）**。

## 5. 与"设计原则"的偏离

根据用户（系统设计者）描述的应有行为：

| 设计原则 | 当前实际 |
|---------|---------|
| Step 3 晋升后 cell 进入 Step 5 pool，**不降级** | ✅ lifecycle 符合（但见下一条） |
| 每层数据都应**累加（pool）**，不是快照 | ❌ sliding_window 累加机制没问题，但源 enriched_records 缺失导致池子没水 |
| 升级前走 Path B（数据齐全），升级后 Path A（简化）—— 升级后 pool 应**继续增长** | ❌ 升级后（batch 3-7）的累积量比升级前（batch 2）的 18 行还少，与设计相反 |
| enriched_records 各 batch 应是**独立增量**，每次 ETL 写入自己 batch 的行 | ❓ 是否曾做过破坏性 reset/重建？ |

## 6. 三种可能的根因（请数据 agent 结合开发文档逐一排查）

### 假设 1：某次历史操作曾 TRUNCATE / DELETE 过 enriched_records 部分 batch

**排查方向**：

### 假设 2：ETL → Enrichment 的批次推进过程中，部分批次跳过或失败

**排查方向**：
- 查 `rebuild5/backend/app/enrichment/pipeline.py` 对 batch 1/2/4/6 是否真正执行过完整流程

- 比对 legacy 源表（`legacy.网优项目_lac定位北京明细数据_20251201_20251207`）是否覆盖 batch 1-7 所有日期

### 假设 3：Step 3 晋升时使用的数据来源不是 enriched_records

**排查方向**：
- 查 `rebuild5/backend/app/profile/pipeline.py` 的 Path A / Path B 实现
- 确认 Path B 是否直接从 legacy 或 etl_cleaned 读取（绕过 enriched_records）
- 如果是，那么"trusted_cell_library batch 2 obs=18"是合法的（从 legacy 算出），但 enriched_records 丢失是另一个独立问题

## 10. 数据 agent 排查结论（2026-04-18 修订）

> 本节为基于 `02_基础画像.md`、`03_流式质量评估.md`、`05_画像维护.md`、`runbook_v5.md` 和当前代码的重新评估。

### 10.1 先澄清：Step 3 与 Step 5 用不同事实层，这本身不是 bug

以下现象本身并不能直接证明“数据被删”：

- `trusted_cell_library(batch=t)` 里某 cell 历史上有 `independent_obs=18`
- 但 `enriched_records` / `cell_sliding_window` 里没有同 batch 的 18 条记录

因为设计上本来就存在两条事实链：

1. **Step 3 候选域晋级证据**
   - 来自 **Step 2 的 Path B / `profile_base`**
   - 文档依据：
     - `02_基础画像.md`：Path B -> `profile_base` -> Step 3
     - `03_流式质量评估.md`：Step 3 只评估候选域

2. **Step 5 已发布对象维护证据**
   - 来自 **Step 2 的 Path A -> Step 4 -> `enriched_records` -> `cell_sliding_window`**
   - 文档依据：
     - `02_基础画像.md`：Path A -> `path_a_records` -> Step 4
     - `05_画像维护.md`：Step 5 的事实层输入是 `enriched_records`

所以：

> 不能用 `enriched_records` / `cell_sliding_window` 去反推 Step 3 某个历史 batch 为什么能晋级。

### 10.2 但用户指出的“Step 5 数据变少、拿不到有效数据”问题是真实存在的

问题不在“历史数据被删”，而在**Step 3 -> Step 5 的当前批数据桥缺失**。

设计期望（来自用户原始设计说明）是：

1. **严格数据先进入 Step 3**
   - 只有质量较好的 Path B 记录参与晋级判断

2. **一旦晋级成功，当前批该 cell 的更多数据应进入 Step 5**
   - 因为它已经确认是可信对象
   - 后续可以走更宽松的补数/治理逻辑
   - 这样 Step 5 拿到的维护数据会明显比 Step 3 更多

当前实现却是：

- Step 4 `_insert_enriched_records()` **只写 `path_a_records`**
- 当一个 cell 在 **当前批首次晋级** 时，这一批驱动它晋级的 Path B 证据并不会进入 `enriched_records`
- Step 5 `refresh_sliding_window()` 又只从 `enriched_records` 追加窗口

结果就是：

> **新晋级 cell 在首次发布批次，Step 5 只能看到 snapshot 指标，看不到促成晋级的当前批记录级事实。**

之后如果它下一批命中 Path A 的记录又很少，Step 5 窗口就会长期“点数偏少”。

这正是用户说的：

- “Step 5 数据变少”
- “最后 Step 5 并没有获得有效数据”

### 10.3 当前代码中这个问题如何形成

#### Step 2

- `build_path_a_records()` 只为命中上一轮正式库的记录生成 `path_a_records`
- `build_profile_base()` 只为未命中但有有效 GPS 的记录生成 `profile_base`

#### Step 3

- `trusted_snapshot_t` 里会出现当前批首次晋级的新 cell
- 这些新 cell 在本批是合法发布的

#### Step 4

- `_insert_enriched_records()` 只 `FROM rebuild5.path_a_records`
- 因此：
  - **已在上一轮发布过的 cell**，本批命中 Path A，能写进 `enriched_records`
  - **本批首次晋级的 cell**，本批并不在上一轮正式库里，不会进 `path_a_records`，也就不会进 `enriched_records`

#### Step 5

- `publish_cell_library()` 可以直接用 `trusted_snapshot_t` 发布这些新晋级 cell
- 但 `refresh_sliding_window()` 只吃 `enriched_records`
- 所以新晋级 cell 的当前批记录级事实没有进入维护窗口

### 10.4 这不是“把 Path B 永久混入 Step 5”，而是缺了一次性的当前批 bridge

这个 bug 的本质不是：

- “让 Step 5 永久直接读取 Step 3 的口径”

而是：

- **在当前批首次晋级成功后，缺少一次性的 Step3 -> Step5 数据桥，把该批促成晋级的记录送入 Step 5 维护事实层**

所以正确修法不是重写主流程，而是**补一条一次性的当前批 bridge**。

### 10.5 正确修复方案（建议给独立 agent 执行）

#### 方案总则

1. **保留现有主链边界**
   - Step 2：Path A / Path B 分流不改
   - Step 3：候选域评估不改
   - Step 4：已发布对象的 donor 补数逻辑保留
   - Step 5：维护窗口继续主要消费 `enriched_records`

2. **新增一条 Step3 -> Step5 的当前批 seed bridge**
   - 只为“本批新晋级并发布的 cell”补当前批记录级事实
   - 不回灌 Step 2 / Step 3
   - 只供 Step 5 维护使用

#### 推荐实现

新增一张**批次级持久表**，例如：

`rebuild5.snapshot_seed_records`

字段口径建议与 `enriched_records` 对齐，至少包含：

- `batch_id`
- `source_row_uid`
- `record_id`
- `event_time_std`
- `dev_id`
- `operator_code / lac / bs_id / cell_id / tech_norm`
- `gps_valid`
- `lon_final / lat_final`
- `gps_fill_source_final`
- `rsrp_final / rsrq_final / sinr_final / pressure_final`
- `seed_source`（建议取值：`trusted_snapshot_seed`）

#### 生成规则

在当前批 Step 3 完成后、Step 5 开始前：

1. 读取当前批 `trusted_snapshot_cell(batch_id=t)` 中 `lifecycle_state in ('qualified','excellent')` 的 cell
2. 回到当前批原始作用域（`step2_batch_input` 或当前批对应的 `etl_cleaned` slice）
3. 取这些 cell 在当前批的所有记录
4. 对这些记录应用“已确认晋级后可走更宽松补数逻辑”的填充规则
   - 可以复用 Step 4 的 fill 逻辑思路
   - 但 donor 改成当前批 `trusted_snapshot_cell` / 当前批已确认对象
5. 写入 `snapshot_seed_records(batch_id=t)`

#### Step 5 窗口更新修改

`refresh_sliding_window()` 不再只追加 `enriched_records(batch_id=t)`，而是：

```text
cell_sliding_window
  += enriched_records(batch_id=t)
  += snapshot_seed_records(batch_id=t)
```

并且要避免重复：

- 若某条记录已经在 `enriched_records` 中出现，则 `snapshot_seed_records` 不再重复插入
- 推荐按 `batch_id + source_row_uid` 去重

#### 语义解释

这样修改后：

- Step 3 仍然只用严格证据晋级
- 但一旦本批晋级成功，该 cell 本批所有记录就能以“已确认可信对象”的身份进入 Step 5
- Step 5 从首次发布批次开始就能拿到足够事实层数据

这正符合用户原始设计意图：

> 严格数据用于晋级；晋级后当前批所有数据进入 Step 5，后续按补数/治理逻辑精算。

### 10.6 不建议的错误修法

下面这些都不建议：

1. **把 `profile_base` 直接当成 Step 5 事实层**
   - `profile_base` 是 Cell 级聚合，不是记录级维护事实

2. **把所有 Path B 记录永久写入 `enriched_records`**
   - 会模糊 Step 3 和 Step 4 的边界
   - `enriched_records` 应继续保持“维护事实层”的语义

3. **用 Step 5 窗口点数去否定 Step 3 历史晋级**
   - 这只是口径不同，不是历史晋级直接无效

### 10.7 对当前问题的最终归类

对应本文 §6 的三类假设：

- 假设 1：历史 TRUNCATE / DELETE 造成 `enriched_records` 缺失  
  -> **不是主因**

- 假设 2：某些 batch 的 Step 4 失败导致未写入  
  -> **不是主因**

- 假设 3：Step 3 使用的数据本来就不是 `enriched_records`  
  -> **成立，但这只是背景，不是完整答案**

最终根因应表述为：

> **当前实现缺少“本批首次晋级对象 -> Step 5 维护事实层”的桥接写入。**
>
> 不是历史数据被删，而是新晋级 cell 在首次发布批次没有把当前批记录级事实带进 Step 5，导致 Step 5 维护窗口长期偏瘦，最终表现为“数据变少、有效数据不足”。 

### 10.8 修复后需要重跑的范围

如果按上述 bridge 方案修复，建议至少重跑：

- Step 3
- Step 4
- Step 5
- 标签引擎

原因：

- Step 3 重新生成当前批 `trusted_snapshot_t`
- Step 4 / seed bridge 重新生成本批维护事实层
- Step 5 重新构建 `cell_sliding_window`
- 标签 / 多质心结果都依赖新的窗口事实

若要最稳妥，仍建议按 7 天全链从头重跑。
