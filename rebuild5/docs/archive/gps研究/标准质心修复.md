## 1. 修复前的两个核心 bug（已诊断证据）

### Bug 1：`min_pts=10` 兜底分支误用 Step 3 资格门槛语义

**位置**：`rebuild5/backend/app/maintenance/window.py` 函数 `build_cell_core_gps_stats`（约 line 386-418，构建 `cell_core_points`），原 SQL：

```sql
CASE
    WHEN s.total_pts < s.min_pts THEN TRUE        -- bug：8 < 10 触发，所有点强制 is_core=true
    ELSE d.dist_to_center0_m <= s.keep_threshold_m
END AS is_core
```

`s.min_pts` 来自 `antitoxin_params.yaml` 的 `core_mad_filter.min_pts`，**原值 10**。这个 10 实际是 Step 3 晋升门槛（`profile_params.yaml` 的 `cell.qualified.min_independent_obs: 10`）的同名误植。

**影响范围**：batch 7 共 466,955 cell 进入 `cell_core_mad_stats`，其中 **296,895 cell（63.6%）** 的 `total_pts < 10` 触发兜底，**MAD 完全跳过、所有点强制保留**。

**典型受害 case**：cell 5792399381
- dedup 后 8 点（其中 1 点是 36 km 外的 GPS 飞跃污染）
- 8 < 10 → 跳过 MAD → p90 = **11,090.79m**

### Bug 2：Step 4 GPS 异常检测"半成品" — 检测但不阻断

**位置**：`rebuild5/backend/app/enrichment/pipeline.py` 函数 `_insert_gps_anomaly_log`（约 line 297-339）。

Step 4 把"距离 trusted donor > 2200m"的点写入 `rebuild5.gps_anomaly_log` 表，**但**：
1. `anomaly_type` 字段始终是 `'pending'`（未决策）
2. `enriched_records.gps_valid` 仍然是 `true`（不回写）
3. `enriched_records` 没有 `is_anomaly` / `gps_anomaly_blocked` 字段

→ Step 5 读 `cell_sliding_window` 时按 `gps_valid=true` 取点，**完全不知道 Step 4 已经识别这些点为异常**，污染点直接传入 dedup → MAD 计算。

**典型受害 case**：cell 5792399381
- Step 4 已正确把 batch 5 的两个 36 km 飞跃点写入 `gps_anomaly_log`
- `distance_to_donor_m=35975.75`、`anomaly_threshold_m=2200`
- 但下游 Step 5 完全无视

## 2. 修复方案（已落地）

### 2.1 修复 1 — `min_pts` 改为统计学下限 4

**配置**：`rebuild5/config/antitoxin_params.yaml`
```yaml
core_mad_filter:
  k_mad: 3.0
  k_mad_small: 2.5
  k_mad_medium: 1.5
  k_mad_large: 2.5
  small_upper: 50
  large_lower: 200
  # 统计学下限：PERCENTILE_CONT 算 MAD 至少需要 4 个点才有意义。
  # 注意：这不是 Step 3 的"晋升资格"门槛 —— Step 5 收到的都是已晋升 cell，
  # 不应再用 Step 3 口径反向兜底。低于此值才跳过 MAD（直接全保留）。
  min_pts: 4
```

**代码注释**：`window.py` 在原 `is_core` 计算前加注释（保持 SQL 不变 — 兜底分支保留是因为 < 4 点连 PERCENTILE_CONT 都不稳定）：
```sql
CASE
    -- min_pts 是统计学下限（默认 4），不是 Step 3 资格门槛。
    -- 已晋升 cell 进 Step 5 后总应走 MAD；只有点数少到 PERCENTILE_CONT 算不出有意义 MAD 时才跳过。
    WHEN s.total_pts < s.min_pts THEN TRUE
    ELSE d.dist_to_center0_m <= s.keep_threshold_m
END AS is_core
```

### 2.2 修复 2 — B 方案：Step 4 ↔ Step 5 异常协同（占比阈值）

**设计原则**（用户提出）：
> 异常点占比 < 10% **且** 绝对数 ≤ 2 → 偶发飞跃，dedup 时直接排除
> 异常占比 ≥ 10% → 视为稳定副簇（多质心/碰撞），保留交给 MAD/DBSCAN

**配置**：`antitoxin_params.yaml` 新增段：
```yaml
gps_anomaly_filter:
  # 复用 Step 4 写入 gps_anomaly_log 的判定，在 Step 5 的 cell_core dedup 阶段做闭环。
  # 设计：异常点占比小 = 偶发飞跃 → 直接排除；占比大 = 稳定副簇（多质心/碰撞）→ 保留交给 MAD/DBSCAN。
  max_anomaly_ratio: 0.10        # 异常占比 < 此值且 异常点数 ≤ max_anomaly_count 才排除
  max_anomaly_count: 2           # 绝对数兜底：临界占比下，绝对数太少也按"偶发"处理
```

**Loader**：`rebuild5/backend/app/profile/logic.py`
```python
def load_gps_anomaly_filter_params(params: dict[str, Any] | None = None) -> dict[str, float]:
    payload = params or load_antitoxin_params()
    cfg = payload.get('gps_anomaly_filter', {})
    return {
        'max_anomaly_ratio': float(cfg.get('max_anomaly_ratio', 0.10)),
        'max_anomaly_count': int(cfg.get('max_anomaly_count', 2)),
    }
```

**核心 SQL**：`window.py` `build_cell_core_gps_stats` 的 `cell_core_gps_day_dedup` CTAS 改为三层 CTE：

```sql
CREATE UNLOGGED TABLE rebuild5.cell_core_gps_day_dedup AS
WITH anomaly_uids AS (
    -- 去重 Step 4 的 anomaly 标记，避免后续 LEFT JOIN 产生 fan-out
    SELECT DISTINCT source_row_uid
    FROM rebuild5.gps_anomaly_log
    WHERE distance_to_donor_m > anomaly_threshold_m
),
dedup_with_anomaly AS (
    SELECT DISTINCT ON (
        w.operator_code, w.lac, w.bs_id, w.cell_id, w.tech_norm,
        COALESCE(NULLIF(w.dev_id, ''), w.record_id),
        DATE(w.event_time_std)
    )
        w.operator_code, w.lac, w.bs_id, w.cell_id, w.tech_norm,
        COALESCE(NULLIF(w.dev_id, ''), w.record_id) AS dedup_dev_id,
        DATE(w.event_time_std) AS obs_date,
        w.event_time_std, w.lon_final, w.lat_final,
        (au.source_row_uid IS NOT NULL) AS is_anomaly
    FROM rebuild5.cell_sliding_window w
    LEFT JOIN anomaly_uids au ON au.source_row_uid = w.source_row_uid
    WHERE w.lon_final IS NOT NULL AND w.lat_final IS NOT NULL
      AND w.gps_valid AND w.event_time_std IS NOT NULL
    ORDER BY w.operator_code, w.lac, w.bs_id, w.cell_id, w.tech_norm,
             COALESCE(NULLIF(w.dev_id, ''), w.record_id), DATE(w.event_time_std),
             w.event_time_std, w.record_id
),
filter_decision AS (
    SELECT operator_code, lac, bs_id, cell_id, tech_norm,
        (COUNT(*) FILTER (WHERE is_anomaly) > 0
         AND COUNT(*) FILTER (WHERE is_anomaly) <= {max_anomaly_count}
         AND COUNT(*) FILTER (WHERE is_anomaly)::double precision
             / NULLIF(COUNT(*), 0) < {max_anomaly_ratio}
        ) AS exclude_anomaly
    FROM dedup_with_anomaly
    GROUP BY operator_code, lac, bs_id, cell_id, tech_norm
)
SELECT d.operator_code, d.lac, d.bs_id, d.cell_id, d.tech_norm,
       d.dedup_dev_id, d.obs_date, d.event_time_std,
       d.lon_final, d.lat_final
FROM dedup_with_anomaly d
LEFT JOIN filter_decision f
  ON f.operator_code = d.operator_code AND f.lac = d.lac
 AND f.bs_id IS NOT DISTINCT FROM d.bs_id AND f.cell_id = d.cell_id
 AND f.tech_norm IS NOT DISTINCT FROM d.tech_norm
WHERE NOT (COALESCE(f.exclude_anomaly, FALSE) AND d.is_anomaly)
```

**关键 JOIN 键**：`cell_sliding_window.source_row_uid = gps_anomaly_log.source_row_uid`（已验证两表均含此字段、可精确匹配）。

**双层防御**：
1. anomaly_ratio < 10% AND count ≤ 2 → SQL 直接排除（轻量）
2. 否则 → 全部保留 → MAD（修复后总是生效）继续过滤离群点

- MAD 自适应 k：n<50 用 2.5，50≤n<200 用 1.5，n≥200 用 2.5

- 修复后基础质心算法路径完整链：
  ```
  cell_sliding_window
    → cell_core_gps_day_dedup（B 方案：anomaly_uids + filter_decision）
    → cell_core_initial_center
    → cell_core_point_distance
    → cell_core_mad_stats（min_pts=4）
    → cell_core_points（is_core 标记）
    → cell_core_gps_stats（核心质心）
    → cell_radius_stats（p50/p90）
    → cell_metrics_window
    → trusted_cell_library
  ```

## 8. 设计原则回顾（用户的核心 insight）

1. **Step 5 不应反向兜底 Step 3 的资格门槛** —— 已晋升 cell 在 Step 5 总应走 MAD，不应因当前样本少就跳过
2. **绝对距离硬边界（如 ">5km 直接丢"）破坏多质心 / 碰撞识别** —— 不能用
3. **占比 + 绝对数双阈值是正确的过滤维度** —— 异常占比小（< 10% 且数 ≤ 2）= 偶发飞跃，占比大 = 真副簇（让 MAD/DBSCAN 处理）
4. **Step 4 检测产物必须被 Step 5 消费** —— 否则 anomaly_log 是浪费
5. **配置必须 yaml 化** —— 硬编码阈值未来不易发现/调整

## 13. 设计原则扩展（第二阶段）

在 §8 基础上新增：

6. **质心有效性门槛是规则基石** —— `dev_day_pts ≥ 10` 避免"假多簇"，所有标签区分建立在此之上
7. **补数污染多质心判定** —— `ss1_own` 是设备历史位置衍生，`trusted_cell` 是邻居 cell 传染，都不能进聚类
8. **先单质心再多质心** —— 分层排查减少边界 case
9. **严格 migration 定义（overlap=0 + 跳后稳 + 不回头）** —— 宁漏毋错，避免周期切换误判
10. **dynamic 规则现在定而不等样本** —— 样本 0 命中不是设计错误，是为未来放大数据做准备
11. **uncertain 是合法的未分类状态** —— 留给运维人工标注 + 后续规则迭代
