# 第一阶段修改意见：全局 UI 收口说明

## 目标

第一阶段 UI 只做“真实、可解释、少误导”的轻量收口，不做重度产品化重构。

适用范围：

- Step 1 到 Step 5 已有页面
- 页面标题、字段口径、状态名、支持能力边界
- 当前支持 / 待开发说明

## 全局原则

### 1. 先保证口径真实

- 页面显示的状态名、阈值、统计含义必须和当前代码一致
- 如果后端还没实现，就不要在页面上暗示已经支持

### 2. 先做轻量 UI 对齐

当前阶段只做：

- 页面标题与说明文案修正
- 状态名与术语统一
- 当前支持 / 待开发说明
- 异常样本 / 分页 / 摘要口径收口

当前阶段不做：

- 大规模交互重构
- 复杂图表重绘
- 完整工作台式联动
- 为未来研究结果提前固化 UI 结构

### 3. 研究前不固化高级标签 UI

以下标签在多质心和高级规则研究完成前，只做轻量展示，不做重度解释性 UI：

- `is_multi_centroid`
- `collision`
- `migration`
- `dynamic`
- 其它依赖 Step 5 深度治理结论的高级标签

原因：

- 这些标签边界后续还会随着研究收敛
- 现在先做重度 UI，后面大概率返工

## 当前跨模块全局问题

### G-001 旧术语残留

现象：

- 部分页面仍残留 `active` 旧术语
- 当前系统已统一收口为 `qualified`

处理原则：

- 页面统一改成当前正式术语
- 不再混用旧术语和新术语

### G-002 页面能力强于当前实现

现象：

- 文档或页面曾写“可筛选 / 可钻取 / 可编辑”
- 当前很多页面实际上只支持摘要 + 列表 + 展开

处理原则：

- 当前支持的就保留
- 当前没实现的直接标为“待开发”
- 不做假按钮、不做误导性说明

### G-003 统计口径跨阶段混用

现象：

- 有些页面展示的是汇总口径，但标题写成了单阶段纯口径

处理原则：

- 标题和说明优先服从当前真实数据来源
- 如无法拆开阶段统计，就不要把页面命名成“纯解析 / 纯评估 / 纯维护”的强口径

### G-004 构建阻塞仍来自全局旧问题

现象：

- 当前前端整体 build 失败主要仍来自：
- `config/*.vue` 的 `dataset_mode`
- `etl/Fill.vue` 的类型不一致
- `GovernanceOverview.vue` 的未使用变量

处理原则：

- 这些属于全局工程收尾问题
- 不应阻塞单模块页面口径收口
- 后续可作为独立一轮前端工程修复处理

## 推荐执行顺序

```text
本轮模块修复
    -> 全链路重跑
    -> 轻量 UI 对齐
    -> 多质心 / 高级标签研究
    -> 深度 UI 对齐
```

说明：

- 轻量 UI 对齐在前，保证当前页面真实可用
- 深度 UI 对齐放到研究后，避免重复返工
