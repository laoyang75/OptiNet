## 二、发现的问题

### 问题 1：BS/LAC library 发布策略与 Cell 不一致

**现象**：
- `trusted_cell_library`：只发布 qualified + excellent（270,810 / 876,352 = 31%）
- `trusted_bs_library`：发布全部状态含 observing（118,178 行）
- `trusted_lac_library`：发布全部状态 + dormant（9,633 行，含 4,378 dormant）

**原因**：`publish_cell.py` 第 208 行和第 483 行只 INSERT `lifecycle_state IN ('qualified','excellent')` 的 Cell，这是正确的——只有毕业的 Cell 才发布到正式库。但 BS/LAC 的发布代码（`publish_bs_lac.py`）从 snapshot 全量复制，包含了 observing 和 waiting 状态。

**建议**：检查 `publish_bs_lac.py` 中 BS/LAC 的 INSERT 逻辑，确认是否应该也只发布 qualified + excellent，还是有意发布全状态。

### 问题 2：碰撞检测有两个盲区

**碰撞检测由两层组成**：
1. `collision_id_list`：检测 cell_id 出现在多个 `(operator_code, lac)` 组合中 → 5,806 个 Cell
2. `_detect_absolute_collision`：检测同一 `(operator, tech, lac, cell_id)` 出现在不同 bs_id → **始终为 0**

**盲区 A：同组合地理碰撞**

案例：`cell_id=673563657, op=46001, lac=73775, bs_id=164444`
- 深圳（114.17, 22.31）：25 个 GPS 点
- 北京（116.53, 40.06）：14 个 GPS 点
- 距离 1,980km，p90 = 1,980,431m

这个 Cell 只有一个 `(op, lac)` 组合、一个 bs_id，所以**两层碰撞检测都检测不到**。collision_id_list 不检测它（只有 1 个 combo），_detect_absolute_collision 也不检测它（只有 1 个 bs_id）。

**规模**：p90 > 50km 且不在 collision_id_list 中的 Cell = **547 个**。

**盲区 B：`_detect_absolute_collision` 永远为 0**

`trusted_cell_library` 的主键是 `(operator_code, tech_norm, lac, cell_id)`，每个组合只有一行。所以 `COUNT(DISTINCT bs_id) > 1` 在这个表上**不可能为 true**。该检测完全失效。

**建议**：需要增加基于 GPS 点分布的地理碰撞检测——即使 cell_id 只有一个组合，如果 GPS 点分散在距质心 > N km 的多个簇中，也应标记为碰撞。这正是多质心检测应该做的事情。

### 问题 3：多质心检测流程中断

**现象**：`cell_centroid_detail` 表为空（0 行），`is_multi_centroid` 全部为 false。

**原因**：`publish_cell_centroid_detail()` 函数在创建 `_cell_centroid_valid_clusters` 阶段崩溃了，中间表没有生成。已生成的中间表：
- `_cell_centroid_candidates`：2,162 个候选 Cell ✅
- `_cell_centroid_points`：73,165 个 GPS 点 ✅
- `_cell_centroid_grid_points`：✅
- `_cell_centroid_clustered_grid`：DBSCAN 产出了 7 个簇 ✅
- `_cell_centroid_labelled_points`：✅
- `_cell_centroid_valid_clusters`：❌ 不存在（崩溃点）
- `_cell_centroid_classification`：❌ 不存在

DBSCAN 检测到 183 个 Cell 有多个簇，但后续的验证/分类流程没有执行完。

**候选筛选条件过严**的问题也存在：
- 要求 `active_days >= 2`，但 120 个 p90 > 100km 的 Cell 的 active_days = 1（drift_pattern='insufficient'）被排除
- 要求 drift_pattern 在 `(large_coverage, migration, collision, moderate_drift)` 中，但 `stable`（158 个极端 Cell）和 `insufficient`（120 个）不在列表中
- 334 个 p90 > 100km 的极端 Cell，只有 44 个（13%）被选为候选

### 问题 4：Step 4 补数 batch=2 结果异常

**现象**：batch=2 有 3,369,880 条 Path A 记录，但 donor_matched_count = 0（补数为 0）。

**原因**：batch=1 的 `anchor_eligible_cell_count = 0`。虽然 116,701 个 Cell 发布到正式库，但没有一个满足 anchor_eligible 条件（需要 gps_valid_count >= 10, distinct_dev_id >= 2, p90 < 1500m, span >= 24h）。batch=2 时正式库中仍然没有可用锚点，所以补数为 0。

batch=3 时 anchor_eligible = 95,473，补数才开始工作（696,568 匹配）。

**建议**：anchor_eligible 阈值是否需要和晋级规则一起简化？目前 anchor 需要 4 个条件全满足，但晋级规则已简化为只看 obs。

### 问题 5：p90 极端值分布

trusted_cell_library batch=3（270,810 行）：

| p90 范围 | Cell 数 | 占比 |
|---------|---------|------|
| = 0 | 11,801 | 4.4% |
| 0 ~ 500m | 206,069 | 76.1% |
| 500 ~ 1500m | 28,194 | 10.4% |
| 1.5 ~ 5km | 13,806 | 5.1% |
| 5 ~ 100km | 10,606 | 3.9% |
| > 100km | 334 | 0.1% |
| 中位数 | 151m | — |

**p90 = 0 的 11,801 个 Cell**：这些 Cell 可能只有 1 个 GPS 点（p50 = p90 = 0 因为只有一个点无法计算距离）。

**建议**：

**方案 A — Step 2 源头过滤（推荐）**：在 `build_profile_base` 中增加两阶段质心法，过滤距粗质心 > 3000m 的离群 GPS 点后再算 p50/p90。这样从源头消除脏数据对半径和质心的影响。详见 `fix3/03_GPS质心计算调研建议.md`。

**方案 B — Step 5 标记但保留**：在 Step 5 发布时对 p90 > 5000m 的 Cell 标记 `gps_anomaly_type = 'extreme_spread'`，不修改 p90 值，但在 UI 上显示警告。这样保留了原始计算结果，同时提醒用户。

**方案 C — 混合方案**：Step 2 过滤 > 10km 的极端点（消除明确脏数据），Step 5 对 1.5km ~ 10km 的做多质心检测决定是否是碰撞。

## 三、antitoxin_hit 与 is_collision 矛盾

| 指标 | 值 |
|------|-----|
| antitoxin_hit = true | 40,248 |
| is_collision = true | 0 |
| is_multi_centroid = true | 0 |
| is_dynamic = true | 2,261 |

antitoxin 在工作（40,248 个被阻断），但碰撞和多质心都是 0。防毒化的触发条件不依赖碰撞检测（它看质心偏移/p90 膨胀/设备突增），所以 antitoxin 独立生效。碰撞检测失效不影响 antitoxin，但会导致治理页面碰撞信息缺失。
