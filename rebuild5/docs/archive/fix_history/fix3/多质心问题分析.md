## 我对技术优化方向的判断

下面这部分，是我认为最适合交给开发 agent 的“优先级意见”。

## 第一优先级：不要先继续收窄候选集，而是先拆成“发现入口”和“续算入口”

你文档里的“继续收窄候选集”方向是合理的，但**不能直接作为第一刀**。

原因很简单：
如果候选集只依赖现有的 `gps_anomaly_type / is_dynamic / is_multi_centroid / centroid_pattern / drift_pattern`，那它更像“对已知异常对象续算”，而不是“发现新多簇对象”。
这样做的风险是：**新出现的 multi-centroid cell 可能根本进不了候选集。**

所以更稳的做法不是单纯收窄，而是拆成两类：

### A. 续算候选集

给已经可疑、已经被打过标、或者与上批明显变化的对象使用。
这部分可以很窄，追求速度。

### B. 发现候选集

给还没有被打过多簇标签、但出现便宜异常信号的对象使用。
这部分不能完全依赖现有标签，应该引入更廉价的“疑似多中心”指标，比如：

* 网格热点数
* bbox 对角线长度
* top1 网格占比
* p90 半径变化
* active_days 分散度

也就是说：

**不要把候选收窄理解成“删更多条件”，而要理解成“把发现和续算分流”。**

这是我认为你当前方案里最值得加强的一点。

---

## 第二优先级：你说的“两级聚类”方向是对的，但要改写成“DBSCAN 之前的廉价判疑”

这里有一个关键细节：

你文档里的流程已经不是“直接对原始点做 DBSCAN”，而是先经过 `ST_SnapToGrid`，然后在 `_cell_centroid_grid_points` 上做聚类。

所以“先按 SnapToGrid 做热点，再决定是否 DBSCAN”这件事，**其实已经做了一半**。
开发 agent 不该重复做同一层，而应该继续往前加一层更便宜的过滤：

### 真正建议补的是：

在进入 `ST_ClusterDBSCAN` 之前，先做 cheap suspiciousness gate，例如：

* 网格数 `< N` 直接判单中心
* bbox 对角线 `< X 米` 直接判单中心
* top1 网格占比 `> Y%` 且 top2 极弱，直接判单中心
* 活跃天过少，直接不做多簇
* p90 半径和上批变化极小，直接沿用上批结果

所以这里建议你把文档里的第二优先级，改成：

> 不是“再做一遍网格预处理”，而是在现有网格化之后、DBSCAN 之前，加一层更便宜的可疑性判定。

这个表述会更准确，也更容易让 agent 落地。

## 第三优先级：避免每批回扫 lookback window，这才是最有可能带来量级收益的一刀

这是我最认同、也最建议提到最前面的方向。

当前流程里，候选 cell 会回扫 `cell_sliding_window` 的 lookback batches，再做点到网格、聚类、稳定簇统计。
这意味着每批都在重复做大量 IO 和几何转换。

这类问题，最有效的优化通常不是再调 SQL，而是**改变输入形态**。

### 我建议 agent 优先评估这种结构改法：

建立持久化的、按 batch 维护的网格贡献表，例如：

* 粒度：`cell + batch + snap_grid`
* 指标：`obs_count / dev_count / active_days / maybe x/y`

每批只写入新增 batch 的网格统计；
在 lookback window 内做滚动聚合，而不是每次从原始 GPS 重新扫一遍。

这个方向的价值在于：

1. 减少 `cell_sliding_window` 大表扫描
2. 避免重复 `ST_Transform + ST_SnapToGrid`
3. 为后续聚类和 cluster stats 提供更小、更稳定的输入

这是最有希望把“分钟级瓶颈”往下打的一条。

## 第四优先级：重写 `_cell_centroid_valid_clusters`，而不是只在外围继续抠条件

你文档里已经明确指出，当前最慢的是：

`CREATE UNLOGGED TABLE rebuild5._cell_centroid_valid_clusters AS ...`

这意味着优化重点应该直接落在这段 SQL 的执行结构上，而不是只在候选层面继续压缩。

我建议 agent 重点看四个具体方向：

### 1. 把大 CTE 拆成真正的中间表

当前这段 SQL 里：

* `cell_totals`
* `cluster_obs`
* `cluster_centers`
* `cluster_radius`
* `cluster_stats`
* `filtered`
* `ranked`

虽然逻辑清楚，但对于 planner 和大数据量执行来说，不一定是最优的。
更稳的做法是拆成多个 `UNLOGGED TABLE` 或临时中间表，并在关键阶段：

* 建索引
* `ANALYZE`
* 观察行数

尤其是你前面已经在别的阶段采用“中间表 + 索引 + 末尾清理”的思路了，这里继续保持一致，会比让所有逻辑压在一个大 CTE 里更容易控性能。

### 2. 在 `_cell_centroid_clustered_grid` 里直接带出更多字段

你现在在 cluster stage 里只保留了 key、`snap_geom_3857`、`cluster_id`。
这会导致后面的 `cluster_centers`、`cluster_radius` 还要再回连 `_cell_centroid_grid_points`。

建议直接把下面这些一起带进去：

* `snap_obs_count`
* `snap_x`
* `snap_y`

这样后续：

* 聚簇中心可以直接做加权平均
* 半径可以直接用数值距离算
* 不需要再多一次大 join

这是一个非常具体、且很可能立刻见效的改法。

### 3. 尽量用数值坐标代替重复几何函数

既然已经在 3857 下处理，而且点是网格点，那么很多地方没必要继续大量调用：

* `ST_X`
* `ST_Y`
* `ST_Distance`

可以把 `snap_x / snap_y` 作为数值列直接存下来，后续中心、半径都按数值算。
尤其半径如果只是点到中心的欧式距离，直接 `sqrt(dx*dx + dy*dy)` 往往比几何函数更轻。

### 4. 不要重复从点表计算 `total_obs`

`cell_totals.total_obs` 很可能可以直接复用上游已经存在的 `window_obs_count`。
如果上游口径一致，就没必要为了算 total_obs 再对 `_cell_centroid_points` 做一次 group by。

这是那种“单点不惊艳，但很值得做”的优化。

## 第五优先级：`COUNT(DISTINCT ...)` 相关统计要重点怀疑

从你给的 SQL 看，慢段里有这些统计：

* `COUNT(DISTINCT dev_id)`
* `COUNT(DISTINCT obs_date)`

这类操作在大数据量下很容易拖慢聚合，特别是还带 cluster 分组。

我建议 agent 检查一个方向：

### 先做更细的去重输入，再做 cluster 统计

例如预先压成：

* `cell + obs_date + dev_id + snap_grid`

这样：

* `dev_count`
* `active_days`

就不必在更大颗粒的原始点集上做 `COUNT(DISTINCT ...)`

如果原始数据里同设备同天同格点重复很多，这一刀收益可能很明显。

## 第六优先级：把“同步主链必须产出的东西”和“可以延迟的分类”拆开

你文档里提到“把分类和主质心回写拆开”，这个方向有价值，但我会把它放在**中优先级**，不是最高优先级。

原因是：

* 它对**缩短 end-to-end 发布时间**很有帮助
* 但它**不一定直接解决最慢的 `_valid_clusters` 阶段**

所以更准确的建议是：

### 同步阶段保留

* 主质心
* stable cluster count
* 基础 detail

### 延迟阶段处理

* `moving`
* `migration`
* `dual_cluster`
* `multi_cluster` 的细分类判定

这样做的前提是：
下游业务链路不要求这些标签必须在同一次 Step5 同步完成。

所以这条不是先做不先做的问题，而是要先确认**消费方 SLA**。
