## 3. 物理地址（省市区）显示修复

### 现象

Cell/BS/LAC 维护页和服务层页面需要展示物理地址（省、市、区县），但当前实现有两个问题：
- `trusted_cell_library` 表本身没有省市区列
- 当前 CellMaintain.vue 通过 LEFT JOIN `rebuild4.sample_cell_profile` 按 `cell_id` 关联地址，覆盖率仅 **0.7%**（30万 Cell 只匹配到 2,150 条）

### 现有可用数据

`rebuild4_meta.lac_location_snapshot`（4,552 条，100% 有省市区）是 rebuild4 已有的 LAC→省市区映射表。通过 `(operator_code, lac)` 关联：

| 层级 | 匹配率 | 说明 |
|------|--------|------|
| Cell | **92.6%**（27.8万/30万） | 通过所属 LAC 关联 |
| BS | **88.6%**（13.9万/15.7万） | 通过所属 LAC 关联 |
| LAC | **31.3%**（3,802/12,139） | 直接匹配（rebuild4 数据集范围较小） |

### 修复方案

**后端查询修改**（不改表结构，不改数据库，只改 SQL JOIN）：

把 Cell/BS 查询中的地址 JOIN 从：
```sql
-- 当前（覆盖率 0.7%）
LEFT JOIN rebuild4.sample_cell_profile geo 
    ON t.cell_id::text = geo.cell_id
```

改为：
```sql
-- 修复后（覆盖率 92.6%）
LEFT JOIN rebuild4_meta.lac_location_snapshot loc
    ON t.operator_code = loc.operator_code AND t.lac = loc.lac::bigint
```

LAC 查询直接加同样的 JOIN 即可。

### 涉及文件

| 文件 | 改动 |
|------|------|
| `backend/app/maintenance/queries.py` | `get_maintenance_cells_payload` 的 LEFT JOIN 改成 `lac_location_snapshot` |
| `backend/app/maintenance/queries.py` | `get_maintenance_bs_payload` 加 LEFT JOIN `lac_location_snapshot` |
| `backend/app/maintenance/queries.py` | `get_maintenance_lac_payload` 加 LEFT JOIN `lac_location_snapshot` |
| `backend/app/service_query/queries.py` | Cell/BS/LAC search 和 detail 查询加 LEFT JOIN `lac_location_snapshot` |

### 前端影响

前端 **不需要改动**：
- `CellMaintain.vue` 已有 `province_name / city_name / district_name` 展示逻辑（第 190、202 行）
- `StationQuery.vue` Cell 详情中如果后端返回了这些字段也能自动展示（SELECT *）
- 只要后端查询返回 `province_name / city_name / district_name` 列，UI 就能显示

### 注意事项

- `lac_location_snapshot` 中的 `lac` 是 text 类型，`trusted_cell_library` 中是 bigint，JOIN 时需要 `loc.lac::bigint`
- rebuild5 新增数据集（如非北京区域）可能没有对应的 `lac_location_snapshot` 记录，此时地址显示为 `-`，属正常
- 后续如需完整覆盖，应在 Step 5 发布流程中基于质心坐标做逆地理编码，写入 `trusted_cell_library` 自身的地址列
