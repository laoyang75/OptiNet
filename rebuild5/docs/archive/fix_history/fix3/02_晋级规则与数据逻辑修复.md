# Fix3-02：晋级规则与数据逻辑修复

> 本文档交付给数据处理流程执行。修改完成后需要从 Step 1 重跑全链路。
> 涉及文件：`clean.py`、`pipeline.py`、`logic.py`、`profile_params.yaml`，以及 `candidate_cell_pool` schema。

---

## 一、Step 3 候选池合并逻辑修复（最高优先级）

Step 3 候选池的设计哲学：**新数据进入 → 与池子历史证据合并 → 重新评估 → 合规放行 / 不合规继续等待**。每批新数据加入后，Cell 的质量指标只应变好或保持，不应因单次脏数据永久恶化。

当前实现存在 3 个 Bug，导致大量 Cell 被错误阻断或虚增。

### Bug 1（P0）：p50/p90 半径合并使用 GREATEST — 脏数据永久污染

**位置**：`evaluation/pipeline.py` → `_candidate_eval_input` 构建段（约第 438–447 行）

**当前代码**：
```sql
ELSE GREATEST(c.p50_radius_m, h.p50_radius_m) END AS p50_radius_m,
ELSE GREATEST(c.p90_radius_m, h.p90_radius_m) END AS p90_radius_m,
```

**问题**：`GREATEST` 取两批中最大的 p90。一旦某批数据出现脏 GPS 点（p90 爆炸到 100km），该值永久保存在池子里，后续无论多少批干净数据进来，p90 始终保持在最大值，Cell 永远无法晋级。

**影响规模**：observing 中满足其他 qualified 条件、仅被 p90 阻断的 Cell：**32,781 个被错误卡住**。

**修复**：改为按 `gps_valid_count` 加权平均。观测点越多的批次权重越大，少量脏点被稀释，随历史数据增加自然收敛到真实值。

```sql
-- p50 修复
CASE
    WHEN c.p50_radius_m IS NULL THEN h.p50_radius_m
    WHEN h.p50_radius_m IS NULL THEN c.p50_radius_m
    WHEN COALESCE(c.gps_valid_count, 0) + COALESCE(h.gps_valid_count, 0) = 0
        THEN COALESCE(c.p50_radius_m, h.p50_radius_m)
    ELSE (
        c.p50_radius_m * COALESCE(c.gps_valid_count, 0)
        + h.p50_radius_m * COALESCE(h.gps_valid_count, 0)
    ) / NULLIF(COALESCE(c.gps_valid_count, 0) + COALESCE(h.gps_valid_count, 0), 0)
END AS p50_radius_m,

-- p90 修复（完全同上，将 p50 换成 p90）
CASE
    WHEN c.p90_radius_m IS NULL THEN h.p90_radius_m
    WHEN h.p90_radius_m IS NULL THEN c.p90_radius_m
    WHEN COALESCE(c.gps_valid_count, 0) + COALESCE(h.gps_valid_count, 0) = 0
        THEN COALESCE(c.p90_radius_m, h.p90_radius_m)
    ELSE (
        c.p90_radius_m * COALESCE(c.gps_valid_count, 0)
        + h.p90_radius_m * COALESCE(h.gps_valid_count, 0)
    ) / NULLIF(COALESCE(c.gps_valid_count, 0) + COALESCE(h.gps_valid_count, 0), 0)
END AS p90_radius_m,
```

### Bug 2（P1）：observed_span_hours 是批次内跨度之和，不是真实首尾跨度

**位置**：`pipeline.py` 约第 416 行

**当前代码**：
```sql
COALESCE(c.observed_span_hours, 0) + COALESCE(h.observed_span_hours, 0) AS observed_span_hours,
```

**问题**：每批 `observed_span_hours` = 该批 `MIN(obs_minute)` 到 `MAX(obs_minute)` 的时间跨度。多批累加后得到的是"各批内部跨度之和"，而不是"首次见到到最近见到的真实跨度"。导致跨多天的 Cell 可能一直不满足 `span >= 24h`。

**修复方案（需要 schema 变更）**：

1. 在 `candidate_cell_pool` 中增加两列：
```sql
ALTER TABLE rebuild5.candidate_cell_pool 
  ADD COLUMN first_obs_at TIMESTAMPTZ,
  ADD COLUMN last_obs_at  TIMESTAMPTZ;
```

2. 在 Step 2 `profile_base` 输出中增加 `first_obs_minute` 和 `last_obs_minute`。

3. 在 `_candidate_eval_input` 合并时：
```sql
LEAST(c.first_obs_at, h.first_obs_at) AS first_obs_at,
GREATEST(c.last_obs_at, h.last_obs_at) AS last_obs_at,
EXTRACT(EPOCH FROM (
    GREATEST(c.last_obs_at, h.last_obs_at) - LEAST(c.first_obs_at, h.first_obs_at)
)) / 3600.0 AS observed_span_hours,
```

### Bug 3（P2）：independent_devs 跨批次直接相加会虚增

**位置**：`pipeline.py` 第 410–413 行

**当前代码**：
```sql
LEAST(
    COALESCE(c.independent_obs, 0) + COALESCE(h.independent_obs, 0),
    COALESCE(c.independent_devs, 0) + COALESCE(h.independent_devs, 0)
) AS independent_devs,
```

**问题**：同一设备在多批次出现时被重复计数。

**修复**：当两批都有数据时，取两批设备数的最大值（保守估计）：
```sql
CASE
    WHEN c.cell_id IS NOT NULL AND h.cell_id IS NOT NULL THEN
        GREATEST(c.independent_devs, h.independent_devs)
    WHEN c.cell_id IS NOT NULL THEN c.independent_devs
    ELSE h.independent_devs
END AS independent_devs,
```

## 二、ETL 清洗规则补充

**现象**：`etl_cleaned` 中存在 `operator_code = 0/''/NULL`、`lac = 0/NULL` 的记录未被拦截。

**数据验证**：
```
etl_cleaned 4500万行：
  operator_code 缺失/空: 437万 (9.7%)
  lac = 0 或 NULL: 14.8万 (0.3%)
  bs_id = 0: 3.2万 (0.07%)
  cell_id = 0: 0（已被拦截）
```

**修复**：在 `rebuild5/backend/app/etl/clean.py` 增加规则：

| 字段 | 条件 | 动作 | 说明 |
|------|------|------|------|
| `cell_id` | `= 0` | 删行 | 和 `IS NULL` 同等处理 |
| `lac` | `= 0` | 置空 | 保留记录，但不参与 LAC 聚合 |
| `operator_code` | `= '' 或 = '0'` | 置空 | 保留记录，由补齐阶段尝试补回 |

> `bs_id = 0` 不删行——原始数据中这些 Cell 的 bs_id 确实缺失，但 Cell 本身有效。
> 这些 Cell 在 BS 上卷时自然被排除（bs_id=0 不会匹配任何有效 BS）。

涉及文件：`rebuild5/backend/app/etl/clean.py`

## 三、Cell 晋级规则

### 当前规则（有问题）

```yaml
waiting:   obs < 3  OR  dev < 2
qualified: obs >= 3 AND dev >= 2 AND p90 < 1500m AND span >= 24h
excellent: obs >= 8 AND dev >= 3 AND p90 < 500m  AND span >= 24h
```

问题：条件过于复杂，且 p90/span/devs 受 Bug 1-3 严重影响，大量 Cell 被错误阻断。

### 新规则

```yaml
waiting:   obs < 1（无观测）
observing: 1 <= obs < 10
qualified: obs >= 10
excellent: obs >= 30
```

**CASE 语句**：
```sql
CASE
    WHEN independent_obs >= 30 THEN 'excellent'
    WHEN independent_obs >= 10 THEN 'qualified'
    WHEN independent_obs >= 1  THEN 'observing'
    ELSE 'waiting'
END AS lifecycle_state
```

**取消的条件**：
- `min_distinct_devices` — 取消（受 Bug 3 虚增影响，且独立设备数在短期数据中意义有限）
- `max_p90_radius_m` — 取消（受 Bug 1 脏数据污染影响，p90 阈值暂不作为晋级条件）
- `min_observed_span_hours` — 取消（受 Bug 2 累加错误影响，跨度暂不作为晋级条件）

> 注意：p90、span、devs 这些指标仍然计算和存储，只是暂时不作为晋级门槛。
> Bug 1-3 修复后，后续可以重新引入作为辅助质量判断。

涉及文件：
- `rebuild5/config/profile_params.yaml`
- `rebuild5/backend/app/evaluation/pipeline.py` — Cell CASE 语句（出现多处，搜索 `lifecycle_state` 的 CASE 块）
- `rebuild5/backend/app/profile/logic.py` — 阈值加载适配

## 四、BS 晋级规则

### 当前规则

```
excellent: 未定义（0 结果）
qualified: ≥ 1 优秀 Cell OR ≥ 3 合格+ Cell
observing: 有 GPS 证据的 Cell ≥ 1
waiting:   其他
```

### 新规则

```
excellent: 下属优秀 Cell ≥ 1
qualified: 下属合格 Cell ≥ 2（含优秀）
observing: 有 GPS 证据的 Cell ≥ 1
waiting:   其他
```

**CASE 语句**：
```sql
CASE
    WHEN COUNT(*) FILTER (WHERE c.lifecycle_state = 'excellent') >= 1 THEN 'excellent'
    WHEN COUNT(*) FILTER (WHERE c.lifecycle_state IN ('qualified', 'excellent')) >= 2 THEN 'qualified'
    WHEN COUNT(*) FILTER (WHERE c.gps_valid_count > 0) >= 1 THEN 'observing'
    ELSE 'waiting'
END AS lifecycle_state
```

涉及文件：
- `rebuild5/config/profile_params.yaml` — 新增 `bs.excellent` 阈值段
- `rebuild5/backend/app/evaluation/pipeline.py` — BS CASE 语句（出现多处：初始快照 + 当前快照 + Step 5 维护，搜索所有 BS lifecycle CASE 块）

## 五、LAC 晋级规则

### 当前规则

```
excellent: 未定义（0 结果）
qualified: 合格 BS ≥ 3 OR 合格 BS 占比 ≥ 10%
observing: 非 waiting BS ≥ 1
waiting:   其他
```

### 新规则

```
excellent: 下属优秀 BS ≥ 30
qualified: 下属优秀 BS ≥ 10
observing: 非 waiting BS ≥ 1
waiting:   其他
```

**CASE 语句**：
```sql
CASE
    WHEN COUNT(*) FILTER (WHERE b.lifecycle_state = 'excellent') >= 30 THEN 'excellent'
    WHEN COUNT(*) FILTER (WHERE b.lifecycle_state = 'excellent') >= 10 THEN 'qualified'
    WHEN COUNT(*) FILTER (WHERE b.lifecycle_state <> 'waiting') >= 1 THEN 'observing'
    ELSE 'waiting'
END AS lifecycle_state
```

涉及文件：同 BS 部分。
