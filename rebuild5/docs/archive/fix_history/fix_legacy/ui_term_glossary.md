# UI 术语对齐表（最终版）

> 目标不是把所有词都改得“像小白教程”，而是确保**同一件事全系统只叫一个名字**。  
> 原则：**对齐第一，理解够用，不矫枉过正**。  
> 适用范围：`rebuild5` 的 UI 页面、状态标签、统计卡、表头、帮助文案。

## 1. 使用规则

### 1.1 三层显示规则

1. **标准显示**
   - 用在：页面标题、模块标题、表头、正文说明。
   - 要求：优先中文，直接表达业务含义。

2. **短标签 / Tag**
   - 用在：状态 Tag、图例、统计卡、窄列标签。
   - 要求：可以适度缩短，但必须还能让人一眼看懂。
   - 允许保留少量约定俗成的短写，比如：`合格+`、`A/B/C`，但只能作为**次级标识**，不能单独承担含义。

3. **技术辅注**
   - 用在：tooltip、灰字括号、调试页、接口对照说明。
   - 要求：这里才允许出现 `anchor_eligible`、`baseline_eligible`、`trusted_snapshot_t-1` 之类实现词。

### 1.2 保留英文的边界

建议可以保留在 UI 中的英文非常少，主要只有：

- 行业缩写：`Cell`、`BS`、`LAC`
- 标识字段：`cell_id`、`bs_id`（但主表头仍优先用“小区 ID”“基站 ID”）

以下词不建议直接作为 UI 一级表达：

- `donor`
- `Path A/B/C`
- `active`
- `qualified+`
- `anchor_eligible`
- `baseline_eligible`
- `snapshot`
- `trusted`

### 1.3 两条总原则

- **同一个概念跨页面必须同名**：不要一个页面叫“补齐”，另一个页面叫“补数”，第三个页面又叫“fill”。
- **不要把内部代号直接扔给用户**：像 `donor`、`Path A` 这类词，本身不具备自解释性，必须翻译成有逻辑的中文。

## 2. 流程与路径术语

| 内部/旧写法 | 标准显示 | 短标签 / Tag | 解释 | 使用边界 |
|-------------|----------|--------------|------|----------|
| `donor` | 补数来源小区 | 来源 / 补数来源 | Step 4 中拿来补当前记录缺失字段的历史可信小区 | UI 主文案不再直接出现 `donor` |
| `source cell` | 补数源小区 | 来源命中 | Step 2 Path A 命中后，传给 Step 4 使用的已确认来源小区 | 这是 Step 4 语境词；不要拿来替代 Step 2 的“命中可信库” |
| `Path A` | 命中可信库 | 命中 / A | 命中上一轮正式库，不再进入 Step 3 重评，直接进入 Step 4 | 主文案优先写“命中可信库”；若图中保留字母，只能写成“命中可信库 A” |
| `Path B` | 进入评估 | 评估 / B | 未命中正式库，但具备可用于空间计算的有效 GPS 证据，进入 Step 3 | 不要写“信息饱满”；关键是“有有效 GPS 证据” |
| `Path C` | 信息不足丢弃 | 丢弃 / C | 未命中正式库，且缺少可用于空间计算的有效 GPS 证据 | 不要只写 `Path C` |
| `routing` | 分流 | 分流 | Step 2 决定记录走哪条处理链 | 页面标题、导航统一用“分流”或“基础画像与分流” |
| `routing` 页面名 | 基础画像与分流 | 分流 | Step 2 页面整体职责：命中判断 + 三路分流 + 基础画像统计 | 不再在 UI 中突出“路由” |

### 建议

- `Path A/B/C` 可以保留给流程图、统计图例做辅助标识，但**中文解释必须在前**。
- 对普通页面卡片，直接显示：`命中可信库` / `进入评估` / `信息不足丢弃` 即可。

## 3. 生命周期状态术语

| 内部状态值 | 治理页标准显示 | 服务页标准显示 | 短标签 / Tag | 解释 |
|------------|----------------|----------------|--------------|------|
| `waiting` | 等待 | 证据不足 | 等待 | 证据太少，当前还不能形成稳定判断 |
| `observing` | 观察 | 积累中 | 观察 | 已有一些有效证据，但还没达到放行门槛 |
| `qualified` | 合格 | 达标 | 合格 / 达标 | 已达到可信使用门槛 |
| `excellent` | 优秀 | 高精度 | 优秀 / 高精 | 质量更高，可用于更高精度服务 |
| `dormant` | 休眠 | 休眠 | 休眠 | 暂停更新，但数据仍保留在库中 |
| `retired` | 退出 | 已退出 | 退出 | 已退出正式库，再次出现要重新积累 |
| `qualified+` | 合格及以上 | 达标及以上 | 合格+ / 达标+ | 合并统计 `qualified` 和 `excellent` |
| `active` | 合格 / 窗口内活跃 | 不直接显示 | 活跃 | 旧实现词，必须按语境拆开，不能裸露 `Active` |

### 建议

- **治理页**保留：等待、观察、合格、优秀、休眠、退出。
- **服务页**优先转成：证据不足、积累中、达标、高精度、休眠、已退出。
- `qualified+` 只有在统计口径里才允许出现，且 UI 上统一写“合格+”或“达标+”，不要继续写英文原词。

## 4. 资格术语

| 内部/旧写法 | 标准显示 | 短标签 / Tag | 解释 | 使用边界 |
|-------------|----------|--------------|------|----------|
| `anchor_eligible` | 锚点资格 | 锚点 | 该 Cell 能否作为补数来源，向其他缺字段记录输出位置和信号知识 | 仅治理页、规则页、详单页显示 |
| `baseline_eligible` | 基线资格 | 基线 | 该 Cell 能否参与正式库基线刷新；即使是优秀对象，被防毒化阻断后也会临时失去该资格 | 仅治理页、规则页、详单页显示 |
| `is_registered` | 已注册 | 已注册 | 系统已经承认这个对象存在 | 内部诊断场景为主 |
| `eligible` | 具备资格 | 具备资格 | 通用实现词，必须说清是哪种资格 | 页面不要单独裸露 `eligible` |

### 更直白的人话

- 锚点资格：**能不能作为补数来源**
- 基线资格：**能不能影响正式结果**

### 建议

- 服务页默认不直接显示 `锚点资格`、`基线资格` 两个字段。
- 如果必须展示，也要放在高级信息区，不放在首屏核心信息里。

## 8. 最后的审稿标准

修改或审阅任何 UI 文案时，用下面这句话做最后检查：

> **第一次看到这个词的人，能不能在 2 秒内知道它在说什么。**

如果不能，就按下面顺序处理：

1. 先换成中文标准显示
2. 如果空间不够，再换成短标签
3. 如果还需要保留原实现词，把它放到 tooltip、括号或调试信息里

## 9. 一句话结论

- `donor`、`Path A/B/C`、`active` 这类词，不适合继续作为 UI 主表达。
- `Cell / BS / LAC` 这类行业缩写可以保留，但中文优先。
- Tag 可以适度缩写，但**缩写必须建立在标准中文已经统一**的前提下。
