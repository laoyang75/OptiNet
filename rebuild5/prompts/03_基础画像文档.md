## 核心概念

### 这个步骤的定位

Step 2 是 ETL 之后的第一个业务步骤。它的职责：

1. **与可信库碰撞** — 判断新数据的 Cell 是否已在可信赖库中
2. **分流路由** — 命中的数据直接去 Step 4 补数，未命中的计算指标后去 Step 3 评估
3. **基础指标计算** — 只对未命中可信库的数据计算质心、半径等基础指标
4. **全局 cell_id 碰撞检测** — 发现同一 cell_id 跨运营商/LAC 出现的真碰撞
5. **分流统计** — 统计各路径的数据量和比例

### 数据版本从这里开始

Step 1（ETL）是无版本的通用工具。从 Step 2 开始，所有处理都绑定到当前数据集（如 `sample_6lac`、`beijing_7d`）。数据版本标记贯穿 Step 2-5 的所有产出。

### 可信库碰撞规则

匹配条件：`cell_id` + 至少一个（`operator_code` 或 `lac`）相同。

**为什么不只用 cell_id**：不同运营商可能分配相同的 cell_id（CellID 碰撞）。只用 cell_id 匹配会导致错误命中。

**碰撞 cell_id 的处理**：

Step 2 **不做**全局碰撞检测——全局 cell_id 碰撞检测在 **Step 5（画像维护 5.5）** 中完成，作为进阶标签的一个独立子步骤。Step 5 会扫描可信赖库中所有 cell_id，找出同一 cell_id 出现在不同 (operator_code, lac) 下的情况，标记 `is_collision_id = true`。

Step 2 只在碰撞匹配时做**局部防护**：
- 正常 cell_id（未被 Step 5 标记过碰撞）：cell_id + 至少一个 (operator_code 或 lac) 匹配即可
- 已标记碰撞的 cell_id（`is_collision_id = true`）：必须三字段全匹配（cell_id + operator_code + lac）且 GPS 位置接近（距离 < 2200m，可配置）
- 碰撞 ID + 三字段匹配但 GPS 不接近 → 数据丢弃

### 三条分流路径

| 路径 | 条件 | 去向 | 说明 |
|------|------|------|------|
| Path A | 命中可信库 | Step 4 知识补数 | Cell 已评估过，不需要重算，新数据用于补充和累积 |
| Path B | 未命中 + 有 GPS | Step 3 流式评估 | 新 Cell 或未达标 Cell，有位置数据可以计算指标 |
| Path C | 未命中 + 无 GPS | 丢弃 | 数据不全，可信库未建时无法补数，没有构建价值 |

```
# Step 2: 基础画像 + 分流

## 模块定位
  - 分流路由节点，不只是计算
  - 从这一步开始属于数据版本上下文
  - 读取 trusted_snapshot_t-1（冻结快照原则）

## 2.1 可信库碰撞
  ### 匹配规则
  - 匹配条件：cell_id + 至少一个 (operator_code 或 lac)
  - 匹配目标：trusted_snapshot_t-1 中的可信赖库
  - 命中 = Cell 已在可信赖库中，不需要重新评估
  
  ### 碰撞 cell_id 的局部防护
  - 全局碰撞检测不在 Step 2 做，在 Step 5.5 做
  - Step 2 读取 Step 5 产出的碰撞标记（is_collision_id）
  - 正常 cell_id：两字段匹配即可
  - 碰撞 cell_id：三字段全匹配 + GPS 接近（< 2200m，可配置）
  - 不满足 → 数据丢弃
  - 冷启动时（无 Step 5 碰撞标记）：所有 cell_id 按正常规则处理
  
  ### 碰撞相关统计
  - 碰撞 cell_id 命中数量
  - 碰撞数据中有效匹配 vs 丢弃的比例

## 2.2 分流路由
  ### 三条路径
  - Path A: 命中可信库 → Step 4
  - Path B: 未命中 + 有 GPS → 计算指标 → Step 3  
  - Path C: 未命中 + 无 GPS → 丢弃
  
  ### 分流判定流程图（ASCII）
  
  ### 分流统计
  - Path A 数量和占比
  - Path B 数量和占比（其中数据完整 vs 部分缺失的比例）
  - Path C 数量和占比
  - 碰撞丢弃数量

## 2.3 基础指标计算（仅 Path B）
  ### 独立观测点去重
  - 去重 key: (cell_id, 分钟)
  - 为什么用分钟级而不是小时级（rebuild4 研究结论）
  - 去重后每个观测点的聚合方式
  
  ### 质心计算
  - 算法：PERCENTILE_CONT(0.5)（中位数）
  - 为什么用中位数而不是 AVG（rebuild4 研究结论：中位数偏差 8.6m vs AVG 61.4m）
  - 分别对经度和纬度取中位数
  
  ### 空间半径
  - P50: 观测点到质心距离的中位数
  - P90: 观测点到质心距离的 90 分位
  - 距离计算公式（使用全局约定中的经纬度转米系数）
  
  ### 基础统计字段
  完整列出 profile_base 的字段清单：
  - 维度: operator_code, operator_cn, lac, bs_id, cell_id, tech_norm
  - 观测统计: independent_obs, independent_devs, independent_days, record_count
  - 时间: observed_span_hours, active_days
  - 空间: center_lon, center_lat, p50_radius_m, p90_radius_m
  - 信号: rsrp_avg, rsrq_avg, sinr_avg
  - 数据质量: gps_valid_count, gps_original_count, gps_original_ratio, gps_valid_ratio, signal_original_ratio

  ### 不做（移到 Step 5）
  - 漂移分析
  - 碰撞分类（collision/stable/migration 等标签）
  - 多质心检测
  - 置信度分级
  - Cell 规模分级

## 2.4 分流统计 + UI 页面
  ### 统计指标
  - 当前数据集标记
  - 总输入量（来自 Step 1）
  - Path A/B/C 各自的数量和占比
  - Path B 中数据完整度（GPS覆盖率、信号覆盖率）
  - 碰撞检测结果（碰撞 cell_id 数量、有效/丢弃比例）
  
  ### UI 页面功能
  - 数据版本标记显示（当前数据集）
  - 分流比例可视化（饼图或条形图）
  - 规则配置区：
    · 碰撞匹配的 GPS 接近阈值
    · 观测点去重的时间窗口（分钟）
    · 质心算法选择（中位数/均值/加权）
  - 基础指标统计表（Cell 数量、平均观测数、GPS 覆盖率等）

## 数据表
  - 输入: etl_cleaned（来自 Step 1）
  - 参考: trusted_cell_library（上一轮已发布的可信小区正式库，发布版本对应 trusted_snapshot_t-1）
  - 输出: profile_base（基础指标，仅 Path B）
```
