# 多质心聚类算法调研

## 推荐时机

本调研**不应作为当前轮审计中的即时修复任务**直接展开，而应放在以下前置条件满足后再进行：

1. 完成本轮模块审计（M1-M5）的代码与文档收口
2. 完成一次按当前规则的**全链路重跑**
3. 确认 Step 5 已能稳定产出可用的：
   - `cell_sliding_window`
   - `cell_daily_centroid`
   - `trusted_cell_library`
   - `gps_anomaly_log`
4. 先完成一轮**轻量 UI 对齐**
   - 只做页面名称、口径说明、字段含义、当前支持/待开发说明
   - **不要**在这一轮先固化多质心、碰撞、迁移等高级标签的最终展示方式
5. 再开展本调研，并根据调研结果决定：
   - 多质心算法
   - 多质心阈值
   - `is_multi_centroid / collision / migration / dynamic` 等标签之间的边界
6. 调研完成后，再做一轮**深度 UI 对齐**
   - 把最终确定的标签体系、解释文案、详情展开、图表和筛选器接入页面

## 推荐顺序

```text
本轮重审与修复
    -> 全链路重跑
    -> 轻量 UI 对齐（口径/文案/待开发）
    -> 多质心与高级标签调研
    -> 深度 UI 对齐（基于最终标签体系）
```

说明：
- 如果把“多质心 / 碰撞 / 迁移 / dynamic”相关 UI 先做重度对齐，后面调研结果一旦改标签边界，就会导致前端重复返工。
- 因此当前阶段的 UI 对齐，建议只做到“信息真实、口径一致、能力边界清楚”。

## 背景

Step 5 中对 P90 >= 800m 的 Cell 需要做多质心分析，判断一个 cell_id 是否在物理上对应多个独立位置（多质心）。

当前实现仅用 P90/P50 比值做启发式判断，没有真正的空间聚类。centroid_detail 表的数据是伪造的（质心 ± P50 偏移生成两个假质心，obs_count 硬编码）。

## 前置数据要求

启动本调研前，必须确认以下数据已经由真实链路产出，而不是占位或伪造结果：

- `cell_sliding_window` 中有足够的治理后观测
- `cell_daily_centroid` 中有真实日质心数据
- `trusted_cell_library` 中已有可参考的 `p90_radius_m / active_days / drift_pattern`
- `cell_centroid_detail` 当前若仍是伪造结果，只能作为“待替换占位表”，不能作为研究依据

结论：
- **研究输入必须来自真实窗口观测和真实日质心，不得基于伪造 centroid_detail 做算法结论**

## 需求

1. 对触发范围内的 Cell（P90 >= 800m），取其滑动窗口中的原始 GPS 观测点
2. 做空间聚类，找出候选质心簇
3. 簇间距离 >= 某个阈值（如 200m-500m）才算独立簇
4. 各簇需要有：中心点、观测量、设备数、活跃天数、占比
5. 主簇按观测量/设备数投票产生

## 候选算法

- DBSCAN（密度聚类，自动发现簇数）
- HDBSCAN（层次密度聚类，对簇大小差异更鲁棒）
- K-Means + Gap Statistic（需要预估簇数）
- 自定义距离阈值聚类（简单直接）

## 数据准备

等数据充足后：
1. 找到明显异常的多质心 Cell（如 P90 > 1000m 且 active_days >= 5）
2. 提取这些 Cell 的原始 GPS 观测点
3. 手动验证几个 case 的真实质心分布
4. 基于验证结果选择算法和参数

建议增加：
5. 同时抽取几个 `collision / migration / large_coverage / stable` case 做对照，避免把“多质心”研究成单一异常的代名词
6. 研究结论需要能反向回答：
   - 哪些对象应该标 `is_multi_centroid`
   - 哪些对象仍应归为 `collision`
   - 哪些对象更像 `migration` 或 `dynamic`
