## 核心概念

### 这个步骤的定位

Step 3 的核心目标是**构建和维护持久可信赖的 Cell/BS/LAC 关系库**。

它不只是"打标签"——它是整个系统可信赖库的入口。一个 Cell 从 Step 2 进来时只有基础指标，经过 Step 3 评估后才能进入可信赖库，才能在后续 Step 4 中被用于知识补数。

### 全量评估

每批新数据进来时，Step 3 对**所有已注册的 Cell**做全量重新评估，不只是新进入的。原因：
- 单一时间片段的数据不足以评估，需要累积
- 新数据加入后，已有 Cell 的观测量、设备数、时间跨度都会变化
- 必须基于累积后的完整指标重新判定状态

### 输入只有 Cell

Step 3 直接接收的数据只有 Cell 级（来自 Step 2 的 profile_base）。BS 和 LAC 完全由 Cell 聚合派生，不直接接收原始数据。

### 与 Step 5 的边界

Step 3 只做**状态判定和入库**，不做深度分析：
- 不做漂移分析（stable/collision/migration 分类）
- 不做碰撞确认
- 不做多质心检测
- 不做防毒化

这些都是 Step 5 的维护工作。Step 3 关注的是"这个 Cell 够不够格进入可信赖库"。

### 入库过滤

在评估前，Step 3 可以配置过滤规则，控制哪些数据有资格参与评估：
- **制式过滤**：当前主链路只处理 4G/5G，2G/3G 可配置为不入库
- **地区过滤**：可按 LAC 范围限制评估范围（如只处理北京地区）
- **GPS 要求**：无有效 GPS 的数据不参与质心和半径评估

### 定期清理

Step 3 维护一个清理机制，处理长期无更新的数据：
- 等待（waiting）状态的 Cell 超过 N 天无新数据 → 直接删除（不进入 dormant，因为从未进入可信赖库）
- 已入库的 Cell 超过 M 天无新数据 → 标记 dormant（进入 Step 5 的退出流程）
- 清理阈值可配置

## 文档结构要求

```
# Step 3: 流式质量评估

## 模块定位
  - 构建持久可信赖的 Cell/BS/LAC 关系库
  - 全量评估：每批数据进来后对所有已注册 Cell 重新评估
  - 输入只有 Cell，BS/LAC 完全从 Cell 聚合派生
  - 从 Step 2 接收 Path B 数据（未命中可信库 + 有 GPS + 基础指标）
  - 遵循冻结快照原则：读取 trusted_snapshot_t-1，批末产出 trusted_snapshot_t

## 入库前置过滤（可配置）
  ### 制式过滤
  - 默认只处理 4G/5G
  - 2G/3G 可配置为过滤或放行
  - 过滤后的数据不参与后续评估，记录过滤统计
  
  ### 地区过滤
  - 可按 LAC 白名单或运营商范围限制
  - 用于控制评估范围（如开发阶段只处理 6 个样本 LAC）
  
  ### GPS 有效性
  - 无有效 GPS 的 Cell 不参与质心和半径评估
  - 但仍然注册（is_registered = true），等待后续数据补充

## 3.1 Cell 质量评估
  ### 评估流程
  - 对所有已注册的 Cell 做全量重评估
  - 基于 profile_base 的累积指标（independent_obs, distinct_dev_id, p90_radius_m, observed_span_hours）
  - 判定 lifecycle_state 和三层资格
  
  ### 生命周期判定规则（可配置阈值）
  - waiting: independent_obs < 3 或 distinct_dev_id < 2
  - observing: 不满足 waiting 条件，但不满足 qualified
  - qualified: obs >= 3, devs >= 2, P90 < 1500m, span >= 24h，且无碰撞阻断（阻断信号来自 Step 5.1 的 collision_id_list）
  - excellent: 在 qualified 基础上，obs >= 8, devs >= 3, P90 < 500m
  全部阈值从 profile_params.yaml 读取
  
  ### 三层资格判定
  - is_registered: 首次出现可解析的 (operator_code, lac, cell_id) 即 true
  - anchor_eligible: gps_valid >= 10, devs >= 2, P90 < 1500m, span >= 24h, 且无碰撞阻断
  - baseline_eligible: anchor_eligible + 无防毒化异常 + 成熟条件
  
  ### 入库判定
  - qualified 或 excellent → 写入本批冻结快照，供 Step 5 维护并发布正式库
  - anchor_eligible = true → 可被 Step 4 用于知识补数
  - waiting/observing → 留在评估池，继续等待数据累积

## 3.2 BS 级联评估
  ### 级联规则（可配置阈值）
  - BS 不直接接收原始数据，完全由下属 Cell 聚合
  - waiting: 下属 Cell 均为 waiting
  - observing: 至少 1 个 Cell 有 GPS 数据
  - qualified: 1 个 excellent Cell 或 3 个 qualified Cell（可配置）
  - excellent: BS 的 excellent 阈值当前未独立参数化，预留
  
  ### BS 属性聚合
  - BS 质心 = 下属 Cell 质心的中位数
  - BS P50/P90 = Cell 到 BS 质心距离的百分位
  - BS 分类（collision_bs/dynamic_bs/large_spread/normal_spread）
  - BS 资格 = 下属 Cell 资格的上卷（BOOL_OR）
  
  ### 入库判定
  - qualified+ → 进入 trusted_bs_library

## 3.3 LAC 级联评估
  ### 级联规则（可配置阈值）
  - LAC 完全由下属 BS 聚合
  - waiting: 下属 BS 均为 waiting
  - observing: 至少 1 个非 waiting BS
  - qualified: 3 个 qualified BS 或 qualified BS 占比 >= 10%（可配置）
  - excellent: LAC 的 excellent 阈值当前未独立参数化，预留
  
  ### LAC 属性聚合
  - LAC 质心 = 下属 BS 质心的中位数
  - LAC 面积估算
  - LAC 异常 BS 占比
  
  ### 入库判定
  - qualified+ → 进入 trusted_lac_library

## 3.4 定期清理
  ### 等待项清理（可配置）
  - waiting 状态的 Cell 超过 N 天（默认 30 天）无新数据 → 直接删除
  - 这些 Cell 从未进入可信赖库，不走 dormant/retired 流程
  - 删除只移除评估池记录，不影响原始 etl_cleaned 数据
  
  ### 已入库项超时
  - qualified/excellent Cell 超过 M 天无新数据 → 标记 dormant
  - dormant 处理由 Step 5.4 接管
  
  ### 地区限制
  - 可配置只保留特定地区的 LAC
  - 超出范围的 Cell/BS/LAC 不入库

## 3.5 Snapshot 冻结
  ### 批末统一更新
  - 批次内不改变任何状态
  - 批末将本批评估结果冻结为 trusted_snapshot_t
  - 三层都要有 snapshot：Cell / BS / LAC
  
  ### Snapshot 内容
  Cell snapshot 字段: cell_id, operator_code, lac, bs_id, lifecycle_state, 
    anchor_eligible, baseline_eligible, center_lon, center_lat, 
    p90_radius_m, position_grade, independent_obs, distinct_dev_id
  BS snapshot 字段: bs_id, operator_code, lac, lifecycle_state, anchorable,
    cell_count, active_cell_count, good_cell_count, classification,
    center_lon, center_lat, gps_p90_dist_m, position_grade
  LAC snapshot 字段: operator_code, lac, lifecycle_state, bs_count,
    active_bs_count, cell_count, center_lon, center_lat, area_km2,
    anomaly_bs_ratio, position_grade
  
  ### Diff 计算
  - 与 trusted_snapshot_t-1 对比，记录三层各自的变化
  - Cell diff: 新增/晋升/降级/删除，质心偏移，P90 变化
  - BS diff: 新增/晋升/降级，下属 Cell 数量变化
  - LAC diff: 新增/晋升/降级，下属 BS 数量变化

## 3.6 产出可信赖库
  ### 输出表
  - trusted_cell_library: qualified + excellent 的 Cell（含完整画像字段 + 三层资格）
  - BS 准入结果: qualified + excellent 的 BS（写入本批冻结快照）
  - LAC 准入结果: qualified + excellent 的 LAC（写入本批冻结快照）
  - trusted_snapshot_t: 冻结快照（供下一批 Step 2/3/4 读取）
  - snapshot_diff: 三层各自的变化记录
  
  ### 冻结快照中的 Cell 核心字段（trusted_snapshot_t）
  - 标识: cell_id, operator_code, lac, bs_id, tech_norm
  - 状态: lifecycle_state, is_registered, anchor_eligible, baseline_eligible
  - 碰撞阻断: is_collision_id（来自 Step 5.1，首次运行为 false）
  - 空间: center_lon, center_lat, p50_radius_m, p90_radius_m
  - 信号: rsrp_avg, rsrq_avg, sinr_avg
  - 统计: independent_obs, distinct_dev_id, active_days, observed_span_hours
  - 质量: position_grade, gps_confidence, signal_confidence

## 3.7 处理流程统计
  ### 统计指标
  - 输入量（来自 Step 2 Path B）
  - 过滤量（制式过滤、地区过滤、无 GPS）
  - 评估总量（参与评估的 Cell 数）
  - 各状态分布: waiting / observing / qualified / excellent
  - 晋升数量: 本批新晋升到 qualified / excellent 的 Cell 数
  - BS 状态分布和晋升数
  - LAC 状态分布和晋升数
  - 清理数量: 等待超时删除的 Cell 数
  - 时间趋势: 随批次推进的各状态数量变化（即收敛曲线）

## UI 页面要求
  ### 3 个独立页面

  #### Cell 评估页

  - LAC 级联规则展示

  - 规则配置区支持查看当前参数值（本期只读，后续可改为可编辑）

## 与其他步骤的接口
  - 从 Step 2 接收: profile_base（Path B 数据）
  - 从全局读取: trusted_snapshot_t-1, collision_id_list（来自 Step 5.1）

  - 输出到 Step 5: trusted_snapshot_t + 当前批 Cell/BS/LAC 准入结果
  - 输出到流转页面: snapshot + diff（三层）

## 数据表
  - 输入: profile_base（来自 Step 2 Path B）
  - 参考: trusted_snapshot_t-1, collision_id_list
  - 输出: trusted_cell_library, trusted_bs_library, trusted_lac_library
  - 输出: trusted_snapshot_t（Cell/BS/LAC 三层）
  - 输出: snapshot_diff（Cell/BS/LAC 三层的变化记录）
  - 统计: step3_run_stats
  - 配置: profile_params.yaml 中的 lifecycle, anchorable, bs_lifecycle, lac_lifecycle 段
```
