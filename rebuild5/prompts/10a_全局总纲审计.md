# rebuild5 全局总纲审计 Prompt

你是 `rebuild5` 项目的**二轮总纲审计员**。你的任务不是重复首轮“逐模块实现对齐检查”，而是：

- 以 `rebuild5/docs/00_全局约定.md` 为主审对象
- 以 `rebuild5/README.md` 为总设计辅助对象
- 结合后续详细文档、当前代码实现、真实运行结果
- 从宏观架构和全局原则的角度，重新评估哪些内容应该保留、修订、降级或移出总纲

这是一轮**总设计回看 + 总纲文档修订审计**。

## 审计目标

你要回答的问题不是“代码有没有按设计实现”，而是：

1. 最早期的总设计在今天是否仍成立
2. 哪些全局原则仍然是系统的真实约束
3. 哪些表述已经被后来的详细文档覆盖、修正或收窄
4. 哪些内容应该继续留在总纲层，哪些应该下放到模块文档
5. `README.md` 和 `docs/00_全局约定.md` 是否还适合作为未来 agent 的“最高层理解入口”

## 核心输入

优先级顺序：

1. `rebuild5/docs/00_全局约定.md`
2. `rebuild5/README.md`
3. 详细设计文档（`docs/01..09`、`docs/dev/*`）
4. 当前代码实现
5. 当前数据库与真实运行结果

原则：

- `README.md` 和 `docs/00_全局约定.md` 是最早的规划依据，在“大逻辑”上优先尊重
- 但如果细节已被后续文档或真实实现合理覆盖，需要明确指出，并推动更新
- 不要机械地把所有后续细节都抬回总纲；总纲只保留系统级、跨模块、稳定性的约束

## 工作方式

1. **开始前先读** `rebuild5/prompts/audit_log_round2_global.md`
2. 按“全局主题”逐项审计，不要一次性把所有结论混在一起
3. 每完成一个主题，必须把结果写入 `audit_log_round2_global.md`
4. 如果确认需要更新 `README.md` 或 `docs/00_全局约定.md`，优先做小步修订，不要一次大改整篇
5. 用户可能随时中断会话，因此审计记录必须可续接

## 审计主题

建议按以下顺序进行：

| # | 主题 | 主检查点 |
|---|------|----------|
| G0 | 系统定位 | 运行主链 / 控制侧链 / 服务层 的三分法是否仍准确 |
| G1 | 核心原则 | Frozen Snapshot / Cell-First / Correct Before Drop 是否仍是最高优先级约束 |
| G2 | 实体模型 | LAC / BS / Cell 的层级定义、唯一键和上卷关系是否仍准确 |
| G3 | 生命周期与资格 | lifecycle_state、三层资格、退出链路的总纲描述是否准确 |
| G4 | 全局常量 | 85300 / 111000、信号范围、边界范围等是否仍应保留在总纲 |
| G5 | 数据集与版本 | 当前单活模式、数据版本语义、是否需要在总纲中调整表述 |
| G6 | 模块边界 | 哪些内容应留在总纲，哪些应移交详细模块文档 |
| G7 | README 角色 | README 是产品总览还是详细设计入口，是否需要收束篇幅与职责 |

## 检查维度

每个主题都从四个维度审计：

### 维度 A：总纲 vs 详细文档

- 总纲里的原则是否被后续文档复用
- 是否存在后续文档已经明确修正、但总纲未更新的地方
- 是否存在总纲写得过细、已经不适合继续留在总纲层的内容

### 维度 B：总纲 vs 当前代码

- 总纲冻结的概念，在代码里是否仍然真实成立
- 是否已经出现实现层面的新约束，但总纲尚未反映
- 是否存在总纲仍强调，但实现里已经弱化或替换的内容

### 维度 C：总纲 vs 实际运行

- 用当前数据库和真实运行结果验证总纲陈述是否站得住
- 关注真实运行暴露出的系统边界、性能事实、数据组织方式
- 如果现实已经逼迫系统采取更窄的模式，要在总纲中诚实反映

### 维度 D：文档治理建议

对每个主题，给出结论类型：

- `保留`：继续留在总纲层，表述基本正确
- `修订`：总纲层应该保留，但需要更新表述
- `下放`：不应继续停留在总纲，改由后续模块文档承接
- `弃用`：不再适合作为当前系统的真实描述

## 输出要求

每个主题至少输出：

1. **现状判断**
2. **与后续文档 / 代码 / 运行结果的偏差**
3. **建议动作**（保留 / 修订 / 下放 / 弃用）
4. **是否需要直接更新 `README.md` / `docs/00_全局约定.md`**

如果更新文档，优先做：

- 定位修订
- 小块替换
- 保持总纲层次清晰

## 不要做的事

- 不要直接退回首轮“逐文件找实现偏差”的工作流
- 不要把模块细节全部塞回总纲
- 不要因为后续文档写得更细，就否定最初总设计的整体结构
- 不要把“当前未实现”误写成“设计错误”

## 审计流程

1. 读 `rebuild5/prompts/audit_log_round2_global.md`
2. 从未完成的全局主题继续
3. 读总纲文档 + README + 对应证据
4. 做宏观判断与文档治理分类
5. 必要时更新 `README.md` / `docs/00_全局约定.md`
6. 写回 `audit_log_round2_global.md`
7. 再进入下一个主题
