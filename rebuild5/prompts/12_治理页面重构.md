# Step 5 治理页面重构 Prompt

## 背景

Step 5 画像维护模块经过审计重构后，功能大幅扩展：漂移分类 6 类（基于日质心 max_spread/ratio）、碰撞两层检测、防毒化三维度对比、密度感知退出管理、气压均值重算等。现有 3 个维护页面（Cell/BS/LAC）无法承载全部治理信息，且缺少跨层级全局视角。

需要新增"治理总览"页面，同时改造现有 3 个维护页面。Step 4 知识补数页面（KnowledgeFill.vue）不在此次重构范围内。

## 设计文档

完整维护页面需求见 `rebuild5/docs/05_画像维护.md` → "UI 页面要求" 章节。

## 重构任务

### 1. 新增 GovernanceOverview.vue（治理总览）

定位：**跨层级治理事件看板** —— Step 5 跑完后，治理人员第一个看的页面。回答"这次维护发生了什么重要的事"。

#### 第一区：版本信息 + 运行按钮

- 当前数据集、维护版本（snapshot_version_prev → snapshot_version）、运行时间
- "运行 Step 5" 按钮

#### 第二区：关键变化卡片（4-6 张）

| 卡片 | 数据源 | 说明 |
|------|--------|------|
| 发布 Cell 数 | stats.published_cell_count | 本批发布的 Cell 总量 |
| 碰撞 Cell 数 | stats.collision_cell_count | is_collision=true 的数量 |
| 防毒化阻断 | 新端点 antitoxin-hits count | antitoxin_hit=true 的数量 |
| 退出预警 | 新端点 exit-warnings count | 即将 dormant 的 Cell 数量 |
| 多质心 | stats.multi_centroid_cell_count | is_multi_centroid=true |
| 动态 Cell | stats.dynamic_cell_count | is_dynamic=true |

#### 第三区：碰撞全局表

- 数据源：`GET /api/maintenance/collision`
- 展示：cell_id、combo 数量、dominant_combo、combo 明细（展开 JSON）
- 碰撞是跨 Cell/BS 的全局事件，放在总览页最合适

#### 第四区：防毒化阻断列表

- 新端点：`GET /api/maintenance/antitoxin-hits?page=&page_size=`
- 展示：Cell 标识、触发维度（质心偏移 / P90 膨胀 / 设备突增）、偏差值、上一版值 vs 新值
- 这里不需要全量列表，展示 top N 最严重的即可

#### 第五区：退出预警列表

- 新端点：`GET /api/maintenance/exit-warnings?page=&page_size=`
- 展示：Cell 标识、active_days_30d、consecutive_inactive_days、密度等级、预计状态变化
- 高密度 Cell 优先排序（它们的消失更值得关注）

#### 第六区：漂移分类全局分布

- 复用现有 drift_distribution 数据
- 6 类分布条形图（从 CellMaintain.vue 的漂移条形图移入或复用）

### 6. 后端新增端点

文件：`rebuild5/backend/app/routers/maintenance.py` + `rebuild5/backend/app/maintenance/queries.py`

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/maintenance/antitoxin-hits` | GET | antitoxin_hit=true 的 Cell 列表，含触发维度 |
| `/api/maintenance/exit-warnings` | GET | consecutive_inactive_days 接近阈值的 Cell 列表 |

antitoxin-hits 查询逻辑：
```sql
SELECT operator_code, lac, cell_id, antitoxin_hit,
       -- 需要与上一版对比计算触发维度
       -- 具体实现参考 publish_cell.py 中的防毒化逻辑
FROM rebuild5.trusted_cell_library
WHERE batch_id = (SELECT MAX(batch_id) ...) AND antitoxin_hit = true
```

exit-warnings 查询逻辑：
```sql
SELECT operator_code, lac, cell_id,
       active_days_30d, consecutive_inactive_days,
       CASE WHEN active_days_30d >= 20 THEN 'high'
            WHEN active_days_30d >= 10 THEN 'mid'
            ELSE 'low' END AS density_level
FROM rebuild5.trusted_cell_library
WHERE batch_id = (SELECT MAX(batch_id) ...)
  AND lifecycle_state NOT IN ('dormant', 'retired')
  AND consecutive_inactive_days > 0
ORDER BY
  CASE WHEN active_days_30d >= 20 THEN consecutive_inactive_days / 3.0
       WHEN active_days_30d >= 10 THEN consecutive_inactive_days / 7.0
       ELSE consecutive_inactive_days / 14.0 END DESC
```

## 技术约束

- Vue 3 + TypeScript
- 不引入新 UI 库，使用现有自定义样式和组件（SummaryCard/StatusTag/PageHeader/Pagination/PercentBar）
- 保持现有分页和筛选模式
- 趋势图和分布图用 CSS bar chart 或简单 SVG，不引入图表库
- 后端 queries.py 中使用 `_safe_fetchone` / `_safe_fetchall` 模式处理可能不存在的表

## 设计与验证要求

### 设计一致性

使用 `/frontend-design` skill 进行页面开发，确保与现有页面风格一致。现有设计系统参考：
- CSS 变量：`--c-primary` / `--c-border` / `--c-surface` / `--c-bg` / `--c-text-muted` 等（见 `style.css`）
- 状态颜色：excellent=绿 / qualified=蓝 / observing=黄 / waiting=灰 / dormant=橙 / retired=红
- 通用组件：`SummaryCard` / `StatusTag` / `PageHeader` / `Pagination` / `PercentBar`（见 `components/common/`）
- 布局类：`.card` / `.data-table` / `.grid` / `.grid-3` / `.grid-4` / `.gap-lg` / `.flex`
- 字体：`.font-mono` / `.font-semibold` / `.text-xs` / `.text-muted` / `.text-secondary`

**严格要求**：除 GovernanceOverview.vue（新增页面）外，所有改造在现有 Vue 文件上直接修改。GovernanceOverview.vue 的设计风格必须与现有治理页面（CellMaintain/BSMaintain/LACMaintain）完全一致。

### Playwright 验证

每个页面改造完成后，使用 Playwright MCP 工具进行截图验证：
1. 启动前端开发服务器（如未启动）
2. 导航到对应页面路由
3. 截图确认：布局正确、组件渲染正常、风格与其他页面一致
4. 如发现风格偏差，立即修正
