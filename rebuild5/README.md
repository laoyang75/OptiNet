# rebuild5 — 流式 IP 画像系统

> 一个以 Cell 为最小动态主语、以冻结可信快照为准入依据、以知识补数和画像维护形成闭环的运行主链；同时保留初始化、局部重算、完整回归和对比验证作为控制侧链。

## 系统定位

通过持续处理手机 SDK 上报的基站信号数据，构建和维护一套**可信赖的运营商基站数据库**。

系统分三个层次：
- **运行主链**（Step 1-5）：数据接入、画像构建、质量评估、补数、维护 — 日常稳态运行
- **控制侧链**：初始化、局部重算、完整回归、对比验证 — 非日常操作，保证系统可控
- **服务层**（Step 6）：基于管道产出的可信赖数据库，对外提供运营商查询、分析与报表服务

```
┌──────────────────── 运行主链（日常稳态）───────────────────────┐
│                                                                 │
│  1.数据源   2.基础画像   3.流式评估   4.知识补数   5.画像维护    │
│  接入        计算        质量分级     可信赖库补数  防毒/退出/标签│
│                                                                 │
│  ⚠ 核心约束：当前批次只读取上一版冻结快照，不可自我强化         │
│                                                                 │
└───────────────────────────┬─────────────────────────────────────┘
                            │ 产出：可信赖运营商基站数据库
                            ▼
┌──────────────────── 服务层（商业交付）──────────────────────────┐
│                                                                 │
│  6.服务层：运营商数据库与分析服务                              │
│  基站位置查询 / 信号覆盖分析 / 区域统计 / API 服务              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

┌──────────────────── 控制侧链（非日常操作）─────────────────────┐
│                                                                 │
│  · 冷启动初始化：首次导入数据集时的全量处理                     │
│  · 局部重算：规则变更后对受影响的 Cell/BS 重新评估              │
│  · 完整回归：解析规则或清洗规则变更后的全量重跑                 │
│  · 对比验证：3天起跑 vs 7天起跑 vs 直接7天，验证收敛一致性     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 全局约定

### 实体层级

| 层级 | 英文 | 中文 | 说明 | 唯一键 |
|------|------|------|------|--------|
| L1 | LAC | 位置区 | 运营商管理区域，包含多个基站 | (operator_code, lac) |
| L2 | BS | 基站 | 物理站点，包含多个小区 | (operator_code, lac, bs_id) |
| L3 | Cell | 小区 | 最小分析单元，一个天线扇区 | (operator_code, lac, cell_id) |

### 运营商编码

| operator_code | 中文 | 制式 |
|---------------|------|------|
| 46000 | 中国移动 | 4G/5G |
| 46001 | 中国联通 | 4G/5G |
| 46011 | 中国电信 | 4G/5G |

### 生命周期状态

流转方向：`waiting → observing → qualified → excellent`，可退出到 `dormant → retired`。

| 状态 | 中文 | 含义 | 适用层级 |
|------|------|------|---------|
| waiting | 等待 | 数据不足，尚无法评估 | Cell/BS/LAC |
| observing | 观察 | 有一定数据，正在积累 | Cell/BS/LAC |
| qualified | 合格 | 达到使用门槛，可用于补数 | Cell/BS/LAC |
| excellent | 优秀 | 高质量，可用于高精度服务 | Cell/BS/LAC |
| dormant | 休眠 | 长时间无新数据 | Cell/BS/LAC |
| retired | 退出 | 确认退出，移出可信赖库 | Cell/BS/LAC |

### 三层资格（独立于生命周期状态）

生命周期状态回答"这个对象处于什么阶段"，三层资格回答"这个对象能被用于什么"。两者独立维护，不混为一个字段。

| 资格 | 中文 | 判定条件 | 用途 |
|------|------|---------|------|
| `is_registered` | 已注册 | 首次出现可解析的 (operator, lac, cell_id) | 系统承认此对象存在 |
| `anchor_eligible` | 锚点资格 | GPS≥10, 设备≥2, P90<1500m, 跨度≥24h, 无碰撞 | 可用于 Step 4 知识补数 |
| `baseline_eligible` | 基线资格 | anchor_eligible=true 且 lifecycle_state='excellent' | 可参与 Step 5 画像刷新 |

**关键区分**：一个 Cell 可以是 `lifecycle_state = qualified` 但 `anchor_eligible = false`（比如观测量够但 P90 太大）。也可以是 `anchor_eligible = true` 但 `baseline_eligible = false`（锚点可用但还不够成熟到影响基线）。

### 冻结快照原则

**这是整个系统最重要的约束**。

```
当前批次的状态判定与知识补数，只能参考上一轮已冻结 / 已发布版本，
不能边判边刷本批结果。

具体规则：
· Step 3 质量评估：读取 trusted_snapshot_t-1
· Step 4 知识补数：读取上一轮已发布的 trusted_cell_library（其版本对应 trusted_snapshot_t-1）
· 本批次新晋升的对象，只能在下一批开始参与补数和准入判断
· 批次内不改状态，批末统一更新
```

**为什么**：如果同一批次刚晋升的 Cell 立刻被拿来补同批次的新报文，会形成"本批证据判本批自己"的自我强化，导致数据质量不可控。

## 数据集选择与运行管理

### 数据集选择

系统维护多个数据集的元数据，但当前实现仍是“单活数据集模式”：实际运行只认 `config/dataset.yaml` 中指定的当前数据集，页面暂不支持在线切换。

| 数据集 | 规模 | 用途 | 状态 |
|--------|------|------|------|
| `sample_6lac` | 6 个 LAC，~42 万行，7 天 | 开发调试，快速验证 | 已完成 |
| `beijing_7d` | 北京全区，7 天 | 跑通全流程 | 待接入 |
| `beijing_30d` | 北京全区，30 天 | 稳定性验证 | 待接入 |
| `national_30d` | 全国，30 天 | 生产级验证 | 待接入 |

**当前实现（单活模式）**：
```
1. 修改 `config/dataset.yaml`
2. 清理共享 Step 0-5 结果表
3. 重新执行全流程
4. 页面只展示当前单活数据集的结果；其他数据集仅保留元数据与历史记录
```

**未来规划**：多数据集切换能力尚未开发，设计方案见 `rebuild5/docs/dev/05_多数据集切换方案.md`。

**数据集元数据**：每个数据集记录来源、导入时间、记录数、LAC 范围、时间范围，作为后续所有处理的上下文。

### 数据版本

数据版本 = 数据集标记。标识“当前系统处理的是哪批数据”，贯穿 Step 2 到 Step 5 全流程。当前实现只保证单个活跃数据集的结果一致性。

- Step 1（ETL）不受数据版本约束 — 它是通用工具，统计跟随数据源
- Step 2 开始，所有处理和产出都绑定到当前数据集

```
数据集: beijing_7d (示例当前)
  → Step 2-5 的共享结果表都属于当前单活数据集
  → 若改跑别的数据集，需先清理共享产物后重跑
```

同一个数据集可以用不同参数运行多次（调整阈值后重跑），每次产出独立的运行记录。多数据集并存切换为后续开发项。

## 管道层：五步处理

### Step 1: 数据源接入

**目标**：将多种格式的原始数据转换为统一的结构化格式。

```
原始报文（多种数据源）
    │
    ├── 1.1 数据源注册
    │   配置数据源类型、字段映射、导入规则
    │
    ├── 1.2 解析（炸开）
    │   一条原始报文 → 多条结构化记录
    │   · cell_infos JSON → 每个基站一行
    │   · ss1 文本 → 每个信号组一行
    │
    ├── 1.3 基础清洗
    │   去除无效值，但不做补数
    │   · CellID = 0 或溢出 → 删除行
    │   · 信号值越界 → 置空
    │   · GPS 越出国界 → 标记无效
    │   · 运营商编码异常 → 置空
    │
    └── 1.4 字段对齐（同报文内互补）
        同一条原始报文内的不同数据源字段互补
        · cell_infos 有 GPS → 补给同报文的 ss1 行
        · ss1 有信号 → 补给同报文的 cell_infos 行
        注意：这是结构性对齐，不是知识补数
```

**输入**：原始报文表
**输出**：`etl_cleaned`（结构化、清洗后）
**复用**：rebuild4 的 pipeline.py step1_parse + step2_clean + step3_fill 的对齐部分

---

### Step 2: 基础画像 + 分流

**目标**：对 Step 1 产出的数据做命中判断、基础指标计算和分流路由。从这一步开始属于数据版本上下文。

```
etl_cleaned (来自 Step 1)
    │
    ├── 2.1 正式库命中判断
    │   用 cell_id + 至少一个（operator_code 或 lac）匹配
    │   上一轮已发布的可信小区正式库（trusted_cell_library）中存在 → 命中
    │   碰撞 cell_id 防护（读取 Step 5.1 产出的 collision_id_list）：
    │     已标记碰撞的 cell_id → 三字段全匹配 + GPS 接近
    │     不满足 → 数据丢弃
    │   ⚠ 全局碰撞检测在 Step 5.1 完成，不在此步
    │
    ├── 2.2 分流路由
    │   ┌────────────────────────────────────────────┐
    │   │ 命中上一轮正式库 → Path A: 直接流入 Step 4 │
    │   │                    不需要重算，只做数据补充  │
    │   │                                            │
    │   │ 未命中 + 有 GPS → Path B: 计算基础指标     │
    │   │                   → 流入 Step 3 评估       │
    │   │                                            │
    │   │ 未命中 + 无 GPS → Path C: 丢弃             │
    │   │                  数据不全，无法构建可信对象 │
    │   └────────────────────────────────────────────┘
    │
    ├── 2.3 基础指标计算（仅 Path B）
    │   · 独立观测点去重 (cell_id, 分钟)
    │   · 中位数质心 PERCENTILE_CONT(0.5)
    │   · P50/P90 半径
    │   · 基础统计（观测数、设备数、信号均值等）
    │
    └── 2.4 分流统计
        · 总数据量
        · 命中上一轮正式库比例 → Step 4
        · 未命中有 GPS → Step 3
        · 未命中无 GPS → 丢弃
        · 数据完整度分布
```

**不做**：漂移分析、多质心、分类标签、全局碰撞检测——这些移到 Step 5。

**输入**：`etl_cleaned` + 上一轮已发布的 `trusted_cell_library` + `collision_id_list`（二者版本对应 `trusted_snapshot_t-1`）
**输出**：
- Path A 数据 → Step 4（命中上一轮正式库的记录）
- Path B 数据 + `profile_base` → Step 3（未命中但有 GPS 的，含基础指标）
- Path C 数据 → 丢弃（未命中且无 GPS，数据不全无法构建可信对象）

---

### Step 3: 流式质量评估

**目标**：基于基础指标，对 Cell/BS/LAC 逐级评估质量等级。每批新数据进来后增量评估，产出本批冻结快照和准入结果，供 Step 5 维护并发布正式库。

```
profile_base
    │
    ├── 3.1 Cell 分级（可配置阈值）
    │   ┌──────────┬────────────────────────────────┐
    │   │ waiting  │ 观测 < 3 或 设备 < 2           │
    │   │ observing│ 基本条件满足但不够              │
    │   │ qualified│ 观测≥3, 设备≥2, P90<1500m,     │
    │   │          │ 跨度≥24h, 非碰撞                │
    │   │ excellent│ 观测≥8, 设备≥3, P90<500m       │
    │   └──────────┴────────────────────────────────┘
    │
    ├── 3.2 BS 级联晋升（可配置）
    │   · 1 个 excellent Cell → BS 可晋升
    │   · 或 3 个 qualified Cell → BS 可晋升
    │   · BS 状态 = 下属 Cell 最优状态的聚合
    │
    ├── 3.3 LAC 级联晋升（可配置）
    │   · N 个 active BS 或 active 比例 ≥ X%
    │   · LAC 状态 = 下属 BS 的聚合
    │
    ├── 3.4 三层资格判定
    │   每个 Cell 独立维护三个资格字段：
    │   · is_registered: 首次出现即注册
    │   · anchor_eligible: GPS≥10, 设备≥2, P90<1500m, 跨度≥24h, 非碰撞
    │   · baseline_eligible: anchor_eligible=true 且 lifecycle_state='excellent'
    │   资格与 lifecycle_state 独立，不混用
    │
    ├── 3.5 流转 Snapshot 记录
    │   每批评估后保存三层状态快照
    │   · Cell/BS/LAC 各自的 lifecycle_state + 三层资格
    │   · 与上一批的 diff（谁晋升了、谁新增了、资格变化）
    │   · 用于流转可视化页面展示
    │
    └── 3.6 冻结快照输出
        本批评估完成后，将结果冻结为 trusted_snapshot_t
        Step 5 读取本批冻结快照做维护发布
        Step 2 和 Step 4 只能在下一轮读取 Step 5 发布后的正式库
```

**输入**：`profile_base` + `trusted_snapshot_t-1`（上一版冻结快照）+ `collision_id_list`（来自 Step 5.1）
**输出**：
- `trusted_snapshot_t`（第 t 批冻结快照，供 Step 5 维护发布）
- 流转 snapshot + diff（三层）
- 当前批 Cell / BS / LAC 准入结果（供 Step 5 维护并发布正式库）

---

### Step 4: 知识补数

**目标**：新报文进来时，用上一轮已发布的可信小区正式库中已验证的知识补充缺失字段。这是系统的自改进能力。第一轮初始化时，因为还没有正式库，所以 Step 4 不生效。

```
新报文 + trusted_cell_library
    │
    ├── 4.1 Cell 匹配查询
    │   新报文的 cell_id 是否命中上一轮已发布正式库？
    │   · 在 → 进入补数流程
    │   · 不在 → 保留原始数据，走 Step 2 积累
    │
    ├── 4.2 GPS 补数
    │   报文无 GPS → 用 Cell 的已知质心位置补充
    │   补数置信度 = Cell 的 position_grade
    │   标记 gps_source = 'trusted_cell'
    │
    ├── 4.3 信号参考补数
    │   报文无信号 → 用 Cell 的信号均值作为参考
    │   标记 signal_source = 'trusted_cell'
    │
    └── 4.4 运营商信息补数
        报文运营商字段缺失 → 用 Cell 的已知运营商补充
```

**与 Step 1 补数的区别**：

| | Step 1 字段对齐 | Step 4 知识补数 |
|-|----------------|----------------|
| 数据来源 | 同一条报文内的其他字段 | 上一轮已发布的可信小区正式库 |
| 可靠性 | 中（依赖原始报文质量） | 高（经过质量评估和维护的正式对象） |
| 补数范围 | 仅同 record_id 的行 | 任何命中正式库的 Cell |
| 触发时机 | 数据导入时 | 正式库建立后的持续运行 |

**关键约束**：
- 补数使用的是上一轮已发布的 `trusted_cell_library`，其版本对应 `trusted_snapshot_t-1`
- 补数后的报文进入**治理后事实候选层**，不直接回灌本批 Step 2 的基础画像计算
- 补数后的记录是否能参与画像刷新，需要 `baseline_eligible` 判定

**输入**：Path A 记录 + 上一轮已发布的 `trusted_cell_library`
**输出**：补数后的记录正常入库，Step 5 下次重算 Cell 画像时作为该 Cell 的有效观测参与计算

## 控制侧链

运行主链（Step 1-5）是日常稳态流程。控制侧链是非日常操作，保证系统在规则变更、数据扩展、质量回归等场景下仍然可控。

### 冷启动初始化

首次导入新数据集时的全量处理。与日常运行的区别：
- 没有 `trusted_snapshot_t-1` 可以参考
- 没有上一轮已发布的 `trusted_cell_library`，因此 Step 2 不会出现 Path A 命中
- Step 4 知识补数在初始化期间不生效（无正式库可读）
- Step 3 先产出第一版冻结快照，Step 5 再发布第一版正式库和 `collision_id_list`

### 局部重算

场景：某个晋级阈值调整后，只需要重新评估受影响的 Cell/BS/LAC。

- 不需要重跑 Step 1（数据已入库）
- 从 Step 2 或 Step 3 开始，只对参数影响范围内的实体重算
- 产出新的 snapshot，与旧 snapshot 对比验证

### 完整回归

场景：解析规则变更（Step 1.2）或清洗规则变更（Step 1.3），需要从头重跑。

- 从 Step 1 开始，用新规则重新处理全量数据
- 产出完整的新 snapshot，与旧 snapshot 做全量对比
- 确认无回归后，切换到新版本

**为什么不能省略**：如果只做日常维护而不支持完整回归，规则变更后就只能“从变更点往后跑”，之前的数据不会被重新处理，口径不统一。

### 对比验证

场景：验证系统收敛一致性。

- 用同一份数据，分别做“3天起跑滚到7天”和“直接7天全量”，对比最终结果是否一致
- rebuild4 的流式评估实验已验证过一次（Day 7 偏差 0.00m），rebuild5 每次大规模数据扩展后应重新验证
- 对比指标：Cell 覆盖率、质心偏差、生命周期一致率

## 服务层

### Step 6: 服务层 - 运营商数据库与分析服务

**目标**：管道层产出的可信赖数据库是内部引擎。服务层是面向外部使用者的查询和分析界面。

```
可信赖运营商基站数据库
    │
    ├── 6.1 基站位置查询
    │   给定 cell_id → 返回位置（经纬度）、精度（P90）、可信度
    │   给定坐标 → 返回附近基站列表
    │
    ├── 6.2 信号覆盖分析
    │   区域内基站密度、信号质量分布
    │   运营商覆盖对比
    │
    ├── 6.3 区域统计报表
    │   按省/市/区统计基站数量、质量分布
    │   趋势分析（月度变化）
    │
    └── 6.4 API 服务
        对外提供标准 API
        · 基站位置查询 API
        · 批量匹配 API
        · 覆盖评估 API
```

**与管道层的关系**：
- 管道层（Step 1-5）= 数据工厂，生产和维护数据
- 服务层（Step 6）= 门店，对外提供数据产品
- 前端 UI 中，管道层的页面是"系统管理员视角"，服务层的页面是"业务用户视角"
