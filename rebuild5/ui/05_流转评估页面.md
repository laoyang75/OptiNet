# 流转评估页面

本文件定义 Step 3 的页面组：流转总览、流转快照、观察工作台、Cell 评估、BS 评估、LAC 评估。

## 页面组定位

| 项目 | 说明 |
|------|------|
| 页面组角色 | 展示对象如何在 waiting / observing / qualified / excellent 中流动 |
| 主要用户 | 治理分析师 |
| 核心问题 | 为什么晋升、为什么停留、为什么某层状态改变 |
| 页面数量 | 6 页 |

## 页面一：流转总览

| 项目 | 说明 |
|------|------|
| 用户目的 | 快速理解三层对象整体收敛情况 |
| 核心模块 | 三层状态分布、晋升数量、锚点数量、趋势曲线、差异摘要 |
| 关键功能 | 选择当前批次、切换层级、从总览跳到对应评估页 |
| 设计重点 | 这是“管理驾驶舱”，先总览，不展开单对象 |

## 页面二：流转快照

| 项目 | 说明 |
|------|------|
| 用户目的 | 查看某一版 `trusted_snapshot_t` 的完整状态 |
| 核心模块 | 当前快照信息、上一版对比、diff 分类、新增 / 晋升 / 降级对象列表 |
| 关键功能 | 切换快照版本、查看新旧差异、跳到单对象明细 |
| 设计重点 | 重点突出“版本差异”，不是重复总览页 |

## 页面三：观察工作台

| 项目 | 说明 |
|------|------|
| 用户目的 | 找出仍在 `waiting / observing` 的对象，并看离晋升还差什么 |
| 核心模块 | 观察对象列表、差距说明、阻断原因、推荐下一步 |
| 关键功能 | 按状态、LAC、运营商筛选，按“差多少”排序 |
| 设计重点 | 页面语言要偏“诊断”，不是只列状态 |

## 页面四：Cell 评估

| 项目 | 说明 |
|------|------|
| 用户目的 | 理解单个 Cell 的状态、资格和晋升差距 |
| 核心模块 | 状态分布、Cell 列表、晋升进度、近期变化、规则展示、清理设置 |
| 关键功能 | 按 `anchor_eligible`、`baseline_eligible`、状态筛选，查看单个 Cell 详情 |
| 设计重点 | 要能明确告诉用户“还差观测量 / 设备数 / 跨度 / P90 / 碰撞阻断” |

## 页面五：BS 评估

| 项目 | 说明 |
|------|------|
| 用户目的 | 理解 BS 为什么能或不能晋升 |
| 核心模块 | 状态分布、BS 列表、下属 Cell 构成、晋升进度、聚合画像 |
| 关键功能 | 展开看下属 Cell、查看 qualified+ Cell 数 |
| 设计重点 | 强调“BS 完全由下属 Cell 上卷而来” |

## 页面六：LAC 评估

| 项目 | 说明 |
|------|------|
| 用户目的 | 理解区域层整体是否稳定 |
| 核心模块 | 状态分布、区域质量概览、异常 BS 占比、晋升进度、近期变化 |
| 关键功能 | 展开看下属 BS、查看区域整体质量指标 |
| 设计重点 | 把 LAC 做成区域质量看板，而不是单对象详情页 |

## 页面组之间的关系

```text
流转总览
  -> 流转快照
  -> 观察工作台
  -> Cell 评估
  -> BS 评估
  -> LAC 评估
```

## 页面成功标准

| 标准 | 表现 |
|------|------|
| 先总后分 | 用户先看整体，再钻到对象 |
| 晋升可解释 | waiting / observing 为什么没晋升必须说清楚 |
| 三层有边界 | Cell / BS / LAC 页面各自关注不同粒度，不相互混淆 |
