# 基础画像与分流页面

本文件定义 Step 2 的唯一页面：基础画像与分流。

## 页面定位

| 项目 | 说明 |
|------|------|
| 页面角色 | 路由总览页，展示数据为何进入 Path A / B / C，以及 Path B 的基础画像质量 |
| 主要用户 | 治理分析师 |
| 核心问题 | 为什么命中正式库、为什么继续评估、为什么被丢弃 |

## 页面目的

用户进入这个页面，不是为了看复杂算法，而是为了回答：
- 本批数据有多少直接命中上一轮正式库？
- 有多少进入待评估链路？
- 有多少因为数据不全被丢弃？
- 碰撞 `cell_id` 的局部防护拦掉了多少错误命中？

## 页面模块

| 模块 | 用户要看什么 |
|------|--------------|
| 数据版本头部 | 当前数据集、运行批次、引用正式库版本 |
| 分流概览卡片 | 输入总量、Path A/B/C 数量与占比 |
| 碰撞防护卡片 | `collision_candidate_count`、成功命中、待定、丢弃 |
| 分流可视化 | 三条路径的比例 |
| Path B 基础画像统计 | 平均观测点、设备数、跨度、P50/P90 |
| 数据完整度区 | Path B 的 GPS / 信号原始覆盖率 |
| 当前规则口径区 | 只读展示碰撞阈值、去重窗口、质心算法 |
| 规则说明区 | 命中规则、碰撞局部防护规则、Path C 丢弃规则 |
| 支持状态区 | 明确当前支持能力与待开发功能 |

## 页面功能

| 功能 | 说明 |
|------|------|
| 查看当前批次路由总览 | 当前已支持 |
| 查看碰撞防护摘要 | 当前已支持 |
| 查看 Path B 基础画像摘要 | 当前已支持 |
| 按运营商 / LAC 筛选 | 待开发 |
| 查看碰撞样本 | 待开发 |
| 查看 Path B 列表 | 待开发 |
| 跳转 Step 3 | 待开发 |
| 跳转 Step 4 | 待开发 |

## 页面设计重点

| 重点 | 说明 |
|------|------|
| 先看流量去向 | 用户第一眼应该看到 A/B/C 比例 |
| 再看丢弃原因 | 丢弃必须有解释，不允许只有数量 |
| 再看 Path B 质量 | 帮用户判断 Step 3 是否有足够输入 |
| 规则可解释 | 用户要知道“为什么判成 Path A / B / C” |
| 能力边界清楚 | 未支持的筛选、样本、跳转能力要明确标成待开发 |

## 页面成功标准

| 标准 | 表现 |
|------|------|
| 看得懂分流 | 用户知道记录在三个方向上怎么流动 |
| 找得到异常 | 用户能快速定位碰撞误命中和 Path C 丢弃原因 |
| 接得上上下游 | 页面能顺畅跳到 Step 3 和 Step 4 |
