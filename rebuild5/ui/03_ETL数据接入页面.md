# ETL数据接入页面

本文件定义 Step 1 对应的五个页面：数据源注册、字段审计、解析、清洗、补齐。

## 页面组定位

| 项目 | 说明 |
|------|------|
| 页面组角色 | 展示原始数据如何进入统一结构化事实 |
| 主要用户 | 系统管理员 |
| 核心问题 | 数据源是否正常、字段是否合理、解析清洗是否正确 |
| 页面数量 | 5 页 |

## 页面一：数据源注册

| 项目 | 说明 |
|------|------|
| 用户目的 | 知道接了哪些原始源、来源范围和状态是否正常 |
| 核心模块 | 数据源列表、源表信息、状态标签、时间范围、记录数 |
| 关键功能 | 新增 / 编辑 / 停用数据源，查看数据源元数据 |
| 设计重点 | 这是配置入口，不是统计页；重点是“当前接了什么” |

## 页面二：字段审计

| 项目 | 说明 |
|------|------|
| 用户目的 | 判断原始字段哪些被保留、解析、丢弃 |
| 核心模块 | 字段覆盖率矩阵、字段类型、`keep / parse / drop` 决策、样本值 |
| 关键功能 | 按字段分类筛选、按覆盖率排序、展开看样本值 |
| 设计重点 | 要能快速看出“哪个字段缺失严重”“哪个字段决定被丢弃” |

## 页面三：解析

| 项目 | 说明 |
|------|------|
| 用户目的 | 判断 `cell_infos` 和 `ss1` 的解析扩展是否正常 |
| 核心模块 | 输入 / 输出数量、来源贡献分布、字段覆盖率变化 |
| 关键功能 | 按来源类型看行数和覆盖率，查看解析扩展比 |
| 设计重点 | 突出“解析前后多出了什么结构化信息” |

## 页面四：清洗

| 项目 | 说明 |
|------|------|
| 用户目的 | 看 ODS 规则删掉了哪些数据、通过率是否合理 |
| 核心模块 | 16 条规则列表、命中数、删除数、通过率、规则说明 |
| 关键功能 | 展开单条规则看条件和动作，查看命中统计 |
| 设计重点 | 要让用户知道“为什么数据被删”而不是只看总数 |

## 页面五：补齐

| 项目 | 说明 |
|------|------|
| 用户目的 | 看 Step 1 同报文内字段对齐是否提升了覆盖率 |
| 核心模块 | 补齐前后覆盖率对比、来源分布、时间约束说明 |
| 关键功能 | 查看 `<=1分钟` 与 `>1分钟` 的边界、查看来源分布 |
| 设计重点 | 必须强调这是同报文结构性对齐，不是历史知识补数 |

## 页面组之间的关系

```text
数据源注册
  -> 字段审计
  -> 解析
  -> 清洗
  -> 补齐
```

说明：
- 这五页共同回答“Step 1 是否产生了可信的 `etl_cleaned`”。
- 不要把 Step 4 的 donor 补数信息混进 Step 1 页面。
