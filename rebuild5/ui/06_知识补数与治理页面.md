# 知识补数与治理页面

本文件定义 Step 4 和 Step 5 的页面组：知识补数、Cell 维护、BS 维护、LAC 维护。

## 页面组定位

| 项目 | 说明 |
|------|------|
| 页面组角色 | 解释正式库命中后的补数效果，以及治理链路如何维护最终正式库 |
| 主要用户 | 治理分析师 |
| 核心问题 | 补了什么、为什么异常、为什么被维护成 stable / collision / migration / retired |
| 页面数量 | 4 页 |

## 页面一：知识补数

| 项目 | 说明 |
|------|------|
| 用户目的 | 看 Step 4 到底补了哪些字段、哪些记录被标成 GPS 异常 |
| 核心模块 | 补数概览卡片、来源分布图、donor 质量分布、字段级补数表、异常样本列表 |
| 关键功能 | 按运营商 / LAC / donor 质量 / 是否异常筛选 |
| 设计重点 | 强调“这是基于正式库的知识补数，不是同报文补齐” |

## 页面二：Cell 维护

| 项目 | 说明 |
|------|------|
| 用户目的 | 看哪些 Cell 被标成漂移、碰撞、多质心或进入退出链路 |
| 核心模块 | 漂移分类分布、碰撞 Cell 列表、GPS 异常时序结果、多质心列表、退出管理、滑动窗口状态、规则命中 |
| 关键功能 | 按异常类型筛选、查看单个 Cell 的规则命中与时间演化 |
| 设计重点 | 这是治理工作台，不是图表报表页；重点是“为什么这样判” |

## 页面三：BS 维护

| 项目 | 说明 |
|------|------|
| 用户目的 | 看异常是否从 Cell 上卷到 BS |
| 核心模块 | BS 分类分布、面积异常列表、下属 Cell 健康度、退出预警、规则命中统计 |
| 关键功能 | 展开查看下属 Cell 贡献、查看异常 BS 的来源 |
| 设计重点 | 突出“BS 异常来自下层传播” |

## 页面四：LAC 维护

| 项目 | 说明 |
|------|------|
| 用户目的 | 看区域层整体是否被异常污染 |
| 核心模块 | 区域质量趋势、异常 BS 比例变化、LAC 级变动提醒、多数据源对比预留 |
| 关键功能 | 看区域变化曲线、展开查看异常 BS 构成 |
| 设计重点 | 把 LAC 做成区域治理看板 |

## 页面组关系

```text
知识补数
  -> Cell 维护
  -> BS 维护
  -> LAC 维护
```

## 页面成功标准

| 标准 | 表现 |
|------|------|
| 补数可审计 | 用户能看出补了什么、来自哪个 donor |
| 异常可解释 | 用户能看出为什么是 collision / migration / retired |
| 治理有层次 | Cell、BS、LAC 维护页各自承担不同层级解释 |
