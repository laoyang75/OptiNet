# 数据集与运行管理

本文件定义“当前数据集”和“运行历史”两页。它们属于全局管理层，是整个系统 UI 的入口。当前版本仅支持单活数据集，不支持在线切换。

## 页面组定位

| 项目 | 说明 |
|------|------|
| 页面组角色 | 统一解决当前数据集展示、运行记录、版本追溯和控制操作展示 |
| 主要用户 | 系统管理员、治理分析师 |
| 核心问题 | 系统现在跑哪批数据？当前版本来自哪次运行？ |
| 页面数量 | 2 页 |

## 页面一：数据集选择

### 页面目的

让管理员明确看到系统当前正在处理哪一个数据集，并理解当前版本仍是单活模式。

### 页面模块

| 模块 | 用户要看什么 |
|------|--------------|
| 当前数据集卡片 | 当前正在使用的数据集、最近运行、当前发布版本 |
| 数据集列表 | 所有可选数据集及其状态 |
| 元数据面板 | 来源、导入时间、记录数、LAC 范围、时间范围 |
| 运行准备状态 | 是否已导入、是否可运行、是否已有历史结果 |
| 单活模式提示区 | 明确说明页面不支持在线切换；如需更换数据集需清理后重跑 |

### 页面功能

| 功能 | 说明 |
|------|------|
| 查看当前数据集 | 明确当前系统上下文 |
| 查看候选数据集 | 对比多个数据集的规模和用途 |
| 查看单活提示 | 知道当前实现为什么不支持在线切换 |
| 跳转运行历史 | 看当前数据集以往跑过什么版本 |

### 页面成功标准

| 标准 | 表现 |
|------|------|
| 选得清 | 用户一眼知道当前系统跑哪批数据 |
| 说得清 | 用户知道当前不支持在线切换，不会误解为可切换产品能力 |
| 看得全 | 元数据足够支持是否切换的判断 |

## 页面二：运行历史

### 页面目的

让用户知道“这次结果是怎么来的”，同时统一承接控制侧链的初始化、局部重算、完整回归和对比验证信息。

### 页面模块

| 模块 | 用户要看什么 |
|------|--------------|
| 历史列表 | 每次运行的时间、运行类型、数据集、状态 |
| 结果摘要 | 覆盖率、对象数、异常统计、版本号 |
| 对比入口 | 选择两次运行做结果对比 |
| 发布状态区 | 当前版本是否已发布、是否验证通过 |
| 影响范围区 | 是否局部重算、影响了哪些 LAC 或对象 |
| 参数说明区 | 用了哪份参数文件 |

### 页面功能

| 功能 | 说明 |
|------|------|
| 按运行类型筛选 | 日常运行 / 初始化 / 局部重算 / 完整回归 / 对比验证 |
| 查看单次运行详情 | 看清楚该运行的输入、范围、版本和结果摘要 |
| 比较两次运行 | 看对象数量、覆盖率、状态分布差异 |
| 查看发布状态 | 区分验证中、已发布、已回退 |
| 跳到对应页面 | 从某次运行跳到当时的流转、维护或服务页面 |

### 控制侧链如何呈现

| 控制操作 | 在页面中如何表达 |
|----------|------------------|
| 冷启动初始化 | 作为特殊 `run_type`，显示“无上一轮正式库”的说明 |
| 局部重算 | 显示影响范围、局部对象或 LAC 列表 |
| 完整回归 | 显示“从 Step 1 全量重跑”的标记 |
| 对比验证 | 显示方案 A / 方案 B 的比较结果入口 |

### 页面成功标准

| 标准 | 表现 |
|------|------|
| 查得到 | 任意一个线上版本都能追溯到来源运行 |
| 对得上 | 页面里的版本标识和下游页面一致 |
| 看得懂 | 控制操作不会散落到别的页面里 |
