# Step 04: 修复观察/异常工作台数据（样本模式）

## 前置条件
- `00`-`03` 已完成
- 阅读 `99_系统上下文.md`

## 工具要求
- 你必须使用 `PG17 MCP` 执行所有 SQL
- 你必须使用 Playwright 检查页面

## 目标
observation_workspace_snapshot 的三阶段进度真实，anomaly 数据来自真实检测。
在样本模式下，仅处理 6 个样本 LAC 范围内的对象。

## 修复内容

### 5.1 重新生成 observation_workspace_snapshot

对样本范围内每个 waiting/observing 的 Cell，基于真实指标计算三阶段进度：

- **Stage 1 (existence_eligible)**: 基于 record_count 和 active_days
- **Stage 2 (anchorable)**: 基于 gps_valid_count/10, distinct_dev_id/2, (1500-p90)/1500
- **Stage 3 (baseline_eligible)**: 基于 anchorable + 累计 batch 数

### 5.2 重新生成异常对象汇总

从样本范围 obj_cell/bs 中 health_state != 'healthy' 的对象生成：
- severity: collision_confirmed=high, collision_suspect=medium, gps_bias=low
- downstream_impact: BS 碰撞影响下属 Cell 数量

### 5.3 重新生成异常记录汇总

从样本范围 fact_rejected / fact_pending_issue 的 route_reason 聚合。

## 验证

对比 rebuild2 的 GPS anomaly 数据：
```sql
-- rebuild2 中标记为 gps_anomaly 的 Cell 在 rebuild4 中应为 gps_bias
SELECT r2.operator_code, r2.lac, r2.cell_id, r2.gps_anomaly, r2.gps_anomaly_reason,
  r4.health_state
FROM rebuild4.sample_dim_cell r2
JOIN rebuild4.obj_cell r4 ON r2.operator_code = r4.operator_code 
  AND r2.lac = r4.lac AND r2.cell_id::text = r4.cell_id
WHERE r2.gps_anomaly = true
LIMIT 20;
```
