# ETL Step 3: 补齐（同报文补齐）

## 做什么

同一条原始报文（record_id）内，不同来源的行互相补齐缺失字段。

## 核心思路

一条原始报文被"炸开"成多行后，同一个 cell 可能出现在 cell_infos 和 ss1 中。它们描述的是同一个物理基站，但各自携带的信息不同：

```
原始报文 record_id = abc123
├── cell_infos → cell 12345  有 GPS，有信号，有 WiFi
├── ss1 组1    → cell 12345  有信号（可能更详细），有 GPS（可能没有）
└── ss1 组2    → cell 12345  有信号，无 GPS
```

**为什么用 cell_id 做匹配**：在同一条报文中（同一次手机上报），同一个 cell_id 出现多次，一定是同一个物理基站。cell 碰撞（不同基站拥有相同 cell_id）在同一条报文内几乎不会发生。

## 补齐规则

### cell_infos 行之间

同一 record_id + 同一 cell_id 的 cell_infos 行（来自同一个 JSON，时间相同）：

- **全部字段都可以补齐**：lac、运营商编码、RSRP、RSRQ、SINR、Dbm、WiFi 名称、WiFi MAC、GPS
- 因为它们时间完全相同，描述的是同一瞬间的同一基站

### ss1 行的时间约束

ss1 每一组有自己的时间戳（ts_block），可能和 cell_infos 的时间有差异。补齐规则根据时间差区分：

**时间差 ≤ 1 分钟**：
- 全部字段可补齐：运营商、LAC、信号（RSRP/RSRQ/SINR）、WiFi、GPS
- 原因：1 分钟内手机位置和信号环境变化很小

**时间差 > 1 分钟**：
- **只补齐**：运营商编码、LAC
- **不补齐**：信号（RSRP 等）、WiFi、GPS
- 原因：超过 1 分钟，手机可能已移动，信号和 WiFi 环境已变化，但运营商和 LAC（位置区）相对稳定

### 补齐字段清单

| 字段 | ≤ 1分钟 | > 1分钟 | 说明 |
|------|---------|---------|------|
| operator_code | 可补 | 可补 | 运营商编码，短时间内不变 |
| lac | 可补 | 可补 | 位置区编码，短时间内不变 |
| rsrp | 可补 | 不补 | 信号强度，随位置快速变化 |
| rsrq | 可补 | 不补 | |
| sinr | 可补 | 不补 | |
| dbm | 可补 | 不补 | |
| lon/lat (GPS) | 可补 | 不补 | GPS 位置，随移动变化 |
| wifi_name | 可补 | 不补 | WiFi 随位置变化 |
| wifi_mac | 可补 | 不补 | |

## 补齐优先级

同一 record_id + cell_id 内有多个候选值时：
1. **GPS**：`raw_gps`（cell_infos 原始上报）> `ss1_own`（ss1 本组 GPS）
2. **信号**：取最强值（RSRP 最大 = 信号最强）
3. **运营商/LAC**：取 cell_infos 行的值（通常更可靠）

## 统计

补齐前后对比（样本数据）：

| 指标 | 补齐前 | 补齐后 | 提升 |
|------|--------|--------|------|
| GPS 覆盖率 | 84.8% | 97.5% | +12.7% |
| RSRP 覆盖率 | 86.9% | 94.6% | +7.7% |

## 代码位置

`pipeline.py → step3_fill()`

## 注意事项

- 补齐**不增减行数**，只填充空值
- 补齐后的值带**来源标记**（original / same_cell / none），方便区分
- 只在同一报文内补齐，**不跨报文**
- cell_id 碰撞在同一报文内几乎不会发生，这是使用 cell_id 做匹配的前提
