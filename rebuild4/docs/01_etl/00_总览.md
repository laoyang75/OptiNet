# ETL 数据处理 — 总览

## 这是什么

ETL 是 rebuild4 数据处理的第一步。它把手机 SDK 上报的原始数据，转换成可以分析的结构化数据。

简单说就是三步：**解析 → 清洗 → 补齐**。

## 数据从哪来

手机 SDK 每隔一段时间上报一条记录，包含 27 个字段。其中两个核心字段：
- `cell_infos`：JSON 格式，手机实时扫描到的所有基站
- `ss1`：文本格式，后台采集的信号数据

这两个字段里藏着基站、信号、GPS 等关键信息，需要解析出来。

## 三步处理

```
原始数据（27 列，每条记录 = 一次手机上报）
    │
    ▼
┌─────────────────────────────┐
│ Step 1: 解析（炸开）          │
│                               │
│ 一条原始记录 → 多条结构化记录  │
│ cell_infos JSON → 每个基站一行 │
│ ss1 文本 → 每个信号组一行      │
│                               │
│ 例：249,201 条 → 689,361 条   │
└──────────────┬──────────────┘
               │
               ▼
┌─────────────────────────────┐
│ Step 2: 清洗                  │
│                               │
│ 去掉无效数据：                │
│ · 无效运营商编码 → 置空       │
│ · LAC=0 或溢出值 → 置空      │
│ · CellID=0 或溢出 → 删除行   │
│ · 信号值越界 → 置空          │
│ · GPS 越界 → 标记无效        │
│                               │
│ 例：689,361 → 687,788        │
└──────────────┬──────────────┘
               │
               ▼
┌─────────────────────────────┐
│ Step 3: 补齐                  │
│                               │
│ 同一条原始报文内互补：         │
│ · cell_infos 有 GPS → 补给 ss1│
│ · ss1 有信号 → 补给 cell_infos│
│ 条件：同 record_id + 同 cell_id│
│                               │
│ GPS 覆盖率：84.8% → 97.5%    │
│ RSRP 覆盖率：86.9% → 94.6%   │
└─────────────────────────────┘
```
