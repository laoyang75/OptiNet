# 流转线数据准备与门禁设计（Claude 草案）

状态：可合并草案  
更新时间：2026-04-06

## 1. 目标

rebuild4 不能再出现“系统完成但无数据可评估”的情况。  
因此数据准备必须前置为正式主任务，而不是上线前补丁。

## 2. 最小可用数据集

### 2.1 P0：继承真相基线

内容：
- rebuild2 字段质量与 ODS 规则真相表
- trusted 过滤损耗真相表
- 质量差异与来源差异读模型

作用：
- 支撑治理页、初始化页、对照页
- 作为 rebuild4 数据准备完成与否的判定基础

### 2.2 P1：真实初始化集

内容：
- 至少 1 个 `real run`
- 至少 1 个 `real initialization batch`
- 至少 1 个 `baseline_version_out`
- 至少 1 组 batch 级四分流统计
- 至少 1 套对象聚合结果（LAC / BS / Cell）

作用：
- 支撑 `/runs`、`/flow/overview`、`/baseline`、`/objects`、画像页

### 2.3 P2：真实快照集

内容：
- 上述真实 batch 对应的多个 `snapshot_timepoint`
- 每个 timepoint 的阶段摘要与关键指标

作用：
- 支撑 `/flow/snapshot`

### 2.4 P3：真实增量集

内容：
- 至少 1 个引用上一版 baseline 的 `real incremental batch`
- 能观察对象状态变化、baseline 承接与四分流变化

作用：
- 支撑“动态系统”而非一次性初始化系统的验收

### 2.5 P4：评估场景集

内容：
- 明确标记为 `synthetic` 的评估 batch / snapshot / scenario

作用：
- 仅用于 UI 与流程评估补充
- 不能替代 P1-P3 的正式验收功能

## 3. Gate 设计

## 3.1 Gate A：真相源冻结

输入：
- `rebuild4/docs/01_foundation/*`
- `rebuild4/docs/02_research/*`

输出：
- 单一真相源清单
- 三线工作边界

校验：
- 文档校验：不存在并行主 prompt
- 路径校验：核心引用收敛到 `rebuild4/docs`

未通过：
- 不允许进入 schema 或 UI 实施

## 3.2 Gate B：数据准备基线冻结

输入：
- 数据清洗线真相基线
- 过滤损耗与质量差异说明

输出：
- P0 数据准备定义
- rebuild4 需补建元数据清单

校验：
- SQL 校验：关键数字可复算
- 文档校验：字段与规则冲突有裁决

未通过：
- 不允许进入治理页与初始化页实现

## 3.3 Gate C：实体与映射矩阵冻结

输入：
- 流转线合同
- UI 页面矩阵

输出：
- 页面/API/表/data_origin 绑定矩阵
- compare / governance 定位裁决

校验：
- 文档校验：每个核心页面都有主语与来源定义
- API 设计校验：返回结构含 `data_origin`

未通过：
- 不允许进入页面实现

## 3.4 Gate D：schema 与读模型落地

输入：
- 前三 Gate 冻结内容

输出：
- `rebuild4` / `rebuild4_meta` schema
- 读模型与控制元数据表

校验：
- SQL 校验：表结构、索引、外键/关联键存在
- API 校验：最小读接口可返回空状态或数据

未通过：
- 不允许接入 UI 页面

## 3.5 Gate E：P0 继承真相入库

输出：
- 字段质量、ODS、trusted 损耗、质量差异读模型已可查询

校验：
- SQL 校验：摘要数字与冻结文档一致
- API 校验：治理/初始化接口可出数
- 页面校验：治理页能显示继承真相

未通过：
- 初始化页不得显示“数据已准备完成”

## 3.6 Gate F：P1 真实初始化跑通

输出：
- 至少 1 个 `real run`
- 至少 1 个 `real initialization batch`
- 对应 baseline 与对象结果

校验：
- SQL 校验：run/batch/baseline/object/fact 均有真实记录
- API 校验：`/runs`、`/flow/overview`、`/baseline` 能返回 real 数据
- 页面 Playwright：上述页面首屏可见 real 上下文

未通过：
- 不得把 synthetic 提升成默认正式主语

## 3.7 Gate G：P2 真实快照 + P3 真实增量跑通

输出：
- 至少 1 套真实 snapshot 链
- 至少 1 个真实 incremental batch

校验：
- SQL 校验：snapshot 与 incremental 链完整
- API 校验：`/flow/snapshot` 和 overview 的 delta 可返回 real 数据
- 页面 Playwright：能切时间点、能看批次变化、能显示 baseline 承接

未通过：
- `/flow/snapshot` 不得作为正式通过页

## 3.8 Gate H：支撑治理与对照页验收

输出：
- compare / governance 定位与来源状态冻结
- 页面完成支撑性验收

校验：
- API 校验：来源状态可见
- 页面 Playwright：fallback/synthetic/real 提示符合合同

未通过：
- 不得声称 compare / governance 为正式实时模块

## 4. 门禁设计原则

1. 每一 Gate 都必须单独通过，不允许默认往下推进
2. 任何页面实现前，先确认该页面的最小数据前提是否满足
3. 任何 synthetic 数据只能补评估，不能补正式验收
4. 任何 fallback 数据只能保系统可打开，不能补正式主语
