# 流转线实体与语义合同（Claude 草案）

状态：可合并草案  
更新时间：2026-04-06

## 1. rebuild4 的系统主语

rebuild4 不是静态结果库，而是“本地动态治理系统”。
系统主语不是页面，不是 scenario，也不是某个 compare 结果，而是：

1. `run`：一次规则、合同、输入边界固定的执行上下文
2. `batch`：一次 run 内的处理批次
3. `baseline_version`：当前批次只读引用的上一版冻结基线
4. `snapshot_timepoint`：批次内可复看的阶段快照
5. `object`：LAC / BS / Cell 等治理对象
6. `fact_layer`：标准化、治理后、观察中、问题中、拒绝的事实分流

## 2. 核心实体定义

### 2.1 `run`

定义：
- 一次具有明确输入范围、规则版本、合同版本、目标模式的执行上下文

必须字段：
- `run_id`
- `run_mode`（初始化 / 增量 / 回放 / 评估）
- `contract_version`
- `rule_set_version`
- `data_origin`
- `started_at` / `finished_at`
- `status`

合同要求：
- 每个页面如果以 run 为上下文，必须可见 `run_id`
- 同一 run 下的 batch 必须共享合同版本与规则版本

### 2.2 `batch`

定义：
- run 内一次实际处理单元，是页面与流程观察的核心粒度

必须字段：
- `batch_id`
- `run_id`
- `batch_type`（初始化 / 增量 / 重算 / 评估）
- `baseline_version_in`
- `baseline_version_out`
- `data_origin`
- `status`
- 四分流统计摘要

合同要求：
- `/flow/overview` 的主语必须是 batch
- `/flow/snapshot` 的主语必须是 batch 内 timepoint
- 不允许用 scenario 名称替代 batch 主语

### 2.3 `snapshot_timepoint`

定义：
- 批次处理过程中某个冻结观察点

必须字段：
- `snapshot_id`
- `batch_id`
- `phase_code`
- `phase_order`
- `captured_at`
- `data_origin`
- 指向四分流及关键指标的摘要

合同要求：
- snapshot 必须可追溯到真实 batch
- synthetic snapshot 只能在明确评估模式中展示
- fallback 不能冒充正式 timepoint

### 2.4 `baseline_version`

定义：
- 在批末冻结、供下一批次只读引用的稳定参考版本

合同要求：
- 当前批次只能读取 `baseline_version_in`
- 当前批次产出的 `baseline_version_out` 只能服务下一批
- 页面必须明确显示当前查看的是哪一版 baseline

### 2.5 `fact_layer`

正式分流：
1. `fact_standardized`
2. `fact_governed`
3. `fact_pending_observation`
4. `fact_pending_issue`
5. `fact_rejected`

合同要求：
- `/flow/overview`、`/flow/snapshot`、`/runs` 必须使用这些层级，而不是临时 sample/full 标签
- 页面与 API 返回必须能追到对应事实层

## 3. `data_origin` 正式合同

### 3.1 取值与定义

- `real`：来自正式数据准备与正式流程的真实运行结果
- `synthetic`：为评估、场景回放或界面验证专门构造/回放的数据
- `fallback`：仅为系统可打开或功能占位提供的临时回退数据

### 3.2 使用边界

`real`：
- 可作为正式验收主语
- 可驱动 baseline、snapshot、overview、objects、profiles 等正式流程

`synthetic`：
- 可用于评估模式
- 必须显式展示 scenario / 构造方式 / 使用限制
- 不得默认取代 real 成为正式主语

`fallback`：
- 只能作为降级占位
- 必须显式警示
- 不得用于正式功能验收结论

### 3.3 强制绑定

所有核心页面与 API 必须绑定：
- 页面主语
- API 主语
- 表主语
- `data_origin`

若四者无法一一对应，该模块不得进入正式验收。

## 4. rebuild4 必须解决的 rebuild3 偏差

### 4.1 页面能渲染但主语不真实

整改要求：
- `overview` 只看 batch
- `snapshot` 只看 batch 内 timepoint
- scenario 只能是评估上下文，不是正式主语

### 4.2 synthetic 与 fallback 混用

整改要求：
- synthetic 允许存在，但只能以评估模式存在
- fallback 允许存在，但只能以降级模式存在
- compare / governance 若暂未真实化，必须在合同中声明“降级模块”

## 5. compare / governance 的位置

### 5.1 `compare`

定位：
- 支撑性验证模块，不是首页主工作流

正式要求：
- 仅当存在清晰 compare 语义、真实输入与明确对比对象时才可标记 `real`
- 在此之前可以保留 `synthetic` 或 `fallback`，但必须显式提示

### 5.2 `governance`

定位：
- 支撑治理解释模块，承接数据清洗线与流程线之间的证据展示

正式要求：
- 若展示的是来自 rebuild2 的已验证继承真相，可标记为 `real`，并在页面说明中注明“继承自 rebuild2 已验证数据”
- 若用临时样例占位，必须标记 `fallback`

## 6. 流程线完成判定

以下条件同时满足才算“流转线合同冻结”：

1. 实体定义冻结
2. `data_origin` 使用边界冻结
3. 页面/API/表/data_origin 绑定规则冻结
4. compare / governance 的正式级别冻结
5. Gate 与停机线冻结
