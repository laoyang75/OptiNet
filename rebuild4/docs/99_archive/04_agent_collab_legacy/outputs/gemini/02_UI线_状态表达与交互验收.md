# UI 线状态表达与交互验收（Gemini 草案）

状态：可合并草案  
更新时间：2026-04-06

## 1. 四种页面状态表达

### 1.1 `real`

页面必须展示：
- `run_id`
- `batch_id`
- `baseline_version`
- `data_origin = real`
- 更新时间或捕获时间

界面要求：
- 无降级警示
- 可以作为正式验收结果截图与结论依据

### 1.2 `synthetic`

页面必须展示：
- 明确 banner
- scenario 名称
- 适用范围与限制
- `data_origin = synthetic`

界面要求：
- banner 不能弱化
- 不得在首屏省略 synthetic 提示
- 不得把 synthetic 写成“当前正式结果”

### 1.3 `fallback`

页面必须展示：
- 显式降级说明
- 当前为何降级
- 哪些能力未正式接通
- `data_origin = fallback`

界面要求：
- 不得伪装成普通数据卡片
- 不得用于正式验收通过结论

### 1.4 `empty`

页面必须展示：
- 当前缺少什么前置条件
- 需要哪个数据准备步骤完成后才能有数
- 可以去哪一个页面或步骤继续操作

界面要求：
- honest empty，不得静默空白
- 不得自动拿 fallback 盖掉空状态

## 2. 页面必须展示的上下文

以下内容为页面级硬约束：

1. 主语标签（当前看的是 run / batch / snapshot / object / baseline）
2. `data_origin`
3. 关键主键（`run_id`、`batch_id`、对象 ID 或 baseline_version）
4. 时间上下文（开始/结束/捕获时间）
5. 若为 synthetic/fallback，需展示原因与限制

## 3. 关键交互闭环

### 3.1 主流程闭环

`/runs -> /flow/overview -> /flow/snapshot -> /objects -> /object/:kind/:id -> /baseline`

验收要求：
- 每一步都能带着上下文继续前进
- 页面跳转后主语不丢失
- 上下文标签能帮助用户确认自己在看什么

### 3.2 支撑治理闭环

`/initialization -> /governance -> /compare`

验收要求：
- initialization 解释“数据是否准备好”
- governance 解释“为什么这么准备、过滤了什么、继承了什么”
- compare 只在正式对照条件成立时作为验证入口

### 3.3 对象闭环

`/objects -> /object/:kind/:id -> /lac-profile|/bs-profile|/cell-profile`

验收要求：
- 对象详情与画像页主键一致
- 画像页字段名、状态名、说明层不漂移

## 4. 常见跑偏防线

1. 不能把 compare 再做成首页主角
2. 不能让 snapshot 脱离 batch 存在
3. 不能让页面只像 UI_v2，却没有真实主语
4. 不能让 synthetic/fallback 提示缩成不明显的小标签
5. 不能让 empty 页面没有下一步指引

## 5. 页面级交互验收模板

每个页面至少检查：

1. 首屏主语是否明确
2. `data_origin` 是否可见
3. 是否可进入下一层页面
4. 关键筛选器/切换器是否真的驱动数据变化
5. 空状态/降级状态是否诚实表达
