# 数据清洗线真相基线（Codex 草案）

状态：可合并草案  
更新时间：2026-04-06  
适用范围：rebuild4 数据清洗整理规则线

## 1. 结论先行

rebuild4 可以继承 rebuild2 的数据清洗成果，但必须先把“文档口径”升级成“可查询真相”。  
本线冻结结论如下：

1. `field_audit` 以数据库当前真实结果为准：`keep=17 / parse=3 / drop=7`。
2. `target_field` 以数据库当前真实结果为准：`55` 行，不再使用“约 60”。
3. `parse_rule / compliance_rule` 在元数据层当前为空，判定为“待补建资产”，不是“已存在资产”。
4. ODS 规则必须拆“定义层”和“执行层”两张口径：定义 `26` 条，执行统计当前覆盖 `24` 条。
5. trusted 过滤损耗必须成为正式可见能力，不再只展示过滤后的结果。

## 2. 真相源优先级（本线）

按以下顺序裁决冲突：

1. `rebuild2_meta.*` 与 `rebuild2.*` 的实时结果（最高优先）
2. `rebuild4/docs/02_research/01_rebuild2_字段质量调查.md`
3. `rebuild4/docs/01_foundation/02_rebuild4_新增需求与数据准备清单.md`
4. `rebuild4/docs/reference/rebuild2_baseline/*`（历史说明，仅作解释）

## 3. 冲突清单与裁决

### 3.1 字段审计冲突

- 历史文档口径：`keep 19 / parse 3 / drop 5`
- 当前实表口径：`keep 17 / parse 3 / drop 7`
- 裁决：rebuild4 正式口径采用 `17 / 3 / 7`，并在文档中保留“历史口径差异”说明。

当前 `drop` 字段（按实表）：

- 当前数据最终经度
- 当前数据最终纬度
- android_ver
- 基带版本信息
- arp_list
- imei
- gps定位北京来源ss1或daa

### 3.2 目标字段数量冲突

- 历史文档口径：“约 60”
- 当前实表口径：`55`
- 裁决：rebuild4 冻结 `55`，并要求导出正式字段清单。

### 3.3 解析/合规规则冲突

- 历史文档口径：`parse_rule=64`，`compliance_rule=17`
- 当前实表口径：两表当前均为 `0`
- 裁决：判定为“需补建的正式元数据”，禁止继续按“已落库规则”叙述。

### 3.4 ODS 规则冲突

- 历史文档口径存在多个版本（26 条内部分类不同）
- 当前实表口径：
  - 定义层 `ods_clean_rule=26`
  - 执行层 `ods_clean_result` 当前覆盖每表 24 条
  - 已定义但未进入执行统计：`NULL_WIFI_NAME_INVALID`、`NULL_WIFI_MAC_INVALID`
- 裁决：rebuild4 必须强制显示“定义层 vs 执行层差异”，并建立缺口跟踪。

## 4. 可直接继承的成果

1. 字段审计结果：`field_audit`
2. 目标字段字典：`target_field`
3. ODS 清洗规则定义：`ods_clean_rule`
4. ODS 清洗执行统计：`ods_clean_result`
5. L0 质量缓存：`l0_stats_cache`
6. trusted 过滤基础：`dim_lac_trusted` 与 `l0_lac` 的过滤关系
7. 过滤后/过滤前关键统计：数量、制式分布、来源分布、GPS/信号保有量

## 5. 不能直接继承、必须补建的资产

1. 可查询的 `parse_rule` 正式元数据
2. 可查询的 `compliance_rule` 正式元数据
3. ODS 规则定义层-执行层差异台账
4. 按来源/制式/字段三维的质量读模型
5. 过滤损耗可追溯链路（规则 -> 影响 -> 最终结果）

## 6. 字段质量解释基线（防误读）

以下解释必须写入 rebuild4 说明层：

1. `CellID` 空值接近 0，不是当前主要质量风险字段。
2. 重点解释字段应是 `operator / gps / sinr / dbm / cell_ts`。
3. `ss1` 与 `cell_infos` 的质量差异是结构性差异，不是单一脏数据事件。
4. 2G/3G 下 `RSRP/SINR` 大面积为空，属于协议/制式语义，不应直接判定为异常数据。

## 7. 必须进入 rebuild4 的损耗事实

trusted 过滤（以 `l0_lac` 为准）：

- 总量：43,771,306
- 命中 trusted：30,082,381
- 被过滤：13,688,925（31.27%）

被过滤数据仍含有效信息（关键事实）：

- 有 RSRP：12,017,352
- 有 GPS：11,350,552
- 4G：10,004,716
- 5G：3,357,500

结论：过滤不是“去垃圾”一句话能解释，必须提供规则与业务边界说明。

## 8. 交付约束（本线）

rebuild4 数据清洗线在统一任务书中，至少要提供三类可验收对象：

1. 真相基线文档与冻结口径
2. 元数据与统计补建清单
3. SQL/API/页面一致性验收清单

若缺任一类，不得判定“数据清洗线完成”。
