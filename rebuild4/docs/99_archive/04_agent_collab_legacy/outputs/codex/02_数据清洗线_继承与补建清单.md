# 数据清洗线继承与补建清单（Codex 草案）

状态：可合并草案  
更新时间：2026-04-06

## 1. 直接继承项

### 1.1 直接继承的数据资产

1. `rebuild2_meta.field_audit`
2. `rebuild2_meta.target_field`
3. `rebuild2_meta.ods_clean_rule`
4. `rebuild2_meta.ods_clean_result`
5. `rebuild2_meta.l0_stats_cache`
6. `rebuild2.dim_lac_trusted`
7. `rebuild2.l0_gps`
8. `rebuild2.l0_lac`
9. `rebuild2.dwd_fact_enriched`

### 1.2 直接继承的事实解释

- rebuild2 已经完成字段 keep / parse / drop 的第一轮裁决
- rebuild2 已经完成 ODS 清洗规则定义与大部分执行统计
- rebuild2 已经提供 trusted 过滤边界
- rebuild2 已经产出 L0 质量统计与 enriched 事实

### 1.3 继承约束

- 继承的是“真实结果 + 可追溯说明”，不是旧文档里的模糊描述
- 继承必须保留“过滤前 -> 清洗后 -> trusted 后 -> enriched 后”的链路
- 继承后必须能被 rebuild4 页面与 API 消费

## 2. rebuild4 需要补建的正式元数据

## 2.1 字段决策基线表

建议表：`rebuild4_meta.source_field_decision`

输入：
- `rebuild2_meta.field_audit`
- `rebuild2_meta.target_field`
- 历史文档差异说明

输出：
- 每个原始字段的正式裁决状态
- `keep / parse / drop` 理由
- 对应 target field 映射
- 当前裁决来源与版本

校验：
- SQL：字段总数与分布和冻结口径一致
- 文档：差异字段有裁决说明
- 页面：未来治理页能按字段看到裁决与理由

## 2.2 解析规则目录表

建议表：`rebuild4_meta.parse_rule_catalog`

输入：
- rebuild2 相关脚本与文档中的解析规则
- 实际已落表字段的解析来源

输出：
- 规则名称、作用字段、输入字段、解析逻辑摘要、适用来源

校验：
- SQL：规则数量 > 0 且字段映射可回溯
- API：规则目录接口可返回规则清单
- 页面：治理页可查看规则定义

## 2.3 合规规则目录表

建议表：`rebuild4_meta.compliance_rule_catalog`

输入：
- rebuild2 文档中的合规规则
- 当前 ODS 与 trusted 实际筛选边界

输出：
- 规则定义、触发条件、处理动作、作用阶段

校验：
- SQL：规则目录存在且每条规则有唯一 code
- API：规则详情可查询
- 页面：治理页可查看合规规则说明

## 2.4 ODS 规则定义与执行差异表

建议表：
- `rebuild4_meta.ods_rule_catalog`
- `rebuild4_meta.ods_rule_effect_summary`
- `rebuild4_meta.ods_rule_gap_registry`

输入：
- `rebuild2_meta.ods_clean_rule`
- `rebuild2_meta.ods_clean_result`

输出：
- 定义层规则清单
- 执行层按目标表统计
- 未执行 / 未补录差异清单

校验：
- SQL：定义层总数 = 26，执行层差异被明确记录
- API：规则统计接口可按表、规则、来源筛选
- 页面：治理页能展示定义数、执行数和缺失项

## 2.5 trusted 过滤损耗汇总表

建议表：`rebuild4_meta.trusted_filter_loss_summary`

输入：
- `rebuild2.l0_lac`
- `rebuild2.dim_lac_trusted`

输出：
- 总量、命中量、过滤量、过滤比例
- 过滤记录中的 GPS / 信号保有量
- 过滤来源分布、制式分布

校验：
- SQL：能稳定复算 43,771,306 / 30,082,381 / 13,688,925
- API：损耗摘要接口返回总量与维度拆分
- 页面：治理页能解释“过滤但仍有效”的记录

## 2.6 来源质量汇总表

建议表：`rebuild4_meta.source_quality_summary`

输入：
- `rebuild2.l0_gps`
- `rebuild2.l0_lac`
- `rebuild2_meta.l0_stats_cache`

输出：
- 按来源、来源明细、Cell 来源、制式的空值率与记录量

校验：
- SQL：能区分 `cell_infos` 与 `ss1` 的结构性差异
- API：质量摘要接口支持分组查看
- 页面：字段质量页能说明“结构性差异不是单点故障”

## 3. rebuild4 数据准备线最低前置

要让 rebuild4 完成后立即有数据可评估，至少要先准备：

1. 一份冻结的字段质量真相基线
2. 一份冻结的 ODS 规则定义与执行差异台账
3. 一份 trusted 过滤损耗读模型
4. 一份按来源 / 制式 / Cell 来源的质量汇总读模型
5. 一套能供治理页与初始化页直接消费的 API 返回结构

## 4. 建议并入页面的能力

### 4.1 数据治理页

必须看到：
- 字段 keep / parse / drop
- ODS 规则影响量
- trusted 过滤损耗
- 来源质量差异

### 4.2 初始化页

必须看到：
- 初始数据准备的前置状态
- trusted 与 quality 基线是否到位
- 若缺任何关键元数据，不允许标记“准备完成”

### 4.3 对照 / 验证页

必须看到：
- 过滤前后数量差异
- 关键字段空值差异
- 规则执行前后变化点

## 5. 本线完成判定

数据清洗线只有同时满足以下条件才算完成：

1. 冻结口径文档已完成
2. 元数据补建清单已冻结
3. SQL 校验清单已冻结
4. 页面/API 映射要求已传递给 UI 与流程线
5. trusted 损耗与质量差异已经被纳入 rebuild4 正式可见范围
