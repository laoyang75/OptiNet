# 统一重构基线

状态：统一稿
更新时间：2026-04-06
编制说明：基于 `Codex / Claude / Gemini` 三线草案合并，数据库事实以 `PG17 MCP execute_sql` 查询为准。

## 1. rebuild4 的统一目标

rebuild4 不是 rebuild3 的补丁版，也不是只把 UI_v2 再实现一遍。

统一目标只有一个：

- 构建一个可持续运行的动态治理系统
- 让使用者能够研究数据在不同阶段的流转参数与状态变化
- 在系统完成后立即具备可评估数据，而不是只交付空页面或仅 synthetic 演示

## 2. 唯一真相源规则

### 2.1 文档真相源

正式执行时只认：

- `rebuild4/docs/01_foundation/`
- `rebuild4/docs/02_research/`
- `rebuild4/docs/UI_v2/`
- `rebuild4/docs/reference/`
- `rebuild4/docs/05_unified/`
- `rebuild4/docs/06_execution/`

其中真正的执行总控文档只允许有一份：

- `rebuild4/docs/06_execution/01_rebuild4_人工步骤任务文档.md`

### 2.2 数据库真相源

数据库事实一律通过 `PG17 MCP execute_sql` 查询确认。

规则：

- 计数、分布、差异、缺口：以 `execute_sql` 为准
- 表结构、字段、索引：可以用 `search_objects` 辅助
- 不允许用 shell 输出、旧文档抄录或页面展示反推数据库事实

## 3. 三条主线的统一边界

### 3.1 数据清洗整理规则线

负责：

- 继承 rebuild2 的字段质量成果
- 冻结 keep / parse / drop 基线
- 解释 ODS 规则定义层与执行层
- 把 trusted 过滤损耗与来源差异纳入正式可见范围

### 3.2 流转处理流程线

负责：

- 冻结 run / batch / baseline_version / snapshot_timepoint / object / fact_layer 合同
- 定义初始化、增量、快照、回放、比较、治理之间的关系
- 规定 real / synthetic / fallback / empty 的使用边界
- 把 Gate 变成真正的停机线

### 3.3 操作 UI 线

负责：

- 以 `UI_v2` 为页面语义基线
- 冻结页面 -> API -> 表/视图 -> data_origin 映射矩阵
- 明确 empty / synthetic / fallback / real 的展示与交互规则
- 让 Playwright 进入正式验收

## 5. rebuild4 的统一实体合同

### 5.1 主实体

正式冻结：

- `run`
- `batch`
- `baseline_version`
- `snapshot_timepoint`
- `object`（LAC / BS / Cell）
- `fact_layer`

### 5.2 正式事实分流

正式冻结为五层：

- `fact_standardized`
- `fact_governed`
- `fact_pending_observation`
- `fact_pending_issue`
- `fact_rejected`

### 5.3 页面主语硬约束

- `/flow/overview` 的主语必须是 `batch`
- `/flow/snapshot` 的主语必须是 `batch + snapshot_timepoint`
- `/runs` 的主语必须是 `run`，并可下钻到 `batch`
- `scenario` 只能作为评估上下文，不能抢占正式主语

## 6. `data_origin` 统一合同

### 6.1 允许值

仅允许：

- `real`
- `synthetic`
- `fallback`
- `empty`

### 6.2 统一定义

- `real`：正式真实结果，或已导入 rebuild4 并可追溯的数据继承真相
- `synthetic`：评估/回放/演示用构造结果
- `fallback`：仅保证系统可打开或功能占位的临时数据
- `empty`：真实缺少前置数据时的诚实空状态

### 6.3 关键裁决

- 若治理页使用的是可追溯的 rebuild2 继承真相，则 `data_origin = real`，同时返回 `origin_reason = inherited_from_rebuild2`
- 若 compare/governance 只是临时样例或手填占位，则必须 `fallback`
- `/flow/overview` 与 `/flow/snapshot` 默认只允许 `real` 或 `empty`
- `synthetic` 只能在用户明确进入“评估模式”后出现，且必须有 banner

## 7. compare / governance 的统一定位

### 7.1 `compare`

- 归入 P2 支撑模块
- 只有具备真实 compare 输入、真实对照语义、可追溯来源时，才允许标记 `real`
- 在此之前只能 `synthetic` 或 `fallback`

### 7.2 `governance`

- 归入 P2 支撑治理模块
- 若展示的是 rebuild2 继承真相，属于正式能力，不是占位页
- 但必须显式标注来源说明，避免误解为“实时由 rebuild4 流程刚刚生成”

## 8. 统一反跑偏规则

1. 不再并行执行多份总 prompt
2. 不允许先做空页面，再事后补主语与数据来源
3. 不允许把 synthetic 静默升格成正式主语
4. 不允许只凭页面外观判断“符合 UI_v2”
5. 每一步都必须有校验；涉及页面的步骤必须有 Playwright
6. 任一 Gate 未通过，不得默认进入下一阶段
