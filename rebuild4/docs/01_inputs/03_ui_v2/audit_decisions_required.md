## Q1: 对象浏览页 vs LAC/BS/Cell 画像页的导航关系

**背景**：来自原始待确认问题 Q1，对应审计项 G7。当前设计已经把两类页面都做出来，但 IA 还没拍板。

**三方现状**：
- 冻结文档：要求页面围绕对象/基线/异常/验证组织，但未规定对象页与画像页必须合并还是分离。
- UI v2：`docs/rebuild3/UI_v2/design_notes.md:92` 认为对象浏览是治理视角，09/10/11 是数据质量视角；`docs/rebuild3/Docs/questions_for_review.md:5` 仍保留三种组织方案。
- rebuild2：只有分离的 LAC/BS/Cell 画像页，没有治理态对象页。

**如果不处理**：如果不先定导航关系，前端路由、侧边栏层级和对象详情复用边界都会反复改。

**选项**：
- A) 对象浏览做主入口，画像页作为对象详情内的 Tab — 治理与画像上下文最一致，但对象详情会变重。
- B) 对象浏览与画像页并列导航 — 信息架构清楚，但侧边栏层级更深。
- C) 合并成一个页面，治理/画像两个大 Tab — 入口最少，但首屏信息密度最高。

**我的建议**：推荐选 B。因为 UI v2 已经明确把“治理状态视角”和“数据质量视角”拆成两种主语，且对象详情还承担异常/资格/下游影响，继续并列更利于后端读模型分层。


选择 B

## Q2: 画像页是否保留 rebuild2 的 classification/confidence 字段

**背景**：来自原始待确认问题 Q2，对应审计项 A2、A5、A6、F10、F11。

**三方现状**：
- 冻结文档：对象层只冻结了 `health_state`、`anchorable`、`baseline_eligible`。
- UI v2：BS/Cell 画像仍大量使用“异常分类 + GPS 可信度 + 信号可信度”。
- rebuild2：`_sample_bs_profile_summary`、`_sample_cell_profile_summary` 直接提供 `classification_v2`、`gps_confidence`、`signal_confidence`。

**如果不处理**：如果两套字段并行但角色不清，会让对象页和画像页出现两套状态真相。

**选项**：
- A) 完全删除旧字段，只保留 rebuild3 新模型 — 状态最统一，但会损失现有质量分档资产。
- B) 保留空间/信号质量指标，但把 classification/confidence 降级为解释标签 — 兼顾迁移成本与新模型一致性。
- C) 新旧两套字段并行长期保留 — 迁移最轻，但最容易制造口径漂移。

**我的建议**：推荐选 B。空间精度、P50/P90、原始率等画像指标值得保留，但 `classification_v2` 和 `gps_confidence` 不应再成为主状态字段。

选择:b

## Q3: 全局搜索的范围

**背景**：来自原始待确认问题 Q3。

**三方现状**：
- 冻结文档：强调对象、批次、基线三类主语，但没有强制全文搜索。
- UI v2：所有页面都宣称支持搜索；待确认问题里保留了对象主键 / 批次 / 全文三种范围。
- rebuild2：旧画像页基本按主键筛选，不提供统一全局搜索。

**如果不处理**：如果范围定义过宽，后端要为异常类型、文案、状态做全文索引；过窄又无法快速跳批次。

**选项**：
- A) 只搜对象主键 — 实现最简单，但批次/运行定位要另找入口。
- B) 搜对象主键 + run_id + batch_id — 覆盖最常见使用场景，复杂度可控。
- C) 做全文搜索 — 最强，但需要额外索引和结果归类。

**我的建议**：推荐选 B。对象、run、batch 正好对应 UI 三条核心主语，能满足大多数排障链路。

选择 B 但需要调整,针对特定页面实际上应该搜索的是 lac bs cell 因为这个的目的是快速定位到具体的对象 或者检查上一次的问题是否解决.

## Q4: 初始化页面是否放入主导航

**背景**：来自原始待确认问题 Q4，对应审计项 G1。

**三方现状**：
- 冻结文档：初始化是冷启动入口，但不是日常主链路。
- UI v2：目前把 `12_initialization.html` 放在侧边栏一级导航。
- rebuild2：没有对应的初始化总览页。

**如果不处理**：若把初始化放得太高，会稀释“流转总览 -> 对象/异常/基线”的日常工作路径。

**选项**：
- A) 保留一级导航 — 查找方便，但长期使用权重偏高。
- B) 并入运行/批次中心作为特殊 run 类型 — 最贴近运行语义，导航更紧凑。
- C) 放到系统/辅助分组 — 最不干扰主流程，但入口更深。

**我的建议**：推荐选 B。初始化本质上是特殊 run，放进运行/批次中心最符合“先看运行，再看对象”的主叙事。

选择 B

## Q5: 侧边栏是否按功能分组

**背景**：来自原始待确认问题 Q5。

**三方现状**：
- 冻结文档：只冻结了页面主语，没有规定侧边栏层级。
- UI v2：当前导航项已达 12 个。
- rebuild2：旧前端多为 step 导航，不具参考价值。

**如果不处理**：如果不分组，侧边栏会越来越像页面清单；如果分组过深，又会增加点击成本。

**选项**：
- A) 按治理主线 / 对象 / 工作台 / 验证分组 — 信息架构最清楚。
- B) 平铺 + 对象画像做二级菜单 — 折中方案，能控制导航长度。
- C) 保持完全平铺 — 实现最简单，但可扩展性最差。

**我的建议**：推荐选 B。对象画像天然属于“对象”分组，二级菜单能在不牺牲入口的前提下收住导航长度。

选择 B

## Q6: delta 展示的统一范围

**背景**：来自原始待确认问题 Q6。

**三方现状**：
- 冻结文档：04c 强调所有关键页面都要表达变化，而不仅是总量。
- UI v2：目前首页、批次中心、对象列表、等待页、异常页都在展示 delta。
- rebuild2：旧页面更偏静态表格，不形成约束。

**如果不处理**：如果全页面硬上 delta，视觉噪音会上升；如果只保留首页，很多工作台会失去“变化”语义。

**选项**：
- A) 所有关键页面都显示 delta — 最贴近 04c，但信息密度最高。
- B) 只在首页和批次中心显示 — 页面更安静，但工作台会回退成静态列表。
- C) 关键页面默认显示，次级页面可折叠/隐藏 delta — 兼顾信息量与可读性。

**我的建议**：推荐选 C。让首页和批次中心强制显示 delta，其他工作台默认显示但允许折叠，是最稳的折中方案。

选择 C

## Q7: `classification_v2` 如何映射到 rebuild3 `health_state` 与异常标签

**背景**：来自审计项 A5。当前 UI 画像页和 rebuild2 数据都仍依赖旧分类。

**三方现状**：
- 冻结文档：对象级异常只认可 `collision_suspect / collision_confirmed / dynamic / migration_suspect / gps_bias`；`single_large`、`normal_spread` 属于记录级异常。
- UI v2：BS/Cell 画像直接展示“动态BS / 碰撞确认 / 碰撞嫌疑 / 面积大 / GPS噪声”。
- rebuild2：数据库里还存在额外的 `collision_uncertain`。

**如果不处理**：如果不明确映射，后端会同时维护对象健康和旧分类两套真相。

**选项**：
- A) 所有旧分类都硬映射成 `health_state` — 实现简单，但会错误抬高记录级异常的等级。
- B) 只把 dynamic/collision* 映射为 `health_state`，其余降为 anomaly tags — 最符合冻结文档。
- C) 完全保留旧分类，不做迁移 — 最省事，但与冻结版冲突最大。

**我的建议**：推荐选 B。另建议把 `collision_uncertain` 临时并到 `collision_suspect`，同时在审计记录里保留原值。

选择 B

## Q8: `gps_confidence / signal_confidence` 如何迁移到 `anchorable / baseline_eligible`

**背景**：来自审计项 A6。

**三方现状**：
- 冻结文档：资格字段必须是 `anchorable` 与 `baseline_eligible`。
- UI v2：BS/Cell 画像仍显式展示 GPS 可信度。
- rebuild2：旧画像表提供 `high / medium / low / none` 四级分档。

**如果不处理**：如果直接把四级分档改成布尔值，会丢掉质量信息；若继续把分档当资格，又会违背冻结版。

**选项**：
- A) 把 high/medium 映射为 true，low/none 映射为 false — 简单，但会掩盖异常和时间跨度等条件。
- B) 资格由规则矩阵计算，confidence 仅保留为画像质量指标 — 最符合冻结版。
- C) 彻底删除 confidence，不再保留 — 状态最纯粹，但损失现有质量信号。

**我的建议**：推荐选 B。资格由对象状态 + 样本门槛 + 异常矩阵决定，confidence 只做画像解释指标。

选择 B

**如果不处理**：如果不提前拍板，流转快照页只能依赖实时计算，无法支持重跑前后对照。

- C) 不做持久化快照，只实时查询 — 最省开发，但性能与可追溯性最差。

**我的建议**：推荐选 A。快照页已经是 UI v2 的核心差异化能力，值得用一张轻量 stage-snapshot 表支撑。

## Q12: LAC 页面是否允许使用区域自定义健康标签

**背景**：来自审计项 A2。LAC 画像引入了 `覆盖不足` 这种冻结版没有的状态。

**三方现状**：
- 冻结文档：对象健康只冻结了 7 个 `health_state`。
- UI v2：LAC 页使用“健康 / 碰撞嫌疑 / 覆盖不足”等区域标签。
- rebuild2：没有 LAC 级健康状态，只能通过异常 BS 和密度聚合得到区域诊断。

**如果不处理**：如果区域标签直接替代 `health_state`，对象模型会再次分裂。

- A) LAC 也严格只用冻结的 `health_state` — 最统一，但会损失区域诊断表达。
- B) LAC 保留 `health_state`，另加 `region_quality_label` 作为派生标签 — 兼顾统一模型与区域解释。
- C) LAC 继续使用独立健康标签体系 — 页面最直观，但与冻结版冲突。

**我的建议**：推荐选 B。让 `health_state` 继续保持统一枚举，把 `覆盖不足` 降级为区域质量派生标签。

选择 B
