# 三线重构策略

状态：策略草案（保留为领域拆解参考，不再作为 agent 分工方案）  
更新时间：2026-04-06

---

## 1. 为什么拆成三条线

你现在的判断是对的，rebuild4 最适合拆成三条线：

1. **数据清洗整理规则线**
   - 继承 rebuild2 的成果
   - 回答字段质量、清洗规则、过滤损耗、来源差异、可信边界
2. **流转处理流程线**
   - 承接 rebuild3 的核心设计
   - 回答 run / batch / snapshot / baseline / 四分流 / 状态机 / 增量链路
3. **操作 UI 线**
   - 承接 UI_v2
   - 把前两条线转成可观察、可操作、可验收的工作台

这样拆的好处是：

- 每条线都有明确边界
- 即使 3 个 agent 做同题评估，也能用这三条线检查是否覆盖完整
- 最后统一时更容易识别冲突来自哪条领域

---

## 2. 三条线各自的产物

### 2.1 数据清洗整理规则线

应该产出：

- 字段质量真相基线
- keep / parse / drop 正式清单
- ODS 规则定义层 / 执行层对照
- trusted 过滤损耗与来源分析
- rebuild4 需要继承/补建的元数据表清单

### 2.2 流转处理流程线

应该产出：

- 初始化与增量的统一语义
- run / batch / baseline / snapshot 合同
- 四分流与状态迁移矩阵
- real / synthetic / fallback 的正式定义
- 数据准备最小可用集
- 各 Gate 的通过条件

### 2.3 操作 UI 线

应该产出：

- 页面 -> API -> 表 -> data_origin 映射矩阵
- 页面最小可用数据条件
- 空状态 / synthetic / fallback 展示规则
- 页面验收标准
- Playwright 验收清单

---

## 3. 三条线如何合并

合并时不是“谁写得多听谁的”，而是按冲突来源裁决：

- 业务语义冲突：流程线优先，但不得违反 Tier 0
- 数据质量冲突：数据清洗线优先，以数据库真实结果为准
- 页面表达冲突：UI 线优先，但不得伪造主语或越权解释数据

最终统一时，必须沉淀成：

1. 一份统一主任务书
2. 一份统一映射矩阵
3. 一份统一验证清单

---

## 4. 强制校验规则

无论哪条线，最后进入人工步骤任务书时都必须补齐：

- 文档校验
- 数据库 / SQL 校验
- API 校验
- 页面 Playwright 校验
- 差异记录与回写

不允许出现：

- 只验证 SQL 不验证页面
- 只验证页面不验证数据来源
- 只验证首页不验证下钻链路

---

## 5. 在当前流程里的正确用法

这份文档现在的作用是：

1. 作为领域检查框架，提醒每一轮都必须覆盖数据 / 流转 / UI 三部分
2. 作为合并时的冲突定位辅助，帮助判断分歧属于哪条线
3. 不再直接决定“哪个 agent 负责哪一块”

当前正式执行流程已经切换为：

1. 3 个 agent 做同题独立评估
2. 主控统一合并
3. 必要时由人类做 A / B / C 裁决
4. 最终冻结成执行任务书
