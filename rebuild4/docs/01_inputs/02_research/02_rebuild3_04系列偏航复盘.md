# rebuild3 04 系列偏航复盘

状态：已复盘  
更新时间：2026-04-06

---

## 1. 这组文档是什么

`rebuild3/docs/04*` 不是一份文档，而是一条“反复纠偏的历史轨迹”。

主要节点：

- `04a_prompt_UI设计.md`：第一版 UI prompt，产物输出到 `UI/`
- `04b_prompt_开发实施.md`：第一版开发实施 prompt
- `04c_UI目的与改版说明.md`：明确指出 UI 偏差
- `04d_prompt_UI设计_重写版.md`：第二版 UI prompt，切换到 `UI_v2/`
- `04e_prompt_开发实施_正式重构流程版.md`：正式重构流程版，加入 Gate 与样本先行
- `04f_prompt_开发实施_UI优先纠偏与归档版.md`：承认 04e 后仍跑偏，改用 UI-first + 归档纠偏

这组文档说明的不是“当初文档不清楚”，而是“执行阶段的控制机制失效”。

## 2. 每一轮在修什么

### 2.1 `04a`

目标：

- 先做 UI 设计
- 用静态 HTML 落设计稿
- 生成一套完整页面与启动器原型

问题：

- 这版仍然容易滑向传统 dashboard / replay 导向
- 输出目录是 `UI/`，后来被证明不是最终 UI 基线

### 2.2 `04b`

目标：

- 在 UI 审查通过后进入开发
- 先建目录、再迁文档、再搭全栈项目骨架

问题：

- 一份 prompt 同时要求目录、配置、schema、后端、前端、launcher、开发顺序
- 任务面过大，执行时容易优先抓“容易落地的代码”，而不是先锁死真相源和主语

### 2.3 `04c`

价值：

- 这是最重要的一份纠偏思想文档
- 它明确指出：
  - 首页太像 dashboard
  - replay / compare 被抬太高
  - 没有把“非全量重跑”写成核心语义
  - 页面优先级没有按决策节奏组织

问题：

- 它提出了正确方向，但没有直接变成唯一执行约束

### 2.4 `04d`

目标：

- 重写 UI prompt
- 明确 `UI_v2` 才是新基线
- 把首页改成“流转总览”，把 replay 降级为验证模块

价值：

- 明确把产品心智从“对比调试器”拉回“流式治理工作台”

问题：

- 新 UI 基线形成了，但旧的 `04a/UI` 并没有被彻底制度化退役

### 2.5 `04e`

目标：

- 正式化开发流程
- 引入 Gate A-G
- 强调样本先行、双跑对比、通过后全量

价值：

- 首次把“门禁”和“样本先行”系统化写出来

问题：

- 仍然过大：一份 prompt 负责从目录迁移一直到全量偏差评估
- Gate 是写出来了，但没有真正成为“未过不许继续”的停机线

### 2.6 `04f`

目标：

- 承认 `04e` 之后正式交付仍明显偏离
- 尤其是 UI 与启动器没有按 `UI_v2` 基线落地
- 引入 UI-first 审计、映射矩阵、归档策略

价值：

- 明确说出“不要再把理解页面语义当成实现了 UI_v2”
- 强调页面级验收和归档治理

问题：

- 又叠加了一层新 prompt
- 执行链继续变长，文档树继续膨胀

## 3. 真正的问题不在内容，而在执行结构

### 3.1 真相源不断切换，但旧文档没有真正退役

发生过的切换：

- `UI/` -> `UI_v2/`
- `docs/rebuild3/` -> `rebuild3/docs/`
- 普通开发 prompt -> 正式重构流程 prompt -> UI-first 纠偏 prompt

问题：

- 新文档出来后，旧文档仍然在目录里保持“可执行状态”
- 执行者容易同时参考多份 prompt
- 结果就是：真相源不是唯一的

### 3.2 一份 prompt 试图解决太多问题

尤其是 `04b`、`04e`、`04f`，都同时想解决：

- 目录迁移
- 项目骨架
- 页面实现
- 后端实现
- 样本验证
- 全量验证
- 启动器
- 归档

这种写法的问题不是不完整，而是：

- 临界路径模糊
- 优先级容易漂
- 执行过程中很容易“先做会动的部分”

### 3.3 Gate 有写出来，但没有真正成为硬门禁

`04e` 已经提出 Gate A-G，这是对的。

但实际问题是：

- Gate 只是写在 prompt 里
- 没有变成单独文件、单独验收、单独停机条件
- 所以更像说明文，而不是控制器

### 3.4 UI 目标、数据目标、验证目标没有被一张矩阵绑死

现象：

- 一轮强调 UI
- 一轮强调样本双跑
- 一轮强调启动器
- 一轮强调归档

缺少的东西：

- 页面 -> API -> 表 -> data_origin -> 最小可用数据 -> 验收方式

没有这张矩阵，执行时就很容易再次分叉。

### 3.5 路径迁移本身消耗了大量执行注意力

`04b` / `04e` 都把“先迁 docs 路径”写成强步骤。

这类动作本身不是错，但副作用是：

- 执行注意力先被目录结构占用
- 真实主语、真实数据、真实验收反而被放后面

rebuild4 现在直接把资料收拢在 `rebuild4/docs`，就是在主动切断这类路径漂移。

## 4. rebuild4 要如何避免重蹈覆辙

### 4.1 只保留一份唯一主任务书

rebuild4 应只认一份最终执行文档，例如：

- `rebuild4/docs/03_final/01_rebuild4_最终执行任务书.md`

其余文件只作为：

- 输入
- 调查
- agent 草案
- 合并记录

不能再并行生效。

### 4.2 先做矩阵，再做实现

在任何开发前，必须先冻结一张矩阵：

- 页面
- API
- 来源表/汇总表
- `data_origin`
- 最小可用数据
- 是否允许 synthetic
- 是否允许 fallback
- 验收方式（含 Playwright）

### 4.3 Gate 必须变成单独停机线

每个 Gate 要有：

- 单独文档
- 单独通过条件
- 未通过不得继续

而不是只写在长 prompt 中。

### 4.4 三条线拆开，但统一收口

你已经明确的三条线是正确的：

1. 数据清洗整理规则（继承 rebuild2）
2. 流转处理流程（rebuild3 核心设计）
3. 操作 UI（兼顾前两者）

rebuild4 应把这三条线拆开各自出稿，再统一合并，而不是一份 prompt 同时混做。

### 4.5 每一步都必须可校验

尤其必须写进任务书：

- SQL / 元数据校验
- API 响应校验
- 页面级 Playwright 校验
- data_origin 是否被前端消费
- 页面主语是否与 API 主语一致

### 4.6 不再把“理解了 UI”当作“实现了 UI”

这是 `04f` 给出的最关键教训之一。

rebuild4 中必须强制：

- UI_v2 原型文件要有逐页映射
- 页面实现要有逐页差异说明
- 没有逐页比对，就不算通过

---

## 5. 一句话结论

rebuild3 的 04 系列不是因为文档不够，而是因为多轮纠偏后没有收敛成唯一执行机制；rebuild4 必须把这些思想提炼后重写成一份唯一主任务书，并用矩阵和 Gate 真正卡住执行路径。
