# rebuild3 预实施任务书（最终冻结版）

> 状态：最终冻结版（UI 设计前置）  
> 适用对象：人类评审、UI 设计协作者、后续实施 agent  
> 重要说明：这不是最终实施任务书。当前阶段先冻结业务边界、数据结构边界、读模型边界和验证路径；UI 方案确认后，再生成最终实施任务书并进入编码。

---

## 1. 本文档的用途

这份文档服务于三个目标：

1. 把 rebuild3 在实现前必须一致的业务口径冻结下来
2. 给 UI 设计提供稳定的页面主语、读模型和交互边界
3. 为 UI 确认后的最终实施任务书提供结构化输入

因此，当前阶段的任务目标应理解为：

- 冻结规则
- 冻结结构
- 冻结验证方式
- 准备 UI 设计输入

而不是：

- 直接做全量 SQL 实现
- 直接重写前后端
- 直接开始最终开发

---

## 2. 当前阶段总边界

### 2.1 这阶段必须完成

1. 统一 rebuild3 的术语、主语和状态表达
2. 冻结对象注册、事实分层、版本绑定、幂等键的最小方案
3. 输出 UI 设计需要的页面清单、页面读模型、关键状态说明
4. 冻结本地验证路径和对比指标
5. 明确哪些项是实现前阻塞，哪些项可以在实施过程中参数化迭代

### 2.2 这阶段明确不做

1. 不做最终页面开发
2. 不做全量数据库迁移和历史回灌执行
3. 不做最终 API 全量实现
4. 不做云端化设计和复杂调度系统
5. 不在 UI 设计前直接进入 Vue 页面实现
6. 不在 UI 设计前冻结最终接口细节与缓存策略细节

这 5 项在最终实施计划里必须被正式写入，但当前阶段已经冻结了方向与表达方式。

| 编号 | 阻塞项 | 当前冻结方式 | 备注 |
|---|---|---|---|
| D1 | `event_time` 口径 | 默认以 `ts_std` 作为系统事件时间；保留原始时间证据列 | UI 和规划阶段可直接使用 |
| D2 | 源适配边界 | 冻结为双模式接口：原始 27 列为标准契约，rebuild2 标准化结果可作为验证/回放快路径 | 不等于长期跳过标准化模块 |

当前阶段建议拆成 5 个部分：前 4 个部分属于“现在要完成的冻结工作”，第 5 部分属于“UI 确认后再展开”。

## Phase A：冻结业务与状态口径

目标：先把“系统在说什么”统一掉，避免 UI、实现和评审各自理解为不同系统。

### A1. 冻结对象主语

**输出**：

- Cell / BS / LAC 职责说明
- 冷启动与长期治理主语边界说明

**冻结结果**：

- Cell = 最小治理主语
- BS = 空间锚点
- LAC = 区域边界与区域健康主语
- 初始化可用 LAC 预边界，不改变长期主语语义

**验收标准**：

- 后续所有文档不再出现对象职责互相冲突
- UI 页面不再围绕旧 Step，而是围绕对象 / 决策 / 基线组织

### A2. 冻结二维状态模型

**输出**：

- `lifecycle_state` 枚举
- `health_state` 枚举
- UI 派生状态说明

**冻结结果**：

- `lifecycle_state = waiting / observing / active / dormant / retired / rejected`
- `health_state = healthy / insufficient / gps_bias / collision_suspect / collision_confirmed / dynamic / migration_suspect`
- `watch` 仅作为 UI 派生展示态，不作为持久化枚举

**验收标准**：

- 不再使用单一大状态枚举同时表达生命周期和健康度
- UI 设计能稳定表达 active 但异常 / active 但禁锚等情况

### A3. 冻结三类资格

**输出**：

- 存在资格
- 锚点资格
- baseline 资格矩阵

**冻结结果**：

系统中的每个对象、每类异常都必须能回答：

1. 能否被系统承认存在
2. 能否做 GPS / 信号锚点
3. 能否参与 baseline / profile

**验收标准**：

- 不再出现“对象存在就默认能做锚点”的隐式逻辑
- UI 页面对同一个对象能够同时展示存在状态、健康状态、锚点状态、baseline 状态

### A4. 冻结 LAC / BS 的“注册”和“激活”区别

**输出**：

- BS 注册与激活说明
- LAC 注册与激活说明

**冻结结果**：

- **注册**：对象在注册表中有了合法存在记录
- **激活**：对象满足当前可用于治理 / 汇总 / 展示的活动条件
- 对象是否可做锚点，不等价于是否已注册

**验收标准**：

- 不再出现“BS ≥ 1 个 trusted Cell 就等于长期可锚定”的歧义
- UI 设计可以区分“已存在但未激活”和“已激活但禁锚”

### A5. 冻结研究期口径与长期口径边界

**输出**：

- 北京样本研究期特殊规则说明
- 长期运行口径说明

**冻结结果**：

- 北京 GPS 边界 / 研究期 LAC 预过滤 = 研究期策略
- 长期运行 = 参数化、轻量过滤、依赖等待/观察/异常分流机制

**验收标准**：

- 后续实施计划不会把研究期规则硬编码成长期主规则

## Phase B：冻结数据结构与版本口径

目标：把后续 UI、API、SQL、回放和验证都依赖的最小骨架确定下来。

### B1. 冻结 schema 边界

**冻结结果**：

- `rebuild3`：对象、事实、画像、异常、热层/长层/归档层
- `rebuild3_meta`：run、batch、contract、rule、baseline、source_adapter 等控制元数据
- `rebuild2 / rebuild2_meta / legacy`：只读

**验收标准**：

- 域数据与元数据职责明确
- 不再出现把运行控制字段散落到业务明细表中的设计

### B2. 冻结对象注册快照与历史模型

**必备对象快照表**：

- `obj_cell`
- `obj_bs`
- `obj_lac`

**必备历史表**：

- `obj_state_history`
- `obj_relation_history`

**对象快照表至少要支持**：

- 业务主键
- `lifecycle_state`
- `health_state`
- `anchorable`
- `baseline_eligible`
- 当前有效关系
- 最近活跃时间
- 版本归属
- 关键统计摘要（样本数、设备数、质心/中心点、活跃天数等）

**历史表至少要支持**：

- 状态变更留痕
- 父子关系变化留痕
- 迁移 / 分裂 / 退役回放

**验收标准**：

- UI 可以展示当前快照
- 审查页可以展示状态时间线
- 后续可支持迁移 / 分裂关系回放

### B3. 冻结事实分层

**冻结结果**：

1. `fact_standardized`：不可变标准事件层
2. `fact_governed`
3. `fact_pending_observation`
4. `fact_pending_issue`
5. `fact_rejected`

**必须满足**：

- 同一标准事件只能有一个主归宿事实层
- `fact_governed` 中必须显式记录：
  - `gps_source`
  - `signal_source`
  - `baseline_eligible`
  - `anomaly_tags`
- `fact_pending_observation` 用于等待 / 观察
- `fact_pending_issue` 用于对象级问题 / 异常复核
- `fact_rejected` 用于结构不合规和明确拒收

**验收标准**：

- 页面和后端都能明确回答“这条事件去哪一层”
- 不再靠口头描述区分 waiting / issue / reject

### B4. 冻结热层 / 长层 / 归档层

**冻结结果**：

- 热层：`cell_detail_hot` 或等价 per-Cell 热明细层
- 长层：日级或更粗粒度的对象活跃与成熟度汇总层
- 归档层：完整治理后历史

**验收标准**：

- 不再把 per-Cell 热明细误当成长期主存储
- UI 和读模型能够区分“近期窗口”和“长期趋势”

### B5. 冻结版本、批次与幂等口径

**冻结结果**：

- 每个对象、事实、画像都能追溯：
  - `run_id`
  - `contract_version`
  - `rule_set_version`
  - `baseline_version`
- `run_id` 与 `batch_id` 分离
- 批次显式记录时间窗口与来源
- 标准事件幂等键显式落库，不只依赖原始 `record_id`

**验收标准**：

- 2 小时批次可以重复回放、去重、比对
- 后续 UI 可展示批次 / run / baseline 绑定关系

## Phase C：给 UI 设计准备稳定输入

目标：让 UI 设计围绕真实对象与真实决策，不再围绕旧 Step 页面做视觉微调。

### C1. 冻结页面主导航清单

当前建议的页面主语如下：

| 页面 | 主语 | 目标 |
|---|---|---|
| 运行中心 | run / batch / baseline | 看运行、触发动作、查看版本绑定 |
| 对象浏览 | Cell / BS / LAC | 看当前对象状态、资格、画像、关系 |
| 对象详情抽屉/页 | 单对象 | 看状态历史、关系历史、样本摘要、异常与资格 |
| 日更审查 / 批次决策 | batch / decision | 看四分流、晋升/降级、异常时间线 |
| 等待/观察池 | candidate / observation | 看累计进度、晋升缺口、拒收/转问题路径 |
| 异常看板 | issue / anomaly | 看 collision / dynamic / migration 等问题对象 |
| baseline / 画像 | baseline / profile | 看当前基线版本、画像覆盖、可信度与刷新来源 |
| 回放对比 | replay / convergence | 看 3+4 vs 7 天收敛差异 |

**验收标准**：

- 页面清单覆盖初始化、增量、异常、基线、验证五类场景
- 页面不再围绕 old Step 导航

### C2. 冻结页面读模型草案

#### 1）运行中心读模型

至少需要：

- `run_id`
- `run_type`
- `status`
- `window_start / window_end`
- `contract_version`
- `rule_set_version`
- `baseline_version`
- `input_rows / output_rows`
- 各步骤耗时与统计
- 关联批次列表
- 当前 baseline 摘要

#### 2）对象列表读模型

至少需要：

- 对象主键
- `lifecycle_state`
- `health_state`
- `anchorable`
- `baseline_eligible`
- 最近活跃时间
- 样本量 / 设备数 / 活跃天数
- 父对象 / 子对象数量
- 异常标签
- GPS / 信号可信度摘要

#### 3）对象详情读模型

至少需要：

- 基本信息与当前快照
- 画像摘要
- 状态历史时间线
- 关系历史时间线
- 最近批次中的事实分布
- 锚点与 baseline 资格原因
- 相关异常事件
- 关键位置/信号统计摘要

#### 4）等待/观察池读模型

至少需要：

- Cell 业务主键
- 首次出现时间
- 当前累计 GPS 点数
- 当前设备数
- 活跃天数
- 当前质心 / 半径
- 距离存在资格 / 锚点资格 / baseline 资格的差距
- 最近批次推进情况
- 当前建议动作（继续观察 / 晋升 / 转问题 / 拒收）

#### 5）异常看板读模型

至少需要：

- `anomaly_type`
- 对象主键
- 当前 `health_state`
- 发现批次
- 严重级别
- 锚点状态
- baseline 状态
- 最近一次证据更新时间
- 建议动作
- 是否影响下游对象

#### 6）baseline / 画像读模型

至少需要：

- `baseline_version`
- 绑定的 `rule_set_version`
- 覆盖对象数量
- 各对象等级分布
- GPS / 信号可信度分布
- 异常占比
- 刷新触发原因
- 与上一版 baseline 的差异摘要

#### 7）回放对比读模型

至少需要：

- 对比任务标识
- 两个运行结果版本
- 对象收敛指标
- 空间收敛指标
- 信号收敛指标
- 异常收敛指标
- 决策收敛指标
- 主要差异对象清单

**验收标准**：

- UI 设计不需要直接阅读底层大表
- 页面都能基于清晰读模型设计

### C3. 冻结关键交互状态

UI 设计必须能表达下面这些状态，而不只是展示统计数字：

- 空状态
- 运行中状态
- 批次冻结中状态
- 等待态
- 观察态
- 已 active 但异常态
- 锚点禁用态
- baseline 禁用态
- 回放对比态
- 异常待复核态
- 拒收态

**验收标准**：

- UI 方案必须显式设计这些状态的视觉表达与文案

### C4. 冻结样本场景集

UI 设计阶段至少要覆盖下面 8 类样本场景：

1. 正常 active Cell / BS / LAC
2. 待晋升 Cell（证据接近门槛）
3. `collision_suspect` BS（禁锚）
4. `collision_confirmed` BS（进入问题事实）
5. `dynamic` 对象（低精度、禁 baseline）
6. `migration_suspect` 对象（有关系历史变化）
7. `fact_rejected` 记录（结构不合规）
8. baseline 刷新前后差异对象

**验收标准**：

- UI 原型必须能承载这些真实场景
- 不能只用“正常 happy path”设计页面

### C5. 冻结页面与系统主语的映射关系

| 页面 | 围绕什么组织 |
|---|---|
| 运行中心 | 运行 / 批次 / 版本 |
| 对象浏览 | 对象 |
| 日更审查 | 决策 / 事实 |
| 等待池 | 候选对象 / 观察事实 |
| 异常看板 | 问题对象 / 问题事实 |
| baseline / 画像 | 基线 / 画像 |
| 回放对比 | 结果差异 / 验证 |

**验收标准**：

- 页面设计不会退回到旧 Step 导航心智

### C6. 冻结前端组件边界草案

推荐至少拆成下面 4 类组件：

1. **页面级组件**  
   运行中心、对象浏览、等待池、异常看板、baseline 视图、回放对比

2. **共享展示组件**  
   状态徽标、资格徽标、对象卡片、时间线、指标卡、表格、分页、空状态、错误状态

3. **详情面板 / 抽屉组件**  
   Cell 详情、BS 详情、LAC 详情、异常事件详情、baseline 详情

4. **筛选与上下文组件**  
   对象筛选器、批次切换器、baseline 版本切换器、对比条件筛选器

**验收标准**：

- UI 设计能自然映射到 Vue 组件边界
- 避免重新出现超大单文件页面

## Phase D：冻结验证与回放方案

目标：让后续实施不是“写完再看”，而是从第一天起就服务于收敛验证。

### D1. 冻结验证路径

固定为：

1. 第 1~3 天初始化起跑
2. 第 4~7 天按 2 小时批次增量滚动
3. 对比直接 7 天初始化结果

### D2. 冻结对比指标

至少包括：

- 对象收敛指标
- 空间收敛指标
- 信号收敛指标
- 异常收敛指标
- 决策收敛指标
- 热层稳定性指标

### D3. 冻结执行顺序原则

后续实施必须遵守：

1. 先样本后全量
2. 先回放可追溯性，再追求性能
3. 先冻结读模型，再定最终 API
4. 先 UI 确认，再生成最终实施任务书

## Phase E：UI 确认后再做的事

目标：在 UI 稳定后，把当前预实施任务书升级成真正可执行的最终实施任务书。

### E1. 需要在 UI 确认后补齐的内容

1. 数据库 DDL 与索引任务拆解
2. run / batch / baseline / source 元数据任务拆解
3. 标准化、初始化、增量、异常、画像模块任务拆解
4. API 与缓存读模型任务拆解
5. 前端页面与组件任务拆解
6. 验证、回归与性能基准任务拆解
7. 依赖顺序、验收标准和回归标准

### E2. UI 确认后的推荐实施模块（供后续规划参考）

> 注意：以下是推荐模块边界，不是当前阶段直接执行的 backlog。

| 模块 | 主要内容 |
|---|---|
| M0 元数据与 schema | `rebuild3 / rebuild3_meta`、版本与批次注册、幂等键 |
| M1 标准化事件层 | 源适配、标准化、解析错误留痕 |
| M2 初始化对象链路 | Cell 晋升、BS / LAC 派生、首轮对象快照 |
| M3 事实治理 | GPS 修正、信号补齐、资格门控 |
| M4 异常与问题链路 | collision / dynamic / migration / gps_bias 检测与分流 |
| M5 增量链路 | 四分流、等待池、观察池、级联更新 |
| M6 画像与 baseline | 三级画像、baseline 冻结、增量刷新 |
| M7 API / 读模型 | 运行、对象、异常、baseline、回放对比接口 |
| M8 前端 SPA | Vue 3 + TS 页面与组件实现 |
| M9 回放验证 | 3+4 vs 7 天对比、误差解释、收敛评估 |

## 6. 当前建议 agent 先做的实际工作

如果后续让 agent 在“当前阶段”继续工作，只允许它优先做下面这些任务：

1. 整理并冻结状态机、资格矩阵、异常分流矩阵
2. 起草对象注册表、事实层、基线层、版本层的数据结构文档
3. 输出 UI 设计需要的页面清单、页面读模型、关键状态说明
4. 准备“3 天起跑，滚到 7 天，对比直接 7 天”的验证方案和指标说明
5. 对两套旧稿中仍有差异的表述继续做去歧义整理

## 7. 当前阶段禁止 agent 直接做的事情

为了避免过早进入错误实现，当前明确禁止 agent 直接推进下面这些工作：

1. 直接实现全量数据库建表和全量历史回放
2. 在没有 UI 方案和组件边界的情况下直接开始写 Vue 页面
3. 直接把 rebuild2 的静态可信库页面硬改成最终运营页面
4. 未冻结最终实施任务书前开始大规模 API 落地
5. 未将异常资格矩阵写成明确规则前开始写 GPS 修正和信号补齐门控逻辑
6. 未完成 UI 设计前开始写最终页面和最终 API

## 8. 本阶段的验收标准

这份“预实施任务书”是否完成，不看代码量，看下面 6 件事是否已经清楚：

1. 人和 agent 对对象主语、状态机、资格矩阵说法一致
2. UI 设计已经拿到稳定输入，不需要再反复追问业务对象含义
3. 数据结构层已经能回答：
   - 事件写去哪一层
   - 对象状态写去哪一层
   - 关系历史写去哪一层
4. 回放验证路径已经明确，不再停留在抽象口号
5. 当前阶段的非目标已经明确，不会提前滑入最终开发
6. 审阅通过后，能自然进入“先做 UI 设计”的下一阶段

## 9. UI 设计完成后，最终实施任务书必须包含什么

UI 设计完成后，最终实施任务书至少要补齐：

4. 回放验证与指标任务拆解
5. 开发顺序与依赖关系
6. 每项任务的验收标准和回归标准
