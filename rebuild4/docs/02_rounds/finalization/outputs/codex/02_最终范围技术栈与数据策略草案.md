## 2. 最终技术栈与基础框架约束

### 2.1 冻结技术栈

| 层级 | 冻结选择 | 约束说明 |
|---|---|---|
| 数据库 | PostgreSQL 17 | 主计算、主存储、主校验环境；所有计数以 PG17 为准 |
| 后端 | Python 3.11+ + FastAPI | 负责 run/batch 编排、读模型 API、Gate 执行输出 |
| 查询风格 | SQL-first | 领域真相、聚合、校验以 SQL / 参数化查询为主，不以 ORM 领域模型为主语 |
| 前端 | 全新 `Vue 3 + TypeScript + Vite` 项目 | 承接 D-002；不 fork rebuild3 前端骨架 |
| 前端共享状态 | Pinia（仅 UI 共享状态） | 只承接筛选条件、抽屉状态、上下文等共享 UI 状态，不承接来源语义判断 |
| 页面验收 | Playwright | finalization 不重跑页面，但 `03_final/04_最终校验清单.md` 仍以 Playwright 为正式页面验收工具 |
| 数据库事实复核 | `PG17 MCP execute_sql` | 所有计数、比例、差集、缺口统一以 execute_sql 为准 |
| 结构复核 | `PG17 MCP search_objects` | 只用于 schema / table / column / index 等结构确认 |

### 2.2 明确禁止的技术方向

以下技术方向未被输入与 merged 文档授权，不进入本轮范围：

1. Kafka / Redis / ClickHouse / Spark / 云调度系统。
2. 以 `rebuild3` 前端旧路由、旧 store、旧 axios 拦截层为基础继续改造。
3. 以 sample / synthetic / fallback 结果顶替主流程 real 数据。
4. 用 shell 输出或页面文案替代数据库事实。

### 2.3 前端框架的复用边界（D-002 回写版）

1. **允许复用**：通过审计的纯渲染组件，如 badge、delta indicator、无取数逻辑的展示组件。
2. **禁止复用**：旧路由、旧 Pinia 结构、旧 axios 层、任何包含 `fallback` / `synthetic` 判断逻辑的代码。
3. **来源合同归属**：`data_origin / origin_detail / subject_scope / subject_note` 由 API 顶层 envelope 提供，前端只消费，不重新发明。
4. **主流程空状态**：没有 real 数据时必须诚实空状态，而不是由前端拼 fake 卡片。

### 2.4 launcher 的最终落点

launcher 不能在 final 冻结时“因为不计入 Gate 就消失”。其正确落点应冻结为：

| 主题 | 冻结结论 |
|---|---|
| 身份 | 独立运维入口，不属于数据主链页面 |
| 是否计入本轮正式完成标准 | 否 |
| 是否保留技术约束 | 是 |
| 启动方式 | 独立 Python 脚本启动，不依赖主系统前后端 |
| 通信方式 | WebSocket：`/ws/launcher/status`、`/ws/launcher/logs/:service` |
| REST 接口 | `/api/launcher/services*` 系列只负责服务控制与日志，不进入数据合同 |
| 在 final 文档中的落点 | `01_最终技术栈与基础框架约束.md` 说明其框架位置；`05_本轮范围与降级说明.md` 说明其不进入 G0-G7 |

## 3. 数据生成与回灌策略

### 3.1 总原则

1. **主链业务结果只生成，不回灌。**
   - `fact_*`、`obj_*`、`baseline_*`、`profile_*` 必须由 rebuild4 自己生成。
   - 严禁从 `rebuild3*` 直接回灌为 rebuild4 current。
2. **治理解释层允许一次性定向回灌。**
   - `field_audit_snapshot`、`target_field_snapshot`、`ods_rule_snapshot`、`ods_execution_snapshot`、asset 注册与规则 seed 可以定向进入 `rebuild4_meta`。
3. **compare 的 synthetic / fallback 只存在于辅助验证链。**
   - 不进入主流程、不回写主链、不充当完成证据。

### 3.2 输入源冻结

| 场景 | 冻结输入 | 说明 |
|---|---|---|
| real `full_initialization` | `rebuild2.l0_gps` + `rebuild2.l0_lac` | 唯一正式双输入 |
| real `rolling` | 真实窗口化输入 | 具体窗口切分由实施决定，但必须满足 `data_origin=real` 与 `completed batch > 1` |
| governance 快照 | `rebuild2_meta.*` + 数据库 introspection | 只读参考源回灌到 `rebuild4_meta` |
| legacy 参考信息 | `rebuild2` / `rebuild3` 旧质量字段 | 只作为 `legacy_ref` 解释层返回，不进入主状态 |

### 3.3 G1：治理快照与规则 seed 回灌

这一阶段允许、且必须发生的回灌只有以下几类：

1. `field_audit_snapshot <- rebuild2_meta.field_audit`（应为 27 行）
2. `target_field_snapshot <- rebuild2_meta.target_field`（应为 55 行）
3. `ods_rule_snapshot <- active rebuild2_meta.ods_clean_rule`（应为 26 行）
4. `ods_execution_snapshot <- rebuild2_meta.ods_clean_result`（应保留 24 个 distinct `rule_code`）
5. `asset_table_catalog` / `asset_field_catalog` <- 数据库 introspection 生成真实基数
6. `asset_usage_map` / `asset_migration_decision` <- 在真实基数之上人工补注语义
7. `parse_rule` / `compliance_rule` <- 从文档 / SQL / Python 提取 seed 写入 `rebuild4_meta`

**冻结限制**：

- 不允许把“人工补录”当唯一正道；人工只能补语义，不能制造基数事实。
- 不允许回写 `rebuild2_meta` 源表。
- `parse_rule` 与 `compliance_rule` 必须非空后才可通过 `G1`。

### 3.4 G2：real `full_initialization` 生成策略

冻结策略如下：

1. `run_type = full_initialization`
2. `data_origin = real`
3. 1 个 run 只允许 1 个 logical batch（`batch_seq = 1`）
4. 初始化页固定承接 11 步语义：
   - 冻结 `run_id` 与版本绑定
   - 全量标准化事件
   - 记录级合规检查
   - 研究期 LAC 预边界/区域缩边界
   - Cell 候选累计与晋升
   - 由 active Cell 派生 BS
   - 由 active BS 派生 LAC
   - 首轮事实治理
   - 异常检测与资格收敛
   - 完整回归一次
   - 生成 baseline v1
5. 本阶段必须同时产出：
   - `fact_standardized`
   - 四分流事实表
   - `obj_cell / obj_bs / obj_lac`
   - `batch_snapshot`
   - `batch_flow_summary`
   - `batch_transition_summary`
   - `batch_anomaly_*`
   - `baseline_version` v1 + `baseline_*`
   - 首个 `trusted_loss_summary` + `trusted_loss_breakdown`

### 3.5 G3：real `rolling` 与 `batch_snapshot` 策略

1. `run_type = rolling`
2. 至少要形成 1 条 completed real rolling run
3. completed batch 必须 `> 1`
4. 每个 completed batch 恰好写入 11 个 snapshot metric：
   - `fact_standardized`
   - `fact_governed`
   - `fact_pending_observation`
   - `fact_pending_issue`
   - `fact_rejected`
   - `obj_cell`
   - `obj_bs`
   - `obj_lac`
   - `baseline_cell`
   - `baseline_bs`
   - `baseline_lac`
5. 每个 completed batch 都要诚实写入 `baseline_refresh_log`，包括“未刷新”的原因
6. `flow-snapshot/timepoints` 只允许 real completed batch，不再复用 `rebuild3_meta.v_flow_snapshot_timepoints`

### 3.6 baseline 版本链与 diff 策略

| 主题 | 冻结结论 |
|---|---|
| 存储方式 | `baseline_*` 按 `baseline_version` 追加存储 |
| current 读取方式 | 允许提供 `baseline_current_*` view，但正式历史必须来自实体表 |
| diff 结构 | `baseline_diff_summary` + `baseline_diff_object` |
| 仅 1 个 baseline 时 | `previous_available=false`；`subject_note=暂无上一版 real baseline 可比` |
| 禁止事项 | 不允许拿 rebuild2 / rebuild3 / synthetic baseline 冒充上一版 |

### 3.7 trusted loss 指标命名冻结

为避免 final 文档继续使用含糊口径，建议在 `trusted_loss_summary` 中把指标写成：

| 指标名 | 定义 | 当前 PG17 基线 |
|---|---|---:|
| `total_rows` | `l0_lac` 总量 | 43,771,306 |
| `trusted_rows` | trusted 命中量 | 30,082,381 |
| `filtered_rows` | trusted 未命中量 | 13,688,925 |
| `filtered_pct` | `filtered_rows / total_rows` | 31.27% |
| `filtered_with_rsrp` | 被过滤且 `RSRP` 非空 | 12,017,352 |
| `filtered_with_lon_lat` | 被过滤且经纬同时非空 | 11,350,552 |

并在 `trusted_loss_breakdown` 至少冻结两类维度：

1. `tech`
2. `source_combo`

### 3.8 compare 的降级与不回灌原则

1. compare 只负责验证 / 对照，不负责产出主链 current。
2. compare 可返回：
   - `real`
   - `synthetic`
   - `fallback`（仅空 envelope）
3. `synthetic` 必须页面 banner 明示“仅供辅助验证”。
4. `fallback` 时 `data = null` 或空列表，不允许伪造差异对象结果。
5. compare 的任何结果都不回写 `fact_*` / `obj_*` / `baseline_*`。

## 5. 残留缺口

以下缺口仍存在，但不构成当前必须新增人类裁决：

1. `parse_rule` / `compliance_rule` 目前在 PG17 仍为 0 行，是否能在 G1 形成非空 seed 取决于实施。
2. 当前 PG17 仍不存在 real rolling run，G3 能否通过取决于真实输入窗口能否形成。
3. launcher 的技术边界已足够明确，但其实现不应阻塞主数据链冻结。

我的建议是：**直接进入 finalization merged 与 `03_final/` 正式文件生成，不新增方向性裁决。**
