## 4. 必须在 finalization 直接回写的收口项

### FZ-001：`03_final/` 文件命名仍有历史漂移

**问题**  
round1 与 round3 多处仍把最终执行入口写成：

- `rebuild4/docs/03_final/01_rebuild4_最终执行任务书.md`
- `rebuild4/docs/03_final/02_rebuild4_最终校验清单.md`

但 runbook 与 finalization `92_taskbook_generation_prompt.md` 已冻结最终文件集合为 `00_05` 六份文件，其中任务书与清单应为：

- `rebuild4/docs/03_final/03_最终执行任务书.md`
- `rebuild4/docs/03_final/04_最终校验清单.md`

**冻结结论**  
这不是新的方向分歧，而是 finalization 必须直接修正的命名回写。所有 `G0/WP0/C-G0-01/最终冻结清单` 中的旧路径都应改为六文件冻结集。

### FZ-002：`contract_version` 的锚点需要从“两份文档”升级为“冻结包 manifest”

**问题**  
round3 `WP0` / `G0` 仍按“两份文档 hash”表述 `contract_version`，但 finalization 最终冻结集是 6 份文件；同时 round2 对 `contract_version` 的建模只有一个 `is_current=true` 当前版本。

**冻结结论**  
`contract_version.doc_path` 应指向 `rebuild4/docs/03_final/00_最终冻结基线.md`；`content_hash` 应表示 `03_final/00_05` 冻结包 manifest 的组合 hash，而不是只覆盖两份任务书/清单文件。这样才能与 finalization 的六文件冻结结构一致。

### FZ-003：trusted 损耗指标中的“有 GPS”口径仍然含糊

**问题**  
`PG17` 复核显示：

- 被过滤且经纬同时非空 = `11,350,552`
- 被过滤且 `GPS有效 = true` = `10,982,466`

前三轮 merged 反复引用的是 `11,350,552`，但文案写的是“有 GPS”，这会把两个不同口径混在一起。

**冻结结论**  
final 文档中应把该指标固定写成：**“被过滤且经纬同时非空”**，并在 `trusted_loss_summary` / `03_数据生成与回灌策略.md` 中用同一命名回写，避免把模糊文案继续带入正式任务书。

### FZ-004：启动器的落点需要显式写进 final 文档，而不是只在 merged 中被动排除

**问题**  
UI_v2 明确把启动器放在“日常使用主链路”的第一步，且 `launcher_doc.md` 已冻结其独立 Python 脚本、REST 与 WebSocket 需求；但 round2/round3 主要以“它不纳入数据合同与 Gate 成功标准”表述，容易在最终冻结时被忽略。

**冻结结论**  
launcher 应在 final 文档中被明确写成：

- **基础框架中的独立运维入口**；
- **不纳入 G0-G7 与数据合同成功标准**；
- **但保留独立 Python 脚本 + `/api/launcher/*` + `/ws/launcher/*` 约束**；
- **落点为 `01_最终技术栈与基础框架约束.md` + `05_本轮范围与降级说明.md`**。

### FZ-005：数据生成与回灌策略仍散落在 WP2-WP4 / G1-G3 中，需汇总成单独 final 文档

**问题**  
round2/round3 已分别冻结 governance 快照、规则 seed、双输入 initialization、rolling、snapshot、baseline refresh、trusted loss 等策略，但当前仍散落在多个工作包和 Gate 中。

**冻结结论**  
这些内容应在 finalization 汇总为单独的 `03_final/02_数据生成与回灌策略.md`，并明确：

1. 哪些是一次性回灌到 `rebuild4_meta` 的治理快照；
2. 哪些是每次 run/batch 生成的正式业务结果；
3. 哪些结果严禁从 `rebuild3*` 回灌冒充 current；
4. compare 的 synthetic/fallback 只存在于辅助验证链，不回灌主链。

## 5. 最终冻结口径

### 5.1 finalization 之后应固定不再变化的内容

1. **唯一冻结包**：`03_final/00_05` 六份文件共同生效。
2. **唯一数据库事实基线**：所有计数、差集、损耗、缺口继续以 `PG17 MCP execute_sql` 为准；结构确认用 `PG17 MCP search_objects`。
3. **唯一页面证据基线**：只复用 round3 已归档截图，不重新跑页面。
4. **唯一裁决集**：只承认 round1 `D-001 ~ D-003` 与 round2/round3“无新增裁决”的结论。
5. **唯一主链来源**：rebuild4 正式主链只接受 `rebuild2.l0_gps + rebuild2.l0_lac` 重跑生成的 real 数据；`rebuild3*` 永远只读参考。

### 5.2 当前仍保留的执行风险

以下是**执行风险**，不是当前必须新增的人类裁决：

1. `parse_rule` / `compliance_rule` 仍为 0 行，真正落地仍取决于 G1 是否能产出非空 seed。
2. real rolling 输入窗口当前仍不存在，G3 能否通过取决于实施阶段是否取得可用窗口。
3. governance 与 trusted loss 的表结构方案已冻结，但仍需真正落库并通过 G5。

### 5.3 本稿建议

**本轮不新增人类裁决文件。** finalization 应直接把以上 5 个收口项写回 merged 与 `03_final/`，再进入最终正式文件生成。
