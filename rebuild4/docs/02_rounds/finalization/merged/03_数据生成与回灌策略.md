# 数据生成与回灌策略

状态：finalization merged 冻结稿  
更新时间：2026-04-06  
输入范围：前三轮 merged、全部 decisions、`PG17`（2026-04-06 复核）。

---

## 1. 总策略

本轮冻结的数据生成与回灌策略只有一条主线：

1. **主链业务结果只允许由 rebuild4 自产。**
2. **治理解释层允许一次性定向回灌到 `rebuild4_meta`。**
3. **compare 的 synthetic / fallback 只服务辅助验证，不回写主链。**
4. **`rebuild3*` 永远只读参考，不得冒充 rebuild4 current。**

---

## 2. 输入源与目标表冻结

| 场景 | 冻结输入 | 目标位置 | 约束 |
|---|---|---|---|
| real `full_initialization` | `rebuild2.l0_gps` + `rebuild2.l0_lac` | `rebuild4` + `rebuild4_meta` | 唯一正式双输入 |
| real `rolling` | 真实窗口化输入 | `rebuild4` + `rebuild4_meta` | 必须满足 `data_origin=real` 且 `completed batch > 1` |
| governance 快照 | `rebuild2_meta.*` + 数据库 introspection | `rebuild4_meta` | 允许一次性快照与 seed 回灌 |
| legacy 参考信息 | `rebuild2` / `rebuild3` 旧质量字段 | API `legacy_ref` | 只作参考，不进入主状态 |
| compare | `compare_job` + `compare_result` | `rebuild4_meta` | 只服务辅助验证，不回写主链业务表 |

## 3. G1：治理快照与规则 seed 回灌

- `NULL_WIFI_NAME_INVALID`

它们必须在最终治理能力里继续以“定义存在、执行统计缺失”的事实出现，不能通过裁剪定义层来“消灭差集”。

## 4. G2：real `full_initialization` 生成策略

### 4.1 对外合同

| 项 | 冻结值 |
|---|---|
| `run_type` | `full_initialization` |
| `data_origin` | `real` |
| 批次语义 | 1 个 logical run + 1 个 logical batch（`batch_seq = 1`） |
| 守恒约束 | `fact_standardized = fact_governed + fact_pending_observation + fact_pending_issue + fact_rejected` |
| baseline | 首次 real baseline 生成 `v1`，并登记为 current |

### 4.2 初始化页面固定承接的 11 步语义

1. 冻结 `run_id` 与版本绑定。  
2. 全量标准化事件。  
3. 记录级合规检查。  
4. 研究期 LAC 预边界 / 区域缩边界。  
5. Cell 候选累计与晋升。  
6. 由 active Cell 派生 BS。  
7. 由 active BS 派生 LAC。  
8. 首轮事实治理。  
9. 异常检测与资格收敛。  
10. 完整回归一次。  
11. 生成 baseline `v1`。  

### 4.3 必须产出的正式结果

- `fact_standardized`
- 四分流事实表
- `obj_cell / obj_bs / obj_lac`
- `batch_snapshot`
- `batch_flow_summary`
- `batch_transition_summary`
- `batch_anomaly_*`
- `baseline_version` `v1` + `baseline_*`
- 首个 `trusted_loss_summary` + `trusted_loss_breakdown`

## 5. G3：real `rolling` 与 `batch_snapshot` 策略

### 5.1 rolling 冻结规则

| 项 | 冻结值 |
|---|---|
| `run_type` | `rolling` |
| `data_origin` | `real` |
| 完成标准 | 至少 1 条 completed rolling run，且 completed batch `> 1` |
| timepoints | 只允许 real completed batch，不得复用 `scenario_replay` |
| baseline 时序 | 当前批只读取上一版冻结 baseline；新版本仅供后续批次使用 |

### 5.2 每个 completed batch 的固定快照形状

每个 completed batch 都必须恰好写入 11 个 metric，覆盖：

- `fact_standardized`
- `fact_governed`
- `fact_pending_observation`
- `fact_pending_issue`
- `fact_rejected`
- `obj_cell`
- `obj_bs`
- `obj_lac`
- `baseline_cell`
- `baseline_bs`
- `baseline_lac`

并继续冻结四个 stage：`input / routing / objects / baseline`。

### 5.3 rolling 必须同步产出的控制层结果

- `batch_flow_summary`
- `batch_transition_summary`
- `batch_anomaly_*`
- `baseline_refresh_log`
- `trusted_loss_summary` / `trusted_loss_breakdown` 的新快照

## 6. baseline 版本链与 diff 策略

| 主题 | 冻结结论 |
|---|---|
| 存储方式 | `baseline_*` 按 `baseline_version` 追加存储 |
| current 读取方式 | 可提供 `baseline_current_*` view，但正式历史必须来自实体表 |
| diff 结构 | `baseline_diff_summary` + `baseline_diff_object` |
| 仅 1 个 baseline 时 | `previous_available=false`；`subject_note=暂无上一版 real baseline 可比` |
| 明确禁止 | 不允许拿 rebuild2 / rebuild3 / synthetic baseline 冒充上一版 |

## 7. trusted loss 指标与命名冻结

### 7.1 正式指标命名

为消除“有 GPS”的歧义，final 文档与正式任务书统一使用以下命名：

| 指标名 | 定义 | 当前 PG17 基线 |
|---|---|---:|
| `total_rows` | `l0_lac` 总量 | 43,771,306 |
| `trusted_rows` | trusted 命中量 | 30,082,381 |
| `filtered_rows` | trusted 未命中量 | 13,688,925 |
| `filtered_pct` | `filtered_rows / total_rows` | 31.27% |
| `filtered_with_rsrp` | 被过滤且 `RSRP` 非空 | 12,017,352 |
| `filtered_with_lon_lat` | 被过滤且经纬同时非空 | 11,350,552 |

**历史稿中的 `filtered_with_gps` 视为废弃歧义写法。** 从本稿开始，正式任务书、正式校验清单与正式表设计统一使用 `filtered_with_lon_lat`。

### 7.2 breakdown 最小要求

`trusted_loss_breakdown` 在 `v1` 必须至少覆盖两类维度：

1. `tech`
2. `source_combo`

当前 PG17 冻结基线为：

- `tech`：`4G=10,004,716`、`5G=3,357,500`、`2G=233,772`、`3G=92,937`
- `source_combo`：`sdk / daa / cell_infos = 7,015,342`、`sdk / daa / ss1 = 5,951,068`、`sdk / dna / cell_infos = 722,515`

## 8. compare 的降级与不回灌原则

1. compare 只负责验证 / 对照，不负责生成主链 current。
2. compare 允许返回 `real`、`synthetic` 与 `fallback`。
3. `synthetic` 必须有 banner，明确“仅供辅助验证”。
4. `fallback` 只允许空 envelope，`data = null` 或空列表，不允许伪造差异对象结果。
5. compare 的任何结果都不得回写 `fact_*`、`obj_*`、`baseline_*`。

---

## 9. 本轮结论

1. rebuild4 的正式业务结果只有“生成”，没有从 rebuild3 主链“回灌 current”这条路径。
2. 真正允许回灌的只有治理快照、规则 seed 与 asset 注册等 `rebuild4_meta` 基础底座。
3. init、rolling、baseline、trusted loss 与 compare 的边界已经足够明确，可直接进入 `03_final/` 正式任务书编制。
