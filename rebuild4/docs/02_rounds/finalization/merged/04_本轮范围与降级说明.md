## 2. 明确降级范围

### 2.1 compare 降级边界

| 项 | 冻结边界 |

| 模块身份 | 辅助验证模块，不是日常主工作流 |
| 允许来源 | `real / synthetic / fallback` |
| 页面要求 | 首屏必须显示“辅助验证”身份；`synthetic` / `fallback` 必须出现 banner |
| 数据要求 | `fallback` 只允许空 envelope，不展示伪结果 |
| 完成定义 | `G7` 只验降级身份成立，不验 compare 成为主链正式能力 |

### 2.2 legacy 信息降级边界

| 项 | 冻结边界 |
|---|---|
| legacy 字段 | 只允许通过 API `legacy_ref` 嵌套返回 |
| 页面展示 | 只允许灰色参考区 / 详情参考区 |
| 明确禁止 | 不进入主状态模型、不进入主筛选条件、不参与 Gate 通过判断 |

## 3. 明确排除范围

以下内容不计入本轮正式完成标准：

1. **launcher / 运维健康检查**：保留为独立运维入口，但不进入 G0-G7 与统一数据合同成功标准。
2. **`rebuild3_sample_meta` 等历史兼容资产**：只作历史参考，不进入 rebuild4 主链。
3. **`rebuild2` / `rebuild3*` 的 current 化、转正化、回写行为**：全部禁止。
4. **旧前端整体 fork**：禁止复用 rebuild3 项目骨架、路由、Store、axios 与隐式 fallback 逻辑。
5. **`scenario_replay`、mock 数据、手动拼装结果**：禁止作为主链正式完成证据。

## 4. 阶段顺序与停机线

### 4.1 严格串行的阶段顺序

正式执行顺序冻结为：

`Phase 0 (G0) -> Phase 1 (G1) -> Phase 2A (G2) -> Phase 2B (G3) -> Phase 3 (G4 / G5 / G6) -> Phase 4 (G7) -> Phase 5 (总体验收)`

冻结含义：

1. 不允许跳过前置 Gate 提前展开后续阶段。
2. 不允许在 `G2` 之前启动 rolling。
3. 不允许在 `G4` 之前让主流程页面接正式 API。
4. 不允许在 `G5` 之前把 governance 计入正式完成。
5. 不允许在 `G7` 之前把 compare 当成主导航主语或正式完成证据。

### 4.2 launcher 与主链的关系

launcher 可以独立实现、独立上线、独立维护，但不得影响以下冻结事实：

- 主链成功标准仍由 `G0-G7` 决定。
- launcher 不承担主链数据主语。
- launcher 不改变 API envelope、数据来源语义与页面矩阵。

## 5. 仅在异常发生时才允许新增裁决

本轮 merged 明确：**当前无需新增人类裁决问题。**

只有以下 3 类异常真正发生时，才允许重新开启裁决文件：

| 异常类型 | 触发条件 | 届时才允许上抛的内容 |
|---|---|---|
| 规则 seed 不可用 | 遍历文档、SQL、Python 后仍无法生成非空、可审计 seed | 是否放宽 seed 来源或接受临时占位值 |
| real rolling 窗口缺失 | 客观上无法形成 `completed batch > 1` 的 real rolling 数据 | 是否允许人工切片或接受 `G3` 暂缓 |
| 页面一级主语冲突 | 同一页面在冻结矩阵下仍出现不可消解的一级主语冲突 | 是修矩阵还是拆页面 |

在以上异常没有发生前，禁止为了实施细节、措辞差异或局部工程便利重新上抛方向问题。
