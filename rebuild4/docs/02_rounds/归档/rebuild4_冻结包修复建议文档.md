# rebuild4 冻结包修复建议文档

## 0. 评审范围与结论

本建议文档**只基于以下 6 份最终文件文本**做交叉评审，不假设代码、数据库、页面或历史文档中的任何实现细节：

- `00_最终冻结基线.md`
- `01_最终技术栈与基础框架约束.md`
- `02_数据生成与回灌策略.md`
- `03_最终执行任务书.md`
- `04_最终校验清单.md`
- `05_本轮范围与降级说明.md`

结论：这套冻结包已经有清晰主方向，但还存在一批**会直接导致执行偏差、实现理解分叉、验收口径松动**的问题。  
其中我判断有 **5 个阻断级问题**，如果不先修，后续代码 agent 很容易在“文档字面上都能自洽、但实际实现并不一致”的状态下推进。

## 1. 阻断级问题

### FR-01：`G0` 与 `G1` 的顺序自相矛盾，任务书按字面无法执行

**证据**

- `00_最终冻结基线.md:L112-L115` 明确写明：`rebuild4 / rebuild4_meta schema 尚不存在`，且该缺口归属 `G1`。
- `03_最终执行任务书.md:L28-L33` 的 `P0-G0-02` 却要求在 `Phase 0 / G0` 就向 `rebuild4_meta.contract_version` 写入记录，并初始化 `gate_definition`、`gate_check_item`、`gate_run_result`。
- `03_最终执行任务书.md:L46-L51` 的 `P1-G1-01` 又明确把 `rebuild4`、`rebuild4_meta` schema 与关键表创建放在 `Phase 1 / G1`。

**为什么这是阻断问题**

这不是“措辞不优雅”，而是**执行顺序不可落地**：  
如果严格按任务书，`P0-G0-02` 依赖的 schema / table 在 `P1-G1-01` 之前并不存在。  
不同工程师会出现两种实现分叉：

1. 有人会“偷跑”先建表再说，实际已经破坏了 Phase 顺序。
2. 有人会按字面卡死，认为 `G0` 根本无法过。

**建议修改**

把 `G0` 和 `G1` 的职责拆清楚，只保留一种解释空间：

- 方案 A：把 `contract_version / gate_*` 这类**最小控制表 bootstrap**前移到 `G0`，并在 `P0-G0-02` 中明确“只允许创建控制层最小元表，不允许进入业务表 DDL”。
- 方案 B：保留当前 `G1` 创建 schema/table 的顺序，但把 `P0-G0-02` 改为“生成待写入 manifest / gate seed 文件”，真正入库动作移到 `P1-G1-01` 之后。
- 两种方案只能保留一种，不能继续并存。

**需要本地核查**

- 当前代码或 SQL 脚手架里，是否已经存在一个“预建的 meta bootstrap 表”机制。
- `contract_version` / `gate_definition` / `gate_check_item` / `gate_run_result` 是否确实计划落在 `rebuild4_meta`，还是原本有独立 bootstrap schema。
- 如果已有 bootstrap 方案，需要把它正式写进六文件，而不是靠工程师默认常识补齐。

### FR-02：正式冻结包并未真正闭环，执行期仍可“重新生成/重新解释”冻结包

**证据**

- `00_最终冻结基线.md:L103-L108` 把 `03_final/00_05` 六文件定义为**唯一真相包**，并把新增裁决条件限制为 3 类异常。
- `03_最终执行任务书.md:L21-L26` 的 `P0-G0-01` 却仍要求把 merged 正文、裁决收口项、round3 基线**再收敛为**六份正式文件。
- `03_最终执行任务书.md:L35-L40` 的 `P0-G0-03` 又要求复核 `01_待裁决问题清单.md` 与六文件正式包；但 `01_待裁决问题清单.md` **不在六文件正式包内**。
- `00_最终冻结基线.md:L126-L143` 虽然给了 `package_id = rebuild4-final-freeze-2026-04-06-v1`，但**没有定义后续修文时如何升版**。

**为什么这是阻断问题**

表面上说“冻结”，但任务书仍允许在执行期：

1. 再次生成正式包；
2. 继续依赖包外文件；
3. 没有版本递增规则。

这会让执行团队在遇到冲突时，用“只是补一行文档”“只是重写 manifest”“只是参考待裁决清单”这种方式，**悄悄重解释冻结口径**。

**建议修改**

把 `G0` 改成“只读校验入口”，不要再把它写成“重新生成正式包”：

- `P0-G0-01` 改为：**校验** 六文件、校验 manifest、校验文件命名，不再允许“收敛生成”动作。
- `P0-G0-03` 不再依赖包外的 `01_待裁决问题清单.md`。  
  做法二选一：
  - 要么把“本轮裁决清单为空”直接写回 `00_最终冻结基线.md` 或 `05_本轮范围与降级说明.md`；
  - 要么把该清单纳入正式 manifest，成为第 7 份正式文件。
- 增补一条**版本升版规则**：六文件任一内容发生实质变化，必须生成新的 `package_id` 和新的 per-file version，旧版本改为 `is_current=false`。

**需要本地核查**

- 当前仓库里是否真的存在 `01_待裁决问题清单.md`，且是否为空。
- `contract_version` 表是否支持多版本留存与 current 切换。
- 现有流程里是否已经有人把 `P0-G0-01` 理解成“可以继续改正式文档”。

### FR-03：统一 API 合同还没有冻结到“后端只能有一种写法”的程度

**证据**

- `01_最终技术栈与基础框架约束.md:L43-L50` 规定顶层必须包含 `data_origin / origin_detail / subject_scope / subject_note / context.* / data`。
- `03_最终执行任务书.md:L149-L154` 与 `04_最终校验清单.md:L63-L65` 又把它写成 6 个顶层字段，其中是 `context`，不是 `context.*`。
- `01_最终技术栈与基础框架约束.md:L54-L56` 规定 `data_origin` 只允许 `real / synthetic / fallback`，同时主流程 API 只允许 `real` 或“诚实空状态”。
- `02_数据生成与回灌策略.md:L162-L165` 与 `03_最终执行任务书.md:L217-L222` 只对 compare 的 `fallback` 空 envelope 做了明确约束；**主流程的“诚实空状态”没有给出统一 JSON 形状**。

**为什么这是阻断问题**

这会直接导致前后端、测试对 API 合同理解不一致：

- `context.*` 会被一些人理解成“扁平化多个顶层字段”，另一些人理解成“顶层 `context` 对象”。
- 当主流程无 real 数据时：
  - 有人可能返回 `data_origin=real + data=[]`
  - 有人可能返回 `data_origin=null`
  - 有人可能返回 `204`
  - 有人可能返回 `data_origin=fallback`
- 这些写法都能自称“诚实空状态”，但前端行为会完全不同。

**建议修改**

在 `01_最终技术栈与基础框架约束.md` 中补一个**强制 JSON 样例**，至少冻结：

1. 顶层固定就是：
   ```json
   {
     "data_origin": "...",
     "origin_detail": "...",
     "subject_scope": "...",
     "subject_note": "...",
     "context": {},
     "data": ...
   }
   ```
2. 明确 `context` 必须是对象，不再使用 `context.*` 写法。
3. 明确“诚实空状态”的统一规则：
   - 列表型 API：`data=[]`
   - 详情型 API：`data=null`
   - 汇总型 API：`data={}` 是否允许，需要明确
   - HTTP 状态码固定 200 还是允许 204，需要明确
4. 明确主流程空态时 `data_origin` 应该取什么值；不要让工程师自己猜。

**需要本地核查**

- 现有后端是否已经有统一 response model / pydantic schema。
- 现有前端对 `data_origin` 与 `context` 的消费逻辑是什么。
- 是否已经存在历史 API 用了扁平化 `context_*` 字段；若有，需要评估迁移影响。

### FR-04：`subject_scope` 与“页面主语矩阵”没有随冻结包一起落地，主语边界仍可漂移

**证据**

- `01_最终技术栈与基础框架约束.md:L55` 把 `subject_scope` 固定为 `run / batch / object / baseline / governance / compare / initialization`。
- `03_最终执行任务书.md:L152` 的核心 API 列表却包含 `flow-overview / flow-snapshot / observation / anomaly / profiles` 等。
- `04_最终校验清单.md:L63` 又要求 `subject_scope` 与“页面主语矩阵”一致。
- `00_最终冻结基线.md:L12` 说 round2 已经把“页面 / API / 表 / 来源矩阵”收敛为可执行合同，但这份矩阵**并没有出现在六文件正式包里**。

**为什么这是阻断问题**

当前文档没有给出“页面 → API → subject_scope”的唯一映射。  
例如：

- `flow-overview` 是 `run` 主语还是 `batch` 主语？
- `profiles` 是 `object` 主语还是独立 `profile` 主语？
- `observation` / `anomaly` 页面是 `object` 主语、`batch` 主语，还是混合主语？

这会让后端、前端、测试都能各自自圆其说，但最后互相验收不一致。

**建议修改**

补一张**正式附录矩阵**，至少包含以下列：

- `page_id`
- `route`
- `api_name`
- `subject_scope`
- `context 必填键`
- `empty-state 规则`
- `是否允许 compare 降级语义`
- `对应 Gate`

没有这张矩阵，`subject_scope 与页面主语矩阵一致` 这句话无法落地。

**需要本地核查**

- 当前 router、页面设计稿、API handler 命名之间，是否已经隐含存在一版矩阵。
- `observation / anomaly / profiles / flow-*` 在现有实现里分别被谁当成主语。
- 是否已经有 FE 根据旧理解把这些页归到了不同导航层级。

### FR-05：`03_最终执行任务书.md` 与 `04_最终校验清单.md` 在 G4/G5/G6 上明显脱节，验收会流于“抽检”

**证据**

- `03_最终执行任务书.md:L152-L167` 的 `G4` 已经覆盖核心 API、主流程空态、governance 与 compare 语义。
- `03_最终执行任务书.md:L174-L204` 的 `G5/G6` 又分别定义 governance 与 baseline/profile 的正式能力。
- `05_本轮范围与降级说明.md:L12-L15` 还把 `M3/G4`、`M4/G6`、`M5/G5` 分成不同模块。
- 但 `04_最终校验清单.md:L63-L83` 的校验只写成：
  - “抽检核心 API”
  - “抽检主流程、objects、baseline、initialization”
  - “governance 至少提供 12 组正式接口”
  - “profile 响应主状态来自新模型”
  没有逐 endpoint、逐字段、逐页面绑定到任务书原文。

**为什么这是阻断问题**

这里会同时出现两类偏差：

1. **阶段边界漂移**  
   有人会说：`G4` 已经把 governance / baseline / profile 接口接上了，所以算完成；  
   另一些人会说：必须到 `G5/G6` 才算完成。
2. **验收抽样过松**  
   例如 `G4-02` 任务书包含 `governance` 与 `profile` 的 real/空态约束，但清单只抽检 `主流程 / objects / baseline / initialization`，直接把 governance/profile 漏掉了。

**建议修改**

- 在 `03_最终执行任务书.md` 明确：
  - `G4` 只验**统一 envelope 与来源语义**
  - `G5` 才验 governance **功能完整性**
  - `G6` 才验 baseline/profile **功能完整性**
- 在 `04_最终校验清单.md` 中把“抽检/至少”改成**逐项点名**：
  - `G4`：列出 12 个核心 API，逐个验 envelope 和 source 语义
  - `G5`：点名 12 个 governance endpoint 名称，不接受“数量够了但名字不对”
  - `G6`：点名 `profile_lac / profile_bs / profile_cell` 以及必须下沉到 `legacy_ref` 的字段

**需要本地核查**

- 当前是否已经有 API contract test，只是文档没写。
- governance 的实际 endpoint 命名是否与文档完全一致。
- G4/G5/G6 是否已经被不同成员拆分开发；若已拆分，更需要先修文档边界。

## 2. 高优先级修复项

### FR-06：页面/API 清单没有闭合，`12 个正式页面` 与实际路由/API 数量对不上

**证据**

- `05_本轮范围与降级说明.md:L15` 写的是“`12 个正式页面` 通过 Playwright 验收”。
- 但 `03_最终执行任务书.md:L228-L310` 的 Playwright 范围里，实际出现了：
  - `/flow-overview`
  - `/flow-snapshot`
  - `/runs`
  - `/objects`
  - `/objects/:id`
  - `/observation-workspace`
  - `/anomaly-workspace`
  - `/baseline`
  - `/profiles/lac`
  - `/profiles/bs`
  - `/profiles/cell`
  - `/initialization`
  - `/governance`
  - `/validation/compare`
  共 **14 个路由/详情入口**（还不算 P00 主页/导航检查）。
- `03_最终执行任务书.md:L152` 的核心 API 还包含 `batches`，但 Phase 5 并没有对应的独立页面验收。

**为什么这是高风险问题**

这会让不同人对“12 个正式页面”的理解不同：

- 有人把 `objects/:id` 当详情页，不单独计数；
- 有人把三类 `profiles` 当 1 页；
- 有人把 `batches` 当 `/runs` 的内嵌详情，不需要单独 API 验收；
- 有人会认为 `P00` 算 1 页。

结果是：**范围、工时、验收都会漂**。

**建议修改**

补一张**正式页面清单**，一行一个 `page_id`，明确：

- 是否计入“12 个正式页面”
- 是否有详情页
- 详情页是否单独算 page
- 绑定哪些 API
- 哪些 Playwright case 覆盖它

并同时补一张**核心 API 清单**，明确 `batches` 是独立 API、独立页面，还是 `/runs` 的子资源。

**需要本地核查**

- 当前前端 router 实际有多少 route。
- 现有原型里 `profiles` 是一个页签页还是 3 个独立路由。
- `batches` 是否已有独立接口与 handler。

### FR-07：观察工作台的“`三段式`”表述自相矛盾，会让不同工程师做出不同 UI / 状态机

**证据**

- `03_最终执行任务书.md:L266` 写的是 `waiting / observing / 晋升 / 转问题三段式资格推进`。
- `03_最终执行任务书.md:L267` 又写“页面可见等待中、观察中、本批晋升、本批转问题或等价三段式结构”。
- `04_最终校验清单.md:L107` 也只保留了“三段式资格推进”这句抽象表述。

**为什么这是高风险问题**

字面上其实是**四个状态**，但文档又反复说“三段式”。  
这会产生至少三种实现：

1. 四列展示：等待 / 观察 / 晋升 / 转问题
2. 三列展示：前两列 + “终态”一列（内含晋升/转问题）
3. 三段流程图，但终态不做分桶

这些实现都能自称“符合三段式”，但前端布局、API 返回、测试断言都会不一样。

**建议修改**

直接把状态机冻结成一版，不要再用“或等价”：

- 方案 A：明确为 **四状态**
- 方案 B：明确为 **三阶段 + 第三阶段双结果桶**

如果选择方案 B，文档中必须写成：
“第三阶段包含 `promoted_this_batch` 与 `converted_to_issue_this_batch` 两个结果桶”。

**需要本地核查**

- 现有 UI 原型到底是四列还是三段。
- 现有后端是否已经把这两个结果做成不同枚举。
- 测试脚本是否已经按某一种布局写死。

### FR-08：异常工作台的关键语义没有被上游数据合同冻结，前后端会各自发明字段

**证据**

- `03_最终执行任务书.md:L273-L275` 要求异常页具备：
  - object / record 双视角
  - `severity`
  - `downstream_impact`
  - `禁锚 / 禁基线` 语义
- 但 `02_数据生成与回灌策略.md:L80`、`L118` 对异常只写到了泛化的 `batch_anomaly_*`。
- 六文件中没有任何地方冻结：
  - 异常级别枚举
  - `downstream_impact` 的字段结构
  - `禁锚 / 禁基线` 的布尔字段还是规则解释字段
  - object anomaly 与 record anomaly 的关联键

**为什么这是高风险问题**

当前异常页看似“需求很清晰”，但实际上只是 UI 要求清晰，**数据合同没冻结**。  
后端可以返回任何自定义结构，前端也可以自己拼字段名；最后测试只能看“页面上有没有类似词语”。

**建议修改**

在 `02_数据生成与回灌策略.md` 或新增附录中，补一份**异常合同最小集**：

- object anomaly 必填字段
- record anomaly 必填字段
- `severity` 允许值
- `downstream_impact` 结构
- `forbid_anchor` / `forbid_baseline` 是否是正式字段名
- object/record 双视角如何互相跳转

**需要本地核查**

- 当前数据库或代码里是否已经存在 anomaly schema。
- 是否已有旧系统异常字段可复用到 `legacy_ref`，还是必须新建正式字段。
- 前端原型上显示的异常文案是否依赖旧字段名。

### FR-09：rolling 的“真实窗口化输入”没有落到可执行级别，`G3` 完成标准仍留了很大自由发挥空间

**证据**

- `02_数据生成与回灌策略.md:L17` 只写 `real rolling = 真实窗口化输入`。
- `02_数据生成与回灌策略.md:L92` 只要求 `completed batch > 1`。
- `03_最终执行任务书.md:L120-L143` 与 `05_本轮范围与降级说明.md:L64-L68` 也都只约束了“不能在 G2 前启动 rolling / 不能混入 scenario_replay”，但没有给出窗口切分规则。

**为什么这是高风险问题**

只要没有冻结这些参数，工程师就可以用不同方式完成 `G3`：

- 1 小时批次
- 2 小时批次
- 自然日批次
- 按事件时间切
- 按导入时间切
- 是否处理迟到数据
- 时区按 UTC 还是本地时区

这些差异会直接影响：

- timepoints
- baseline 刷新触发点
- batch 数量
- 页面图表形状
- compare 对照口径

**建议修改**

给 `02_数据生成与回灌策略.md` 补一个 **rolling_window_contract**，至少明确：

- 使用哪个时间字段切窗
- 时区
- 窗口粒度
- 闭区间/开区间规则
- 迟到数据策略
- 批次重跑的幂等键
- 是否允许人工切片作为临时方案

**需要本地核查**

### FR-10：`parse_rule / compliance_rule` 的 seed 来源仍然是“文档 / SQL / Python”，没有正式来源白名单

**证据**

- `02_数据生成与回灌策略.md:L32-L39` 把 `parse_rule` 来源写成“文档 / SQL / Python 抽取 seed”。
- `03_最终执行任务书.md:L67-L72` 的 `P1-G1-04` 也是“从文档、SQL、Python 提取 seed”。
- 但 `00_最终冻结基线.md:L103-L108` 又要求六文件是唯一正式冻结包。

**为什么这是高风险问题**

这会让团队在执行时重新去历史文档、旧 SQL、旧脚本里“找规则”，等于把**正式规则来源重新打开**了。  
而且一旦多个来源冲突，六文件里没有任何优先级规则：

- 先信文档？
- 先信 SQL？
- 先信 Python？
- 冲突时谁裁决？

**建议修改**

把 seed 规则改成“两步冻结”：

1. 允许从历史来源抽取，但必须先生成一份**正式 seed 导出物**（CSV/SQL/YAML 均可）。
2. 正式执行只承认这份导出物，不再直接从“文档 / SQL / Python”多源即席抽取。

同时增加：
- 来源白名单
- 冲突优先级
- 生成脚本路径
- 导出物校验方式
- 导出物是否纳入 manifest

**需要本地核查**

- 当前实际能抽到 seed 的来源有哪些文件。
- 各来源之间是否已经出现冲突。
- 现有代码是否支持“从冻结 seed 导入”，还是只能临时解析历史脚本。

### FR-11：launcher 虽被定义了接口和职责，但没有进入任何正式执行/验收闭环，边界仍然留白

**证据**

- `01_最终技术栈与基础框架约束.md:L98-L108` 给 launcher 定义了身份、启动方式、WebSocket、REST API。

**为什么这是高风险问题**

这会让 التنفيذ团队出现两种完全不同的理解：

1. “不计入正式完成” = 可以先不做，接口定义只是说明文字
2. “既然已经冻结了 WS / REST 路径” = 至少要做一个可用最小版

- **保留为非阻断交付**：给 launcher 增加一个独立附录，写明“最小必须实现”和“仅做非阻断验收”，避免它既像范围内又像范围外。

**需要本地核查**

### FR-12：页面证据口径与 Phase 5 Playwright 口径没有被明确区分，容易让人误判“不能重新验页面”

**证据**

- `00_最终冻结基线.md:L105` 写的是“唯一页面证据来源：只允许承接 round3 merged 结论与已归档截图，不重新跑页面、不新增页面心证”。
- 但 `03_最终执行任务书.md:L228-L310` 与 `04_最终校验清单.md:L97-L119` 又明确要求做 Phase 5 Playwright 页面验收。

**为什么这是高风险问题**

这两段话并非必然逻辑冲突，但**没有被明确区分**：

- 一种理解：冻结阶段不再新增“方向判断”的页面心证，但执行期仍然可以跑 Playwright 验收。
- 另一种理解：既然“不重新跑页面”，那 Phase 5 只能参考旧截图，不能新增页面证据。

测试团队与实施团队很可能各自采用不同理解。

**建议修改**

在 `00_最终冻结基线.md` 明确拆成两个概念：

1. **冻结期页面判断依据**：只承认 round3 merged + 已归档截图，不再用页面重新裁方向。
2. **执行期页面验收证据**：允许使用 Playwright 新增验收证据，但只能验证冻结包已经写明的结论，不能反过来改口径。

**需要本地核查**

- 当前是否已经有 Playwright 证据产物目录规划。
- 是否有人把“不能重新跑页面”理解成“不能新增页面验收证据”。
- 现有测试脚本是否会生成截图并被视为正式归档物。
