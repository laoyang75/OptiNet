## 1. 风险总览

当前最大风险不是“信息缺失”，而是“已有信息还没有被写成硬约束”。

按阻塞程度统一排序，rebuild4 当前最重要的 8 项风险/缺口如下：

| 编号 | 风险 / 缺口 | 阻塞级别 | round2 必须完成什么 |
|---|---|---|---|
| R-01 | 唯一任务书、唯一矩阵、独立 Gate 尚未冻结 | P0 | 把执行入口、矩阵、Gate 独立成文 |
| R-02 | real 数据标准未冻结，当前正式动态主链不足 | P0 | 写清 initialization / rolling / snapshot / baseline 的 real 定义 |
| R-03 | rebuild2 元数据真相源存在口径分裂 | P0 | 冻结字段质量真相基线，以数据库为准 |
| R-04 | `parse_rule` / `compliance_rule` 仍为空表 | P0 | 给出正式元数据承接方案，不能继续只靠 Markdown/脚本 |
| R-05 | ODS 规则存在“26 条定义、24 条执行统计”的闭环缺口 | P1 | 明确缺失规则归属，拆开定义层与执行层 |
| R-06 | trusted 过滤解释层尚未进入正式交付范围 | P1 | 让治理页/API能解释损耗、来源、制式和追溯链 |
| R-07 | 页面/API/表/来源矩阵尚未冻结，主语仍可能漂移 | P1 | 为每个页面绑定唯一主语、唯一 API、唯一验收方式 |
| R-08 | `data_origin`、支撑治理层边界与历史兼容资产归属仍不够明确 | P1 | 冻结来源合同，明确 governance 正式范围、compare 降级边界、兼容资产归属 |

## 2. 逐项风险与缺口

### R-01 唯一任务书与 Gate 还没有成为真正的停机线

**结论**  
这是当前最高优先级的流程风险。如果不先冻结唯一主任务书、唯一矩阵和独立 Gate，rebuild4 仍会重演 rebuild3 的“先实现后纠偏”。

**依据**  
`02_rebuild3_04系列偏航复盘.md` 已明确指出：问题不在于文档不足，而在于 Gate 只是写在 prompt 里，没有成为“未过不许继续”的停机线。

**风险**  
会再次出现：

- 文档与实现并行漂移；
- 主流程先用 synthetic / fallback 顶住；
- round2、round3 继续在实现后补定义。

**round2 关闭标准**  
至少同时产出：

1. 唯一任务书骨架；
2. 页面/API/表/来源矩阵；
3. 独立 Gate 文件；
4. 唯一校验清单骨架。

**校验方式**  
如果 round2 仍然只有长篇说明文，没有独立 Gate 和矩阵，则该风险未关闭。

### R-02 real 数据标准未冻结，当前正式动态主链不足

**结论**  
这是当前最高优先级的数据风险。rebuild4 不能把现有 rebuild3 run / batch / baseline 计数，直接等同于“正式可交付的 real 主链”。

**依据（PG17，2026-04-06）**

- `rebuild3_meta.run` = 4，其中 `full_initialization` = 1，`scenario_replay` = 3；
- `rebuild3_meta.batch_snapshot` = 1,562，覆盖 142 个 batch、4 个 stage、11 个 metric；
- `rebuild3_meta.batch` 当前按 run 仅覆盖：1 个 full init batch、7/61/73 个 scenario batch；
- 当前不存在 real rolling run。

**风险**  
如果不先冻结标准，后续很容易再次发生：

- synthetic 继续进入主流程页；
- initialization 与 rolling 被混写；
- baseline 看起来可比，实际上没有 real 连续版本链。

**round2 关闭标准**  
写清并验收：

1. real initialization 的定义；
2. real rolling run 的定义；
3. real `batch_snapshot` 的定义；
4. real baseline diff 的前置条件。

**校验方式**  
PG17 能直接验证是否已经存在：1 条 real initialization、1 条 real rolling run、多个 completed batch、真实写入的 `batch_snapshot` 与可追溯的 baseline 链。

### R-03 rebuild2 元数据口径分裂，不先裁决会污染正式任务书

**结论**  
这是当前最高优先级的解释层风险。rebuild4 不能继续把旧文档数字和数据库事实并列当真。

**依据（PG17，2026-04-06）**

- `field_audit` = 27，决策分布 `keep=17 / parse=3 / drop=7`；
- `target_field` = 55；
- 历史文档仍保留 `keep 19 / parse 3 / drop 5`、`约 60 字段` 等口径。

**风险**  
如果 round2 不先冻结真相表，后续会出现：

- 页面文案与数据库事实不一致；
- API 返回的数字和说明文案不一致；
- 基础数据治理页失去可审计性。

**round2 关闭标准**  
产出一份统一的字段质量真相基线，至少包含：

1. 当前真实字段总数；
2. `keep / parse / drop` 明细；
3. `target_field` 正式清单；
4. 文档旧口径与数据库现口径的差异说明。

**校验方式**  
所有字段与规则说明必须能回指到数据库元数据表，而不是只回指历史 Markdown。

### R-04 `parse_rule` / `compliance_rule` 空表，解释层会天然断档

**结论**  
这是当前最高优先级的元数据缺口。若不先补正式承接方案，rebuild4 的“基础数据治理”能力会天然缺一段。

**依据（PG17，2026-04-06）**

- `rebuild2_meta.parse_rule` = 0；
- `rebuild2_meta.compliance_rule` = 0。

**风险**  
如果继续只依赖文档和脚本：

- 解析规则与合规规则无法被查询、审计、追溯；
- 前端只能展示静态说明或空状态；
- 新系统仍旧缺正式元数据真相层。

**round2 关闭标准**  
必须给出正式承接方案，至少写清：

1. 目标元数据表结构；
2. 文档 / 脚本到元数据表的映射清单；
3. 规则最小字段集合；
4. 后续如何补齐执行统计。

**校验方式**  
round2 必须落盘正式元数据设计；round3 / 实施阶段必须能让 PG17 查询到非空正式表。

### R-05 ODS 规则存在“定义层完整、执行层不完整”的闭环缺口

**结论**  
这是当前高优先级的规则闭环风险。rebuild4 不能把“26 条定义”直接包装成“26 条已执行事实”。

**依据（PG17，2026-04-06）**

- active `ods_clean_rule` = 26；
- 执行统计仅覆盖 24 个 `rule_code`；
- 缺失的 2 条规则为：`NULL_WIFI_MAC_INVALID`、`NULL_WIFI_NAME_INVALID`。

**风险**  
如果不显式拆层：

- 页面会误报规则覆盖度；
- 用户无法理解为何定义存在、统计为空；
- 后续实现容易把缺口静默吞掉。

**round2 关闭标准**  
必须拆成两层：

1. 规则定义层；
2. 规则执行统计层；

并对差集逐条标注“未执行 / 未补录 / 已废弃候选”。

**校验方式**  
用 `ods_clean_rule` 与 `ods_clean_result` 做差集校验；差集要么归零，要么被正式登记说明。

### R-06 trusted 过滤解释层缺失，会继续只展示结果不解释损耗

**结论**  
这是当前高优先级的业务可解释性风险。若不把 trusted 过滤损耗做成正式能力，rebuild4 仍然无法回答“为什么这些数据被排除”。

**依据（PG17，2026-04-06）**

- `l0_lac` = 43,771,306；
- trusted 命中 = 30,082,381；
- 被过滤 = 13,688,925（31.27%）；
- 被过滤中仍有 `RSRP` = 12,017,352；
- 被过滤中仍有 GPS = 11,350,552；
- 被过滤前三来源组合：`sdk / daa / cell_infos`、`sdk / daa / ss1`、`sdk / dna / cell_infos`。

**风险**  
如果只展示最终治理结果：

- 用户仍然看不到 trusted 前后损耗；
- 仍然无法理解“为什么这些有 GPS / 信号的数据没进去”；
- 数据解释链在最关键处断掉。

**round2 关闭标准**  
把以下能力列入正式范围：

1. 过滤漏斗；
2. 来源损耗分布；
3. 制式损耗分布；
4. 从最终结果反查上游过滤动作。

**校验方式**  
治理页 / 初始化页 / 对象详情至少一处能返回 trusted 前后数量、损耗比例、来源拆分和追溯信息。

### R-07 页面/API/表/来源矩阵未冻结，页面主语仍可能继续漂移

**结论**  
这是当前高优先级的架构风险。没有矩阵，执行阶段最容易再次退回“先把页面做出来，再补主语和来源说明”。

**依据**  
`design_notes.md` 与 `03_三线重构策略.md` 都要求页面围绕对象 / 决策 / 事实 / baseline 组织，并且要求把页面、API、表、来源和验收绑在一张矩阵里。

**风险**  
会再次发生：

- overview、snapshot、runs、baseline 上下文不一致；
- 页面有壳子但没有最小可用数据定义；
- 验收只能看页面是否打开，不能验证主语是否真实。

**round2 关闭标准**  
所有正式页面都必须能在矩阵中找到：

1. 唯一主语；
2. 唯一 API；
3. 唯一数据底座；
4. 允许的来源状态；
5. 最小数据前提；
6. 验收方式。

**校验方式**  
最终任务书逐页复核；如果某页不能回到矩阵唯一行，说明该风险仍未关闭。

### R-08 `data_origin`、支撑治理层边界与历史兼容资产归属仍需显式写实

**结论**  
这是当前高优先级的边界风险。`governance` / `compare` 的范围、`data_origin` 的消费方式、历史兼容资产的正式归属，如果不写清，就会继续形成灰区。

**依据**

- `design_notes.md` 已明确将 `验证/对照` 降级为辅助模块；
- `新增需求与数据准备清单.md` 要求 compare / governance 要么 real 接数，要么显式降级；
- `01_rebuild4_核心输入整理.md` 已把 `rebuild2 / rebuild2_meta / rebuild3*` 统一定义为只读参考源；
- `rebuild3_review` 材料反复提示 `rebuild3_sample_meta` 等历史兼容资产不能再偷渡成正式主语。

**风险**  
如果不写清：

- fallback 会再次被默认为“临时可用”；
- compare / governance 会继续膨胀 scope；
- 历史 sample / compatibility 资产可能再次污染正式主流程。

**round2 关闭标准**  
必须同时写清：

1. 每个核心读模型必须返回的来源字段；
2. `governance` 正式完成标准；
3. `compare` 降级边界与诚实表达；
4. 每个兼容资产的只读 / 验证 / 禁用归属。

**校验方式**  
检查 API 响应、页面 banner、矩阵和 schema 边界文档是否四处一致。

## 3. 当前最需要先关的 4 个缺口

如果 round2 只能优先关闭 4 项，统一建议顺序如下：

1. **R-01：唯一任务书 + 独立 Gate + 唯一矩阵**
2. **R-02：real 数据标准冻结**
3. **R-03 / R-04：字段质量真相基线 + parse/compliance 正式承接方案**
4. **R-07：页面/API/表/来源矩阵**

统一原因：这 4 项不关，后面的 governance、baseline diff、compare、画像与解释层都会继续建立在不稳定前提上。

---

## 4. round2 统一校验重点

round2 至少要能让下面这些问题从“现在说不清”变成“已经可校验”：

1. rebuild4 的 real initialization / rolling / snapshot / baseline 到底各是什么；
2. `parse_rule` / `compliance_rule` 到底如何成为正式元数据；
3. `governance` 与 `compare` 的正式 / 降级边界是什么；
4. 无上一版 baseline 时，页面与 API 如何诚实表达；
5. trusted 损耗解释层落在哪些页面 / API / 模块；
6. 每个页面的主语、API、数据底座和验收方式是什么。

## 5. 统一校验方式

本文件中的数据库事实，应继续以 `PG17 MCP execute_sql` 复核；本轮最关键的 5 组复核对象为：

1. `rebuild3` 事实层与对象层行数；
2. `rebuild3_meta.run / batch / batch_snapshot / baseline_version`；
3. `rebuild2_meta.field_audit / target_field / parse_rule / compliance_rule`；
4. `rebuild2_meta.ods_clean_rule / ods_clean_result` 的差集；
5. `rebuild2.l0_lac` 对 `dim_lac_trusted` 的过滤损耗。

只有当 round2 把这些事实转写成正式合同，并清楚标记哪些是数据库事实、哪些是冻结原则、哪些是已降级范围，本轮识别出的主要风险才算真正开始关闭。
