## 1. 统一结论

### 1.1 总判断

rebuild4 当前可以进入 **round2 细化设计**，但 **不能直接进入编码**。

阻塞点不是“资料不足”，而是“正式实施前必须冻结的合同还没有写成唯一可执行版本”。本轮统一后的第一优先级不是挑页面开工，而是先把下面 4 组前置合同冻结下来：

1. **唯一真相源与唯一任务书合同**
   - `rebuild4/docs/03_final/01_rebuild4_最终执行任务书.md` 与 `rebuild4/docs/03_final/02_rebuild4_最终校验清单.md` 是最终冻结出口；在此之前，round2 / round3 只能继续收敛，不得并行生效多份执行稿。
2. **来源语义合同**
   - `real / synthetic / fallback` 必须是页面、API、读模型共同遵守的合同字段，而不是实现内部约定。
3. **最小可用数据合同**
   - 初始化、滚动批次、快照、baseline、支撑治理层各自达到什么条件才算“可验收”。
4. **页面/API/表/来源矩阵合同**
   - 每个页面都必须能回到唯一主语、唯一 API、唯一数据底座、唯一来源状态和唯一验收方式。

### 1.2 本轮直接冻结的方向

结合三家输出与已冻结输入，round1 可以直接拍板以下方向，不再上抛给人类裁决：

1. **先冻结合同，再准备 real 数据，再接主流程读模型，最后才做页面层。**
2. **`governance` 纳入本轮正式范围，`compare` 明确降级为辅助验证模块。**
   - 依据：`design_notes.md` 已将“验证/对照”明确降级为辅助模块；`新增需求 A` 又要求把 rebuild2 清洗/过滤解释层做成正式可见能力，因此 `governance` 不能再一起降级。
3. **`rebuild2` / `rebuild2_meta` / `rebuild3*` 只作为只读参考源。**
   - rebuild4 的正式动态对象、事实、run、batch、snapshot、baseline 必须由 rebuild4 自己产出，不能把旧库结果直接冒充为 rebuild4 的正式当前态。
4. **`parse_rule` / `compliance_rule` 不能继续停留在 Markdown + 脚本口径。**
   - round2 必须给出可查询、可审计、可回溯的正式元数据承接方案，默认落入 `rebuild4_meta` 或等价正式元数据层。
5. **`/baseline` 在只有 1 个 real baseline 时允许上线，但只能展示当前版本，并明确写出“暂无上一版可比”。**
   - 不允许拿 rebuild2 结果或 synthetic baseline 冒充“上一版”。
6. **没有被允许使用 synthetic / fallback 的页面，必须进入诚实空状态。**
   - 不允许再用 fallback 伪装“页面有结果”。
7. **Gate 必须独立成文。**
   - 不再接受“Gate 写在长 prompt 里但没有停机效力”的写法。

### 1.3 rebuild4 的最小成功标准

本轮统一后的最小成功标准建议冻结为 6 条：

1. **唯一执行入口已冻结**
   - round3 合并后，只有 `rebuild4/docs/03_final/01_rebuild4_最终执行任务书.md` 与 `rebuild4/docs/03_final/02_rebuild4_最终校验清单.md` 具备最终执行效力。
2. **主流程来源语义已冻结且前后端一致**
   - 主流程页必须明确 `data_origin`、`origin_detail`、`subject_scope`，且前端真实消费。
3. **rebuild2 清洗解释层已进入正式交付范围**
   - 至少能查询字段审计、ODS 规则定义/执行差异、trusted 过滤损耗、来源损耗和结果追溯。
4. **系统完成后立即有可评估 real 数据**
   - 至少 1 个 real `full_initialization` run；
   - 至少 1 条 real rolling run；
   - 该 rolling run 下多于 1 个 completed batch；
   - 每个 completed batch 都有 real `batch_snapshot`。
5. **baseline 语义诚实可验**
   - 当前页面展示的 baseline 必须可追溯到 `run_id / batch_id / rule_set_version`；
   - 若暂无上一版 real baseline，必须诚实显示“暂无上一版可比”。
6. **支撑治理层边界已明确**
   - `governance` 进入本轮正式完成标准；
   - `compare` 若仍未 real 接数，必须显式降级且不计入本轮正式完成标准。

## 3. round2 必须产出的统一交付件

round2 的目标不是开始写代码，而是把 round1 的判断细化成可执行合同。统一后，至少应交付下面 5 个包：

### 包 A：数据语义与元数据合同

必须写清：

1. `run / batch / baseline_version / contract_version / rule_set_version` 的正式结构；
2. `real / synthetic / fallback` 的字段定义、返回位置与页面表达；
3. rebuild2 解释层元数据如何承接到 rebuild4；
4. `parse_rule` / `compliance_rule` 的正式元数据结构与承接路径。

### 包 B：数据准备合同

必须写清：

1. real `full_initialization` 的输入源、窗口与完成定义；
2. real rolling run 的输入源、最小 batch 序列与完成定义；
3. `batch_snapshot` 的真实写入条件；
4. baseline 刷新前提；
5. 无上一版时的页面与 API 表达。

### 包 C：页面/API/表/来源矩阵

每个页面至少要落下：

1. 页面主语；
2. 对应 API；
3. 读模型表/视图；
4. `data_origin` 允许范围；
5. 最小可用数据；
6. 空状态 / synthetic / fallback 规则；
7. SQL / API / Playwright 验收方式。

### 包 D：Gate 与校验基线

每个 Gate 至少包含：

1. Gate 名称与前置输入；
2. 通过条件；
3. SQL 校验项；
4. API 校验项；
5. 页面校验项；
6. 未通过时的停机规则。

### 包 E：支撑治理层边界

必须明确：

1. `governance` 的正式数据底座；
2. `compare` 的降级边界、banner、来源说明和不计入完成标准的规则；
3. 历史兼容资产（如 `rebuild3_sample_meta`）的只读/验证/禁用归属。

## 4. 统一实施顺序

### 4.1 编码前的顺序

1. **Phase 0：冻结任务书、矩阵、校验清单、Gate**
2. **Phase 1：schema 与元数据底座**
   - `rebuild4` / `rebuild4_meta` schema
   - 字段治理、ODS 规则、trusted 损耗解释层承接
   - `parse_rule` / `compliance_rule` 方案落盘
3. **Phase 2：流转主链**
   - 初始化、滚动批次、四分流、对象注册、资格计算、baseline 刷新、real `batch_snapshot`
4. **Phase 3：读模型 API**
   - overview / snapshot / runs / objects / observation / anomaly / baseline / governance
   - 所有核心响应返回 `data_origin`
5. **Phase 4：前端工作台**
   - 先主流程层，再治理支撑层；
   - `compare` 只按已降级边界提供诚实能力。

### 4.2 模块执行顺序

推荐以模块而不是页面数量推进：

1. `M0`：schema / meta / 版本合同 / Gate 框架
2. `M1`：数据准备链路（initialization、rolling、snapshot、baseline）
3. `M2`：rebuild2 清洗解释层读模型
4. `M3`：主流程页读模型（overview、snapshot、runs、objects、workspaces）
5. `M4`：baseline / profile
6. `M5`：支撑治理层（governance 正式完成；compare 仅按降级范围实现）

统一原因：如果 `M1` 和 `M2` 不先完成，后续页面即使可渲染，也会再次退回“页面先行、主语失真、数据后补”的 rebuild3 路径。

## 5. 统一校验方式

### 5.1 round1 结论复核点

本文件中的关键结论，应通过以下方式复核：

1. **文档复核**
   - Tier 0 / Tier 1 / Tier 2 / Tier 3 是否支持同一结论；
2. **PG17 复核**
   - rebuild3 事实层、对象层、run/batch/baseline 数量；
   - rebuild2 字段治理、ODS 规则差异、trusted 过滤损耗；
3. **边界复核**
   - 是否清楚区分了“数据库事实”“冻结原则”“仍待 round2 细化的合同”。

### 5.2 round2 的入场与出场条件

只有当下列条目都已写成可校验文档，才建议进入 round3：

1. 最小成功标准已变成逐项可验收条目；
2. 页面/API/表/来源矩阵已成稿；
3. real initialization / rolling / snapshot / baseline 方案已冻结；
4. `governance` / `compare` 边界已写入任务书；
5. Gate 已独立成文。

---

## 6. 最终建议

round1 合并后的唯一主线应写成：

**先冻结合同，再准备 real 数据，再接主流程读模型，最后才做页面与辅助模块。**

如果 round2 不能把这条顺序写死，rebuild4 仍然会重复 rebuild3 已经证明代价过高的路径：先有页面，再补主语，再补来源说明，最后才发现数据并没有准备好。
