## D-001：rebuild4 是否复用 rebuild3 的数据，还是从 rebuild2 重新跑起

**背景**：

当前 rebuild3 已有大量数据：
- fact_standardized：43,771,306 行
- obj_cell：573,561 / obj_bs：193,036 / obj_lac：50,153
- 4 个 baseline_version

但 rebuild3 的数据有语义约束：
- 只有 1 个 real full_initialization
- 3 个 scenario_replay 是 synthetic 语义
- 四分流合计与 standardized 一致（43,771,306 = 43,771,306），数据完整性可信

**为什么必须现在裁决**：

这个决定影响 Phase 1 的所有工作内容——是"迁移 + 标记"还是"重新初始化"。如果推迟，Phase 1 的 schema 设计、数据准备、Gate A 通过条件全部悬空。

**选项**：

- **A：复用 rebuild3 数据作为 rebuild4 的初始状态**
  - rebuild4 schema 中通过视图或物化视图引用 rebuild3 数据
  - rebuild4_meta 中标记这些数据的 data_origin = inherited
  - 新增的 real incremental run 从此基础上继续
  - 优点：启动快，已有数据可立即支撑 UI 评估
  - 风险：继承了 rebuild3 的 scenario 数据语义

- **B：从 rebuild2 的 l0 层重新跑 rebuild4 的完整流转**
  - rebuild4 自己做 full_initialization，输入源为 rebuild2.l0_gps / l0_lac
  - 所有对象、事实、baseline 由 rebuild4 自己产出
  - 优点：数据语义完全清晰，不继承历史包袱
  - 风险：初始化需要时间（4300 万行标准化 + 四分流 + 对象注册），且需要验证流转引擎

- **C：混合方案——rebuild3 数据作为只读参考，rebuild4 从 rebuild2 重新跑但降低初始化范围**
  - rebuild4 先用 rebuild2 的一个子集（如 1 天数据）完成 real initialization
  - rebuild3 数据保留在只读 schema 中，供对照验证
  - 后续逐步扩大到全量
  - 优点：兼顾速度和语义清晰
  - 风险：子集初始化的代表性

## D-002：compare / governance 本轮是否降级

**背景**：

rebuild3 中 /compare 和 /governance 仍有 fallback 历史包袱。UI_v2 已把"验证/对照"放在导航最后一项。

**为什么必须现在裁决**：

如果不裁决，执行过程中会反复纠结"是否要把 compare 做到 real 数据"，消耗注意力。如果明确降级，Phase 4 的页面数量和 Phase 3 的 API 数量都会减少。

**选项**：

- **A：本轮必须做到 real 数据**
  - compare 和 governance 都必须接 real 数据
  - 优点：交付完整度高
  - 风险：工作量增加，可能影响主链路进度

- **B：明确降级为 Phase 6 的后续任务**
  - 本轮不做 compare 和 governance 的 real 接数
  - 页面保留但显式标记"尚未接入真实数据"
  - 优点：聚焦主流程
  - 风险：交付物看起来不完整

- **C：governance 本轮做，compare 降级**
  - governance（基础数据治理）与数据清洗解释层强相关，必须本轮完成
  - compare（验证/对照）降级到后续
  - 优点：保留清洗解释层的完整性，同时不在验证对照上花太多时间

## D-003：rebuild2 的 parse_rule / compliance_rule 补建到什么程度

**背景**：

当前 `rebuild2_meta.parse_rule` = 0 行，`rebuild2_meta.compliance_rule` = 0 行。文档宣称分别有 64 行和 17 行。这些规则在脚本和文档中存在，但没有进入可查询元数据。

**为什么必须现在裁决**：

这决定了 Phase 1 的元数据补建工作量，也决定了基础数据治理页面的展示粒度。

**选项**：

- **A：完整补建——从脚本反推所有规则，填入 rebuild4_meta**
  - 每条规则有 rule_code / rule_type / description / affected_fields / affected_rows
  - 优点：元数据完整，页面可以展示完整规则清单
  - 风险：脚本到元数据的反推需要逐条验证

- **B：按需补建——只补建在基础数据治理页面中需要展示的规则**
  - 优先补建影响行数最大的规则
  - 其余规则以文档引用形式说明
  - 优点：工作量可控
  - 风险：不完整，后续可能需要再补

- **C：不补建元数据表，只做文档级说明**
  - 在基础数据治理页面中以静态文档形式展示规则清单
  - 不追求可查询
  - 优点：最快
  - 风险：违反"可查询、可审计"的原则

## D-004：batch_snapshot 的写入时机与范围

**背景**：

当前 rebuild3_meta.batch_snapshot 有 1,562 行，覆盖 142 个 batch，4 个 stage（input / routing / objects / baseline），11 个 metric。这是流转快照页面的底座。

**为什么必须现在裁决**：

snapshot 的写入时机决定了快照数据的粒度和可用性。如果写入不够及时或不够完整，流转快照页面会有数据空洞。

**选项**：

- **A：每个 batch 完成后立即写入，覆盖全部 4 个 stage x 11 个 metric**
  - 与 rebuild3 当前设计一致
  - 优点：快照粒度最细
  - 风险：写入量大，需要考虑性能

- **B：只在 batch 完成后写入 routing + objects 阶段的核心 metric**
  - input 和 baseline 阶段可以从原始表回算
  - 优点：减少写入量
  - 风险：回算增加查询复杂度

- **C：保持 A 的覆盖范围，但允许异步写入**
  - batch 完成后异步触发 snapshot 写入
  - 不阻塞主流转链路
  - 优点：兼顾完整性和性能
  - 风险：短暂的数据延迟

## D-005：rebuild4 前端是否从 rebuild3 前端代码继承

**背景**：

rebuild3 已有 Vue 3 + TypeScript 前端工作台。rebuild4 的 UI 需要按 UI_v2 design_notes 重新实现。

**为什么必须现在裁决**：

这影响 Phase 4 的工作方式——是在 rebuild3 代码上改，还是重新写。rebuild3 的前端经历了多轮偏航修正（04 系列复盘已记录），代码中可能残留旧心智。

**选项**：

- **A：从 rebuild3 前端代码继承，按 UI_v2 改造**
  - 复用项目骨架、路由、Pinia store、共享组件
  - 逐页改造为 UI_v2 设计
  - 优点：启动快，组件可复用
  - 风险：旧代码心智可能污染新实现

- **B：全新前端项目，UI_v2 design_notes 为唯一设计输入**
  - Vue 3 + TypeScript + Vite 全新搭建
  - 不引用 rebuild3 前端代码
  - 优点：没有历史包袱
  - 风险：工作量大，已验证的组件需要重写

- **C：新项目，但允许从 rebuild3 前端复制验证过的共享组件**
  - 项目骨架全新
  - LifecycleBadge / HealthBadge / DeltaIndicator 等验证过的组件允许复制
  - 优点：干净起点 + 复用验证成果
  - 风险：需要逐组件审计是否符合 UI_v2

## D-006：三层画像页（LAC/BS/Cell）的数据来源

**背景**：

UI_v2 设计中画像视角层（LAC 画像 / BS 画像 / Cell 画像）与主流程层的对象浏览页是互补关系。画像页偏数据质量视角，对象浏览偏治理状态视角。

画像数据在 rebuild2 中有基础研究（GPS 修正、信号补齐、异常分类），在 rebuild3 中有部分继承（classification_v2 / gps_confidence / signal_confidence 降级为解释层）。

**为什么必须现在裁决**：

画像页的数据来源决定了它们的展示是 rebuild2 的静态研究结果还是 rebuild4 的动态计算结果。

**选项**：

- **A：画像页完全使用 rebuild4 自己的动态计算结果**
  - rebuild4 自己做 GPS 修正、信号统计、异常分类
  - rebuild2 的旧分类只作为对照参考
  - 优点：数据新鲜度高，与治理状态同步
  - 风险：需要重新实现画像计算逻辑

- **B：画像页以 rebuild2 研究结果为底座，rebuild4 只做增量更新**
  - 初始画像从 rebuild2 继承
  - 后续新增对象的画像由 rebuild4 计算
  - 优点：工作量可控
  - 风险：新旧画像的计算口径可能不一致

- **C：本轮画像页降级为只读展示 rebuild2 结果**
  - 画像页不做动态计算
  - 直接读取 rebuild2 的画像数据
  - 页面明确标注 data_origin = inherited
  - 优点：最快
  - 风险：画像与治理状态脱节
