## 2. 逐项分析

### R-01：rebuild2 元数据口径分裂

**现象**：

同一事实在不同来源中有不同数字：

| 事实 | 文档口径 | 数据库真实值 | 差异 |
|------|---------|------------|------|
| field_audit keep/parse/drop | 19/3/5 | 17/3/7 | keep 差 2，drop 差 2 |
| target_field 数量 | "约 60" | 55 | 差 5 |
| ODS 规则分类 | "1+19+4+2" 或 "1+17+4+2" | 1+22+3 | nullify 和 convert 数量不同 |

**根因**：rebuild2 文档在不同时期写成，期间规则有调整，但文档未同步更新。数据库是执行结果，文档是写作时的理解。

**影响**：如果 rebuild4 不先裁决口径，后续任何引用 rebuild2 字段数量的文档、页面、API 都会产生不一致。

**解决方式**：
- 统一以 `rebuild2_meta` 数据库为真相源
- 在 rebuild4 任务书中写死一份"字段质量真相基线"，不再引用旧文档数字
- 差异处需要一行说明（"文档曾记录 X，数据库当前为 Y，以 Y 为准"）

**校验方式**：`SELECT decision, count(*) FROM rebuild2_meta.field_audit GROUP BY decision` 已验证

### R-02：parse_rule / compliance_rule 空表

**现象**：

- `rebuild2_meta.parse_rule`：0 行
- `rebuild2_meta.compliance_rule`：0 行
- 文档宣称 parse_rule 64 行、compliance_rule 17 行

**根因**：这两类规则在 rebuild2 中以文档和脚本形式存在，未被填充到元数据表。

**影响**：
- rebuild4 无法直接复用这些规则的元数据
- 数据清理解释层中"解析规则"和"合规规则"将没有可查询底座
- 基础数据治理页面将无法展示这部分

**解决方式**：
- Phase 1 必须补建：把文档/脚本中的规则回填到 `rebuild4_meta` 对应表
- 补建后必须与 rebuild2 脚本实际行为交叉验证
- 不能只从文档抄，必须从脚本逻辑反推

**校验方式**：`SELECT count(*) FROM rebuild2_meta.parse_rule` = 0，`SELECT count(*) FROM rebuild2_meta.compliance_rule` = 0 已验证

---

### R-03：真正的 real 动态主链不足

**现象**：

当前 rebuild3_meta.run 中：
- `full_initialization`：1 个
- `scenario_replay`：3 个

没有真正的 real rolling/incremental run。batch_snapshot 的 142 个 batch 全部来自 full initialization 或 scenario replay。

**根因**：rebuild3 开发重心在系统结构和页面，增量链路只在 scenario 模式下验证过。

**影响**：
- rebuild4 如果直接复用 rebuild3 数据作为初始状态，需要明确这些数据的语义
- UI 上"批次趋势"、"增量变化"等功能在 real 数据下未被验证
- "可研究流转参数"这个核心承诺在 real 增量场景下没有实证

**解决方式**：
- Phase 2 必须产出至少 1 条 real incremental run，含多个 completed batch
- 不能用 scenario_replay 冒充 real incremental
- 验收标准：flow-overview 能以 real batch 为主语

---

### R-04：scenario_replay 仍是 synthetic 语义

**现象**：

3 个 scenario_replay run 的 baseline 命名包含 `SCN-SMOKE_INIT1D_STEP2H`、`SCN-INIT1D_STEP2H`、`SCN-INIT2D_STEP2H`，这些都是参数化场景，不是真实时间窗口数据。

**根因**：scenario replay 是 rebuild3 的评估模式，用于验证不同初始化/步长参数下的系统行为。

**影响**：
- 如果 rebuild4 继续在 UI 中把 scenario 数据当成默认展示，用户会误解
- 如果不明确区分 real 和 scenario，data_origin 合同就是假的

**解决方式**：
- 在 rebuild4 的 run 表中必须有 `run_type` 字段明确区分
- UI 上必须显式标记 scenario run
- 主流程页默认只展示 real run，scenario 放入独立评估模式

---

### R-05：data_origin 合同未实施

**现象**：

设计文档已经定义了 data_origin = real / synthetic / fallback，但当前 rebuild3 的 API 中并未普遍返回此字段。rebuild3 的 /compare、/governance 仍有 fallback 历史包袱。

**根因**：rebuild3 开发过程中优先保证页面可用，data_origin 是后期提出的要求。

**影响**：
- 用户无法判断页面展示的是 real 还是 fallback
- 后续验收时无法自动化检查数据真实性
- 违反"不允许 fallback 冒充 real"的核心约束

**解决方式**：
- Phase 3 每个读模型 API 必须返回 data_origin
- 前端必须消费并展示
- 验收标准：每个页面都能通过 API 响应确认来源

---

### R-06：映射矩阵未建立

**现象**：

当前没有一张完整的"页面 -> API -> 表 -> data_origin -> 最小可用数据 -> 验收方式"矩阵。

**根因**：rebuild3 的 04 系列偏航复盘已明确指出此问题——"UI 目标、数据目标、验证目标没有被一张矩阵绑死"。

**影响**：
- 执行时容易"先做会动的部分"
- 验收时没有逐行对照标准
- 不同 agent 对同一页面的数据前提理解可能不一致

**解决方式**：
- Phase 0 必须冻结此矩阵
- UI_v2 design_notes 中的读模型建议是良好起点
- 矩阵必须包含：是否允许 synthetic、是否允许 fallback、空状态是否可接受

---

### R-07：Gate 未成为硬门禁

**现象**：

rebuild3 的 04e 已经提出 Gate A-G，但实际执行中 Gate 只是写在 prompt 里，没有成为"未过不许继续"的停机线。

**根因**：Gate 作为长 prompt 的一部分，容易被跳过。没有单独文档、单独验收、单独停机条件。

**影响**：
- Phase 之间没有真正的质量卡点
- 容易出现"先做后面的，再回来补前面的"
- 最终验收时发现基础不牢，返工代价大

**解决方式**：
- 每个 Gate 单独文档
- 每个 Gate 有明确的 SQL / API / Playwright 验证命令
- 未通过不得进入下一 Phase

---

### R-08：旧文档未制度化退役

**现象**：

rebuild3 中 `UI/` -> `UI_v2/` 的切换、多版 prompt 并存，导致执行者可能同时参考多份不一致的文档。

**根因**：新文档出来后，旧文档仍在目录中保持"可执行状态"。

**影响**：
- 如果 rebuild4 不明确哪些文档是唯一主真相源，历史会重演

**解决方式**：
- rebuild4 已把资料收拢在 `rebuild4/docs/`，这是正确的
- Phase 0 冻结时必须明确：只有 `rebuild4/docs/03_final/` 下的文档是执行真相源
- 其余一切都是输入、调查、草案

---

### R-09：页面先行数据后补的路径惯性

**现象**：

rebuild3 中反复出现"页面能打开，但数据不真实"的问题。

**根因**：开发者倾向于先做可见的 UI，data_origin 和数据准备被推迟。

**影响**：
- 如果 rebuild4 不在任务书层面强制先做数据再做 UI，惯性会再次生效

**解决方式**：
- Phase 1-2（数据底座 + 流转引擎）必须在 Phase 4（前端工作台）之前完成
- Gate A / B 是 Phase 4 的前置条件
- "有数据可评估"是正式交付条件，不是附属工作

---

### R-10：ODS 规则定义-执行层不闭合

**现象**：

- 定义层：26 条规则
- 执行层：每张 L0 表 24 条（共 48 行）
- 缺失：`NULL_WIFI_NAME_INVALID`、`NULL_WIFI_MAC_INVALID` 无执行统计

**根因**：这两条规则可能在后续脚本中未执行，或统计未补录。

**影响**：
- rebuild4 的基础数据治理页如果直接写"26 条规则"，会与实际执行不一致
- 用户可能追问为什么有规则没有统计

**解决方式**：
- rebuild4 必须拆开"规则定义层"和"规则执行层"
- 缺失的 2 条必须写明原因（未执行 / 统计未录 / 已废弃）
