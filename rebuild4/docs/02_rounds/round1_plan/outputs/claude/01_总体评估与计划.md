## 1. rebuild4 的核心定位

rebuild4 = 继承 rebuild2 已验证的数据解析/清洗/可信库研究成果，重新构建一个可持续运行、可研究流转参数、且系统完成后立即有数据可用的本地动态治理系统。

这句话有四个关键词，任何一个丢失都会导致整个系统偏航：

| 关键词 | 边界 | 如果丢失会怎样 |
|--------|------|---------------|
| 继承 rebuild2 已验证成果 | 字段审计、ODS 清洗、trusted 过滤、GPS 修正、信号补齐、画像与异常研究 | 系统看起来有结果，但用户无法解释"为什么是这些数据" |
| 动态数据库 | 初始化 + 增量 + baseline 版本承接，不是一次性静态重跑 | 退化为每次全量重跑的报表系统 |
| 可研究流转参数 | 用户可对比不同 run / batch / 场景 / 时间点的流转变化 | 只能看最终结果，无法调参观察 |
| 完成后立即有可用数据 | 不能只有 schema 和空页面 | 重复 rebuild3 "先有壳子后补数据"的偏移 |

## 5. 必须在本轮冻结的关键约束

以下约束如果不在第 1 轮明确，后续每一轮都会反复争论：

### 5.1 状态与资格体系

已冻结，rebuild4 不应重新发明：
- `lifecycle_state`：waiting / observing / active / dormant / retired / rejected
- `health_state`：healthy / insufficient / gps_bias / collision_suspect / collision_confirmed / dynamic / migration_suspect
- 三层资格：existence_eligible / anchorable / baseline_eligible
- `watch` 只能是 UI 派生态

### 5.2 四分流语义

四分流必须使用全称：
- fact_governed / fact_pending_observation / fact_pending_issue / fact_rejected
- 四分流合计 = fact_standardized（不变量，已验证）

### 5.3 baseline 原则

- 当前批次只读上一版冻结 baseline
- 当前批次不能一边判定一边改写本批 baseline
- baseline 刷新是批末行为

### 5.4 data_origin 合同

每个核心读模型必须返回：
- `data_origin`：real / synthetic / fallback
- `origin_detail`
- `subject_scope`

页面必须消费并展示来源状态。

### 5.5 schema 命名

- `rebuild4`：核心数据
- `rebuild4_meta`：控制元数据
- `rebuild2` / `rebuild2_meta` / `rebuild3*`：只读参考源
