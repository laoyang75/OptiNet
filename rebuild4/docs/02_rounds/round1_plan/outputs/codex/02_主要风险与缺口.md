# 主要风险与缺口

状态：round1 独立输出（Codex）  
更新时间：2026-04-06  
约束：仅基于 `rebuild4/docs` 输入材料与 `PG17 MCP`；本轮未访问页面

---

## 1. 风险总览

当前风险不是“信息缺失”，而是“已有信息没有被写成硬约束”。

按阻塞程度排序，rebuild4 当前最重要的缺口共有 9 项：

1. 主任务书仍未冻结为唯一执行入口
2. 主流程 real 数据标准未冻结，当前 rebuild3 现实仍以 synthetic/fallback 为主
3. rebuild2 字段与规则真相源存在口径分裂
4. trusted 过滤解释层尚未进入正式交付范围
5. 页面/API/表/来源矩阵尚未冻结
6. baseline 版本链的最小验收标准未冻结
7. compare/governance 的正式范围未裁决
8. `rebuild3_sample_meta` 等历史兼容包袱的正式归属未说明
9. Gate 尚未外化为单独停机线文件

---

## 2. 详细风险与缺口

### R-01 过早进入编码，重复 rebuild3 的“先实现后纠偏”路径

**结论**  
这是当前最高优先级流程风险；如果不先冻结唯一主任务书与 Gate，rebuild4 仍会重复 rebuild3 的偏航历史。

**依据**  
文档侧：`00_rebuild4_重构准备总说明.md` 与 `02_rebuild3_04系列偏航复盘.md` 都明确指出 rebuild3 的核心问题不是资料不足，而是执行阶段没有把真相源、Gate、矩阵做成硬停机线。  
数据库侧：无须新增数据库事实支撑该结论；这是流程结构性风险，而不是数据量问题。

**风险**  
会再次出现：

- 文档和实现并行演化
- synthetic/fallback 先进入页面主语
- 实施后再做多轮 repair

**应对**  
round2 必须先产出：

1. 唯一主任务书骨架
2. 唯一校验清单骨架
3. 独立 Gate 文件
4. 页面/API/表矩阵

**校验方式**  
检查 round2 输出中是否已经有独立 Gate 文件和唯一主任务书草案；若仍然只是长文档叙述，没有停机线，则该风险未关闭。

### R-02 主流程 real 数据标准未冻结，当前现实仍不足以直接承接正式主语

**结论**  
这是当前最高优先级数据风险；rebuild4 不能默认把现有 rebuild3 run/batch 快照直接视为正式主语。

**依据**  
PG17（2026-04-06）验证：

- `rebuild3_meta.run` = 4（1 条 `full_initialization`，3 条 `scenario_replay`）
- `rebuild3_meta.batch_snapshot` = 1,562 行，覆盖 142 个 batch
- `rebuild3_meta.baseline_version` = 4（1 条 `full_initialization`，3 条 `scenario_replay`）

文档侧：`rebuild3_round2_repair_execution_note.md` 明确说明当前仍然没有 real timepoint snapshot，也没有可作为 `/flow/overview` 主语的 real batch；`scenario_replay` 与 initialization 汇总快照仍按 synthetic 处理。

**风险**  
如果不先冻结 real 数据标准，后续极容易再次发生：

- synthetic 进入主流程页
- initialization 与 rolling 被混写
- baseline 看起来可比，实际上没有真实上一版

**应对**  
round2 必须明确写出：

1. real initialization 的定义
2. real rolling run 的定义
3. real `batch_snapshot` 的定义
4. real baseline diff 的前提

**校验方式**  
在 round2 中列出每个主流程页对应的最小 real 数据前提，并能用 PG17 复核是否满足。

### R-03 rebuild2 字段真相源分裂，不能直接继承旧文档口径

**结论**  
这是当前最高优先级解释层风险；如果不先重建字段质量真相表，rebuild4 会把历史口径冲突带入正式任务书。

**依据**  
PG17（2026-04-06）验证：

- `field_audit`：总字段 27，keep 17，parse 3，drop 7
- `target_field`：55
- `parse_rule`：0
- `compliance_rule`：0

文档侧：rebuild2 旧文档中仍存在 `keep 19 / parse 3 / drop 5`、`约 60 字段`、`parse_rule 64`、`compliance_rule 17` 等口径。

**风险**  
如果 round2 不先裁决，后续会出现：

- UI 文案与数据库事实不一致
- 治理页和说明页使用不同字段总数
- parse/compliance 规则无法落成正式元数据资产

**应对**  
建议把“字段质量真相表”单列为 round2 必做件：

1. 当前真实字段数
2. keep/parse/drop 明细
3. target field 正式清单
4. parse/compliance 元数据补建方案

**校验方式**  
以 PG17 为准重跑字段统计；所有说明文案必须能回指到元数据表，而不是只回指历史 Markdown。

### R-04 ODS 规则存在“定义层完整、执行层不完整”的闭环缺口

**结论**  
这是当前高优先级规则闭环风险；如果不在 round2 明确写清，后续治理解释仍然会有“规则存在，但没有执行统计”的断层。

**依据**  
PG17（2026-04-06）验证：

- active `ods_clean_rule` = 26
- 其中 `delete=1`、`nullify=22`、`convert=3`
- `ods_clean_result` 对 `l0_gps` 的 distinct rule_code = 24
- `ods_clean_result` 对 `l0_lac` 的 distinct rule_code = 24
- 已定义未进入执行统计的规则有 2 条：
  - `NULL_WIFI_MAC_INVALID`
  - `NULL_WIFI_NAME_INVALID`

**风险**  
如果 rebuild4 直接对外宣称“26 条规则已执行”，就是把未闭合规则包装成已完成事实。

**应对**  
round2 应把规则体系拆成两层：

1. 规则定义层
2. 规则执行统计层

并逐条写清：哪些已执行、哪些未进入闭环、缺口原因是什么。

**校验方式**  
对 `ods_clean_rule` 与 `ods_clean_result` 做差集校验；只有差集为 0，或差集已被明确登记，才算通过。

### R-05 trusted 过滤解释层缺失，会让新系统继续只展示结果、不解释损耗

**结论**  
这是当前高优先级业务可解释性风险；如果不把 trusted 过滤损耗变成正式页面/读模型能力，rebuild4 仍然无法回答“为什么这些数据被排除”。

**依据**  
PG17（2026-04-06）验证：

- `rebuild2.l0_lac` 总行数 = 43,771,306
- trusted 命中 = 30,082,381
- filtered out = 13,688,925（31.27%）
- filtered out 中仍有 `RSRP` = 12,017,352
- filtered out 中仍有 GPS = 11,350,552
- filtered out 主要来源组合：
  - `sdk / daa / cell_infos` = 7,015,342
  - `sdk / daa / ss1` = 5,951,068
  - `sdk / dna / cell_infos` = 722,515

**风险**  
如果只展示 final enriched / governed 结果，用户依然看不到：

- trusted 前后损耗多少
- 仍有有效 GPS/信号的数据为何被过滤
- 不同来源通道对损耗的贡献

**应对**  
将以下能力提升为正式范围：

1. 过滤漏斗
2. 来源损耗分布
3. 制式损耗分布
4. 最终结果反查上游过滤动作

**校验方式**  
对治理页/初始化页/API 逐项验证是否能返回 trusted 前后数量、损耗比例、来源拆分与规则追溯。

### R-06 页面/API/表/来源矩阵尚未冻结，UI_v2 仍有被“实现简化”稀释的风险

**结论**  
这是当前高优先级架构风险；如果没有矩阵，后续最容易退回“先把页面做出来，再补主语和来源说明”。

**依据**  
文档侧：`design_notes.md` 和 `03_三线重构策略.md` 已明确要求页面围绕对象/决策/事实/baseline 组织，并要求矩阵化。  
流程侧：`02_rebuild3_04系列偏航复盘.md` 已明确指出缺少矩阵是 rebuild3 反复偏航的核心原因之一。

**风险**  
会再次发生：

- overview 与 snapshot 的主语不同步
- 页面能打开，但无最小可用数据定义
- 页面没有下钻链路或出口

**应对**  
round2 必须冻结完整矩阵，并把它变成 round3 和编码阶段的硬依赖。

**校验方式**  
检查每个页面是否都能在矩阵中找到唯一主语、唯一 API、唯一数据底座和唯一验收方式。

### R-07 baseline 版本链最小验收标准未冻结，容易再次拿历史对照冒充“上一版”

**结论**  
这是当前中高优先级语义风险；baseline 页若不先冻结无上一版时的规则，会再次出现假 diff。

**依据**  
PG17（2026-04-06）验证：

- `baseline_version` 总数 = 4
- 其中 `full_initialization` baseline = 1
- 其中 `scenario_replay` baseline = 3

文档侧：repair 文档已明确禁止再用 rebuild2 结果冒充上一版 baseline 差异。

**风险**  
如果 round2 不提前冻结规则，后续很容易出现：

- 用 synthetic baseline 当 real previous baseline
- 用 rebuild2 对照结果当上一版 diff
- 页面展示有“版本差异”，但实际上不是同一语义链路

**应对**  
先裁决：

1. 只有 1 个 real baseline 时页面怎么展示
2. 什么时候才允许进入 baseline diff 功能
3. synthetic baseline 是否只保留为验证模式

**校验方式**  
核查 baseline 页面/API 是否能区分“当前版本展示”和“真实版本比较”两件事。

### R-08 compare/governance 的正式范围未裁决，支撑治理层容易变成隐式无限 scope

**结论**  
这是当前中高优先级范围风险；如果不在 round2 明确裁决，支撑治理层会再次变成“页面都存在，但不知道哪些必须 real”。

**依据**  
文档侧：`02_rebuild4_新增需求与数据准备清单.md` 已明确提出 compare/governance 至少二选一，要么建设 real 数据链，要么明确降级。  
当前现实：repair 文档仍把 `/compare`、`/governance` 视为 fallback，仅做透明化处理。

**风险**  
如果不裁决：

- 任务书会持续膨胀
- 主流程与支撑层抢优先级
- compare/governance 会以“先 fallback 后补 real”的方式再次拖累主系统

**应对**  
在 round2 明确裁决：

1. governance 是否属于最小成功标准
2. compare 是否属于最小成功标准
3. 若降级，降级后的诚实表达与验收方式是什么

**校验方式**  
检查最终任务书是否把 compare/governance 标注为“正式完成项”或“显式降级项”，不能保持模糊。

### R-09 `rebuild3_sample_meta` 等历史兼容包袱若不明确归属，会继续污染正式语义边界

**结论**  
这是当前中优先级边界风险；如果不在 round2 把历史兼容 schema/对象的正式归属写清，rebuild4 仍会留下“兼容资产临时可用、但没人知道它是否正式”的灰区。

**依据**  
文档侧：`Claude_field_baseline.md`、`Claude_reaudit_output.md` 与 `rebuild3_round2_repair_execution_note.md` 都反复提到 `rebuild3_sample_meta` 的存在与正式归属未清。repair 文档还明确把“继续清理 `rebuild3_sample_meta` 的正式归属”列为下一轮单独立项事项。

**风险**  
如果不裁决：

- 兼容 schema 可能再次被拿来充当正式主流程来源
- 文档里说的是一套语义，实际实现里又偷偷挂回 sample 资产
- round3 任务书很难定义哪些对象可删、哪些必须保留

**应对**  
在 round2 明确登记：

1. 哪些历史兼容 schema/表仍保留
2. 每个兼容资产的用途是验证、迁移过渡还是正式只读参考
3. 哪些资产禁止再进入正式主流程页

**校验方式**  
检查 round2 的 schema 边界与来源合同中，是否已经对兼容资产逐项标注“正式/只读参考/验证用途/废弃候选”。

## 3. 当前最需要优先关闭的 4 个缺口

如果只能先关 4 个缺口，建议顺序如下：

1. **R-01 唯一任务书 + Gate 文件化**
2. **R-02 real 数据标准冻结**
3. **R-03 字段质量真相表与 parse/compliance 承接方案**
4. **R-06 页面/API/表/来源矩阵**

原因：这 4 项不关，后面的 compare/governance、baseline diff、解释层页面都会继续建立在不稳定前提上。

---

## 4. round2 必须补齐的缺口清单

round2 完成后，至少要能回答下面这些“现在还不能正式回答”的问题：

1. rebuild4 的 real initialization / rolling / snapshot / baseline 到底各是什么
2. parse_rule / compliance_rule 到底是否重建为正式元数据表
3. governance 与 compare 哪个进入最小成功标准
4. baseline 无上一版时页面与 API 如何诚实表达
5. trusted 损耗解释层落在哪些页面/模块
6. 每个页面的主语、API、数据底座和验收方式是什么

---

## 5. 校验方式

本文件中的数据库事实，应以 `PG17 MCP execute_sql` 复核；重点复核以下 5 组：

1. `rebuild3` 事实层与对象层行数
2. `rebuild3_meta.run / batch / batch_snapshot / baseline_version`
3. `rebuild2_meta.field_audit / target_field / parse_rule / compliance_rule`
4. `rebuild2_meta.ods_clean_rule / ods_clean_result` 差集
5. `rebuild2.l0_lac` 对 `dim_lac_trusted` 的过滤损耗

只有当 round2 把这些事实转写成正式合同，并明确标记哪些是文档规则、哪些是数据库事实，本轮识别出的主要风险才算真正开始关闭。
