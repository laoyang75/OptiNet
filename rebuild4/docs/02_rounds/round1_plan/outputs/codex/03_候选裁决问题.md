## 1. 使用说明

以下问题不是“现在无法继续”，而是“进入 round2/round3 前最好由合并稿或人工裁决一次”，以避免后续再次出现范围漂移。

每个问题都给出 A / B / C 选项、推荐项、依据、风险和裁决后校验方式。

## D-001：`governance` 与 `compare` 哪个进入 rebuild4 的最小成功标准？

**问题**  
支撑治理层是否要求两个模块都建设 real 数据链，还是只要求其中一个进入最小成功标准？

**选项**

- **A**：`governance` 与 `compare` 都进入最小成功标准，二者都必须 real 接数
- **B**：`governance` 进入最小成功标准，`compare` 明确降级为验证辅助模块（推荐）
- **C**：二者都不进入最小成功标准，统一保留为显式降级模块

**推荐**  
推荐 **B**。

**依据**  

1. rebuild4 的新增硬要求之一，是把 rebuild2 的字段治理、ODS 清洗、trusted 损耗解释层正式纳入系统；这更接近 `governance` 的职责。
2. `compare` 在 UI_v2 中被明确降级为验证/对照模块，不应反客为主。
3. 当前 repair 文档已明确 `/compare`、`/governance` 仍是 fallback；若 round2 不做范围裁决，二者容易一起膨胀。

**风险**  

- 选 A：范围显著增大，主流程 real 数据链可能被拖慢
- 选 C：rebuild2 清洗解释层会缺少正式落点，新增需求 A 难以闭合

**裁决后校验**  
在 round2 的页面/API/表矩阵中检查：

- `governance` 是否已有 real 数据底座
- `compare` 是否被标注为正式项或降级项
- 对降级模块是否已写清 banner、数据来源和不计入完成标准的规则

## D-002：parse/compliance 规则应该如何进入 rebuild4 正式范围？

**问题**  
当前 `rebuild2_meta.parse_rule = 0`、`compliance_rule = 0`；rebuild4 是否要在 round2 就补建正式元数据底座？

**选项**

- **A**：round2 即冻结“补建正式元数据表”的方案，把文档/脚本规则转录为 `rebuild4_meta` 可查询资产（推荐）
- **B**：继续以 Markdown + 脚本为准，元数据表后补
- **C**：本轮不承接 parse/compliance，只承接 field_audit 与 ODS/trusted

**推荐**  
推荐 **A**。

**依据**  

1. PG17 当前事实：`parse_rule = 0`、`compliance_rule = 0`，说明这部分并没有现成的数据库真相层。
2. 如果继续只依赖 Markdown/脚本，rebuild4 仍然无法把解析/合规逻辑变成可查询、可审计、可回溯的正式能力。
3. rebuild4 本轮强调“不要把未验证推测写成事实”，这要求规则元数据尽量落库。

**风险**  

- 选 B：round2 会留下一个很大的“规则说明层缺口”
- 选 C：rebuild4 的数据解释链会天然断一段

**裁决后校验**  
检查 round2 是否产出：

1. parse/compliance 元数据表草案
2. 文档/脚本到元数据表的映射清单
3. 每条规则的最小字段结构与验收方式

## D-003：主流程 real 数据的最低验收门槛是什么？

**问题**  
rebuild4 完成时，主流程 real 数据到底最低要达到什么程度，才能算“完成后立即有数据可用”？

**选项**

- **A**：至少 1 条 real `full_initialization` run + 1 条 real rolling run（且该 run 下 >1 个 completed batch）+ 每个 completed batch 有 real `batch_snapshot` + baseline 链可追溯（推荐）
- **B**：只要求 real initialization；rolling 允许 synthetic 评估模式兜底
- **C**：只要页面能展示 synthetic/fallback，就算阶段完成

**推荐**  
推荐 **A**。

**依据**  

1. `02_rebuild4_新增需求与数据准备清单.md` 已明确把“系统完成后立即有可用数据”列为正式交付条件。
2. UI_v2 的主链路本质上依赖批次变化、时间快照和异常下钻，不可能只靠 initialization 单点就闭合。
3. rebuild3 的历史已经说明：如果 rolling 仍由 synthetic 占位，主流程页最终会回到“能看但不真实”的状态。

**风险**  

- 选 B：主流程 real 语义仍会断在 initialization 之后
- 选 C：rebuild4 直接重复 rebuild3 的核心问题

**裁决后校验**  
用 PG17 检查是否存在：

1. real initialization run
2. real rolling run
3. real completed batch 序列
4. real `batch_snapshot`
5. current baseline 的版本追溯链

## D-004：只有 1 个 real baseline 时，`/baseline` 页面应如何定义完成？

**问题**  
如果 rebuild4 首版交付时仍只有 1 个 real baseline，页面是“允许只展示当前版本”，还是“必须等到两版 real baseline 才算完成”？

**选项**

- **A**：允许只展示当前 real baseline；差异区明确显示“暂无上一版可比”（推荐）
- **B**：必须等到至少 2 个 real baseline，`/baseline` 才算正式完成
- **C**：允许用 rebuild2 或 synthetic baseline 暂代上一版

**推荐**  
推荐 **A**。

**依据**  

1. repair 文档已经明确禁止用 rebuild2 结果或 synthetic baseline 冒充上一版 diff。
2. “诚实空状态/诚实无对比”比“伪差异”更符合 rebuild4 的整体原则。
3. 若强制选 B，会把 baseline 页是否可上线与数据积累周期绑定过死，反而拖慢主流程交付。

**风险**  

- 选 B：会提高首次交付门槛，影响主流程先落地
- 选 C：会再次制造伪对比语义

**裁决后校验**  
检查 round2/round3 是否明确写出：

- 当前 baseline 展示必备字段
- 无上一版时的页面与 API 表达
- 进入 baseline diff 功能的前置条件

## D-005：rebuild2 资产在 rebuild4 中的复用边界是什么？

**问题**  
rebuild4 应把 rebuild2 资产视作“运行时真相”，还是“解释层与冷启动参考”？

**选项**

- **A**：rebuild2 资产只读复用，主要作为解释层与冷启动参考；rebuild4 的动态对象、事实、baseline 必须自己产出（推荐）
- **B**：直接把 rebuild2 的 enriched/final 结果作为 rebuild4 的长期运行真相
- **C**：在 rebuild4 开工前先把 rebuild2 全链路完全重做一遍，再开始动态治理

**推荐**  
推荐 **A**。

**依据**  

1. rebuild4 的正式定位是“继承 rebuild2 已验证成果”，不是“继续以 rebuild2 静态结果代替动态系统”。
2. 若选 B，rebuild4 会重新退化成“静态结果展示系统”。
3. 若选 C，范围会急剧膨胀，且不符合 round1/round2 的阶段目标。

**风险**  

- 选 B：动态 run/batch/baseline 语义被弱化
- 选 C：项目周期与复杂度大幅上升

**裁决后校验**  
检查 round2 的表/视图设计中：

- 哪些 rebuild2 表被标注为只读参考
- 哪些 rebuild4 表负责正式动态产出
- 哪些页面从 rebuild2 读解释层，哪些页面从 rebuild4 读正式主语

## D-006：Gate 应该采用什么粒度落盘？

**问题**  
rebuild4 是否沿用“在长任务书内描述 Gate”的写法，还是必须把 Gate 做成独立停机线文件？

**选项**

- **A**：每个 Gate 都独立成文，含通过条件、校验 SQL/API/页面、未通过停机规则（推荐）
- **B**：继续把 Gate 写在长任务书里，但不单独出文件
- **C**：不再强调 Gate，只保留推荐顺序

**推荐**  
推荐 **A**。

**依据**  

1. `02_rebuild3_04系列偏航复盘.md` 已明确指出 rebuild3 的问题之一，是 Gate 被写进 prompt 但没有成为真正停机线。
2. rebuild4 round1 的核心目标本来就是“明确哪些事必须先做，哪些事不能先做”。
3. 若 Gate 不独立，后续最容易被“先做会动的部分”绕开。

**风险**  

- 选 B：Gate 仍会退化成说明文
- 选 C：几乎等于放弃流程控制

**裁决后校验**  
检查 round2/round3 是否至少产出：

1. Gate 清单
2. 每个 Gate 的输入/输出
3. 通过标准
4. 未通过后的停机动作

## 2. 建议优先裁决顺序

如果合并稿需要控制裁决数量，建议按下面顺序优先裁决：

1. `D-001`：支撑治理层正式范围
2. `D-003`：real 数据最低验收门槛
3. `D-002`：parse/compliance 元数据策略
4. `D-004`：baseline 单版本完成定义
5. `D-005`：rebuild2 资产复用边界
6. `D-006`：Gate 文件化粒度

原因：前 3 项直接决定 round2 的范围和建模方式；后 3 项更多决定交付边界与流程控制方式。

---

## 3. 校验方式

这些裁决问题一旦拍板，应立即回写到 round2 细化设计里，并且至少反映到以下 4 处：

1. 页面/API/表矩阵
2. 数据准备计划
3. Gate 文件
4. 最终任务书的“最小成功标准”章节

如果裁决结果没有回写到这 4 处，后续仍会在实施阶段再次变成口头约定。
