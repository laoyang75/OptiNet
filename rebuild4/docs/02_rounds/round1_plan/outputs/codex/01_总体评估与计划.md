# 总体评估与计划

状态：round1 独立输出（Codex）  
更新时间：2026-04-06  
约束：仅基于 `rebuild4/docs` 输入材料与 `PG17 MCP`；本轮未访问页面

---

## 1. 结论

### 1.1 总判断

rebuild4 当前**可以进入第 2 轮细化设计**，但**不应直接进入编码**。

原因不是“资料不足”，而是“实施前必须先冻结的合同还没有被写成唯一可执行版本”。当前最需要先冻结的不是页面样式，而是下面 4 组前置合同：

1. **真相源合同**
   - rebuild3 Tier 0 业务/状态/版本语义
   - UI_v2 页面与主工作流
   - rebuild2 清洗/过滤/可信库解释层
2. **来源语义合同**
   - `real / synthetic / fallback`
   - 页面/API 的 `data_origin`、`subject_scope`、`subject_note`
3. **最小可用数据合同**
   - 哪些页面必须有 real 数据才算完成
   - 哪些页面允许 synthetic 评估模式
   - 哪些页面只能显式 fallback，且不计入最小成功标准
4. **页面到数据底座合同**
   - 页面 -> API -> 表/视图 -> `data_origin` -> 最小可用数据 -> 验收方式

换句话说，rebuild4 的 round1 不是“决定先写哪个页面”，而是“决定什么条件满足后才允许开始写”。

### 1.2 rebuild4 的最小成功标准

rebuild4 的最小成功标准建议冻结为以下 6 条；任一条未满足，都不应宣称 rebuild4 完成：

1. **唯一任务书已冻结**
   - 只认 `rebuild4/docs/03_final/01_rebuild4_最终执行任务书.md` 为唯一主任务书
   - 旧草案、调查稿、agent 输出不再并行生效
2. **主流程页来源语义已冻结且前后端一致**
   - `flow/overview`、`flow/snapshot`、`runs`、`initialization`、`baseline` 都明确 `real/synthetic/fallback`
   - synthetic 不能再占据正式主语位置
3. **rebuild2 清洗解释层已成为正式能力**
   - 至少能查询 keep/parse/drop、ODS 规则定义/执行、trusted 过滤损耗、来源损耗、结果追溯
4. **系统完成后立即有可评估数据**
   - 至少有 1 条 real `full_initialization` run
   - 至少有 1 条 real rolling run，且该 run 下有多于 1 个 completed batch
   - 每个 completed batch 都有 real `batch_snapshot`
5. **baseline 语义诚实可验**
   - 当前 baseline 可追溯到 `run_id / batch_id / rule_set_version`
   - 若暂无上一版 real baseline，页面必须诚实显示“无上一版可比”，不能用 rebuild2 或 synthetic 冒充
6. **支撑治理层范围已明确**
   - `governance`、`compare` 哪些进入本轮正式完成标准，哪些显式降级，已经在任务书中拍板

### 1.3 当前最关键的判断

当前 rebuild4 最值得利用的不是“rebuild3 页面已能跑”，而是**rebuild2 + rebuild3 已经把两个最重要的输入准备好了**：

- 一条可继承的数据治理解释链
- 一条已经被证明会跑偏的实施历史

因此 rebuild4 的核心目标不应再是“把 rebuild3 继续补齐”，而应是：

**继承 rebuild2 已验证的数据研究成果，吸收 rebuild3 的偏航教训，先冻结合同与最小数据标准，再进入动态治理系统重建。**

## 3. 必须先做什么，不能先做什么

## 3.1 必须先做的事

### P0. 冻结唯一真相源与唯一任务书路径

必须先把以下内容写进后续正式任务书：

1. Tier 0 / Tier 1 / Tier 2 / Tier 3 的优先级
2. 哪些文档只做输入，哪些文档具有执行效力
3. 最终唯一主任务书与最终唯一校验清单的路径

### P1. 冻结来源语义合同

必须先写清：

- 哪些读模型允许 `synthetic`
- 哪些读模型只允许 `real`
- 哪些读模型允许 `fallback` 但不计入正式完成
- 前端如何显示 `data_origin`

### P2. 冻结最小可用数据标准

必须先写清：

- real initialization 的最小定义
- real rolling run 的最小定义
- `batch_snapshot`、`baseline_version`、compare/governance 的完成边界

### P3. 冻结页面-数据映射矩阵

必须先形成矩阵：

- 页面
- API
- 来源表/视图
- 主语
- `data_origin`
- 最小可用数据
- 空状态规则
- 验收方式（SQL / API / Playwright）

### P4. 冻结 Gate，而不是只写“建议顺序”

必须把 Gate 拆成单独停机线，未通过不得继续，不再允许长 prompt 内部隐式推进。

## 3.2 不能先做的事

在上述 4 组合同冻结前，下面这些事都不应该先做：

1. 不应直接开始 rebuild4 编码
2. 不应先做页面壳子再倒推数据与主语
3. 不应默认让 synthetic 占据主流程页
4. 不应把 rebuild2 对照结果当“上一版 baseline”
5. 不应把旧文档冲突数字直接并列写进任务书
6. 不应把 compare/governance 的 fallback 历史包袱继续留成隐式行为

## 4. 建议的实施顺序

## 4.1 第 2 轮（细化设计）应该完成什么

round2 的目标不是写代码，而是把 round1 的判断细化为**可执行合同**。

建议 round2 至少拆成 5 个包，并按下面顺序推进：

### 包 A：数据语义与元数据合同

需要细化：

1. `run / batch / baseline / contract_version / rule_set_version` 的正式结构
2. `real / synthetic / fallback` 合同字段
3. rebuild2 解释层元数据的承接方式
4. parse/compliance 元数据是补建到 `rebuild4_meta`，还是只承接文档/脚本

### 包 B：数据准备合同

需要细化：

1. real initialization 数据从哪里来
2. real rolling batch 数据从哪里来
3. 哪些旧表只读复用，哪些结果必须 rebuild4 重算
4. `batch_snapshot` 的真实写入条件
5. baseline 刷新前提与无上一版时的页面表达

### 包 C：页面/API/表矩阵

需要细化：

1. 主流程层每个页面的主语
2. 页面对应 API
3. API 对应读模型表/视图
4. 空状态 / synthetic / fallback 展示规则
5. 页面下钻路径与入口出口

### 包 D：校验与 Gate

需要细化：

1. Gate 名称
2. 通过条件
3. SQL 校验项
4. API 校验项
5. 页面 Playwright 校验项
6. 未通过时的停机规则

### 包 E：支撑治理层边界

需要细化：

1. `governance` 是否进入最小成功标准
2. `compare` 是否进入最小成功标准
3. 若降级，降级到什么程度仍算诚实交付

## 4.2 第 3 轮（最终执行任务）应该完成什么

round3 应把 round2 已冻结的合同转成真正可执行的任务书：

1. 模块拆解
2. 依赖顺序
3. 验收口径
4. 回归口径
5. 不同模块之间的阻塞关系

## 4.3 真正进入编码后的推荐顺序

在 round3 冻结后，推荐按下面顺序执行，而不是按页面数量推进：

1. `M0`：schema / meta / 版本合同 / Gate 框架
2. `M1`：数据准备链路（initialization、rolling、snapshot、baseline）
3. `M2`：rebuild2 清洗解释层读模型
4. `M3`：主流程页读模型（overview、snapshot、runs、objects、workspaces）
5. `M4`：baseline / profile
6. `M5`：支撑治理层（governance / compare，按已裁决范围推进）

原因：如果 `M1` 和 `M2` 不先完成，后面的页面即使能显示，也大概率再次变成“看上去有结果、实际主语不真实”。

---

## 5. 第 2 轮必须继续细化的重点

round2 至少必须把下面 8 项写实、写细、写成可校验条目：

1. **rebuild4 的 real 数据定义**
   - initialization、rolling、snapshot、baseline 各自什么条件才算 real
2. **rebuild2 解释层承接方案**
   - `field_audit`、`target_field`、`ods_clean_rule`、`ods_clean_result`、trusted 损耗如何入新系统
3. **parse/compliance 元数据空表问题的处理方案**
   - 是补建元数据表，还是把脚本/文档正式转录成元数据
4. **页面 -> API -> 表/视图矩阵**
   - 尤其是 overview / snapshot / runs / initialization / baseline / governance / compare
5. **空状态与降级策略**
   - 哪些页面可以 synthetic，哪些页面只能 fallback，哪些页面必须空状态
6. **baseline 版本链策略**
   - 无上一版时如何展示；什么时候才允许做版本差异页
7. **Gate 文件化**
   - 不再把 Gate 隐含在长 prompt 里
8. **验收清单**
   - SQL、API、Playwright 三层都要落盘

## 6. 校验方式

## 6.1 round1 结论的校验方式

本文件的关键结论可由以下方式复核：

1. **文档复核**
   - 复核 Tier 0 / UI_v2 / rebuild2 / rebuild3_review 是否支持同一结论
2. **PG17 复核**
   - 复核事实层规模、run/batch/baseline 数量、字段治理元数据、ODS 规则差异、trusted 过滤损耗
3. **边界复核**
   - 检查本文是否把“文档结论”和“数据库事实”混写成同一来源

## 6.2 round2 的入场校验

只有当以下问题都被 round2 写清，才建议进入 round3：

1. 最小成功标准是否变成了可验收条目
2. 页面-数据矩阵是否已经成稿
3. 数据准备与 baseline 链是否已有正式方案
4. compare/governance 是否已被明确裁决
5. Gate 是否已经独立成文

---

## 7. 最终建议

rebuild4 的第一优先级应明确写成：

**先冻结合同，再准备 real 数据，再接主流程读模型，最后才做页面与支撑模块。**

如果 round2 不能把这条顺序写死，rebuild4 很容易再次回到 rebuild3 的偏移路径：

- 先有页面
- 再补主语
- 再补来源说明
- 最后才发现数据并没有准备好

这条路径在 rebuild3 已经验证过代价过高，rebuild4 不应重复。
