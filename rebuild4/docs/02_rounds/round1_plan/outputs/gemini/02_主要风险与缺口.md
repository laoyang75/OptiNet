## 1. 风险清单表与缺口评估

**结论**：rebuild4 当前主要面临数据源语义混淆、解析/合规规则元数据缺失、以及流程监控退化为静态报表三大核心风险。其中，数据清理执行统计与定义的不匹配是一个明显的缺口。

**依据**：
- `rebuild4/docs/01_inputs/02_research/01_rebuild2_字段质量调查.md` 明确指出 `parse_rule` / `compliance_rule` 数据库中为空 (0行)。通过 `PG17 MCP execute_sql` 查询证实 `rebuild2_meta.parse_rule` 等表确实未实例化。
- ODS清洗定义了 26 条规则，但在 `ods_clean_result` 的统计中只有 24 条有数据，存在 2 条数据的断层。
- `rebuild4/docs/01_inputs/03_ui_v2/design_notes.md` 中指出的风险1：首页容易退化为 KPI 仪表盘。

**风险**：
如果不在 rebuild4 中补建 `parse_rule` / `compliance_rule`，数据清洗说明将永远只能依靠口头传达或脚本硬编码，无法在工作台中进行审计。

**校验方式**：
要求最终产出中包含完整的规则定义元数据表，并通过前端 `基础数据治理` 页面真实呈现每条规则的过滤量与流向。

## 2. 每项风险的触发条件与预防方式

### 风险 A：真实数据与回退数据 (fallback/synthetic) 混淆

**结论**：必须彻底隔离这三种数据源的语义，不允许页面默认 fallback 冒充 real。

**依据**：`rebuild4/docs/01_inputs/01_foundation/02_rebuild4_新增需求与数据准备清单.md` 第 3 节。

**风险**：
如果 API 没有强制要求带有 `data_origin` 字段，且前端也没有强制根据该字段显示对应警告标识，就会触发数据主语漂移。
**触发条件**：开发为了快速跑通页面，直接复用 `scenario_replay` 数据作为主面板数据，且不标明 "synthetic 评估模式"。
**预防方式**：
在数据模型合同中强制要求所有响应包含 `data_origin` (枚举: real / synthetic / fallback)；前端遇到非 real 时必须在页面顶部强制渲染 GlobalStatusBar。

**校验方式**：
使用 Playwright 测试时，断言页面顶部存在状态说明条；拦截 API 请求验证 `data_origin` 字段的存在性。

### 风险 B：`parse_rule` 和 `compliance_rule` 元数据的实表缺口

**结论**：由于 rebuild2 阶段这些规则存在于文档而非数据库元数据中，直接开发工作台时会导致“基础数据治理”模块无数据可查。

**依据**：`rebuild4/docs/01_inputs/02_research/01_rebuild2_字段质量调查.md` 2.3节。

**风险**：
开发时发现缺数据，直接在前端写死静态表格或提示“暂无数据”。
**触发条件**：执行任务书未将“补建 parse_rule / compliance_rule”设为强制前置交付物。
**预防方式**：
在正式任务书中要求后端第一步先从旧代码/文档中梳理并 insert `rebuild4_meta.parse_rule` 等表的数据，确保基础元数据有记录。

**校验方式**：
`PG17 MCP execute_sql` 查询上述两表，行数必须大于 0。

### 风险 C：数据准备未被视为一等任务

**结论**：重构如果依然采用先开发 UI 再接入真实数据的方式，将导致极大的沟通与修补成本。

**依据**：`rebuild4/docs/01_inputs/01_foundation/01_rebuild4_核心输入整理.md` 3.3 与 5.3。

**风险**：
交付时空有 UI 框架，无法评估“流转总览”、“批次中心”以及 delta 变化。
**触发条件**：测试环境或开发环境缺乏一套完整的、带上下文 (run, batch, baseline) 的基准数据。
**预防方式**：
由数据线专门负责先行准备 real initialization 和 real rolling batch，这必须作为卡住 Gate A 的强制前置工作。

**校验方式**：
执行验收时，查询 `batch_snapshot` 表必须有对应的完整时间点记录，且四分流分布数字有意义。

## 3. 我可能遗漏的点

- `ods_clean_rule` 中缺失的两条规则 (`NULL_WIFI_NAME_INVALID`, `NULL_WIFI_MAC_INVALID`)，究竟是被废弃了还是脚本确实漏了统计，在文档中未提供明确的排查结果。这里存在潜在的隐患，可能在重跑 ODS 逻辑时导致总数对不上。
- 对于 `trusted` 过滤损耗，由于只统计了保留的与拦截的数量，并未明确“拦截数据如何在系统中二次复活”或归档机制。
