## 1. 候选裁决问题清单

以下问题直接影响后续开发边界和任务书的编撰，属于“必须通过人类裁决”的关键分歧点，而非开发人员可自行决定的实现细节。

### 1.1 ODS 规则遗漏的统计数据处置方式

**结论**：需要裁决 `ods_clean_rule` 中定义了但未执行统计的 2 条规则 (`NULL_WIFI_NAME_INVALID`, `NULL_WIFI_MAC_INVALID`) 在 rebuild4 中应如何展示。

**依据**：`rebuild4/docs/01_inputs/02_research/01_rebuild2_字段质量调查.md` 2.4节指出，当前定义了26条，但执行统计中只有24条。通过 `PG17 MCP execute_sql` 查询证实 `ods_clean_rule` 为 26，而 `ods_clean_result` 是针对24条覆盖的48条结果。

**风险**：
如果不裁决，后端 API 可能无法对齐总规则数和有效统计数，导致前端“基础数据治理”页的数据总数或者行项显示异常（如出现 NaN 或错乱）。

**候选选项**：
- **A**: 维持 26 条的元数据展示，对缺失统计的 2 条标记为 "未执行/未统计"，并展示为空或0。
- **B**: 将元数据定义裁剪为 24 条，让前后端展示保持完美一致，彻底放弃废弃的 2 条。

**校验方式**：
无论选择哪一项，都必须通过 API `GET /api/governance/ods_rules` 返回的具体对象列表及状态值来断言。

---

### 1.2 `parse_rule` / `compliance_rule` 缺失元数据的重建粒度

**结论**：需要裁决 rebuild4 中补建解析与合规规则元数据表的执行标准。

**依据**：`rebuild4/docs/01_inputs/02_research/01_rebuild2_字段质量调查.md` 第 2.3 节明确两表当前为空。

**风险**：
如果只是要求补建，开发者可能只插入极简的假数据以应付检查，从而使后续真正的清洗逻辑与这部分元数据仍然脱节。

**候选选项**：
- **A**: 强制要求根据 rebuild2 的已有 Python 脚本或原始文档，原样逆向抽取真实的 64条/17条 记录落表，且必须在重跑清洗时产生对应的 result 统计记录。
- **B**: 允许 rebuild4 初期先作为纯静态元数据进行只读展示，暂不要求它立刻与最新的数据清洗运行时强绑定，留待后续版本接轨。

**校验方式**：
通过 `PG17 MCP execute_sql` 查看这两个表的具体列与行数；若选A，还需查询是否有对应的 `_result` 表记录生成。

---

### 1.3 `fallback` 数据在 UI 中的展示容忍度

**结论**：需要裁决如果在 `/compare` 或 `/governance` 页面暂时无法供给真实 (real) 数据时，系统该如何处理。

**依据**：`rebuild4/docs/01_inputs/01_foundation/02_rebuild4_新增需求与数据准备清单.md` 第 3 节明确要求前置冻结 fallback 语义。

**风险**：
若不明确底线，开发者很可能依然默认展示 fallback 数据且不做明显标记，混淆用户视听。

**候选选项**：
- **A**: 绝对禁止 fallback。如果拿不到 real 或被允许的 synthetic 数据，直接呈现“空状态”或“无可用数据”画面。
- **B**: 允许降级呈现 fallback 数据，但前端必须全屏展示强烈的灰色警告水印，且右上角强制标记 `data_origin: fallback`。

**校验方式**：
利用 Antigravity 浏览器截取上述页面时进行视觉比对断言，确保空状态或水印明显存在。
