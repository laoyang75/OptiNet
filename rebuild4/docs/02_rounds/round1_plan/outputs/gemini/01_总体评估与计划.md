## 2. rebuild4 总体目标判断

**结论**：rebuild4 不是对 rebuild3 页面的修补，而是要求：**继承 rebuild2 已验证的数据解析/清洗/可信库研究成果，重新构建一个可持续运行、可研究流转参数、且系统完成后立即有数据可用的本地动态治理系统。**

**依据**：
- `rebuild4/docs/01_inputs/01_foundation/00_rebuild4_重构准备总说明.md` (第 2 节：rebuild4 必须重新对齐的核心定位)。

**风险**：
如果仍然按 "先写前端 UI 壳子，再随便塞 mock 数据" 的方式推进，将会重蹈 rebuild3 偏航的覆辙。由于未将 "准备好可用数据" 提升为一等公民任务，最终会导致系统虽然能跑通空数据，但无法评估真实流转。

**校验方式**：
最终验收时，检查系统必须存在 real initialization 数据和 real rolling batch 数据；检查 `/flow/overview` 等页面必须展示真实的四分流统计，不允许仅在 synthetic 下有数据。

---

## 3. 三阶段推进建议

**结论**：rebuild4 的开发与交付必须分为三个阶段严格推进：
1. **数据底座与元数据阶段**：优先复原并固化 rebuild2 的清洗、过滤、质量字典元数据。
2. **流转核心阶段**：基于真实/可靠的 baseline 构建 run、batch、snapshot 以及四分流。
3. **UI 与 API 映射阶段**：最后实现工作台，确保页面下钻链路和数据主语与 API 的完全对齐。

**依据**：
- `rebuild4/docs/01_inputs/02_research/03_三线重构策略.md` 指出应将重构拆解为数据清洗整理规则线、流转处理流程线、操作 UI 线。
- `rebuild4/docs/01_inputs/02_research/02_rebuild3_04系列偏航复盘.md` 指出必须用矩阵和 Gate 真正卡住执行路径。

**风险**：
如果三条线并行开发但不按顺序验收，会导致 UI 层先跑通但 API 喂的是假数据，或者流转核心缺乏真实清洗后的 trusted 源作为输入。

**校验方式**：
引入严格的 Gate 机制 (例如 Gate A, Gate B, Gate C)，前一个阶段必须通过 SQL / API / 文档的联合校验 (而不是仅仅口头通过) 才能进入下一阶段。

## 4. 必须前置冻结项

**结论**：进入开发前，必须冻结以下四项内容，不容更改：
1. 真实字段与规则清单 (55 个目标字段，27 个字段审计决策，26 条 ODS 规则)。
2. `real / synthetic / fallback` 数据来源的正式定义与隔离。
3. `UI_v2` 中规定的所有下钻路径及其最小可用数据集要求。
4. 页面 -> API -> 表/视图 -> 数据来源 -> 验收项 的映射矩阵。

**依据**：
- `rebuild4/docs/01_inputs/01_foundation/02_rebuild4_新增需求与数据准备清单.md`
- `rebuild4/docs/01_inputs/02_research/02_rebuild3_04系列偏航复盘.md`

**风险**：
如果上述项不在开发前彻底冻结，在执行中遇到卡点时，系统极其容易向 synthetic / fallback 模式降级，导致交付物不符合“立刻有数据可用”的目标。

**校验方式**：
开发执行书出台前，要求人类审阅一张完整的 "映射与冻结矩阵表"；没有该表的批准，直接阻断代码编写。
