## 裁决对 round2 的约束

### D-001 → Phase 1 / Phase 2 合同约束

- `rebuild4` 的正式 `full_initialization` run 输入源 = `rebuild2.l0_gps`（38,433,729 行）+ `rebuild2.l0_lac`（43,771,306 行）
- 不得引用 `rebuild3.fact_*` 或 `rebuild3.obj_*` 作为 rebuild4 的初始对象/事实
- Gate A 通过条件必须包含：至少 1 个 rebuild4 自产的 real `full_initialization` run 完成
- `rebuild3_meta.run`（4 行）/ `rebuild3_meta.batch_snapshot`（1,562 行）等保留为只读历史参考，不进入 rebuild4 主链

### D-002 → Phase 0 / Phase 4 合同约束

- rebuild4 前端为**全新 Vue 3 + TypeScript + Vite 项目**，不从 rebuild3 前端 fork
- 以下内容禁止从 rebuild3 直接复制：路由配置、Store（Pinia）结构、axios 拦截层、任何包含 `fallback` / `synthetic` 判断逻辑的代码
- 以下组件允许在通过逐组件 UI_v2 合规审计后复制：`LifecycleBadge`、`HealthBadge`、`DeltaIndicator` 等纯渲染展示组件
- 组件审计标准：该组件不包含数据获取逻辑、不包含 `data_origin` 判断、样式符合 UI_v2 design_notes

### D-003 → /governance 页 API 合同约束

- ODS 规则展示分两个 API 端点：
  - `GET /api/governance/ods_rules` — 返回全部 26 条规则定义（含 `NULL_WIFI_MAC_INVALID`、`NULL_WIFI_NAME_INVALID`）
  - `GET /api/governance/ods_executions` — 返回 24 条有执行统计的规则结果
- 前端 `/governance` 页面采用两个独立标签页（或展示区）分别呈现定义层和执行层
- 禁止将 26 条定义与 24 条执行统计混用在同一个 API 响应中，避免数字歧义

## 原始问题与选项记录

### D-001：rebuild4 的数据起点——继承 rebuild3 还是从 rebuild2 重新跑

**为什么必须裁决**

合并稿（`01_统一计划.md` §1.2 第 3 条）已明确"rebuild2/rebuild3 只作为只读参考源，rebuild4 正式动态对象/事实/run/batch 必须由 rebuild4 自己产出"，但**没有明确 rebuild4 的 Phase 1 到底从哪套原始数据起跑**。这个分歧直接决定：
- Phase 1 schema 设计的第一步内容（迁移 vs 初始化）
- Phase 2 流转主链的输入源
- Gate A 通过的数据前提
- 执行时间估算差异超过 1 个量级

**选项**

- **A（已选）：从 rebuild2.l0_gps / l0_lac 重跑 rebuild4 完整初始化**
  - 方案：rebuild4 自己做 real `full_initialization`，输入源为 rebuild2 的 l0 层，所有对象、事实、baseline 由 rebuild4 算出
  - 影响：数据语义完全清晰，无历史包袱；初始化需要时间，Phase 1 开工成本高；但这是唯一能让 rebuild4 真正闭合 real 主链的选项

- **B：以只读视图引用 rebuild3 的对象/事实层作为初始态，rebuild4 增量向前**
  - 方案：rebuild4_meta 中标记 data_origin=inherited；新 real rolling run 从此基础继续；rebuild3 的 73 个 scenario_replay batch 不继承
  - 影响：启动快；但继承了 rebuild3 的语义包袱，real 主链论证复杂

- **C：从 rebuild2.l0 重跑，但先用 1 天子集完成 real initialization，再扩量**
  - 方案：Phase 1 先用小窗口跑通全流转链路，通过 Gate A 后再扩到全量
  - 影响：兼顾速度和语义清晰；子集代表性有限，Gate A 条件需单独定义

**裁决：A** ✅

### D-002：前端工作台——从 rebuild3 继承改造，还是全新项目

**为什么必须裁决**

这是唯一会在 Phase 0（冻结任务书阶段）就必须写进合同的前端决策。一旦 Phase 4 开始，中途切换会导致大量返工。合并稿完全没有覆盖此问题。

**选项**

- **A：从 rebuild3 前端代码继承，按 UI_v2 逐页改造**
  - 方案：复用 rebuild3 的骨架、路由、Pinia store、共享组件；逐页替换为 UI_v2 设计
  - 影响：启动快；rebuild3 旧心智（fallback、synthetic 展示逻辑）可能污染新实现

- **B：全新前端项目，UI_v2 design_notes 为唯一设计输入，不引用任何 rebuild3 代码**
  - 方案：Vue 3 + TypeScript + Vite 全新搭建；rebuild3 代码完全不引用
  - 影响：干净起点；已验证组件需重写，工作量增加 1～2 周

- **C（已选）：新项目骨架，但允许逐组件审计后复制 rebuild3 的验证过的组件**
  - 方案：项目全新，路由/store/页面全部重写；符合 UI_v2 的纯展示组件可审计后复制
  - 影响：干净起点 + 保留部分复用收益；需逐组件审计，防止整体框架污染

**裁决：C** ✅

### D-003：ODS 清洗规则的定义层/执行层差集处置方式

**为什么必须裁决**

- PG17 事实：`ods_clean_rule` = 26 条定义；`ods_clean_result` 覆盖 24 个 rule_code（缺 `NULL_WIFI_MAC_INVALID`、`NULL_WIFI_NAME_INVALID`）
- 如果不裁决，API 层面临两种自相矛盾的实现选择，前端无法写出稳定的规则列表组件

**选项**

- **A：维持 26 条定义展示，对缺失统计的 2 条显式标记"未执行/未统计"**
  - 方案：API 返回 26 行，2 行 `execution_status = "not_executed"`；前端混合状态渲染
  - 影响：符合可审计原则；前端组件设计略复杂

- **B：将定义层裁剪为 24 条，彻底放弃 2 条已废弃规则**
  - 方案：软删除 2 条规则，API 和前端只展示 24 条
  - 影响：展示最简单；但掩盖历史事实，违反可审计原则

- **C（已选）：保留 26 条定义，将定义层和执行层分离为两个独立 API / 展示区**
  - 方案：`GET /api/governance/ods_rules`（26 条定义）+ `GET /api/governance/ods_executions`（24 条执行统计）；前端两区分开展示
  - 影响：API 无歧义，语义最清晰；前端需两区视图设计，工作量最大

**裁决：C** ✅
