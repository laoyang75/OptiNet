## 0. 任务书总判断

rebuild4 可以进入正式实施，但**不得并行散开做“全量 DDL + 全量 API + 全量前端”**。  
本轮统一执行顺序冻结为：

`Phase 0 / G0 合同冻结 -> Phase 1 / G1 schema 与元数据底座 -> Phase 2A / G2 real full_initialization -> Phase 2B / G3 real rolling -> Phase 3 / G4-G6 API 与治理合同 -> Phase 4 / G7 前端联调 -> Phase 5 总体验收与冻结`

统一执行口径如下：

1. 编码前先冻结唯一执行入口：本轮 `merged/` 两份统一稿进入 `03_final/` 后，才允许开始正式开发。
2. rebuild4 正式主链必须从 `rebuild2.l0_gps + rebuild2.l0_lac` 重跑；`rebuild3*` 只读参考，不得补丁式转正。
3. 主流程、对象、baseline、profile、initialization、governance 只允许 `real`；没有 real 就诚实空状态。
4. compare 保留，但只作为辅助验证模块；允许 `synthetic` / `fallback`，且必须显式 banner，不计入正式完成标准。
5. `data_origin / origin_detail / subject_scope / subject_note` 必须固定在 API 顶层 envelope，由前端真实消费并影响展示。
6. `parse_rule` / `compliance_rule`、asset 注册表、ODS 定义/执行差集、trusted 损耗统计都必须在 `rebuild4_meta` 正式落表。
7. `baseline_*` 必须按 `baseline_version` 追加存储；`classification_v2`、`gps_confidence`、`signal_confidence` 只允许以 `legacy_ref` 参考字段返回。
8. 任何步骤都必须同时具备执行动作、验收动作、停机动作和回退动作；缺一项视为任务书未完成。

## 3. 统一来源合同与页面/API/表/来源矩阵

### 3.1 统一 API envelope

```json
{
  "data_origin": "real | synthetic | fallback",
  "origin_detail": "run_id=RUN-..., batch_id=BATCH-...",
  "subject_scope": "batch | run | object | baseline | governance | compare | initialization",
  "subject_note": null,
  "context": {
    "run_id": "RUN-...",
    "batch_id": "BATCH-...",
    "baseline_version": "BASELINE-...",
    "contract_version": "rebuild4-contract-v1",
    "rule_set_version": "rebuild4-rule-v1"
  },
  "data": {}
}
```

冻结规则：

1. `data_origin`、`origin_detail`、`subject_scope`、`subject_note` 固定在 API 顶层，不能埋进 `data`。
2. 主流程、对象、baseline、画像、initialization、governance 只允许 `real`；没有 real 就返回诚实空状态。
3. compare 允许 `real` / `synthetic`；`synthetic` 必须 banner；`fallback` 仅允许返回诚实空 envelope，且 `data = null`。
4. 页面必须真实消费 `data_origin` 并影响展示，但无需为此发起新的方向性裁决；推荐承接方式是统一 context bar、空状态文案和 compare/governance banner。

### 3.2 页面 / API / 读模型 / 来源矩阵

| 页面 | 主语 | 主 API | 主要读模型 | 允许来源 | 最小可用数据 | 空状态规则 |
|---|---|---|---|---|---|---|

## 8. 裁决结论

**本轮无新增人类裁决问题。**

已直接在本文中冻结、无需再次上抛的争议点：

1. `data_origin` 是否需要页面承接：需要，由统一 context bar / 空状态 / banner 承接；这属于实现细节，不再单独裁决。
2. `full_initialization` 是否允许内部 checkpoint：允许作为执行引擎细节存在，但不改变“1 run + 1 logical batch”对外合同。
3. introspection 扫描范围是否只限 rebuild4：不只限；应覆盖 rebuild4 正式会读取的 schema，但仍不改变只读参考原则。
4. rolling 数据窗口如何切分：属于实施细节，只要满足 `data_origin=real`、completed batch `> 1` 与 Gate 通过条件即可。

只有出现以下异常时，才允许新建人类裁决文件：

1. `parse_rule` / `compliance_rule` 无法从现有资料提取出非空、可审计正式 seed；
2. 客观上无法形成满足 `G3` 的 real rolling 输入窗口；
3. 同一页面在已冻结矩阵下仍出现不可消解的一级主语冲突。

在未出现上述异常前，统一按本任务书与 `02_统一最终校验清单.md` 直接执行。
