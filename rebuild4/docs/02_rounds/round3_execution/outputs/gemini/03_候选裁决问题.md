# 候选裁决问题 (Gemini 独立输出)

## 结论
通过细致对齐文档及 UI_v2 的物理验证，只发现一处真正卡住“最终任务书”的实施悖论，急需裁决。

## 依据
- Antigravity Browser 对 10 份 UI_v2 HTML 的实地检查表明，所有页面均 **不存在* `data_origin` 和 `real/synthetic` 等来源提示位。
- `01_统一细化设计.md` 强制要求前端通过页面明确传递来源信息，这产生了明显断层。

## 风险
- 若按“遵循 UI_v2 作为基础页面”执行，必然导致缺失验证层字段；按“遵循元数据合同”执行，必须大规模重构 UI，增加工作量，存在交付跑偏的风险。

## 校验
- 人类介入判别，并下达后续任务书 UI 改造权限的优先级。

## 若失败如何处理
- 如果悬而未决，前端页面开发模块将被无限期 Block，等待上游达成共识。

## 必须要裁决的卡点问题

### D-001: UI_v2 前端页面是否必须引入重大的侵入性改造？
- **背景**：根据 API 及数据元数据“包A”的强校验要求，每个响应必须带有 `data_origin` (即 `real` / `synthetic` 等) 并在页面明示。但在前端 UI_v2 设计基线（如 `01_flow_overview.html`, `11_cell_profile.html`）中，当前页面并未给来源声明留出可用且醒目的视觉位置。
- **影响**：如不改造，无法满足“页面层追溯”的核心通过标准；若是大刀阔斧改造，偏离原有设计。
- **选项**：
  - **A**: 强逻辑优先。强制在所有页面的面包屑下方或左上角全局悬浮添加标牌，UI 人员必须按新合同重画组件。
  - **B**: 妥协型。保留原 UI_v2 主视觉和排版框架，仅在左下角、右上角等边缘信息模块通过 Hover 浮窗提供 `data_origin` 追溯提示，减少侵入感。
  - **C**: 搁置该硬线要求。认为该字段仅存在于 API 网络请求与 Network 抓包层即可，页面 UI 渲染免于强绑定。
