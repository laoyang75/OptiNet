## 1. 最终实施方向（阶段与步骤）

### 结论
实施必须严格按照：Phase 1 (Schema 与元数据) -> Phase 2A (Full Init) -> Phase 2B (Rolling) -> Phase 3 (API 读模型) -> Phase 4 (页面联调) 的顺序进行。严禁跨阶段先行。

### 依据
- 文档依据：根据 `01_统一细化设计.md` 规定的“同一实施顺序”，避免重蹈数据缺失的覆辙。
- PG17依据：`rebuild4` schema 为空，无法进行任何前端开发。

### 风险
- 若允许前端提前进场，将由于 `data_origin` 和真实数据的缺失，导致 Mock 数据泛滥，最终失去溯源能力。

### 校验
- 每一阶段必须强制阻断（Gate G1-G7），未通过上一阶段基线检查前，拒绝对下一阶段发放执行许可。

### 若失败如何处理
- 必须立刻停机并回滚当前 Phase 的改动。严禁使用 `synthetic` 数据或 mock 接口蒙混过关。

## 2. 分阶段执行步骤设计

### Phase 1: schema 与元数据底座
- **输入**：`02_统一细化校验基线.md` 中的 Schema 结构与 rebuild2 的治理事实。
- **输出**：在 PG17 创建 `rebuild4` 和 `rebuild4_meta` 库，并沉淀资产注册表（`asset_*`）、字段审计表等治理快照。
- **PG17/API/页面校验**：`SELECT COUNT(*) FROM information_schema.schemata` 必须为大于 0。
- **若失败如何处理**：回退 DDL 脚本，重新对齐 `02_统一细化校验基线.md` 中的设计字典。

### Phase 2A: Full Initialization (实测数据)
- **输入**：`rebuild2.l0_gps` 和 `rebuild2.l0_lac`。
- **输出**：1 个 completed logical batch；四分流与对象模型完成初始化；写出 `baseline v1`。
- **PG17/API/页面校验**：`SELECT COUNT(*) FROM rebuild4.fact_standardized` 必须与 l0 总行数守恒。验证首个基准版本。
- **若失败如何处理**：清空当前 batch ID 产生的所有写入记录，修复初始化管道 BUG 后重新走 `full_initialization`。

### Phase 2B: Rolling Batch
- **输入**：后续增量业务事实以及 `baseline v1`。
- **输出**：至少 1 个 completed rolling run 及其下的多个 completed batch。
- **PG17/API/页面校验**：完成批次大于 1 且各有 11 个 snapshot metric 数据落库。
- **若失败如何处理**：截断错误 Rolling batch 发起的事务，从上一个有效 checkpoint 恢复。

### Phase 3: 核心 API 读模型
- **输入**：完成初始化的对象实体表及流程统计表。
- **输出**：提供带有标准 `data_origin` envelope 的 API 端点（如 `/api/flow-overview`）。
- **PG17/API/页面校验**：API response JSON 的顶层必须清晰带有 `data_origin` 和 `subject_scope`。
- **若失败如何处理**：禁止联调，并强制 API 层开发人员重新包裹标准 envelope 合同。

### Phase 4: 前端联调与视图挂载
- **输入**：完整、真实、含 envelope 合同的后端 API。
- **输出**：所有页面均能真实展示 `real` 状态的内容及指标。
- **PG17/API/页面校验**：UI 必须正确显示 "real" 来源标识；所有降级必须在页面标出 Banner；无数据必须是诚实空状态。
- **若失败如何处理**：禁止发布，直至页面能准确呈现底层 `data_origin` 标签为止。
