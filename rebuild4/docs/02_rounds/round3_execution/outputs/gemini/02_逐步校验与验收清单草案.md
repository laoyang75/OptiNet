# 逐步校验与验收清单草案 (Gemini 独立输出)

## 结论
为了保证 rebuild4 不会退化为前端驱动的数字堆砌，验收清单必须覆盖文档、数据、API、页面及链路 5 层。任何一层的脱节都意味着任务不达标。

## 依据
- `01_统一设计细化.md` 明确规定了必须实现真实的数据流向记录。
- 实际页面检查显示，前端尚未显式承接 `data_origin` 等强制语义，这是核心缺口。

## 风险
- 如缺失 API 和页面的强制一致性检查，UI 开发者易回退到直接把 `synthetic` 数据包装为 `real`，从而击穿可信度底线。

## 校验
- 测试手段：使用 `execute_sql` 在 PG17 层面验证数据量，利用 `Playwright` 或 `Antigravity Browser` 捕捉 UI 层面是否可见来源印记。

## 若失败如何处理
- 如果任一单项校验未通过，视该项功能对应所处的 Phase 阻断，拒绝推进到下一阶段。

---

## 1. 文档检查
- [ ] ODS 和 Parse/Compliance 设计表是否已完整定义（包含对 rebuild2 历史资产的确凿承接）。
- [ ] Gate (G0-G7) 标准是否已脱离草案进入强制审查列表。
- [ ] COMPARE 模块是否已经显式降级为验证工具，移出核心关键路径。

## 2. 数据检查
- [ ] Schema 检查：`rebuild4` 与 `rebuild4_meta` 是否已被实例化。
- [ ] 初始化数据守恒：`fact_standardized` = 四分流（`governed + observation + issue + rejected`）。
- [ ] Snapshot 完整性：校验每个 completed batch 必须且只包含 11 行 core metric 快照。
- [ ] Trusted 损耗量：计算出的被过滤量和总计过滤占比（如 31.27%）应可独立呈现及核对。

## 3. API 检查
- [ ] Envelope 标准检查：所有 `/api/*` 返回体第一层级必须带有 `data_origin`、`origin_detail` 及 `subject_scope`。
- [ ] 降级检查：当缺乏有效 `real baseline` 时，必须返回 `previous_available = false`，禁止混淆。
- [ ] 治理注册表接口：`/api/governance/*` 必须返回来自真实表结构的扫描记录，而非人工写死的空数据。

## 4. 页面检查
- [ ] **可视化 `data_origin`**：流转快照及各类画像页面，必须在肉眼可见的位置声明数据来源于 `real`、`synthetic` 还是 `fallback`。
- [ ] **诚实空状态体现**：`/runs` 及 `/baseline` 在没有真实跑批时，应呈现空白并提示无数据，不允许展示 Mock 案例。
- [ ] **Legacy字段的降级显示**：确保旧版本分类等字段只存在于参考信息面板，非核心主维度内。

## 5. 链路检查
- [ ] Drill-down 穿透验证：`/runs` -> 锁定单一条目 -> 点击查阅 `/flow/overview` -> 定位某节点进入 `/flow/snapshot` -> 点击流转事实进入 `/objects` -> 点击单个实体到 `detail` -> 对比该实体所属基线记录 `/baseline`。
- [ ] 上述穿透链中，所有上下文参数（如 `batch_id`, `run_id`）必须在整个路由和 API 发起参数中保持连贯传递，不得中途中断。

---

## 我最可能遗漏的部分
- **权限及会话边界的连带测试**：在进行 API 链路层穿透时，可能忽视了因为跨域请求或 Token 环境引发的数据串扰，导致合成跑批数据污染正式视角。
- **并发环境下的快照竞态**：11 步的 Batch 推进在高并发 Rolling 下是否会产生脏视图，这份静态校验单难以直接保证原子性验证。
