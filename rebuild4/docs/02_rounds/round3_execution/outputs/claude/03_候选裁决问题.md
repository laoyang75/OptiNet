## 1. 候选问题检查记录

以下是本轮编写任务书过程中识别的"可能产生多解释"的点，以及为什么最终不需要上抛：

### D3-001：full_initialization 的执行引擎是否需要 checkpoint

**为什么检查**：`rebuild2.l0_gps`（3,843 万行）+ `rebuild2.l0_lac`（4,377 万行）的完整初始化在单节点 PostgreSQL 上可能耗时较长。如果中间失败，是否需要 checkpoint / 分段提交。

**为什么不上抛**：round2 合并稿 §1.1.5 已明确："若实施阶段需要 checkpoint、分片、分段提交，那是执行引擎内部机制，不进入 `batch` 语义层，不改变 Gate B。" 这意味着 checkpoint 是纯实现细节，不影响合同层面的 "1 run + 1 logical batch" 语义。

**任务书处理**：Phase 2A 只要求 1 条 completed run + 1 个 logical batch。执行引擎内部如何分片由实施者决定。

### D3-002：parse_rule / compliance_rule 的 seed 数据来源清单

**为什么检查**：round2 合并稿要求"seed 脚本从现有文档 / SQL / Python 提取"，但没有列出具体哪些文档、哪些脚本文件。

**为什么不上抛**：这是实施细节而非方向性裁决。seed 的目标表结构已在 round2 校验基线 §2.3 冻结（`parse_rule` 和 `compliance_rule` 的完整字段清单）。实施者只需遍历 `rebuild4/docs/01_inputs/04_reference/rebuild2_baseline/` 下的文档和 `rebuild2` 现有 SQL/Python 脚本提取规则。

**任务书处理**：Phase 1 G1-S9/G1-S10 要求非空即可。具体 seed 方式由实施者根据可用资料自主选择。

### D3-003：governance asset 注册表的 introspection 范围

**为什么检查**：round2 合并稿 §1.1.10 冻结"数据库 introspection 建真实基数"，但未明确 introspection 的范围是只扫描 `rebuild4` / `rebuild4_meta`，还是也包含 `rebuild2` / `rebuild3`。

**为什么不上抛**：round1 合并稿 §1.2.3 已冻结"rebuild2 / rebuild2_meta / rebuild3* 只作为只读参考源"。governance 的 asset 注册表应反映 rebuild4 实际使用的全部资产（包括只读引用的旧 schema 表），因此 introspection 范围应包含所有 rebuild4 会读取的 schema。这不改变任何方向性决策。

**任务书处理**：Phase 1 步骤 9 的 introspection seed 应覆盖 `rebuild4` / `rebuild4_meta` / `rebuild2` / `rebuild2_meta` / `rebuild3` / `rebuild3_meta`。

### D3-004：前端组件审计的具体标准

**为什么检查**：D-002 裁决允许"经逐组件审计后复制 rebuild3 的验证过的组件"，但审计标准写得较高层（不包含数据获取逻辑、不包含 data_origin 判断、样式符合 UI_v2）。实施时可能产生"这个组件算不算纯渲染"的灰区。

**为什么不上抛**：这是 Phase 4 的实施细节。round1 裁决 D-002 已经给出了充分的三条审计标准。如果实施中遇到边界 case（如组件内部含有 `computed` 属性引用 store），直接判定为"非纯渲染"，不复制即可。这不影响任何前置 Phase 或 Gate。

**任务书处理**：Phase 4 步骤 3 保留审计标准原文，不需要扩展。

### D3-005：rolling batch 的数据窗口来源

**为什么检查**：round2 合并稿冻结了 rolling 的合同（`run_type = rolling`，多个 completed batch），但未明确 rolling batch 的数据窗口来自哪里。是继续读 `rebuild2.l0_*` 的不同时间切片，还是需要新的输入源。

**为什么不上抛**：这取决于 rebuild4 部署后的实际数据流入方式。当前阶段，为了满足 Gate G3（至少 1 条 completed rolling run + completed batch > 1），实施者可以将 `rebuild2.l0_*` 按时间窗口切分成多个 batch 来模拟 rolling。这是数据准备细节，不改变来源合同（`data_origin = real`，因为数据本身确实来自 real l0 层）。

**任务书处理**：Phase 2B 只约束 Gate 通过条件，不强制指定窗口切分策略。
