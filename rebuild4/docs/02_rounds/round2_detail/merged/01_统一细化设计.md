## 1. 本轮统一冻结结论

### 1.1 结构与方向结论

1. **rebuild4 必须从新 schema 启动。**
   - `PG17` 复核：当前不存在 `rebuild4`、`rebuild4_meta` schema。
   - 统一结论：round3/实施阶段必须先建 schema，再建表、API、Gate；不允许在 `rebuild3*` 上继续补丁式演进。

2. **real 主链必须从 `rebuild2.l0_gps + rebuild2.l0_lac` 重跑。**
   - `PG17` 复核：`rebuild2.l0_gps = 38,433,729`，`rebuild2.l0_lac = 43,771,306`。
   - 统一结论：D-001 已裁决的“双输入重跑”继续有效；`rebuild3.fact_*`、`rebuild3.obj_*`、`rebuild3_meta.*` 只保留为历史参考，不进入 rebuild4 正式主链。

3. **来源合同必须前置为 API 顶层合同，而不是隐藏语义。**
   - `PG17` 复核：`rebuild3` / `rebuild3_meta` 当前没有 `data_origin`、`origin_detail`、`subject_scope` 三类列；`rebuild3.fact_standardized.sample_scope_tag` 全量 `43,771,306` 行均为 `NULL`。
   - 统一结论：rebuild4 的来源语义由 API 顶层 envelope + `run` / `batch` / `baseline_version` / `compare_job` 元数据共同承接；不再靠 `run_type`、`scenario_key`、`sample_scope_tag` 间接猜测。

4. **`run_type` 冻结为 3 个正式值。**
   - 冻结值：`full_initialization`、`rolling`、`scenario_replay`。
   - 统一结论：不预留 `other`；后续若真出现新 run 类型，必须先修订合同与 Gate，再改 DDL / API /前端。

5. **`full_initialization` 对外合同冻结为“1 个 real run + 1 个 logical batch”。**
   - 统一结论：初始化语义必须保持单批次、单基线起点；若实施阶段需要 checkpoint、分片、分段提交，那是执行引擎内部机制，不进入 `batch` 语义层，不改变 Gate B。

6. **`batch_snapshot` 冻结为 4 个 stage + 11 个必备 metric。**
   - `PG17` 复核：`rebuild3_meta.batch_snapshot = 1,562`，覆盖 142 个 batch；stage 只有 `input / routing / objects / baseline`；metric 只有 11 个：
     `fact_standardized`、`fact_governed`、`fact_pending_observation`、`fact_pending_issue`、`fact_rejected`、`obj_cell`、`obj_bs`、`obj_lac`、`baseline_cell`、`baseline_bs`、`baseline_lac`。
   - 统一结论：v1 先冻结这 11 项为 Gate 基线；未来若要新增 metric，必须走合同变更，不能静默扩展。

7. **baseline 历史能力冻结为“按 `baseline_version` 追加存储 + current view”。**
   - `PG17` 复核：`rebuild3_meta.baseline_version = 4`，但 `rebuild3.baseline_cell`、`baseline_bs`、`baseline_lac` 当前都只有 `BASELINE-FULL-V1` 这 1 个实体版本。
   - 统一结论：rebuild4 的 `baseline_*` 必须按版本追加存储，并补 `baseline_diff_summary` / `baseline_diff_object`；再提供 current view 给 API 使用。不得只保留 current、再靠临时 diff 现算。

8. **governance 进入正式完成标准，compare 保持降级模块。**
   - `PG17` 复核：`rebuild3_meta.compare_result = 0`；`asset_field_catalog`、`asset_table_catalog`、`asset_usage_map`、`asset_migration_decision` 全为 `0`。
   - 统一结论：governance 不能继续停留在 mock 或空注册表；compare 允许存在，但仅作为辅助验证，不计入正式完成标准。

9. **parse / compliance 规则的正式承接位置冻结在 `rebuild4_meta`。**
   - `PG17` 复核：`rebuild2_meta.parse_rule = 0`、`rebuild2_meta.compliance_rule = 0`；但两表已有既定列结构。
   - 统一结论：rebuild4 直接沿用旧表字段语义并补版本字段，在 `rebuild4_meta` 建正式表；`rebuild2_meta` 保持只读参考，不回写，不把 Markdown/脚本继续当唯一真相源。

10. **governance 注册表填充方式冻结为“数据库 introspection 建真实基数 + 人工补充语义”。**
    - 统一结论：字段/表目录、主键、层级、是否存在等事实来自数据库；`usage_desc`、`migration_decision`、`semantic_desc` 等业务语义允许人工补充。拒绝纯人工台账，也拒绝等待代码扫描作为 phase1 前提。

11. **trusted 损耗查询冻结为预聚合读模型，不走实时大表联查。**
    - `PG17` 复核：`l0_lac` 总量 `43,771,306`，trusted 命中 `30,082,381`，被过滤 `13,688,925`（`31.27%`）。
    - 统一结论：治理页使用 `trusted_loss_summary` + `trusted_loss_breakdown` 等预聚合结果；初始化完成后和每个 rolling batch 完成后刷新，不把 1,300 万级过滤结果的多维统计留给页面实时查询。

12. **异常工作台读模型必须拆粒度。**
    - `PG17` 复核：`rebuild3_meta.batch_anomaly_summary` 同时混有 `cell_object`、`bs_object`、`lac_object` 与 `record` 级条目。
    - 统一结论：rebuild4 拆成 `batch_anomaly_object_summary`、`batch_anomaly_record_summary`、`batch_anomaly_impact_summary`；不继续沿用单张混合表。

13. **BS / Cell 画像的旧分类字段只保留为解释层。**
    - 输入文档已冻结：`classification_v2`、`gps_confidence`、`signal_confidence` 必须降级为灰色参考信息。
    - 统一结论：这些字段不进入 rebuild4 核心状态模型，不进入 `obj_bs` / `obj_cell` 主状态字段；API 仅以 `legacy_ref` 嵌套返回，页面以灰色参考列 + 详情区“rebuild2 参考信息”展示。

14. **本轮合并不再新增 round2 人类裁决文件。**
    - 三家候选问题中，真正影响方向和结构的条目已在本稿内冻结：run 枚举、initialization 批次语义、snapshot 枚举、baseline 历史策略、governance 填充方式、parse/compliance 承接位置、异常粒度、compare 语义、legacy 字段位置均已收口。

## 3. 数据语义与元数据合同（包 A）

### 3.1 实体分层

| 层级 | 正式实体 | 说明 |
|---|---|---|
| 版本与运行控制层 | `contract_version`、`rule_set_version`、`source_adapter`、`run`、`batch`、`initialization_step_log` | 控制一次运行到底绑定哪版合同、哪版规则、哪类输入 |
| 主业务层 | `fact_standardized`、四分流事实表、`obj_cell`、`obj_bs`、`obj_lac`、`baseline_version`、`baseline_*` | rebuild4 正式主链 |
| 治理解释层 | `field_audit_snapshot`、`target_field_snapshot`、`ods_rule_snapshot`、`ods_execution_snapshot`、`parse_rule`、`compliance_rule`、`asset_*`、`trusted_loss_*` | governance 正式底座 |
| 异常与快照层 | `batch_snapshot`、`batch_flow_summary`、`batch_transition_summary`、`batch_anomaly_*` | 首页、快照页、异常工作台读模型 |
| 验证层 | `compare_job`、`compare_result` | compare 降级模块专用 |
| Gate 层 | `gate_definition`、`gate_check_item`、`gate_run_result` | Gate 独立成文与可执行承接 |

### 3.2 统一来源合同

所有核心 API 统一使用顶层 envelope：

```json
{
  "data_origin": "real | synthetic | fallback",
  "origin_detail": "来源说明",
  "subject_scope": "run | batch | object | baseline | governance | compare | initialization",
  "subject_note": null,
  "context": {
    "run_id": "RUN-...",
    "batch_id": "BATCH-...",
    "baseline_version": "BASELINE-...",
    "contract_version": "rebuild4-contract-v1",
    "rule_set_version": "rebuild4-rule-v1"
  },
  "data": {}
}
```

冻结规则：

| 页面层 | 允许 `data_origin` | 备注 |
|---|---|---|
| 主流程层、对象层、baseline、画像、initialization、governance | `real` | 没有 real 就诚实空状态 |
| compare | `real` / `synthetic` | synthetic 必须 banner |
| compare 的降级空响应 | `fallback` | 仅表示“当前无法形成可比结果”，不表示有 fallback 数据集 |
| launcher / 运维健康检查 | 不纳入本合同 | 独立于数据主链 |

### 3.3 版本与来源绑定合同

1. `contract_version` 与 `rule_set_version` 必须是显式实体，不能只是 run 表中的散落字符串。
2. `run` 必须绑定：`run_type`、`contract_version`、`rule_set_version`、`source_adapter`、`data_origin`。
3. `batch` 必须绑定：`run_id`、`batch_type`、`batch_seq`、`window_start/end`、`status`。
4. `baseline_version` 必须绑定：`run_id`、`batch_id`、`rule_set_version`、`refresh_reason`、`is_current`。
5. 正式主链业务表通过 `run_id` / `batch_id` / `baseline_version` 回溯来源；不要求在每张业务主表重复存储 `data_origin`，但所有 read model 与 API 必须可无损推出来源。

### 3.4 rebuild2 解释层承接合同

必须冻结两类承接方式：

1. **一次性冻结快照，之后只读**
   - `field_audit_snapshot` ← `rebuild2_meta.field_audit`（27）
   - `target_field_snapshot` ← `rebuild2_meta.target_field`（55）
   - `ods_rule_snapshot` ← active `rebuild2_meta.ods_clean_rule`（26）
   - `ods_execution_snapshot` ← `rebuild2_meta.ods_clean_result`（48）

2. **rebuild4 自建正式元数据**
   - `parse_rule`
   - `compliance_rule`
   - 规则 seed 脚本从现有文档 / SQL / Python 提取，产物落到 `rebuild4_meta`，并挂 `rule_set_version`。

统一结论：不采用“Admin 人工补录”作为唯一正道；人工补录只能作为校对/修订辅助，不能取代正式 seed 流程。

### 3.5 legacy 参考字段合同

1. `classification_v2`、`gps_confidence`、`signal_confidence` 不属于 rebuild4 主状态。
2. 这三类字段不进入 `obj_bs` / `obj_cell` / `obj_lac` 的核心状态列。
3. API 如需返回，只能放在 `legacy_ref` 内，例如：

```json
{
  "data": {
    "item": {
      "health_state": "healthy",
      "anchorable": true,
      "legacy_ref": {
        "classification_v2": "dynamic_bs",
        "gps_confidence": "medium",
        "signal_confidence": "high",
        "source_note": "rebuild2 reference"
      }
    }
  }
}
```

4. 页面上仅 BS / Cell 画像允许灰色参考列；对象主状态、筛选条件、Gate 均不得依赖这些 legacy 字段。

## 4. 数据准备合同（包 B）

### 4.1 real `full_initialization` 合同

#### 对外语义

- `run_type = full_initialization`
- `data_origin = real`
- 输入源固定为 `rebuild2.l0_gps + rebuild2.l0_lac`
- 1 个 logical run 下只允许 1 个 logical batch（`batch_seq = 1`）
- 完成后必须生成首个 real `baseline_version`

#### 页面必须看到的 11 个步骤

初始化页的 11 步只冻结“步骤语义”，不接受 UI mock 数字作为事实：

1. 冻结 `run_id` 与版本绑定
2. 全量标准化事件
3. 记录级合规检查
4. 研究期 LAC 预边界/区域缩边界
5. Cell 候选累计与晋升
6. 由 active Cell 派生 BS
7. 由 active BS 派生 LAC
8. 首轮事实治理
9. 异常检测与资格收敛
10. 完整回归一次
11. 生成 baseline v1

#### 输出要求

必须同时写入：

- `fact_standardized`
- `fact_governed`
- `fact_pending_observation`
- `fact_pending_issue`
- `fact_rejected`
- `obj_cell` / `obj_bs` / `obj_lac`
- `batch_snapshot`
- `batch_flow_summary`
- `batch_transition_summary`
- `batch_anomaly_*`
- `baseline_version` 与 `baseline_*`
- `initialization_step_log`
- `trusted_loss_summary` / `trusted_loss_breakdown`

#### 特别说明

`12_initialization_doc.md` 中的 `12,782 / 3,382 / 152` 等数字是 UI mock，不是 round2 冻结事实；正式页面只能读 rebuild4 实际 run 结果。

### 4.2 real `rolling` 合同

1. `run_type = rolling`。
2. 一个 rolling run 下必须有多个 completed batch；round3 Gate 要求 `completed batch > 1`。
3. 每个 completed batch 都必须写：
   - `batch_snapshot`
   - `batch_flow_summary`
   - `batch_transition_summary`
   - `batch_anomaly_*`
   - `baseline_refresh_log`
4. rolling batch 只读取上一版冻结 baseline；当前 batch 计算出的 baseline 只服务后续 batch。

### 4.3 baseline 诚实合同

1. `baseline_*` 按 `baseline_version` 追加存储，主键为 `(baseline_version, object_id)`。
2. `baseline_current_*` 可以是 view，但正式历史只能从 `baseline_*` 实体表读取。
3. `GET /api/baseline/current/diff` 必须返回：
   - `previous_available = true/false`
   - `diff_summary`
   - `diff_objects`（必要时分页）
4. 若当前只有 1 个 real baseline，则：
   - `previous_available = false`
   - `subject_note = 暂无上一版 real baseline 可比`
   - 不允许拿 rebuild2 / rebuild3 / synthetic baseline 冒充上一版

### 4.4 governance / trusted 合同

1. governance 页必须同时有资产注册、字段治理、ODS 规则、parse/compliance、trusted 损耗 5 类正式能力。
2. trusted 损耗只读取预聚合结果：
   - `trusted_loss_summary`
   - `trusted_loss_breakdown`
3. `trusted_loss_breakdown` 至少冻结两个维度：
   - `tech`
   - `source_combo`
4. `filtered_with_rsrp`、`filtered_with_gps` 作为 summary 主指标单独保留。

## 5. 页面 / API / 表 / 来源矩阵（包 C）

| 页面 | 主语 | 主 API | 主要读模型 / 表 | 允许来源 | 最小可用数据 | 诚实空状态 / 降级规则 |

| 异常工作台 | 对象级 / 记录级异常 | `GET /api/anomaly-workspace`、`GET /api/anomaly-workspace/summary` | `batch_anomaly_object_summary` + `batch_anomaly_record_summary` + `batch_anomaly_impact_summary` | `real` | 异常摘要表非空 | 为空时显示“当前无异常” |
| baseline 中心 | 当前 baseline 与版本差异 | `GET /api/baseline/current`、`GET /api/baseline/current/diff`、`GET /api/baseline/history` | `baseline_version` + `baseline_*` + `baseline_diff_*` | `real` | 至少 1 个 real baseline | 无 baseline 时显示“尚未产出基线版本” |
| LAC / BS / Cell 画像 | 画像对象 | `GET /api/profiles/lac|bs|cell` | `profile_*` + `obj_*` + `baseline_*` + `legacy_ref` | `real` | 对应 profile 读模型非空 | 无 profile 时显示“暂无画像数据” |
| 基础数据治理 | 资产 / 字段 / 规则 / 损耗 | `GET /api/governance/*` | `asset_*` + `field_audit_snapshot` + `target_field_snapshot` + `ods_*` + `parse_rule` + `compliance_rule` + `trusted_loss_*` | `real` | Gate G5 全部通过 | 元数据尚未初始化时显示“元数据快照尚未初始化” |
| 验证/对照 | compare 任务 | `GET /api/validation/compare` | `compare_job` + `compare_result` | `real` / `synthetic`；仅空响应可 `fallback` | 至少两次可比运行，或显式降级 | 不足两次运行时返回 fallback 空 envelope + banner |
| 初始化数据 | full initialization run | `GET /api/initialization/latest`、`GET /api/initialization/:run_id` | `run` + `batch` + `initialization_step_log` + `batch_snapshot` | `real` | 至少 1 条 completed full init | 无 initialization 时显示“尚未执行初始化” |

补充说明：

1. governance 页不使用“一个巨型 API”，而是按模块 lazy-load 多个 endpoint。
2. compare 页允许 synthetic 结果，但 synthetic 只能作为显式评估模式，不允许静默出现。
3. launcher 属于运维运行页，不纳入本轮数据合同与 Gate 成功标准；只需要独立健康检查接口。

## 6. 支撑治理层边界与执行顺序（包 E）

### 6.1 governance 正式范围

governance 至少要正式交付以下 9 组能力：

1. 概览统计：表数、字段数、使用数、待迁移数
2. 字段目录：`asset_field_catalog`
3. 表目录：`asset_table_catalog`
4. 使用关系：`asset_usage_map`
5. 迁移状态：`asset_migration_decision`
6. 字段审计 / 目标字段：`field_audit_snapshot`、`target_field_snapshot`
7. ODS 定义 / 执行：`ods_rule_snapshot`、`ods_execution_snapshot`
8. parse / compliance：`parse_rule`、`compliance_rule`
9. trusted 损耗：`trusted_loss_summary`、`trusted_loss_breakdown`

### 6.2 compare 降级边界

1. compare 不得承担主流程入口职责。
2. compare 的结果只服务“验证 / 对照”，不反向写回主链业务表。
3. compare 可以有 `synthetic` 结果，但必须：
   - 顶层 `data_origin = synthetic`
   - 页面 banner 明示“仅供辅助验证”
4. compare 只有在“当前请求无法形成真实对比结果”时，才允许返回 `data_origin = fallback` 的空 envelope；此时 `data = null`，不得伪造结果列表。

### 6.3 统一实施顺序

1. **Phase 0**：冻结 round2 合并稿、矩阵、Gate
2. **Phase 1**：创建 `rebuild4` / `rebuild4_meta` schema，落版本表、治理快照表、parse/compliance、asset 注册表
3. **Phase 2A**：跑 real `full_initialization`
4. **Phase 2B**：跑 real `rolling`
5. **Phase 3**：实现读模型 API 与来源 envelope
6. **Phase 4**：新前端项目联调（承接 D-002）
7. **Phase 5**：Gate + Playwright 验收，生成 round3 最终执行任务书
