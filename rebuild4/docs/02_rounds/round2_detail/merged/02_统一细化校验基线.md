# 统一细化校验基线

状态：round2 合并稿  
更新时间：2026-04-06  
合并依据：三家 round2 输出交叉比对；涉及计数、差集、结构现状的条目已回到 `PG17 MCP execute_sql` / `search_objects` 复核。  
用途：把 round2 统一设计收敛为 DDL / API / Gate / SQL 可直接执行的校验基线。

## 1. 冻结的命名、枚举与公共约束

### 1.1 统一命名原则

1. **外部业务标识统一使用稳定文本 ID。**
   - `run_id`、`batch_id`、`baseline_version`、`contract_version`、`rule_set_version` 对 API 和 Gate 都使用稳定文本键。
   - 数据库内部如需 surrogate key，可自行增加，但不得替代这些外部稳定键。

2. **主表只保留业务必要字段，来源合同由元数据和 API envelope 承接。**
   - `data_origin` / `origin_detail` / `subject_scope` / `subject_note` 必须出现在 API 顶层。
   - `run`、`baseline_version`、`compare_job` 等元数据表必须能无损承接来源语义。
   - 正式主链业务表通过 `run_id` / `batch_id` / `baseline_version` 回溯来源，不强制每张业务表冗余 `data_origin`。

3. **UI 派生语义不落业务主表。**
   - 不持久化字段：`watch`、`delta_color`、`empty_state_message`、页面级 banner 文案。
   - legacy 参考字段不进入主状态模型，只能放在 API `legacy_ref` 内。

### 1.2 冻结枚举

| 合同项 | 冻结值 | 备注 |
|---|---|---|
| `run_type` | `full_initialization` / `rolling` / `scenario_replay` | 严格三值 |
| `data_origin` | `real` / `synthetic` / `fallback` | `fallback` 仅用于显式降级空响应 |
| `run.status` | `pending` / `running` / `completed` / `failed` | |
| `batch.status` | `pending` / `processing` / `completed` / `failed` | |
| `batch_snapshot.stage_name` | `input` / `routing` / `objects` / `baseline` | 严格四值 |
| `lifecycle_state` | `waiting` / `observing` / `active` / `dormant` / `retired` / `rejected` | 系统级枚举；具体页面可只展示子集 |
| `health_state` | `healthy` / `insufficient` / `gps_bias` / `collision_suspect` / `collision_confirmed` / `dynamic` / `migration_suspect` | 与 Tier 0 保持一致 |
| `fail_action` | `reject` / `flag` / `warn` | `compliance_rule` 使用 |
| `severity` | `critical` / `high` / `medium` / `low` | governance / compare / anomaly 共用 |

### 1.3 冻结的 11 个快照 metric

| stage | required metrics |
|---|---|
| `input` | `fact_standardized` |
| `routing` | `fact_governed`、`fact_pending_observation`、`fact_pending_issue`、`fact_rejected` |
| `objects` | `obj_cell`、`obj_bs`、`obj_lac` |
| `baseline` | `baseline_cell`、`baseline_bs`、`baseline_lac` |

Gate 规则：每个 completed batch 恰好覆盖这 11 项；未来若要新增 metric，只能通过合同升级，不能静默扩容。

### 1.4 统一响应 envelope

```json
{
  "data_origin": "real",
  "origin_detail": "run_id=RUN-..., batch_id=BATCH-...",
  "subject_scope": "batch",
  "subject_note": null,
  "context": {
    "run_id": "RUN-...",
    "batch_id": "BATCH-...",
    "baseline_version": "BASELINE-...",
    "contract_version": "rebuild4-contract-v1",
    "rule_set_version": "rebuild4-rule-v1"
  },
  "data": {}
}
```

统一校验：

- `data_origin` 必须在顶层，不能埋进 `data`
- `subject_scope` 表示本次响应的业务主语，不表示“数据是否完整”
- `fallback` 只允许返回诚实的空 / 降级响应，不允许伴随伪造结果集

## 5. Gate 与 SQL / API 校验基线

### 5.1 Gate 清单

| Gate | 阶段 | 通过条件 | 未通过时动作 |
|---|---|---|---|
| `G0_CONTRACT_FREEZE` | 编码前 | round2 合并稿、矩阵、Gate、校验基线齐备 | 禁止开始 DDL / API 编码 |
| `G1_SCHEMA_META_READY` | Phase 1 | schema 已建；快照表已冻结；parse/compliance 非空；asset 注册表非空 | 禁止开始 full init |
| `G2_REAL_INITIALIZATION_READY` | Phase 2A | 1 条 completed real full init；1 个 logical batch；四分流守恒；baseline v1 已产出 | 禁止开始 rolling |
| `G3_REAL_ROLLING_READY` | Phase 2B / 3 | 至少 1 条 completed rolling run；completed batch > 1；每个 completed batch 都有 11 个 snapshot metric | 禁止开始 overview / batches / objects 联调 |
| `G4_API_ORIGIN_CONTRACT` | Phase 3 | 核心 API 顶层 envelope 齐备；主语与矩阵一致 | 禁止前端联调 |
| `G5_GOVERNANCE_FORMALIZED` | Phase 3 / 4 | governance endpoint 全部可用；ODS 定义/执行分离；trusted summary 可查 | governance 不计完成 |
| `G6_BASELINE_HONESTY` | Phase 3 / 4 | baseline 历史可追溯；无上一版时诚实返回 `previous_available=false` | baseline / profile 不得上线 |
| `G7_COMPARE_DEGRADED` | Phase 4 | compare 仅辅助验证；synthetic 明示；fallback 只做空响应降级 | compare 不得算正式完成 |

### 5.2 关键 SQL 校验语句

```sql
-- Q01: rebuild4 schema 就绪
SELECT COUNT(*)
FROM information_schema.schemata
WHERE schema_name IN ('rebuild4', 'rebuild4_meta');

-- Q02: field_audit 真相基线
SELECT decision, COUNT(*)
FROM rebuild2_meta.field_audit
GROUP BY decision;

-- Q03: ODS 定义/执行差集
SELECT rule_code
FROM rebuild2_meta.ods_clean_rule
WHERE is_active
EXCEPT
SELECT DISTINCT rule_code
FROM rebuild2_meta.ods_clean_result;

-- Q04: trusted 过滤主指标
SELECT
  COUNT(*) AS l0_total,
  COUNT(*) FILTER (WHERE t.lac IS NOT NULL) AS trusted_match,
  COUNT(*) FILTER (WHERE t.lac IS NULL) AS filtered_out
FROM rebuild2.l0_lac l
LEFT JOIN rebuild2.dim_lac_trusted t
  ON l."运营商编码" = t.operator_code
 AND l."标准制式" = t.tech_norm
 AND l."LAC" = t.lac::bigint;

-- Q05: 四分流守恒（rebuild4 落地后执行）
SELECT
  COUNT(*) AS standardized_rows,
  (SELECT COUNT(*) FROM rebuild4.fact_governed WHERE batch_id = $1)
+ (SELECT COUNT(*) FROM rebuild4.fact_pending_observation WHERE batch_id = $1)
+ (SELECT COUNT(*) FROM rebuild4.fact_pending_issue WHERE batch_id = $1)
+ (SELECT COUNT(*) FROM rebuild4.fact_rejected WHERE batch_id = $1) AS routed_rows
FROM rebuild4.fact_standardized
WHERE batch_id = $1;

-- Q06: completed batch 的 snapshot 覆盖
SELECT b.batch_id, COUNT(*) AS snapshot_metrics
FROM rebuild4_meta.batch b
LEFT JOIN rebuild4_meta.batch_snapshot s ON s.batch_id = b.batch_id
WHERE b.status = 'completed'
GROUP BY b.batch_id;

-- Q07: baseline current 唯一性
SELECT COUNT(*)
FROM rebuild4_meta.baseline_version
WHERE is_current = true;

-- Q08: governance 核心表非空
SELECT
  (SELECT COUNT(*) FROM rebuild4_meta.field_audit_snapshot) AS field_audit_rows,
  (SELECT COUNT(*) FROM rebuild4_meta.target_field_snapshot) AS target_field_rows,
  (SELECT COUNT(*) FROM rebuild4_meta.ods_rule_snapshot WHERE is_active) AS ods_rule_rows,
  (SELECT COUNT(*) FROM rebuild4_meta.parse_rule) AS parse_rule_rows,
  (SELECT COUNT(*) FROM rebuild4_meta.compliance_rule) AS compliance_rule_rows;
```

### 6.2 本轮关闭的歧义

本校验基线已统一关闭以下 round2 分歧：

1. `run_type` 不允许自由扩展
2. `full_initialization` 是 1 个 logical batch
3. `batch_snapshot` 是 4 stage + 11 metric
4. `data_origin` 固定在顶层 envelope
5. governance endpoint 集合固定，不再等待“后续再定”

7. parse/compliance 的正式落点固定在 `rebuild4_meta`
8. compare 的 synthetic / fallback 语义已明确区分

统一结论：round2 合并后，不再需要新增人类裁决文件即可进入 round3 任务书收束。
