## 逐项说明

### D2-001：`run_type` 枚举是否允许后续扩展

**为什么必须裁决**

round2 细化中定义了 `run_type` CHECK 约束为 `('full_initialization', 'rolling', 'scenario_replay')`。但后续可能出现需求如 `partial_rerun`（局部重跑）、`hotfix_run`（修复运行）等。

如果不裁决，实现时可能绕过 CHECK 约束发明新类型，导致 Gate 条件失效（Gate B/C 的 SQL 校验只检查特定 run_type）。

**选项**

- **A：严格三值，不可扩展**
  - 方案：CHECK 约束写死三个值；需要新类型时必须修改合同并更新 Gate
  - 优点：Gate 条件明确，不会被新类型绕过
  - 缺点：灵活性低，修改成本高

- **B：允许扩展，但新类型必须先写入合同并更新 Gate**
  - 方案：CHECK 约束可扩展，但每次扩展必须同步更新所有 Gate SQL
  - 优点：保留灵活性
  - 缺点：依赖人工纪律维护一致性

- **C：三值 + 预留 `other`，`other` 不进入任何 Gate 统计**
  - 方案：允许 `other` 类型存在但所有正式统计排除
  - 优点：实验性运行不影响正式主链
  - 缺点：`other` 容易被滥用

**建议**：A。rebuild4 必须比 rebuild3 更严格。

### D2-002：`full_initialization` 下是单 batch 还是允许多 batch

**为什么必须裁决**

D-001 裁决选择从 `rebuild2.l0_gps`（38,433,729 行）+ `rebuild2.l0_lac`（43,771,306 行）重跑完整初始化。数据量超过 8000 万行。

- 如果是单 batch：全量数据在一个批次中处理，原子性强但处理时间长、内存压力大
- 如果允许多 batch：可以按时间窗口或数据源拆分，但 initialization 的"初始快照"语义需要重新定义

这直接影响 Gate B 的通过条件（"至少 1 个 completed batch" vs "所有 initialization batch 都 completed"）。

**选项**

- **A：单 batch**
  - 方案：`full_initialization` run 下严格只有 1 个 batch
  - 优点：语义最清晰；Gate B 条件最简单；baseline 只有一个初始版本
  - 缺点：8000 万行单批处理的性能和可恢复性挑战

- **B：允许多 batch，但 baseline 只在最后一个 batch 完成后产出**
  - 方案：initialization 可拆成多个时间窗口 batch，但 `baseline_version` 只在所有 batch 完成后创建
  - 优点：更好的可恢复性和进度可见性
  - 缺点：中间 batch 的 snapshot 语义不清晰（initialization 还在进行中）

- **C：单 batch，但允许 checkpoint 机制**
  - 方案：仍然是单 batch，但处理引擎内部支持 checkpoint 恢复
  - 优点：对上层保持单 batch 语义，底层解决可恢复性
  - 缺点：引入了隐含的内部状态

**建议**：A。保持语义最简；性能问题在 Phase 2 实施时用 SQL 优化解决，不应在合同层引入复杂性。如果 A 在实施中确实不可行，再升级为 B，但必须先回到合同层修改。

### D2-003：`batch_snapshot` 的 stage 和 metric 枚举是否冻结

**为什么必须裁决**

当前 rebuild3 的 `batch_snapshot` 有 4 个 stage（`input` / `routing` / `objects` / `baseline`）和 11 个 metric。rebuild4 继承这些还是允许扩展？

如果不冻结，快照页的 UI 无法提前设计（不知道会有多少 stage/metric），API 响应结构也不稳定。

**选项**

- **A：冻结为当前 4 stage + 11 metric，不可扩展**
  - 优点：UI 可提前完全设计
  - 缺点：完全失去扩展性

- **B：stage 冻结 4 个，metric 允许扩展但必须登记**
  - 方案：stage 是硬枚举；metric 在 `rebuild4_meta` 中维护一张 `metric_registry` 表
  - 优点：核心结构稳定，细粒度可扩展
  - 缺点：需要额外维护 metric 登记

- **C：stage 和 metric 都允许扩展，但扩展需要更新合同**
  - 优点：最灵活
  - 缺点：UI 需要动态适配任意数量的 stage/metric

**建议**：B。stage 是骨架性的（对应处理管线的固定阶段），应该冻结；metric 是度量性的，合理扩展是正常需求。

### D2-004：`data_origin` 在 API 响应中的确切位置

**为什么必须裁决**

round1 已要求所有核心 API 返回 `data_origin`，但没有定义它在 JSON 响应中的位置。

- 方案 A：顶层 `{ "data_origin": "real", "data": { ... } }`
- 方案 B：嵌套在 data 内 `{ "data": { "data_origin": "real", ... } }`

如果不裁决，不同 API 可能采用不同位置，前端需要写两套解析逻辑。

**选项**

- **A：顶层，与 `data` 同级**
  - 方案：`{ "data_origin": "real", "origin_detail": "...", "subject_scope": "...", "subject_note": null, "data": { ... } }`
  - 优点：前端统一拦截层即可提取；不污染业务数据结构
  - 缺点：所有 API 必须遵守统一包装

- **B：嵌套在 `data` 内**
  - 方案：`{ "data": { "data_origin": "real", ... } }`
  - 优点：不需要统一包装
  - 缺点：前端需要在每个 API 消费处单独提取；与业务字段混在一起

**建议**：A。这是 rebuild4 最核心的语义边界字段，必须在统一位置，不应与业务数据混合。

### D2-005：governance 页需要多少个 API 端点

**为什么需要裁决**

D-003 已裁决 ODS 规则定义和执行统计分两个 API。但 governance 页还需要展示：

1. 字段审计（27 行）
2. 目标字段（55 行）
3. ODS 规则定义（26 条）——D-003
4. ODS 执行统计（24 条）——D-003
5. 解析规则（parse_rule）
6. 合规规则（compliance_rule）
7. trusted 过滤损耗

合计至少 7 个 API 端点。如果不提前定义，实现时可能出现两个极端：要么全塞一个巨型 API，要么拆成过多碎片。

**选项**

- **A：7 个独立端点**
  - 方案：每个子模块一个 API
  - 优点：语义最清晰；前端可按需加载
  - 缺点：前端 governance 页要发 7 个请求

- **B：3 个聚合端点**
  - 方案：`/governance/field_quality`（字段审计 + 目标字段）+ `/governance/rules`（ODS 定义 + 执行 + parse + compliance）+ `/governance/trusted_filter`
  - 优点：请求数少
  - 缺点：违反 D-003 裁决（ODS 定义和执行混在一个 API）

- **C：4 个端点 = D-003 的 2 个 + 字段质量 1 个 + 规则/过滤 1 个**
  - 折中方案

**建议**：A。7 个端点看似多，但每个都很小（数据量在 27-55 行级别），网络开销可忽略。语义清晰优先于请求数量。且 D-003 裁决已经确立了"分离"的原则。

### D2-006：画像页旧分类字段如何展示

**为什么需要裁决**

UI_v2 `design_notes.md` 已明确"rebuild2 旧分类（classification_v2 / gps_confidence / signal_confidence）降级为解释层"。BS 画像和 Cell 画像的 `_doc.md` 中将这些字段列在主表格中但标注为灰色参考。

但具体的降级方式没有定义：

1. 是放在主表格的最后几列（灰色）？
2. 还是放在行展开详情里？
3. 还是放在独立的"历史参考" tab 里？
4. API 返回时是与主数据同级还是嵌套在 `legacy_ref` 里？

**选项**

- **A：主表格末尾列，灰色样式，API 中嵌套在 `legacy_ref`**
  - 方案：表格中可见但灰色弱化；API 返回 `{ ..., legacy_ref: { classification_v2: ..., gps_confidence: ..., signal_confidence: ... } }`
  - 优点：一眼可见但不抢主状态
  - 缺点：主表格列数增加

- **B：仅在行展开详情中展示**
  - 方案：主表格不展示旧字段；点击展开后在详情区域以灰色展示
  - 优点：主表格更干净
  - 缺点：需要展开才能看到，对比不便

- **C：不展示，完全从 rebuild4 移除**
  - 方案：rebuild4 不读取 rebuild3 的旧分类字段
  - 优点：最干净
  - 缺点：丢失了与历史的对照能力

**建议**：A。`_doc.md` 已经把这些字段列在主表格中，且用户可能需要在新旧体系之间做映射理解。但必须通过 API 的 `legacy_ref` 嵌套和前端的灰色样式双重降级，避免被误认为主状态。

### D2-007：空状态 UI 组件统一

**为什么需要裁决**

矩阵中为每个页面定义了空状态文案，但没有定义统一的空状态 UI 组件。UI_v2 `design_notes.md` 的组件清单中有 `EmptyState.vue`，但没有定义具体样式。

如果不统一，13 个页面可能出现 13 种空状态风格。

**选项**

- **A：统一组件 + 统一样式，但允许自定义文案和图标**
- **B：各页面自行设计空状态**
- **C：round3 再统一**

**建议**：C（降低 round2 负担）。round2 冻结原则即可：所有页面必须使用 `EmptyState.vue` 共享组件，具体样式在 Phase 4 实现时统一。
