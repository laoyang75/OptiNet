## 1. 总判断

round1 已冻结方向和裁决，round2 的任务是把这些方向写成**不可多解释的合同**。

当前最大风险不是遗漏，而是**同一个词在不同位置可能有不同语义**。如果 round2 不把实体定义、来源边界、Gate 条件和页面主语全部收敛到唯一解释，round3 的执行任务书仍然会是"看起来完整但可以被自由解读"的文档。

## 2. 数据语义与元数据合同（包 A）

### 2.1 核心实体定义

以下实体必须在 round2 冻结唯一定义，不允许后续实现时再发明新语义。

#### run

| 字段 | 定义 | 约束 |
|---|---|---|
| `run_id` | 唯一标识一次完整运行 | UUID / BIGSERIAL |
| `run_type` | `full_initialization` / `rolling` / `scenario_replay` | 枚举，不可扩展 |
| `data_origin` | `real` / `synthetic` | 枚举；`scenario_replay` 必须标记为 `synthetic` |
| `input_source` | 初始化输入来源描述 | 例：`rebuild2.l0_gps + rebuild2.l0_lac` |
| `contract_version` | 本次运行绑定的合同版本 | 不可空 |
| `rule_set_version` | 本次运行绑定的规则集版本 | 不可空 |
| `status` | `running` / `completed` / `failed` | 枚举 |
| `created_at` / `completed_at` | 时间戳 | |

**语义边界**：

- `full_initialization` = 从 `rebuild2.l0_gps`（38,433,729 行）+ `rebuild2.l0_lac`（43,771,306 行）重跑完整初始化（D-001 裁决）
- `rolling` = 在已有 initialization 基础上接收增量窗口数据
- `scenario_replay` = 仅用于验证/对照目的的模拟运行，不进入正式主链

**必须冻结的细节**：`run_type` 的三个值不能在实现时被静默扩展。如果后续需要新类型（如 `partial_rerun`），必须先回到合同层修改。

#### batch

| 字段 | 定义 | 约束 |
|---|---|---|
| `batch_id` | 唯一标识一个批次 | |
| `run_id` | 所属 run | FK |
| `batch_seq` | 批次在 run 内的序号 | 从 1 递增 |
| `window_start` / `window_end` | 时间窗口 | |
| `status` | `pending` / `processing` / `completed` / `failed` | 枚举 |
| `is_rerun` | 是否为重跑批次 | boolean |
| `rerun_source_batch` | 若重跑，来源批次 | nullable FK |
| `input_count` | 输入记录数 | |
| `created_at` / `completed_at` | 时间戳 | |

**语义边界**：

- `initialization` run 下只有 1 个 batch（全量）
- `rolling` run 下有多个 batch，每个覆盖一个时间窗口
- batch 内的处理是原子的：要么全部完成，要么全部失败

#### baseline_version

| 字段 | 定义 | 约束 |
|---|---|---|
| `version_id` | 版本标识 | |
| `run_id` / `batch_id` | 产出该版本的运行和批次 | |
| `rule_set_version` | 绑定的规则集 | |
| `object_coverage` | Cell / BS / LAC 覆盖数 | JSONB |
| `created_at` | 冻结时间 | |
| `is_current` | 是否为当前生效版本 | boolean |
| `data_origin` | `real` / `synthetic` | |

**原则冻结（继承 Tier 0）**：

1. 当前批次只读上一版冻结 baseline
2. 当前批次不能一边判定一边改写本批 baseline
3. baseline 刷新是批末行为，服务下一批次

#### batch_snapshot

| 字段 | 定义 | 约束 |
|---|---|---|
| `batch_id` | FK | |
| `stage_name` | `input` / `routing` / `objects` / `baseline` | 枚举 |
| `metric_name` | 指标名 | |
| `metric_value` | 指标值 | numeric |
| `created_at` | 写入时间 | |

**写入条件**：batch `status = completed` 后立即写入，不允许延迟或批量补写。

**当前 rebuild3 参考**（PG17 验证，2026-04-06）：

- `rebuild3_meta.batch_snapshot` = 1,562 行
- 覆盖 142 个 batch、4 个 stage（`input` / `routing` / `objects` / `baseline`）、11 个 metric
- 这些数据仅作为历史参考，不进入 rebuild4 主链

#### contract_version / rule_set_version

这两个版本号必须是显式对象，不能只是 run 表里的字符串。

| 字段 | 定义 |
|---|---|
| `version_id` | 标识 |
| `content_hash` | 合同/规则内容的哈希 |
| `description` | 变更说明 |
| `created_at` | 创建时间 |
| `is_current` | 是否当前生效 |

**为什么必须独立**：如果版本号只是 run 表里的字符串，后续无法回答"这个结果是在哪个版本的合同下产出的"。

### 2.2 `data_origin` 合同

`data_origin` 是 rebuild4 最核心的语义边界字段。round1 已冻结 `real / synthetic / fallback` 三值，round2 必须写死以下合同：

| 值 | 定义 | 允许出现位置 |
|---|---|---|
| `real` | 由 rebuild4 自己的 real run 产出 | 所有主流程页、所有核心 API |
| `synthetic` | 由 `scenario_replay` 或显式评估模式产出 | 仅验证/对照页、显式标注的评估模式 |
| `fallback` | 数据不足时的降级填充 | **仅 compare 页允许**；其他页面必须进入诚实空状态 |

**所有核心 API 响应必须返回的来源字段**：

```json
{
  "data_origin": "real",
  "origin_detail": "run_id=xxx, batch_id=yyy",
  "subject_scope": "full" | "partial" | "empty",
  "subject_note": "可选说明"
}
```

**不可协商的硬约束**：

1. 主流程页（流转总览、运行/批次中心、对象浏览、对象详情、等待/观察、异常、基线/画像）只允许 `real`
2. 画像页（LAC/BS/Cell 画像）只允许 `real`
3. 支撑治理层中 `governance` 允许 `real`；`compare` 允许 `real` + `synthetic`；降级时 compare 可用 `fallback` 但必须 banner 标注
4. 没有 real 数据的页面**不允许**用 synthetic/fallback 冒充——必须进入诚实空状态
5. 前端必须消费 `data_origin` 字段并在页面上可见地展示来源状态

**若不冻结此项的后果**：后续实现时会再次出现 rebuild3 的"先用 fallback 顶住页面，等有数据再替换"的路径。

## 3. 数据准备合同（包 B）

### 3.1 real `full_initialization` 定义

| 项目 | 定义 |
|---|---|
| 输入源 | `rebuild2.l0_gps`（38,433,729 行）+ `rebuild2.l0_lac`（43,771,306 行） |
| 处理链 | 标准化 → 合规检查 → LAC/Cell/BS 对象注册 → 四分流 → 异常检测 → baseline 冻结 |
| 完成定义 | run `status = completed` + 至少 1 个 batch `status = completed` + baseline_version 已创建 |
| 产出 | rebuild4 自产的对象层（`obj_cell` / `obj_bs` / `obj_lac`）+ 事实层（四分流）+ 首版 baseline |
| 不允许 | 引用 `rebuild3.fact_*` 或 `rebuild3.obj_*` 作为初始对象/事实（D-001 裁决） |

### 3.2 real rolling run 定义

| 项目 | 定义 |
|---|---|
| 输入源 | rebuild4 initialization 完成后的增量窗口数据 |
| 最小序列 | 至少 1 条 real rolling run，下含多于 1 个 `completed` batch |
| 批次间关系 | 每个 batch 使用上一 batch 产出的 baseline 作为输入 |
| 完成定义 | run `status = completed` + 所有 batch `status = completed` |

### 3.3 `batch_snapshot` 写入条件

| 条件 | 规则 |
|---|---|
| 触发时机 | batch `status` 变为 `completed` 后立即写入 |
| 必须覆盖的 stage | `input` / `routing` / `objects` / `baseline` |
| 不允许 | 延迟补写、估算拼装、从其他 batch 复制 |

### 3.4 baseline 刷新前提

| 条件 | 规则 |
|---|---|
| 可刷新 | batch `status = completed` + 存在满足条件的 baseline_eligible 对象 |
| 不可刷新 | 当前批次正在处理中、或没有新的 baseline_eligible 变化 |
| 无上一版时 | API 返回 `subject_scope = "single_version"` + `subject_note = "暂无上一版可比"` |
| 页面表达 | 显示当前版本信息 + "暂无上一版可比" 文案；**禁止**显示 rebuild2 结果或 synthetic baseline 作为"上一版" |

### 3.5 最小可用数据集（round2 冻结）

rebuild4 正式交付前必须同时满足以下 6 条：

| 编号 | 条件 | 验证方式 |
|---|---|---|
| D-MIN-1 | 至少 1 个 real `full_initialization` run，`status = completed` | `SELECT count(*) FROM rebuild4_meta.run WHERE run_type = 'full_initialization' AND data_origin = 'real' AND status = 'completed'` ≥ 1 |
| D-MIN-2 | 至少 1 条 real `rolling` run | 同上，`run_type = 'rolling'` |
| D-MIN-3 | rolling run 下多于 1 个 `completed` batch | `SELECT count(*) FROM rebuild4_meta.batch WHERE run_id = <rolling_run> AND status = 'completed'` > 1 |
| D-MIN-4 | 每个 completed batch 都有 `batch_snapshot` | `LEFT JOIN` 无空行 |
| D-MIN-5 | 至少 1 个 real `baseline_version`，`is_current = true` | 直接查询 |
| D-MIN-6 | `parse_rule` 和 `compliance_rule` 非空 | `count(*) > 0` |

### 4.4 矩阵语义一致性校验点

以下是我在审查中发现的**必须在 round2 冻结前对齐的语义风险点**：

1. **`流转总览` vs `流转快照` 的主语区别**
   - 流转总览 = 当前 batch 的实时聚合
   - 流转快照 = 某个历史 batch 完成后写入的 `batch_snapshot`
   - 如果不明确区分，实现时会把两者混成一个 API

2. **`对象浏览` vs 画像页的视角区别**
   - 对象浏览 = 治理状态视角（lifecycle / health / 资格）
   - 画像页 = 数据质量视角（P50/P90 / GPS / 信号 / 来源构成）
   - 两者可以互链，但**主语不同、API 不同、读模型不同**

3. **`基础数据治理` 的多 API 边界**
   - D-003 裁决要求规则定义和执行统计分两个 API
   - 加上字段审计、trusted 过滤、parse_rule、compliance_rule，该页至少需要 6 个 API 端点
   - 如果不提前定义，实现时会把所有内容塞进一个巨型 API

4. **`验证/对照` 是唯一允许 fallback 的页面**
   - 必须有显式 banner 说明当前数据来源
   - 降级时 banner 不能被隐藏或静默

| Gate 0 | 合同冻结门禁 | round3 最终任务书已冻结 | 阻断编码开始 |
| Gate A | 元数据底座门禁 | schema 已创建 + parse_rule / compliance_rule 非空 + 元数据快照已冻结 | 阻断数据准备开始 |
| Gate B | 初始化完成门禁 | real full_initialization completed + baseline_version 存在 | 阻断 rolling run 开始 |
| Gate C | 滚动批次门禁 | real rolling run completed + 多个 completed batch + 所有 batch 有 snapshot | 阻断读模型 API 开发 |

### 5.2 Gate 0：合同冻结门禁

**通过条件**：

1. `rebuild4/docs/03_final/01_rebuild4_最终执行任务书.md` 已冻结
2. `rebuild4/docs/03_final/02_rebuild4_最终校验清单.md` 已冻结
3. 页面/API/表/来源矩阵已成稿
4. 所有 Gate 已独立成文

**SQL 校验项**：无（文档层）

**未通过停机规则**：不得开始 Phase 1 schema 创建。

### 5.3 Gate A：元数据底座门禁

**通过条件**：

1. `rebuild4` / `rebuild4_meta` schema 已创建（当前 PG17 验证：**尚未创建**）
2. `rebuild4_meta.parse_rule` 行数 > 0
3. `rebuild4_meta.compliance_rule` 行数 > 0
4. `rebuild4_meta.field_audit_snapshot` 行数 = 27（与源一致）
5. `rebuild4_meta.ods_rule_snapshot` 行数 = 26（与源一致）
6. `rebuild4_meta.ods_result_snapshot` 行数 = 48（与源一致）

**SQL 校验项**：

```sql
-- 全部必须返回 true
SELECT EXISTS(SELECT 1 FROM information_schema.schemata WHERE schema_name = 'rebuild4');
SELECT EXISTS(SELECT 1 FROM information_schema.schemata WHERE schema_name = 'rebuild4_meta');
SELECT count(*) > 0 FROM rebuild4_meta.parse_rule;
SELECT count(*) > 0 FROM rebuild4_meta.compliance_rule;
SELECT count(*) = 27 FROM rebuild4_meta.field_audit_snapshot;
SELECT count(*) = 26 FROM rebuild4_meta.ods_rule_snapshot;
SELECT count(*) = 48 FROM rebuild4_meta.ods_result_snapshot;
```

**未通过停机规则**：不得开始 full_initialization run。

### 5.4 Gate B：初始化完成门禁

**通过条件**：

1. 至少 1 个 real `full_initialization` run，`status = completed`
2. 该 run 下至少 1 个 `completed` batch
3. `rebuild4.obj_cell` / `obj_bs` / `obj_lac` 行数 > 0
4. 四分流表均有数据
5. 至少 1 个 `baseline_version`，`data_origin = 'real'`

**SQL 校验项**：

```sql
SELECT count(*) >= 1 FROM rebuild4_meta.run
  WHERE run_type = 'full_initialization' AND data_origin = 'real' AND status = 'completed';
SELECT count(*) > 0 FROM rebuild4.obj_cell;
SELECT count(*) > 0 FROM rebuild4.obj_bs;
SELECT count(*) > 0 FROM rebuild4.obj_lac;
SELECT count(*) > 0 FROM rebuild4.fact_governed;
SELECT count(*) >= 1 FROM rebuild4_meta.baseline_version WHERE data_origin = 'real';
```

**未通过停机规则**：不得开始 rolling run。

### 5.5 Gate C：滚动批次门禁

**通过条件**：

1. 至少 1 条 real `rolling` run，`status = completed`
2. 该 run 下多于 1 个 `completed` batch
3. 所有 `completed` batch 都有 `batch_snapshot`
4. baseline_version 存在版本链（≥ 2 个版本）

**SQL 校验项**：

```sql
SELECT count(*) >= 1 FROM rebuild4_meta.run
  WHERE run_type = 'rolling' AND data_origin = 'real' AND status = 'completed';
-- completed batch > 1
SELECT count(*) > 1 FROM rebuild4_meta.batch b
  JOIN rebuild4_meta.run r ON b.run_id = r.run_id
  WHERE r.run_type = 'rolling' AND r.data_origin = 'real' AND b.status = 'completed';
-- 所有 completed batch 都有 snapshot
SELECT count(*) = 0 FROM rebuild4_meta.batch b
  LEFT JOIN rebuild4_meta.batch_snapshot s ON b.batch_id = s.batch_id
  WHERE b.status = 'completed' AND s.batch_id IS NULL;
```

**未通过停机规则**：不得开始读模型 API 开发。

### 5.6 Gate D：读模型 API 门禁

**通过条件**：

1. 矩阵中所有核心 API 已实现
2. 所有 API 响应包含 `data_origin` 字段
3. API 主语与矩阵定义一致
4. ODS 规则定义（26 条）和执行统计（24 条）分两个 API 返回（D-003 裁决）

**API 校验项**：

```
GET /api/flow-overview → 响应包含 data_origin
GET /api/governance/ods_rules → 返回 26 行
GET /api/governance/ods_executions → 返回 24 行
# 所有核心 API 逐一检查
```

**未通过停机规则**：不得开始前端页面开发。

### 5.7 Gate E：前端验收门禁

**通过条件**：

1. 主流程页全部通过 Playwright 验收
2. 所有页面正确消费并展示 `data_origin`
3. 空状态正确展示
4. 无 fallback 冒充 real 的情况

**Playwright 校验项**：逐页检查（详见最终校验清单）

**未通过停机规则**：不得宣布正式交付。

## 6. 支撑治理层边界（包 E）

### 6.1 `governance` 正式范围

`governance`（基础数据治理）进入 rebuild4 正式完成标准。其正式数据底座如下：

| 子模块 | 数据源 | API | 说明 |
|---|---|---|---|
| 字段审计 | `rebuild4_meta.field_audit_snapshot` | `GET /api/governance/field_audit` | 展示 27 个字段的 keep/parse/drop 决策 |
| 目标字段 | `rebuild4_meta.target_field_snapshot` | `GET /api/governance/target_fields` | 展示 55 个目标字段 |
| ODS 规则定义 | `rebuild4_meta.ods_rule_snapshot` | `GET /api/governance/ods_rules` | 26 条规则定义（D-003） |
| ODS 执行统计 | `rebuild4_meta.ods_result_snapshot` | `GET /api/governance/ods_executions` | 24 条有执行统计的规则（D-003） |
| 解析规则 | `rebuild4_meta.parse_rule` | `GET /api/governance/parse_rules` | 新建正式元数据 |
| 合规规则 | `rebuild4_meta.compliance_rule` | `GET /api/governance/compliance_rules` | 新建正式元数据 |
| trusted 过滤损耗 | `rebuild4_meta.trusted_filter_snapshot` | `GET /api/governance/trusted_filter` | 过滤漏斗 + 来源/制式损耗 |

### 6.2 `compare` 降级边界

`compare`（验证/对照）明确降级为辅助验证模块，**不计入 rebuild4 正式完成标准**。

| 项目 | 规则 |
|---|---|
| 允许数据来源 | `real` + `synthetic`；不足时可 `fallback` |
| 降级 banner | 必须显示；文案："当前对比使用的数据包含非正式来源（synthetic/fallback），仅供参考验证" |
| 降级条件 | 不足 2 个可对比的 real run |
| 不计入完成标准 | compare 功能可用但不作为 rebuild4 正式验收条目 |

### 6.3 历史兼容资产归属

| 资产 | 归属 | rebuild4 中的角色 |
|---|---|---|
| `rebuild3.fact_*` / `rebuild3.obj_*` | 只读历史参考 | 不进入 rebuild4 主链；可用于验证对比 |
| `rebuild3_meta.run`（4 行） | 只读历史参考 | 不继承为 rebuild4 的 run |
| `rebuild3_meta.batch_snapshot`（1,562 行） | 只读历史参考 | 不继承为 rebuild4 的 snapshot |
| `rebuild3_meta.baseline_version`（4 行） | 只读历史参考 | 不继承为 rebuild4 的 baseline |
| `rebuild3_sample_meta` | **禁用** | 不得以任何方式进入 rebuild4 主流程 |
| `rebuild2 / rebuild2_meta` | 只读参考源 + 初始化输入源 | l0 层作为输入；元数据作为快照来源 |

## 9. 必须在 round2 冻结的细节（否则后续会多解释空间）

以下是我在审查中识别出的**若不冻结会导致多解释空间**的具体细节：

| 编号 | 细节 | 当前状态 | 若不冻结的后果 |
|---|---|---|---|
| F-01 | `run_type` 枚举值是否可扩展 | 未明确 | 实现时可能发明 `partial_rerun` / `hotfix` 等新类型 |
| F-02 | initialization batch 是 1 个还是可以多个 | 未明确 | 实现时可能把 initialization 也做成多 batch |
| F-03 | `batch_snapshot` 的 stage/metric 枚举 | 当前 rebuild3 有 4 stage / 11 metric，但未冻结为合同 | 实现时可能随意增删 |
| F-04 | `data_origin` 在 API 响应中的位置 | "顶层还是嵌套"未定义 | 前端消费逻辑不一致 |
| F-05 | governance 页的 API 数量（至少 6 个端点） | 仅 D-003 定义了 2 个 ODS 端点 | 实现时可能把所有内容塞一个 API |
| F-06 | 画像页的旧分类字段（classification_v2 / gps_confidence / signal_confidence）展示方式 | "降级为解释层"但未定义如何降级 | 可能仍然做成主状态 |
| F-07 | `contract_version` / `rule_set_version` 是字符串还是独立对象 | 未明确 | 若只是字符串则无法追溯 |
| F-08 | trusted 过滤损耗的展示维度 | 已知总损耗 31.27%，但未定义哪些页面展示 | governance 页可能只展示总数，不展示来源/制式拆分 |
| F-09 | `compare` 降级 banner 的触发条件和文案 | 已说"必须 banner"但未定义具体条件 | 实现时可能把 banner 做成可关闭 |
| F-10 | 空状态的 UI 表达统一性 | 各页面各自描述 | 可能出现 10 种不同的空状态风格 |

这些细节建议在 round2 合并稿中逐项冻结，或上抛 round2 裁决。
