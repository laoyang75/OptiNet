## 2. 候选裁决问题列表

### D-004: Parse Rule 与 Compliance Rule 的元数据填充动作与来源归属
**结论**
当前 PG17 中 `rebuild4_meta.parse_rule` 与 `compliance_rule` 必须存在，但现网实际为空。必须确定使用何种途径、由谁将原 Markdown 文档或 Python 清洗脚本中的内容转化为 PG17 中的正式行数据。

**依据**
`rebuild4/docs/02_rounds/round1_plan/merged/02_统一风险与缺口.md` 的 R-04 要求给出规则的正式元数据承接方案。

**风险**
如果只是声明有这几张表而不裁决“怎么把数据塞进去”，执行期表里将永远是空数据，进而导致 Governance 页和 Gate 2 完全卡死。

**校验**
检查执行后的 `rebuild4_meta.parse_rule` 表数据量必须 > 0，且规则详情完整。

**选项**
- **A**: **手动提取法**。由研发人员结合现有代码库，一次性提取并编写人工 INSERT SQL 脚本在 schema 迁移期同步落库。
- **B**: **文档解析法**。编写专门的自动化脚本，依靠大模型或结构化文本抽取，从原有 Markdown / Python 中转换。
- **C**: **前端补录法**。优先提供一个无 Gate 阻断的 Admin 补录界面，由专家直接在初始化阶段通过系统后台填写。

### D-005: Trusted 过滤损耗多维分析（1300万数据）的高性能存储与查询策略
**结论**
Governance 模块必须能够清晰呈现高达 1300 万记录规模的 Trusted 过滤损耗（包含制式分布、来源分布等）。因为直接对原事实表进行高并发聚合查询易引发崩溃，需要裁决使用何种技术手段支撑该视图分析。

**依据**
`rebuild4/docs/02_rounds/round1_plan/merged/01_统一计划.md` 第 2.4 节确认 Trusted 被过滤数据达 13,688,925 条。

**风险**
API 查询过慢或者前端请求超时，将直接摧毁支撑治理层的用户体验，使解释层失效。

**校验**
在包含海量数据的测试环境下，请求对应的 `/api/governance/trusted_loss` 耗时应维持在亚秒级 (< 1s)。

**选项**
- **A**: **预计算物化视图 (Materialized View)**。依托 PG17 构建按 `制式` / `来源` / `批次` 维度的物化视图，并和 run/batch 快照流保持同步刷新。
- **B**: **周期性跑批统计表 (Summary Table)**。在 `full_initialization` 或每轮 rolling run 后，由定时任务聚合生成统计行存入 `fact_trusted_loss_summary` 表供前端秒级查询。
- **C**: **直接关联实时查询**。仅做最佳的索引结构优化，让 API 依然在 `l0_lac` 与 `dim_lac_trusted` 之间左连动态统计。

### 可能遗漏点
- 问题 D-005 无论选 A 还是 B，都会对现有的 rolling batch snapshot 的工作流产生代码与调度层面的强侵入依赖。
- 从旧项目迁移的代码可能因为缺乏文档维护而包含更多隐藏的、未能识别成 parse 规则的逻辑。

### 你认为最危险的 3 个未决问题
1. **Rule 元数据自动化或手动录入的失真度**。从代码和散乱 Markdown 转录的规则是否真正准确对应业务意图？
2. **预计算刷新时机。** 当选择 D-005 A/B 方案时，刷新任务由于数据量太大可能拉长 rolling run 的耗时。
3. **如果在补录过程中发现历史文档相互矛盾**，有没有一套最终兜底原则指导研发进行去伪存真操作。
