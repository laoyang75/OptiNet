## 2. 实体清单 (Entity List)
### 结论
rebuild4 的核心实体必须脱离对 rebuild3 的直接依赖，完全自建自产。实体必须分为“核心业务实体”与“流转元数据实体”两层。
- **业务实体**：`obj_cell`, `obj_bs`, `obj_lac`，以及事实底座 `fact_standardized`, `fact_governed`, `fact_pending_observation`, `fact_pending_issue`, `fact_rejected`。
- **流转元数据实体**：`run`, `batch`, `batch_snapshot`, `baseline_version`，均属于 `rebuild4_meta` 范畴。
- **治理与规则实体**：`ods_clean_rule`, `ods_clean_result`, `parse_rule`, `compliance_rule`, `field_audit`, `target_field`。

### 依据
`rebuild4/docs/02_rounds/round1_plan/merged/01_统一计划.md` §1.2 明确裁决“rebuild4 正式动态对象、事实、run、batch 等必须由 rebuild4 自己产出”。依据 PG17 查询确认，原有 `rebuild3` 的 `full_initialization` 仅1次、`scenario_replay` 3次，历史跑批数据不合法，须全量重建。

### 风险
如果不进行实体绝对隔离，将不自觉地继承 rebuild3 带有 synthetic 语义的历史包袱数据，最终导致主流程数据失真。

### 校验
必须在 PG17 `rebuild4` 及 `rebuild4_meta` schema 下，能够完整检索出独立的表定义，且所有记录必须关联到自身新建的 real `full_initialization` run。

## 3. `data_origin` 合同
### 结论
`data_origin` 及其核心枚举 (`real`, `synthetic`, `fallback`) 必须固化为数据库字典与 API 核心响应合同字段，并禁止使用前端弱类型绕过。所有主流程前端页面在读取列表或概览时，若遇到不合规来源或查无数据，强制渲染为诚实空状态。

### 依据
`rebuild4/docs/02_rounds/round1_plan/merged/01_统一计划.md` §1.3 成功标准 2 规定：“主流程页必须明确 `data_origin`、`origin_detail`、`subject_scope`，且前端真实消费。”

### 风险
如果未在数据库和 API 层施加强约束保障 `data_origin` 输出，前端必定会在面临空数据时再次通过硬编码 fallback 逻辑填充假界面。

### 校验
后端所有核心对象的查询 API 返回的 JSON Schema 必须将 `data_origin` 标记为必填且在允许的枚举值中。针对数据库表中的记录，执行 `SELECT DISTINCT data_origin FROM rebuild4.fact_standardized` 必须只返回规定内枚举。

## 4. 页面/API/表主语对齐要求
### 结论
页面、API、表数据源三者必须形成 1:1:1 映射闭环关系，不得交叉借用。
- **数据准备与状态页 (Initialization/Rolling)**: 主语为 `run` / `batch`，对应 `/api/runs`，底层为 `rebuild4_meta.run`。
- **快照与基线比对页 (Snapshot / Baseline)**: 主语为 `batch_snapshot` 与 `baseline_version`，对应 API `/api/snapshots`、`/api/baselines`，底层为 `rebuild4_meta.batch_snapshot` 与 `rebuild4_meta.baseline_version`。
- **核心数据对象页 (Objects/Overview)**: 主语为 `obj_cell` / `obj_bs` / `obj_lac`，通过强关联当前最新 real baseline 过滤得出，对应 API `/api/objects`。
- **清洗与治理页面 (Governance)**: 主语为规则（ODS/Parse/Compliance），对应 API `/api/governance/*`，分别映射 `rebuild4_meta` 中定义的规则及执行统计表。

### 依据
`rebuild4/docs/02_rounds/round1_plan/merged/02_统一风险与缺口.md` R-07 指出：若页面/API/表/来源矩阵未冻结，页面主语仍可能继续漂移，导致上下文断裂。

### 风险
若页面脱离主语约束，随意向后端索要“只要有数据就行”的接口响应，就会在数据层还未准备好 real 数据时，提前渲染出无逻辑支撑的脏数据页面。

### 校验
验收测试（如 Playwright 链路），检查所有获取数据的 API 请求必须携带特定的 `run_id` 或 `baseline_version` 作为请求载荷，并且这些上下文必须对应 PG17 中真实的 UUID 或版本号。

### 你认为最危险的 3 个未决问题
1. **历史 parse_rule 与 compliance_rule 元数据如何平滑入库？** 只有 26 条 ODS 有确切明细，而前两者在 PG17 中为空表，从 markdown / 脚本向 PG17 提取录入的具体责任边界未定。
