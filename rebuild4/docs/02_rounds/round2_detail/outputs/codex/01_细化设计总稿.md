## 1. 直接冻结结论

### 1.1 本轮应直接冻结的结构结论

1. **rebuild4 必须以新 schema 启动，而不是在 rebuild3 上继续修补。**
   - `PG17 execute_sql` 结果：`information_schema.schemata` 中当前不存在 `rebuild4`、`rebuild4_meta`。
   - 结论：round2 必须把 `rebuild4` / `rebuild4_meta` 的表、字段、API、Gate 写成唯一结构合同；round3/实施阶段先建 schema 再编码。

2. **real 主链必须从 rebuild2 的双输入源重跑，而不是继承 rebuild3 当前结果。**
   - `rebuild2.l0_gps = 38,433,729`；`rebuild2.l0_lac = 43,771,306`。
   - `rebuild3.fact_standardized = 43,771,306`，只等于 `l0_lac`，不能覆盖 D-001 已裁定的“双输入重跑”要求。
   - 结论：`full_initialization` 输入合同必须同时绑定 `l0_gps + l0_lac`，且由 rebuild4 自产对象、事实、baseline。

3. **`data_origin / origin_detail / subject_scope` 必须成为 API 与读模型的硬字段。**
   - `execute_sql` 查询 `information_schema.columns` 结果：`rebuild3` / `rebuild3_meta` 当前不存在 `data_origin`、`origin_detail`、`subject_scope` 三列。
   - `rebuild3` 当前只能通过 `run_type`、`scenario_key`、`snapshot_source`、`sample_scope_tag` 间接猜来源；其中各 `fact_*` 的 `sample_scope_tag` 当前全部为 `NULL`。
   - 结论：rebuild4 不允许继续靠隐含语义区分 real / synthetic / fallback。

4. **governance 必须进入正式完成标准，compare 必须正式降级。**
   - `rebuild3_meta.compare_result = 0`；`rebuild3_meta.asset_field_catalog = 0`、`asset_table_catalog = 0`、`asset_usage_map = 0`、`asset_migration_decision = 0`。
   - 结论：governance 不能再复用空注册表；必须在 rebuild4_meta 中正式补齐。compare 可以保留，但不得阻塞主链完成标准。

5. **baseline 历史能力不能继续停留在“版本表有记录、对象表无历史”的状态。**
   - `rebuild3_meta.baseline_version = 4`，但 `rebuild3.baseline_cell / baseline_bs / baseline_lac` 当前都只有 `BASELINE-FULL-V1` 这 1 个版本。
   - `batch_baseline_refresh_log` 显示：1 条 `full_initial_baseline` 触发、3 条 `scenario_initialization` 触发、138 条 `rolling_2h_snapshot` 未触发。
   - 结论：rebuild4 必须把 baseline 历史存储方式冻结为“版本可追溯、页面诚实显示有无上一版”。

### 1.2 本轮输出要解决的不是“原则”，而是“合同”

本轮必须把下面 5 个包冻结成可执行结构：

- 包 A：数据语义与元数据合同
- 包 B：数据准备合同
- 包 C：页面 / API / 表 / 来源矩阵
- 包 D：Gate 与校验基线
- 包 E：governance / compare 边界合同

   - 结论：必须补齐版本化存储或正式定义 diff 落盘方案。

4. **快照页当前结构偏向 scenario，而不是正式主链。**
   - `v_flow_snapshot_timepoints` 共有 141 行，全部来自 `scenario_replay`，不含 full init。
   - 结论：rebuild4 的流转快照合同必须以 real `batch_snapshot` 为准，不允许沿用 scenario-only 视图。

## 3. 包 A：数据语义与元数据合同

### 3.1 需要冻结的核心实体

必须冻结为正式实体的最小集合：

1. `contract_version`
2. `rule_set_version`
3. `run`
4. `batch`
5. `standardized_event`
6. `routed_fact`
7. `cell / bs / lac`
8. `baseline_version`
9. `governance_rule`（字段 / parse / compliance / ODS / trusted）
10. `gate`

### 3.2 统一来源语义合同

所有业务 API（launcher 以外）统一返回：

```json
{
  "data_origin": "real | synthetic | fallback",
  "origin_detail": "具体来源说明",
  "subject_scope": "run | batch | object | baseline_version | governance_asset | validation_compare",
  "context": {
    "run_id": "...",
    "batch_id": "...",
    "baseline_version": "...",
    "contract_version": "...",
    "rule_set_version": "..."
  },
  "data": {}
}
```

冻结规则：

- 主流程页、对象页、画像页、baseline 页、initialization 页、governance 页：只允许 `data_origin = real`
- compare 页：允许 `real` 或 `synthetic`，但必须显示 banner；禁止 `fallback`
- 未满足最小数据条件的页面：返回空状态，不允许把 `fallback` 伪装成正式结果

### 3.3 版本实体必须独立成表

当前 `rebuild3_meta` 只有 `run` / `batch` / `baseline_version`，并没有独立的 `contract_version`、`rule_set_version`、`source_adapter` 表。

round2 冻结为：

- `rebuild4_meta.contract_version`
- `rebuild4_meta.rule_set_version`
- `rebuild4_meta.source_adapter`（记录 `l0_gps` / `l0_lac` / 后续 rolling 输入源）

这样做的原因：

- 避免只在事实表里散落字符串版本号
- 让 Gate 可以直接校验“本批次绑定的是哪版合同”
- 让 initialization / rolling / rerun 能回查到同一来源配置

### 3.4 parse / compliance 必须从“空壳表”升级成正式元数据

当前 `rebuild2_meta.parse_rule = 0`、`compliance_rule = 0`，但表结构已经存在。

rebuild4 应冻结为：

- `rebuild4_meta.parse_rule`
- `rebuild4_meta.compliance_rule`
- `rebuild4_meta.parse_execution`（可选，至少预留）
- `rebuild4_meta.compliance_execution`（可选，至少预留）

最小字段要求见 `02_实体字段接口与Gate细化.md`。

风险说明：如果 round2 不先冻结字段集合，round3/实施阶段又会回到“文档里有规则，数据库里没有规则”的 rebuild3 老路。

校验动作：

- SQL：两张规则表非空；每条规则有 `rule_code` / `target_field` / `tech_scope` / `is_active`
- API：`/api/governance/parse_rules`、`/api/governance/compliance_rules` 返回非空
- 页面：governance 页能按规则类型、字段、制式筛选（round3 执行）

## 4. 包 B：数据准备合同

### 4.1 real `full_initialization` 合同

冻结定义：

- `run_type = full_initialization`
- `data_origin = real`
- 输入源 = `rebuild2.l0_gps + rebuild2.l0_lac`
- 输出必须同时写入：
  - `rebuild4.fact_standardized`
  - 四分流事实表
  - `obj_cell / obj_bs / obj_lac`
  - `baseline_version`
  - `baseline_*`
  - `batch_snapshot`
  - `initialization_step_log`

通过条件：

- 至少 1 条 completed real init run
- 至少 1 个 completed init batch
- 四分流合计守恒 = standardized 总量
- 生成 1 个可追溯的 real baseline version

### 4.2 real rolling 合同

冻结定义：

- `run_type = rolling`
- `batch_type = rolling_2h`（v1）
- 同一个 rolling run 下必须存在多于 1 个 completed batch
- 每个 completed batch 都必须写入 `batch_snapshot`、`batch_flow_summary`、`batch_transition_summary`

通过条件：

- `rebuild4_meta.batch` 中至少 1 个 rolling run 拥有 `completed batch_count > 1`
- 每个 completed batch 均有 11 个快照指标
- 首页与批次中心不需要临时聚合脚本补数据

### 4.3 baseline 诚实合同

冻结规则：

- `baseline_*` 必须可回指 `baseline_version`
- `baseline_version` 必须可回指 `run_id + batch_id + rule_set_version`
- 若仅有当前版 real baseline，则 `GET /api/baseline/current/diff` 返回：
  - `previous_available = false`
  - `message = 暂无上一版 real baseline 可比`
- 禁止拿 rebuild2 或 synthetic baseline 充当上一版

### 4.4 initialization 页面合同

现有 UI 文档要求“初始化流程 + 汇总 + 四分流 + 特殊口径说明”，但当前数据库没有对应的步骤日志表。

因此 round2 冻结新增：

- `rebuild4_meta.initialization_step_log`
- `rebuild4_meta.initialization_note`

否则 `/initialization` 只能重新拼 Markdown，不具备数据库可验证性。

## 5. 包 C：页面 / API / 表 / 来源矩阵（摘要）

| 页面 | 主语 | 主要 API | 读模型/表 | 允许来源 | 最小可用数据 |
|---|---|---|---|---|---|
| 流转总览 | 当前 batch | `GET /api/flow-overview` | `batch` + `batch_flow_summary` + `batch_anomaly_*` | real | 至少 1 个 completed real batch |
| 流转快照 | timepoint/batch | `GET /api/flow-snapshot` | `batch_snapshot` + `v_flow_snapshot_timepoints` | real | 每 completed batch 11 个快照指标 |
| 运行/批次中心 | run / batch | `/api/runs/current` `/api/batches` `/api/batches/:id/detail` | `run` + `batch` + `batch_transition_summary` | real | 至少 1 条 real run |
| 对象浏览 | 当前对象快照 | `/api/objects` `/api/objects/summary` | `obj_cell` / `obj_bs` / `obj_lac` | real | 对象快照非空 |
| 对象详情 | 单对象 | `/api/objects/:key/detail` | `obj_*` + `obj_state_history` + `obj_relation_history` + `profile_*` | real | 对象存在 |
| 等待/观察工作台 | 等待/观察 Cell | `/api/observation-workspace` | `v_observation_candidate` | real | `obj_cell` 有 waiting/observing 数据 |
| 异常工作台 | 对象级 / 记录级异常 | `/api/anomaly-workspace` | `v_anomaly_object` + `v_anomaly_record` | real | anomaly summary / detail 非空 |
| 基线/画像 | 当前 baseline | `/api/baseline/current` `/api/baseline/current/diff` | `baseline_version` + `baseline_*` + `baseline_diff_*` | real | 至少 1 个 real baseline |
| LAC / BS / Cell 画像 | 画像对象 | `/api/profiles/lac|bs|cell` | `profile_*` + `obj_*` + `baseline_*` | real | 对应 profile 读模型非空 |
| 基础数据治理 | 字段/规则/资产 | `/api/governance/*` | `asset_field_catalog` + `parse_rule` + `compliance_rule` + `ods_*` + `trusted_loss_*` + `asset_*` | real（可指向 reference schema） | governance meta 表非空 |
| 验证/对照 | 两次运行比较 | `/api/validation/compare` | `compare_job` / `compare_result` | real 或 synthetic | 需显式 banner；不计入正式完成标准 |
| 初始化数据 | full init run | `/api/initialization/:run_id` | `run` + `batch` + `initialization_step_log` | real | 至少 1 条 completed full init |

风险说明：如果没有这张矩阵，后续最容易再次出现“页面能开、数据主语不真实、来源字段靠猜”的偏移。

校验动作：

- 文档：矩阵必须进入 round3 最终任务书
- SQL：每个矩阵行必须能映射到至少 1 张表/视图
- API：每个 endpoint 返回统一来源字段
- 页面：round3 用 Playwright 验证主下钻链路

## 6. 包 D：Gate 与校验基线

### 6.1 必须独立的 Gate 列表

| Gate | 阻塞阶段 | 通过条件 | 失败时动作 |
|---|---|---|---|
| G0 合同冻结 Gate | 编码前 | 本轮 3 份文档齐备；矩阵、字段、Gate 独立成文 | 禁止进入 schema / API 编码 |
| G1 schema/meta Gate | Phase 1 | `rebuild4` / `rebuild4_meta` 已建；版本表、规则表、asset 表非空或可解释 | 禁止进入 full init |
| G2 real initialization Gate | Phase 2 | 1 条 completed real init run；四分流守恒；产生 baseline | 禁止进入 rolling/API |
| G3 real rolling Gate | Phase 2/3 | 至少 1 条 rolling run，且 completed batch > 1；每批有 snapshot | 禁止首页/批次中心验收 |
| G4 API 来源合同 Gate | Phase 3 | 核心 API 返回 `data_origin / origin_detail / subject_scope` | 禁止前端联调 |
| G5 governance Gate | Phase 3/4 | governance API 非空；ODS 定义/执行分离；trusted 损耗可查询 | governance 不得算完成 |
| G6 baseline 诚实 Gate | Phase 3/4 | 当前 baseline 可追溯；无上一版时诚实空提示 | baseline/profile 不得上线 |
| G7 compare 降级 Gate | Phase 4 | compare 只走独立入口；若 synthetic 必有 banner；不计入正式完成标准 | compare 从导航保留但不得算主链完成 |

### 6.2 本轮建议固定的 SQL 校验项

1. **事实守恒**：`fact_standardized = fact_governed + fact_pending_observation + fact_pending_issue + fact_rejected`
2. **快照覆盖**：每个 completed batch 必须有 11 个 `batch_snapshot` 指标
3. **来源合同**：核心读模型必须存在 `data_origin / origin_detail / subject_scope`
4. **ODS 分层**：`ods_rule` 26 条定义与 `ods_execution` 24 条统计分离展示
5. **trusted 损耗**：能重算 `43,771,306 / 30,082,381 / 13,688,925 / 31.27%`
6. **baseline 诚实**：若上一版缺失，则 diff API 返回 `previous_available = false`

### 6.3 本轮建议固定的 API 校验项

- `/api/flow-overview`：必须带 `batch_id`、`baseline_version`、`data_origin`
- `/api/batches/:id/detail`：必须返回晋升/降级/级联更新摘要
- `/api/objects/:key/detail`：必须一次返回 snapshot / history / profile / anomaly / impact
- `/api/governance/ods_rules` 与 `/api/governance/ods_executions`：必须拆 endpoint
- `/api/baseline/current/diff`：必须有 `previous_available`

页面校验在 round3 执行；本轮只冻结检查项，不访问页面。

## 7. 包 E：governance / compare 边界

### 7.1 governance 正式范围

governance 本轮必须正式交付以下 6 组能力：

1. 字段目录（来自 `asset_field_catalog` / `target_field` / `field_audit`）
2. parse 规则目录
3. compliance 规则目录
4. ODS 定义层 / 执行层双视图
5. trusted 损耗与来源解释
6. 资产使用与迁移决策目录

依据：

- `field_audit = 27`
- `target_field = 55`
- `parse_rule = 0`
- `compliance_rule = 0`
- `asset_* = 0`

风险：如果 governance 继续只靠文档和脚本，rebuild4 仍然无法解释“为什么数据变成现在这样”。

### 7.2 compare 降级边界

compare 只允许承担：

- 局部修复后的辅助验证
- real vs real 对比（优先）
- synthetic 验证模式（允许但必须 banner）

compare 不允许承担：

- 主流程首页入口
- 正式完成标准
- fallback 兜底展示

依据：`rebuild3_meta.compare_result` 当前为 0；当前并无正式 compare 数据链。

## 9. round3 入场条件

只有当下列条目全部满足，才建议进入 round3：

1. `02_实体字段接口与Gate细化.md` 中的表、字段、API、Gate 已被接受
2. `rebuild4` / `rebuild4_meta` 建表清单已从文档映射到实施任务

4. governance 与 compare 的边界已不再依赖口头解释
5. baseline 历史策略已被裁决或明确接受当前推荐方案

如果以上任一项未完成，就不应进入“直接写代码”阶段。
