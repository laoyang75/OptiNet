## D-201：baseline 历史存储模式

**为什么必须裁决**

`PG17 execute_sql` 结果：

- `rebuild3_meta.baseline_version = 4`
- 但 `rebuild3.baseline_cell / baseline_bs / baseline_lac` 当前都只有 `BASELINE-FULL-V1` 1 个实体版本

这说明“版本登记”和“版本实体明细”当前是脱节的。若不先裁决，`/baseline/current/diff` 以及画像页的历史引用都会反复改。

**选项**

- **A：保留当前 baseline_* 只存当前版，历史比较全部靠临时 diff 计算**
  - 优点：写入最轻
  - 缺点：无法直接审计历史版本；每次 diff 都要重新扫描对象集

- **B（建议） ：baseline_* 改为按 `baseline_version` 追加存储，并额外提供 current view**
  - 优点：版本明细与 `baseline_version` 对齐；既能直接查询历史，又能对首页/API暴露“当前版”视图
  - 缺点：存储量会上升，但 baseline 刷新频率低，可控

- **C：把 baseline 历史单独拆到 `baseline_history_*`，当前表仍只存 current**
  - 优点：职责分离清晰
  - 缺点：对象字段需要维护两套表结构，实施成本最高

**建议：B**

原因：当前 `baseline_*` 本身已经带 `baseline_version` 字段，改成追加式版本存储是最小结构变更，同时能直接解决“版本表有记录但实体表没有历史”的根问题。

## D-202：governance 注册表填充方式

**为什么必须裁决**

`PG17 execute_sql` 结果：

- `rebuild3_meta.asset_field_catalog = 0`
- `rebuild3_meta.asset_table_catalog = 0`
- `rebuild3_meta.asset_usage_map = 0`
- `rebuild3_meta.asset_migration_decision = 0`

而 governance 页设计文档明确要求“概览卡 + 字段目录 + 表目录 + 使用关系 + 迁移状态”四类读模型。如果不先裁决，governance 只能继续停留在 mock。

**选项**

- **A：完全人工维护**
  - 优点：最容易控制说明文案
  - 缺点：维护成本高，字段和表一变就过时

- **B（建议）：数据库 introspection 自动登记 + 人工补充层级/迁移结论/使用说明**
  - 优点：既能保证表/字段基数真实，又能补齐业务语义
  - 缺点：需要一套首次初始化脚本

- **C：优先代码扫描生成，再人工修正**
  - 优点：API/页面使用关系可自动化
  - 缺点：当前 rebuild4 代码尚未生成，无法作为 phase1 起点

**建议：B**

原因：当前事实层和对象层的真实表、字段数量必须以数据库为准；层级、迁移结论、usage_desc 再由人工补充最稳。

## D-203：parse / compliance 规则正式承接位置

**为什么必须裁决**

`PG17 execute_sql` 结果：

- `rebuild2_meta.parse_rule = 0`
- `rebuild2_meta.compliance_rule = 0`

同时 `search_objects` 显示两张表在旧 schema 中已经有列结构，但为空。现在需要决定：rebuild4 是继续往旧 schema 补，还是在新 schema 正式承接。

**选项**

- **A：继续使用 `rebuild2_meta.parse_rule / compliance_rule` 作为唯一真相源**
  - 优点：表名延续性最好
  - 缺点：违反“rebuild2 只读参考源”的 round1 结论

- **B（建议）：在 `rebuild4_meta` 新建正式版本化规则表，`rebuild2_meta` 保留只读参考**
  - 优点：符合 rebuild4 独立主链要求；可以挂 `rule_set_version` 和 `contract_version`
  - 缺点：需要一次性迁移/补录规则

- **C：继续把 Markdown / 脚本当主真相源，数据库只做缓存**
  - 优点：短期省事
  - 缺点：直接违背 round1 `R-04` 关闭标准

**建议：B**

原因：这是唯一同时满足“rebuild2 只读参考”“rebuild4 正式可查询”“规则可版本化”的方案。

## D-204：anomaly summary 是否拆成对象级 / 记录级 / 影响级三张表

**为什么必须裁决**

`PG17 execute_sql` 结果显示 `rebuild3_meta.batch_anomaly_summary` 里同时存在：

- `bs_object:active`
- `cell_object:gps_bias`
- `record:structural_rejected`
- `record:single_large`

这说明当前表把“对象状态汇总”和“记录级异常汇总”混在一个结构里。若不裁决，异常工作台 API 会长期含糊不清。

**选项**

- **A（建议）：拆成 `batch_anomaly_object_summary`、`batch_anomaly_record_summary`、`batch_anomaly_impact_summary`**
  - 优点：和 UI 文档“双视角 Tab + 下游影响”完全一致；API 粒度最清楚
  - 缺点：要新建 3 张表/视图

- **B：继续用 1 张混合表，通过 `anomaly_level` + `anomaly_name` 区分**
  - 优点：实现最省
  - 缺点：对象状态、异常类型、下游影响继续混写，难以校验

- **C：不落盘，页面临时聚合**
  - 优点：表最少
  - 缺点：重复计算严重，且难以形成 Gate

**建议：A**

原因：这是唯一能把“对象级异常”“记录级异常”“影响路径”三种主语分开的方案，也最符合 UI_v2 文档。

## D-205：compare 页是否允许 synthetic 结果

**为什么必须裁决**

当前事实：

- `rebuild3_meta.compare_result = 0`
- round1 已冻结：compare 降级为辅助验证模块，不计入正式完成标准

但 compare 页面仍在 IA 中存在。需要明确：若第二个 real baseline 尚未形成，compare 是否允许以 synthetic 方式提供辅助验证。

**选项**

- **A：只允许 real vs real；不满足条件时 compare 页直接空状态**
  - 优点：语义最纯
  - 缺点：辅助验证价值较低，页面长期空白

- **B（建议）：允许 synthetic compare，但必须 `data_origin = synthetic` + banner，且不计入正式完成标准**
  - 优点：保留辅助验证价值；不污染主流程
  - 缺点：需要非常严格的来源表达

- **C：复用 rebuild3 或旧脚本结果作为 compare 页面参考**
  - 优点：最快有内容
  - 缺点：最容易再次出现“reference 结果冒充 rebuild4 正式结果”

**建议：B**

原因：round1 已明确 compare 是辅助模块，不是正式完成标准；在此边界下允许 synthetic 是可以接受的，但必须被来源合同和页面 banner 强约束。
