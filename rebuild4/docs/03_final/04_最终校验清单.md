
# 最终校验清单

## 1. 使用规则

1. 先做文档与 SQL 校验，再做 API 合同测试，最后做 Playwright 页面验收；前置失败时后置动作一律暂停。
2. 所有计数、比例、差集、缺口都直接校验冻结数值：`38,433,729`、`43,771,306`、`27`、`55`、`26`、`24`、`31.27%`、`11,350,552` 等数字必须原样出现。
3. 主流程、对象、baseline、profile、initialization、governance 只允许 `real` 或诚实空状态；只有 compare 允许 `synthetic` / `fallback`。
4. Gate 顺序固定为 `G4 -> G5 -> G6 -> G7 -> G8`；`G4` 只验 API 顶层合同，`G5` 才验 governance 完整性，`G6` 才验 baseline/profile 完整性，`G7` 只验 compare 降级身份，`G8` 负责页面总体验收。
5. `G0` 在控制表尚不存在前，唯一正式启动凭证就是 `rebuild4/runtime_artifacts/gate/{package_id}/G0/attempt_001/index.json`；`gate_run_result.evidence_ref` 必须固定指向 `rebuild4/runtime_artifacts/gate/{package_id}/{gate_code}/{attempt_id}/index.json`；各 step 的原始证据由同目录下的 `{step_id}.*` 提供，且必须被当前 attempt 的 `index.json` 收录。`G4-G7` 使用 JSON / HTTP 合同测试证据，`G8` 使用 Playwright trace / screenshot / HTML 报告。
6. `G8` 只有在 `P5-GPW-00` 到 `P5-GPW-11` 全部 passed 且 `G8/{attempt_id}/index.json` 汇总完成后才允许写成 `passed`；其中 `P5-GPW-11` 不计入 12 个正式页面计数，但仍计入 compare 辅助验证模块的正式放行。单页失败时可以继续收集其他页面诊断证据，但 `G8` 总状态必须保持 `failed`。
7. 任一检查项失败，都必须执行对应停机动作，不允许口头豁免。
