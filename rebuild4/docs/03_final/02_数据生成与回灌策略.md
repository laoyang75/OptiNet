# 数据生成与回灌策略

## 1. 总策略

本轮冻结的数据生成与回灌策略只有一条主线：

1. **主链业务结果只允许由 rebuild4 自产。**
2. **治理解释层允许一次性定向回灌到 `rebuild4_meta`。**
3. **compare 的 synthetic / fallback 只服务辅助验证，不回写主链。**
4. **`rebuild3*` 永远只读参考，不得冒充 rebuild4 current。**

## 2. 输入源与目标表冻结

| 场景 | 冻结输入 | 目标位置 | 约束 |
|---|---|---|---|
| real `full_initialization` | `rebuild2.l0_gps` + `rebuild2.l0_lac` | `rebuild4` + `rebuild4_meta` | 唯一正式双输入 |
| real `rolling` | 真实窗口化输入 | `rebuild4` + `rebuild4_meta` | 必须满足 `data_origin=real` 且 completed batch `> 1` |
| governance 快照 | `rebuild2_meta.*` + 数据库 introspection | `rebuild4_meta` | 允许一次性快照与 seed 回灌 |
| legacy 参考信息 | `rebuild2` / `rebuild3` 旧质量字段 | API `legacy_ref` | 只作参考，不进入主状态 |
| compare | `compare_job` + `compare_result` | `rebuild4_meta` | 只服务辅助验证，不回写主链业务表 |

### 3.2 ODS 差集必须显式保留

正式冻结以下差集：

- `NULL_WIFI_MAC_INVALID`
- `NULL_WIFI_NAME_INVALID`

它们必须在最终治理能力里继续以“定义存在、执行统计缺失”的事实出现，不能通过裁剪定义层来“消灭差集”。

### 3.3 canonical_seed_contract：正式执行唯一 seed 输入

执行期只允许引用六文件，因此 G1 的正式 seed 输入不再依赖包外资料，而是**本包内冻结的 canonical seed 合同 + `seed_artifact_manifest` current 记录**。

#### 3.3.1 canonical seed artifact 固定形状

| 项 | 冻结值 | 约束 |
|---|---|---|
| artifact 路径 | `rebuild4/runtime_artifacts/canonical_seed/{package_id}/{seed_manifest_id}/canonical_seed.csv` | 必须绑定 `seed_manifest_id`，禁止不同 manifest 共用同一路径 |
| artifact 格式 | `UTF-8 CSV` | 不接受 Excel、临时 SQL dump 或人工表格 |
| schema version | `canonical_seed_v1` | 必须写入 `seed_artifact_manifest.schema_version` |
| row 粒度 | 1 行 = 1 条待导入规则 | 通过 `rule_family` 区分 `parse_rule / compliance_rule` |
| 必填列 | `rule_family`、`rule_code`、`source_field`、`target_field`、`rule_logic`、`fail_action`、`severity`、`source_reference`、`conflict_resolution_note`、`is_active` | 缺任一列都不得导入 |
| artifact hash | `sha256` | 必须登记到 `seed_artifact_manifest.artifact_hash` |
| producer | `python:seed_builder` | 必须登记到 `seed_artifact_manifest.producer` |

#### 3.3.2 `seed_artifact_manifest` 的 current 约束

| 字段 | 要求 |
|---|---|
| `artifact_uri` | 必须指向唯一 current canonical seed artifact，且路径必须包含 `seed_manifest_id` |
| `artifact_hash` | 与 artifact 文件内容一致 |
| `schema_version` | 固定 `canonical_seed_v1` |
| `row_count_parse` | `> 0` |
| `row_count_compliance` | `> 0` |
| `approved_by` | 优先记录与 `producer` 不同的 `G1` gate owner；单线程执行时允许固定写 `single_owner_exception` |
| `approval_note` | 必须由 `G1` gate owner 写明“冲突已在 artifact 内裁平、批准导入 current rule_set_version” |
| `is_current` | 恰好 1 行为 `true` |

#### 3.3.3 `seed_builder` 构建指南

| 主题 | 冻结要求 |
|---|---|
| 单一执行主体 | current canonical seed 只允许由 1 个 `python:seed_builder` 进程或 1 名指定 Seed Builder 生成；SQL、人工填表与多脚本并行拼装都不属于正式合同 |
| 批准职责分离 | Seed Builder 只负责生成 artifact 与填写 `conflict_resolution_note`；`seed_artifact_manifest.approved_by` 与 `approval_note` 必须由 `G1` gate owner 填写；若只有 1 名执行者，则 `approved_by` 固定写 `single_owner_exception` 并在 `approval_note` 标明自审降级原因 |
| 正式输入 | 只允许读取已经在 G1 固化的 `field_audit_snapshot`、`target_field_snapshot`、`ods_rule_snapshot`，以及本节内嵌的构建规则；包外文档只能写入 `source_reference`，不能在 Gate 执行期再次作为输入分支 |
| parse 家族最小来源 | 每个 `field_audit_snapshot.decision=parse` 的字段都必须在 canonical seed 中至少生成 1 条 active `parse_rule` 行；同一 `source_field + target_field` 只允许 1 条 current 行 |
| compliance 家族最小来源 | 所有会进入 `fact_rejected`、`fact_pending_issue`、`batch_anomaly_*` 或 `trusted_loss_*` 的正式校验逻辑，都必须先在 canonical seed 中拥有唯一 `rule_code`；不得存在“代码里生效、seed 里缺席”的隐式规则 |
| 冲突裁平 | 同一 `rule_code` 或同一 `source_field + target_field` 若出现多来源冲突，必须由 Seed Builder 先裁成 1 条 canonical 行，并把取舍原因写入 `conflict_resolution_note`；`G1` gate owner 在复核通过后再把批准意见写入 `approval_note`；未裁平不得导入 |
| 伴随产物 | 除 `canonical_seed.csv` 外，必须在同一 `seed_manifest_id` 目录下同步生成 `canonical_seed.build.json`，至少记录 `package_id`、`seed_manifest_id`、`producer`、`approved_by`、`generated_at`、`artifact_hash`、`row_count_parse`、`row_count_compliance`、`source_summary`、`conflict_count` |
| 不可用上抛 | 若构建结束后仍无法得到单一 current artifact，属于“规则 seed 不可用”，只能按 `05_本轮范围与降级说明.md` 重新上抛，不允许开发期临场补规则 |

#### 3.3.4 canonical seed -> `parse_rule / compliance_rule` 固定映射

family 分流与逐列映射必须固定如下，不允许实现者自行挑列导入：

| canonical seed 列 | `parse_rule`（`rule_family=parse_rule`） | `compliance_rule`（`rule_family=compliance_rule`） |
|---|---|---|
| `rule_family` | 只允许 `parse_rule`；命中后只写 `parse_rule`，不得再落 `compliance_rule` | 只允许 `compliance_rule`；命中后只写 `compliance_rule`，不得再落 `parse_rule` |
| `rule_code` | `rule_id = rule_code`，并保留 `rule_code` 原值 | `rule_id = rule_code`，`rule_code = rule_code` |
| `source_field` | `source_field = source_field` | `source_field = source_field` |
| `target_field` | `target_fields = ARRAY[target_field]`，不得改写成多值自由拆分 | `target_field = target_field`，且 `check_field = target_field` |
| `rule_logic` | `parse_logic = rule_logic` | `check_condition = rule_logic` |
| `fail_action` | 原值保留到 `parse_rule.fail_action`，不得在导入时丢弃 | `fail_action = fail_action` |
| `severity` | 原值保留到 `parse_rule.severity`，不得在导入时丢弃 | `severity = severity` |
| `source_reference` | `source_reference = source_reference` | `source_reference = source_reference` |
| `conflict_resolution_note` | 保留到导入审计日志或 build artifact，不得在导入后失踪 | 同左 |
| `is_active` | `is_active = is_active` | `is_active = is_active` |

补充冻结：

1. `parse_rule` 允许从 canonical seed 保留 `fail_action / severity` 作为治理追溯字段，但 parse 执行阶段不得把它们误解释为 compliance 行为。
2. `compliance_rule.check_field` 的正式来源固定为 `target_field`，`source_field` 仅用于追溯原始来源，不得再让实现者在 `source_field` 与 `target_field` 之间自由选择。
3. 同一 canonical seed 行不得同时导入两张目标表；同一 `rule_code` 也不得跨 family 复用为两个 current 规则。

#### 3.3.5 `rule_set_version` 的 current 与可执行性

`rule_set_version.is_current=true` 在 `P1-G1-04` 可以先作为 bootstrap 占位 current 存在，但在满足以下条件前不得被任何 run / API / current 选择逻辑视为“可执行 current”：

1. `seed_manifest_id` 已回指 current `seed_artifact_manifest`；
2. `is_executable=true`；
3. 对应 `G1` 已正式 passed。

换言之，`seed_manifest_id IS NULL` 或 `is_executable=false` 的 current `rule_set_version` 只代表“占位 current”，不代表“可执行 current”。

冻结规则如下：

1. `parse_rule` 与 `compliance_rule` 只允许从 current canonical seed artifact 导入。
2. 包外文档、历史 merged 文稿与自由脚本只允许作为 `source_reference` 的历史溯源，不得成为 Gate 执行期的正式输入。
3. `parse_rule` 与 `compliance_rule` 的每条正式记录都必须带 `source_reference`，用于追溯其 canonical seed 来源。
4. 一旦历史来源之间发生冲突，必须先在 canonical seed artifact 中裁平；不得把冲突留给开发时临场判断。
5. `G1` 验收既要校验 current `seed_artifact_manifest`，也要校验对应 `canonical_seed.build.json` 真实存在且其 `artifact_hash / row_count_* / approved_by` 与 manifest 一致。

## 5. G3：real `rolling` 与 `batch_snapshot` 策略

### 5.1 rolling 冻结规则

| 项 | 冻结值 |
|---|---|
| `run_type` | `rolling` |
| `data_origin` | `real` |
| 完成标准 | 至少 1 条 completed rolling run，且 completed batch `> 1` |
| timepoints | 只允许 non-rerun real completed batch，不得复用 `scenario_replay` |
| baseline 时序 | 当前批只读取上一版冻结 baseline；新版本仅供后续批次使用 |

### 5.1.1 raw 时间跨度预检结论

进入 `G2 / G3` 前必须先验证双输入的 raw 时间跨度是否足以支撑至少两个 2 小时窗口。当前 PG17 已确认：

| 来源表 | 预检列 | 最早时间 | 最晚时间 | 跨度 |
|---|---|---|---|---:|
| `rebuild2.l0_lac` | `上报时间` | `2025-12-01T00:00:00Z` | `2025-12-07T23:59:59Z` | `167.9997` 小时 |
| `rebuild2.l0_gps` | `上报时间` | `2025-12-01T00:00:00Z` | `2025-12-07T23:59:59Z` | `167.9997` 小时 |

因此，在当前冻结输入上，`rolling_window_contract` 已具备切出多个 real completed batch 的时间覆盖条件；后续标准化得到的 `fact_standardized.event_time` 不得把该覆盖范围缩减到 `< 4` 小时而不显式上抛异常。


### 5.1.2 standardized `event_time` 阻断检查

在进入 `G3` 之前，必须再对 `fact_standardized.event_time` 做一次 blocking 检查，而不能仅依赖 raw 时间跨度预检。

| 检查项 | 冻结要求 |
|---|---|
| 检查对象 | 当前 completed `full_initialization` batch |
| 检查表达式 | `MAX(event_time) - MIN(event_time)` |
| 通过标准 | `>= 4` 小时 |
| 失败后果 | 直接阻断全部 `G3` rolling 执行 |
| 证据归档 | 原始 step 证据固定为 `rebuild4/runtime_artifacts/gate/{package_id}/G2/{attempt_id}/P2A-G2-06.json`，并必须被 `rebuild4/runtime_artifacts/gate/{package_id}/G2/{attempt_id}/index.json` 收录 |

因此，`rolling_window_contract` 的正式起跑条件是：**raw 时间跨度预检通过 + standardized `event_time` 阻断检查通过**。

### 5.1.3 `current run / current batch` 选取合同

| 主题 | 冻结结论 |
|---|---|
| current 唯一来源 | 只允许读取 `rebuild4_meta.current_pointer`；任何 API 不得靠“数据库里已经出现 completed rolling run”自动推导 current |
| current run | `current_pointer.current_run_id` 指向的 run；`G2` 初始化时先指向唯一 completed real `full_initialization` run，只有 `G3` handoff 才允许切到 rolling |
| current batch | `current_pointer.current_batch_id` 指向的 batch；初始化阶段固定为 `batch_seq=1`；`G3` 完成 handoff 前允许跟随当前 run 下最新 non-rerun completed batch，`G3` 签发后到 `G8` 结束前必须冻结为同一验收批次 |
| rerun 可见性 | rerun batch 可以在 `/runs` 详情中查看，但不得进入默认 `/flow-overview` 与 `/flow-snapshot/timepoints` |
| 明确排除 | `scenario_replay`、synthetic、fallback 都不得成为 current run / current batch |

### 5.1.4 current 切换原子性合同

| 主题 | 冻结结论 |
|---|---|
| 切换触发 | 只有当第一个 rolling batch 已 completed，且对应 `batch_snapshot`、`batch_flow_summary`、`baseline_refresh_log`、timepoints 都已经落库后，才允许 current 从 initialization 切到 rolling |
| 切换方式 | `/api/flow-overview`、`/api/runs/current`、`/api/flow-snapshot/timepoints` 必须在同一次 `current_pointer` 更新中原子切换；切换前继续返回 initialization current，切换后统一返回 rolling current |
| 失败处理 | 若任一 current 视图仍停在 initialization、另一个已经切到 rolling，则视为 mixed current，直接阻断 `G4` |
| 验收冻结 | `P2B-G3-05` 一旦签发当前验收批次，`G4-G8` 期间不得再推进 `current_pointer`；若必须继续 rolling，只能回到 `G3` 重签新的验收轮次 |

### 5.2 rolling_window_contract

| 主题 | 冻结结论 |
|---|---|
| 系统事件时间 | `fact_standardized.event_time`，默认来源为 `ts_std` |
| 时区 | `Asia/Shanghai` |
| 窗口粒度 | 固定 2 小时 |
| 边界规则 | 左闭右开 `[window_start, window_end)` |
| late data 定义 | `event_time < 已关闭窗口的 window_end`，且该记录在原窗口 completed 之后才进入标准事实层 |
| late data 策略 | 迟到数据只允许以“增量补入”方式进入 rerun batch；原批不改写，rerun 通过 `is_rerun=true` + `rerun_source_batch_id` 追溯，且 rerun 结果不代表整窗重算快照 |
| rerun 触发 | 由 rolling 编排在写下一批前自动扫描并注册 rerun；手工 rerun 不属于正式合同 |
| 幂等键 | 事件级幂等键写入标准事实层并保持全局唯一；因此 rerun 只能补入此前未落库的 late data，不能整窗重写已落库事件；批次级幂等由 `run_id + batch_seq + window_start + window_end` 共同约束 |
| 手工切片 | 不属于正式合同；若客观上无法形成 real 窗口，只能按冻结包允许的异常上抛 |

### 5.3 每个 completed batch 的固定快照形状

每个 completed batch 都必须恰好写入 11 个 metric，覆盖：

- `fact_standardized`
- `fact_governed`

- `batch_anomaly_object_summary`

## 6. 观察工作台与异常工作台的正式合同

### 6.1 观察工作台：固定为“三阶段 + 第三阶段双结果桶”

观察工作台的资格推进不再允许写成“或等价结构”。正式合同固定如下：

| 阶段 | 正式名称 | 含义 | 页面/接口必须出现的核心字段 |
|---|---|---|---|
| Stage 1 | `existence_eligible` | 存在资格 | `existence_progress`、`missing_detail` |
| Stage 2 | `anchorable` | 锚点资格 | `anchorable_progress`、`missing_detail`、`trend_direction` |
| Stage 3 | `baseline_eligible` | 基线资格 | `baseline_progress`、`suggested_action`、`stall_batches` |

第三阶段的输出结果桶固定为：

- `promoted_this_batch`
- `converted_to_issue_this_batch`

因此页面可以展示 waiting / observing / 结果桶，但接口和验收必须始终以“三阶段 + 双结果桶”解释，不再允许“或等价三段式”的自由发挥。

### 6.1.1 观察工作台必须有正式 read model

`/api/observation-workspace` 不允许临时拼接 `obj_*`、`batch_*` 与异常表后再即席推导口径；正式执行必须先产出 `rebuild4_meta.observation_workspace_snapshot`，再由 API 直接读取。

| 字段 | 要求 |
|---|---|
| `run_id` / `batch_id` | 固定指向当前 real batch 语义 |
| `object_key` / `object_type` | 对象主键与类型 |
| `lifecycle_state` | 当前对象生命周期 |
| `current_stage` | 固定只允许 `existence_eligible / anchorable / baseline_eligible / promoted_this_batch / converted_to_issue_this_batch` |
| `existence_progress` | Stage 1 进度 |
| `anchorable_progress` | Stage 2 进度 |
| `baseline_progress` | Stage 3 进度 |
| `missing_detail` | Stage 1/2 缺口说明 |
| `trend_direction` | Stage 2 趋势方向 |
| `suggested_action` | Stage 3 建议动作 |
| `stall_batches` | Stage 3 停滞批次数 |
| `created_at` | 快照生成时间 |

冻结规则：

1. 每个 completed batch 都必须刷新一版 `observation_workspace_snapshot`。
2. `promoted_this_batch / converted_to_issue_this_batch` 只能作为 `current_stage` 的正式结果桶值出现，不得由前端再自由重算。
3. `missing_detail / stall_batches / suggested_action` 的最终口径以该 read model 为准，禁止前后端各自从底表二次推导。

### 6.1.2 对象晋升与三层派生合同

| 主题 | 冻结结论 |
|---|---|
| `obj_cell` 初次注册 | 当 `fact_standardized` 首次出现可解析的 `(operator_code, lac, bs_id, cell_id)` 组合时注册 Cell 对象，初始 `lifecycle_state=waiting` |
| `existence_eligible` | 当前对象在 real completed 批次中已有可归属标准化记录，且对象 key 完整 |
| `anchorable` | 同时满足 `gps_valid_count >= 10`、`distinct_dev_id >= 2`、`p90_radius_m < 1500`、`observed_span >= 1 day`，且不存在 `forbid_anchor=true` 的对象异常 |
| `baseline_eligible` | 已 `anchorable` 且不存在 `forbid_baseline=true` 的对象异常；在 `G3` rolling 场景下还必须累计满足至少 6 个 completed real batch；`G2` full initialization 允许基于冻结全历史窗口一次性计算首批 bootstrap 结果 |
| `lifecycle` 输出 | `G2` 初始化只允许把对象输出为 `waiting / observing / active` 三态；`active` 仅在 `baseline_eligible=true` 时产生；rolling 后续维护 `dormant / retired / rejected` 不改变上述晋升入口 |
| BS 派生 | `obj_bs` 只允许由 active Cell 按 `(operator_code, lac, bs_id)` 聚合派生；没有 active Cell 的 BS 不得凭空生成 |
| LAC 派生 | `obj_lac` 只允许由 active BS 按 `(operator_code, lac)` 聚合派生；没有 active BS 的 LAC 不得凭空生成 |
| `health_state` | 对象健康度由当前未解除异常的最高严重度决定；若来源规则 severity 为 `critical`，在对象异常域中必须映射为 `high` 后再写入 |

### 6.2 异常工作台：最小数据合同

#### 对象级异常最小字段

| 字段 | 含义 |
|---|---|
| `object_key` | 对象主键 |
| `object_type` | `cell` / `bs` / `lac` |
| `anomaly_type` | 例如 `collision_suspect`、`collision_confirmed`、`dynamic`、`migration_suspect`、`gps_bias` |
| `severity` | 固定为 `high / medium / low` |
| `discovered_batch` | 首次发现批次 |
| `forbid_anchor` | 是否禁锚 |
| `forbid_baseline` | 是否禁基线 |
| `evidence_trend` | 恶化 / 稳定 / 改善 / 新发现 |
| `downstream_impact` | 受影响对象列表或其聚合摘要 |
| `suggested_action` | 系统推荐动作 |

#### 记录级异常最小字段

| 字段 | 含义 |
|---|---|
| `anomaly_type` | 记录级异常类型 |
| `affected_rows` | 当前累计条数 |
| `new_rows_current_batch` | 本批新增条数 |
| `fact_destination` | `fact_governed` / `fact_rejected` 等正式去向 |
| `anchor_impact` | 对锚点资格的影响 |
| `baseline_impact` | 对基线资格的影响 |

#### 影响链最小字段

| 字段 | 含义 |
|---|---|
| `object_key` | 上游异常对象 |
| `affected_object_key` | 受影响对象 |
| `affected_object_type` | 受影响对象类型 |
| `impact_type` | 影响类型 |
| `impact_count` | 影响数量 |

冻结规则如下：

1. 页面和接口都必须使用正式字段名 `forbid_anchor`、`forbid_baseline`、`downstream_impact`，不允许前后端各自发明近义字段。
2. 对象级与记录级是固定双视角；记录级视图以类型聚合为正式主语，不以逐条记录作为页面主语。
3. `severity` 的正式枚举固定为 `high / medium / low`。
4. 若来源规则 severity 为 `critical`，在对象级/记录级异常域中必须先映射为 `high`，再写入正式异常读模型。

## 9. compare 的降级与不回灌原则

1. compare 只负责验证 / 对照，不负责生成主链 current。
2. compare 允许返回 `real`、`synthetic` 与 `fallback`。
3. `synthetic` 必须有 banner，明确“仅供辅助验证”。
4. `fallback` 只允许空 envelope，`data = null`，不允许伪造差异对象结果。
5. compare 的任何结果都不得回写 `fact_*`、`obj_*`、`baseline_*`。

## 10. 数据复用与检测策略

### 10.1 执行前数据检测

每个 Phase 在执行前必须先检测数据库是否已存在所需数据。如果存在且通过完整性校验，则跳过数据生成步骤，直接进入下一阶段。

| Phase | 检测 SQL | 通过条件 | 不通过时动作 |
|---|---|---|---|
| G1 | `SELECT COUNT(*) FROM rebuild4_meta.contract_version WHERE is_current=true` | = 1 | 执行完整 G1 |
| G1 快照 | `SELECT COUNT(*) FROM rebuild4_meta.field_audit_snapshot` | = 27 | 跳过快照回灌 |
| G1 seed | `SELECT COUNT(*) FROM rebuild4_meta.parse_rule` + `compliance_rule` | 均 > 0 | 跳过 seed 导入 |
| G2 | `SELECT COUNT(*) FROM rebuild4.fact_standardized` | > 0 且守恒成立 | 跳过标准化 |
| G2 对象 | `SELECT COUNT(*) FROM rebuild4.obj_cell` | > 0 | 跳过对象生成 |
| G2 baseline | `SELECT baseline_version FROM rebuild4_meta.baseline_version WHERE is_current=true` | = 'v1' | 跳过 baseline |
| G2 trusted loss | `SELECT total_rows FROM rebuild4_meta.trusted_loss_summary LIMIT 1` | = 43771306 | 跳过 |
| G3 | `SELECT COUNT(*) FROM rebuild4_meta.batch WHERE run_id LIKE 'RUN-ROLL%' AND status='completed'` | > 1 | 跳过 rolling |

### 10.2 守恒校验公式

```sql
SELECT
  (SELECT COUNT(*) FROM rebuild4.fact_governed WHERE batch_id='BATCH-INIT-001')
+ (SELECT COUNT(*) FROM rebuild4.fact_pending_observation WHERE batch_id='BATCH-INIT-001')
+ (SELECT COUNT(*) FROM rebuild4.fact_pending_issue WHERE batch_id='BATCH-INIT-001')
+ (SELECT COUNT(*) FROM rebuild4.fact_rejected WHERE batch_id='BATCH-INIT-001')
= (SELECT COUNT(*) FROM rebuild4.fact_standardized WHERE batch_id='BATCH-INIT-001')
AS conservation_holds;
```

### 10.3 完全重建路径

如果需要从零重建数据库：

```sql
DROP SCHEMA IF EXISTS rebuild4 CASCADE;
DROP SCHEMA IF EXISTS rebuild4_meta CASCADE;
```

然后按 `03_最终执行任务书.md` 的 Phase 0 到 Phase 2B 顺序执行全部数据步骤。

## 11. LAC 位置快照回灌

### 11.1 来源与目的

- 来源表：`rebuild2._sample_lac_location`（6 行，包含 LAC 的省/市/区位置信息）
- 目标表：`rebuild4_meta.lac_location_snapshot`

```sql
CREATE TABLE rebuild4_meta.lac_location_snapshot (
    snapshot_id SERIAL PRIMARY KEY,
    operator_code TEXT,
    tech_norm TEXT,
    lac TEXT,
    province_name TEXT,
    city_name TEXT,
    district_name TEXT,
    district_cnt INTEGER,
    city_cnt INTEGER,
    districts_list TEXT,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

### 11.3 使用规则

1. LAC 画像展开行直接显示 `province_name`、`city_name`、`district_name`。

5. 本表在 G1 阶段一次性回灌完成，后续不再更新。
