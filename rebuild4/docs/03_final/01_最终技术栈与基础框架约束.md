# 最终技术栈与基础框架约束

## 1. 冻结结论

本轮最终技术栈与基础框架约束已经收口到唯一方案：

- 数据库事实与正式存储统一落在 PostgreSQL 17。
- 后端统一采用 `Python 3.11+ + FastAPI` 提供 RESTful JSON API，并对统一 envelope 负责。
- 前端统一采用全新 `Vue 3 + TypeScript + Vite` 项目，不从 rebuild3 前端整体 fork。
- 页面验收统一以 Playwright 为正式入口。
- launcher 被明确冻结为独立运维入口，但本轮只保留接口保留名与边界，不计入 G0-G8 完成标准。

## 2. 最终技术栈冻结

### 2.1 分层冻结表

| 层级 | 冻结选择 | 最终约束 |
|---|---|---|
| 数据库 | PostgreSQL 17 | 主计算、主存储、主校验环境；所有计数与比例以 PG17 为准 |
| 正式 schema | `rebuild4` + `rebuild4_meta` | rebuild4 业务主表与控制 / 元数据层必须分层 |
| 只读参考 schema | `rebuild2`、`rebuild2_meta`、`rebuild3`、`rebuild3_meta` | 只读参考，不得 current 化或反写 |
| 后端 | `Python 3.11+ + FastAPI` | run / batch 编排、读模型 API、Gate 执行输出统一由后端负责 |
| 查询风格 | SQL-first | 领域真相、聚合、校验、损耗统计以 SQL / 参数化查询为主，不以 ORM 领域模型为主语 |
| 前端 | 全新 `Vue 3 + TypeScript + Vite` 项目 | 承接 `D-002`，项目骨架、路由、状态、请求层全部重建 |
| 前端共享状态 | Pinia | 只承接 UI 共享状态，不承接来源语义判断 |
| 页面验收 | Playwright | 最终正式页面验收入口 |
| 执行证据 | 工作区相对文件路径 | Gate 证据固定写入 `rebuild4/runtime_artifacts/gate/{package_id}/{gate_code}/`，数据库只登记 repo-relative 路径 |
| 合同版本 | 六文件冻结包 manifest | `contract_version` 只锚定六文件正式包，不再使用两文件 hash 口径 |

### 2.2 明确禁止的技术方向

以下方向未被输入、merged 与裁决授权，明确不进入本轮范围：

1. Kafka / Redis / ClickHouse / Spark / 云调度系统等新增基础设施。
2. 以 rebuild3 的路由、Pinia 结构、axios 拦截层或 fallback / synthetic 判断逻辑为起点继续改造。
3. 以 `rebuild3*` 或 `scenario_replay` 结果顶替 rebuild4 正式主链 real 数据。
4. 用 shell 输出、页面文案或人工台账替代数据库事实。
5. 将 compare 结果、legacy 字段或 launcher 状态冒充主链业务真相。

## 3. 统一 API 合同

### 3.1 顶层 envelope 固定形状

所有正式 API 顶层都必须固定包含以下 6 个字段：

- `data_origin`
- `origin_detail`
- `subject_scope`
- `subject_note`
- `context`
- `data`

**正式写法只允许下面这一种顶层结构：**

```json
{
  "data_origin": "real",
  "origin_detail": "run_id=RUN-..., batch_id=BATCH-...",
  "subject_scope": "batch",
  "subject_note": null,
  "context": {
    "contract_version": "rebuild4-final-freeze-2026-04-06-v6",
    "rule_set_version": "RULESET-...",
    "run_id": "RUN-...",
    "batch_id": "BATCH-..."
  },
  "data": {}
}
```

冻结约束如下：

1. `context` 必须是顶层对象；不再使用 `context.*` 这种可被误读为扁平化字段的写法。
2. `subject_scope` 只允许 `run / batch / object / baseline / governance / compare / initialization` 七值。
3. API 合同权威统一以 `03_最终执行任务书.md` 的附录 B 与附录 B.1 为准；附录 C 只负责页面绑定并必须逐项镜像，不允许两套口径并行。
4. 附录 B.1 列的是请求 path/query 参数；附录 B 与附录 C 列的是响应 envelope 中的 `context` 归一化键。请求参数与 `context` 键允许不同名，但映射必须按 endpoint-scoped 明细固定，同名 query 参数不得跨 endpoint 自由套用。
5. 主流程 API 只允许 `real` 或诚实空状态；`synthetic` / `fallback` 只允许出现在 compare 降级语义中。

### 3.2 诚实空状态合同

| 场景 | 适用 API | HTTP 状态码 | `data_origin` | `data` 形状 | 附加规则 |
|---|---|---:|---|---|---|
| 列表型空态 | `/api/flow-snapshot/timepoints`、`/api/batches`、`/api/objects`、`/api/profiles/*`、`/api/observation-workspace`、`/api/anomaly-workspace`、`/api/governance/fields`、`/api/governance/tables`、`/api/governance/field_audit`、`/api/governance/target_fields`、`/api/governance/ods_rules`、`/api/governance/ods_executions`、`/api/governance/parse_rules`、`/api/governance/compliance_rules` | 200 | `real` | `[]` | 必须写明 `subject_note`，禁止 204 |
| 汇总/概览型空态 | `/api/flow-overview`、`/api/flow-snapshot`、`/api/runs/current`、`/api/baseline/current`、`/api/baseline/current/diff`、`/api/initialization/latest`、`/api/governance/overview`、`/api/governance/trusted_loss?breakdown_type=overview` | 200 | `real` | `{}` | 必须写明为什么为空，例如“暂无已完成批次” |
| 显式 key 不存在 | `/api/objects/:key/detail` 等显式主键详情 | 404 | 不属于 honest empty | `null` 或错误对象 | 这是“对象不存在”，不是“诚实空状态” |
| compare fallback | `/api/validation/compare` | 200 | `fallback` | `null` | 必须有 banner，且不得展示伪结果 |

补充规则：

1. 主流程、对象、baseline、profile、initialization、governance 的 honest empty 一律使用 `data_origin=real`。
2. `HTTP 204 No Content` 在本冻结包中不作为正式空态写法。
3. compare 之外，任何页面或 API 出现 `fallback` 都视为阻断级失败。
4. `flow-snapshot`、`runs`、`governance` 这类混合家族不得再用“整个家族统一 `data={}`”的简写偷渡；列表 endpoint 与汇总 endpoint 必须逐条声明空态形状。

### 3.3 统一错误响应格式

成功态与错误态必须严格分开：

1. 只有 `HTTP 2xx` 才使用 six-field envelope。
2. `HTTP 4xx / 5xx` 不得伪装成 honest empty，也不再套 six-field envelope。
3. “无数据”必须走 `HTTP 200 + honest empty`，不能借错误响应偷渡。

正式错误体最小结构固定如下：

```json
{
  "error_code": "object_not_found",
  "error_message": "object_key=CELL-... not found",
  "error_context": {
    "request_path": "/api/objects/:key/detail",
    "contract_version": "rebuild4-final-freeze-2026-04-06-v6"
  }
}
```

| 场景 | HTTP 状态码 | 最小结构 | 约束 |
|---|---:|---|---|
| 参数非法 / 合同不满足 | 400 | `error_code`、`error_message`、`error_context` | `error_code` 固定为 `invalid_request` 或 `invalid_parameter` |
| 显式 key 不存在 | 404 | `error_code`、`error_message`、`error_context` | `/api/objects/:key/detail` 固定返回 `object_not_found` |
| 服务内部异常 | 500 | `error_code`、`error_message`、`error_context` | `error_code` 固定为 `internal_error`，不得泄露堆栈 |

### 3.4 页面主语与 `subject_scope` 的绑定原则

1. `flow-overview` / `flow-snapshot` / `observation-workspace` / `anomaly-workspace` 的正式主语统一归为 `batch`。
2. `runs` 的正式主语统一归为 `run`，但 `GET /api/batches/:id/detail` 的详情 envelope 必须固定为 `batch`，不能继承父路径的 `run`。
3. `objects` 与三类 profile 的正式主语统一归为 `object`。
4. `baseline` 统一归为 `baseline`，`governance` 统一归为 `governance`，`compare` 统一归为 `compare`，`initialization` 统一归为 `initialization`。
5. 页面/API/表/来源矩阵已经在正式包内闭合，不再允许实现者引用包外矩阵文件自行补主语。

## 4. governance、compare 与 launcher 的边界

### 4.1 governance 与 compare

| 模块 | 冻结约束 |
|---|---|
| governance | 必须采用多 endpoint lazy-load；不可回退为巨型单接口 |
| ODS | 定义层与执行层必须分离为 `ods_rules` 与 `ods_executions` 两组接口 |
| parse / compliance | 作为 governance 正式能力，必须有独立接口，且正式数据来自 `rebuild4_meta` |
| trusted loss | 正式接口命名固定为 `trusted_loss`；不再使用 `trusted_filter` 旧命名 |
| compare | 正式辅助验证模块；不计入 12 个正式页面，但仍属于 `G7/G8` 的正式放行范围；允许 `real / synthetic / fallback`，其中 `fallback` 只允许空 envelope |

### 4.2 launcher 的最终落点

launcher 不能因为“不计入 Gate”而在最终文档中消失，但它在本轮的身份必须只有一种解释：

| 主题 | 冻结结论 |
|---|---|
| 身份 | 独立运维入口，不属于数据主链页面 |
| 启动方式 | 独立 Python 脚本启动，不依赖主系统前后端 |
| WebSocket | `/ws/launcher/status`、`/ws/launcher/logs/:service` |
| REST API | `/api/launcher/services*` 系列仅负责服务控制、状态与日志 |
| 与数据合同关系 | 不进入统一数据来源合同，不承接主链业务主语 |
| 与 Gate 关系 | 不计入 G0-G8 成功标准，不阻断正式数据链冻结 |
| 本轮实现义务 | 仅冻结接口保留名与边界；本轮不要求进入 `03_最终执行任务书.md` 的正式执行/验收闭环 |

## 5. 前端框架与复用边界

### 5.1 允许复用的范围

仅允许在逐组件审计通过后复制以下类型的纯展示组件：

- badge / tag / indicator 类纯渲染组件
- 不含取数逻辑、来源语义判断与 fallback 逻辑的展示卡片
- 与 `UI_v2` 一致的基础视觉组件

### 5.2 禁止复用的范围

以下内容一律禁止直接复制或整体继承：

- rebuild3 路由结构
- rebuild3 Pinia store 结构
- rebuild3 axios 请求层 / 拦截层
- 任意包含 `fallback` / `synthetic` / 主链语义判断的逻辑代码
- 旧前端项目骨架、隐式页面主语和旧状态机

### 5.3 前端正式约束

1. 主流程无 real 数据时必须显示诚实空状态，不能拼 fake 卡片。
2. legacy 信息只允许通过 `legacy_ref` 进入嵌套参考区，不进入主筛选与主状态模型。
3. compare 页必须显式显示“辅助验证”身份，`synthetic` / `fallback` 必须出现 banner。
4. governance 页必须真实消费 ODS 定义 / 执行分离、trusted loss 与规则 seed 的正式接口。
5. `/objects/:key` 是 `/objects` 的详情态，不单独计入 12 个正式页面；三类 profile 路由单独计页但允许共用一个 Playwright 用例覆盖。

### 7.4 GlobalStatusBar

顶栏下方全宽状态条，按系统状态切换显示模式：

| 系统状态 | 显示模式 | 样式 |
|---|---|---|
| 正常（无处理中任务） | 隐藏 | 不占用空间 |
| 批次处理中 | 蓝色脉冲 + "正在处理批次..." | blue pulse animation |

### 7.5 DataOriginBanner 全页面覆盖

所有页面都必须包含 `DataOriginBanner` 组件。当 `data_origin=real` 时该组件隐藏（不占用空间），采用防御性编程确保来源判断始终存在。当 `data_origin=synthetic` 时显示 amber 警告横幅，`data_origin=fallback` 时显示 rose 警告横幅。

## 8. 正式执行约束

1. 任何正式 API、任务书步骤或校验清单都只使用 `03_final/00_05` 正式命名。
2. 前端新项目、统一 envelope、ODS 分离、launcher 独立入口都已经升级为最终正式约束。
3. trusted 损耗指标在正式文档中统一使用 `filtered_with_lon_lat` 命名，不再使用旧歧义写法。
4. 页面/API 的正式主语、上下文键与空态合同，必须以本冻结包内矩阵为唯一依据，不得回到包外文件自行解释。
