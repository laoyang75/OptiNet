# 最终执行任务书


## 1. 执行总则

1. 严格按 `Phase 0 (G0) -> Phase 1 (G1) -> Phase 2A (G2) -> Phase 2B (G3) -> Phase 3 (G4 -> G5 -> G6) -> Phase 4 (G7) -> Phase 5 (G8)` 串行推进，任何 Gate 未通过都不得提前进入后续阶段。
2. G0 只负责冻结包只读校验；G1 的第一个原子动作必须是 raw 时间跨度预检；只有 `P1-G1-01` 通过后才允许首个数据库写入落到 `P1-G1-02`。
3. 主链 real 数据只允许来自 `rebuild2.l0_gps=38,433,729` 与 `rebuild2.l0_lac=43,771,306` 两个冻结输入；`rebuild3*` 永远只读参考。
4. 所有涉及计数、比例、差集、缺口的正式说明都直接写入 PG17 冻结值，不允许再写“以数据库为准”替代具体数字。
5. trusted 损耗统一使用 `trusted_loss` / `filtered_with_lon_lat=11,350,552` 这一正式命名；compare 是正式辅助验证模块但不计入 12 个正式页面；launcher 仅冻结边界，不进入正式 Gate 闭环。
6. 本包当前版本为 `rebuild4-final-freeze-2026-04-06-v6`；后续若修改任一正式文件，必须整包升版。

## 2. 步骤编号规则

- 步骤锚点使用 `P<phase>-G<gate>-<seq>`，例如 `P2A-G2-03`。
- Phase 5 正式归属 `Gate G8`；Playwright 专用锚点继续使用 `P5-GPW-00` 到 `P5-GPW-11`，但 `gate_check_item` 与 `gate_run_result` 必须统一挂在 `gate_code=G8` 下。
- `04_最终校验清单.md` 必须与本文件步骤锚点一一对应，不允许缺项或多项。

## 3. 证据归档与 Gate 写入规则

1. 文档中的 `artifacts/gate/...` 是简写；物理落地路径固定为工作区相对目录 `rebuild4/runtime_artifacts/gate/{package_id}/{gate_code}/{attempt_id}/`。
2. `attempt_id` 是 Gate 级重试维度，固定从 `attempt_001` 开始递增；同一 Gate 的不同 attempt 禁止共享 `index.json` 或 step 证据目录。
3. `G0` 在控制表尚不存在前，唯一正式 passed 状态就是 `rebuild4/runtime_artifacts/gate/{package_id}/G0/attempt_001/index.json`；`P1-G1-04` 只能原样镜像该 artifact 到 `gate_run_result`，不得另起口头状态。
4. `gate_run_result.evidence_ref` 只登记 repo-relative 路径，默认指向该 Gate 当前 attempt 的 `index.json` 汇总清单；原始 step 证据固定落在同目录下的 `{step_id}.json/.png/.zip/.html`。
5. 文档中若写 `执行主体=SQL`，正式含义一律是“SQL + 同一步 Python evidence_writer”：SQL 负责查询/DDL，Python 负责把本 step 的结果序列化到 `{step_id}.*` 并更新当前 attempt 的 `index.json`；不得把落证动作挪到别的 step。
6. 每个 Gate 只能有 1 名 gate owner 写 `gate_run_result.status`；多工程师或多 sub-agent 可以协作，但只有 gate owner 能签发 `passed/failed` 与汇总清单，且 `gate_run_result.gate_owner` 必须与 `gate_definition.gate_owner` 一致。
7. 任何 Gate 在证据文件不存在、不可读、为空或未被当前 attempt 的 `index.json` 收录时，均不得写 `status=passed`。

### 3.1 冻结 Gate owner 角色矩阵

| Gate | 固定 `gate_owner` | `backup_role` | handoff_to |
|---|---|---|---|
| `G0` | `release_owner` | `seed_owner` | `G1` |
| `G1` | `seed_owner` | `release_owner` | `G2` |
| `G2` | `pipeline_owner` | `seed_owner` | `G3` |
| `G3` | `pipeline_owner` | `api_owner` | `G4` |
| `G4` | `api_owner` | `pipeline_owner` | `G5` |
| `G5` | `governance_owner` | `api_owner` | `G6` |
| `G6` | `baseline_owner` | `api_owner` | `G7` |
| `G7` | `compare_owner` | `release_owner` | `G8` |
| `G8` | `frontend_owner` | `api_owner` | 发布收口 |

补充规则：

1. `gate_definition.gate_owner` 只允许使用上表角色名，不允许实现者在运行时自定义。
2. 单线程执行时，同一人可以兼任多个角色，但表里仍必须写冻结角色名。
3. `seed_artifact_manifest.approved_by` 优先要求独立于 `producer`；若只有 1 名执行者，则固定写 `single_owner_exception` 并在 `approval_note` 说明原因。

## Phase 3 内部顺序与职责冻结

1. Phase 3 必须严格按 `G4 -> G5 -> G6` 推进；允许多工程师或多 sub-agent 协作，但同一时刻每个 Gate 只允许 1 名来自冻结角色矩阵的 gate owner 负责签发 `gate_run_result` 与最终接口合同。
2. `G4` 的最小骨架深度固定为：路由注册、query/path 参数校验、six-field / 错误响应 envelope、honest empty、request 参数到 `context` 键的归一化映射、HTTP 合同测试 stub；不在 `G4` 宣称 governance 数字聚合、baseline 历史计算或 profile 业务派生已经完成。
3. `G5` 只负责 `/api/governance/*`、治理快照、ODS 拆分、seed 与 `trusted_loss` 透传；不得回头重命名 `G4` 已冻结的主流程接口。
4. `G6` 只负责 `/api/baseline*` 与 `/api/profiles/*`；不得在 `G6` 修改 governance 路由、参数名或 compare 语义。
5. `G5/G6` 的业务聚合、页面联调与证据归档，只有在 `G4` 的 HTTP 合同测试通过且 `G4/{attempt_id}/index.json` 已存在后才允许展开。

### Phase 5 验收方法：Playwright 对比验收

每个页面完成后，必须同时截图 rebuild3 UI_v2 和 rebuild4 的相同路由，进行并排对比验收：

```
rebuild3 UI_v2:  http://127.0.0.1:47122/{route}
rebuild4:        http://127.0.0.1:47132/{route}
```

验收标准是"结构一致"而不仅仅是"能打开"：

| 验收维度 | 验收标准 |
