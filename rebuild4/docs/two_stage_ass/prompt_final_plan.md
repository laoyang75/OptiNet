## 重要背景

### 系统开发历史

这个系统的 ETL 管道和画像管道（`profile.py`）是由 Claude 开发并验证的。三个独立 agent 的审计结果存在偏差：

- **Claude 的审计**与实际开发逻辑自洽性最强，应作为**主线参考**
- **Codex 和 Gemini 的审计**是完整代码审计，可能包含大量对现有代码的泛化评价。其中有价值的发现应吸收，但不应让它们把方案拉偏

### 核心意图：流式评估 → 可观察模块

人类对 Q2 的决策纠正了一个关键误解。原始审计把问题框定为"两套数据体系选哪个"，但人类的实际意图是：

**流式评估实验（Day 1→7 收敛曲线）已经证明了数学正确性，下一步应该把这个实验过程固化成一个可控制、可配置、可观察的流式画像模块。**

具体来说：

1. **已完成**：`profile.py` 能全量计算画像（一次性出结果）
2. **已验证**：流式评估实验证明逐天累积 = 批量结果（eval_stream_convergence 表）
3. **缺失的关键一步**：把实验的逐天累积逻辑固化成 profile.py 的一个运行模式，产出可观察的 snapshot 序列和 diff 数据
4. **连接点**：流转总览/快照/观察工作台的概念是对的，它们的数据源应该是流式 snapshot 的 diff，不是旧的 batch_flow_summary

这意味着：
- 流转可视化页面**不应该下线**，而是改接流式 snapshot 数据
- `profile.py` 需要增加 streaming 模式（按天/按窗口累积构建 snapshot）
- 每个 snapshot 之间的 diff = 流转数据（哪些 Cell 状态变了）
- 参数可配置 → 改参数重跑 streaming → 观察不同参数下的收敛行为

### 流式评估实验的关键资产

以下是实验中已经写过并验证过的逻辑（在 `eval_stream_convergence` 表中可查），方案应复用而非重写：

- 按天切分 `etl_filled`（`DATE(ts_std)`），7 天数据
- 逐天累积构建 Cell 画像：独立观测去重 → 中位数质心 → P50/P90 → 生命周期判定
- 每天 snapshot 与基准对比：覆盖率、质心偏差、Active 召回率
- 收敛曲线数据结构（day, stream_cell_count, cell_coverage_pct, cell_centroid_median_m, active_recall_pct, ...）

详见 `rebuild4/docs/02_profile/06_流式评估.md`。
