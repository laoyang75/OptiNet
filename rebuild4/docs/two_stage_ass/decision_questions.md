## Q1：系统长期应采用哪套对象模型作为单一权威语义

**问题**：系统当前同时存在 `etl_dim_*` 画像链路和 `obj_*` 工作台治理链路，二者数据语义不同、生产方式不同，没有自然收敛。当前 `profile.py` 生成最新画像产物（`etl_dim_cell/bs/lac`），而对象工作台等页面仍读取 `obj_*`，造成页面心智模型割裂。

**为什么需要人类决策**：这是研究目标的根本取舍——是把系统定义为"IP 画像研究平台"（以 `etl_dim_*` 为核心），还是保留"治理监控平台"（以 `obj_*` + 批次中心为核心），还是两套并存分别服务不同场景。

**选项**：

- **A**：全面切换到 `etl_dim_*` — 将首页、工作台、对象浏览等页面全部改接 `etl_dim_*`，废弃 `obj_*` 的前台展示角色。
- **B**：维持双域并存，通过 UI 边界标识区分。
- **C**：以 `etl_dim_*` 为主权威，`obj_*` 降级为只读参考。

**推荐选项**：C

**人类决策**：A

## Q2：流转可视化页面应如何与流式画像模块对接

> ⚠️ 本问题经人类纠正重述。原始 Q2 将流转页面框定为"过期数据是否下线"的选择题，但人类的实际意图是：**流式评估实验（Day 1→7 收敛曲线）验证了数学正确性，但这个实验过程本身就应该固化为可控、可配置、可观察的流式模块**。流转可视化页面的概念是对的，问题是它的数据源应该从旧的 `batch_flow_summary` 切换到流式画像 snapshot 的 diff 数据。

**问题**：`profile.py` 当前是全量一次性计算，产出最终画像但不保留中间过程。而流式评估实验已经证明逐天累积构建画像是可行的（Day 7 与批量结果完全一致）。研究人员需要的是：将这个逐步累积的过程变成一个可观察的模块——能看到每天新数据进来后，哪些 Cell 从 waiting 变成 active，质心偏移了多少，覆盖率怎么增长。这恰恰是流转总览、流转快照、观察工作台原本要展示的信息。

**为什么需要人类决策**：这决定了 profile.py 是保持"黑盒全量计算"还是增加"可观察的流式模式"。后者意味着需要：(1) profile.py 支持按时间窗口累积运行并存储 snapshot；(2) 计算 snapshot 之间的 diff 作为流转数据；(3) 流转页面改接这些 snapshot/diff 数据。这是一个非平凡的工作量决策。

**选项**：

- **A**：将流式评估逻辑固化到 profile.py — 增加 streaming 模式，按天/按批次产出 snapshot 序列，自动计算 diff，流转页面改接 snapshot 数据。后果：流转过程可观察、参数可调整重跑，但需要设计 snapshot 存储和 diff 计算逻辑。
- **B**：保持 profile.py 全量计算不变，流转页面改为展示"画像统计概览" — 流转总览改为展示当前画像的 lifecycle 分布统计（不展示时间序列变化），观察工作台改为展示 etl_dim_cell 中 waiting/observing 的列表。后果：工作量小，页面有数据但失去"流转过程"的观察能力。
- **C**：当前阶段下线流转页面，后续作为独立模块重建 — 先跑通画像和对象浏览，流式观察能力作为下一阶段独立交付。后果：本轮工作量最小，但研究人员暂时无法观察流转过程。

**推荐选项**：A

**推荐理由**：流式评估实验的核心逻辑已经写过一遍（eval_stream_cell_profiles 的 SQL），固化成模块的增量工作量可控。而且这是整个系统的差异化价值——如果只看静态画像，和直接查表没有区别。

**人类决策**：A — 流式评估实验证明了正确性，下一步就应该把它变成可控可配置可观察的模块。放大到更多数据时肯定会出问题，需要这个观察能力来发现和调整。

**选项**：

- **A**：仅提炼为 `profile.py` 内的 Python 常量块。
- **B**：外化到配置文件（`.yaml`），与代码版本一起管理，无 UI。
- **C**：外化到数据库配置表，纳入 `rule_set_version` 版本管理。

**推荐选项**：B

**人类决策**：B — 配合 Q2=A 的流式模式，改参数后重跑 streaming 观察效果，配置文件足够。
