## 关键背景：第一轮开发的经验教训

本项目已经经历过一轮完整开发，以下是核心教训：

### 数据库已存在，可直接复用
- `rebuild4` 和 `rebuild4_meta` schema 已在 PG17 中存在，包含完整的控制层、业务数据、治理快照、trusted loss 等
- **开发前必须先按 `02_数据生成与回灌策略.md` 第 10 节的检测表逐项校验**，数据存在则跳过生成步骤
- 四分流守恒已验证：82,205,035 = 67,963,244 + 9,493,333 + 1,461 + 4,746,997
- 如果需要新增 `lac_location_snapshot` 表（第 11 节），从 `rebuild2._sample_lac_location` 回灌

### 前端和 API 是本轮重点
第一轮失败的根因：冻结文档只定义了数据合同（envelope 形状、context 键、空态规则），没有定义 UI 交互规范。本轮文档（v7）已补齐：
- `01` 第 7 节定义了 CSS token、10 个共享组件、侧边栏布局
- `03` 每个 P5-GPW 步骤都有 **UI 结构规范**，精确到组件名、配色、布局
- `04` 每个 GPW 校验项都附带了 UI 结构验证点

### UI 设计的权威来源
rebuild4 的 UI 设计以 rebuild3 UI_v2 的实际实现为基准（`rebuild3/frontend/src/pages/*.vue`），不以设计文档描述为准。rebuild3 是中间状态（失败的重构），但其 UI_v2 前端实现质量高于其后端。画像部分核心代码参考 rebuild2。

### Playwright 验收是对比验收
每个页面完成后，必须同时截图 rebuild3 UI_v2（http://127.0.0.1:47122）和 rebuild4（http://127.0.0.1:47132）的同一路由，进行并排对比。验收标准是**结构一致**，不是"能打开"。

## 开发规则

**每个阶段完成后，必须做以下三件事，再进入下一阶段：**

1. **比对文档**：将已完成内容与 `03_最终执行任务书.md` 中当前阶段的任务要求逐条核对；
2. **比对校验清单**：检查 `04_最终校验清单.md` 中对应阶段的验收项（含 UI 结构验证点）；
3. **声明进入下一阶段**。

**Phase 5 (G8) 特殊规则：**
- 每个页面完成后，用 Playwright 同时截图 rebuild3:47122 和 rebuild4:47132 的同一路由
- 验收维度：布局结构、组件类型、数据字段、交互模式、空态表现、配色系统
- 只有并排对比通过后，该页面才算完成

**严禁以下行为：**
- 使用通用 auto-table 渲染代替文档指定的专用组件（如 LifecycleBadge、FactLayerBadge 等）
- 返回扁平 DB 行让前端自己聚合，后端必须预计算 summary、delta、flow distribution
- 跳过 UI 结构规范中指定的任何区域或组件
- 偏离 `05_本轮范围与降级说明.md` 中标明的降级决策
