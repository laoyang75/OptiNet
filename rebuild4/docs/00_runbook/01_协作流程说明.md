# 协作流程说明

## 1. 为什么改成三轮同题评估

因为当前项目已经具备三类基础：

- 数据库已经有可核对事实
- UI_v2 已经存在
- rebuild2 / rebuild3 的需求与历史经验已经足够多

所以新的目标不是“边做边想”，而是：

1. 先统一计划
2. 再统一细节
3. 最后统一执行任务

## 2. 每一轮都固定做什么

每一轮都固定经历 5 个步骤：

1. 我准备该轮 prompt
2. 3 个 agent 同题独立输出
3. 我进行合并
4. 如有冲突，产出裁决文件
5. 你裁决后进入下一轮

## 3. 什么问题值得进入裁决文件

只有下面这几类问题值得让你裁决：

- 会影响整体方向的问题
- 会影响后续 schema / API / 页面合同的问题
- 3 个 agent 明显分歧且我不能单方面裁决的问题
- 影响最终执行成本或验收口径的问题

下面这些不应该进裁决文件：

- 纯措辞差异
- 结构顺序差异
- 我可以根据已冻结原则直接裁决的问题

## 4. 这一套机制要避免什么

它主要是为了避免 rebuild3 里出现的 4 类问题：

- 同时执行多份 prompt
- agent 各做一块，最后没有互证
- 人类被迫持续微操
- 页面、API、数据库真相分离
