# Q01: 数据起点决策

## 问题

重构后的治理链路，从哪一层数据开始执行？

## 背景

当前 PG 上存在两个起点层级的数据：

### Layer0（真正原始层）
- `Y_codex_Layer0_Lac`：1.2亿行，48GB，36个字段
- `Y_codex_Layer0_Gps_base`：1.3亿行，53GB，36个字段
- 包含中文列名（记录id、运营商id、原始lac、数据来源、北京来源）
- 包含原始信号字段（sig_rsrp, sig_rsrq, sig_sinr, sig_rssi）
- Layer0_Lac 和 Layer0_Gps_base 结构完全相同，区别是 Lac 基于 LAC 解析，Gps_base 基于 GPS 解析

### Layer2_Step06（合规过滤后）
- `Y_codex_Layer2_Step06_L0_Lac_Filtered`：2178万行，13GB，37个字段
- 在 Layer0 基础上增加了：is_connected, operator_id_raw, device_id, tech_norm
- 已经完成运营商过滤、LAC编码过滤、位置过滤
- 行数约为 Layer0 的 18%（过滤掉了 82% 的数据）

## 我的理解

- 如果从 Layer0 开始：支持真正的完整回归，但 101GB 数据量大，本地处理时间长；需要重新实现标准化解析逻辑（中文字段名→英文、运营商识别、制式识别等）
- 如果从 Layer2_Step06 开始：跳过了标准化和合规过滤，直接从已过滤的 2178万行开始；不能验证"原始数据被过滤了多少"这个环节；但开发速度更快，链路更短
- 如果两层都保留但分阶段实现：一期先从 Layer2_Step06 开始跑通后续链路；二期再把 Layer0→Layer2 的标准化逻辑补回来

## 选项

### A. 从 Layer0 开始（完整起点）
- 重构包含标准化解析步骤（Layer0 → 标准化 → 合规过滤 → 后续链路）
- 优点：完整覆盖，支持真正的完整回归
- 缺点：工作量大，需要额外处理中文字段名映射

### B. 从 Layer2_Step06 开始（过滤后起点）
- 跳过标准化和合规过滤，直接从 Step06 开始
- 优点：开发快，数据量小，后续链路可以更快跑通
- 缺点：不支持完整回归到 Layer0；run 必须标注 origin_scope=filtered_start

### C. 分阶段实现（推荐）
- 一期：从 Layer2_Step06 开始，先跑通 可信LAC→Cell→BS→GPS→信号→回归→画像
- 二期：补回 Layer0→标准化→合规过滤 的逻辑
- 优点：一期交付快，二期补完整性
- 缺点：一期的"完整回归"步骤实际上回归到 Step06 而不是 Layer0

## 请选择：A / B / C

选择 A 
这个处理没有那么长时间，重构就是要确保流程每一步都准确清晰，且文档化。
