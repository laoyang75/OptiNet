# Q02: 新旧表命名策略

## 问题

重构后的表是否继续使用 `Y_codex_*` 命名？还是采用新的命名体系？

## 背景

当前 56 张表使用 `Y_codex_Layer{N}_Step{NN}_{描述}` 命名，例如：
- `Y_codex_Layer2_Step04_Master_Lac_Lib`
- `Y_codex_Layer3_Step30_Master_BS_Library`
- `Y_codex_Layer4_Final_Cell_Library`
- `Y_codex_Layer5_BS_Profile`

还有观察表 `Y_codex_obs_*` 和 SMOKE 测试表 `*__MCP_SMOKE`。

开发基础文档建议工作台元数据表使用 `wb_*` 前缀，字段治理使用 `meta_*` 前缀。

## 我的理解

- 旧命名包含 Layer/Step 编号，UI 文档明确要求"不以 Step 表命名"
- 但旧表中的数据是有效资产，完全改名会增加迁移复杂度
- 重构是全新系统，应该有机会建立更清晰的命名

## 选项

### A. 完全新命名
治理链路产物表使用新命名体系：
```
raw_standardized          # 标准化后原始记录
dim_lac_trusted           # 可信 LAC
dim_cell_stats            # Cell 统计
dim_bs_trusted            # 可信 BS
fact_gps_corrected        # GPS 修正后记录
fact_signal_filled        # 信号补齐后记录
fact_final                # 完整回归最终明细
profile_lac / profile_bs / profile_cell  # 画像
```
工作台表：`wb_*`，字段治理表：`meta_*`

旧 `Y_codex_*` 表保留在 PG15 不动，PG17 上全部用新名。

### B. 保留旧名 + 新增工作台表
治理链路产物表继续用 `Y_codex_*` 命名（数据从 PG15 迁移过来直接用）。
仅新增 `wb_*` 和 `meta_*` 表。
UI 通过工作台的步骤注册映射到旧表名。

### C. 混合方案
- PG17 上保留旧 `Y_codex_*` 表作为只读参考
- 新建一套新命名的表作为重构后的治理链路产物
- 新旧表共存，新表是实际写入目标，旧表用于对比验证
- 验证通过后逐步废弃旧表

## 请选择：A / B / C

选择 A
应该放在一个独立的 schema 中，至于过程文件，我希望是用另一种命名方式， 当前的书库中的的命名包含了大量无用的中间过程。包括冒烟测试一类的。
