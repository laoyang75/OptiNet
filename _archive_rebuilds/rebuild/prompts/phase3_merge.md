## 3. 合并逻辑

### 3.1 方案裁决规则

对每个维度的实施方案，按以下逻辑裁决：

| 情况 | 裁决方式 |
|------|---------|
| 3 个 agent 方案一致 | 直接采纳 |
| 2 个一致 + 1 个不同 | 采纳多数方案，记录少数意见供参考 |
| 3 个方案各不相同 | 生成决策问题文件（ABC选项），交用户裁决 |
| 某 agent 方案明显过度工程化 | 标注"[Agent名] 方案过重"，不采纳该方案 |
| 某 agent 方案明显过于简陋 | 标注"[Agent名] 方案不充分"，不采纳 |

### 3.2 关键决策点（必须裁决）

以下是三个 agent 最可能产生分歧的关键问题。请逐项裁决：

#### 决策1：合规规则存储方案
- 选项A：扩展 meta_field_registry 表
- 选项B：新建 meta_source_field_compliance 独立表
- 选项C：复用 meta_field_mapping_rule（rule_type='compliance'）
- **裁决标准**：简单性 > 扩展性 > 规范性。这是调试工具，不是生产系统。

#### 决策2：合规率计算源表
- 选项A：raw_records（2.5亿行，最全但最慢）
- 选项B：fact_filtered（2180万行，已过滤但保留主要字段）
- 选项C：按字段分别选源表
- **裁决标准**：数据代表性 × 计算可行性。如果 fact_filtered 已包含所有待检字段且数据量合理，优先选 fact_filtered。

#### 决策3：合规率计算策略
- 选项A：全量计算 + 快照存储
- 选项B：抽样计算（10%采样率）
- 选项C：按分区/批次增量计算
- **裁决标准**：fact_filtered 2180万行的全量 COUNT+CASE 在 PG17 上通常 30-60秒可完成，如果可行则选全量。

#### 决策4：前端模块化方案
- 选项A：ES Module（import/export）
- 选项B：IIFE + 全局命名空间
- 选项C：script 标签顺序加载 + 共享全局对象
- **裁决标准**：浏览器兼容性 + 开发体验。现代浏览器支持 ES Module。

#### 决策5：workbench.py 拆分粒度
- 粗粒度（3个模块）vs 中粒度（5个模块）vs 细粒度（7+个模块）
- **裁决标准**：每个模块 200~400 行最佳。根据 2,045 行总量，5个模块（每模块~400行）最合理。

#### 决策6：P3 字段治理前端组织
- 选项A：Tab 切换（"原始字段定义" | "过程字段注册"）
- 选项B：上下两个独立区域
- 选项C：统一表格，用 field_type 列区分
- **裁决标准**：参考 V2 fields.html 的设计意图。

### 3.3 质量校验

- 如果某个 agent 对字段治理（维度F）的分析过于笼统（没有 DDL、没有具体字段列表），标注"审计不充分"
- 如果某个 agent 的拆分方案只是"建议拆分"但没有具体的模块划分和函数分配，标注"缺乏可执行性"
- 优先采信包含代码级细节的方案

## 5. 注意事项

- **最终计划必须具体到文件名和函数名**，不允许模糊的"需要重构"之类的结论
- 任务排序必须考虑依赖关系：数据库变更 → 后端 API → 前端 UI
- 代码拆分应在功能开发之前完成（否则在大文件上改代码很痛苦）
- 合并时如果发现某个 agent 的方案有独到的细节（其他两个没想到的），应保留
- 字段治理的 DDL 必须与现有 meta schema 表结构兼容
- 估算工作量时考虑 AI 辅助开发的效率（通常比手写快 3-5 倍）
- 所有方案必须遵循"AI 开发纪律"（单文件≤500行、函数≤80行、不引入框架等）
- **不要修改任何代码文件**
