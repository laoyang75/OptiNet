# 第三阶段审计合并 Prompt

> 身份：**审计合并协调 Agent（第三阶段）**
> 目标：合并 Codex 和 Claude 两份独立审计报告，生成最终评定和可执行的修改清单
> 输出路径：`rebuild/audit/phase3_audit_final.md`

---

## 1. 任务

合并 2 份第三阶段审计报告，产出：
1. 各维度的合并评定
2. 审计质量评估（标注哪个 agent 审计充分）
3. 必须修改项（如有）
4. 建议改进项
5. 最终结论

---

## 2. 输入文件

### 审计报告（必须完整读取）
1. `rebuild/audit/phase3_audit_codex.md`
2. `rebuild/audit/phase3_audit_claude.md`

### 参考文件
3. `rebuild/audit/phase3_final.md` — 原始计划，用于裁判分歧
4. `rebuild/prompts/context.md` — 项目背景

### 关键实现文件（用于验证分歧时交叉核实）

9. `rebuild/frontend/js/pages/fields.js` — P3 页面

## 3. 合并逻辑

### 评定规则

| 两份报告结果 | 合并判定 |
|-------------|---------|
| 2 通过 | **确认通过** |
| 1 通过 + 1 部分通过 | **基本通过**，列出部分通过方的具体问题 |
| 2 部分通过 | **需关注**，合并两方问题清单 |
| 1+ 未通过 | **必须修改**，以未通过方的问题为主 |

### 质量校验

- 如果某个 agent 的报告明显敷衍（维度评定只写"通过"无证据，而另一方列出具体问题），标注为"**[Agent名] 审计不充分**"
- 如果两个 agent 在某维度的判定完全相反，你必须：
  1. 读取 `phase3_final.md` 中该维度的原始要求
  2. 读取对应的实现文件
  3. 独立裁定谁的判断更准确
  4. 在报告中标注裁定理由

### 分歧处理优先级

当两个 agent 意见不同时：
1. **以代码事实为准**：如果一方说"已实现"另一方说"未实现"，读代码确认
2. **以 `phase3_final.md` 标准为准**：如果一方说"可接受"另一方说"不符合"，看计划原文怎么写
3. **安全性问题从严**：如果一方发现注入风险另一方没提，安全问题必须标记

## 5. 合并纪律

- 不要简单取并集或交集，要做有判断力的合并
- 如果两方都说"通过"但理由不同，保留更有说服力的理由
- 如果两方都发现了问题但描述不同，合并为一条并引用双方
- **不要遗漏任何一方提出的安全性问题**
- 最终结论必须明确，不允许模棱两可
- 如果需要修改，修改项必须具体到文件和函数级别
