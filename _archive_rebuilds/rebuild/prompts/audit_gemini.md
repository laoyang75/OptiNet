## 1. 项目上下文

你正在审计一个**网优数据治理系统重构项目**的第一阶段文档。

### 项目目标
在本地 PostgreSQL 17 上完整重构现有网优数据治理系统，建设"治理链路调试与验证工作台"。核心链路：LAC→Cell→BS→GPS→信号→回归→画像。

### 技术栈
- 数据库：PostgreSQL 17（本地，192.168.200.217:5433/ip_loc2）
- 后端：FastAPI + Worker
- 前端：Next.js 或原生 HTML
- 调度：cron / APScheduler

### 治理链路（8步）
1. 数据起点 → 2. 可信LAC → 3. Cell统计 → 4. 可信BS → 5. GPS修正 → 6. 信号补齐 → 7. 完整回归 → 8. 画像/基线

### 核心业务原则（4条）
1. 有效 cell_id = 有效上报（不因GPS漂移丢弃记录）
2. 修正优于丢弃（优先修正、补齐、标记）
3. 层层收敛、互相印证（正向LAC→Cell→BS→GPS，反向BS→Cell，完整回归）
4. 基线驱动（冷启动建基线，后续撞基线）

### 版本体系（5+1个标识）
run_id、parameter_set_id、rule_set_version、sql_bundle_version、contract_version、baseline_version

### 4种运行模式
1. 全链路重跑  2. 局部重跑  3. 样本重跑  4. 伪日更运行

### 关键数据量
- Layer0_Gps_base: 1.3亿行 53GB
- Layer0_Lac: 1.2亿行 48GB
- Layer4_Final_Cell_Library: 3050万行 27GB

### 数据库规划
4个schema: legacy（旧表只读）、pipeline（治理产物）、workbench（工作台元数据）、meta（字段治理）

## 3. 审计维度

你必须对以下 5 个维度逐一评估：

### 维度 A：完整性
- 5 个缺口（数据起点决策、新旧表映射、步骤SQL参数映射、指标注册表、工作台DDL）是否每个都有对应文档？
- 每个文档的内容是否到位（不是仅有标题/占位符）？
- 决策问题（Q01~Q04）是否全部有明确答案？

### 维度 B：一致性
- 文档之间的**表名**是否一致？（如 Doc02 定义的新表名是否在 Doc03、Doc04、Doc05 中统一使用）
- **字段名**是否一致？（同一字段在不同文档中是否用同一名称）
- **步骤编号/名称**是否一致？（Doc03的步骤与Doc04的指标、Doc05的step_registry是否对齐）
- **参数名和值**是否一致？（Doc03的参数是否与Doc05的parameter_set初始数据一致）

### 维度 C：可执行性
- 仅凭这些文档，能否直接开始编写：
  - pipeline schema 的 DDL（CREATE TABLE）？
  - workbench/meta schema 的 DDL？
  - FastAPI 后端的步骤执行 API？
  - 前端步骤工作台页面？
- 是否有任何地方需要"猜测"或"假设"才能开始编码？

### 维度 D：业务正确性
- 文档内容是否符合 4 条核心业务原则？
- 步骤顺序和依赖关系是否合理？
- 参数值是否在合理范围内（如距离阈值、百分位阈值）？
- 指标的计算逻辑是否正确（分子分母关系、聚合层级）？

### 维度 E：遗漏检测
- 是否有明显遗漏的表/字段/步骤/指标？
- 是否有需要但未定义的索引策略？
- 4种运行模式在文档中是否都有体现？
- 伪日更模式的设计是否有所涉及？
- 错误处理、重试机制是否有定义？
