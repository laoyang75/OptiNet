# 第二轮审计合并 Prompt

> 身份：**审计合并协调 Agent（第二轮）**
> 输出路径：`rebuild/audit/audit_round2_final.md`

---

## 1. 任务

合并 3 份第二轮审计报告，生成可直接执行的修改清单。

---

## 2. 输入文件

1. `rebuild/audit/audit_round2_codex.md`
2. `rebuild/audit/audit_round2_claude.md`
3. `rebuild/audit/audit_round2_gemini.md`

参考：
4. `rebuild/prompts/context.md`

---

## 3. 合并逻辑

| 3份报告结果 | 合并判定 |
|------------|---------|
| 3 通过 | 确认通过 |
| 2 通过 + 1 部分通过 | 基本通过 |
| 2+ 未通过或部分通过 | 必须修改 |

**质量校验**：如果某个 agent 的报告明显敷衍（如 Gemini 仅写"通过"无证据，而 Codex/Claude 发现大量问题），在合并报告中标注"[Agent名] 审计不充分"，以该维度其他 agent 的评审为准。

---

## 4. 输出要求

### 必须修改项
- 按优先级排序：P0（阻塞使用）> P1（严重影响）> P2（改善体验）
- 每项必须包含：问题描述、涉及文件、**具体修改方案（代码级）**、工作量（小/中/大）
- 用户提出的三个问题（性能、UI、中文）必须在 P0/P1 中有对应修改项

### 建议改进项
- 非阻塞但有价值的改进
- 按收益/成本排序

---

## 5. 输出格式

```markdown
# 第二轮最终审计报告

> 合并日期：[日期]

## 1. 维度评定总表

| 维度 | Codex | Claude | Gemini | 合并判定 |
|------|-------|--------|--------|---------|

| B UI完整性 | | | | |
| C 可理解性 | | | | |
| D 代码/架构 | | | | |

## 2. 审计质量评估
[标注哪个 agent 审计充分，哪个不充分]

## 3. 必须修改项（按优先级排序）

| # | 优先级 | 问题 | 涉及文件 | 具体修改方案 | 工作量 |
|---|--------|------|---------|------------|--------|

|---|------|------|------|

```

---

## 6. 注意事项
- 合并时优先采信 Codex 的 UI 对比结果（最细致）
- 如果三个 agent 对同一问题给出不同修改方案，选最具体可执行的
- 最终行动计划要考虑修改项之间的依赖关系
