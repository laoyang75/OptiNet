# 重构项目上下文

> 最后更新：2026-03-23
> 任何新会话或新 agent 读此文件即可接上上下文

## 1. 项目目标

在本地 PostgreSQL 17 上完整重构现有网优数据治理系统，建设"治理链路调试与验证工作台"。

**一句话**：把已经跑通的治理链路（LAC→Cell→BS→GPS→信号→回归→画像），整理成一个便于人理解、便于 AI 协作、便于重复验证的本地工作台。

## 2. 技术栈

- 数据库：PostgreSQL 17（本地）
- 后端：FastAPI + 独立 Worker 进程
- 前端：Next.js（或原生 HTML，视开发阶段决定）
- 调度：cron / APScheduler（一期不是主角）

## 3. 治理链路（8个步骤）

| 序号 | 步骤名 | 业务目的 |
|------|--------|---------|
| 1 | 数据起点 | 原始数据接入/临时起点 |
| 2 | 可信LAC | 从全量数据筛出结构可信的LAC集合 |
| 3 | Cell统计 | 在可信LAC范围内聚合出行为稳定的Cell |
| 4 | 可信BS | 基于Cell聚合出BS空间锚点，碰撞检测 |
| 5 | GPS修正 | 用BS中心点修正/回填GPS |
| 6 | 信号补齐 | 二阶段信号字段补齐 |
| 7 | 完整回归 | 用纯化后的逻辑重放全量数据 |
| 8 | 画像/基线 | LAC/BS/Cell三级画像 |

后续追加：伪日更（新数据撞基线）

## 4. 核心业务原则

1. 有效 cell_id = 有效上报（不因GPS漂移丢弃记录）
2. 修正优于丢弃（优先修正、补齐、标记）
3. 层层收敛、互相印证（正向LAC→Cell→BS→GPS，反向BS→Cell，完整回归）
4. 基线驱动（冷启动建基线，后续撞基线）

## 5. 版本体系（5+1个标识）

- run_id：运行实例
- parameter_set_id：参数集
- rule_set_version：规则集版本
- sql_bundle_version：SQL资源版本
- contract_version：字段契约版本
- baseline_version：基线版本（进入基线阶段后）

## 6. 4种运行模式

1. 全链路重跑（冷启动/大改规则后验证）
2. 局部重跑（某步骤往后）
3. 样本重跑（验证特定样本集）
4. 伪日更运行（历史数据模拟增量）

## 7. UI 结构（V2）

4个主页面 + 3个全局抽屉：
- P1: 治理链路总览
- P2: 步骤工作台（核心页，所有步骤共用模板，8个区块）
- P3: 字段治理
- P4: 样本研究
- D1: 版本与运行抽屉
- D2: SQL查看抽屉
- D3: 样本/对象详情抽屉
