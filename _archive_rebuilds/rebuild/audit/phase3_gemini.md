# Gemini 第三阶段规划审计报告

> 审计日期：2026-03-24

## 1. 各维度分析

### F. 字段治理（最高优先级）
**选型推荐：** **选项B（新建 `meta.meta_source_field_compliance` 独立表）**
- **理由**：`meta_field_registry` 已经用于映射物理表的字段（包含源字段和标准字段映射），承载的是物理和加工属性；合规规则属于纯粹的"业务约束第一层"（如 `RSRP` 必须在 `[-140,-44]`）。新建独立表可以将**业务规范**与**物理字段**解耦，结构更清晰。若追求极简，**选项A**（在 `meta_field_registry` 加 `compliance_rules` JSONB 字段）亦可，但考虑到未来可能对不同运营商有不同规则，选项B 或 JSONB 扩展更具弹性。务实角度推荐 **选项A (加 JSONB 列)**，改动成本最低。

**合规率计算方案：** **基于 `raw_records` 采样计算 (TABLESAMPLE)**
- **源表**：必须在 `raw_records` (2.5亿行) 算，因为 `fact_filtered` 已经是合规过滤后的结果。
- **策略/性能**：绝对**不能**在每次 API 请求或常规重跑时全表扫描 2.5 亿行。建议在后端增加预计算逻辑：使用 `SELECT ... FROM pipeline.raw_records TABLESAMPLE SYSTEM(1)` (1% 随机采样，约 250万行) 来计算合规率，响应时间可控制在 1-3 秒内。
- **存储**：结果直接存入 `meta.meta_field_health`（已有此表，含 `null_rate`, `min_value`, `is_anomalous` 等字段），绑定 `run_id`。

**API设计：**
- `GET /api/v1/fields` (增强：返回合规配置与最新 health 快照)
- `GET /api/v1/fields/{field_name}/compliance` (获取解析规则历史与健康度趋势)
- `POST /api/v1/fields/compute-health` (触发采样预计算)

**前端方案：**
- 遵循 V2 设计，在 P3 (字段治理) 中**采用单列表、行内展开**设计。
- 点击行后，动态拉取该字段的详情并渲染出 5 个子面板：【基本信息】、【映射与解析规则】、【健康度趋势】、【影响步骤】、【变更历史】。

### B. 性能
**评估与建议：**
1. **合规率预计算**：2.5亿行表强依赖 PG17 的并行扫描和 `TABLESAMPLE`。应作为异步 Job 运行或通过专门的初始化脚本算好存入 `meta_field_health`，工作台直接读缓存。
2. **快照刷新策略**：当前三层缓存机制（DB快照 -> 本地文件缓存 -> 内存缓存）基本达标，无需大改。重点是将新增的 `samples` 和 `fields` 接口接入缓存体系。

### C. 代码架构

**`workbench.py` (2,045行) 拆分方案：**
- **当前瓶颈**：所有 DB 查询、指标聚合、运行上下文管理堆砌在一个类或模块中，缺乏单一职责。
- **拆分计划** (新建 `services/` 子目录或模块)：
  1. `run_service.py`：负责 `wb_run` 的生命周期和版本上下文提取。
  2. `metric_service.py`：负责所有 `_compute_*` (层级快照、指标计算、异常统计)。
  3. `field_service.py`：分离 `list_fields`, `get_field_detail`, `ensure_field_registry` 等逻辑。
  4. `sample_service.py`：专门管理样本集的生成、查询和明细。
  5. `workbench.py` 退化为 Facade，负责调度上述子服务，将行数控制在 300 行以内。

**`app.js` (1,184行) 拆分方案：**
- **约束**：不引入现代前端框架。
- **模块化方案** (使用 ES Modules 或按文件引入)：
  1. `core/utils.js`: 格式化 (`fmt`, `pct`, `timeAgo`) 和 DOM 工具。
  2. `core/api.js`: `fetch` 封装及内存/本地缓存管理。
  3. `components/`: 抽取通用的渲染函数 (如 `renderMetricTable`, `runSummaryCard`, 抽屉开关)。
  4. `pages/`: 按照路由分为 `dashboard.js`, `fields.js`, `samples.js`。通过主入口 `app.js` 的 hash 路由分发执行具体的 page 初始化逻辑。

### D. 中文化
**方案：**
- **元数据初始化**：目前 `meta_field_registry.description` 全空。应编写一个 `scripts/init_meta_dict.json` 静态文件（包含如 LAC、Cell、RSRP 的中文定义和合规范围）。
- **同步机制**：在后端服务启动或通过专门的命令，将 JSON 字典全量 `upsert` 到 `meta_field_registry`，实现业务含义的自动填充。

### E. 业务逻辑
**检查结果：**
- **指标注册**：`wb_step_metric` 已有 123 条数据，与 Doc04 贴合较好，但可能缺失新规划的合规率相关的全局指标。
- **质量门控**：`wb_gate_result` 当前在数据库和后端逻辑中存在感极低，未见严格拦截机制，需在第三阶段补齐质量门的判断与打标逻辑。

### 决策问题
1. **合规规则持久化**：采用扩展 JSONB 还是新建表？建议用户确认为 **JSONB 扩展** 以追求交付效率。
2. **采样比例**：2.5亿行采用 `1%` 采样是否足够？是否可以接受每次只评估 250 万条数据以换取秒级响应？

## 3. 实施建议
**务实至上：**
作为本地调试工具，应**坚决抵制过度工程化**。对合规率不需要做到绝对精确，利用 PG 的 `TABLESAMPLE` 给出近似的健康度（偏差<0.1%）是最优雅的解法。
前端拆分尽量采用原生的 ES Modules，不用加 Webpack/Vite 等构建工具，保持零依赖，维护好全局的 `index.html` script 引入顺序即可。
