## 1. 项目目标

在本地 PostgreSQL 17 上完整重构现有网优数据治理系统，建设**治理链路调试与验证工作台**。

目标不是做"自动化运营平台"或"正式云端数仓"，而是：
- 把已验证的治理链路装进更清晰的结构
- 每一步的规则、SQL、参数、数据变化可见
- 调整规则后能快速重跑并比较差异
- 通过样本验证判断逻辑是否正确
- 为后续上线和云端迁移留下清晰边界

## 3. 核心业务原则

### 原则 1：有效 cell_id = 有效上报

cell_id 是终端与基站的真实连接记录。如果 cell_id 有效（格式正确、运营商合规、在可信 LAC 范围内），该上报就是有效的——即使 GPS 漂移。

推论：
- 不因 GPS 漂移丢弃整条记录
- GPS 的价值低于 cell 连接事实本身
- 信号强度（RSRP/RSRQ/SINR 等）与 GPS 准不准无关，必须保留

### 原则 2：修正优于丢弃

字段有问题时优先修正（用更可靠的数据源覆盖），不丢弃整条记录。无法修正时标记为异常（仍保留）。

体现：
- GPS 漂移 → 用 BS 中心点修正
- 信号缺失 → 用时间最近的同 Cell/同 BS 数据补齐
- 严重碰撞桶 → 仍回填 GPS 但强标注

### 原则 3：层层收敛、互相印证

不是单向流水线，而是多层交叉验证的闭环：
- 正向：LAC → Cell → BS → GPS（从大到小逐级确认）
- 反向：BS → Cell（用 BS 画像反向纯化 Cell）
- 回归：用纯化后的结果重新处理全量数据

### 原则 4：基线驱动的日常运营

冷启动完成后，日常不再重跑全部。新数据与基线对比 → 好数据沉淀 → 变化发现 → 定期刷新基线。

## 4. 治理链路（8个步骤）

### Step 1：数据起点（原始接入/临时起点）

- 14天或更长的网优明细数据
- 包含：seq_id, cell_ts, tech, 运营商id, cell_id, lac, bs_id, GPS(lon/lat), 信号(rsrp/rsrq/sinr/rssi)

### Step 2：可信 LAC

从全量数据筛出结构可信的 LAC 集合。

怎么做：
1. 运营商过滤：只保留五大核心 PLMN（移动/联通/电信/广电/铁路）
2. LAC 编码过滤：剔除 NULL、异常长度、黑名单
3. 位置过滤：只保留北京区域 GPS
4. 稳定性过滤：活跃天数 ≥ 阈值 且 上报量 ≥ 阈值

产出：可信 LAC 维表（主键：operator + tech + lac）

### Step 3：Cell 统计与可信筛选

在可信 LAC 范围内，按 (运营商, 制式, cell_id, lac) 聚合，筛出行为稳定的 Cell。

怎么做：
1. 只在可信 LAC 明细中聚合
2. 统计每 Cell 的活跃天数、GPS 空间分布、上报量
3. 剔除：位置离散、偶尔出现、样本量过小
4. 标记：一个 Cell 出现在多个 LAC 下 → 映射异常

产出：Cell 统计维表（主键：operator + tech + cell_id + lac）

### Step 4：可信 BS

基于可信 Cell 聚合出 BS 空间锚点。

怎么做：
1. 按 (制式, bs_id, lac) 聚合所有可信 Cell
2. bs_id 推算规则：4G = cell_id ÷ 256, 5G = cell_id ÷ 4096
3. 计算 BS 中心点（GPS 中位数，不是平均值）
4. 计算覆盖半径（P50/P90）
5. 碰撞检测：Cell GPS 聚落分成不连通簇 → 编码碰撞
6. 标注风险等级

产出：可信 BS 维表（含中心点、覆盖半径、碰撞标记）

### Step 5：GPS 修正

用 BS 中心点作为锚点，校验和修正每条记录的 GPS。

怎么做：
1. 每条记录通过 (制式, bs_id, lac) 关联 BS 主库
2. 计算原始 GPS 到 BS 中心距离
3. 距离 ≤ 阈值（统一 1500m）→ 保留原 GPS（注：完整回归 Step40 使用更严格的分制式阈值 4G:1000m, 5G:500m）
4. 距离 > 阈值 → 钉成 BS 中心点
5. GPS 缺失 → 回填 BS 中心点
6. 所有情况都保留整条记录
7. 碰撞 BS 下的记录仍回填，但强标注 is_severe_collision

### Step 6：信号补齐

二阶段信号字段补齐（Layer3 摸底版 + Layer4 精确版）。

怎么做：
- **Layer3 摸底版（Step33）**：同 Cell 中位数 → 同 BS 中位数（快速评估信号完整度）
- **Layer4 精确版（Step41，完整回归中）**：同 cell_id 下时间最近的有信号记录作为 donor → 同 BS Top Cell 的时间最近记录
- 记录补齐来源和 donor 标识

补齐字段：sig_rsrp, sig_rsrq, sig_sinr, sig_rssi, sig_dbm, sig_asu_level, sig_level, sig_ss

### Step 7：完整回归

用纯化后的 BS/Cell 结果，把全量原始数据从头再处理一遍。

为什么：确保最终产出基于同一套规则统一处理，不存在"部分旧逻辑部分新逻辑"。

### Step 8：画像/基线

从完整回归后的最终明细中，按 LAC/BS/Cell 三级聚合生成画像。

| 级别 | 主键 | 核心指标 |
|------|------|---------|
| LAC | (运营商,制式,lac) | 上报量、活跃天数、BS数、Cell数、GPS覆盖率 |
| BS | (运营商,制式,bs_id,lac) | GPS中心点、覆盖P50/P90、碰撞标记、移动Cell数 |
| Cell | (运营商,制式,cell_id,lac) | GPS位置、信号中位数、活跃天数、样本量、移动标记 |

后续追加：**伪日更**（新数据撞基线，四分流：范围内/漂移/新增/异常）

## 5. 版本体系

| 标识 | 说明 |
|------|------|
| run_id | 一次运行的唯一标识 |
| parameter_set_id | 本次运行使用的参数集 |
| rule_set_version | 本次运行使用的规则集合版本 |
| sql_bundle_version | 本次运行对应的 SQL 资源版本 |
| contract_version | 本次运行绑定的数据字段契约版本 |
| baseline_version | 基线版本（进入基线阶段后） |

## 6. 4种运行模式

1. **全链路重跑**：大改规则后验证整体效果、冷启动重放
2. **局部重跑**：修改某步规则后从该步骤往后重跑
3. **样本重跑**：验证特定样本集（GPS漂移/碰撞BS等）
4. **伪日更运行**：用历史数据模拟增量，验证四分流

---

## 7. 产品原则

1. 人理解得快，优先于系统看起来高级
2. 规则、SQL、参数、结果变化必须同屏或近距离可见
3. 先服务"调试与验证"，再服务"自动化运营"
4. 以步骤为主线，以对象和问题为辅助钻取
5. 保留现有结果资产，增量建设工作台元数据
