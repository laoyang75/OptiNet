# rebuild3 文档审核结论（最终冻结版）

> 产出日期：2026-04-03  
> 用途：说明两套文档的差异、冲突点与最终收敛决策  
> 适用对象：人类评审、UI 设计协作者、后续实施 agent

## 一、审核结论总览

本次两套文档并不是互相否定，而是**定位不同**：

- 第一套（`01/02/03`）更偏**实现展开版**，优点是结构完整、模块清晰、数值和表族较具体。
- 第二套（`04/05/06`）更偏**预实施/审阅版**，优点是阶段边界明确，强调“先冻结口径、先做 UI、再出最终实施任务书”。

最终冻结版采用的原则是：

1. **节奏上**以第二套为主：先冻结业务与结构，再做 UI，再生成最终实施计划。
2. **内容上**吸收第一套的细节：流程、对象、版本、验证路径、表族与模块拆解继续保留。
3. **冲突点上**做显式收敛，不再让 UI、开发计划和实现阶段分别理解成不同系统。

## 二、已收敛的关键冲突

| 主题 | 第一套文档倾向 | 第二套文档倾向 | 最终冻结决策 |
|---|---|---|---|
| 文档定位 | 可直接展开实现 | 先做 UI 前置冻结 | **以“UI 前置的最终冻结版”定位为准** |
| 生命周期状态 | 包含 `watch` | 不建议把“观察中”混入生命周期 | **持久化 lifecycle 不含 `watch`，UI 可派生展示 `watch`** |
| 事实分层 | `fact_governed / fact_waiting / fact_rejected` | 强调四层事实 | **冻结为 4 层事实：`fact_governed / fact_pending_observation / fact_pending_issue / fact_rejected`** |
| 标准化层 | 已明确 | 已明确 | **保留独立不可变的 `fact_standardized`** |
| 元数据存放 | 主要放在 `rebuild3` | 建议单独 meta 边界 | **冻结为 `rebuild3` + `rebuild3_meta` 双 schema** |
| 对象历史 | 通用状态日志为主 | 要求显式状态/关系历史 | **冻结为“快照表 + 状态历史 + 关系历史”** |
| 数据访问方式 | `asyncpg` + 纯 SQL | 允许沿用既有 `SQLAlchemy asyncio` 依赖 | **坚持 SQL-first，不使用 ORM 领域模型；连接层可沿用现有 `asyncpg` / `SQLAlchemy asyncio` 能力** |
| 批次事务 | 倾向单批次单事务 | 未强约束 | **改为“批次是逻辑事务单位，数据库可按步骤做幂等事务”** |
| 初始化顺序 | `LAC -> Cell -> BS` | 强调增量要 `Cell` 先行 | **冻结为：Cell 是长期治理最小主语；初始化允许 LAC 预边界作为冷启动策略，不改变长期主语** |
| 当前阶段任务 | 已列完整开发 Phase | 强调当前不直接编码 | **冻结为：现在输出的是最终冻结版，不是最终实施任务书** |

## 三、最重要的 8 条最终冻结决策

### 1. rebuild3 的真正主语

最终统一为：

**对象 → 决策 → 事实 → 基线 → 状态流转**

不再把系统理解为旧式 Step / Layer 页面搬家。

### 2. Cell / BS / LAC 的职责

- **Cell**：动态治理最小主语
- **BS**：空间锚点
- **LAC**：区域边界与区域健康主语

初始化允许研究期使用 LAC 预边界，但这只是**冷启动缩边界策略**，不是长期主语顺序。

### 3. 生命周期与健康状态分离

最终冻结为：

- `lifecycle_state = waiting / observing / active / dormant / retired / rejected`
- `health_state = healthy / insufficient / gps_bias / collision_suspect / collision_confirmed / dynamic / migration_suspect`

`watch` 不再作为持久化 lifecycle 状态。  
如需表达“active 但需重点观察”，UI 应派生展示为：

- `lifecycle_state = active`
- `health_state != healthy`
- 可显示 `watch` 徽标或标签

### 4. 三类资格必须显式分开

系统必须同时表达：

1. **存在资格**：对象是否被承认存在
2. **锚点资格**：对象能否参与 GPS 修正 / 信号补齐
3. **基线资格**：对象或事实能否进入 baseline/profile

最终冻结要求：这三类资格不能只靠口头约定，必须能在对象快照或读模型中直接回答。

### 5. 事实层必须是四层，而不是一张总明细

最终冻结为：

- `fact_standardized`：不可变标准事件层
- `fact_governed`：正式治理后事实
- `fact_pending_observation`：未知但合规、处于等待/观察链路的事实
- `fact_pending_issue`：对象级异常或待复核问题事实
- `fact_rejected`：结构不合规或明确拒收

原则：**同一标准事件只能有一个主归宿事实层**。

### 6. 元数据与领域数据分 schema

最终冻结为：

- `rebuild3`：对象、事实、画像、异常、热层/长层/归档层
- `rebuild3_meta`：run、batch、contract、rule、baseline、source_adapter 等控制元数据

`rebuild2` / `rebuild2_meta` / `legacy` 保持只读。

### 7. 当前批次只能看上一版冻结 baseline

最终冻结原则：

1. 当前批次准入与修正只看上一版 baseline
2. 当前批次结束后才允许生成下一版 baseline
3. 新 baseline 只能供下一批次使用

这条原则贯穿准入、修正、补齐、状态转移、画像刷新。

### 8. 当前最终文档的定位

本轮最终文档的定位不是“直接开始最终编码”，而是：

1. 冻结业务口径
2. 冻结数据结构边界
3. 给 UI 设计提供稳定输入
4. 为 UI 确认后的最终实施任务书做准备

## 四、仍然保留为参数/策略而非硬编码的内容

以下内容可以在最终实施计划中继续细化，但当前已经明确其**表达方式**：

| 项目 | 当前冻结方式 |
|---|---|
| `event_time` | 默认以 `ts_std` 作为系统事件时间；保留原始时间证据列，不丢失升级空间 |
| 源适配模式 | 冻结为“双模式接口”：原始 27 列是标准契约入口，rebuild2 标准化结果可作为验证/回放快路径 |
| Cell 三层门槛 | 冻结为“存在 / 锚点 / 基线成熟”三层结构，数值阈值参数化 |
| 热层保留策略 | 冻结为“热层 + 长层 + 归档层”；每 Cell 的热层 N 值参数化，可按流量分层 |
| 异常资格矩阵 | 冻结为显式矩阵，不允许在代码里隐式散落 |

## 五、对后续工作的直接影响

### 对 UI 设计的影响

UI 设计必须围绕以下主语展开，而不是延续 rebuild2 的旧 Step 导航：

- 运行与批次
- 对象注册与状态
- 等待/观察池
- 异常与问题
- 基线与画像
- 回放与对比

### 对后续实施计划的影响

UI 确认后再输出最终实施任务书，届时必须：

1. 以本轮冻结口径为业务来源
2. 以 UI 确认稿为页面和读模型来源
3. 再把数据库、API、前端和验证任务拆成正式 backlog
