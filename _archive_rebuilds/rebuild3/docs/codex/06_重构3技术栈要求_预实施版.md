# rebuild3 技术栈要求（预实施版）

> 状态：预实施版  
> 适用对象：Agent、架构/实现评审、后续 UI 设计协作者  
> 重要说明：本文约束的是“当前阶段与后续首轮实施的技术边界”，不是未来全国化云端系统的最终技术蓝图

## 1. 总原则

rebuild3 的技术方案，当前必须遵守下面 6 条原则：

1. 以现有本地工程底座为主，不另起一套全新技术体系。
2. 以 PostgreSQL 为核心计算与存储底座，保持 SQL-first。
3. 以后端编排和读模型为核心，不把大规模统计逻辑搬到前端或 Python 内存循环里。
4. 前端重构明确选择 `Vue 3 + TypeScript`，重点复用业务逻辑、读模型和接口，不复用旧前端结构本身。
5. 旧 `legacy` / `rebuild2` / `rebuild2_meta` 资产保留，用作初始化输入、对照输入和回归输入。
6. 当前先满足本地验证、UI 设计和后续首轮实施，不提前做云端平台化和复杂基础设施。

## 2. 当前必须沿用的基础技术栈

结合现有仓库和已落地代码，当前应明确要求如下：

| 层 | 当前要求 | 说明 |
| --- | --- | --- |
| 数据库 | PostgreSQL 17 | 继续作为主计算、主存储、主回放环境 |
| 后端 | Python + FastAPI | 继续沿用当前 `rebuild2/backend` 的服务组织方式 |
| 数据访问 | `asyncpg` / `SQLAlchemy asyncio` | 与现有依赖保持一致，不另起新栈 |
| 数据模型 | Pydantic 2 | 保持 API 出入参和配置模型统一 |
| 前端 | Vue 3 + TypeScript + Vite | 前端重构的默认选型，用组件化解决旧工作台页面膨胀问题 |
| 路由 | Vue Router | 用对象视图、批次视图、异常视图替代旧 Step 导航式堆叠 |
| 状态管理 | Pinia | 统一承接对象筛选、批次上下文、抽屉状态和页面共享状态 |
| 静态托管 | Vite 构建产物 + FastAPI `StaticFiles` | 开发期使用 Vite，交付仍可由 FastAPI 托管静态产物 |
| 配置 | `.env` + DB 元数据表 | 敏感配置在 `.env`，规则与版本在表内显式存储 |
| 文档 | Markdown + Mermaid | 当前阶段文档必须可直接审阅和版本管理 |

## 3. 数据层技术要求

### 3.1 schema 策略

首轮应遵循“增量新增，不覆盖旧库”的原则。

要求：

1. 不直接覆盖 `legacy.*`。
2. 不直接重写 `rebuild2.*` 既有研究结果表。
3. rebuild3 应新增独立 schema 或等价表族边界。
4. 元数据和运行控制信息应与事实层、对象层分开存放。

推荐方向：

- `rebuild3`：对象、事实、画像、决策结果
- `rebuild3_meta`：source、run、batch、contract、rule、baseline 绑定和控制元数据

### 3.2 标准事件层要求

`ods_std_event` 或等价层必须满足：

1. 标准化事件不可变。
2. 必须保留原始来源信息和解析来源信息。
3. 关键治理字段不能只放 JSON，必须落成可索引的显式列。
4. 必须有标准事件级幂等键。

至少需要保留：

- `source_name`
- `source_record_id`
- `parsed_from`
- `source_group_seq`
- `source_cell_seq`
- `event_time`
- `operator`
- `tech`
- `lac`
- `cell_id`
- `bs_id`

### 3.3 对象层要求

对象层必须是 rebuild3 的核心，不允许继续只靠聚合维表临时代替。

最少要有：

1. `obj_cell_registry`
2. `obj_bs_registry`
3. `obj_lac_registry`
4. `*_state_history`
5. `*_relation_history`

对象表至少要显式支持：

- 业务主键
- `lifecycle_state`
- `health_state`
- 锚点资格
- baseline 资格
- 当前有效关系
- 版本归属
- 最近活跃时间

### 3.4 事实层要求

事实层必须至少分成 4 类，不允许再只用一张“总明细”表达所有语义：

1. `fact_governed`
2. `fact_pending_observation`
3. `fact_pending_issue`
4. `fact_rejected`

要求：

1. 事实分层必须可追溯到决策结果。
2. `fact_governed` 中必须显式记录 `baseline_eligible`。
3. 异常标签、修正来源、补齐来源必须显式记录。
4. 同一标准事件只能落到明确的一层事实归宿。

### 3.5 热层 / 长层 / 归档层要求

首轮技术方案必须允许 3 层数据保留：

| 层 | 要求 |
| --- | --- |
| 热层 | 服务近期画像、等待池、观察池、异常复核，按窗口滚动保留 |
| 长层 | 服务活跃节奏、成熟度、退役判断，按日或更粗粒度保留 |
| 归档层 | 服务审计、回放、完整回归，保留完整治理后事实历史 |

结论：

- per-Cell 热明细可以有
- 但不能只有 per-Cell 热明细

## 4. 版本、批次与幂等要求

这是首轮实施前必须先冻结的技术底座。

### 4.1 最小版本体系

所有对象、事实、决策结果、画像，都必须能追溯到下面 4 个版本标识：

1. `run_id`
2. `contract_version`
3. `rule_set_version`
4. `baseline_version`

### 4.2 batch 要求

要求：

1. `run_id` 与 `batch_id` 分离。
2. 批次必须显式记录窗口范围和来源。
3. 2 小时批次不能只靠文件名或表名隐式推断。
4. 同一批次必须能被重复回放和结果比对。

### 4.3 event_time 要求

在最终拍板前，技术实现必须先支持“双时间口径”：

1. 一个系统事件时间，用于当前增量治理。
2. 一组原始时间证据，用于追溯和后续升级。

也就是说：

- 当前可以先用 `ts_std` 作为系统事件时间
- 但不能丢掉 `cell_infos` / `ss1` 内的原始时间证据

### 4.4 幂等要求

标准事件级幂等键必须显式落库，且不得只依赖原始 `record_id`。

原因：

- 一条原始记录会拆成多条标准事件
- `cell_infos` 和 `ss1` 有展开序号
- 同一 run、同一 batch 必须可重放、可去重、可比对

## 5. 后端实现要求

### 5.1 服务组织要求

后端继续沿用 FastAPI 路由组织，但职责要从“Step 查询接口”升级为“对象/决策/基线读模型接口”。

首轮至少要预留下面 6 类接口能力：

1. 源与批次读模型
2. 对象注册读模型
3. 等待/观察池读模型
4. 事实分流读模型
5. 异常与问题读模型
6. baseline / profile 读模型

### 5.2 执行方式要求

长耗时任务不应强绑定在单个同步 HTTP 请求里。

要求：

1. 批次执行以 run / batch 注册为主。
2. UI 主要读取状态和结果，不直接承受全量大查询。
3. 复杂统计继续采用 SQL 预聚合、缓存表、读模型返回。

### 5.3 计算职责要求

重计算应优先放在 PostgreSQL 中完成，Python 负责：

- 编排
- 校验
- 数据搬运
- API 输出
- 状态汇总

不建议在首轮实现里用 Python 在内存中处理大规模明细聚合。

## 6. 前端与 UI 设计阶段要求

### 6.1 当前默认前端要求

当前默认要求仍然是：

- Vue 3
- TypeScript
- Vite
- Vue Router
- Pinia
- 构建产物继续可由 FastAPI 静态托管

原因很直接：

1. 当前旧工作台已经出现页面文件过大、结构难持续收敛的问题。
2. rebuild3 是重构，不是保留旧前端结构继续堆页面。
3. 这次前端应复用业务逻辑、读模型和接口，而不是复用旧 DOM 拼装方式。
4. `Vue 3 + TypeScript` 更适合把对象页、批次页、等待池、异常抽屉等复杂工作台界面收敛成稳定组件。

### 6.2 UI 设计阶段允许做什么

UI 设计阶段可以做：

1. 静态高保真页面稿。
2. 基于对象主语重组页面结构，不受旧导航结构约束。
3. 新增对象、批次、等待池、异常视图原型。
4. 读模型驱动的页面结构设计。
5. 同步定义 Vue 组件边界、页面路由边界和共享状态边界。

### 6.3 UI 设计阶段默认不引入什么

在已确定 `Vue 3 + TypeScript` 的前提下，默认不引入：

1. React / Next 等另一套前端主栈。
2. Nuxt、SSR、微前端等超出当前工作台需求的前端架构。
3. 全量重量级组件库迁移。
4. 为了追求“更现代”而引入不必要的前端基础设施。

前端栈本身不再作为待定项，当前待定的是组件边界、页面信息架构和读模型组织方式。

## 7. API 与页面的协作要求

首轮必须坚持“页面读模型化”，不允许页面直接围绕底层大表做临时拼装。

要求：

1. 每个页面先定义读模型，再决定 API。
2. 页面展示对象、决策、事实、基线，不继续围绕旧 Layer 命名组织。
3. 统计类页面优先消费缓存或汇总结果，不直接扫千万级明细。
4. 页面必须能表达空状态、等待态、观察态、异常态、锚点禁用态。

## 8. 校验与验证要求

技术实现必须为下面这条验证路径服务：

1. 第 1 到第 3 天初始化起跑。
2. 第 4 到第 7 天按 2 小时批次增量滚动。
3. 与直接 7 天初始化结果做对比。

因此技术方案至少要支持：

1. 历史回放模式。
2. 批次重放。
3. 对比指标输出。
4. 同版本条件下的重复运行比对。

至少要能输出以下指标类别：

- 对象收敛指标
- 空间收敛指标
- 信号收敛指标
- 异常收敛指标
- 决策收敛指标
- 热层稳定性指标

## 9. 当前明确不要求引入的技术

为了防止技术栈失焦，当前明确不要求、也不建议主动引入下面这些内容：

1. Kafka、Flink、Spark 等流式或大数据平台。
2. Redis、Elasticsearch、ClickHouse 等额外基础设施。
3. 微服务拆分。
4. 云端调度系统。
5. 大一统主数据平台。
6. Nuxt、SSR、微前端、跨端框架等超出当前目标的前端体系。

这些内容后续如果确有必要，可以在首轮本地验证跑通后单独评估。

## 10. 必须预留的未来兼容边界

虽然当前不做云端化，但技术上必须预留下面这些边界：

1. 源适配接口边界。
2. 批次回放接口边界。
3. baseline 全量刷新与局部刷新边界。
4. 局部重算与全量回归边界。
5. UI 读模型边界。
6. 规则实验与异常样本边界。

这意味着当前实现不能把主逻辑写死在：

- 某一张具体原始表名
- 某一个具体城市边界
- 某一种固定批次粒度
- 某一版特定 UI 页面结构

## 11. 结论

当前 rebuild3 的技术栈要求，核心不是“换技术”，而是“把已有技术底座组织成可持续运行的治理系统”。

首轮真正应被强约束的，是下面这些点：

1. PostgreSQL 17 继续做主计算和主存储。
2. Python + FastAPI 继续做编排和读模型服务。
3. 前端明确选择 `Vue 3 + TypeScript` 重构 UI，复用逻辑和接口，不复用旧前端结构。
4. 对象层、事实层、基线层、版本层必须显式分开。
5. 批次、版本、幂等必须先于实现冻结。
6. UI 设计先行，最终实施任务书后置。
