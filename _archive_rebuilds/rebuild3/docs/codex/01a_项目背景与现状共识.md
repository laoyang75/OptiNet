## 一、项目背景与业务目标

### 1.1 项目性质

这是一个**移动网络优化数据仓库项目**，核心目标是从运营商原始报文数据中，持续构建并维护可信的三级基础对象库与画像体系：

| 对象层级 | 说明 | 主键 |
|----------|------|------|
| **LAC**（位置区） | 最大粒度的网络分区，一个城市区域的逻辑分区 | (运营商ID, 制式, LAC编码) |
| **BS**（基站） | 由 cell_id 按规则推算：4G = cell_id ÷ 256，5G = cell_id ÷ 4096 | (运营商ID, 制式, BS_ID, LAC编码) |
| **Cell**（小区） | 最小粒度，一个基站通常有多个扇区/频段 | (运营商ID, 制式, Cell_ID, LAC编码) |

层级关系：运营商 → 制式(4G/5G) → LAC → BS → Cell

**实体关系解读**：
- 原始数据中直接包含 `cell_id`（终端与基站的真实连接标识），而 `bs_id` 需要通过 cell_id 推算
- 一个 LAC 下有多个 BS，一个 BS 下有多个 Cell（扇区）
- **Cell 是原子观测单位**——每条原始记录自带 cell_id，Cell 是数据的天然累计单元
- 聚合方向：Cell→BS→LAC（Cell 质心聚合出 BS 中心，BS 聚合出 LAC 画像）
- 前一阶段（rebuild2）的研究顺序是 LAC→Cell→BS（先画可信边界再逐级下钻），这是研究策略的选择而非唯一路径

### 1.2 业务哲学（已确认，rebuild3 必须继承）

| 原则 | 内容 | 含义 |
|------|------|------|
| 原则1 | **有效 cell_id = 有效上报** | cell_id 是终端与基站的真实连接记录，不因 GPS 漂移丢弃记录。GPS 可以修正，信号数据必须保留 |
| 原则2 | **修正优于丢弃** | 当一条记录的某个字段有问题时，优先用更可靠的数据源修正该字段，而不是丢弃整条记录。只有无法修正时才标记异常（仍保留，由下游按标签决定是否使用） |
| 原则3 | **层层收敛、互相印证** | 数据处理不是单向流水线。正向：LAC→Cell→BS→GPS（从大到小逐级确认）；反向：BS→Cell（用聚合后的 BS 画像反过来纯化 Cell）；完整回归：用纯化后的结果重新处理全量数据确保一致性 |
| 原则4 | **基线驱动日常运营** | 一次性冷启动完成后，日常不再重跑全部流水线。改为：新数据与基线对比 → 好数据沉淀 → 变化发现 → 定期刷新基线 |
| 原则5 | **先建可信对象，再修正事实** | 必须先通过多层收敛确定哪些 LAC/BS/Cell 是可信的，才能用它们的画像去修正具体记录的 GPS 和信号 |
| 原则6 | **正向收敛 + 反向纯化 + 必要时完整回归** | 三段式闭环：正向聚合确认对象 → 反向用对象审视细节 → 用最终结论重新处理全量数据 |

### 1.3 当前数据源

| 项 | 值 |
|----|-----|
| 原始表 | 两张原始源表（LAC 表 1796 万行 + GPS 表 2217 万行），北京地区一周数据 |
| 原始字段 | 27 列，两表结构一致（仅最后一列列名不同） |
| 核心嵌套字段 | **cell_infos**（JSON 格式，手机实时基站扫描结果，包含 cell_id/lac/运营商/信号强度/制式等）、**ss1**（紧凑分号分隔字符串，后台采集的信号+时间+GPS+基站+AP 组合信息） |
| 展开逻辑 | 一条原始记录中 cell_infos 可包含多个基站扫描结果，ss1 可包含多组采集数据。解析后每个结果/每组数据各产出一行标准化记录 |
| 展开比 | 约 2~2.4 倍（原始 4013 万行 → 标准化后约 8220 万行） |

**cell_infos 解析要点**：JSON 对象，每个 key 是一个基站扫描结果，包含 cell_identity（Ci/Nci→cell_id, Tac/Lac→lac, mno→运营商）、signal_strength（rsrp/rsrq/sinr/dbm 等）、type（lte→4G, nr→5G）、isConnected（1=当前连接）。

**ss1 解析要点**：以分号分组，每组以 `&` 分 5 个元素——①信号块（制式+信号强度，按 `+` 分多条）、②时间戳（unix 秒）、③GPS（经度,纬度,时间）、④基站块（制式,cell_id,lac,plmn，按 `+` 分多条）、⑤AP 信息。信号和基站按制式类型匹配（l信号→l基站，n信号→n基站），产出配对行。当基站块值为 `1` 时表示继承上一组的基站信息。

## 二、当前项目结构与开发现状

### 2.1 项目演进脉络

项目经历了三个阶段：

1. **lac_enbid_project（legacy 阶段）**：最初版本，在 PostgreSQL 上实现了 Layer_0 到 Layer_5 共 20+ 步的处理流水线。产出了基本可用的结果但层次过多、边界模糊、正向/反向逻辑交织、冷启动与日常未分离。此阶段的 28 张表保留在数据库 `legacy` schema 中，仅供参考。

2. **rebuild2（当前活跃阶段）**：核心理念是"先看数据，再建结构；每建一层，可视化确认一层"。从原始 27 列数据出发重新审视字段、重新解析、重新构建可信库和画像。已完成字段治理、可信库构建、GPS 修正与信号补齐、BS 异常分类研究，画像基线正在进行中。

3. **rebuild3（即将启动）**：不是推翻 rebuild2，而是把 rebuild2 研究出的静态规则嵌入到一个"初始化 + 2 小时增量"的动态治理系统中。

### 2.2 技术栈

| 组件 | 技术 | 说明 |
|------|------|------|
| 数据库 | PostgreSQL 17 | 本地服务器 |
| 后端 | Python FastAPI + asyncpg | REST API 服务 |
| 前端 | 原生 HTML + CSS + ES Modules | 无构建工具，轻量级 |

### 2.3 当前 UI / API / Agent 协作形态

当前系统采用 **AI Agent 驱动的阶段化开发模式**：

- 每个阶段有一份明确的 prompt 文档，描述前置条件、业务规则、执行计划和产出物
- AI Agent 读取 prompt 后设计规则、生成 SQL/代码
- 人类通过 UI 页面审核规则和结果，确认后 Agent 执行下一步
- 大表操作（千万级行）通过服务器 psql 直接执行，统计结果预算到缓存表供 UI 读取
- 每步产出都有 UI 展示：字段审计页、漏斗统计页、画像展示页等

## 三、rebuild2 已完成的工作详情

> 以下按阶段详细说明 rebuild2 的每项工作内容、方法和产出，以便无代码访问权限的读者完整理解。

### 3.1 字段治理与标准化（已完成）

**做了什么**：从两张原始源表（各 27 列）出发，完成了完整的字段审计和标准化。

**具体工作**：
1. **字段决策**：27 列原始字段逐一审计，决定保留（16 列）、丢弃（10 列，如 ip/imei 等隐私或无关字段）、或解析（cell_infos + ss1 两个嵌套字段）
2. **解析规则制定**：
   - cell_infos JSON 解析：提取 22 个字段（cell_id、lac、运营商、制式、PCI、频点、带宽、全部信号字段）
   - ss1 紧凑字符串解析：5 元素拆解、信号按制式匹配基站、继承逻辑处理
3. **派生字段计算**：bs_id（4G: cell_id/256, 5G: cell_id/4096）、sector_id、运营商中文标签、制式标准化
4. **26 条 ODS 清洗规则**：
   - 删除行：主服务基站无 CellID 的行
   - 置空（19 条）：CellID 为 0/占位值、LAC 为 0/溢出值、运营商非白名单、RSRP 正值/零值/超低值、RSRQ/SINR/Dbm 超范围、ASU 溢出等
   - 转换（4 条）：时间戳标准化
   - 运营商（2 条）：无效编码置空、非五大运营商置空
5. **合规规则制定**（17 条）：每个关键字段的有效性定义（运营商白名单、LAC/CellID 合法范围、GPS 边界框、信号有效范围等）

**产出**：
- 标准化后的明细表：GPS 表 3843 万行 + LAC 表 4377 万行（共约 8220 万行，约 60 个字段）
- 元数据表：字段决策（27 行）、目标字段定义（~60 行）、解析规则（64 行）、合规规则（17 行）、清洗规则（24 行）

### 3.2 可信库构建（已完成）

**做了什么**：从标准化数据中，按 LAC→Cell→BS 的顺序逐级筛选，构建可信对象库。

**具体工作**：

**Step 1 — 可信 LAC 筛选**：
- 从 GPS 表（3843 万行，GPS 数据充足）出发
- 按 (运营商编码, 标准制式, LAC) 聚合
- 层层过滤：运营商白名单（五大运营商）→ LAC 编码合法（非 NULL、非黑名单）→ GPS 区域（北京范围）→ 稳定性（活跃天数 ≥ 阈值、上报量 ≥ 阈值）
- 产出：1,057 个可信 LAC

**Step 2 — 可信 Cell 筛选**：
- 只在可信 LAC 范围内的明细中，按 (运营商, 制式, CellID, LAC) 聚合
- 统计每个 Cell 的活跃天数、GPS 空间分布、上报量
- 剔除：位置离散、偶尔出现、样本量太小的 Cell
- 标记异常：一个 Cell 出现在多个 LAC → 映射异常

**Step 3 — 可信 BS 构建**：
- 按 (制式, BS_ID, LAC) 聚合所有可信 Cell
- 计算 BS 中心点（GPS **中位数**而非平均值——中位数抗离群点更强）
- 计算覆盖半径（GPS 点到中心距离的 P50/P90）
- 碰撞初步检测：同 bs_id 下的 Cell 的 GPS 聚落分成不连通的几簇 → 可能是编码碰撞
- 产出：193,036 个 BS（含中心点、覆盖半径、初步风险标记）

**Step 4 — 全局表过滤回写**：
- 用可信库过滤 LAC 表（4377 万行），提取属于可信 LAC 范围内的记录
- 产出：合规过滤后的全局明细（约 3008 万行）

### 3.3 GPS 修正 + 信号补齐 + BS 异常分类研究（已完成）

**做了什么**：三部分工作——(A) 用 BS 中心点修正每条记录的 GPS 并补齐信号；(B) 对空间散布异常的 BS 进行深入分类研究。

#### A. BS 精算 + GPS 修正 + 信号补齐

1. **BS 精算**：对 193,036 个 BS 重新精算中心点和覆盖范围（使用全量数据，剔除极端离群点后重新计算中位数）
2. **Cell 校验**：573,561 个 Cell 逐一验证其与所属 BS 的空间关系
3. **GPS 修正**：
   - 每条记录通过 (制式, bs_id, lac) 关联到 BS
   - 计算原始 GPS 到 BS 中心点的距离
   - 距离 ≤ 阈值（4G: 1000m, 5G: 500m）→ 保留原 GPS
   - 距离 > 阈值 → GPS 漂移，**钉成 BS 中心点**
   - 原始 GPS 缺失 → **回填 BS 中心点**
   - **所有情况都保留整条记录，信号字段原封不动**
4. **信号补齐**（两阶段策略）：
   - 第一阶段：在同一个 cell_id 下，找时间最近的有信号记录，逐字段补齐（RSRP/RSRQ/SINR/Dbm 等）
   - 第二阶段：如果同 Cell 没有 donor，退化到同 BS 下数据量最大的 Cell，找其时间最近的有信号记录补齐
   - 记录补齐来源和 donor 标识

产出：GPS+信号补齐后的全局明细表（3008 万行），每条记录附带 GPS 来源标记（original / cell_center / bs_center）和信号来源标记（original / cell_fill / unfilled）。

#### B. BS 异常分类研究（Cell 质心法 v2）

**背景**：在 BS 精算过程中发现部分 BS 的 GPS 空间散布异常（P90 偏移 > 1500m），需要分析原因。

**分析漏斗**：
1. **数据充分性过滤**：193,036 个 BS 中，39,096 个数据不足（GPS 点 ≤ 3 或仅单设备），144,348 个空间正常，9,591 个进入异常候选分析
2. **异常候选条件**：GPS 点 ≥ 10、设备 ≥ 2、且满足 P90 偏移 > 1500m 或最大偏移 > 5000m

**Cell 质心法分类算法（5 步）**：

- **Step 1**：提取 9,591 个候选 BS 的有效 GPS 记录（含 CellID），共 203 万条
- **Step 2**：按 500m 网格（0.005°）对每个 Cell 独立聚合 GPS 点
- **Step 3**：对每个有 ≥ 3 个 GPS 点的 Cell 计算**加权质心**（按网格点数加权平均经纬度）和**空间跨度**（网格 min/max 经纬度换算距离）
  - 跨度 > 1500m → 标记为**移动 Cell**（高铁/车载/移动基站场景）
  - 跨度 ≤ 1500m → 标记为**固定 Cell**
- **Step 4**：在 BS 级别，只看固定 Cell 的质心分布，计算所有固定 Cell 质心的最大间距
  - 无固定 Cell → `dynamic_bs`（动态 BS）
  - < 500m → `normal_spread`（GPS 噪声，回归正常）
  - 500-1500m → `single_large`（面积大/低精度）
  - > 1500m → 潜在碰撞，进入设备分裂验证
  - 仅 1 个固定 Cell → `single_large`
- **Step 5**：对潜在碰撞 BS（2,199 个），验证设备是否空间分裂
  - 以 BS 级质心经度为分界，将固定 Cell 分为 A/B 两组
  - 检查每个设备（设备标识）出现在哪些组
  - 交叉率 = 同时出现在两组的设备数 / 总设备数
  - < 5% → `collision_confirmed`（确认碰撞：不同物理位置有各自的用户群）
  - 5-20% → `collision_suspected`（疑似碰撞）
  - > 20% → `collision_uncertain`（不确定）

**碰撞的本质**：运营商 BS 编码碰撞 = 不同物理基站共用同一个 (运营商编码, 制式, LAC, 基站ID) 组合键。已验证这不是数据处理造成的——碰撞 BS 下的 CellID 也完全分裂（一组 Cell 只在 A 侧出现，另一组只在 B 侧），确认是运营商 BS 编码复用。

**最终分类结果**：

| 分类 | BS 数 | 记录数 | 含义 |
|------|-------|--------|------|
| `dynamic_bs` | 5,124 | 983,153 | 全部 Cell 都是移动的（高铁/车载），GPS 画像精度低 |
| `collision_confirmed` | 2,013 | 726,109 | 确认碰撞（设备交叉率 < 5%），需虚拟拆分 |
| `collision_suspected` | 173 | 77,567 | 疑似碰撞（交叉率 5-20%） |
| `single_large` | 1,360 | 429,450 | 面积大或低精度（500-1500m 或仅 1 个固定 Cell） |
| `normal_spread` | 908 | 265,621 | GPS 噪声导致初始异常标记，实际正常 |
| `collision_uncertain` | 13 | 2,415 | 不确定（交叉率 > 20%） |

### 3.4 画像基线（进行中）

**做了什么**：基于 GPS+信号补齐后的 3008 万行明细，按 LAC / BS / Cell 三级聚合，生成画像。

**设计原则**：
- 所有 BS 都生成画像（包括异常 BS），但附带异常分类标记和可信度等级
- 画像即"基线"——日常运营中新数据的每个字段都要和画像指标对比
- 可信度分两维度：GPS 可信度（high/medium/low/none，取决于 GPS 点数量和异常标记）、信号可信度（high/medium/low，取决于原始信号占比）

**各级画像核心指标**：

| 级别 | 核心指标 |
|------|----------|
| LAC 画像 | 上报量、活跃天数、下属 BS 数/Cell 数、GPS 覆盖中心与半径、信号聚合(RSRP P10/P50/P90)、异常 BS 占比、GPS 来源比例 |
| BS 画像 | GPS 精算中心点、覆盖半径(P50/P90)、下属 Cell 数、设备数、信号聚合、异常分类标记、可信度等级 |
| Cell 画像 | GPS 中心点、信号分布(RSRP P10/P50/P90)、活跃天数、样本量、所属 BS 的分类标记 |

**当前进度**：LAC 画像样本已确认结构，BS 画像设计中，Cell 画像待启动。

## 五、当前协作逻辑的核心特征

### 5.1 必须保留的协作模式

1. **prompt 驱动的阶段化开发**：每个阶段有明确的 prompt、前置条件、执行计划、产出物定义
2. **先样本后全量**：大表操作先用小规模样本（如 6 个 LAC 的子集）验证逻辑和 UI 展示，确认后全量执行
3. **UI 审核确认**：每步规则和结果在 UI 上展示，人工确认后再进入下一步
4. **预算缓存**：千万级表的统计结果预算到缓存表，页面读缓存，避免前端超时
5. **元数据驱动**：字段定义、解析规则、清洗规则均存储在数据库元数据表中，代码从元数据读取

## 六、rebuild3 与 rebuild2 的边界

### 6.1 rebuild2 的定位

rebuild2 是**基于静态数据的研究与规则确认**：
- 用一批固定的历史数据（北京一周），从头验证整套处理逻辑
- 每一步产出稳定的中间结果和最终结论
- 产出的是**规则、阈值、口径、算法**——不是可持续运行的系统
- 所有处理都是一次性的：跑完一遍就得到结果，没有"新数据来了怎么办"的回答

### 6.2 rebuild3 的定位

rebuild3 是**将 rebuild2 的研究成果嵌入到动态/准流式治理系统中**：
- 不是推翻 rebuild2，而是将其成果结构化、动态化
- 核心变化：从"一次性跑完"变为"初始化 + 持续增量"
- 新增能力：对象状态机（注册→等待→观察→晋升→降级→退役）、等待/观察池、画像驱动准入、版本化规则

### 6.3 明确的边界

| 维度 | rebuild2 | rebuild3 |
|------|----------|----------|
| 数据模式 | 静态全量（一次性跑完） | 初始化 + 2 小时增量批处理 |
| 处理顺序 | LAC→Cell→BS（研究策略：先画可信边界） | 混合策略：初始化沿用现有顺序，增量以 Cell 先行（Cell→BS→LAC 联动） |
| 对象管理 | 一次计算产出，无状态流转 | 对象有生命周期，Cell 晋升驱动 BS/LAC 联动 |
| LAC 过滤 | 严格 GPS 边界框（北京）——研究阶段权宜之计 | 可配置参数（未来全国数据时只需过滤海外，通过运营商白名单即可） |
| 规则 | 硬编码在 SQL 和 prompt 中 | 版本化规则，可迭代、可回溯 |
| 画像 | 一次生成的静态快照 | 定期刷新的动态基线，参与准入判定 |
| 异常处理 | 研究后标记分类 | 持续检测、自动分流 |
| 版本管理 | 无 | run_id + contract_version + rule_set_version + baseline_version |
| UI | 阶段性研究审核工具 | 运营级观察与审查平台 |

## 七、本轮重构的目标、非目标、约束

### 7.1 目标

1. 设计 rebuild3 的清晰主框架（分层/分域、对象生命周期、增量骨架）
2. 明确 rebuild2 成果的继承策略（直接复用 / 需改造 / 暂不进入）
3. 统一初始化流程与 2 小时增量流程到同一套逻辑框架
4. 在不破坏当前 UI / DB / Agent 配合逻辑的前提下设计新框架

### 7.2 非目标

1. ❌ 不是立即写实现代码
2. ❌ 不是设计云端企业级大平台
3. ❌ 不是推翻现有业务哲学或 UI/DB 体系
4. ❌ 不是展开字段级、SQL 级、API 级细节
5. ❌ 不是做全国化/全制式的完整设计

### 7.3 约束

1. 技术栈以 PostgreSQL 17 + Python + FastAPI 为基础
2. 当前以本地验证为主，但框架要为未来上云做准备
3. 当前首轮以 4G/5G 为主，方法需能平滑扩展到 2G/3G
4. 保持现有 prompt 驱动的 Agent 协作方式

### 7.4 当前阶段的特殊性（必须理解）

当前只有北京一周数据，这造成了一些**仅在当前阶段存在、未来不会延续的特殊约束**：

| 当前特例 | 原因 | 未来全国数据时 |
|----------|------|----------------|
| 严格的 GPS 边界框过滤（只保留北京范围） | 一个城市 + 一周数据，不严格过滤会混入大量漫游/异常数据，干扰研究 | 只需过滤海外数据（通过运营商白名单即可），国内数据即使有少量噪声也无法累积注册 |
| LAC 级门控过滤 | 研究阶段需要快速画出可信边界 | LAC 可信性变成 Cell 聚合的派生属性，不再是前置门槛 |
| 固定时间窗口（7 天）的统计特征 | 数据只有 7 天 | 时间窗口可配置，低流量 Cell 可能需要 90 天累计 |

**架构原则**：这些参数化约束不应硬编码为系统架构，而应作为可配置参数传入。

### 7.5 验证计划

用现有 7 天数据验证 rebuild3 的初始化+增量闭环：

```
第 1~3 天数据 → 初始化（构建对象库 + 画像基线）
第 4~7 天数据 → 按 2 小时增量逐批跑入
对比：增量跑完 7 天的最终结果 vs 直接用 7 天数据一次性初始化的结果
评估：增量逻辑的收敛性和准确性
```

## 八、文档间口径不一致的标注

### 8.1 数据行数口径

早期方案文档中描述"一周约 3000 万行"，这是概略描述。legacy 阶段的标准化表有 1.33 亿行（含邻区扫描行）。rebuild2 重新解析后去除了邻区行，实际为：原始记录 4013 万行（GPS 2217 万 + LAC 1796 万），标准化展开后约 8220 万行（GPS 3843 万 + LAC 4377 万）。

**结论**：以 rebuild2 重新解析的数字为准。

### 8.2 可信 LAC 数量

legacy 阶段产出 881 个可信 LAC，rebuild2 重新计算后为 1,057 个。差异原因：rebuild2 增加了 ss1 解析（约 30% 原始记录含 ss1 数据），扩大了数据量。

**结论**：以 rebuild2 的 1,057 为准。

### 8.3 "8 域方案"与数仓分层的关系

项目中有两套分层描述：
- **8 域方案**（meta_contract / raw_source / work_run / core_entity / fact_governed / baseline_profile / issue_rule_lab / decision_ops）——本地一期的工程组织方式
- **标准数仓分层**（ODS / DWD / DIM / DWS / ADS）——未来上云的正式分层

已达成共识：**8 域是本地一期的工程主结构，不是正式数仓主结构**。rebuild3 应在此基础上明确二者关系。

### 8.4 Cell 注册表的定位

当前 Cell 只有"统计维表"（聚合统计信息），没有"注册表"（状态管理）。已达成共识：rebuild3 需要将 Cell 提升为有状态的注册对象（最小注册），但不扩成复杂主数据平台。
