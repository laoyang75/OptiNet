# rebuild3 UI v2 三方审计与偏差分析

---

## 背景

rebuild3 是在 rebuild2 基础上的一次完整重构，核心变化是从静态跑批模式转为流式治理模式（2 小时增量批次）。

UI v2 是本次重构的核心设计资产。它定义了系统的信息架构、用户操作链路、数据展示字段。UI 设计直接决定了后端需要哪些表、哪些读模型、哪些 API。因此 UI 与业务规则、数据实现之间的一致性，是进入开发前必须解决的关键问题。

本次审计的目标：对齐 UI v2 设计、rebuild3 冻结文档、rebuild2 实际实现三方之间的所有差异，找出偏差和需要决策的分歧点，确认后进入开发。

---

## 审计原则

1. **冻结文档是业务规则的唯一权威来源** — UI 与冻结文档冲突时，以冻结文档为准
2. **rebuild2 是数据实现的参考** — 了解实际表结构和逻辑，但 rebuild3 不必完全复制 rebuild2
3. **UI v2 是核心设计资产** — UI 提出的合理新需求（如 batch_snapshot 表）不算偏差，应标记为"需补充到实施计划"
4. **名称必须完全一致** — 字段名、状态值、枚举值在三方有任何差异都必须标出
5. **所有输出使用中文**
