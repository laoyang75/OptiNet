# Rebuild3 UI 视觉审美与结构审计偏差报告

**出具日期**: 2026年4月
**审计范围**: Rebuild3 Live (运行于 `127.0.0.1:47122`) 对比 UI_v2 Static HTML 静态设计原型 (`docs/UI_v2/pages/*`)
**核心结论**: 
当前开发的 Live 版本与 `UI_v2` 静态设计原型存在 **巨大的视觉风格与信息架构偏差**。当前开发版本仅仅堆砌了框架组件（如默认色块、默认表格、默认进度条），而完全丢失了 UI_v2 设计中要求的高级感、克制、留白以及定制化的数据可视化表达。这属于“严重降级”。

下面将按维度进行强对比剖析。

---

## 1. 全局视觉风格与布局偏差 (严重度: P0)

### 侧边栏与导航结构 (Sidebar)
*   **UI_v2 设计规范**: 采用了深藏青色（接近 `#2f3554` / `#1e2235`），菜单项无冗余背景，高对比度但非常柔和内敛，不使用巨大醒目的标题块。
*   **Live App 开发版本**: 侧边栏使用了过于明亮、大面积的蓝色（经典后台主题色），且页头包含了一块巨大的 `REBUILD3 流式治理工作台` 色块，极大地破坏了界面的“极致专业、克制”感。子菜单甚至带有一些不必要的英文字段 (`flow` / `batch` 等)。

### 页面骨架与卡片布局
*   **UI_v2 设计规范**: 强调卡片式极简设计，去网格、去厚重边框，数据呈现通过行距和字重来建立层次关系。页面大多采用通排（如初始化流程）结合核心指标的优雅卡片组展示。
*   **Live App 开发版本**: 存在非常生硬的 Dashboard 式二分/三分网格。排版为了填满网格而生搬硬套，如初始化页为了强行分列，把流程挤在左侧窄列，汇总信息挤在右面宽卡。

### 上下文模块 (Context Bar)
*   **UI_v2 设计规范**: 页面顶部“当前上下文”区域呈现如同一行内联的标签组，极致压缩了元数据的空间占用（纯字、极小的内边距），只展示核心的 `run_id`, `batch`, `窗口`, `版本`，显得非常克制。
*   **Live App 开发版本**: 将上下文渲染成庞大的区块，占用超过屏幕 15% 的高度，使用了标准 key-value 表格甚至多列大卡片，喧宾夺主。

---

## 2. 核心页面差异详解

### a. 流转总览页 (`01_flow_overview`)
*   **设计原则**: “时间快照与流转视图”。设计稿在中心区域用定制的 SVG 式节点链条（包含带有汉字的圆形节点：“入”, “注”, “标”, “查”, “分”, “治”, “观”...）高度可视化了流水线引擎的专业流程。
*   **Live App 偏差**: 核心的流转节点树**完全缺失**！取而代之的是通用的“四分流”彩色长条进度栏。这使得业务规则引擎的可观测性降级为普通的大屏粗糙统计，完全偏离了流式监控的设计意图。
*   **底部详情卡片**: 设计稿要求极低密度的清爽卡牌，而 Live 版本出现了巨大的、冗余的“问题入口”和“对照结论”大方框区域，且视觉对齐杂乱无章。

### b. 冷启动 / 初始化列表 (`12_initialization`)
*   **设计原则**: 11 步流程呈现了柔和的浅绿色 `[✓]` 步骤框，文本对齐有序，展示出系统自举时的优雅。
*   **Live App 偏差**: 用最基础的 Ant-Design `(1) (2)` 圆形数字组件顶替，状态仅是用黑字显示 `状态: completed`，视觉极其粗糙。

### c. 等待/观察工作台 (`05_observation_workspace`)
*   **设计原则**: 运用精细的条形图、小红点和悬浮样式，通过进度条的宽度和颜色巧妙传达“资格积累”的情况。
*   **Live App 偏差**: 此部分在 Live 版本中被强行替换为常规的表格加默认长进度条，缺失了悬浮面板、状态转化提示等微交互。

---

## 3. 设计中文化与文案偏差

*   **视觉噪音**: Live 版本直接暴露了大量的拼音和英文常量（如 `gps_bias`、`completed`、`fact_pending_issue`、`collision_suspect` 等），没有经过前端的中文转化。
*   **枚举标签 (Tags)**: 设计规范对“晋升”、“活跃”、“待处理”等打签逻辑设置了精确的语义色。而 Live 代码中使用的颜色明显是随便挑选的基础库色（例如刺眼的红色或不柔和的绿色），偏离了 `.impeccable.md` 约定的莫兰迪色柔和色谱体系。

---

## 4. 下一步整改建议 (Next Steps)

当前系统正处于“功能堆砌”的开发半成品阶段。必须**立即暂停**以功能完善为主的开发堆栈，转入页面 UI 重铸： 

1.  **废弃 Live 版本的底层 Dashboard CSS 框架模板**，彻底清理侧边栏、大头部及基础表单的样式。
2.  **严格对齐静态 HTML 设计稿 (`UI_v2/pages/*。html`)**，将页面级的 Flex 布局和核心字重、字号（Typography）像素级复刻。
3.  **重构可视化核心组件**：
    *   重构 `01_flow_overview` 里的状态机节点渲染（必须实现带圆圈文本图标的流程树）。
    *   重构顶层的 `Context Bar`，必须缩小并实现一行式标签组展现。
4.  **引入全局语义色卡 / 映射器**，彻底将诸如 `active` 等后端常量强制转换为系统级浅色语义中文标签（如 `绿底白字 [活跃]`）。

系统重建的意义不仅在于数据，还在于**专业感与可解释性**，这正是设计稿所传递的。目前的 Live 界面完全失去了这种“Data Governance Workbench”的高端气质。
