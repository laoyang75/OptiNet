> 使用场景：在进入开发之前，先重新完成 rebuild3 的 UI 设计，并确保设计主视角回到“流式治理工作台”，而不是“对比调试器”。

你现在扮演：

- 资深产品设计师

## 第三步：你必须先理解的产品判断

下面这些判断不是建议，而是本次设计的硬约束。

### 1. rebuild3 首先是什么

rebuild3 首先是一个**本地流式治理工作台**，用于观察：

- 数据如何在系统中进入
- 如何在四分流中流转
- 如何累计证据
- 如何触发晋升 / 降级 / 异常 / baseline 刷新
- 修复后哪些节点发生变化

### 2. rebuild3 不首先是什么

它不是：

- 静态报表系统
- 旧 Step 流程的换皮版
- 以回放对比为第一入口的验证器
- 假设每次都全量重跑、没有状态承接的一次性系统

### 3. 本次 UI 的第一目标

本次 UI 的第一目标是帮助用户回答：

1. 当前 run / batch / baseline 是什么
2. 当前批次的数据流到了哪里
3. 哪个节点的数量变化最大
4. waiting / issue / rejected 是否在堆积
5. 哪类对象失去了锚点或 baseline 资格
6. 修复后哪些节点变化符合预期

### 4. 回放对比的定位

回放对比是**验证模式**，不是整个产品的主叙事。  
它可以保留，甚至必须保留，但不能主导首页和整体信息架构。

### 5. 长期运行语义

设计时必须显式考虑下面这个事实：

- 用户会修逻辑、调规则、重跑部分批次
- 但本质上不会每次都完全重跑
- UI 必须支持“局部重跑 + 状态承接 + 修复后观察变化”的工作方式

## 第四步：本次设计必须解决的问题

请围绕下面这些问题完成设计：

1. 如何让用户在进入系统后第一时间看清**当前流转状态**，而不是只看到几张总览卡片？
2. 如何让用户快速发现：
   - 哪个批次变化异常
   - 哪个节点堆积
   - 哪类异常在扩大
   - 哪类对象资格失效变多
3. 如何让用户从“批次变化”一路下钻到“对象、事实、证据、资格原因”？
4. 如何让等待池真正服务“推进与判断”，而不是只做候选列表？
5. 如何让异常看板体现“影响路径”，而不只是罗列问题对象？
6. 如何让 baseline 页面体现“触发原因与影响范围”，而不只是版本对比？
7. 如何保留回放对比能力，但不让它主导整个产品结构？

---

## 第五步：禁止滑回的设计方向

本次设计中，下面这些倾向必须主动避免：

1. **不要把首页做成通用 KPI 仪表盘**
   - 不要只有几个数字、几个环图、一个最近批次列表就结束

2. **不要把回放对比做成最强主页面**
   - 它不是日常使用的第一视角

3. **不要只展示总量，不展示变化**
   - 必须尽量同时表达存量、流量、delta、原因

4. **不要只有对象视角，没有批次变化视角**
   - 用户不能只靠翻对象列表来猜哪里出问题

5. **不要只做静态报表，不做工作台**
   - 页面必须帮助用户定位问题并决定下一步动作

6. **不要让“第一次重构验证”绑死长期 UI 结构**
   - 第一次对照重要，但它不是长期主心智

## 第七步：页面设计硬要求

### A. 首页必须是“流转总览”

首页不是普通 Dashboard，而是当前治理状态的入口页。

至少必须包含：

1. 当前上下文
   - `run_id`
   - 当前 `batch_id`
   - 当前窗口
   - 当前 `baseline_version`
   - 当前 `rule_set_version`

2. 当前批次流转结构
   - 输入量
   - 四分流去向
   - 晋升 / 降级 / 级联变化
   - baseline 刷新触发情况

3. 关键变化
   - 相对上一批次的 delta
   - waiting / observing / issue / rejected 变化
   - health_state 分布变化
   - `anchorable` / `baseline_eligible` 变化

4. 问题入口
   - 变化最大的节点
   - 当前最值得处理的问题
   - 一跳进入等待池 / 异常 / 对象 / baseline

### B. 批次相关页面必须表达“变化”

批次页面至少要能表达：

- 这个 batch 进来了多少
- 出去了多少
- 分流比例和上一批次相比怎么变了
- 哪类对象在本批次被晋升、降级、转问题、拒收
- 哪些变化可能来自规则调整或异常放大

### C. 对象相关页面必须表达“证据与原因”

对象列表和对象详情不能只表达当前状态。

至少要明确表达：

- 当前 `lifecycle_state`
- 当前 `health_state`
- `anchorable`
- `baseline_eligible`
- 最近几个 batch 的事实分布
- 最近一次状态变化和资格变化原因
- 与 BS / LAC / baseline 的关系影响

### D. 等待池必须表达“推进节奏”

等待 / 观察页必须能直接看出：

- 证据累计进度
- 距离晋升还差什么
- 最近几个批次是否有推进
- 是在停滞、前进、回退，还是应该转问题
- 哪类 waiting / observing 正在批量堆积

### E. 异常看板必须表达“影响路径”

异常页不能只列问题对象。

至少要明确表达：

- 异常类型
- 严重度
- 是否禁锚
- 是否禁 baseline
- 最近证据变化
- 影响的下游对象 / 事实 / baseline 候选
- 从异常进入对象详情和相关批次详情的路径

### F. baseline 页面必须表达“触发与影响”

baseline 页面至少要能回答：

- 为什么刷新 / 没刷新
- 与上一版差异主要来自哪里
- 哪些对象被纳入或剔除
- 异常或 waiting 晋升对 baseline 的影响
- 当前 baseline 的稳定性风险

### G. 回放 / 对照页面必须降级为验证模块

回放页保留，但页面定位必须明确为：

- 重构验证
- 局部修复后的定向核对
- 解释差异，而不是主导系统使用流程

### H. 启动器（独立页面）

启动器是独立于主系统的运维控制面板，通过独立 Python 脚本启动。

**布局要求**：

- 服务卡片采用**横向三列小方块布局**（不要竖向三行），每个服务一个紧凑卡片
- 卡片内容：服务名称、状态徽标（绿色脉冲=运行中 / 红色=已停止 / 黄色=启动中）、PID、端口、运行时长
- 每个卡片底部：4 个操作按钮（启动 / 停止 / 重启 / 杀掉端口），其中"杀掉端口"用红色虚线边框标记为危险操作

**必须包含的功能区**：

1. **顶部**：标题 + 一键启动全部 / 一键停止全部 / 一键重启全部
2. **服务区（三列方块）**：后端 API（FastAPI, :8000）、前端开发服务（Vite, :5173）、数据库（PostgreSQL 17, :5432）
3. **日志查看区**：暗色背景、等宽字体、可切换服务 tab、按级别过滤（ALL/INFO/WARN/ERROR）、清空日志、自动滚动

**风格**：与主系统一致但更简洁，是运维面板不是数据分析工具。

### I. 主要下钻路径（必须在设计中体现为可点击入口）

下面这些跳转路径是用户的真实操作链路，页面设计中必须有明确的可点击入口支持：

```
流转总览 → 四分流任一节点 → 该批次该层事实列表
流转总览 → 变化最大节点 → 对应页面（等待池 / 异常 / 对象）
流转总览 → delta 异常指标 → 批次中心对应批次
批次中心 → 某批次晋升列表 → 对象详情
批次中心 → 某批次四分流 → 该层事实明细
等待池 → 某候选对象 → 对象详情（含推进历史）
异常看板 → 某异常对象 → 对象详情 + 影响的下游对象
异常看板 → 某异常类型汇总 → 该类型全部对象列表
baseline → 某差异对象 → 对象详情
baseline → 刷新触发原因 → 对应批次详情
```

不允许做出孤立页面。每个页面至少要有进入路径和下钻出口。

## 第十步：每个 HTML 页面的共同要求

1. 必须是完整可打开的静态 HTML
2. 必须使用 mock 数据，不允许空占位
3. 必须保持导航、上下文条、全局状态表达一致
4. 必须体现桌面优先的本地工作台风格
5. 可以有动效，但动效只服务状态变化和层级引导
6. 不要为了“像设计稿”而牺牲可扫描性

特别注意：

- 不要只做一堆卡片 + 表格 + 图表的堆叠
- 要让页面能体现”流动、变化、积压、下钻”
- 批次和版本上下文尽量在全局可见

### Mock 数据中的 delta 展示硬要求

所有指标卡和关键数字必须同时展示**当前值 + 本批次新增量 + 相对上一批次的 delta**。

格式示例：
- `活跃对象 12,847 | 本批 +3 | 较上批 +1 ↑`
- `等待池 1,283 | 本批 +15 | 较上批 -3 ↓`
- `异常对象 47 | 本批 +2 | 较上批 +2 ↑`（红色上箭头表示恶化）

不允许只放一个孤立数字。如果某个指标确实没有 delta（例如首次初始化），用 “—“ 标记而不是省略。

### design_system.html 的处理

design_system.html 可以沿用 v1 的色彩、字体、间距规范（已在 `.impeccable.md` 中冻结），但需要补充以下 v2 新增组件：
- delta 展示组件（上升/下降箭头、变化量标签、恶化/改善配色）
- 批次流转图组件（四分流可视化 + 节点标注）
- 重跑标记徽标（区分正常批次和重跑批次）
