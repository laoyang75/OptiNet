## 背景

rebuild3 是一个本地桌面流式治理工作台（Vue 3 + FastAPI）。前端实现存在**结构性偏差**——不是文案翻译问题，而是页面结构、组件架构、交互模式与 UI_v2 设计稿有根本性差异。需要逐页对照设计稿进行结构重构。

## 硬约束（不可违反）

1. 遵守 `.impeccable.md` 设计原则（精准·专业·可控，浅色克制工作台）
2. 不改业务术语（lifecycle_state、health_state、anchorable、baseline_eligible 等）
3. 浅色模式、蓝靛蓝主色、不要紫色主导、不要毛玻璃、不要 AI 套壳风格
4. 所有可见文案优先中文，技术字段值允许英文
5. 不要推翻 UI_v2 设计稿，除非与冻结业务文档直接冲突
6. 修完后必须 `cd rebuild3/frontend && npm run build` 验证构建通过

## 之前犯过的错误（必须避免）

1. **只修文案不看结构** — 把 English→Chinese 当作主要工作，完全忽略了页面布局、组件架构、交互模式的结构性偏差
2. **没有对照设计稿 HTML** — 设计稿 HTML 是像素级参考，不是"大概长这样"的参考
3. **把验证/对照内容塞进首页** — SAMPLE vs FULL 对照表不属于流转总览，属于验证/对照页
4. **用通用组件替代专用组件** — 用一个 BadgePill 替代了所有状态徽标，丢失了设计稿中每种状态独有的图标、颜色、进度条
5. **扁平化组件层次** — 设计稿有嵌套组件（如三层资格卡片），实现时全部打平成简单 div
6. **忽略交互模式** — 设计稿中的可展开行、子 Tab、筛选器、时间选择器全部跳过
7. **用错误的数据填充正确的位置** — 比如把 batch scope 的 "full"/"sample" 直接渲染而不做标签映射

## 正确的工作方法

1. **先开设计稿 HTML 截图，再开当前页面截图，逐区块结构对比**
2. **读对应的 *_doc.md 理解每个区块的设计意图**
3. **按设计稿的组件边界拆分代码，不用通用组件糊弄**
4. **交互模式必须实现：可展开行、Tab 切换、筛选器、下钻链接**
5. **API 不够用就先改后端，不要在前端硬凑**
