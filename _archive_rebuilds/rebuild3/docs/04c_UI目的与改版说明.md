# rebuild3 UI 目的与改版说明

> 这份文档用于把 rebuild3 的 UI 目标重新钉死，避免后续设计再次滑回“对比调试器”或“静态报表页”。

---

## 1. UI 的真正目的

rebuild3 的 UI 首先是一个**本地流式治理工作台**，其次才是一个**验证与对照工具**。

它服务的不是“只看最终结果的人”，而是下面两类会持续进入系统的人：

1. **数据运营 / 数据分析人员**
   - 观察对象、事实、异常、baseline 的日常变化
   - 快速发现某个节点的积压、流失、异常波动
   - 判断当前规则、阈值、资格条件是否合理

2. **技术运维 / 重构验证人员**
   - 确认 run / batch / baseline 是否按预期推进
   - 检查修复逻辑后，局部重跑是否带来预期变化
   - 在第一次重构阶段，用旧版本结果做对照验证

### 1.1 这个 UI 要解决的核心问题

用户每天进入这个系统，不是为了欣赏一张“总览报表”，而是为了回答下面这些问题：

1. **当前系统正在跑什么？**
   - 当前 run 是什么
   - 当前 batch 窗口是什么
   - 当前参考的是哪个 baseline

2. **数据在当前批次流到了哪里？**
   - 进入了多少
   - `governed / pending_observation / pending_issue / rejected` 各有多少
   - 相比上一批、上一天、上一版规则，哪个节点变化最大

3. **哪里出现了积压、流失或异常放大？**
   - waiting / observing 是否堆积
   - 哪类 issue 在上升
   - 哪类 rejected 突然变多
   - 哪类对象的 anchorable / baseline_eligible 大量失效

4. **问题具体落在哪一层？**
   - 是记录级问题、对象级问题，还是 baseline 刷新逻辑问题
   - 是 Cell 本身卡住，还是级联影响了 BS / LAC

5. **修复后结果有没有变好？**
   - 修了某条逻辑、调了某个阈值之后
   - 哪些节点数量变化了
   - 哪些对象从 waiting 进入 observing / active
   - 哪些 issue 被消化、哪些新问题出现

6. **第一次重构时，和旧版本差在哪里？**
   - 对照能力是必要的
   - 但它是“验证模式”，不是整个产品的主视角

### 1.2 这不是一个什么系统

rebuild3 的 UI 明确**不是**下面这些东西：

- 不是只在最后展示结果的静态报表
- 不是围绕旧 Step 流程复刻的页面导航
- 不是以“回放对比”作为首页主任务的验证器
- 不是每次都假设全量重跑、没有状态承接的一次性跑批工具

## 2. 使用者怎么使用这个 UI

下面这条链路，才是 rebuild3 UI 应该服务的真实使用方式。

### 2.1 日常使用主链路

1. **先看启动器**
   - 确认前后端、任务服务、日志是否正常

2. **进入主工作台后先看“流转总览”**
   - 看当前 run / batch / baseline 上下文
   - 看当前批次进入量、四分流结果、关键 delta
   - 看 waiting / issue / rejected 是否突然上升

3. **定位异常节点**
   - 某一层数量突增
   - 某一类 health_state 异常扩张
   - 某一类资格失效比例上升

4. **从节点下钻到对象和证据**
   - 先看批次级变化
   - 再看对象列表
   - 再看对象详情中的状态历史、关系历史、事实落点、资格原因

5. **决定动作**
   - 继续观察
   - 转问题
   - 检查规则
   - 修复逻辑
   - 触发局部重跑 / 批次复跑

6. **修复后回来看变化**
   - 核心是看“变化是否发生在预期节点”
   - 不是只看最终总数是否漂亮

### 2.2 第一次重构阶段的辅助链路

1. 做局部修复或参数调整
2. 跑指定批次或指定窗口
3. 查看新旧结果差异
4. 判断差异是否可解释
5. 把验证结果反馈回规则和实现

这条链路重要，但它应该作为**验证视角**存在，而不是压过主工作台。

## 3. 当前设计的主要偏差

当前 `docs/rebuild3/UI/` 下的设计已经比旧 Step 思路好很多，但整体重心仍然偏向“结果展示 + 对比验证”，没有把“流式过程观测”提到第一层。

### 3.1 偏差一：首页仍然过于像仪表盘

当前 `docs/rebuild3/UI/design_notes.md` 把首页定义为“系统全局概览”，`docs/rebuild3/UI/pages/01_dashboard_doc.md` 也强调“5 秒内看整体健康状态”。

这没有错，但还不够。问题在于：

- 它更像静态指标总览，而不是流转工作台入口
- 强调“看整体状态”，但没有强制“看本批次节点流向与变化”
- 缺少“相对上一批 / 上一日 / 上一规则版本”的 delta 表达
- 缺少“积压、流失、滞留、级联影响”的主视图

### 3.2 偏差二：回放对比被抬得太高

当前 `docs/rebuild3/UI/design_notes.md` 把“收敛验证”列为五类核心决策之一，`docs/rebuild3/UI/pages/09_replay_doc.md` 又把综合评分、详细对比、趋势图做得很重。

这在第一次重构时有价值，但长期来看会造成两类误导：

1. 容易让 agent 误以为这个产品首先是“对比工具”
2. 容易让首页和主导航围绕“验证结果”组织，而不是围绕“治理过程”组织

### 3.3 偏差三：没有把”非全量重跑”写成核心语义

你当前真实的工作方式是：

- 修逻辑
- 重跑某些批次或某个阶段
- 带着已有状态继续推进
- 再观察流转变化

但当前设计里，对”状态承接””局部重跑影响范围””修复前后节点变化”表达得不够强。

具体来说，UI 需要能区分：

- **正常推进的批次** vs **修复后重跑的批次**（例如用不同标记或背景色）
- **重跑影响了哪些对象**（哪些对象状态因重跑而发生了变化）
- **重跑前后节点变化对比**（不是全量对比，而是定向看受影响节点的 delta）

### 3.4 偏差四：页面已经按主语分了，但还没按“决策节奏”排优先级

现在的页面清单基本完整，但默认优先级更像：

`仪表盘 -> 运行中心 -> 对象 -> 等待池 -> 异常 -> 批次审查 -> baseline -> 回放`

更合理的优先级应当是：

`流转总览 -> 批次与节点变化 -> 对象/等待/异常下钻 -> baseline 判断 -> 验证对照`

也就是先看**流动与变化**，再看**对象与原因**，最后才看**验证与对照**。

## 4. 明确修改说明

下面这些修改，不是“可选优化”，而是后续 UI 重新设计时应明确落实的方向。

### 4.1 首页语义改为“流转总览”，不再是静态 Dashboard

首页必须从“全局概览页”改成“当前运行态的流转总览页”。

首页至少要明确展示四类信息：

1. **当前上下文**
   - 当前 run
   - 当前 batch
   - 当前 baseline
   - 当前规则版本

2. **当前批次流转**
   - 输入量
   - 四分流去向
   - 晋升 / 降级 / 级联变化
   - baseline 刷新触发情况

3. **变化与风险**
   - 相对上一批次的 delta
   - waiting / issue / rejected 的增减
   - health_state 异常分布变化
   - anchorable / baseline_eligible 失效波动

4. **可下钻的问题入口**
   - 哪个节点变化最大
   - 哪类问题需要优先处理
   - 哪个页面最值得马上进入

### 4.2 把“存量 + 流量 + 变化 + 原因”作为统一设计语言

所有核心页面都不能只给总量。

每个关键区块尽量同时表达：

- **存量**：当前有多少
- **流量**：本批次新进了多少
- **变化**：相比上批/上日增加或减少多少
- **原因**：主要由什么状态、规则、异常导致

如果一个页面只给总数和静态列表，基本就还不够。

### 4.3 把批次变化视角提到和对象视角同级

对象页面重要，但不能让用户只能”先看对象，再猜哪里出问题”。

必须增加明显的**批次变化视角**：

- 哪个 batch 的四分流比例异常
- 哪个 batch 的 waiting 激增
- 哪个 batch 出现异常集中爆发
- 哪个 batch 触发了 baseline 刷新或没有触发
- **近 N 批趋势对比**（例如最近 6 个批次 waiting 数是持续上涨还是稳定），不能只看单批次快照

### 4.4 等待池要改成“推进工作台”，不是候选对象列表

等待 / 观察池不应只是展示候选对象，而应直接回答：

- 它距离晋升还差什么
- 最近几个批次有没有推进
- 是在前进、停滞、回退，还是应该转问题
- 这类对象批量堆积的原因是什么

换句话说，这个页面的关键词应是：

- 门槛差距
- 批次推进
- 卡点原因
- 建议动作

### 4.5 异常看板要增加“影响路径”

异常看板不能只列异常对象。

必须尽量表达：

- 异常类型
- 当前严重度
- 是否禁锚
- 是否禁 baseline
- 影响了哪些下游对象 / 事实 / baseline 候选
- 最近几个批次证据是否在增加

### 4.6 对象详情要补齐“流转证据”

对象详情除了当前快照，还应重点表达：

- 最近批次中的进入事实分布
- 最近状态变化发生在哪个批次
- 最近资格变化原因
- 它在本轮异常或观察中扮演什么角色
- 它对 BS / LAC / baseline 的影响链

### 4.7 baseline 页面要强调“触发与影响”，而不是只看版本

baseline 页面不能只是版本管理。

它应该回答：

- 为什么这次刷新了 / 没刷新
- 新版 baseline 相比上一版变化集中在哪些对象类型
- 哪些变化来自 waiting 晋升，哪些来自异常剔除
- 当前 baseline 的稳定性风险在哪里

### 4.8 回放对比降级为“验证模式”，不再主导整体 IA

回放对比仍然保留，但角色应下调为：

- 用于第一次重构阶段的验证
- 用于修复后的定向解释
- 用于对照旧版本或全量初始化结果

它不应再定义整个系统的主首页、主叙事和主要导航权重。

---

## 5. 建议的新信息架构重心

建议把整个系统的主叙事改为下面这条：

1. **流转总览**
   - 今天 / 当前批次发生了什么

2. **批次与节点变化**
   - 是哪个节点的数量或比例变了

3. **对象与证据下钻**
   - 具体是哪些对象、哪些事实、哪些资格原因

4. **问题处置**
   - 等待继续观察、异常复核、规则修复

5. **baseline 判断**
   - 当前是否应该刷新、刷新会带来什么影响

6. **验证对照**
   - 修复后结果是否更接近期望

说明：启动器仍作为独立入口页面，不纳入主导航叙事。它服务于运维人员管理系统进程，与主工作台的治理叙事分离。

---

## 6. 对后续 Prompt 的要求

后续 UI 设计 Prompt 必须显式加入下面这些硬约束：

1. **先定义 UI 目的和用户使用链路，再设计页面**
2. **首页必须是流转总览，不是通用 Dashboard**
3. **回放对比是辅助验证模块，不是系统主入口**
4. **所有关键页面都必须表达 delta，而不仅是总量**
5. **必须支持“局部重跑 + 状态承接 + 修复后观察变化”的心智**
6. **必须支持从批次异常下钻到对象，再看到证据和资格原因**
7. **不得把系统设计成只适合第一次重构对照，而不适合长期流式运营**

---

## 7. 一句话结论

rebuild3 的 UI 应该优先帮助用户看清**数据如何在系统里流、在哪一层卡、为什么变化、修完后有没有改善**；  
回放对比只是这个工作台里的一个验证视角，而不是整个产品的定义。
