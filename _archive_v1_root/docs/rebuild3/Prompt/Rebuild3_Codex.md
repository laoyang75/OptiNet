## 三、项目背景（请视为硬约束）

### 1）项目性质

这是一个移动网络优化数据仓库项目，核心目标是：

从运营商原始报文数据中，持续构建并维护可信的：
- `BS`（基站）
- `Cell`（小区）
- `LAC`（位置区）

三级基础对象库与画像体系。

### 2）当前数据源

当前本地测试原始表是：
- `网优项目_lac定位北京明细数据_20251201_20251207`

其特点：
- 北京地区一周数据；
- 约 1800 万行原始记录；
- 未来上线后数据源可能变化，但应通过**输入适配层**解决，而不是把主逻辑绑死在单一表名上。

每条原始记录包含两个关键嵌套字段：
- `cell_infos`（JSON）：手机实时基站扫描结果，包含制式、CellID、LAC、运营商、信号强度等；
- `ss1`（紧凑字符串）：后台采集的信号 + 时间 + GPS + 基站 + AP 组合信息。

单条原始记录经解析后，可展开为多条标准化记录（展开比约 24–29 倍）。

### 3）rebuild2 已完成的核心研究（必须复用，不要重新发明）

rebuild2 本质上是**基于静态数据库的研究与规则确认**，已经基本完成：

#### A. 字段解析与清洗（L0）
- 27 个原始字段 → 42 个标准化字段的映射；
- `cell_infos` JSON 解析规则；
- `ss1` 字符串解析规则；
- ODS 清洗规则（删除行 / 置空 / 转换 / 过滤）。

#### B. 可信对象库构建规则（L2）
已验证的规则包括但不限于：
- 运营商白名单；
- LAC / CellID 合法范围；
- GPS 边界框；
- 信号有效范围；
- 可信对象统计口径；
- `dim_lac_trusted`、`dim_cell_stats`、`dim_bs_stats` 等静态研究产物。

#### C. GPS 修正与补齐规则（L3）
包括但不限于：
- `BS_ID` 派生规则：4G = `cell_id / 256`，5G = `cell_id / 4096`；
- BS / Cell 质心逻辑；
- GPS 质量等级；
- GPS 修正阈值；
- 多级回退补齐；
- 同 Cell / 同 BS 的信号补齐路径。

#### D. BS 异常分类研究（L3B）
已验证并需要继承其思想与口径：
- `normal_spread`
- `single_large`
- `dynamic_bs`
- `collision_confirmed`
- `collision_suspected`
- `collision_uncertain`

#### E. 三级画像基线（L5）
- LAC 画像；
- BS 画像；
- Cell 画像；
- 相关可信度与统计口径。

### 4）rebuild3 的目标

rebuild3 不是继续做静态全量研究，而是要把上面的成果嵌入到一个**动态 / 准流式治理系统**中：

#### 阶段一：初始化
- 用一段历史数据（例如 14 天）冷启动；
- 生成初始可信 `lac / bs / cell` 对象库；
- 建立初始画像和基线。

#### 阶段二：增量批处理（当前按 2 小时一批设计）
- 新数据进入后，完成标准化、主键解析、清洗；
- 已在库对象：判断是否可直接进入治理后事实；
- 不在库对象：进入等待 / 观察池继续累计；
- 样本累计满足条件后：评估是否晋升进入正式库；
- 若对象已存在但出现异常：
  - GPS 偏差
  - BS 碰撞
  - 动态 Cell / 动态 BS
  - 迁移
  - 数据不足待观察
  则不能简单覆盖，要进入规则判定与持续累计。

#### 阶段三：持续运维
- 画像基线定期刷新；
- 异常检测；
- 状态流转管理；
- 必要时局部重算与完整回归。

### 5）业务原则（默认继承，不要重新争论）

请默认继承这些原则：
- 有效 `cell_id` = 有效上报；
- 修正优于丢弃；
- 先建可信对象，再修正事实；
- 基线驱动准入与日常治理；
- 正向收敛 + 反向纯化 + 必要时完整回归。

### 6）扩展性原则

- 当前首轮设计以 4G / 5G 为主；
- 方法需要能平滑扩展到 2G / 3G；
- 但不要为了“理论上全支持”而牺牲当前框架清晰度。
