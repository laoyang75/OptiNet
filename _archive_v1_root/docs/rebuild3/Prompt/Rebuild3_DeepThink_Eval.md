# Rebuild3 深度分析评估 Prompt

> 用途：将此 prompt 连同两个 agent 的输出一起提交给深度分析 agent
> 你不能访问代码库，所有必要信息已包含在附件文档中

---

## 你的任务

你将收到**两份独立分析结果**（来自两个不同的 agent），它们都是对同一个项目 rebuild3 重构的框架分析。你需要：

1. **横向对比**两份分析的核心判断，找出共识、分歧和各自盲区
2. **给出你自己的判断**，产出一份**可直接指导 rebuild3 重构的框架文档**
3. 这份文档不是详细设计文档（详细设计由后续可访问代码的 agent 完成），而是**重构指导文档**——明确做什么、按什么顺序做、哪些是控制环节

---

## 项目背景（你必须理解的上下文）

### 这是什么项目

移动网络优化数据仓库。从运营商原始报文数据中，构建并维护 Cell（小区）→ BS（基站）→ LAC（位置区）三级可信对象库和画像体系。

### 当前处在什么阶段

- **rebuild2（已完成）**：基于静态数据（北京一周 4000 万行原始记录）的研究阶段。已验证全套规则：字段解析、可信对象筛选、GPS 修正、信号补齐、BS 异常分类（碰撞/动态/噪声）、三级画像。**规则已确认，算法已验证。**
- **rebuild3（即将启动）**：把 rebuild2 的静态规则嵌入到动态系统中。核心变化：从"一次性跑完"变为"初始化 + 持续增量"。

### rebuild3 的真实定位（极其重要，不要误判）

**rebuild3 不是线上生产系统。** 它是：

1. **全流程调试环境**：用单一数据源（北京一周数据）跑通从原始数据到动态对象库的完整流程
2. **为上线做准备**：调试并确认每个环节的进入/退出控制逻辑，发现问题，调整参数
3. **验证增量逻辑的收敛性**：用 3 天初始化 + 4 天增量的结果，对比直接 7 天初始化的结果

所以你在评估时：
- ❌ 不要按生产级系统的标准评审（高可用、分布式、容灾等不需要）
- ❌ 不要建议复杂的基础设施（消息队列、事件驱动、微服务等不需要）
- ✅ 要聚焦在**逻辑完整性**和**控制环节是否明确**
- ✅ 要关注**增量逻辑是否能收敛到与全量初始化一致的结果**

## 你必须给出判断的核心问题

### A. 处理顺序

两份分析可能对"初始化和增量的处理顺序"有不同判断。当前的共识方向是：
- 初始化：沿用已验证的 LAC→Cell→BS 自顶向下顺序（批量高效，规则成熟）
- 增量：Cell 先行，自底向上级联（Cell→BS→LAC）

**你需要评估**：
1. 这个混合策略是否合理？是否存在初始化和增量逻辑不一致导致的收敛风险？
2. 增量模式下 Cell 级联晋升的 BS/LAC 是否能与初始化产出的 BS/LAC 等价？如果不能，差异是否可接受？

### B. 控制环节识别

rebuild3 的流程中有多个"控制环节"——即需要明确进入条件和退出条件的节点。你需要识别并列举**所有关键控制环节**，至少包括但不限于：

- 记录级合规检查（什么条件通过/拒收）
- Cell 等待池进入条件
- Cell 晋升条件（数据质量指标，不是具体数值）
- BS 级联构建的触发条件
- LAC 级联构建的触发条件
- GPS 修正的前置条件（什么状态的对象可以修正）
- 信号补齐的前置条件
- per-Cell 详细库的滚动策略（何时写入、何时淘汰）
- 画像刷新的触发条件
- 异常检测的触发条件和分流规则
- 对象降级/消失/退役的触发条件
- 完整回归的触发条件

对每个控制环节，说明：
1. 它控制什么（进入/退出/触发）
2. 它需要哪些输入指标
3. 它的输出是什么（状态变更/记录写入/事件触发）
4. 如果判断错误会怎样（假阳性/假阴性的后果）

**注意**：不需要给出具体参数值（那是调试阶段的事），但需要明确"这个环节需要哪些可调参数"。

### C. per-Cell 详细库的设计合理性

这是 rebuild3 新增的核心设计：每个 Cell 保留最近 N 条治理后明细，画像从这里计算。

你需要评估：
1. 这个设计是否合理？与直接从全量事实表聚合画像相比，优劣是什么？
2. 滚动窗口是否会导致画像计算偏差（如季节性变化被丢弃）？
3. 封顶数 N 的选择逻辑是什么？应该基于什么指标？

### D. 验证计划的充分性

当前验证计划：3 天初始化 + 4 天增量 → 对比 7 天直接初始化。

你需要评估：
1. 这个验证方案能覆盖哪些场景？不能覆盖哪些？
2. 对比时应该关注哪些指标？（不只是"结果一样"，而是具体看什么）
3. 7 天数据是否足够验证增量逻辑？可能遗漏什么？

### E. 两份分析的盲区

指出两份分析各自可能忽略的问题。例如但不限于：
- 是否有隐含的循环依赖没有被发现？
- 是否有状态转移路径没有被覆盖？
- 是否有数据一致性风险没有被提及？

## 你的输出要求

产出**一份文档**，结构如下：

### 第一部分：两份分析的对比评估（简短，不超过全文 20%）
- 共识点
- 分歧点及你的判断
- 各自盲区

### 第二部分：rebuild3 重构指导框架（核心，占全文 60%+）

这部分必须能直接交给实现团队（代码级 agent）作为重构指导。包含：

1. **总体策略**：一段话说清楚 rebuild3 做什么、怎么做、按什么顺序
2. **处理流程骨架**：初始化流程 + 增量流程，包含明确的步骤和步骤间关系
3. **完整的控制环节清单**：每个控制环节的进入条件、退出条件、需要的可调参数、判断错误的后果
4. **对象生命周期**：Cell/BS/LAC 的状态定义、所有合法状态转移路径、级联关系
5. **数据流与存储设计原则**：per-Cell 详细库、事实层、画像层的关系和数据流向

### 第三部分：验证计划与风险（占全文约 20%）

1. 验证方案细化：3+4 天验证应该关注的具体指标
2. 已识别风险及缓解建议
3. 进入实现前必须确认的最小决策清单（不超过 5 项）

---

## 写作规范

1. **中文撰写**
2. **自包含**：不引用文件路径，不假设读者能访问代码，所有逻辑内联描述
3. **面向实现**：不写空泛架构词汇，每个结论都要能转化为实现动作
4. **控制环节是重点**：这是本文档的核心价值——让实现者知道"流程中哪些地方需要做判断、判断依据是什么"
5. **不给具体参数值**：参数值是调试阶段的事，但必须明确"这个环节有哪些参数需要调"
6. **不展开详细设计**：表结构、字段定义、SQL 逻辑等留给后续阶段
