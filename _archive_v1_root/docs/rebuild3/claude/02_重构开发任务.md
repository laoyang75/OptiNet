## Phase 0: 基础设施

### T0.1 数据库 Schema 创建

**目标**：在 PG17 上创建 rebuild3 所需的全部表结构。

**输入**：
- 技术栈要求文档中的表族定义

**产出**：

## Phase 1: 初始化链路

### T1.1 标准化事件生成模块

**目标**：将原始记录解析为标准化事件，完全复用 rebuild2 已验证的规则。

**输入**：
- T0.3 提供的原始记录流
- 已冻结的解析规则：
  - cell_infos JSON 解析（22 个字段提取）
  - ss1 紧凑字符串解析（5 元素拆解，信号按制式匹配基站，继承逻辑）
  - 26 条 ODS 清洗规则
  - 派生字段计算（bs_id = 4G: cell_id/256, 5G: cell_id/4096）
  - 17 条合规规则

**产出**：
- 函数 `standardize(batch_id, source_config) -> (std_event_count, error_count)`
- 写入 `fact_standardized` 和 `parse_error`
- 每条标准化记录包含：网络标识 + GPS + 信号 + 合规标记 + 解析来源

**验收标准**：
- 与 rebuild2 的 `l0_gps` / `l0_lac` 结果行数一致（误差 < 0.1%）
- 解析规则 100% 复用（无新增/修改）
- 幂等写入（ON CONFLICT DO NOTHING）
- 错误记录有完整的错误类型和原始片段

**关键规则清单**（直接继承）：
- 运营商白名单：46000/46001/46002/46003/46006/46011/46015/46020
- LAC 合法范围：4G 1~65535, 5G 1~16777215
- CellID 合法范围：4G 1~268435455, 5G 1~1099511627775
- 信号有效范围：RSRP -156~-30, RSRQ -34~3.5, SINR -23~40, Dbm -156~-20, ASU 0~99
- GPS 边界框：中国范围 70-140°E, 15-55°N

### T1.2 可信对象构建模块（LAC→Cell→BS）

**目标**：从标准化事件中按 LAC→Cell→BS 顺序构建初始可信对象库。

**输入**：
- `fact_standardized` 中的合规记录
- 可信筛选规则（复用 rebuild2）

**产出**：
- 函数 `build_trusted_library(run_id) -> (lac_count, cell_count, bs_count)`
- 写入 `obj_lac`, `obj_bs`, `obj_cell`
- 所有对象初始状态：`lifecycle_state=active`, `health_state=healthy`

**步骤**：
1. **可信 LAC**：运营商白名单 → LAC 编码合法 → GPS 区域过滤（可配置，当前北京） → 活跃天数+上报量稳定性 → 写入 `obj_lac`
2. **可信 Cell**：限定在可信 LAC 内 → 按 (运营商,制式,CellID,LAC) 聚合 → 活跃天数/空间分布/上报量过滤 → 计算 Cell 质心（GPS 加权平均） → 写入 `obj_cell`
3. **可信 BS**：按 (制式,BS_ID,LAC) 聚合可信 Cell → 从 Cell 质心取**中位数**作为 BS 中心点 → 计算覆盖半径 P50/P90 → 写入 `obj_bs`

**验收标准**：
- LAC 数量与 rebuild2 的 1,057 接近（允许因参数调整有差异）
- BS/Cell 数量与 rebuild2 接近
- BS 中心点用的是**中位数**而非平均值
- 对象主键无重复
- 状态日志记录了所有注册事件

### T1.3 事实治理模块（GPS 修正 + 信号补齐）

**目标**：对已匹配到可信对象的标准化记录执行 GPS 修正和信号补齐。

**输入**：
- `fact_standardized` 中的合规记录
- `obj_bs`（BS 中心点作为修正锚点）
- `obj_cell`（Cell 状态，决定是否允许修正/补齐）

**产出**：
- 函数 `govern_facts(batch_id, object_library) -> governed_count`
- 写入 `fact_governed`（每条附带 gps_source + signal_source + anomaly_tags）
- 写入 `cell_detail_hot`

**规则**（直接继承）：
- GPS 修正：
  - 距离 ≤ 阈值（4G: 1000m, 5G: 500m）→ 保留原 GPS，gps_source='original'
  - 距离 > 阈值 → 钉成 BS 中心点，gps_source='bs_center'
  - 原始 GPS 缺失 → 回填 BS 中心点，gps_source='bs_center'
  - **所有情况保留整条记录，信号字段不动**
- 信号补齐：
  - 第一阶段：同 cell_id 下时间最近有信号记录逐字段补齐
  - 第二阶段：同 BS 下数据量最大 Cell 的时间最近记录补齐
  - 记录补齐来源和 donor 标识
- **前置门控**：
  - `lifecycle_state=active` 且 `health_state=healthy` → 正常修正
  - `health_state=collision_confirmed/dynamic` → GPS 修正降级或禁用
  - `lifecycle_state=waiting` → 不修正、不补齐

**验收标准**：
- 与 rebuild2 的 `dwd_fact_enriched`（3008 万行）结果对比：GPS 来源分布接近、信号补齐率接近
- 每条记录的 gps_source 和 signal_source 字段正确填充
- 碰撞/动态 BS 的记录正确降级处理

### T1.4 BS 异常分类模块

**目标**：对空间散布异常的 BS 执行 Cell 质心法 v2 分类。

**输入**：
- `obj_bs`（初始构建结果）
- `fact_standardized`（含 GPS 和 CellID）

**产出**：
- 函数 `classify_bs_anomaly(run_id) -> classification_counts`
- 更新 `obj_bs` 的 anomaly_class 和 health_state
- 产出 6 种分类

**算法**（Cell 质心法 v2，直接继承）：
1. 筛选候选 BS：GPS 点 ≥ 10, 设备 ≥ 2, P90 偏移 > 1500m 或最大偏移 > 5000m
2. 按 500m 网格（0.005°）对每个 Cell 独立聚合 GPS 点
3. 每个 Cell（GPS 点 ≥ 3）计算加权质心和空间跨度；跨度 > 1500m = 移动 Cell
4. BS 级只看固定 Cell 质心最大间距：
   - 无固定 Cell → `dynamic_bs`
   - < 500m → `normal_spread`
   - 500-1500m → `single_large`
   - \> 1500m → 进入设备分裂验证
   - 仅 1 个固定 Cell → `single_large`
5. 设备分裂验证：以 BS 质心经度分 A/B 组，计算设备交叉率
   - < 5% → `collision_confirmed`
   - 5-20% → `collision_suspected`
   - \> 20% → `collision_uncertain`

**验收标准**：
- 分类结果与 rebuild2 的 `_research_bs_classification_v2`（9,591 行）对比一致
- 碰撞确认的 BS 自动更新 `health_state=collision_confirmed`, `anchorable=false`

**目标**：用纯化后的对象库重跑事实治理，消除"先粗后细"的语义混杂。

**输入**：

**产出**：
- 函数 `full_regression(run_id) -> governed_count`

**验收标准**：
- 回归后所有记录基于同一套对象状态处理
- 碰撞 BS 的记录 GPS 修正已降级

### T1.6 画像基线生成模块

**目标**：从治理后事实计算三级画像。

**输入**：
- `fact_governed`（baseline_eligible=true 的记录）
- `obj_cell`, `obj_bs`, `obj_lac`

**产出**：
- 函数 `build_profile(scope, baseline_version) -> profile_counts`
- 写入 `profile_lac`, `profile_bs`, `profile_cell`

**画像内容**：
- LAC：覆盖范围中心+半径, BS/Cell 数, 信号聚合(RSRP P10/P50/P90), 活跃天数, 异常 BS 占比, GPS 来源比例
- BS：精算中心点, P50/P90 覆盖, Cell 数/设备数, 信号聚合, 异常分类标记, GPS 可信度(high/medium/low/none), 信号可信度(high/medium/low)
- Cell：GPS 中心, 信号 P10/P50/P90, 活跃天数, 样本量, 所属 BS 分类标记

**可信度评级规则**：
- GPS 可信度：GPS 原始点≥10 且设备≥2 且无碰撞/动态=high, 4-9点或单设备=medium, 1-3点=low, 无GPS=none
- 信号可信度：原始信号占比>80%=high, 50-80%=medium, <50%=low

**验收标准**：
- 三级画像均正确生成
- 可信度评级逻辑正确
- baseline_version 正确记录

## Phase 2: 增量链路

### T2.1 批次注册与快照冻结

**目标**：增量批次开始前，注册批次并冻结当前 baseline 快照。

**产出**：
- 函数 `start_increment(source_name, window_start, window_end) -> (run_id, batch_id, frozen_baseline_version)`
- 本批所有准入和修正只参考冻结快照

---

### T2.2 Cell 匹配与四分流

**目标**：每条增量记录按 Cell 匹配对象库，分成四类。

**产出**：
- 函数 `cell_match_and_route(batch_id) -> (governed, candidate, issue, rejected)`
- 四类分流：
  1. 已知且健康 → 进入事实治理
  2. 已知但异常 → 治理+打标签
  3. 未知但合规 → 进入等待池
  4. 不合规 → 拒收
- 写入 `decision_log`

---

### T2.3 等待池管理与候选累计

**目标**：管理未知但合规的 Cell 候选，累计证据。

**产出**：
- 函数 `manage_waiting_pool(batch_id) -> (new_candidates, updated_candidates)`
- 已在等待池 → 更新累计指标（GPS 点数, 设备数, 活跃天数, 质心收敛度）
- 首次出现 → 新建等待记录
- 记录暂存到 `fact_waiting`

---

### T2.4 Cell 晋升与级联构建

**目标**：评估等待池中的 Cell 是否达标，执行级联晋升。

**产出**：
- 函数 `cascade_promote(batch_id) -> (promoted_cells, updated_bs, updated_lac)`
- Cell 晋升条件（初版建议，待验证调整）：
  - GPS 有效点数 ≥ 10
  - 设备数 ≥ 2
  - 质心 P90 半径 < 1500m（固定 Cell）或标记为移动 Cell
  - 最低时间跨度 ≥ 1 天
- 晋升执行：
  - Cell → `lifecycle_state=active`
  - 计算 Cell 质心
  - 回填 `fact_waiting` 中的暂存记录到 `cell_detail_hot` 和 `fact_governed`
- BS 级联：该 bs_id 下 trusted Cell ≥ 1 → 自动构建/更新 BS（聚合 Cell 质心取中位数）
- LAC 级联：该 LAC 下 trusted BS ≥ 1 → 自动注册/更新 LAC

**验收标准**：
- 级联关系正确（Cell→BS→LAC）
- 回填记录不重复
- 状态日志记录级联原因

---

### T2.5 增量事实治理

**目标**：对已在库 Cell 的增量记录执行 GPS 修正和信号补齐。

**产出**：
- 复用 T1.3 的 `govern_facts`，传入当前批次
- 异常对象的 GPS 修正按锚点资格矩阵降级

---

### T2.6 per-Cell 详细库滚动更新

**目标**：维护每个 Cell 的热明细库，封顶 N 条。

**产出**：
- 函数 `update_cell_detail_base(batch_id) -> updated_cell_count`
- 新记录写入 `cell_detail_hot`
- 超出封顶数的旧记录移除
- 低流量 Cell 保留全部，高流量 Cell 封顶在 N 条

**待确认**：N 的初始值（建议分析 Cell 记录数分布取 P95）

### T2.7 增量异常检测

**目标**：批后检测碰撞/动态/迁移/消失。

**产出**：
- 函数 `detect_anomaly_incremental(batch_id) -> anomaly_events`
- 检测项：
  - GPS 系统性偏移 → migration_suspect
  - Cell 质心法重新评估 → collision
  - Cell 消失（avg_report_interval × 2 未上报）→ 级联影响 BS
  - 动态 Cell/BS 检测
- 写入 `anomaly_event`
- 更新 `obj_cell`, `obj_bs` 的 health_state

---

### T2.8 画像增量刷新

**目标**：按策略触发画像刷新。

**产出**：
- 函数 `refresh_profile(trigger_condition) -> new_baseline_version`
- 从 `cell_detail_hot`（而非全量 `fact_governed`）计算画像
- 触发条件（可配置）：累积批次数达到阈值 / 手动触发
- 冻结新 `baseline_version`

---

### T2.9 增量编排

**目标**：串联 T2.1~T2.8 为完整的 2 小时增量流程。

**产出**：
- 函数 `run_increment(source_name, window_start, window_end) -> run_id`
- 按顺序执行：批次注册 → 标准化 → Cell 匹配+四分流 → 等待池管理 → Cell 晋升+级联 → 事实治理 → 详细库更新 → 异常检测 → （可选）画像刷新
- 结果写入 run_registry 和 decision_log

**验收标准**：
- 完整流程可一键执行
- 每批次的四分流统计正确
- 多批次连续执行结果稳定

## 附录：控制环节清单

以下是实现时必须显式做判断的控制点，每个都需要明确进入条件、退出条件和错判后果：

| # | 控制环节 | 控什么 |
|---|----------|--------|
| 1 | 标准化成功门 | 原始记录能否成为标准化事件 |
| 2 | 记录级合规门 | 事件是否允许进入对象判定链 |
| 3 | 已知对象识别门 | 事件命中 active / dormant / candidate / none |
| 4 | 候选入池门 | 未知但合规对象是否进入 waiting |
| 5 | waiting→observing 门 | 是否从纯累计进入可评估观察 |
| 6 | Cell 晋升门 | Cell 是否变成正式对象 |
| 7 | BS 级联构建门 | BS 是否存在、是否可作锚点 |
| 8 | LAC 级联构建门 | LAC 是否注册、是否激活 |
| 9 | GPS 修正前置门 | 该记录能否使用 BS 锚点修正 |
| 10 | 信号补齐前置门 | 能否补齐、用谁补齐 |
| 11 | governed/pending 分流门 | 记录进正式事实还是待判池 |
| 12 | per-Cell 写入门 | 哪些 governed facts 进入热详细库 |
| 13 | 画像刷新门 | 什么时候刷、刷多大范围 |
| 14 | 异常检测门 | drift/collision/dynamic/migration 的判定 |
| 15 | 降级/退役门 | active ↔ dormant → retired |
| 16 | 完整回归触发门 | 局部重算还是全量回归 |
