# rebuild3 项目背景与现状共识

## 0. 第三方阅读说明

本文会引用当前仓库中的文档路径、数据库表名和历史阶段名，用于说明结论来自哪里；但即使读者无法访问代码或数据库，也可以直接按下面的业务含义理解：

- `rebuild2`：上一轮“本地验证版”。它的重点是把历史数据跑通，验证字段解析、可信对象、GPS 修正、异常分类和画像是否成立。
- `rebuild3`：当前准备进入的“动态治理版”。它的重点不再是静态研究，而是把对象生命周期、增量批次、异常分流和基线刷新组织成可持续运行的系统。
- `legacy`：更早一轮历史研究产物，包括原始表和 `Y_codex_Layer*` 结果表，主要作用是做对照、回归和经验复用。
- `Layer/Step`：历史流水线的阶段命名方式。本文会引用这些名字说明历史成果来自哪里，但 rebuild3 不建议继续把它们当成长期主语。

如果文中出现表名如 `rebuild2.dim_lac_trusted`、`rebuild2.dwd_fact_enriched`，应把它理解为“当前已经存在的验证结果载体”，而不是阅读本文必须先掌握的代码细节。

## 1. 本轮任务理解

本轮不是继续补 SQL、API 或页面细节，而是先把 `rebuild3` 的重构主语从“Layer/Step 研究流水线”切换为“可持续运行的对象治理系统”。

重构目标不是推翻既有成果，而是把已经在 `rebuild2` 和 `lac_enbid_project` 中验证过的字段契约、可信对象规则、GPS 修正思路、异常分类和画像口径，整理成一套同时支持：

- 历史窗口初始化；
- 2 小时增量批处理；
- 后续本地回放验证；
- 未来上云迁移；

的统一骨架。

同时需要明确一个研究边界：

- 当前主要使用“北京、短窗口、本地可控”的历史样本做验证；
- 本轮重点是先把现有数据跑通，而不是一次定义全国长期运行的最终参数；
- 因此很多在 `rebuild2` 中看起来很严格的过滤和口径，本质上是“研究策略”，不能直接等同于全国长期规则。

## 2. 项目背景与业务目标

项目业务目标始终稳定：从运营商原始报文数据中，持续构建并维护可信的三级基础对象库与画像体系：

- `LAC`
- `BS`
- `Cell`

核心业务原则在以下文档中反复一致：

- `docs/data_warehouse/00_业务逻辑与设计原则.md`
- `docs/data_warehouse/02_项目现状与逻辑梳理_外部评审版.md`
- `docs/data_warehouse/最终整合意见文档.md`
- `rebuild2/docs/00_项目定义.md`

可以直接视为 rebuild3 的默认继承前提：

1. 有效 `cell_id` = 有效上报。
2. 修正优于丢弃。
3. 先建可信对象，再修正事实。
4. 基线驱动准入与日常治理。
5. 正向收敛 + 反向纯化 + 必要时完整回归。

但要额外补一条本轮已经明确的新共识：

6. 北京研究样本上的严格 GPS/LAC 过滤，只是帮助快速发现问题的研究约束，不是未来全国长期口径。

## 6. 当前 UI / DB / Agent 协作逻辑的基本形态

### 6.1 UI 逻辑

当前工作台定位清晰：它是“治理链路调试与验证工作台”，不是正式运营平台。

现有 UI 强项是：

- 以步骤为主线；
- 先预览规则和统计，再由用户确认执行；
- 能展示样本、漏斗、阈值、异常与画像。

这套模式在 rebuild3 里不应推翻，而应被重新定义为：

- 初始化视图；
- 规则实验视图；
- 批次回放视图；

### 6.2 DB 逻辑

当前数据库已经天然分成三层：

- `legacy`：历史 Layer/Step 与原始表；
- `rebuild2`：本轮冷启动与样本表；
- `rebuild2_meta`：字段、规则、缓存、构建结果。

如果不看数据库名字，只需要把这三层理解为：

- 历史研究和原始输入层；
- 本轮本地验证结果层；
- 规则、契约和缓存等元数据层。

这说明 rebuild3 没必要从零重建数据库协作方式，而应该：

- 在现有 PG17 上先补对象注册、批次、版本、状态转移和等待/观察表；
- 继续把旧表当成初始化输入和历史对照输入。

### 6.3 Agent 协作逻辑

当前仓库里已经形成了一种稳定协作方式：

1. Prompt 文档定义阶段目标。
2. AI/人类逐阶段确认规则。
3. 临时探索物放到 `lac_enbid_project/Agent_Workspace/`。
4. 稳定结论写回 `docs/` 或 Layer 文档。

rebuild3 应继续采用这种模式，但把输入文档收敛到更少、更稳定的“核心文档”，避免 prompt 过散。

## 7. rebuild2 与 rebuild3 的边界

### 7.1 rebuild2 的本质

rebuild2 的本质是：

- 面向静态数据库的本地研究和工作台验证；
- 核心目标是确认字段、规则、可信对象和异常分类；
- 重点在“把冷启动做对、把样本看清楚”。

它之所以在很多地方采用较严格的区域/GPS 过滤，是因为：

- 当前研究样本只覆盖北京；
- 可用时间窗口短；
- 本轮目标是让结果尽快贴近人的认知，以便快速发现规则问题。

这说明 rebuild2 的严格过滤应理解为“研究期收敛策略”，而不是长期运行的通用生产规则。

### 7.2 rebuild3 的本质

rebuild3 的本质是：

- 把 rebuild2 验证过的资产提升为“动态治理系统骨架”；
- 解决初始化与 2 小时增量共用逻辑的问题；
- 为对象等待、观察、晋升、降级、异常分流建立统一状态机；
- 在不破坏现有 UI / DB / Agent 协作的情况下，引入对象主档、治理后事实、批次与版本体系。

更具体地说，rebuild3 需要把处理顺序区分为两条链路：

- 冷启动初始化阶段，继续以 `LAC -> Cell -> BS` 为主链；
- 日常动态阶段，改成 `Cell` 先判断、`BS` 再传播、`LAC` 做边界和区域约束。

### 7.3 不该在 rebuild3 首轮做的事

以下内容应明确后置：

- 正式云端大平台设计；
- 一次性重写全部前后端 UI / API；
- 全国化参数体系；
- 企业级元数据平台；
- 复杂事件流或消息总线；
- 所有异常类型的二轮拆分算法一次做完。
- `Cell` 消失/退役精确时间参数的一次性冻结。

## 8. 本轮重构的目标、非目标与约束

### 8.1 目标

1. 冻结 rebuild3 的逻辑主框架。
2. 定义初始化与 2 小时增量共用的治理骨架。
3. 明确对象状态机、等待池、观察池、晋升和异常分流。
4. 明确哪些现有资产可以直接复用，哪些要改造成动态状态。
5. 为后续 AI/工程实现准备稳定输入文档。
6. 先完成“3 天起跑，再滚到 7 天，再对比直接 7 天”的本地验证路径定义。

### 8.2 非目标

1. 现在不写大规模实现代码。
2. 现在不展开字段级 / SQL 级 / API 级全设计。
3. 现在不把问题扩写成正式云端数仓或企业平台蓝图。
4. 现在不把北京研究窗口里的严格过滤参数直接冻结成全国长期参数。

### 8.3 关键约束

1. 保留现有业务哲学。
2. 尽量不破坏当前 UI、数据库、Agent 协作模式。
3. 当前先在本地 PG17 验证。
4. 未来要能迁移到云端日常运行。
5. 本地验证应允许“先少天数跑通，再补齐到完整窗口做回归对比”。

## 10. 当前共识结论

当前最值得冻结的共识不是“再造一套新系统”，而是：

1. rebuild2 已经把静态研究、冷启动逻辑和本地工作台打到了可继承状态。
2. rebuild3 需要新增的不是更多研究表，而是对象状态、批次治理、版本绑定和异常旁路。
3. 当前 UI / DB / Agent 体系可以承接 rebuild3，但主语必须从“Step 表”升级为“对象 + 决策 + 事实 + 基线”。
4. 旧 Layer/Step 结果应继续保留，作为初始化输入、规则对照和回归参照，而不应继续充当长期系统主语。
5. 冷启动初始化仍沿用 `LAC -> Cell -> BS` 的研究主链，但动态治理应切换为 `Cell` 为最小主语、`BS` 为锚点、`LAC` 为边界。
6. 北京严格 GPS/LAC 过滤只是研究期收敛策略，不应直接写成全国长期规则。
7. 当前最合理的本地验证路径是“3 天起跑 -> 滚到 7 天 -> 对比直接 7 天”，而不是先把所有长期参数一次定死。
