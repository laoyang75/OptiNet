## 一、核心技术栈

| 组件 | 技术 | 版本 | 说明 |
|------|------|------|------|
| **数据库** | PostgreSQL | 17 | 本地服务器，已运行 |
| **后端** | Python + FastAPI | Python 3.11+, FastAPI latest | REST API 服务 |
| **数据库驱动** | asyncpg | latest | 异步 PostgreSQL 驱动 |
| **前端** | Vue 3 + TypeScript + Vite | Vue 3.4+, TS 5+, Vite 5+ | 组件化开发，解决历史页面臃肿问题 |
| **SQL 执行** | 大表操作通过 psql 直接执行 | — | 千万级行操作不走 API |

### 1.1 不引入的技术

以下技术**不在** rebuild3 首轮范围内：

- ❌ ORM（SQLAlchemy / Tortoise）—— 直接写 SQL
- ❌ 消息队列（Kafka / RabbitMQ）
- ❌ 任务调度框架（Celery / Airflow）
- ❌ 容器化（Docker / K8s）
- ❌ SSR 框架（Nuxt）

### 1.2 与 rebuild2 的关系

**后端**：继承 rebuild2 的模式，不做改变：
- FastAPI 路由组织方式（按功能域分 router）
- asyncpg 连接池管理
- 大表统计结果预算到缓存表，前端读缓存

**前端**：**不再继承** rebuild2 的原生 HTML + ES Modules 方式。
- rebuild2 的前端页面已经变得非常庞大，单文件难以维护
- rebuild3 前端用 Vue 3 + TypeScript **全部重写**，复用的是**业务逻辑和交互设计**，不是代码
- rebuild2 的旧页面保持原样可访问（独立入口），不做迁移

## 二、数据库规范

### 2.1 Schema 组织

| Schema | 用途 | 操作权限 |
|--------|------|----------|
| `rebuild3` | rebuild3 所有新表 | rebuild3 代码读写 |
| `rebuild2` | rebuild2 历史验证结果 | rebuild3 只读（作为初始化输入和对照） |
| `rebuild2_meta` | rebuild2 元数据 | rebuild3 只读（复用字段契约和规则定义） |
| `legacy` | 更早的历史研究结果 | rebuild3 只读 |

**原则**：rebuild3 代码**不修改** rebuild2 / rebuild2_meta / legacy 中的任何表。

### 2.2 命名规范

| 类型 | 命名规则 | 示例 |
|------|----------|------|
| 表名 | 小写下划线，按功能前缀分组 | `obj_cell`, `fact_governed`, `profile_bs` |
| 列名 | 小写下划线 | `lifecycle_state`, `center_lon` |
| 主键 | 业务主键优先，不用自增 ID（除非表间无业务关联） | `(operator_code, tech_norm, lac, cell_id)` |
| 索引 | `idx_{table}_{columns}` | `idx_obj_cell_bs_id` |
| ENUM 类型 | `{domain}_{name}_enum` | `lifecycle_state_enum` |
| 分区 | 按 `batch_id` 分区（仅大表） | `fact_standardized` 按 batch_id range 分区 |

### 2.3 ENUM 类型定义

```sql
-- 生命周期状态
CREATE TYPE rebuild3.lifecycle_state_enum AS ENUM (
    'waiting',      -- 合规但样本不足
    'observing',    -- 已具备最小证据，等待稳定性判断
    'active',       -- 正式对象
    'watch',        -- 正式对象，近期需观察
    'dormant',      -- 长期无数据，休眠中
    'retired',      -- 退役
    'rejected'      -- 明确拒收
);

-- 健康状态
CREATE TYPE rebuild3.health_state_enum AS ENUM (
    'healthy',              -- 健康
    'insufficient',         -- 证据不足
    'gps_bias',             -- GPS 偏差
    'collision_suspect',    -- 疑似碰撞
    'collision_confirmed',  -- 确认碰撞
    'dynamic',              -- 动态对象（移动基站）
    'migration_suspect'     -- 疑似迁移
);

-- BS 异常分类（来自 Cell 质心法 v2）
CREATE TYPE rebuild3.bs_anomaly_class_enum AS ENUM (
    'normal',               -- 正常 BS（未进入异常候选）
    'normal_spread',        -- 异常候选但 GPS 噪声回归正常
    'single_large',         -- 面积大或低精度
    'dynamic_bs',           -- 动态 BS（全部 Cell 都是移动的）
    'collision_confirmed',  -- 确认碰撞
    'collision_suspected',  -- 疑似碰撞
    'collision_uncertain'   -- 不确定碰撞
);

-- 运行类型
CREATE TYPE rebuild3.run_type_enum AS ENUM (
    'init_bootstrap',       -- 冷启动初始化
    'increment_2h',         -- 2小时增量
    'historical_replay',    -- 历史回放
    'partial_recalc',       -- 局部重算
    'full_regression',      -- 完整回归
    'profile_refresh'       -- 画像刷新
);

-- GPS 可信度
CREATE TYPE rebuild3.gps_confidence_enum AS ENUM (
    'high', 'medium', 'low', 'none'
);

-- 信号可信度
CREATE TYPE rebuild3.signal_confidence_enum AS ENUM (
    'high', 'medium', 'low'
);

-- GPS 来源
CREATE TYPE rebuild3.gps_source_enum AS ENUM (
    'original', 'bs_center', 'cell_center'
);

-- 信号来源
CREATE TYPE rebuild3.signal_source_enum AS ENUM (
    'original', 'cell_fill', 'bs_fill', 'unfilled'
);
```

### 2.4 大表性能要求

| 表 | 预计行数级 | 分区策略 | 索引要求 |
|----|-----------|----------|----------|
| `fact_standardized` | 千万级 | 按 batch_id range | (batch_id, record_id, cell_seq) 主键 |
| `fact_governed` | 千万级 | 按 batch_id range | 主键 + (operator_code, tech_norm, lac, cell_id) |
| `cell_detail_hot` | 百万级 | 无分区（热库，定期淘汰） | (operator_code, tech_norm, lac, cell_id, event_time) |
| `obj_cell` | 十万级 | 无分区 | 主键 + (bs_id) + (lifecycle_state) |
| `obj_bs` | 十万级 | 无分区 | 主键 + (lifecycle_state, anchorable) |
| `fact_waiting` | 十万~百万级 | 无分区 | (operator_code, tech_norm, lac, cell_id) |

### 2.5 SQL 编写规范

- **千万级行操作**：通过 `psql` 命令行或后端的批量 SQL 执行，不走 API 接口
- **JOIN 优化**：大表 JOIN 前先确认索引覆盖
- **批量写入**：使用 `COPY` 或 `INSERT ... SELECT` 而非逐行 INSERT
- **幂等写入**：使用 `INSERT ... ON CONFLICT DO NOTHING`
- **事务**：单个批次的处理在一个事务内完成
- **EXPLAIN ANALYZE**：对新写的大表查询必须检查执行计划

### 3.2 API 设计规范

- **RESTful** 风格
- **路径命名**：`/api/v1/{domain}/{action}`
- **返回格式**：统一 JSON，包含 `status`, `data`, `message`
- **分页**：大列表查询必须支持分页（`offset` + `limit`）
- **错误处理**：HTTP 状态码 + 业务错误码

示例路由：

```
POST   /api/v1/run/init          # 触发初始化
POST   /api/v1/run/increment     # 触发增量
GET    /api/v1/run/{run_id}      # 查询运行详情
GET    /api/v1/batch/list        # 批次列表

GET    /api/v1/object/cell       # Cell 对象列表（支持状态筛选）
GET    /api/v1/object/bs         # BS 对象列表
GET    /api/v1/object/lac        # LAC 对象列表
GET    /api/v1/object/{type}/{key}  # 对象详情

GET    /api/v1/profile/bs/{key}  # BS 画像详情
GET    /api/v1/profile/baseline  # 当前基线版本

GET    /api/v1/waiting/list      # 等待池列表
GET    /api/v1/decision/{batch_id}  # 批次判定结果

```

### 3.4 日志与监控

- 使用 Python `logging` 模块
- 每个批次处理记录：开始时间、结束时间、各步骤行数、耗时
- 错误日志包含完整的上下文（batch_id, run_id, 错误位置）
- 大表操作前后记录行数变化

### 3.5 Python 编码规范

- 类型注解：所有函数签名使用类型注解
- 异步：数据库操作使用 async/await
- SQL 参数化：所有 SQL 使用参数化查询，禁止字符串拼接
- 模块边界：core/ 下的模块不依赖 api/，api/ 调用 core/
- 错误处理：业务错误抛出自定义异常，API 层统一捕获转 HTTP 响应

### 4.3 前端编码规范

**Vue 组件**：
- 使用 `<script setup lang="ts">` + Composition API（不用 Options API）
- 组件文件名 PascalCase（`RunCenter.vue`）
- Props 用 `defineProps<T>()` 带类型
- Emit 用 `defineEmits<T>()`
- 模板中禁止复杂逻辑，抽到 computed 或 composable

**TypeScript**：
- 严格模式（`strict: true`）
- API 响应必须定义类型（不用 `any`）
- 业务对象类型集中在 `types/` 下
- 枚举值与后端 ENUM 保持一致

**样式**：
- 组件内用 `<style scoped>` 隔离
- 全局变量用 CSS Custom Properties（`--color-primary` 等）
- 响应式布局：桌面优先（本地开发工具，不需要移动端）
- 状态颜色统一：active=绿, waiting=黄, anomaly=橙, dormant=灰, retired=淡灰, rejected=红

**API 调用**：
- 统一通过 `api/client.ts` 封装（baseURL、超时、错误拦截）
- 返回类型化的 Promise
- 大列表必须分页（composable 封装分页逻辑）

### 4.4 与 rebuild2 前端的关系

- rebuild3 前端是**完全独立的 SPA**，独立端口运行
- **不修改、不迁移** rebuild2 的原生 HTML 页面
- rebuild2 旧页面保持原样可访问（供回溯参考）
- 复用的是 rebuild2 的**业务逻辑和交互设计思路**，不是代码

## 五、测试规范

### 5.1 测试层次

| 层次 | 范围 | 工具 |
|------|------|------|
| **SQL 验证** | DDL 可执行、索引生效、ENUM 正确 | psql 脚本 |
| **单元测试** | core/ 下每个模块的核心函数 | pytest |
| **集成测试** | 初始化/增量完整流程 | pytest + 测试数据集 |
| **收敛性验证** | 3天初始化+4天增量 vs 7天初始化 | 对比脚本 |

### 5.2 测试数据

- 使用 rebuild2 现有的标准化结果作为参照
- 可从 rebuild2 表中抽样构建小规模测试集（如 6 个 LAC 的子集）
- 先用小数据集验证逻辑正确性，再跑全量

### 5.3 验收检查清单

每个模块交付时必须满足：

- [ ] SQL 可在 PG17 上正常执行
- [ ] 函数有类型注解
- [ ] 参数化配置（无硬编码阈值）
- [ ] 幂等写入（重复执行不报错、不重复数据）
- [ ] 日志输出（行数、耗时）
- [ ] 与 rebuild2 结果可对比（适用时）

## 六、协作规范

### 6.1 Agent 协作模式

延续 rebuild2 的协作方式：

1. **Prompt 驱动**：每个开发任务有明确的输入/输出/验收标准
2. **先样本后全量**：大表操作先用小数据集验证，确认后全量执行
3. **UI 审核确认**：每步结果在 UI 上展示，人工确认后再进入下一步
4. **预算缓存**：千万级表的统计结果预算到缓存表供 UI 读取

### 6.2 代码组织原则

- 每个 core/ 模块对应一个开发任务
- 模块间通过函数调用（不用消息/事件）
- 编排逻辑集中在 orchestrator.py
- 配置集中在 config.py

### 6.3 文档约定

- 重要决策记录在代码注释中（为什么这样做，而不只是做了什么）
- 阈值参数旁标注来源（如 `# 来自 rebuild2 验证：4G 城市 GPS 修正阈值`）
- 复用的规则标注来源文档路径

## 七、性能基准

| 操作 | 预期数据量 | 目标耗时 |
|------|-----------|----------|
| 单批次标准化（2小时数据） | ~50 万行 | < 5 分钟 |
| 单批次 Cell 匹配+四分流 | ~50 万行 | < 2 分钟 |
| 单批次事实治理 | ~50 万行 | < 5 分钟 |
| 画像全量刷新 | ~57 万 Cell | < 10 分钟 |
| 初始化全流程（3 天数据） | ~2400 万行 | < 2 小时 |

以上为估算基准，实际以 PG17 本地服务器性能为准。
