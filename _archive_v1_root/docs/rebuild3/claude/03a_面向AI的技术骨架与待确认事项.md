## 一、核心对象族与建议表族

### 1.1 核心对象族

| 对象族 | 说明 | 主键模式 |
|--------|------|----------|
| **LAC** | 位置区，最大粒度的网络分区 | (operator_code, tech_norm, lac) |
| **BS** | 基站，由 cell_id 推算（4G: cell_id/256, 5G: cell_id/4096） | (operator_code, tech_norm, lac, bs_id) |
| **Cell** | 小区，最小粒度，一个基站有多个扇区 | (operator_code, tech_norm, lac, cell_id) |
| **等待对象** | 尚未达标入库的候选对象（合规但样本不足） | 同上，附加 first_seen / sample_count |
| **治理后事实** | GPS 修正 + 信号补齐后的明细记录 | (batch_id, record_id, cell_seq) |
| **画像/基线** | LAC/BS/Cell 三级聚合画像，参与增量数据的准入判定 | 同对象主键 + baseline_version |
| **异常/问题** | 碰撞、动态、漂移、迁移等异常事件 | (event_id) |

### 1.2 建议表族

#### A. 接入层（原始数据标准化）

| 表名建议 | 说明 |
|----------|------|
| `batch_registry` | 批次注册表：batch_id, 时间窗口(起止时间), 来源(源表标识), 原始行数, 标准化行数, run_id |
| `fact_standardized` | 标准化明细：每条原始记录经 cell_infos/ss1 解析 + 26 条 ODS 清洗后的标准化记录，按 batch_id 分区。约 60 个字段，包含网络标识(cell_id/lac/bs_id/运营商/制式)、GPS(经纬度)、信号(RSRP/RSRQ/SINR/Dbm 等)、合规标记等 |
| `source_adapter` | 源表适配配置：源表名→标准化字段映射。当前直连固定的两张原始表，未来换数据源时只改此表不改代码 |

#### B. 对象层（Cell 为核心，BS/LAC 由 Cell 聚合派生）

| 表名建议 | 说明 |
|----------|------|
| `obj_cell` | **核心对象表**。Cell 注册表：主键(operator_code, tech_norm, lac, cell_id) + 状态(pending/waiting/trusted/anomaly/dormant/retired) + 质心经纬度 + 空间跨度 + 是否移动 Cell + 归属 bs_id + GPS 点数 + 设备数 + 活跃天数 + 首次出现时间 + 最后活跃时间 + avg_report_interval（平均上报间隔，用于消失检测） |
| `obj_bs` | BS 注册表（由 Cell 聚合派生）：主键 + 状态 + GPS 中心点(从下属 trusted Cell 质心取中位数) + 覆盖半径(P50/P90) + 异常分类标记 + 可信度等级 + 下属 trusted Cell 数 |
| `obj_lac` | LAC 注册表（由 BS 聚合派生）：主键 + 状态 + 下属 trusted BS 数 + 创建版本 + 最后活跃时间 |
| `obj_waiting_pool` | Cell 等待池：Cell 主键 + 首次出现时间 + 累计 GPS 点数 + 累计设备数 + 累计活跃天数 + 质心收敛度(P90 半径) + 暂存记录计数。**晋升条件基于数据质量（GPS 点数/设备数/质心收敛度），不是固定时间窗口** |
| `obj_state_log` | 对象状态变更日志：对象类型(cell/bs/lac) + 对象主键 + 旧状态 + 新状态 + 变更时间 + 触发原因(如 "cell_promotion_cascade") + run_id |

#### C. 事实层（per-Cell 详细库 + 治理后明细）

| 表名建议 | 说明 |
|----------|------|
| `cell_detail_base` | **per-Cell 详细库（核心新增）**：每个 Cell 保留最近 N 条治理后明细记录，滚动更新。低流量 Cell 保留全部（总量小），高流量 Cell 封顶在 N 条（如 1000 条）。画像从此表计算（而非从全量 fact_governed 聚合），确保画像始终基于最新数据 |
| `fact_governed` | 治理后全量明细：GPS 修正后的经纬度 + GPS 来源标记(original/cell_center/bs_center) + 信号补齐后的值 + 信号来源标记(original/cell_fill/unfilled) + 命中规则标签 + 版本信息。只有 trusted 和 anomaly 状态 Cell 的记录才写入 |
| `fact_waiting` | 等待池明细暂存：waiting 状态 Cell 的原始标准化记录暂存于此，Cell 晋升后回填到 cell_detail_base 和 fact_governed |

#### D. 画像层（三级基线画像）

| 表名建议 | 说明 |
|----------|------|
| `profile_lac` | LAC 画像：覆盖范围(中心经纬度+半径)、BS 数/Cell 数、信号聚合(RSRP P10/P50/P90)、活跃天数、异常 BS 占比、GPS 来源比例 |
| `profile_bs` | BS 画像：精算中心点、覆盖半径(P50/P90)、下属 Cell 数/设备数、信号聚合、异常分类标记、GPS 可信度(high/medium/low/none)、信号可信度(high/medium/low) |
| `profile_cell` | Cell 画像：GPS 中心、信号分布(RSRP P10/P50/P90)、活跃天数、样本量、所属 BS 的分类标记 |
| `baseline_version` | 基线版本记录：版本号, 使用的时间窗口, 刷新时间, 绑定的 rule_set_version, 对象数统计 |

#### E. 运行与版本（追溯与审计）

| 表名建议 | 说明 |
|----------|------|
| `run_registry` | 运行记录：run_id, 运行类型(init/incremental/regression/profile_refresh), 关联 batch_id, 绑定的三个版本号, 开始/结束时间, 状态(running/completed/failed), 输入/输出行数, 运行参数快照 |
| `version_binding` | 版本绑定关系：run_id → contract_version(字段解析口径) + rule_set_version(治理规则口径) + baseline_version(画像基线口径) |

#### F. 异常与判定（运营）

| 表名建议 | 说明 |
|----------|------|
| `anomaly_event` | 异常事件：event_id, 类型(collision/dynamic/migration/gps_drift/data_insufficient), 关联对象主键, 发现批次, 严重度, 状态(open/investigating/resolved) |
| `decision_log` | 判定日志：每批次的四分流结果统计——范围内多少条、漂移多少条、新增多少条、拒收多少条 |

### 2.1 共用模块

│  standardize(batch_id, source_config)                │

### 2.2 初始化独有模块

| 模块 | 说明 |
|------|------|
| `build_trusted_library()` | 从全量标准化数据构建初始可信对象库。按 LAC→Cell→BS 顺序逐级筛选：运营商白名单过滤→LAC 编码/区域/稳定性过滤→在可信 LAC 内聚合 Cell→BS 中位数中心点计算 |
| `classify_bs_anomaly()` | BS 异常分类，执行 Cell 质心法 v2 算法（5 步：提取 GPS+CellID → Cell 级网格聚合 → Cell 质心+空间跨度 → BS 级固定 Cell 质心对比 → 设备分裂验证），产出 6 种分类 |
| `full_regression()` | 完整回归：用纯化后的对象库（含异常分类标记），从标准化记录重跑 govern_facts，确保全量记录基于同一套规则处理 |
| `freeze_baseline()` | 冻结初始基线版本号，所有入库对象标记为 trusted，系统转入增量模式 |

### 2.3 增量独有模块（Cell 先行）

| 模块 | 说明 |
|------|------|
| `cell_match_and_route()` | Cell 级匹配与分流：每条记录按 cell_id 匹配对象库 → Cell 在库(trusted/anomaly)则进入事实治理 → Cell 不在库则进入等待池。记录级合规检查（运营商白名单+CellID/LAC 格式）在此步前置完成 |
| `manage_waiting_pool()` | Cell 等待池管理：累计 GPS 点数/设备数/活跃天数/质心收敛度。**晋升条件基于数据质量，非固定时间窗口**——高流量 Cell 可能 1 天晋升，低流量 Cell 可能 90 天 |
| `cascade_promote()` | **级联晋升（核心新模块）**：Cell 达标 → Cell 晋升 trusted → 自动构建/更新所属 BS（聚合 trusted Cell 质心为 BS 中心点）→ 自动构建/更新所属 LAC → 回填等待池暂存记录到 cell_detail_base 和 fact_governed |
| `update_cell_detail_base()` | per-Cell 详细库滚动更新：新记录写入 → 超出封顶数的旧记录移除 |
| `detect_anomaly_incremental()` | 增量异常检测：GPS 系统性偏移→迁移；Cell 质心法重新评估→碰撞；Cell 消失检测（avg_report_interval × 2 未上报）→ 级联影响 BS |

## 三、各对象最小状态机

### 3.1 状态枚举

| 状态 | 含义 |
|------|------|
| `pending` | 刚发现，未通过基本合规检查（运营商不在白名单、CellID/LAC 格式非法等） |
| `waiting` | 合规但样本不足，正在累计中（等待活跃天数、上报量、GPS 分布达标） |
| `trusted` | 可信，正常参与事实治理（GPS 修正和信号补齐） |
| `anomaly` | 异常观察中（含子类型：collision_confirmed/collision_suspected/dynamic/gps_drift/migration/data_insufficient） |
| `dormant` | 长期无数据，休眠中（连续 14+ 天无上报） |
| `retired` | 退役（确认永久消失，或所属上级对象退役） |

### 3.2 状态转移矩阵

| 当前状态 | 触发条件 | 目标状态 | 动作 |
|----------|---------|----------|------|
| — | 首次发现，运营商/CellID/LAC 不合规 | pending | 记录但不入等待池 |
| — | 首次发现，合规 | waiting | 入等待池，开始累计样本 |
| pending | 后续批次中发现合规数据 | waiting | 入等待池 |
| waiting | 累计达标（天数+量+分布均满足晋升条件） | trusted | 写入对象库，回填暂存记录到事实层 |
| waiting | 超时未达标（如 30 天仍不满足晋升条件） | dormant | 标记休眠 |
| trusted | 正常运行中，持续有数据 | trusted | 更新最后活跃时间和统计 |
| trusted | 检测到碰撞/动态/系统性偏移 | anomaly | 标记异常子类型，不删除对象 |
| trusted | 连续 14 天无上报 | dormant | 标记休眠 |
| anomaly | 异常消解（如碰撞 BS 完成虚拟拆分） | trusted | 恢复正常 |
| anomaly | 确认永久异常（如碰撞无法拆分） | retired | 退役 |
| dormant | 新数据到达 | waiting | 重新评估（需重新累计达标） |
| dormant | 超时（如 90 天无任何数据） | retired | 退役 |

### 3.3 异常子类型

| 子类型 | 说明 | 处理方式 |
|--------|------|----------|
| `collision_confirmed` | 编码碰撞已确认（不同物理基站共用同一 BS ID，设备交叉率 < 5%） | 虚拟拆分：按 Cell 质心空间聚类分组，每组视为独立 BS，分别计算画像 |
| `collision_suspected` | 疑似碰撞（设备交叉率 5-20%） | 继续观察积累数据，或人工确认 |
| `dynamic` | 移动基站（高铁/车载场景，BS 下所有 Cell 空间跨度 > 1500m） | 画像标记低精度，GPS 修正降级 |
| `gps_drift` | GPS 系统性偏移（新数据的 GPS 中心持续偏离基线画像） | 调查是临时偏移还是永久迁移 |
| `migration` | 物理位置永久迁移 | 更新画像或分裂为新对象 |
| `data_insufficient` | 数据量骤降（如曾经活跃的对象突然长期无上报） | 继续观察 |

## 四、判定矩阵

### 4.1 增量四分流判定矩阵

每条增量记录按以下条件判定分流方向：

| 条件组合 | 判定 | 动作 |
|----------|------|------|
| Cell 在对象库（trusted 状态）+ GPS 在该 BS 画像的 P90 覆盖范围内 + 信号在画像 P10~P90 区间 | ① 范围内 | 正常治理（GPS 修正+信号补齐），写入 fact_governed |
| Cell 在对象库 + GPS 偏离 BS 中心 > 修正阈值（4G:1000m, 5G:500m）| ② GPS 漂移 | 治理但打 gps_drift 标签，记录漂移事件 |
| Cell 在对象库 + 信号偏离画像 > 2 个标准差 | ② 信号漂移 | 治理但打 signal_drift 标签 |
| Cell 不在对象库 + LAC 在库 + 运营商/CellID/LAC 合规 | ③ 新增 | 入等待池（已有则累计，首次则新建） |
| Cell 不在对象库 + LAC 也不在库 + 合规 | ③ 新增 | LAC 和 Cell 都入等待池 |
| 运营商不在白名单 / CellID 不合法 / LAC 不合法 | ④ 严重异常 | 拒收，写异常日志 |
| Cell 在对象库但所属 BS 为 collision_confirmed 状态 | ② 标记 | 治理但打 collision 标签（GPS 修正降级） |

### 4.2 晋升判定矩阵（Cell 驱动级联）

晋升是自底向上级联的：Cell 晋升 → 触发 BS 构建/更新 → 触发 LAC 构建/更新。

**Cell 晋升条件**（基于数据质量，非固定时间窗口）：

| 指标 | 条件（初版建议，需通过 3+4 天验证调整） | 说明 |
|------|----------------------------------------|------|
| GPS 有效点数 | ≥ 10 | 确保质心计算有统计意义 |
| 设备数 | ≥ 2 | 排除单设备噪声 |
| 质心收敛度 | P90 半径 < 1500m（固定 Cell）或标记为移动 Cell | 确保 GPS 点不过度离散 |
| 最低时间跨度 | ≥ 1 天（至少跨越 1 个自然日的数据） | 防止瞬时突发直接入库 |
| 记录级合规 | 运营商在白名单 + CellID 在合法范围 | 前置已检查 |

**BS 级联条件**：

| 触发 | 条件 | 动作 |
|------|------|------|
| 其下某 Cell 晋升为 trusted | 该 bs_id 下 trusted Cell ≥ 1 | 自动构建 BS：聚合 trusted Cell 质心取中位数 → BS 中心点，计算 P50/P90 |
| 已有 BS 新增 trusted Cell | — | 更新 BS 中心点和覆盖半径 |

**LAC 级联条件**：

| 触发 | 条件 | 动作 |
|------|------|------|
| 其下某 BS 注册/更新 | 该 LAC 下 trusted BS ≥ 1 | 自动注册/更新 LAC，标记 trusted |

**关键区别**：前一阶段的 LAC→Cell→BS 是"自顶向下的门控"（LAC 不可信则 Cell 不分析）。rebuild3 增量模式是"自底向上的级联"（Cell 达标就晋升，BS 和 LAC 自动派生）。LAC 不再是前置门槛。

### 4.3 降级/退役/消失判定矩阵

| 对象类型 | 疑似消失条件 | 降级条件（→ dormant） | 退役条件（→ retired） |
|----------|-------------|----------------------|----------------------|
| Cell | 当前时间 - last_seen > avg_report_interval × 2 | 疑似消失且确认无新数据 | dormant 超时（参数，如 90 天） |
| BS | 下属所有 Cell 均疑似消失 | 级联降级 | 全部下属 Cell 均已 retired |
| LAC | 下属所有 BS 均 dormant | 级联降级 | 全部下属 BS 均已 retired |

**注意**：消失检测是动态阈值（基于每个 Cell 自身的上报频率），不是固定天数。高流量 Cell 3 天没数据就异常，低流量 Cell 30 天没数据可能正常。具体参数在本地验证阶段不需要太纠结，留待真实增量数据时调优。

| 画像可信度标注 | GPS 可信度：GPS 原始点≥10 且设备≥2 且无碰撞/动态标记=high, GPS 点 4-9 或设备=1 或有 single_large=medium, GPS 点 1-3=low, 无 GPS=none。信号可信度：原始信号占比>80%=high, 50-80%=medium, <50%=low |

### 5.2 需调整后复用

| 规则 | 前一阶段的形态 | rebuild3 的调整方向 |
|------|----------------|-------------------|
| 对象构建顺序 | LAC→Cell→BS（自顶向下门控） | 混合策略：初始化沿用 LAC→Cell→BS（批量高效），增量改为 Cell→BS→LAC 级联晋升（自底向上） |

## 六、补齐、画像、可信度、异常分类之间的依赖关系

```
对象状态 (obj_*)
    │
    ├── 决定是否允许 GPS 修正
    │   trusted → 允许（用该 BS 的中位数中心点修正）
    │   anomaly/collision → 降级修正（碰撞 BS 的中心点是两个物理位置的折中，不可靠）
    │   waiting → 不修正（无可信锚点）
    │
    ├── 决定是否允许信号补齐
    │   trusted → 允许（同 Cell 时间最近 → 同 BS 最大 Cell 回退）
    │   waiting → 不补齐
    │
    └── 影响画像可信度评级
        trusted + GPS 点充足 → high
        anomaly / 数据稀疏 → low
        waiting → none

画像 (profile_*)
    │
    ├── 参与增量准入判定
    │   "新记录的 GPS 在该 BS 画像的 P90 覆盖范围内吗？" → 范围内
    │   "GPS 偏离画像中心多远？" → 漂移判定
    │
    ├── 参与异常检测
    │   "新一批数据的 GPS 中心与基线画像中心偏移多大？" → 迁移疑似
    │
    └── 提供补齐锚点
        BS 画像中的精算中心点 → GPS 修正的参照系

异常分类（6 种分类标记）
    │
    ├── 影响对象状态
    │   collision_confirmed → 对象标记为 anomaly
    │   dynamic_bs → 对象标记为 anomaly
    │
    ├── 影响 GPS 修正策略
    │   collision → 不用混合中心点（它是两个位置的折中）
    │   dynamic → 标记低精度（位置在变）
    │
    └── 影响画像可信度评级
        collision → gps_confidence = low
        dynamic → gps_confidence = medium
        normal_spread → 回归正常（gps_confidence 不降级）
```

**核心依赖链**：
```
对象状态 → 决定补齐策略 → 影响事实质量 → 影响画像质量 → 影响准入判定 → 影响对象状态
```

这是一个闭环。初始化时通过全量聚合打破循环（先全量计算再回归）；增量时通过画像基线作为稳定锚点维持收敛（画像不每批更新，避免引入抖动）。

## 七、run / version / batch / event_time / idempotency 最小建议口径

### 7.1 核心标识

| 标识 | 语义 | 生成规则 | 示例 |
|------|------|----------|------|
| `run_id` | 一次实际运行实例（初始化/增量/回归/画像刷新） | UUID 或递增序列 | `run_20260403_001` |
| `batch_id` | 一个数据批次（2 小时时间窗口） | 时间窗口标识 | `batch_20260403_08`（8:00~10:00） |
| `contract_version` | 字段解析与输出契约口径（27→60 列映射+清洗规则的版本） | 语义版本号 | `v1.0` |
| `rule_set_version` | 正式治理规则集合口径（GPS 修正阈值+补齐策略+碰撞检测算法的版本） | 语义版本号 | `v1.0` |
| `baseline_version` | 对象画像与基线参照口径（画像聚合的数据窗口和规则版本） | 语义版本号 | `v1.0` |

### 7.2 绑定关系

每次 run 必须绑定完整版本上下文：

```
run_registry:
  run_id              -- 唯一标识
  run_type            -- 'init'(初始化) / 'incremental'(增量) /
                      -- 'regression'(完整回归) / 'profile_refresh'(画像刷新)
  batch_id            -- 增量运行关联的批次（初始化为 NULL）
  contract_version    -- 本次运行使用的字段解析口径
  rule_set_version    -- 本次运行使用的治理规则口径
  baseline_version    -- 本次运行参照的画像基线版本（初始化为 NULL）
  started_at          -- 开始时间
  finished_at         -- 结束时间
  status              -- 'running' / 'completed' / 'failed'
  input_rows          -- 输入行数
  output_rows         -- 输出行数
  parameters          -- 运行参数快照（JSON，如阈值配置）
```

### 7.3 事件时间口径

| 时间 | 含义 | 来源 |
|------|------|------|
| `event_time` | 原始上报时间（业务时间） | 原始记录的上报时间戳标准化后 |
| `process_time` | 系统处理时间 | 运行时的当前系统时间 |
| `batch_window_start/end` | 批次时间窗口起止 | 批次注册时确定 |

**关键约定**：所有业务判断（活跃天数计算、"时间最近"补齐、画像时间特征等）以 `event_time` 为准，不用 `process_time`。

### 7.4 幂等性

| 操作 | 幂等键 | 策略 |
|------|--------|------|
| 标准化写入 | (batch_id, record_id, cell_seq) | INSERT ON CONFLICT DO NOTHING（同一批次同一记录不重复写入） |
| 事实治理写入 | (batch_id, record_id, cell_seq) | 同上 |
| 对象状态更新 | (对象主键, run_id) | 记录状态变更日志，不覆盖（保留完整变更历史） |
| 画像刷新 | (对象主键, baseline_version) | INSERT OVERWRITE 同版本（同版本可重算覆盖） |

## 八、需要预留但暂不展开的接口边界

### 8.1 源表适配接口

- **当前**：直连两张固定的原始表（GPS 表 + LAC 表，各 27 列）
- **未来**：数据源可能变化（不同表名、不同字段名、甚至不同格式）
- **预留**：设计 source_adapter 配置表，将源表结构映射到标准化字段，换数据源时只改配置
- **暂不展开**：具体适配器实现代码

### 8.2 画像刷新接口

- **当前**：手动触发全量画像刷新
- **未来**：按周/月自动刷新 + 支持局部刷新（只刷新指定 LAC 或 BS 范围）
- **预留**：build_profile 接受 scope 参数（全量/指定范围）
- **暂不展开**：自动调度逻辑和增量画像更新算法

### 8.3 局部重算接口

- **当前**：手动操作
- **未来**：规则变更时自动评估影响范围并触发局部重算
- **预留**：partial_recompute 接受影响范围参数
- **暂不展开**：影响范围自动评估逻辑

### 8.4 完整回归接口

- **当前**：前一阶段中手动决定何时回归
- **未来**：按触发矩阵自动判断
- **暂不展开**：并行执行策略、增量回归优化

### 8.5 重算/回归触发矩阵

| 变更类型 | 默认动作 | 原因 |
|----------|----------|------|
| 原始字段解析规则变更（如 cell_infos 新增字段提取） | 完整回归 | 影响全部标准化记录 |
| 主键/派生规则变更（如 BS_ID 计算规则） | 完整回归 | 影响全部对象归属 |
| LAC/Cell/BS 可信判定主逻辑变更 | 完整回归 | 影响全部对象库 |
| BS 锚点计算规则变更（如中位数→加权平均） | 完整回归 | 影响全部 GPS 修正 |
| GPS 修正阈值变更（如 4G 从 1000m 改为 800m） | 完整回归 | 影响全部 GPS 修正判定 |
| 仅新增特例旁路规则（如新增一种异常类型） | 局部重算 | 只影响命中该规则的对象 |
| 仅新增问题样本复核 | 局部重算 | 只影响相关对象 |
| 同规则下基线刷新（定期更新画像） | 仅重算画像 | 不回归事实，事实已正确 |
| 增量沉淀（日常 2 小时批次） | 不触发回归 | 正常增量流程 |

**一句话原则**：会改变"全局解释口径"的 → 完整回归；只影响"局部对象或局部规则命中"的 → 局部重算。

## 十、风险点、优先级建议、阻塞项

### 10.1 风险点

| 风险 | 严重度 | 缓解措施 |
|------|--------|----------|
| 等待池膨胀：大量不达标对象堆积占用存储 | 中 | 设置超时清理机制（waiting 超 30 天未达标 → dormant，dormant 超 90 天 → retired） |
| 碰撞 BS 拆分后画像不一致 | 中 | 拆分后独立验证每个虚拟 BS 的画像质量，确认可信度 |
| 初始化与增量规则不一致导致结果不可比 | 高 | 共用模块设计（standardize / govern_facts / build_profile 初始化和增量调用同一份代码） |
| 版本体系过重影响开发速度 | 中 | 首轮最小化：只跟踪 run_id + 3 个 version 标识，不做复杂版本管理 UI |
| 性能问题：8000 万行全量回归耗时 | 高 | 利用 PostgreSQL 并行查询 + 表分区（按 batch_id）+ 预算缓存 |

### 10.2 优先级建议

**第一批（最高优先）—— 用 3 天数据跑通初始化：**
1. 设计并创建 obj_cell（核心）+ obj_bs + obj_lac 表结构（Cell 为中心，BS/LAC 由 Cell 派生）

4. 实现 standardize 模块（复用前一阶段的解析规则和 ODS 清洗规则）
5. 实现 build_trusted_library（初始化用 LAC→Cell→BS 顺序，已验证高效）
6. 实现 govern_facts（GPS 修正 + 信号补齐）
7. 实现 build_profile（从 cell_detail_base 聚合画像）
8. 用前 3 天数据跑通初始化，冻结初始基线

**第二批（高优先）—— 用第 4~7 天数据跑通 Cell 先行增量：**

4. 实现 update_cell_detail_base（滚动更新）
5. 用第 4~7 天数据按批次跑增量

**第三批（中优先）—— 完善治理能力：**

### 10.3 进入实现前的阻塞项

只有以下 3 项真正阻塞实现启动：

1. **确认 raw_source 起点**：从原始 27 列重新解析，还是复用前一阶段的标准化数据？
