### 必须先理解的业务事实

rebuild3 不是静态报表系统，而是一个围绕下面主语运行的动态治理系统：

- 对象（Cell / BS / LAC）
- 决策（四分流、晋升、降级、异常）
- 事实（governed / pending_observation / pending_issue / rejected）
- baseline / profile
- run / batch / version 绑定
- 状态流转（lifecycle_state + health_state）
- 三类资格（存在资格 / 锚点资格 / baseline 资格）

初始化和增量共享一套治理语义，但入口不同。

## 启动器页面专项设计要求

启动器是一个**独立运行的页面**，通过独立 Python 脚本启动后在浏览器中打开（与主系统分离）。  
它是运维人员管理整个 rebuild3 系统进程的控制面板。

### 启动器页面必须包含以下功能区：

#### 1. 服务控制区
每个服务（如后端 API、前端 Vite 服务等）都需要独立控制按钮：
- **启动** — 启动该服务进程
- **停止** — 正常停止该服务进程
- **重启** — 停止后重新启动
- **杀掉端口** — 强制杀掉占用该端口的进程（适合进程卡死的情况）

#### 2. 运行状态区
- 每个服务的当前状态：`运行中 / 已停止 / 启动中 / 错误`
- 进程 PID 展示
- 服务监听端口展示
- 运行时长展示

#### 3. 日志查看区
- 可切换查看各服务的实时日志输出
- 日志自动滚动到最新一行
- 支持清空日志
- 支持按级别过滤（INFO / WARN / ERROR）

#### 4. 整体系统控制
- 一键启动所有服务
- 一键停止所有服务
- 一键重启所有服务

### 启动器设计风格要求
- 风格与主系统一致，但整体更简洁（这是一个运维面板，不是数据分析工具）
- 状态指示必须使用显眼的颜色徽标（绿色=运行中，红色=已停止，黄色=启动中/重启中）
- 按钮要有明确的危险等级区分（普通操作 vs 强制操作）
- 日志区要有等宽字体、背景暗色
