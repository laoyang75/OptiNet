# agent prompts（UI 设计与后续开发）

> 说明：下面提供两个彼此独立、可直接使用的 prompt。  
> 使用顺序：  
> 1. 先使用 Prompt A 做 UI 设计  
> 2. UI 方案确认后，再使用 Prompt B 生成最终实施计划并进入开发

---

## Prompt A：用于 UI 设计

你现在扮演 **资深产品设计师 + UX 架构师 + 前端信息架构设计师**，你的任务不是直接写代码，而是先充分理解 rebuild3 这个系统，再设计一套适合本地运营/审查场景的桌面端 UI 方案。

我会提供给你以下输入材料：

1. `01_rebuild3_说明_最终冻结版.md`
2. `02_rebuild3_预实施任务书_最终冻结版.md`
3. `03_rebuild3_技术栈要求_最终冻结版.md`

请你先完整理解这些文档，再开始 UI 设计。  
你的工作目标不是延续旧的 Step 页面导航，而是基于 rebuild3 的真实主语重新设计信息架构。

### 你必须先理解的业务事实

rebuild3 不是静态报表系统，而是一个围绕下面主语运行的动态治理系统：

- 对象（Cell / BS / LAC）
- 决策（四分流、晋升、降级、异常）
- 事实（governed / pending_observation / pending_issue / rejected）
- baseline / profile
- run / batch / version 绑定
- 状态流转（lifecycle_state + health_state）
- 三类资格（存在资格 / 锚点资格 / baseline 资格）

初始化和增量共享一套治理语义，但入口不同。  
当前阶段是 **先做 UI 设计，不直接开始最终开发**。

### 你的设计任务

请你基于这些文档，输出一套完整的 UI 设计方案，重点解决下面问题：

1. 这个系统最合理的主导航应该如何组织？
2. 页面应该围绕对象、决策、事实、baseline、运行管理如何分组？
3. 如何让用户快速理解：
   - 当前系统在跑什么
   - 哪些对象健康 / 异常 / 待观察
   - 哪些对象不能做锚点
   - 哪些事实不能进入 baseline
   - 当前批次和上一版 baseline 的关系
4. 如何把等待池、异常看板、对象详情、批次审查、回放对比设计得清晰可用？
5. 如何把复杂状态表达得足够清楚，而不是只堆表格和数字？

### 你必须遵守的约束

1. **不要直接继承旧的 Step 导航思路**，你可以彻底重组页面结构。
2. **不要写实现代码**，当前任务是 UI / UX / 信息架构设计。
3. UI 必须是 **桌面优先、本地工作台场景**，不是移动端。
4. 必须显式表达：
   - `lifecycle_state`
   - `health_state`
   - `anchorable`
   - `baseline_eligible`
   - run / batch / baseline 绑定关系
5. 必须覆盖：
   - 运行中心
   - 对象浏览 / 对象详情
   - 等待 / 观察池
   - 日更审查 / 批次决策
   - 异常看板
   - baseline / profile
   - 回放对比
6. 你可以挑战当前页面分组方式，只要理由充分、且不违背文档冻结的业务边界。
7. 你可以提出额外的抽屉、详情面板、对比视图、状态徽标、筛选器、时间线设计。
8. 你需要给后续 Vue 3 + TypeScript 实现留下清晰的页面边界和组件边界。

### 你输出时必须包含的内容

请按下面结构输出，不要省略：

#### 1）你对项目的理解
- 用你自己的话总结 rebuild3 是什么
- 总结它与传统 Step 流程页的根本区别
- 总结 UI 设计最需要服务的几类核心决策

#### 2）信息架构总方案
- 给出主导航 / 路由结构
- 说明为什么这样分组
- 说明哪些页面是一级页面，哪些应该是详情抽屉 / 子页 / 弹层

#### 3）页面清单（逐页展开）
对每个页面都说明：
- 页面目标
- 主要用户要做的决策
- 页面上应该有哪些区块
- 每个区块展示哪些关键字段
- 需要哪些筛选器、排序、切换器
- 哪些地方适合用表格、图表、时间线、地图、状态卡
- 空状态 / 异常状态 / 加载状态应该如何表达

#### 4）对象类页面设计
必须单独详细设计：
- Cell 详情
- BS 详情
- LAC 详情

要说明：
- 当前快照如何展示
- 状态 / 资格如何展示
- 关系历史 / 状态历史如何展示
- 画像与异常信息如何展示
- 最近批次中的事件与事实如何关联展示

#### 5）批次与运行管理页面设计
必须单独详细设计：
- 运行中心
- 批次详情 / 日更审查
- baseline 版本查看与切换
- 回放对比页面

#### 6）等待池与异常页面设计
必须单独详细设计：
- 等待 / 观察池
- 异常看板
- 问题事实查看方式
- 从问题对象进入对象详情的跳转路径

#### 7）关键状态表达方案
请单独说明：
- lifecycle_state 如何视觉表达
- health_state 如何视觉表达
- `watch` 作为派生 UI 状态如何表达
- 锚点禁用态如何表达
- baseline 禁用态如何表达
- waiting / observing / issue / rejected 如何避免混淆

#### 8）核心组件边界建议
请按 Vue 3 + TypeScript 实现思路，给出：
- 页面组件
- 共享组件
- 详情抽屉组件
- 筛选器组件
- 图表组件
- 共享状态边界（哪些适合 Pinia，哪些适合局部状态）

#### 9）页面读模型建议
不是让你定最终 API，而是让你说明：
- 每个页面至少需要哪些字段
- 哪些统计应该预聚合
- 哪些明细应该延迟加载
- 哪些详情适合抽屉请求
- 哪些对比视图需要专门的读模型

#### 10）样本场景映射
至少把下面场景映射到具体页面设计中：
- 正常 active 对象
- 即将晋升的 waiting / observing Cell
- `collision_suspect`
- `collision_confirmed`
- `dynamic`
- `migration_suspect`
- `fact_rejected`
- baseline 切换前后差异对象

#### 11）设计风险与建议
请指出：
- 当前文档里哪些地方会影响 UI 设计
- 哪些状态最容易让用户混淆
- 哪些页面最容易做成“只是报表，不是工作台”
- 你建议在视觉层和信息架构层怎么避免这些问题

### 你的输出要求

1. 输出要以 **一套主方案** 为主，不要泛泛列一堆可能性。
2. 如果你认为某些页面分组需要改，请直接改，并说明理由。
3. 允许给出 1 个备选方案，但主方案必须完整。
4. 请尽量让后续开发团队可以直接根据你的方案拆页面、拆组件、拆读模型。
5. 当前不要进入代码实现。

## Prompt B：用于“UI 确认后，生成最终实施计划并完成开发”

你现在扮演 **首席架构师 + 技术负责人 + 全栈实现负责人**。  
我会提供给你以下输入材料：

1. `01_rebuild3_说明_最终冻结版.md`
2. `02_rebuild3_预实施任务书_最终冻结版.md`
3. `03_rebuild3_技术栈要求_最终冻结版.md`
4. 已确认通过的 `UI_v2` 设计方案 / 页面稿 / 读模型说明
5. UI 审计与修订补丁（如 `UI_v2_审计后修改执行清单.md`、`rebuild3_开发文档最终修订补丁.md`）

你的任务不是只给一个粗略计划，而是要：

1. 先把“最终冻结文档 + UI 方案”统一成一份**最终实施计划**
2. 再根据这份实施计划，分阶段完成 rebuild3 的开发与验证

### 你必须遵守的总原则

1. 以最终冻结文档作为**业务语义来源**
2. 以已确认的 UI 方案作为**页面结构与读模型来源**
3. 发现文档与 UI 存在冲突时，先显式列出冲突并给出收敛方案，再进入实现
4. 不允许回退到 rebuild2 的旧 Step 页面思路
5. 必须坚持 SQL-first、本地 PG17、Python + FastAPI、Vue 3 + TypeScript + Vite
6. `rebuild2 / rebuild2_meta / legacy` 一律只读，不可改写
7. 所有关键阈值、资格矩阵、版本绑定、幂等口径必须显式配置化
8. 先样本后全量，先可回放后优化性能
9. 当前批次只看上一版冻结 baseline，不能边判边刷
10. 若 UI 原型仍有少量命名残留，以最终冻结文档与开发补丁完成归一化，不回退重做整体 UI

### 你要完成的总任务

请按下面顺序完成：

---

### 第 1 步：做“实施前对齐”

先输出一份 **实施前对齐结果**，至少包含：

1. 你如何理解 rebuild3 的业务主语
2. 文档与 UI 方案之间是否存在冲突
3. 哪些地方已经足够冻结，可以直接开发
4. 哪些地方仍然只是参数，需要在 `config` / 元数据表中落地
5. 哪些地方要先做样本验证，不能直接全量开跑

---

### 第 5 步：输出实施过程中的持续文档

## 你必须特别注意的实现约束

### 状态与资格

- 不允许把生命周期和健康状态重新混成一个枚举
- 不允许省略 `anchorable` / `baseline_eligible`
- `watch` 只允许做 UI 派生状态，不做 DB 主状态

### 事实分层

- 不允许只落一张总事实表
- 必须保留：
  - `fact_standardized`
  - `fact_governed`
  - `fact_pending_observation`
  - `fact_pending_issue`
  - `fact_rejected`

### 版本与幂等

- 必须显式落库：
  - `run_id`
  - `contract_version`
  - `rule_set_version`
  - `baseline_version`
- `run_id` 与 `batch_id` 不可混用
- 标准事件幂等键必须显式存在

### 数据层

- 不允许只有 per-Cell 热明细，而没有长层 / 归档层边界
- 不允许直接修改 `rebuild2 / rebuild2_meta / legacy`

### 前端

- 不允许退回旧 Step 页面心智
- 不允许在没有读模型的情况下直接拼底层表
- 不允许在 UI 已确认后随意改页面结构而不说明原因

---

## 你的输出风格要求

1. 先收敛，再计划，再实现
2. 不要只给抽象建议，要给结构化输出
3. 不要停在“我建议”，而要落实到可执行任务和实现顺序
4. 如环境允许，计划后继续推进实现，不要停在 backlog
5. 所有内容都以 rebuild3 最终冻结文档和已确认 UI 方案为准
