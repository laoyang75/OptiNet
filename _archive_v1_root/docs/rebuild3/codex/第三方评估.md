下面把内容整理成你要求的三份文档。我把两组结果分别称为“分析A”和“分析B”。

分析A：更偏实现骨架，强调初始化/增量步骤、等待池、级联晋升、per-Cell 详细库、运行优先级。
分析B：更偏对象治理语言，强调“对象/决策/事实/基线”、研究期与长期口径拆分、Cell 三层门槛、源适配、幂等与批次快照。

# 第一：两份分析的对比评估

## 一、共识点

**1. 对 rebuild3 的定位一致。**
两份分析都没有把 rebuild3 误判为生产级系统，而是把它定义成“把 rebuild2 已验证规则嵌入动态治理骨架”的本地调试/验证系统，重点是对象状态、批次治理、异常旁路和基线刷新，而不是微服务、消息队列或云原生大平台。

**2. 对处理顺序的主判断一致。**
两份分析都认可“初始化沿用 LAC→Cell→BS，增量转为 Cell 先行”的混合策略；都认为初始化是从零建边界，增量是已有对象和基线上的持续判断。

**3. 对系统主语的切换一致。**
两份分析都主张从“Layer/Step 流水线”切换为“对象 + 决策 + 事实 + 画像 + 状态流转”的系统主语，且都把 Cell 视为动态治理的最小主语，BS 视为空间锚点。

**4. 对验证路径一致。**
两份分析都把“3 天初始化 + 4 天增量，对比直接 7 天初始化”作为当前最合理的本地收敛性验证路径。

**5. 对北京口径的边界一致。**
两份分析都明确：北京一周数据上的严格 GPS/LAC 过滤，是研究期收敛策略，不应被硬编码成未来长期规则。

---

## 二、分歧点及我的判断

### 1. 状态机应该是一维还是二维

分析A更偏一维状态：`pending / waiting / trusted / anomaly / dormant / retired`。
分析B更偏二维表达：`lifecycle_state` 与 `health_state` 分离，避免“可信但异常观察中”这类语义挤在一个枚举里。

**我的判断：应采用二维模型。**
原因不是为了“架构好看”，而是为了把三个本来不同的问题拆开：

* 生命周期：这个对象现在是候选、正式、休眠还是退役
* 健康状态：它是否能继续当锚点
* 基线资格：它的事实是否允许进入画像刷新

如果仍用一维状态，后面一定会出现“trusted_but_collision_watch”“trusted_but_not_baseline_eligible”这类状态爆炸。
所以建议最小化为：

* `lifecycle_state`：waiting / observing / active / dormant / retired / rejected
* `health_state`：healthy / insufficient / gps_bias / collision_suspect / collision_confirmed / dynamic / migration_suspect

这比一维状态多一列，但会大幅降低控制环节歧义。

### 2. per-Cell 详细库是否应成为唯一画像来源

分析A主张每个 Cell 保留最近 N 条治理后明细，画像从这里算，强调时效性和存储可控。
分析B则进一步指出，仅有热明细不够，应该同时有热明细、长期汇总、归档事实三层。

**我的判断：分析B更完整，但首轮实现应采用“分析A的核心 + 分析B的补足”。**
也就是：

* **热层**：per-Cell 最近 N 条治理后明细，供当前画像与锚点计算
* **长层**：Cell/BS 日级汇总，供活跃节奏、退役判断、慢变化跟踪
* **档层**：governed archive，供回放和完整回归

结论：per-Cell 详细库合理，但**不应独自承担全部长期记忆**。

### 3. 异常记录应该进 governed_fact 还是 pending_issue

分析A倾向于：异常对象的记录仍可治理并写入事实，只是打标签。
分析B倾向于：当对象身份本身不稳定时，应进入待判定事实或异常池，而不是直接进入正式事实。

**我的判断：两者都对，但适用对象不同。**

应区分两类异常：

* **记录级异常**：如 GPS 偏差、single_large、normal_spread
  这类记录可以进入 `fact_governed`，但带异常标签
* **对象定义级异常**：如 collision_confirmed、dynamic_bs、migration_suspect
  这类不能直接进入可用于画像的正式事实；要么进入 `fact_pending_issue`，要么进入 `fact_governed` 但强制 `baseline_eligible = false`

换句话说，不是“异常都能进”，也不是“异常都不能进”，而是看异常是否已经破坏对象身份和锚点语义。

### 4. LAC 级联条件的表达不一致

分析A内部其实有一个小冲突：一处写“LAC 下 ≥N 个 BS trusted 才 trusted”，另一处又写“trusted BS ≥1 即自动注册/更新 LAC”。
分析B没有把阈值写死，但强调 LAC 是边界和区域约束，不应再做增量前置门槛。

**我的判断：这里要拆成两个动作，而不是一个动作。**

* **LAC existence（存在）**：第 1 个 active BS 出现时即可注册
* **LAC active/trusted（激活）**：active BS 数、稳定性、异常比达到阈值后才变成可用边界对象

这样就把“≥1”与“≥N”的冲突消掉了：
前者是“建档”，后者是“可用”。

### 5. 阻塞项清单孰轻孰重

分析A认为真正阻塞实现的只有 3 项：raw_source 起点、event_time 口径、Cell 初始阈值。
分析B则把源表适配边界和“异常对象是否还能当补齐锚点”也列入高优先级阻塞。

**我的判断：分析B更接近真实实现阻塞。**
因为没有“异常对象是否还能做锚点”的明确口径，GPS 修正、信号补齐、画像刷新三条链都会相互污染；没有源适配边界，初始化入口都不稳定。

## 三、各自盲区

### 分析A的盲区

**1. 对“对象存在 / 对象可作锚点 / 对象可进基线”三层资格分得不够开。**
这会让 waiting、anomaly、trusted 的后续控制环节变得模糊。

**2. 对 event_time、幂等键、源适配的强调不够。**
这些对 2 小时增量回放是底层稳定性问题，不是后置细节。

**3. LAC 级联条件存在内部口径不一致。**
这说明“对象注册”和“对象激活”还没有被明确拆开。

### 分析B的盲区

**1. 架构语言更强，控制环节粒度偏粗。**
它讲清了“为什么”，但对“哪一步进、哪一步出、用什么指标判断”写得不如分析A细。

**2. 对等待池回填、同批次晋升后的事实回补、per-Cell 写入时机写得不够具体。**
这些恰好是实现期最容易出错的地方。

### 两份分析共同的盲区

**1. 都低估了“基线反向参与准入”带来的循环依赖风险。**
如果同一批次边判定边刷新基线，很容易出现自举污染。
我的判断是：**必须用“上一个冻结基线版本”判定当前批，当前批结束后再刷新下一版基线。**

**2. 都没有把“候选事实回填后是否重写历史画像”讲透。**
如果一个 Cell 在第 7 批次晋升，把前 6 批等待池明细回填进正式事实，会不会改写之前已经冻结的画像？
答案应是：**会进入事实归档，但只在下一次画像刷新时生效，不反写旧 baseline_version。**

**3. 都没有明确“等价”的定义。**
增量结果不必与 7 天全量初始化逐行完全一致；真正该比的是：
对象集合、对象中心、覆盖半径、画像分位、异常分类、准入分流比例，是否在可接受误差内收敛。

# 第二：rebuild3 重构指导框架

## 一、总体策略

rebuild3 的实现主线应定义为：

**复用 rebuild2 已经稳定的解析、清洗、GPS 修正、信号补齐、BS 异常分类与画像口径；重构的核心不在算法本身，而在对象管理。初始化阶段允许继续使用 LAC→Cell→BS 的高效冷启动路径；增量阶段统一改成“上一版冻结基线 + 当前批次判定 + 批后级联更新”的 Cell 先行模式。初始化和增量可以走不同的建库入口，但必须共享同一套：记录准入规则、Cell 晋升规则、BS/LAC 计算函数、异常判定语义、画像刷新口径。**

我建议把系统拆成五个稳定主语：

1. **标准化事件层**：不可变输入事实
2. **对象注册层**：Cell/BS/LAC 当前状态与关系
3. **决策分流层**：governed / candidate / issue / rejected
4. **治理后事实层**：修正/补齐后的正式事实
5. **画像基线层**：冻结版本的准入参照

---

## 二、处理流程骨架

## 1. 初始化流程

**Init-1：运行冻结**
冻结 `run_id / contract_version / rule_set_version / source_adapter / replay_window`。
同一轮初始化内，这些不允许变化。

**Init-2：全量标准化**
把 3 天初始化窗口的原始数据统一转成标准化事件；解析成功与失败分流，标准化事件层不可变。

**Init-3：记录级合规检查**
对白名单、主键合法性、时间合法性、GPS 研究口径边界做硬判断。
不合规事件进入 `fact_rejected`，不参与后续对象构建。

**Init-4：研究期 LAC 预边界**
当前北京验证阶段允许用 LAC/GPS 严格边界快速缩小可信区域，但这一步只属于初始化研究模式，不得复制为增量长期前置门槛。

**Init-5：Cell 候选聚合与晋升**
在当前初始化窗口内，对 Cell 形成首轮证据聚合；满足晋升门槛的注册为 active Cell。

**Init-6：BS 派生与首轮分类**
BS 由 active Cell 派生；先算中心点与覆盖半径，再跑异常分类；此时得到“存在的 BS”和“可用作锚点的 BS”。

**Init-7：LAC 派生**
LAC 由 active BS 派生；LAC 的存在与可用也分开处理：先注册，再根据 active BS 数和稳定性激活。

**Init-8：首轮事实治理**
仅对已获得锚点资格的对象执行 GPS 修正和信号补齐。
身份未稳定但记录语义仍成立的，可进入 governed_fact 且 `baseline_eligible = false`；身份本身未定的，进 pending_issue。

**Init-9：完整回归**
用纯化后的对象库与异常结论，对初始化窗口再跑一遍事实治理，消掉“先粗后细”带来的语义混杂。

**Init-10：建立 baseline v1**
从 `baseline_eligible = true` 的 governed facts 生成 per-Cell 热详细库、日级汇总和三级画像，冻结为 `baseline_version = v1`。
初始化结束后，系统进入增量模式。

---

## 2. 增量流程

**Inc-1：批次快照冻结**
注册 `batch_id`，绑定上一版 `baseline_version`。
本批所有准入和修正，都只能参考这个冻结快照。

**Inc-2：标准化**
与初始化使用同一套解析、清洗和派生规则。

**Inc-3：记录级合规检查**
不合规直接拒收，不进入等待池。

**Inc-4：对象查找与四分流**
按 `(operator, tech, lac, cell_id)` 查找 Cell；结果进入四类之一：

* 已知且可治理
* 已知但对象异常待判
* 未知但结构合规，可入候选
* 结构不合规，直接拒收

分析B提出的四分流表达更适合作为增量主骨架。

**Inc-5：候选累计与 Cell 晋升**
waiting/observing Cell 只累计证据，不直接做正式锚点。
在批末执行晋升评估；达标的 Cell 从候选转 active，并触发 backlog backfill。

**Inc-6：BS 级联更新**
凡有 Cell 新晋升或健康状态变化，相关 BS 在批末统一重算中心与半径，并判断是否仍可作锚点。

**Inc-7：LAC 级联更新**
相关 BS 更新后，刷新 LAC 的存在、激活和健康状态。

**Inc-8：事实治理**
对 active 且允许锚点的对象做 GPS 修正和信号补齐；对异常但仍可保留语义的，写入 governed_fact 并带标签；对象身份未定的写入 pending_issue。

**Inc-9：详细库与汇总更新**
governed_fact 写入 hot detail、daily aggregate、archive。
hot detail 只保留画像热窗口；长周期判断靠 daily aggregate。

**Inc-10：异常检测与对象状态转移**
对 collision / dynamic / migration / drift / disappearance 做批后检测，更新 health_state 和 lifecycle_state。

**Inc-11：画像刷新**
不是每批都刷。
满足触发条件时，用 hot detail + daily aggregate 计算新画像，冻结新 `baseline_version`，供下一批使用。

## 三、完整的控制环节清单

下面是实现团队必须显式做判断的控制环节。每个环节都要有进入条件、退出条件、输入指标、输出、可调参数和错判后果。

### 1. 源适配冻结门

* 控什么：本轮 run 的输入口径是否唯一
* 进入：run 创建时
* 输入指标：源表类型、字段映射版本、单表/双表回放模式
* 输出：source_adapter 绑定、run 上下文
* 可调参数：源白名单、字段映射版本、允许的源组合
* 错判后果：同一 run 内混入不同输入语义，后续无法比较

### 2. 标准化成功门

* 控什么：原始记录能否成为标准化事件
* 进入：原始记录读取后
* 输入指标：`cell_infos/ss1` 解析成功率、主键派生成功率
* 输出：`standardized_event` 或 `parse_error`
* 可调参数：解析严格度、继承/兜底规则
* 错判后果：假阳性会污染主键；假阴性会丢掉可用样本

### 3. 记录级合规门

* 控什么：事件是否允许进入对象判定链
* 进入：标准化完成后
* 输入指标：运营商白名单、LAC/CellID 合法范围、时间合法性、GPS 边界
* 输出：compliant / rejected
* 可调参数：白名单、范围、研究期 GPS 边界、时间容忍
* 错判后果：假阳性会把垃圾样本送入等待池；假阴性会压低收敛速度

### 4. 已知对象识别门

* 控什么：事件命中 active / dormant / candidate / none
* 进入：记录合规后
* 输入指标：对象注册表、关系历史、当前生命周期
* 输出：route 决策
* 可调参数：dormant 是否可直接唤醒、关系匹配优先级
* 错判后果：对象碎片化，或把新对象误并入旧对象

### 5. 候选入池门

* 控什么：未知但合规对象是否进入 waiting
* 进入：未命中已知对象后
* 输入指标：结构完整性、去重键、最小字段集
* 输出：candidate 记录、候选累计
* 可调参数：候选保存窗口、去重规则
* 错判后果：等待池膨胀，或新对象发现延迟

### 6. waiting→observing 门

* 控什么：是否从“纯累计”进入“可评估观察”
* 进入：waiting 候选累计到最小证据后
* 输入指标：记录数、GPS 有效点、设备数、时间跨度
* 输出：observing 状态
* 可调参数：最小证据阈值
* 错判后果：过早观察会增加假晋升，过晚观察会拖慢收敛

### 7. Cell 晋升门

* 控什么：Cell 是否变成正式对象
* 进入：observing Cell 到批末评估时
* 输入指标：GPS 点数、设备数、活跃跨度、空间收敛度、LAC 映射稳定性、异常预检
* 输出：Cell active / 继续观察 / rejected
* 可调参数：最低证据阈值、收敛阈值、高流量快晋升条件
* 错判后果：假阳性会制造错误锚点；假阴性会让 valid Cell 长期卡池

### 8. BS 级联构建门

* 控什么：BS 是否存在、是否可作锚点
* 进入：任一下属 Cell 新晋升或健康变化后
* 输入指标：active Cell 数、子 Cell 质心、异常类别
* 输出：BS 注册、中心点、覆盖半径、anchorable 标记
* 可调参数：存在阈值、锚点阈值、重算频率
* 错判后果：会直接污染 GPS 修正

### 9. LAC 级联构建门

* 控什么：LAC 是否注册、是否激活
* 进入：任一下属 BS 更新后
* 输入指标：active BS 数、异常 BS 占比、边界一致性
* 输出：LAC registered / active / watch
* 可调参数：注册阈值、激活阈值、异常占比容忍
* 错判后果：过松会把边界扩脏，过严会压掉真实区域

### 10. GPS 修正前置门

* 控什么：该记录能否使用 BS 锚点修正
* 进入：事实治理前
* 输入指标：BS anchorable、健康状态、原始 GPS 是否存在、距离偏差
* 输出：gps_value、gps_source
* 可调参数：按制式距离阈值、异常状态允许矩阵
* 错判后果：假阳性会“钉错点”，假阴性会保留漂移

### 11. 信号补齐前置门

* 控什么：能否补齐、用谁补齐
* 进入：GPS 修正后
* 输入指标：同 Cell donor、同 BS donor、时间差、donor 健康状态
* 输出：signal_value、signal_source、donor_id
* 可调参数：最大时间差、donor 优先级、异常状态是否禁用
* 错判后果：会把别的对象画像导进来

### 12. governed_fact / pending_issue 分流门

* 控什么：记录进正式事实还是进待判池
* 进入：修正/补齐后
* 输入指标：对象身份稳定性、异常类型、是否仍可解释
* 输出：`fact_governed`、`fact_pending_issue`、`baseline_eligible`
* 可调参数：哪些异常可入正式事实、哪些异常可进基线
* 错判后果：要么污染基线，要么损失可用事实

### 13. per-Cell 详细库写入门

* 控什么：哪些 governed facts 进入热详细库
* 进入：governed_fact 写入后
* 输入指标：baseline_eligible、traffic class、当前热库规模、记录新鲜度
* 输出：hot detail 更新
* 可调参数：N、采样/保留策略、是否保留设备/时间多样性
* 错判后果：热库失真或存储失控

### 14. 详细库淘汰门

* 控什么：超出 N 后淘汰谁
* 进入：热库写入时
* 输入指标：记录时间、设备覆盖、时段覆盖、当前分布稳定性
* 输出：evict 记录、归档/摘要更新
* 可调参数：纯 FIFO 还是“最近优先 + 多样性保留”
* 错判后果：画像被最近高频设备劫持，或保留太旧导致画像钝化

### 15. 画像刷新门

* 控什么：什么时候刷、刷多大范围
* 进入：批末或按周期
* 输入指标：对象变更量、晋升量、异常解除量、时间间隔
* 输出：新 `baseline_version`
* 可调参数：刷新周期、最小变更阈值、全量/局部刷新
* 错判后果：太勤会抖动，太慢会过时

### 16. 异常检测与分流门

* 控什么：drift / collision / dynamic / migration / insufficient 的判定
* 进入：事实沉淀后
* 输入指标：中心偏移、设备交叉、子 Cell 质心聚类、活跃节奏变化
* 输出：health_state、issue_event、split/recompute 触发
* 可调参数：偏移阈值、聚类阈值、设备交叉阈值
* 错判后果：健康对象被冻结，或坏锚点长期存活

### 17. 降级/唤醒/退役门

* 控什么：active ↔ dormant → retired
* 进入：批末对象状态维护
* 输入指标：last_seen、avg_report_interval、子对象活跃数
* 输出：lifecycle_state 转移
* 可调参数：不活跃倍数、dormant 超时、唤醒最小证据
* 错判后果：对象连续性断裂，或僵尸对象堆积

### 18. 完整回归触发门

* 控什么：局部重算还是全量回归
* 进入：规则、契约、主键语义或锚点计算规则变更时
* 输入指标：变更类型、影响范围
* 输出：partial_recompute / full_regression run
* 可调参数：触发矩阵
* 错判后果：混合语义长期共存，导致结果不可比

## 四、对象生命周期

## 1. 推荐的状态表达

我建议采用**二维状态**。

### 生命周期状态 `lifecycle_state`

* `waiting`：合规但证据不足
* `observing`：已具备最小证据，等待正式晋升
* `active`：正式对象
* `dormant`：暂时沉寂，等待唤醒或退役
* `retired`：确认退役，只保留历史
* `rejected`：明确不接受为对象

### 健康状态 `health_state`

* `healthy`
* `insufficient`
* `gps_bias`
* `collision_suspect`
* `collision_confirmed`
* `dynamic`
* `migration_suspect`

**关键原则**：
“active 但异常观察中”不是新生命周期，而是 `lifecycle_state = active` + `health_state != healthy`。

---

## 2. Cell 生命周期

### 合法路径

* `absent -> waiting`
* `waiting -> observing`
* `observing -> active`
* `waiting / observing -> rejected`
* `active -> dormant`
* `dormant -> observing`
* `dormant -> retired`

### 健康状态路径

* `healthy <-> gps_bias`
* `healthy -> collision_suspect -> collision_confirmed`
* `healthy -> dynamic`
* `healthy -> migration_suspect`
* 任意异常恢复后可回到 `healthy`

### 级联关系

* Cell 是唯一直接由原始记录驱动注册的对象
* Cell 晋升触发 BS 更新
* Cell 不直接决定 LAC，只通过 BS 传播

---

## 3. BS 生命周期

### 合法路径

* `absent -> active`：至少有 1 个 active Cell 时注册 BS
* `active -> dormant`：所有 active Cell 不活跃
* `dormant -> active`：任一 Cell 恢复
* `dormant -> retired`：长期无恢复

### 健康状态路径

* `healthy <-> gps_bias`
* `healthy -> collision_suspect -> collision_confirmed`
* `healthy -> dynamic`
* `healthy -> migration_suspect`

### 重要补充

* **BS 存在 ≠ BS 可作锚点**
* `collision_confirmed / dynamic / migration_suspect / insufficient` 下，BS 可继续存在，但默认失去 anchorable 资格

---

## 4. LAC 生命周期

### 合法路径

* `absent -> active`：达到 LAC 激活阈值后
* `active -> dormant`：active BS 数长期低于保持阈值
* `dormant -> active`：active BS 恢复
* `dormant -> retired`：长期无恢复

### 重要补充

LAC 是**派生区域对象**，不是增量前置门槛。
它的主要作用是：

* 约束区域边界
* 承载区域级画像
* 汇总下级异常占比

---

## 5. 级联规则

* **Cell active** → 触发 BS 重算
* **BS active/health 变化** → 触发 LAC 重算
* **Cell 大面积 dormancy** → BS 可进入 dormant
* **BS 大面积 dormancy** → LAC 可进入 dormant
* **对象异常不会自动删除对象**，只先改变 `health_state`
* **split / migration** 必须保留关系历史，不能直接覆盖旧对象

---

## 五、数据流与存储设计原则

## 1. 数据流原则

**原始记录 → 标准化事件 → 决策分流 → 治理后事实 / 待判定事实 / 拒收事实 → 热详细库 / 长期汇总 / 归档 → 画像基线**

三条原则必须守住：

* **标准化事件不可变**
* **批次判定只看上一版冻结基线**
* **画像刷新永远发生在批次判定之后**

这样才能切断循环依赖。分析B提出“对象/决策/事实/基线”的分层方向是对的；分析A提出的标准化、对象匹配、事实治理、画像刷新顺序也对。我的框架是在这两者之间补上“批次快照”和“待判定事实”这两个控制点。

## 2. per-Cell 详细库的设计判断

**结论：合理，但只能作为热层，不能单独承担全部历史语义。**

它相比“每次都从全量事实表重聚合画像”的优势是：

* 画像刷新成本稳定
* Cell 局部变化可局部生效
* BS 锚点能随近期真实变化更新
* 很适合 rebuild3 这种“初始化 + 增量验证”的调试环境。

但它有天然偏差：

* 只保留最近 N 条，容易把高流量时段、少数设备、短时间窗口过度放大
* 慢变化、季节性、长期沉寂信号会被热窗口抹平
* 退役/消失判断不能只靠热详细库

所以我建议：

* **hot detail**：用于当前画像与空间锚点
* **daily aggregate**：用于活跃节奏、长期稳定性、退役判断
* **archive**：用于回放与完整回归

## 3. N 的选择逻辑

N 不应该是一个拍脑袋的常数，应由四类指标共同决定：

1. **画像稳定性**
   当继续增加样本时，中心点、P50/P90 半径、RSRP P10/P50/P90 的变化已经很小，说明 N 足够。

2. **流量分层**
   低流量 Cell 可保留全量；中高流量 Cell 应分层设 N，不宜全网一个常数。分析A已经提出低/中/高流量分层，这是对的。

3. **有效样本覆盖，而不是原始行数**
   N 应尽量代表“设备-时间片”的有效覆盖，而不只是最近 N 行。
   否则一个高频设备可以轻易把热库刷满。

4. **刷新节奏和存储预算**
   N 至少要覆盖一个完整画像刷新窗口内的主要变化；同时不能让热库存储失控。

# 第三：验证计划与风险

## 一、验证方案细化

## 1. 验证方法

统一采用同一套 `contract_version / rule_set_version`，做两条链：

* **链路 A**：第 1～3 天初始化，第 4～7 天按 2 小时增量
* **链路 B**：第 1～7 天直接初始化

在**第 7 天结束时点**做对比。
对比时如有画像刷新，应在同一评价时点强制完成最终刷新，避免因为刷新时刻不同造成伪差异。这个思路与两份分析一致。

## 2. 对比指标

### A. 对象收敛指标

看“对象有没有收敛到同一批成熟对象”：

* active Cell / BS / LAC 数量差
* 对象主键集合重合度
* waiting / observing / active / dormant 分布
* Cell 晋升时延分布
* 等待池积压量与尾部对象数

### B. 空间收敛指标

看“锚点是否等价”：

* BS 中心点距离差分布
* BS 覆盖半径 P50/P90 差分布
* LAC 中心与覆盖范围差分布
* GPS 修正来源占比（original / corrected / fallback）

### C. 信号收敛指标

看“补齐是否等价”：

* 信号补齐率
* donor 来源占比（same cell / same BS）
* RSRP/RSRQ/SINR 的 P10/P50/P90 差异
* baseline_eligible 事实占比

### D. 异常收敛指标

看“坏对象是否被同样识别出来”：

* collision / dynamic / migration_suspect 数量
* 异常对象集合重合度
* anchor-disabled BS 占比
* 进入 pending_issue 的事实比例

### E. 决策收敛指标

看“批次决策骨架是否一致”：

* governed / candidate / issue / rejected 四分流占比
* 同主键对象在两条链中的最终路由一致率
* 同一批次重放的一致率（幂等性）

### F. 热库稳定性指标

看 per-Cell 详细库设计是否成立：

* 画像在最近几次刷新间的波动幅度
* 增大 N 后画像改善是否已进入平台期
* 高流量 Cell 是否被单设备/短时段劫持

## 3. 这个验证方案能覆盖什么

能覆盖：

* 冷启动到增量的基本衔接
* waiting→promotion 的主路径
* Cell→BS→LAC 级联是否能收敛
* 画像能否作为下一批准入依据
* 研究期严格口径下的对象收敛性。

不能覆盖：

* 长周期 dormancy / retired
* 季节性或月级变化
* 迟到数据、乱序数据
* 源表结构变化
* 跨城市阈值差异
* 规则版本升级后的 partial/full regression 策略

**所以 7 天足够验证“收敛骨架”，不够验证“长期运行行为”。**

---

## 二、已识别风险及缓解建议

### 1. 初始化与增量口径不一致

* 风险：同样的对象，初始化能进，增量进不去
* 缓解：复用同一套晋升规则、BS/LAC 计算函数、异常语义；只允许“入口不同”，不允许“计算不同”

### 2. 基线循环依赖

* 风险：当前批边判边刷基线，导致自我强化
* 缓解：批次只看上一版 baseline，当前批结束后才产生下一版

### 3. LAC 逻辑混乱

* 风险：LAC 既当初始化前置门控，又当增量派生结果，语义打架
* 缓解：仅初始化研究模式允许 LAC 预边界；增量时 LAC 只做派生区域对象

### 4. 异常对象继续污染锚点

* 风险：collision_confirmed / dynamic / migration_suspect 继续参与 GPS 修正与画像刷新
* 缓解：把“对象存在资格”“锚点资格”“基线资格”分开控制

### 5. waiting 回填破坏历史一致性

* 风险：Cell 晋升后把历史候选事实回填，导致旧 baseline 被隐式改写
* 缓解：回填只改变事实层和后续画像刷新，不回写旧 baseline_version

### 6. per-Cell 热库偏差

* 风险：N 过小或纯 FIFO，导致画像只代表最近少量高频设备
* 缓解：加入 daily aggregate；N 用稳定性曲线选，不用拍脑袋

### 7. event_time / 幂等没先冻住

* 风险：2 小时重放结果不可复现
* 缓解：实现前先冻结 event_time、batch_id、展开后的幂等键

### 8. 对象拆分/迁移无关系历史

* 风险：后续无法解释“为什么这个 Cell 看起来换了 BS/LAC”
* 缓解：对象关系变更必须写 relation_history，禁止直接覆盖旧归属

---

## 三、进入实现前必须确认的最小决策清单

控制在 5 项内，我的建议是这 5 项：

**1. event_time 与迟到数据口径**
用哪个时间做批次归属、最近 donor、活跃判断、画像时间窗。

**2. 源适配与 raw_source 起点**
首轮到底从原始 27 列重新解析，还是复用 rebuild2 标准化结果；单表/双表回放边界如何定义。
这一项把分析A的“raw_source 起点”与分析B的“源适配边界”合并。

**3. Cell / BS / LAC 的两级阈值**
至少要先定：

* Cell：waiting→observing→active
* BS：存在阈值 vs 锚点阈值
* LAC：注册阈值 vs 激活阈值

**4. 异常对象的锚点与基线资格矩阵**
哪些异常仍可：

* 写 governed_fact
* 参与 GPS 修正
* 参与信号补齐
* 进入 baseline_eligible

**5. per-Cell 热库 + 长期汇总的保留策略**
不是只定一个 N，而是先定：

* 是否必须有 daily aggregate
* N 是否分 traffic class
* 热库淘汰是否允许“最近优先 + 多样性保留”

---

## 最后的总判断

我支持“初始化 LAC→Cell→BS、增量 Cell 先行”的混合策略，但前提是把它理解为**两种入口，不是两套规则**。
真正决定收敛性的，不是顺序本身，而是下面四件事是否统一：

1. Cell 晋升条件是否统一
2. BS/LAC 是否纯由 active child 派生
3. 批次是否只看冻结基线
4. 异常对象是否被正确剥夺锚点资格

如果这四件事统一，3 天初始化 + 4 天增量完全有机会在“成熟对象集合、空间锚点、画像分位和异常分类”上收敛到与 7 天全量初始化近似等价；
如果这四件事不统一，哪怕顺序看起来合理，最终也只会得到“两套口径下都各自自洽、但彼此不可比”的系统。
