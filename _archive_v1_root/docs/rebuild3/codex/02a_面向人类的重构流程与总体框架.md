## 一、一句话定位

**rebuild3 是把前一阶段（rebuild2）在静态数据上研究出的字段解析规则、可信对象筛选方法、GPS 修正策略、信号补齐策略、BS 异常分类算法，嵌入到一个"初始化 + 2 小时增量"共用同一套骨架的动态治理系统中。**

---

## 二、为什么它是"动态治理系统"，而不是"静态重跑系统"

前一阶段（rebuild2）回答的问题是：**规则是什么、阈值是多少、算法对不对。** 它用一批固定的北京一周数据一次跑完，产出静态结论。

rebuild3 回答的问题是：**新数据来了怎么办。** 具体来说：

| 场景 | 前一阶段的做法 | rebuild3 需要的做法 |
|------|----------------|-------------------|
| 一个新 Cell 第一次出现 | 无需处理——所有 Cell 在一次性聚合中直接产出 | 进入等待池 → 累计观察 → 达标后晋升入库 |
| 一个已知 BS 的 GPS 偏了 | 标记在 BS 异常分类表里，人工分析 | 自动检测偏移量 → 判断是正常漂移/迁移/碰撞 → 按规则分流 |
| 基线画像过时了 | 重新跑一遍全链路 | 定期用增量沉淀的数据刷新画像，不重跑全链路 |
| 解析规则改了 | 从头全量重算 | 判断影响范围 → 局部重算 or 完整回归 |

核心区别：**对象有了生命周期，规则有了版本，处理有了批次。**

## 三、总体分层 / 分域

rebuild3 的逻辑结构分为 **4 个基础层 + 3 个治理能力域**：

```
┌─────────────────────────────────────────────────────────────┐
│                    治理能力域（横切）                          │
│  ┌─────────────┐ ┌──────────────┐ ┌──────────────────────┐  │
│  │ 版本与运行   │ │ 规则与实验    │ │ 日常判定与运营        │  │
│  │ 管控        │ │ 管控         │ │                      │  │
│  └─────────────┘ └──────────────┘ └──────────────────────┘  │
├─────────────────────────────────────────────────────────────┤
│                    基础数据层（纵向）                          │
│                                                             │
│  ┌──────────┐    ┌──────────────┐    ┌──────────────────┐   │
│  │ 接入层    │ →  │ 对象层        │ →  │ 事实层           │   │
│  │ (标准化)  │    │ (LAC/BS/Cell) │    │ (治理后明细)     │   │
│  └──────────┘    └──────────────┘    └──────────────────┘   │
│                         ↓                    ↓              │
│                  ┌──────────────────────────────────┐       │
│                  │ 画像层（三级基线 + 可信度）        │       │
│                  └──────────────────────────────────┘       │
└─────────────────────────────────────────────────────────────┘
```

### 基础数据层说明

| 层 | 职责 | 继承自前一阶段的什么 |
|----|------|---------------------|
| **接入层** | 原始数据标准化：批次管理 → cell_infos JSON 解析 → ss1 紧凑字符串解析 → 26 条 ODS 清洗规则 → 合规标记 → 产出标准化记录 | 字段治理阶段的全部解析规则和清洗规则（100% 继承） |
| **对象层** | 管理 LAC/BS/Cell 三级对象的注册、状态、归属。**Cell 是原子单位**，BS 和 LAC 由 Cell 聚合派生 | 可信库构建阶段的筛选和聚合逻辑（改造为：初始化沿用 LAC→Cell→BS 顺序，增量以 Cell 先行驱动 BS/LAC 联动） |
| **事实层** | 两部分：①**per-Cell 详细库**（每个 Cell 保留最近 N 条明细，滚动窗口，封顶控制存储）②治理后事实表（GPS 修正+信号补齐后的完整记录） | GPS 修正+信号补齐阶段的全部规则（100% 继承） |
| **画像层** | 三级画像基线，**从 per-Cell 详细库计算**（不从全量事实表聚合），参与增量数据的准入判定 | 画像基线阶段的聚合逻辑（需改造为支持定期刷新） |

### 治理能力域说明

| 域 | 职责 |
|----|------|
| **版本与运行管控** | 每次运行记录 run_id，绑定 contract_version（字段解析口径）+ rule_set_version（治理规则口径）+ baseline_version（画像基线口径），支持追溯和回滚 |
| **规则与实验管控** | 异常检测规则的版本化管理、规则实验（用历史数据重跑比较新旧效果）、规则固化 |
| **日常判定与运营** | 增量数据的四分流判定（范围内/漂移/新增/严重异常）、等待/观察池管理、对象晋升/降级记录 |

## 四、从原始数据到治理后事实与画像的完整流向

```
原始报文数据（每 2 小时一批）
    │
    ▼
┌──────────────────────────────────────────────┐
│  ① 接入标准化                                  │
│  注册批次 → 解析 cell_infos JSON（提取 cell_id  │
│  /lac/运营商/制式/信号强度等 22 个字段）           │
│  → 解析 ss1 紧凑字符串（5 元素拆解，信号按制式    │
│  匹配基站）→ 应用 26 条 ODS 清洗规则             │
│  → 标记合规性 → 产出标准化记录                    │
└──────────────────────────────────────────────┘
    │
    ▼
┌──────────────────────────────────────────────┐
│  ② 对象匹配                                    │
│  每条标准化记录的 (运营商, 制式, CellID, LAC)     │
│  → 查找对象库中是否已有该 Cell / BS / LAC         │
│  → 分两路：已在库 / 不在库                       │
└──────────────────────────────────────────────┘
    │                    │
    ▼                    ▼
┌───────────┐    ┌───────────────────┐
│ 已在库     │    │ 不在库             │
│ → 进入     │    │ → 进入等待/观察池   │
│   事实治理  │    │ → 持续累计样本     │
│   流程     │    │ → 达标后评估晋升    │
└───────────┘    └───────────────────┘
    │
    ▼
┌──────────────────────────────────────────────┐
│  ③ 事实治理                                    │
│  GPS 修正：                                    │
│    计算原始 GPS 到所属 BS 中心点的距离             │
│    ≤ 阈值(4G:1000m,5G:500m) → 保留原 GPS       │
│    > 阈值 → 钉成 BS 中心点                      │
│    缺失 → 回填 BS 中心点                        │
│  信号补齐：                                     │
│    同 Cell 时间最近有信号记录补齐                  │
│    不足则退化到同 BS 下最大 Cell 补齐              │
│  异常标记 → 写入治理后事实                        │
└──────────────────────────────────────────────┘
    │
    ▼
┌──────────────────────────────────────────────┐
│  ④ 画像更新                                    │
│  定期用治理后事实重算三级画像                     │
│  画像 = 各对象的空间/信号/时间特征聚合            │
│  → 画像参与下一轮增量的准入判定                  │
│  （"这条记录的 GPS 是否在该 BS 画像的正常范围内？"）│
└──────────────────────────────────────────────┘
```

## 五、初始化流程

### 5.1 目标

用一段历史数据冷启动，产出初始的可信对象库和画像基线。初始化完成后系统进入增量模式。

当前验证计划：用 7 天数据中的前 3 天做初始化，后 4 天做增量验证。未来生产环境建议 14 天。

### 5.2 为什么初始化沿用 LAC→Cell→BS 顺序

初始化处理的是**一次性的大批量历史数据**，此时系统中没有任何已有对象。在这种"从零开始"的场景下，前一阶段（rebuild2）验证过的 LAC→Cell→BS 顺序是高效且成熟的：
- LAC 级过滤先排除大量垃圾（当前北京数据需要严格 GPS 过滤；未来全国数据时此步骤会大幅简化，只需通过运营商白名单过滤海外数据）
- 在可信 LAC 范围内聚合 Cell 效率更高（减少无效计算）
- 已有完整的验证数据可对比

**但这只是初始化的预处理策略**。增量流程将转为 Cell 先行（见 §6）。

### 5.3 步骤

```
历史数据（当前验证：3 天；未来生产：14 天）
    │
    ▼
Step 1: 全量标准化
    把全部原始数据通过接入层：
    解析 cell_infos JSON + ss1 字符串 → 应用 26 条 ODS 清洗规则
    产出：全量标准化记录
    │
    ▼
Step 2: 构建初始可信对象（LAC→Cell→BS 顺序）
    从标准化记录中，按层级逐级筛选：
    (a) 可信 LAC：运营商白名单 → LAC 编码合法 → GPS 区域过滤（可配置参数，
        当前北京严格过滤，未来全国只需排除海外）→ 活跃天数+上报量
    (b) 可信 Cell：限定在可信 LAC 内 → 按 (运营商,制式,CellID,LAC) 聚合 → 过滤
    (c) 可信 BS：按 (制式,BS_ID,LAC) 聚合可信 Cell → 计算 GPS 中位数中心点 → P50/P90 覆盖
    产出：初始对象库（LAC ~1000+ / BS ~19万 / Cell ~57万，状态均设为 trusted）
    │
    ▼
Step 3: 事实治理
    用初始对象库对全量标准化记录做：
    GPS 修正（距离 > 阈值钉成 BS 中心点）→ 信号补齐（同 Cell → 同 BS 回退）→ 标记来源
    产出：治理后事实（预计数千万行）
    │
    ▼
Step 4: BS 异常分类
    对空间散布异常的 BS（P90 偏移 > 1500m 等条件筛出的候选）：
    执行 Cell 质心法分类算法：
    → 每个 Cell 独立计算加权质心和空间跨度
    → 跨度 > 1500m 的标为移动 Cell
    → BS 级只看固定 Cell 质心分布
    → 质心间距 > 1500m 的做设备分裂验证（检查设备交叉率）
    → 产出 6 种分类：normal_spread / single_large / dynamic_bs /
       collision_confirmed / collision_suspected / collision_uncertain
    │
    ▼
Step 5: 完整回归
    用纯化后的对象库 + 异常分类标记，从 Step 1 的标准化记录重跑事实治理
    目的：确保所有记录基于同一套规则处理（避免"部分用旧逻辑、部分用新逻辑"）
    产出：最终一致的治理后事实
    │
    ▼
Step 6: 构建初始画像
    从最终治理后事实按 LAC / BS / Cell 三级聚合：
    - LAC 画像：覆盖范围、信号分布、活跃天数、异常 BS 占比
    - BS 画像：精算中心点、覆盖半径 P50/P90、异常分类标记、可信度
    - Cell 画像：GPS 中心、信号 P10/P50/P90、活跃天数、样本量
    每条画像附带可信度评级（GPS 可信度 + 信号可信度）
    │
    ▼
Step 7: 冻结初始基线
    baseline_version = v1
    所有已入库对象标记为 trusted
    系统正式进入增量模式
```

### 5.4 初始化产出

初始化完成后，系统中有：
- **对象库**：带状态的 LAC/BS/Cell 注册表（初始状态均为 trusted）
- **per-Cell 详细库**：每个 Cell 保留最近 N 条明细（初始化时为全量，后续滚动更新）
- **治理后事实**：GPS 修正+信号补齐后的完整记录
- **画像基线**：从 per-Cell 详细库聚合的三级画像（baseline_version = v1）
- **BS 异常分类**：6 种分类标记

### 5.5 与前一阶段的关系

初始化流程复用前一阶段验证过的全部规则和算法，区别在于：
- 加上了 run_id / version 体系（每次运行可追溯）
- 对象有了明确的状态标记（不再只是统计维表）
- 新增 per-Cell 详细库（为增量模式做准备）
- 产出写入 rebuild3 的正式表结构

## 六、2 小时增量流程（Cell 先行）

### 6.1 核心设计思想

增量流程与初始化的关键区别：**增量以 Cell 为原子单位，自底向上驱动**。

初始化用 LAC→Cell→BS 顺序是因为从零开始需要先画边界。但增量时系统中已有对象库和画像基线，每条新记录自带 cell_id，天然以 Cell 为单位到达和累计。因此增量流程采用 **Cell 先行**：记录→Cell 匹配/累计→Cell 晋升→BS 联动→LAC 联动。

### 6.2 触发

每 2 小时一个批次的新数据到达。

### 6.3 步骤

```
新批次数据（2 小时窗口）
    │
    ▼
Step A: 标准化
    与初始化共用同一套解析规则和 ODS 清洗规则
    （cell_infos JSON 解析 + ss1 字符串解析 + 26 条清洗规则）
    + 记录级合规检查（运营商白名单 + CellID/LAC 格式合法 + GPS 边界框）
    不合规记录直接拒收写异常日志，不进入后续流程
    产出：本批次合规标准化记录
    │
    ▼
Step B: Cell 匹配与分流（Cell 先行的核心）
    每条记录按 (运营商, 制式, CellID, LAC) 匹配 Cell 对象库
    │
    ├── Cell 在库（trusted 状态）
    │   → 对比该 Cell 所属 BS 的画像基线
    │   ├── GPS 在 BS P90 范围内 + 信号正常 → ① 范围内
    │   └── GPS 或信号偏离 → ② 漂移标记
    │   → 记录写入该 Cell 的 per-Cell 详细库（滚动更新）
    │
    ├── Cell 在库（anomaly 状态）
    │   → 仍治理但打异常标签
    │   → 记录写入 per-Cell 详细库
    │
    └── Cell 不在库
        → 记录级合规检查已通过（Step A），进入 Cell 等待池
        → 已在等待池 → 累计样本（更新 GPS 点数/设备数/活跃天数等）
        → 首次出现 → 新建等待记录
        → 记录暂存在等待池明细中
    │
    ▼
Step C: 事实治理（仅已在库 Cell 的记录）
    与初始化共用同一套规则：
    GPS 修正（距离>阈值钉成 BS 中心点）→ 信号补齐（同 Cell→同 BS）→ 打标签
    写入治理后事实表
    │
    ▼
Step D: Cell 晋升评估 + 级联晋升（Cell 先行的关键）

    D1. Cell 晋升评估：
    检查等待池中每个 Cell 的累计数据质量：
    - GPS 点数 ≥ 阈值？（高流量 Cell 可能 1 天就够）
    - 设备数 ≥ 阈值？（确保不是单设备噪声）
    - Cell 质心收敛？（P90 半径 < 某阈值）
    注意：晋升条件基于数据质量，不是固定时间窗口。
    高流量 Cell 可能 1 天即晋升；低流量 Cell 可能需要 90 天累计。

    D2. Cell 晋升执行：
    达标 Cell → 状态改为 trusted
    → 计算该 Cell 的质心（加权平均 GPS）
    → 将暂存的等待池明细回填到 per-Cell 详细库和事实层

    D3. BS 级联晋升：
    当一个 BS_ID 下有 ≥ 1 个 Cell 晋升为 trusted
    → 自动构建/更新该 BS：
       聚合其下所有 trusted Cell 的质心 → 计算 BS 中心点（中位数）
       计算覆盖半径 P50/P90
    → BS 状态标记为 trusted

    D4. LAC 级联晋升：
    当一个 LAC 下有 ≥ N 个 BS 为 trusted
    → LAC 状态标记为 trusted
    （LAC 晋升是派生结果，不是前置门槛）
    │
    ▼
Step E: 异常检测
    对本批次治理后事实 + per-Cell 详细库做异常检测：
    - 已有 Cell/BS GPS 系统性偏移 → 可能迁移
    - BS 碰撞检测（新积累数据触发 Cell 质心法重新评估）
    - 动态 Cell/BS 检测（位置持续变化）
    - Cell 消失检测：如果某 Cell 的 avg_report_interval × 2 倍时间未上报
      → 标记疑似消失 → 如果一个 BS 下所有 Cell 消失 → BS 也标记
    产出：异常事件 → 进入规则实验或人工审查
    │
    ▼
Step F: 画像增量更新（按策略触发，非每批执行）
    - 从 per-Cell 详细库（而非全量事实表）计算画像
    - 按周/月或累积数据量达到阈值时触发画像刷新
    - 刷新后产出新的 baseline_version
```

### 6.4 per-Cell 详细库的滚动机制

| Cell 类型 | 策略 | 说明 |
|-----------|------|------|
| 低流量（如郊区） | 保留全部记录 | 总量本身不大（可能 90 天才几百条） |
| 中流量 | 保留最近 N 条（如 1000 条） | 足以计算准确画像 |
| 高流量（如商业区） | 同样封顶在 N 条 | 超出的旧记录不再保留，画像从最新数据计算 |

这个滚动窗口确保：
- **存储可控**：即使 57 万 Cell，每个最多 N 条，总量有上限
- **画像始终基于最新数据**：自然跟踪物理变化（如基站参数调整）
- **GPS 修正锚点保持时效性**：BS 中心点从最近的 Cell 质心计算

### 6.5 初始化与增量的共享关系

| 模块 | 初始化 | 增量 | 是否共用？ |
|------|--------|------|-----------|
| 标准化（解析 + ODS 清洗） | 全量历史数据 | 2 小时批次 | ✅ 完全共用 |
| 记录级合规检查 | Step 1 内含 | Step A 独立步骤 | ✅ 完全共用 |
| 对象构建/匹配 | LAC→Cell→BS（从零构建） | Cell 先行（撞已有库+级联晋升） | ⚠️ 模式不同：初始化是"批量构建"，增量是"逐条匹配+级联" |
| 事实治理（GPS 修正 + 信号补齐） | 全量记录 | 本批次记录 | ✅ 完全共用 |
| BS 异常分类 | 全量 Cell 质心法 | 增量触发检测 | ✅ 算法共用 |
| 画像生成 | 全量聚合 | 从 per-Cell 详细库刷新 | ✅ 聚合逻辑共用 |
| 完整回归 | 初始化最后一步 | 仅规则变更时触发 | ✅ 逻辑共用 |

## 七、等待 / 观察 / 晋升 / 异常分流的大逻辑

### 7.1 对象状态机

每个对象（LAC / BS / Cell）都有以下状态：

```
         首次出现
            │
            ▼
      ┌──────────┐
      │  待注册   │ ← 刚发现，尚未通过基本合规检查
      │ pending  │    （运营商不在白名单、CellID 不合法等）
      └────┬─────┘
           │ 通过基本合规检查
           ▼
      ┌──────────┐
      │  等待中   │ ← 合规但样本不足，持续累计
      │ waiting  │    需要足够的活跃天数、上报量、GPS 分布稳定
      └────┬─────┘
           │ 累计达标（天数/量/分布均满足）
           ▼
      ┌──────────┐     检测到碰撞/动态/系统性偏移
      │  可信     │ ─────────────────────────────→ ┌──────────┐
      │ trusted  │                                  │ 异常观察  │
      └────┬─────┘ ←──────────────────────────────  │ anomaly  │
           │         异常消解（如碰撞拆分完成）       └──────────┘
           │
           │ 连续 14+ 天无任何上报数据
           ▼
      ┌──────────┐
      │  休眠     │ ← 对象可能已下线
      │ dormant  │    如果又收到新数据则重新评估
      └────┬─────┘
           │ 确认永久消失（如 90 天无数据）
           ▼
      ┌──────────┐
      │  退役     │
      │ retired  │
      └──────────┘
```

### 7.2 不同状态下新数据的处理

| 对象状态 | 新数据处理 | GPS 修正 | 信号补齐 | 写入事实层 |
|----------|-----------|----------|----------|-----------|
| trusted | 正常治理 | 用该 BS 中心点修正 | 正常补齐 | ✅ 写入 |
| waiting | 仅累计样本统计 | 不修正（无可信锚点） | 不补齐 | ❌ 暂存等待池 |
| anomaly | 治理但标记异常 | 视异常类型（碰撞 BS 不用混合中心点） | 正常补齐 | ✅ 写入但打标签 |
| dormant | 触发唤醒评估 | 待评估后决定 | 待评估后决定 | 待评估 |
| retired | 拒收 | — | — | ❌ |

### 7.3 晋升评估——Cell 驱动的级联晋升

与初始化的 LAC→Cell→BS 自顶向下不同，增量模式下晋升是**自底向上级联**的：

```
Cell 累计达标 → Cell 晋升 trusted
    → 该 Cell 所属 BS 是否已注册？
        否 → 自动注册 BS，聚合其下 trusted Cell 质心为 BS 中心点
        是 → 更新 BS（新增 Cell 参与中心点计算）
    → 该 BS 所属 LAC 是否已注册？
        否 → 自动注册 LAC（下属 BS ≥ 阈值时标记 trusted）
        是 → 更新 LAC 统计
```

**Cell 晋升条件**（基于数据质量，非固定时间窗口）：

| 指标 | 条件 | 说明 |
|------|------|------|
| GPS 有效点数 | ≥ N（如 10） | 确保质心计算有统计意义 |
| 设备数 | ≥ 2 | 排除单设备噪声 |
| 质心收敛度 | P90 半径 < 阈值 | 确保 GPS 点不过度离散（移动 Cell 除外） |
| 最低时间跨度 | ≥ 1 天 | 防止瞬时突发数据直接入库 |

高流量 Cell 可能 1 天就满足所有条件；低流量 Cell 可能需要 90 天累计。**网络本身是稳定的，慢慢累计是安全的。**

**BS 级联条件**：下属 ≥ 1 个 Cell 为 trusted → BS 自动构建/更新

**LAC 级联条件**：下属 ≥ N 个 BS 为 trusted → LAC 标记为 trusted（LAC 是派生属性，不是前置门槛）

## 八、缺失值补齐与对象健康度的关系

### 8.1 补齐何时可以直接使用库内结果

| 条件 | 可以直接使用 | 说明 |
|------|-------------|------|
| 对象状态 = trusted | ✅ | BS 中心点可信，可直接用于 GPS 修正和信号补齐 |
| 对象状态 = anomaly，异常类型 = normal_spread 或 single_large | ✅ | 标记低精度但仍可使用 |
| 画像 GPS 可信度 = high 或 medium | ✅ | 画像质量足够支撑补齐 |

### 8.2 补齐何时必须继续累计观察

| 条件 | 不可直接使用 | 说明 |
|------|-------------|------|
| 对象状态 = waiting | ❌ | 对象尚未入库，无可信锚点，补齐无意义 |
| 对象状态 = anomaly，异常类型 = collision_confirmed | ⚠️ | BS 中心点是两个不同物理位置的折中值，不代表任何真实位置 |
| 对象状态 = anomaly，异常类型 = dynamic_bs | ⚠️ | 移动基站，位置在持续变化，中心点不稳定 |
| 画像 GPS 可信度 = none 或 low | ❌ | 数据不足（GPS 点 < 4 或仅单设备），不应作为补齐依据 |

## 九、动态 BS / 碰撞 / 迁移 / 漂移等异常在流程中的位置

### 9.1 异常检测时机

| 异常类型 | 初始化时 | 增量时 | 检测方法 |
|----------|---------|--------|----------|
| **BS 碰撞** | Step 4：全量 Cell 质心法分类 | Step E：新积累数据触发重新评估 | Cell 质心法 v2：每个 Cell 独立计算加权质心，固定 Cell 质心间距 > 1500m 且设备交叉率 < 5% = 碰撞 |
| **动态 BS/Cell** | Step 4：Cell 空间跨度 > 1500m | Step E：位置持续变化 | Cell 加权质心的空间跨度超过 1500m 即判为移动 Cell；BS 下无固定 Cell 即判为动态 BS |
| **GPS 漂移（单条记录）** | Step 3：对比 BS 中心点 | Step B/C：对比画像基线 | 原始 GPS 到 BS 中心点距离 > 阈值（4G:1000m, 5G:500m） |
| **迁移（物理位置永久变化）** | 不适用（初始化只有一个时间快照） | Step E：检测 GPS 中心系统性偏移 | 新批次的画像 vs 基线画像的中心点偏移量超过阈值 |
| **数据不足** | Step 2：样本量过滤 | Step D：等待池评估 | 累计天数/上报量不达标 |

### 9.2 异常处理路径

```
检测到异常
    │
    ├── GPS 漂移（单条记录级，最常见）
    │   → 自动修正为 BS 中心点，标记 GPS 来源
    │   → 不改变对象状态（这是记录级问题，不是对象级问题）
    │
    ├── BS 碰撞（确认级）
    │   → 对象标记为 anomaly，异常子类型 = collision
    │   → 虚拟拆分：按 Cell 质心空间聚类分组，每组视为一个虚拟 BS
    │   → 为每个虚拟 BS 分别计算画像
    │
    ├── 动态 BS（移动基站场景）
    │   → 对象标记为 anomaly，异常子类型 = dynamic
    │   → GPS 修正精度降级（中心点不可靠）
    │   → 画像标记低可信度
    │
    ├── 迁移（物理位置永久变化）
    │   → 如果偏移量较小：直接更新对象画像
    │   → 如果偏移量较大：旧对象标记为 retired，新位置作为新对象重新注册
    │
    └── 数据不足
        → 保持 waiting 状态继续累计
        → 超时未达标 → dormant（休眠）→ 最终 retired（退役）
```

## 十一、现在本地应该先做什么，不该急着做什么

### 11.1 应该先做（P0）

1. **冻结对象状态机定义**：Cell 驱动的级联晋升规则、状态枚举（阈值可暂定）
2. **冻结版本体系**：run_id + contract_version + rule_set_version + baseline_version
3. **设计 per-Cell 详细库**：滚动窗口、封顶策略、与画像计算的关系
4. **设计对象层表结构**：Cell 注册表（含质心、累计统计、状态）+ BS/LAC 由 Cell 聚合派生
5. **用 3 天数据跑通初始化**：LAC→Cell→BS 顺序，产出初始对象库+画像基线
6. **用第 4~7 天数据跑增量**：Cell 先行模式，验证级联晋升和四分流判定
7. **对比验证**：增量跑完的结果 vs 直接用 7 天初始化的结果，评估收敛性

### 11.2 不该急着做

1. ❌ 不急着做完整 9+ 页 UI，先做运行中心 + 对象页 + 日更审查页
2. ❌ 不急着做规则实验室完整闭环，先把主流程跑通
3. ❌ 不急着考虑云端迁移细节（框架预留可迁移性即可）
4. ❌ 不急着做字段血缘图引擎
5. ❌ 不急着做复杂权限和审批流
6. ❌ 不急着为 2G/3G 展开完整设计（方法预留扩展性即可）

### 11.3 验证计划与路径

```
现在（框架分析 ← 你在这里）
    → 冻结 Cell 驱动的状态机 + 版本体系 + per-Cell 详细库设计
    → 用第 1~3 天数据跑初始化（LAC→Cell→BS，验证规则继承正确性）
    → 用第 4~7 天数据跑增量（Cell 先行 + 级联晋升）
    → 对比：增量跑完 vs 直接 7 天初始化 → 评估收敛性
    → 根据对比结果调整晋升阈值和分流规则
    → 冻结为 rebuild3 核心文档
```

**为什么这个验证计划有效**：3 天初始化会产出一个"不完整但可用"的基线（部分低流量 Cell 可能还没入库），然后用 4 天增量验证这些 Cell 是否能通过等待池→晋升流程逐步入库，最终收敛到与 7 天直接初始化相近的结果。差异越小说明增量逻辑越健壮。
