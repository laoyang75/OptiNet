## 1. 文档目的

本文档用于记录 2026-03-24 完成的一轮完整修复与优化，回答以下问题：

1. 本次为什么要修。

4. 修完后当前系统能力到了什么程度。

本文档是“本次优化落地说明”，不是新的方案文档。方案背景和长期架构边界，仍以 `docs/data_warehouse/` 和 `rebuild/docs/` 下既有文档为准。

## 2. 触发背景

第二轮合并审计 [`rebuild/audit/audit_round2_final.md`](../../rebuild/audit/audit_round2_final.md) 给出的结论是“需修改后可用”。问题主要集中在四个方向：

1. 首屏和步骤页直接扫大表做实时聚合，性能模型错误。
2. 前端没有缓存、降级和刷新控制，慢查询会被切页行为不断放大。
3. 工作台 UI 只做了简化骨架，缺失规则、SQL、差异、样本、字段治理、样本研究等核心能力。
4. 中文展示、字段元数据、版本上下文和错误处理不完整，导致可理解性和稳定性不足。

本轮修复的目标不是继续扩需求，而是把已经被审计明确指出的阻塞项全部补齐，使 `rebuild/` 版本达到可直接演示、可继续迭代、可进行后续联调的状态。

## 3. 本次优化目标

本轮目标收敛为三条：

1. 把公共查询路径从“实时扫事实表”切换为“读取工作台快照/缓存”。
2. 把前端工作台从简化壳子补到可用版本，至少覆盖总览、步骤、字段、样本、抽屉四类核心场景。
3. 把运行上下文、版本信息、字段中文名、样本入口和缓存刷新能力纳入统一工作台模型。

## 5. 后端详细改造

### 5.1 引入 `services/` 层

本轮不再让所有 router 直接承担 SQL 组织、字段翻译、缓存和上下文拼装，新增了三个服务模块：

### `labels.py`

职责：

1. 维护表名、层名、异常类型、对象级别、状态码、字段名等中文标签映射。
2. 给前端返回 `code + label` 所需的中文展示值。
3. 统一前后端看到的中文语义，减少页面里散落的硬编码翻译。

### `cache.py`

职责：

1. 提供应用内 `AsyncTTLCache`。
2. 支持 `get / set / invalidate / get_or_set`。
3. 为版本上下文、字段列表、样本列表、步骤列表等接口提供短 TTL 缓存。

### `workbench.py`

职责：

1. 承接工作台快照、版本、字段、样本、规则、SQL、差异等所有中间组织逻辑。
2. 统一初始化默认版本、默认规则、默认 SQL 包、默认契约和默认样本集。
3. 回填 `meta.meta_field_registry`。
4. 生成 `workbench` 快照表数据，供公共接口直接读取。

这一步的核心意义是把“工作台读模型”从 API 层拆出来，使后续继续扩展 P3/P4/D1/D2/D3 时不需要再把复杂 SQL 堆回 router。

### 5.2 快照化改造

公共页面不再直接对大表做实时统计，而是通过 `workbench.py` 生成并读取快照数据。

本轮快照写入了以下四类数据：

| 快照表 | 用途 |
|---|---|
| `workbench.wb_layer_snapshot` | 首页层级记录数快照 |
| `workbench.wb_step_metric` | 步骤摘要、步骤指标、步骤卡片 |
| `workbench.wb_anomaly_stats` | 异常摘要和分布统计 |
| `workbench.wb_rule_hit` | 步骤规则命中、规则说明和命中结果 |

相关核心函数包括：

- `ensure_reference_data()`
- `ensure_field_registry()`
- `ensure_snapshot_bundle()`
- `list_layer_snapshot()`
- `list_step_summary()`
- `list_anomaly_summary()`
- `list_operator_tech_distribution()`
- `list_gps_status_distribution()`
- `list_signal_fill_distribution()`
- `refresh_all()`

其中 `ensure_snapshot_bundle()` 是本轮后端性能修复的核心。它的职责是：

1. 找到当前或指定 `run_id`。
2. 在缺快照或强制刷新时重算快照。
3. 用锁避免并发重复构建同一批次快照。
4. 为公共接口输出 `generated_at`、来源 run 和聚合结果。

### 5.3 默认版本与工作台元数据补齐

为避免工作台进入“有页面但无上下文”的空壳状态，本轮在 `workbench.py` 里增加了最小自举逻辑：

1. 默认规则版本：`R-001`
2. 默认 SQL 包版本：`S-001`
3. 默认契约版本：`C-001`
4. 默认样本集：
   - 碰撞BS样本
   - 动态Cell样本
   - GPS漂移记录样本
   - 信号未补齐样本

对于历史 `wb_run` 数据，如果缺少 `parameter_set_id / rule_set_id / sql_bundle_id / contract_id`，系统会补默认值，保证版本上下文页和总览页总能展示可读信息。

### 5.4 字段注册表与中文元数据

字段治理不是靠前端硬编码实现的，而是由服务层在数据库中补齐字段注册表：

1. 从 `information_schema.columns` 读取表字段。
2. 联合 `pg_description` 读取表与字段注释。
3. 将结果 upsert 到 `meta.meta_field_registry`。
4. 对缺注释字段使用 `labels.py` 里的最小中文映射兜底。

这样做带来的结果是：

1. 前端字段页可以按字段名、中文名、表名、状态做检索。
2. 字段详情可以展示字段定义、关联步骤、字段健康状态和上下游语义。
3. 中文展示不再只依赖页面侧硬编码。

### 5.6 稳定性与运行时配置

本轮还补了几项基础设施：

1. `main.py` 增加全局异常处理。
2. `database.py` 接入连接池参数和 `statement_timeout`。
3. `config.py` 增加 `statement_timeout_ms / pool_size / max_overflow / pool_timeout`。
4. `steps.py` 的 `io-summary` 改为批量查表，避免单步页面额外放大数据库开销。

当前系统对“工作台查询”已经有基础保护，但仍保留一个后续项：`config.py` 里仍存在默认数据库连接串占位值，后续建议继续收敛到 `.env` 驱动并移除源码默认明文。

## 6. 前端详细改造

### 6.1 前端数据层重写

`rebuild/frontend/app.js` 已不再是纯直连接口的简单脚本，而是加上了工作台需要的最小数据层：

1. 内存缓存。
2. `sessionStorage` 缓存。
3. TTL 控制。
4. 强制刷新参数。
5. 请求超时与失败提示。
6. 刷新后统一失效缓存。

页面在大多数情况下会优先读取现有快照，再按需刷新，避免每次切换 hash 路由都重新把公共数据打到后端。

### 6.2 页面结构补齐

本轮重写后的工作台不再只剩一个“步骤页样板”，而是恢复了完整主导航：

2. `P2 步骤工作台`
3. `P3 字段治理`
4. `P4 样本研究`

顶部上下文条新增：

2. Compare Run
3. 参数集

5. SQL 版本
6. 契约版本
7. 基线版本
8. 上次刷新时间
9. `版本抽屉`
10. `刷新缓存`
11. `强制重算`

### 6.3 总览页

总览页现在提供以下信息块：

1. 概览统计卡片。
2. 主链路流程图。
3. 当前 Run 摘要。
4. Compare Run 摘要。

这使首页不再只是展示“存在一些层和步骤”，而是能直接回答“当前版本跑了什么、与对比版本差在哪、有哪些异常需要看”。

### 6.4 步骤页

步骤页已按审计要求补齐 A-H 八个区块和操作区：

| 区块 | 内容 |
|---|---|
| A | 步骤定位与说明 |
| B | 输入输出与上下游表 |
| C | 规则命中与规则说明 |
| D | 局部参数与全局参数 |
| E | SQL 资产与文件入口 |
| F | 数据变化与关键指标 |
| G | 对比 run 差异 |
| H | 关联样本区 |

操作区支持：

1. 查看版本抽屉。
2. 查看 SQL 抽屉。
3. 刷新当前步骤快照。
4. 发起样本重跑登记。

### 6.5 字段治理页

字段治理页提供：

1. 字段总数、健康状态等统计卡片。
2. 按字段名 / 中文名搜索。
3. 按表名过滤。
4. 按生命周期状态过滤。
5. 字段详情抽屉。

这部分直接建立了 UI 和 `meta.meta_field_registry` 之间的连接，后续字段注释补齐后可以继续扩展，但当前已满足“一期最小字段治理能力”。

### 6.6 样本研究页

样本研究页提供：

1. 四类系统样本集入口。
2. 样本预览表。
3. 样本详情抽屉。
4. 从样本直接发起重跑登记。

对应的样本范围已经和步骤页联通，因此样本不再是孤立展示，而是能从步骤进入样本、从样本回到步骤。

### 6.7 抽屉体系

当前已实现三个抽屉：

1. D1 版本与运行抽屉。
2. D2 SQL 查看抽屉。
3. D3 字段 / 样本详情抽屉。

这让总览页、步骤页、字段页、样本页之间形成了足够紧密的交互，而不是每个页面只靠静态区块堆信息。

### 6.8 前端容错修复

在浏览器联调阶段发现：步骤页初版使用 `Promise.all`，只要 `/steps/{step_id}/samples` 这样的非核心接口超时，整个页面就会一起失败。

本轮已修复为：

1. `loadStep()` 使用 `Promise.allSettled`。
2. 只有步骤主体、IO、参数、指标四类核心数据失败才判定整页失败。
3. 规则、SQL、差异、样本允许独立降级。

这一步让步骤页在局部接口失败时仍然可用，避免再次出现“页面卡死在加载中”的问题。
