## 1. 业务流程梳理（已在 PG 上验证）

### 1.1 你的核心链路

基于对 `lac_enbid_project` 全部五层代码和文档的深度阅读，你的处理逻辑可以概括为以下**两阶段闭环**：

```
┌─────────────────────── 冷启动（14+ 天数据，构建可信基线）──────────────────────┐
│                                                                               │
│  ① 原始数据标准化（Layer0）                                                    │
│     网优明细表 → 字段标准化 → PLMN/LAC/Cell 解析                                │
│     产出: Y_codex_Layer0_Lac                                                  │
│                                   ↓                                           │
│  ② 确定可信 LAC（Layer2 Step00~06）                                            │
│     运营商过滤 → LAC 格式清洗 → 北京范围GPS过滤                                  │
│     → 活跃天数/访问量统计 → 可信 LAC 候选库                                      │
│     → LAC→Cell 映射 → 异常 Cell 标记                                           │
│     → 反哺到 Layer0 明细（过滤出合规子集）                                       │
│     产出: Y_codex_Layer2_Step04_Master_Lac_Lib                                │
│            Y_codex_Layer2_Step05_CellId_Stats_DB                              │
│            Y_codex_Layer2_Step06_L0_Lac_Filtered                              │
│                                   ↓                                           │
│  ③ 用 Cell 确定 BS（Layer3 Step30~34）                                         │
│     构建 BS 主库（cell→bs 聚合，按 tech|bs_id|lac 为主键）                       │
│     → Cell GPS 修正（用 BS 中心点纠偏）                                         │
│     → 信号字段补齐 → 交叉比对验证                                               │
│     产出: Y_codex_Layer3_Step30_Master_BS_Library                             │
│                                   ↓                                           │
│  ④ GPS 纯化 + Cell 去噪 + 完整回归（Layer4 Step40~44）                          │
│     以 BS 中心点为参照过滤离群 GPS                                              │
│     → 严重碰撞桶强标注（不删除但标注来源和原因）                                   │
│     → 信号二阶段补齐（同Cell时间最近 → 同BS top Cell）                            │
│     → 用最终清洗好的 Cell 做完整回归                                             │
│     → Final vs Raw 对比评估                                                    │
│     产出: Y_codex_Layer4_Final_Cell_Library                                   │
│                                   ↓                                           │
│  ⑤ 构建三级基线画像（Layer5 Step50~52）                                         │
│     LAC 画像: 覆盖范围/GPS稳定性/信号覆盖率/活跃天数/异常标记                      │
│     BS  画像: GPS中心/范围/碰撞标记/移动Cell标记/多运营商共享                      │
│     Cell 画像: GPS位置/信号分布/移动Cell标记/样本量                               │
│     产出: Y_codex_Layer5_Lac_Profile                                          │
│            Y_codex_Layer5_BS_Profile                                          │
│            Y_codex_Layer5_Cell_Profile                                        │
│                                                                               │
└───────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────── 日常运营（每日增量比对）────────────────────────────────┐
│                                                                               │
│  新数据 → 标准化 → JOIN 基线表（Layer5 画像）                                    │
│     ├─ 匹配且在范围内 → 沉淀为可信数据                                          │
│     ├─ 匹配但指标漂移 → 标记为变化数据（Cell 位移/参数变化）                       │
│     ├─ 新 Cell/BS → 标记为新增，进入观察池                                       │
│     └─ 超出基线范围 → 标记为异常/噪点，不入库                                    │
│                                                                               │
│  定期（周/月）用沉淀数据重跑基线 → 更新画像                                       │
│                                                                               │
└───────────────────────────────────────────────────────────────────────────────┘
```

### 1.2 关键特点总结

| 特点 | 说明 |
|---|---|
| **层层收敛、互相印证** | LAC→Cell→BS→GPS，然后 BS 反向纯化 Cell，不是单向流水线 |
| **完整回归** | 用最终 Cell 把全量数据再处理一遍，闭环验证 |
| **可审计** | 每步都有行数/指标/对比报告（Obs Mart 机制） |
| **基线驱动** | Layer5 的画像即为基线，用于日常新数据的准入判断 |
| **标记不删除** | 异常数据（碰撞/漂移/编码异常）标注原因但不丢弃，下游按标记筛选 |

### 4.2 缺失/需补充（⚠️ 需要行动）

| 缺口 | 影响 | 解决方案 |
|---|---|---|
| **无 DROP/ALTER TABLE** | 表结构演进无法通过 MCP | MaxCompute Console 或 odpscmd |

| **PG→MC 数据通道** | 源数据在 PG | Python 脚本 + MCP INSERT；后期用 DataX/Tunnel |
| **跨项目表** | `1568935375600958:` 前缀表不可读 | 申请权限或不依赖 |

## 5. 实施路径（三阶段）

### Phase 1：在 MaxCompute 上复现基线构建（当前可做）

**目标**：将 PG 上验证通过的 Layer0→Layer5 逻辑迁移到 MaxCompute

> SQL 方言差异注意：PG 的 `DO $$...$$` 动态 SQL、Session Setting、`percentile_cont` 等需要改写为 MaxCompute 兼容语法。

1. 建 ODS 层 → 从 PG 导出 14 天数据
2. 建 DIM 层 → 迁移可信维表
3. 建 DWD 层 → 改写 Layer2 + Layer4 SQL
4. 建 DWS 层 → 改写 Layer5 画像 SQL
5. 建 ADS 层 → 实现质量门控

### Phase 2：日常增量比对体系

1. 日增量接入 → 新分区
2. 增量清洗 → 过 DWD 规则
3. 基线比对 → JOIN DWS 画像
4. 变化检测 → 产出 ADS 表
5. 数据沉淀 → 合并到 DWD

### Phase 3：自动化与监控

1. DataWorks 调度上线
2. 基线自更新（周/月）
3. 告警体系
4. 元数据管理

## 6. PG SQL → MaxCompute SQL 主要改写点

| PG 特性 | MaxCompute 替代 |
|---|---|
| `DO $$...$$` | 静态 SQL 或 Script Mode |
| `SET codex.xxx` | SQL 变量或参数表 |
| `percentile_cont(0.5)` | `PERCENTILE(col, 0.5)` |
| `serial / bigserial` | `ROW_NUMBER()` |
| `timestamptz` | `DATETIME` + 时区处理 |
| `jsonb` | `STRING` + `GET_JSON_OBJECT` |
| `CREATE INDEX` | 无用户侧索引 |
| 分片 Session Setting | 天然分区 + Cluster Table |
| `LATERAL` | 子查询改写 |

## 7. 需要确认的事项

1. **数据来源**：14 天的源数据是从哪里取的？是 MaxCompute 跨项目表，还是有其他推送接口？  
2. **日更数据通道**：每天的新数据怎么到达？是否已有 DataWorks 任务？
3. **是否保留 PG 做开发**：PG 做开发验证 → MaxCompute 做生产？还是完全切到 MC？
4. **层次优先级**：建议从 DIM + DWS（基线画像）开始，你倾向先搭哪一层？
5. **数据量级**：Layer0 源表大约多少行？（影响分区策略和成本）
