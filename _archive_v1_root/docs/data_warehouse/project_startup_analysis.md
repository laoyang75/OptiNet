## 二、你的四个核心问题解答

### 问题 1：应该怎么启动？

> **建议：以"闭环最小可跑"为目标启动，而不是从文档整理开始。**

评审报告提了很多 P0 文档（对象状态机、语义分层图、入库决策矩阵），但如果你先写文档再动手，很可能写一个月文档还没跑起来。

**推荐的启动路径是：**

```
Phase 0 (1-2天): 搭建可视化监控台 → 让你能直观看到现有数据全貌
Phase 1 (3-5天): 在现有 Layer5 画像基础上，先跑通"模拟增量→比对→分流"的最小闭环
Phase 2 (1周+):  在闭环跑通后，再回头补文档、理清对象定义、拆分语义层
```

> 核心逻辑：**先让系统"活"起来（能感知变化并做出反应），再在运行中校准结构。**

### 问题 3：继续在本地 PG 上做，还是去服务器？

> **建议：阶段性策略。先在本地 PG 完成逻辑验证，稳定后再迁移。**

| 维度 | 本地 PG | 服务器 (MaxCompute等) |
|------|---------|----------------------|
| **开发效率** | ✅ 极高，秒级反馈 | ❌ 提交等待，调试困难 |
| **数据规模** | ⚠️ 1.3亿可用但吃力 | ✅ 弹性扩展 |
| **UI 联调** | ✅ localhost 直连 | ⚠️ 网络延迟 |
| **成本** | ✅ 免费 | ⚠️ 计算费用 |
| **生产准备** | ❌ 单机不可靠 | ✅ 适合生产 |

**分阶段执行：**

```
阶段 1 (现在): 本地 PG — 所有逻辑开发、UI开发、闭环验证
                ↓ 
阶段 2 (闭环跑通后): 把核心 SQL 移植到 MaxCompute
                     把 UI 改为连接远程数据源
                ↓
阶段 3 (稳定运行后): 全面迁移到服务器
```

**关键原则：保持 SQL 的可移植性。** 当逻辑在本地 PG 验证通过后，SQL 迁移到 MaxCompute 的工作量应该是最小的。

### 问题 4：先梳理清楚现有的？还是直接搭建闭环平台？

> **建议：直接搭建最小闭环，在搭建过程中自然完成梳理。**

理由：

1. **你的治理链路已经跑完了一遍** — Layer0→5 全量数据已经在 PG 里了，画像也已经产出，说明冷启动逻辑是可用的
2. **梳理文档是无底洞** — 评审报告提的 7 个问题都很好，但如果先全部写成文档再动手，你可能会陷入"文档完美主义"
3. **闭环过程会倒逼你梳理** — 当你试图让"新数据 vs 基线画像"的比对跑起来时，你必须明确：对象主键是什么、状态是什么、分流规则是什么——这就是评审报告说的 P0 内容，但它是在实操中被倒逼出来的

## 三、推荐的启动行动计划

### Phase 0：可视化基座 (1-2天)

**目标**：搭建一个能看到"现有治理全貌"的 UI 监控台

1. **数据资产盘点页面**
   - 列出所有 Layer0-5 表的行数、字段数、最后更新时间
   - 展示数据血缘流转图（哪个表依赖哪个表）
   
2. **画像快览页面**  
   - BS 画像核心指标：总数、活跃率、碰撞率、GPS可用率
   - Cell 画像核心指标：总数、覆盖 LAC 数、信号完整率
   - LAC 可信库指标：总数、覆盖占比

3. **健康检查页面**
   - obs_* 系列表有数据吗？
   - 各 Layer 之间行数是否合理？
   - 关键字段缺失率

### Phase 1：最小闭环 (3-5天)

**目标**：在现有画像基线上，跑通"模拟增量→比对→分流"

1. **模拟增量数据注入**
   - 从 Layer0 里抽取出最近 1-2 天的数据作为"新增量"
   - 或从原始数据源再拉一批新日期数据
   
2. **增量比对引擎（核心 SQL）**
   - 新 Cell 撞 Layer5_Cell_Profile → 已知/未知
   - 新 BS 撞 Layer5_BS_Profile → 范围内/漂移
   - GPS 偏差超阈值 → 标记为异常
   
3. **分流结果落表 + UI 展示**
   - 四分流：已知且范围内 / 已知但漂移 / 新对象 / 严重异常
   - UI 上能看到每次比对的分流统计

4. **入库决策（简单版）**
   - 已知且范围内 → 自动沉淀（更新画像统计）
   - 其他三类 → 先标记不入，等人工确认

### Phase 2：结构化升级 (1周+)

**目标**：在闭环能跑后，补齐评审报告要求的结构化内容

- P0-1：写对象定义与状态机（基于 Phase 1 实际跑出来的分流逻辑）
- P0-2：拆开四类语义（在 PG 里建对应的 schema 或表前缀）
- P0-3：把特例旁路独立出来
- P0-4：把入库判定正式化
