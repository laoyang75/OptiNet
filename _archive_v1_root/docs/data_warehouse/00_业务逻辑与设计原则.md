# 网优数仓：业务逻辑与设计原则

> 版本：v1.0 | 日期：2026-03-18  
> 来源：与业务负责人讨论确认  
> 定位：本文件是数仓构建的**行为准则**，所有后续表结构和 SQL 逻辑必须遵循本文件中的原则

---

## 一、核心哲学

### 原则 1：有效 cell_id = 有效上报，充分利用

`cell_id` 是终端与基站之间的**真实连接记录**。如果 cell_id 有效（格式正确、运营商合规、在可信 LAC 范围内），那么这条上报就是有效的——即使它的 GPS 漂移了。

**推论**：
- 不因为 GPS 漂移而丢弃整条记录
- GPS 的价值低于 cell 连接事实本身
- 信号强度（RSRP/RSRQ/SINR 等）是和 cell 连接绑定的，与 GPS 准不准无关，必须完整保留

### 原则 2：修正优于丢弃

当一条记录的某个字段有问题时，优先**修正**（用更可靠的数据源覆盖），而不是丢弃整条记录。只有在无法修正时才标记为异常（仍保留，由下游决定是否使用）。

**体现**：
- GPS 漂移 → 用 BS 中心点修正，不丢弃
- 信号缺失 → 用时间最近的同 Cell/同 BS 数据补齐，不丢弃
- 严重碰撞桶 → 仍回填 GPS 但强标注原因，由下游按标签筛选

### 原则 3：层层收敛、互相印证

数据处理不是单向流水线，而是**多层交叉验证的闭环**：
- 正向：LAC → Cell → BS → GPS（从大到小逐级确认）
- 反向：BS → Cell（用聚合后的 BS 画像反过来纯化 Cell）
- 回归：用纯化后的结果重新处理全量数据（确保一致性）

### 原则 4：基线驱动的日常运营

一次性冷启动完成后，日常不再重跑全部流水线。改为：新数据与基线对比 → 好数据沉淀 → 变化发现 → 定期刷新基线。

## 二、第一步：冷启动——构建可信基线（静态处理）

### 输入

14 天（或更长）的网优明细数据。时间窗口越长，统计特征越稳定，基线越可靠。

### 处理链路与每一步的原因

#### Step 1：确定可信 LAC

**做什么**：从全量数据中筛出"结构可信"的 LAC 集合。

**为什么先做 LAC**：
- LAC 是最大粒度的网络分区标识（一个 LAC 下有多个 BS，一个 BS 下有多个 Cell）
- 如果一个 LAC 本身就是假的（测试网络、国外漫游、异常编码），它下面的所有 Cell/BS 数据都不可信
- 先确定 LAC，等于先画出"可信的地盘边界"，后面所有分析都在边界内做

**怎么做**：
1. 运营商过滤：只保留五大核心 PLMN（移动/联通/电信/广电/铁路专网）
2. LAC 编码过滤：剔除 NULL、异常长度、已知黑名单值
3. 位置过滤：只保留 GPS 落在目标区域（北京）的数据
4. 稳定性过滤：活跃天数 ≥ 阈值 且 上报量 ≥ 阈值

**产出**：可信 LAC 维表（`dim_lac_trusted`）

---

#### Step 2：用可信 LAC 确定可信 Cell

**做什么**：在可信 LAC 范围内，筛出"行为稳定"的 Cell 集合。

**为什么在 LAC 确定之后做**：
- Cell 挂在 LAC 下面，一个假 LAC 会导致其下所有 Cell 数据不可信
- 限定在可信 LAC 范围内分析 Cell，避免垃圾传播

**怎么做**：
1. 只在可信 LAC 的明细数据中，按 `(运营商, 制式, cell_id, lac)` 聚合
2. 统计每个 Cell 的活跃天数、GPS 空间分布、上报量
3. 剔除：位置离散（GPS 点四处散落）、仅偶尔出现（活跃天数太少）、样本量太小的 Cell
4. 标记异常（一个 Cell 出现在多个 LAC 下 → 映射异常）

**产出**：Cell 统计维表（`dim_cell_stats`）

---

#### Step 3：用 Cell 反推 BS（基站）

**做什么**：将可信 Cell 按 BS 维度聚合，构建 BS 主库。

**为什么是 Cell→BS 而不是反过来**：
- 原始数据里有 `cell_id`，但没有直接的 `bs_id`
- `bs_id` 是从 `cell_id` 按规则推算的（4G: cell_id ÷ 256, 5G: cell_id ÷ 4096）
- 所以物理上先有 Cell 数据，再聚合出 BS

**怎么做**：
1. 按 `(制式, bs_id, lac)` 聚合所有可信 Cell
2. 计算 BS 中心点（所有 Cell 的 GPS 中位数，而不是平均值——中位数抗离群点更强）
3. 计算 BS 覆盖半径（GPS 点到中心距离的 P50/P90）
4. 碰撞检测：如果一个 BS 下的 Cell 的 GPS 聚落分成了不连通的几簇 → 可能是编码碰撞（不同物理基站被映射到同一个 bs_id）
5. 标注风险等级（碰撞/编码异常/正常）

**BS 中心点为什么更可靠**：
- 一个 BS 通常有多个 Cell（例如 3 个 120° 扇区）
- 每个 Cell 有多条上报，每条带一个 GPS 点
- GPS 漂移是**偶然的**，大量正常点会将偶然漂移平滑掉
- 聚合后的中心点比任何单条记录的 GPS 都更稳定

**产出**：可信 BS 维表（`dim_bs_trusted`，含中心点/范围/碰撞标记/风险等级）

---

#### Step 4：用 BS 修正 GPS + 标注异常

**做什么**：拿 BS 中心点作为参照系，校验和修正每条原始记录的 GPS。

**为什么到这一步才修正 GPS**：
- GPS 修正需要一个"锚点"（参照系），这个锚点就是 BS 中心点
- BS 中心点必须先通过 Step 1~3 的层层收敛才可靠
- 如果直接用原始数据计算 BS 中心点再去修正，"脏数据修正脏数据"没有意义

**怎么做**：
1. 每条原始记录通过 `(制式, bs_id, lac)` 关联到 BS 主库
2. 计算原始 GPS 到 BS 中心点的距离
3. 判定：
   - 距离 ≤ 阈值（4G: 1000m, 5G: 500m）→ GPS 可信，保留原值
   - 距离 > 阈值 → GPS 漂移，**钉成 BS 中心点**
   - 原始 GPS 缺失 → 同样**回填 BS 中心点**
4. **所有情况都保留整条记录**（cell_id 有效 → 上报有效）
5. 信号强度字段**原封不动保留**（信号与 GPS 准不准无关）

**严重碰撞桶的处理**：
- 如果某个 BS 被标记为碰撞（Step 3），该 BS 下的记录仍然回填 GPS
- 但强标注：`is_severe_collision=true` + 碰撞原因 + GPS 来源
- 由下游决定是否降权或过滤

**这一步同时发现**：
- 哪些 Cell 的 GPS 大多数在漂移（→ 可能是室内覆盖或特殊场景）
- 哪些 Cell 的位置在持续变化（→ 移动 Cell，如高铁/公交）
- 哪些 BS 存在碰撞问题

---

#### Step 5：信号数据补齐

**做什么**：对信号字段（RSRP/RSRQ/SINR/dBm 等）进行二阶段补齐。

**为什么要补齐信号**：
- 部分上报的信号字段是空的（终端未上报或解析失败）
- 但 cell 连接是有效的，信号数据丢了可惜
- 同一个 Cell 在相近时间点的信号状况应该是相似的

**怎么做（二阶段策略）**：
1. **同 Cell 补齐**：在同一个 cell_id 下，找时间最近的有信号记录作为 donor，逐字段 COALESCE 补齐
2. **同 BS 补齐**：如果同 Cell 没有 donor，退化到同 BS 下数据量最大的 Cell，找其时间最近有信号记录补齐
3. 记录补齐来源和 donor 标识，便于追溯

---

#### Step 6：用 BS 反向纯化 Cell（去噪闭环）

**做什么**：用 BS 的画像反过来审视下属 Cell，标记可疑 Cell。

**为什么要"反向"**：
- Step 3 是 Cell→BS（正向聚合），现在反过来
- 如果一个 BS 标记有碰撞 → 其下 Cell 也应被标记
- 如果一个 Cell 的 GPS 行为和所属 BS 的整体画像严重不一致 → 这个 Cell 可能是噪点
- **这就是闭环**：正向聚合确认 BS，反向用 BS 纯化 Cell

---

#### Step 7：完整回归

**做什么**：用纯化后的 BS/Cell 结果，把**全量原始数据**从头再处理一遍。

**为什么要"从头再跑"**：
- 前面每一步可能修改了 BS 库、Cell 列表、碰撞标记等
- 如果只在前面的中间产物上叠加修改，会出现"部分数据用旧逻辑、部分用新逻辑"的不一致
- 完整回归确保最终产出是**基于同一套规则、统一处理**的

**为什么从 Layer0 全量出发（而不是 Layer2 过滤后子集）**：
- 核心原则"有效 cell_id = 有效上报"
- Layer2 的过滤是 LAC 级别的（去掉不合规的运营商/编码/区域）
- 通过 LAC 合规检查的全量数据都应该进入 GPS 修正和信号补齐
- 不应该因为"更保守的过滤"而损失有效上报

---

#### Step 8：构建三级基线

**做什么**：从完整回归后的最终明细库中，按 LAC / BS / Cell 三级聚合，生成画像。

**为什么是最后一步**：
- 基线必须基于"够干净"的数据计算
- 如果用未纯化的数据算均值和范围，基线本身就是偏的
- 完整回归后的数据是最可靠的

**每级画像包含什么**：

| 级别 | 主键 | 核心指标 |
|---|---|---|
| LAC | (运营商, 制式, lac) | 上报量、活跃天数、下属 BS 数、Cell 数、GPS 覆盖率 |
| BS | (运营商, 制式, bs_id, lac) | GPS 中心点、覆盖半径(P50/P90)、碰撞标记、移动 Cell 数、共享运营商 |
| Cell | (运营商, 制式, cell_id, lac) | GPS 位置、信号中位数(RSRP/RSRQ)、活跃天数、样本量、移动标记 |

**这些画像就是"基线"**——日常运营中，新数据的每个字段都要和这些指标对比。

## 三、第二步：日常运营——增量比对与数据沉淀

### 每日处理流程

```
当日新数据到达
    ↓
标准化（和 Layer0 相同的 PLMN/LAC/Cell 解析）
    ↓
按 (运营商, 制式, cell_id, lac) JOIN 基线表
    ↓
┌──────────────────────────────────────────────┐
│  分四种情况处理：                                │
│                                              │
│  ① 基线已知 + 在范围内                          │
│     → 好数据，沉淀入库                          │
│     → GPS 修正（同冷启动逻辑）                   │
│     → 信号数据保留                              │
│                                              │
│  ② 基线已知 + 指标漂移                          │
│     → 标记为"变化数据"                          │
│     → Cell GPS 中心偏移了？                      │
│       → 可能基站搬迁/天线调整                    │
│     → 信号参数变了？                            │
│       → 可能频段调整/功率变更                    │
│     → 仍然沉淀，但打标签                        │
│                                              │
│  ③ 基线中不存在（新 Cell / 新 BS）               │
│     → 标记为"新增"                              │
│     → 进入观察池                                │
│     → 连续多天出现且稳定 → 可升级为正式基线       │
│                                              │
│  ④ 超出基线范围（严重异常）                      │
│     → 标记为噪点/异常                           │
│     → 不入正式库，但保留在异常日志中              │
│                                              │
└──────────────────────────────────────────────┘
    ↓
好数据沉淀 → 用于未来基线刷新
变化记录 → 用于运维关注和基站变更追踪
新增记录 → 观察池，待晋升
```

### 基线更新策略

- **触发条件**：定期（周/月）或当沉淀数据量足够时
- **方式**：用近 N 天的沉淀数据重算三级画像
- **不是全量重跑**：只更新画像统计值（均值/范围/活跃天数），不重跑冷启动的全链路

### 日常运营的核心价值

| 发现 | 含义 | 业务动作 |
|---|---|---|
| Cell GPS 中心偏移 | 可能基站物理搬迁 | 推送给网优工程师确认 |
| 新增 Cell/BS | 新建基站 | 更新网络拓扑 |
| 数据量骤增/骤降 | 设备异常或覆盖调整 | 排查告警 |
| BS 下 Cell 分布变化 | 频段/天线调整 | 更新覆盖模型 |
| 信号强度整体变化 | 功率调整或干扰 | 优化网络参数 |
