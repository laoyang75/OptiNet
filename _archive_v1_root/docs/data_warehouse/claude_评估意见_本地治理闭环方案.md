## 一、数据库现状快照（PG17，端口 5433）

### 1.3 关键发现

1. **Layer0 为空**：`Y_codex_Layer0_Lac` 在 PG17 中行数为 0。这意味着当前 PG17 上无法从原始数据重跑冷启动。如果本地闭环方案要在 PG17 上实施，Layer0 数据需要重新导入或从 PG15 迁移。
2. **obs 体系已有骨架**：10 张 obs 表结构完整，`run_registry` 有 2 条记录，`gate_result` 18 条，`quality_metric` 30 条——说明观测体系已经试跑过，但尚未进入常态化使用。
3. **SMOKE/MCP_SMOKE 测试表**：存在 5 张带 `__L3_SMOKE` 或 `__MCP_SMOKE` 后缀的测试表，说明之前做过分阶段验证。这些表在重构时可以考虑清理。
4. **BS Profile 字段丰富**（54 个字段）：包含碰撞标记、移动 Cell 标记、信号补齐统计、GPS 漂移统计、多运营商共享等，说明 Layer5 画像的信息密度已经很高。

## 二、方案定位评估：完全认同

### 2.1 "本地治理闭环一期"而非"正式数仓一期"——这个定位是正确的

理由：

1. **核心逻辑已验证但结构未固化**。Layer0-5 的冷启动链路已经跑通了 3000 万行数据，说明治理哲学和处理顺序是成立的。但正如评审文档指出的，当前 Layer/Step 结构更像"探索历史"而非"长期语义结构"。在这个状态下迁云，迁的是不稳定的中间态。

2. **三块缺失能力必须在本地补齐**。字段治理、规则实验、UI 观察——这三块不是"锦上添花"，而是从"一次性冷启动脚本"走向"可持续运营系统"的必要条件。在本地 PG 上补这些能力的迭代效率远高于在 MaxCompute 上做。

3. **MaxCompute 的 MCP 能力有硬限制**。无 DROP/ALTER TABLE、无调度、无 UDF——这些限制在逻辑尚未稳定时会严重拖慢迭代。

### 2.2 四个目标的优先级建议

方案提出四个目标，我认为它们之间有隐含的依赖关系：

```
目标 3（字段注册/元数据、规则重跑、异常研究）
    ↓ 依赖
目标 2（冷启动 + 伪日常运营闭环跑通）
    ↓ 依赖
目标 1（保留并整理现有冷启动逻辑）
    ↓ 最终服务于
目标 4（为云端迁移提供清晰逻辑契约）
```

这意味着目标 1 和目标 2 是串行的前置条件，目标 3 可以和目标 2 并行推进，目标 4 是自然产物不需要单独追求。

## 三、三块新增能力的逐项评估

### 3.1 字段/元数据治理系统

**认同点：**

- 字段解析就是治理入口。PLMN/LAC/Cell 解析是 Layer0 的第一步操作，如果字段语义变了（比如新运营商编码、新制式标识），下游所有逻辑都会受影响。把它前移为独立模块而非附属工作，这个判断是对的。
- "看到 4 种变化"（结构变化、健康变化、语义变化、影响变化）的设计思路非常好，它把字段治理从"被动登记"变成了"主动发现"。

**补充建议：**

1. **字段治理和主流程的接口要显式定义**。方案 5.4 节说"冷启动跑前先确定契约版本"，但没有说明契约版本和 run 的绑定机制。建议：每次 run 启动时必须锁定一个 `contract_version`，run 过程中字段解析规则不可变，run 完成后如果发现字段异常，进入下一轮契约版本迭代。

2. **字段健康度的计算时机**。建议分两个时点：
   - **批次级**：每次新数据导入时，自动计算该批次的字段健康度（空值率、异常值率）并写入快照
   - **趋势级**：跨批次对比健康度变化趋势，触发告警

3. **建议增加"字段依赖图"的数据结构**。方案提到"下游影响范围"，但没有给出具体建模方式。建议用一张 `field_dependency` 表：`(field_id, dependent_process, dependent_field, dependency_type)` 来显式描述字段之间的依赖链。这会让"哪些规则需要重跑验证"变得可自动推算。

### 3.2 异常研究/规则实验/重跑比较系统

**认同点：**

- "发现→样本集→假设→版本化→重跑→比较→固化"这个完整闭环设计得很好。
- "最关键的不是规则表，而是比较能力"——这句是整个 issue_rule_lab 域的灵魂，我完全认同。

**补充建议：**

1. **比较能力需要明确"比什么"**。方案列了 5 个比较维度（修复了哪些问题、是否引入误伤、影响范围、画像收敛性、区域特异性），建议再加一个：**对基线版本的影响**——即如果新规则被固化，baseline_v_next 相比 baseline_v_current 会有多大变化。这是决定"是否值得固化"的最终判断依据。

2. **规则版本和 run 版本的关系要理清**。建议区分：
   - `rule_version`：单条规则的版本
   - `rule_set_version`：一组规则的组合版本（类似代码的 release tag）
   - `run` 绑定的是 `rule_set_version`，不是单条规则版本

3. **重跑的粒度控制**。全量重跑成本很高（Layer4 Final 27GB），建议支持两种重跑模式：
   - **采样重跑**：只取问题样本集涉及的对象范围重跑，用于快速验证假设
   - **全量重跑**：规则固化前的最终验证，确保没有全局副作用

4. **问题分类体系建议增加"未知类"**。方案列了 8 种问题类型（GPS 漂移、碰撞 BS、全室内 BS 等），但实际运行中一定会出现不属于任何已知类型的异常。建议预留 `unknown / under_investigation` 类别，避免强制归类导致的误判。

### 3.3 UI 观察与运营系统

**认同点：**

UI 不是辅助工具，而是治理系统的必要组成部分。理由：

1. **复杂治理逻辑的可理解性**。8 个逻辑域、多步链路、正向/反向/回归——这种复杂度仅靠 SQL 查询是无法形成全局认知的。UI 是把离散信息组织成可理解画面的唯一手段。
2. **日常运营的操作入口**。伪日更的分流结果需要人工审查，问题需要人工确认，规则实验需要人工决策——这些都需要 UI 承载。
3. **团队协作与移交**。这个项目的知识目前高度集中在你一个人身上，UI 是降低移交成本最有效的方式。
4. **数仓建设的标配需求**。元数据治理、数据血缘、画像展示、规则管理——在任何正规数仓体系中都需要对应的可视化页面。

**对 9 个页面的逐一评估和补充：**

| # | 页面 | 是否必要 | 补充建议 |
|---|------|---------|---------|
| 1 | 运行中心 | **必要** | 建议增加"运行对比"功能——选两次 run 直接看差异 |
| 2 | 字段治理中心 | **必要** | 这是元数据治理的核心入口，建议包含字段搜索和影响分析功能 |
| 3 | 数据流转/血缘页 | **必要** | 建议做成交互式：点击某个节点能看到该步骤的行数、字段变化、耗时 |
| 4 | 对象观察页 | **必要** | 建议支持"对象对比"——选一个对象看它在不同 run/不同基线版本下的画像变化 |
| 5 | 问题中心 | **必要** | 建议增加"问题聚合视图"——按问题类型统计分布，而非只看队列列表 |
| 6 | 规则实验室 | **必要** | 建议和问题中心做双向链接：从问题跳到对应规则实验，从规则实验跳回问题样本 |
| 7 | 基线中心 | **必要** | 建议增加"基线版本时间线"——纵向看基线如何随时间演进 |
| 8 | 日更模拟/差异审查 | **必要** | 这是本地阶段最高频使用的页面，建议做得最细致，包含可钻取的分流统计 |
| 9 | 审计与版本页 | **必要** | 建议作为横向页面，从任何其他页面都能跳转到对应的审计记录 |

**额外建议：考虑增加第 10 个页面——"全局总览/驾驶舱"**

作为所有页面的入口和系统健康度的一屏汇总。内容包括：
- 最近一次运行状态和关键指标
- 各域数据量变化趋势
- 未处理问题数量
- 基线版本和规则版本当前状态
- 字段健康度告警

这不是新需求，而是你现有 `dashboard.html` 的升级版，在新架构下重新定义它的内容。

## 四、8 个逻辑域的逐一评估

### 4.1 整体评价

8 域设计是对评审建议 7 层架构的落地版本，且做了两个重要改进：
- 把评审没有给出归属的 Step 中间产物，用 `work_run` 域承接——解决了"中间态放哪里"的问题
- 新增 `meta_contract` 域——把字段治理从"后续再说"前移为显式域

与评审建议的对应关系总体清晰，但有几个值得深入讨论的点。

### 4.2 逐域评估

#### meta_contract（字段与契约治理域）

**认同**。独立出来是对的。

**建议调整：** 方案列了 10 类对象（数据源注册、表注册、字段注册、字段映射、解析规则注册、衍生字段定义、字段健康度快照、字段变更日志、下游依赖关系、输出契约版本）。这 10 类的内部也有依赖关系，建议按以下顺序构建：

```
第一批（骨架）：数据源注册 → 表注册 → 字段注册 → 字段映射
第二批（规则）：解析规则注册 → 衍生字段定义 → 输出契约版本
第三批（观测）：字段健康度快照 → 字段变更日志 → 下游依赖关系
```

#### raw_source（原始接入域）

**认同**。保留原始数据是完整回归的前提。

**关键问题：** 当前 PG17 的 `Y_codex_Layer0_Lac` 为空。在新架构中，raw_source 的原始明细表必须有数据才能跑通任何后续流程。这是启动开发前必须先解决的基础问题——要么从 PG15 重新迁移 Layer0 数据，要么从原始数据源重新导入。

**建议增加：** "批次质量摘要"之外，还应有"批次元数据表"——记录每个批次的来源、时间窗口、导入时间、行数、文件路径等。这是 run 追溯的起点。

#### work_run（运行中间域）

**认同定位，但需要明确生命周期管理策略。**

这个域会存储所有 Step 中间产物，是 8 个域中数据量增长最快的。当前仅 Layer3+Layer4 的中间表就已经有 ~60GB。如果每次 run 都保留中间产物，磁盘会快速膨胀。

**建议：**
- 定义 work_run 表的保留策略：最近 N 次 run 的中间产物保留，更早的自动清理或归档
- 中间产物表名应包含 run_id，而非覆盖式写入（这样才能做 run 间对比）
- 考虑用 PG 的 schema 隔离不同 run 的中间产物：`work_run.run_20260322_001.lac_candidates`

#### core_entity（可信对象域）

**认同方向，但有一个核心问题需要回答。**

方案说"Cell 更适合先走统计 + 状态注册，不要过早做成僵硬的正式维表"。这与评审建议指出的"Cell 的可信主档语义不够清楚"是同一个问题。

我的看法是：**Cell 应该有正式的注册表，但它的"可信"含义和 LAC/BS 不同。**

- LAC 的可信 = 编码合规 + 在目标区域 + 活跃稳定 → 是一个**边界判定**
- BS 的可信 = 中心点可计算 + 非严重碰撞 + 覆盖范围合理 → 是一个**空间锚点判定**
- Cell 的可信 = cell_id 有效 + 有稳定的 LAC 归属 + 行为统计正常 → 是一个**连接事实判定**

Cell 不需要像 BS 那样有一个"精确的可信空间位置"，但它需要有一个明确的注册状态：这个 Cell 是否被系统承认为有效观测来源。

**建议：** core_entity 中包含 `cell_registry` 表，字段不需要很多，核心是主键 + 状态 + 归属 LAC + 归属 BS + 置信度 + 版本。Cell 的丰富统计信息仍然放在 baseline_profile 的 Cell Profile 中。

#### fact_governed（治理后事实域）

**认同。**

**补充：** 方案列了"运行版本 / 基线版本 / 规则版本"，建议再加 `contract_version`（契约版本），这样这张事实表就完整记录了它被哪个版本的字段解析规则、治理规则、基线参照系产生。这是未来审计和回溯的关键字段。

同时建议保留对 raw_source 的溯源键（raw_batch_id + raw_row_id），使任何一条治理后事实都能追溯到原始记录。

#### baseline_profile（画像与基线域）

**认同。**

**补充建议：**
1. 基线版本表应记录该版本的"产出 run_id"和"输入数据窗口"，这样才能回答"这个基线是基于哪些数据、什么规则算出来的"。
2. 建议增加"画像差异表"——每次基线刷新时自动计算 baseline_v_new vs baseline_v_old 的差异摘要，沉淀为独立记录。这会直接服务于"基线中心"UI 页面。

#### issue_rule_lab（问题与规则实验域）

**认同设计，是最有创新价值的新增域。**

**补充建议：**
1. 问题事件表建议包含 `source_domain` 字段——标明这个问题是从哪个域发现的（冷启动的 work_run？日更的 decision_ops？字段健康度的 meta_contract？还是人工标记？）。
2. 规则定义表中的 `sql_text` 字段要能支持版本对比（diff），建议存储为完整 SQL 而非增量。
3. "审批/固化记录"在本地阶段可以简化为"人工确认记录"（谁在什么时间确认了这条规则从实验升级为正式），不需要做成审批工作流。

#### decision_ops（日常判定与运营域）

**认同。**

**补充建议：**
1. "对象晋升/降级记录"建议和 core_entity 的状态变更做双向联动——decision_ops 产出晋升/降级决定，core_entity 执行状态变更，变更记录同时写入两边。
2. 建议增加"决策依据快照"——记录每次晋升/降级/拒绝决定的输入数据快照（该对象在判定时的关键指标值），而不是只记录结果。这是后续回溯"当时为什么这样判"的关键。

## 五、对方案中几个关键判断的意见

### 5.1 "不急着用数仓正式形态约束自己"——同意

方案第 8 节的结论是"用数仓逻辑组织，但不用数仓正式形态约束"。这是合理的。具体来说：

- 8 个逻辑域在语义上已经覆盖了 ODS/DWD/DIM/DWS/ADS 的功能
- 但本地阶段不需要严格遵循数仓命名规范，因为你需要快速迭代
- 当逻辑稳定后，从 8 域到数仓层级的映射是自然的：

```
raw_source        → ODS
core_entity       → DIM
fact_governed     → DWD
work_run          → 临时态，不进正式数仓
baseline_profile  → DWS
decision_ops      → ADS（部分）
issue_rule_lab    → 本地研究态，不直接进数仓
meta_contract     → 元数据层（独立于数仓分层）
```

### 5.2 技术栈选择——同意，有一个补充

PG + FastAPI + Next.js + 独立 worker 的技术栈对这个阶段是合理的。

**补充：建议在 PG 中使用 schema 来物理隔离 8 个逻辑域。**

```sql
CREATE SCHEMA meta_contract;
CREATE SCHEMA raw_source;
CREATE SCHEMA work_run;
CREATE SCHEMA core_entity;
CREATE SCHEMA fact_governed;
CREATE SCHEMA baseline_profile;
CREATE SCHEMA issue_rule_lab;
CREATE SCHEMA decision_ops;
```

好处：
- 表名不需要加域前缀，保持简洁
- 权限可以按 schema 控制
- 查询时可以用 `search_path` 切换上下文
- 未来迁云时，schema 直接对应数仓的逻辑分层

### 5.3 调度方案——同意，有一个强调

方案说"关键不是选哪个调度器，而是所有任务都带 run_id / rule_version / baseline_version / contract_version"。这个判断非常重要。

**强调：run_id 体系应该是第一个落地的数据结构**，因为所有域的表都会引用它。建议 `run_registry` 表至少包含：

```
run_id              -- 唯一标识
run_type            -- cold_start / pseudo_daily / rule_experiment / regression
status              -- pending / running / completed / failed
contract_version    -- 字段契约版本
rule_set_version    -- 规则集版本
baseline_version    -- 基线版本（冷启动时为 null）
input_window_start  -- 输入数据起始时间
input_window_end    -- 输入数据结束时间
input_batch_ids     -- 输入批次列表
parameters          -- jsonb，运行参数
started_at
completed_at
summary             -- jsonb，运行摘要（各阶段行数、耗时等）
```

## 六、方案中需要进一步明确的问题

以下问题不影响方案方向，但会影响具体表结构设计和开发实施。建议在开发前讨论确认。

### 6.1 对象状态机的转移条件

方案在 core_entity 中提到了 `trusted / observe / changed / anomaly / retired` 五种状态，但没有定义转移条件。需要回答：

| 转移路径 | 触发条件是什么？ |
|---------|----------------|
| `observe → trusted` | 连续 N 天出现且稳定？N 是多少？"稳定"的量化定义？ |
| `trusted → changed` | GPS 中心偏移超阈值？信号参数变化超阈值？哪些指标的变化算"changed"？ |
| `trusted → anomaly` | "严重异常"的具体判定标准？ |
| `changed → trusted` | 是自动回归还是需要人工确认？ |
| `changed → retired` | 多长时间无回归自动退役？ |
| `anomaly → observe` | 异常解除的条件？ |
| any → `retired` | 长期不出现自动退役？阈值是多少？ |

这不是一次性定义，可以先定一版粗糙的初始规则，在伪日更中根据实际数据逐步修正。但初始版本必须在开发前存在。

### 6.2 冷启动和伪日更的共享边界

冷启动走完整 Step1-8 链路，伪日更走"增量撞基线→四分流"。二者的共享边界应该是：

```
冷启动产出：core_entity（对象注册）+ fact_governed（治理后事实）+ baseline_profile（画像基线）
         ↓ 这三个域的表就是伪日更的输入
伪日更消费：raw_source（新增量）JOIN baseline_profile → decision_ops（分流结果）→ 反馈回 core_entity
```

需要确认：伪日更是否需要写入 fact_governed？即"好数据沉淀"是否意味着直接追加到治理后事实域？如果是，那 fact_governed 会同时包含冷启动产出和日更沉淀的数据，需要有 `source_run_id` 来区分来源。

### 6.3 完整回归的触发条件

冷启动必须做完整回归（Step 7），但伪日更不做。那么以下场景是否需要触发完整回归？

- 规则实验固化了一条新规则
- 基线刷新（用近 N 天沉淀数据重算画像）
- 字段解析规则发生变更
- 发现了大量异常需要重新评估

建议：区分"局部重算"和"完整回归"。只有当规则变更影响全局（如 BS 中心点计算逻辑变化）时才需要完整回归；如果只是新增了一条特例旁路规则，只需要对受影响的对象范围做局部重算。

### 6.4 Layer0 数据缺失的解决方案

当前 PG17 的 Layer0 为空，但 raw_source 域需要原始数据。需要确认：
- 是从 PG15 重新迁移 Layer0？（数据量可能在 50GB 以上）
- 还是从原始文件重新导入？
- 还是暂时用 Layer2_Step06（合规过滤后的 2179 万行）作为 raw_source 的起点？

第三种方案会丢失被过滤掉的原始数据，无法做真正的完整回归。但如果只是为了先跑通闭环流程，可以作为临时方案。

## 七、与前序文档的一致性检查

### 7.1 与「数据治理项目评审与架构建议」的关系

评审建议了 7 层语义结构（原始接入 / 可信实体 / 规则判断 / 画像 / 变化识别 / 入库决策 / 数仓资产沉淀）。本方案的 8 域与之高度一致，主要差异：

| 差异点 | 评审建议 | 本方案 | 评估 |
|-------|---------|-------|------|
| 字段治理 | 放在后续阶段 | 前移为独立域 meta_contract | **本方案更好**，字段治理确实是前置能力 |
| 中间产物 | 没有明确归属 | 独立为 work_run 域 | **本方案更好**，解决了"Step 临时表放哪"的问题 |
| 变化识别+入库决策 | 分为两层 | 合并为 decision_ops | **可接受**，本地阶段合并更高效，未来可拆 |
| 数仓资产层 | 独立为第 7 层 | 不单独建域 | **合理**，本地阶段不需要这一层 |
| 规则实验 | 归入规则/判断层 | 独立为 issue_rule_lab | **本方案更好**，研究态和生产态分开更清晰 |

### 7.2 与「project_startup_analysis」的关系

该文档建议"先让系统活起来，在运行中校准结构"，本方案的思路与之一致（先在本地跑通闭环，再逐步固化迁云）。但本方案比 startup_analysis 更系统化——后者偏"快速搭最小可用"，本方案在补齐结构的同时保持了务实节奏。

### 7.3 评审文档 P0 清单的覆盖情况

| 评审 P0 项 | 本方案是否覆盖 | 说明 |
|-----------|--------------|------|
| P0-1 对象定义+状态机 | **部分覆盖** | core_entity 有状态设计，但转移条件未定义 |
| P0-2 四类语义拆开 | **已覆盖** | 8 域拆分已经完成了语义分离 |
| P0-3 特例旁路正式化 | **部分覆盖** | issue_rule_lab 承接特例研究，但特例的路由策略未正式定义 |
| P0-4 日常入库判定矩阵 | **部分覆盖** | decision_ops 域存在，但判定矩阵的具体规则未写 |

## 八、整体结论与建议

### 8.1 方案成熟度评价

这是一份**方向正确、结构清晰、节奏合理**的方案。它成功地：

1. 找到了正确的阶段定位（本地治理闭环，而非直接上数仓）
2. 继承了评审建议的核心思想并做了更落地的改造
3. 补齐了三块之前缺失的关键能力（字段治理、规则实验、UI 观察）
4. 给出了合理的技术栈和阶段路线

### 8.2 开发前需要补齐的三份短文档

在进入开发之前，建议先补齐以下三份简短文档（不需要长篇大论，每份 1-2 页即可）：

1. **对象状态机定义**：LAC / Cell / BS 各自的状态枚举 + 转移条件 + 转移动作
2. **入库判定矩阵**：伪日更四分流的具体量化标准（什么算"范围内"、什么算"漂移"、什么算"严重异常"）
3. **run_registry 与版本体系设计**：run 表结构 + version 表结构 + 它们与 8 个域的关联方式

这三份文档定义了系统的"骨架接口"，有了它们，8 个域的表结构设计和 UI 页面的数据需求就能并行展开。

### 8.3 开发启动建议

启动顺序建议（不含时间估计）：

```
第一步：建 8 个 schema + run_registry + version 体系
        → 这是所有后续工作的基础设施

第二步：raw_source 域 + 解决 Layer0 数据问题
        → 没有原始数据，后续一切无法验证

第三步：把现有 Layer2-5 核心产出映射到 core_entity / fact_governed / baseline_profile
        → 不重写逻辑，先建视图或重命名，让新结构能跑起来

第四步：冷启动流程改造（加 run_id，写入新域）
        → 验证新结构能承载已有逻辑

第五步：meta_contract 最小集 + issue_rule_lab 最小集
        → 补齐新增能力的骨架

第六步：伪日更闭环（增量比对 → 四分流 → decision_ops）
        → 跑通日常运营循环

第七步：UI 全部页面
        → 基于已经跑通的数据和流程构建可视化

以上各步之间不一定严格串行，有些可以并行推进。
```

### 8.4 我的一个额外建议

当前所有文档都是面向"人类阅读"的叙事体。建议在开发过程中，同步维护一份**机器可读的系统契约文件**（比如 YAML 或 JSON），内容包括：

- 8 个域的表清单和表间依赖
- 每个 run_type 的输入/输出域定义
- 对象状态机的形式化表达
- 字段解析规则的版本化注册

这份文件既是开发时的参照，也是未来自动化验证（比如"这次 run 是否产出了所有应该产出的表"）的基础。
