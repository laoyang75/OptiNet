## 一、项目背景

### 1.1 业务场景

我们在做**移动网络优化**。手机等终端设备在使用网络时，会上报它连接的基站信息（LAC、Cell ID、信号强度等），同时 GPS 定位记录终端的物理位置。

我们的目标是：**为每个基站（BS）、每个小区（Cell）、每个位置区（LAC）建立可信的画像档案**（位置、覆盖范围、信号特征等），用于：
- 网络覆盖质量评估
- 基站变更发现（搬迁、新增、参数调整）
- 数据质量监控

### 1.2 核心数据实体关系

```
运营商（PLMN，如中国移动 46000）
  └── 制式（4G / 5G）
       └── LAC（位置区编码，一个城市区域的逻辑分区）
            └── BS（基站，由 cell_id 按规则推算：4G 取 cell_id÷256，5G 取 cell_id÷4096）
                 └── Cell（小区，一个基站通常有多个扇区/频段，每个扇区是一个 Cell）
```

每条原始上报记录包含：
- `运营商ID`（PLMN 代码）
- `cell_id`（十进制小区编码，**硬事实**——终端确实连接了这个小区）
- `LAC`（位置区编码）
- `GPS`（经纬度，**可能漂移**——受室内、遮挡等影响）
- `信号强度`（RSRP、RSRQ、SINR、dBm 等，与连接事实绑定，是真实的）
- `时间戳`

### 1.3 数据量级

- 当前一周数据约 **3000 万行**（北京区域）
- 14 天数据用于基线构建
- 未来扩展到全国时数据量会成倍增长

## 二、核心设计哲学（已确认）

### 哲学 1：有效 cell_id = 有效上报

cell_id 是终端与基站的真实连接记录，不会漂移。只要 cell_id 有效，这条上报就应该被利用——即使 GPS 飘了。GPS 可以修正，信号数据必须保留。

### 哲学 2：修正优于丢弃

优先修正（用更可靠的数据源覆盖有问题的字段），而不是丢弃整条记录。异常数据标记原因但不删除，由下游按标签筛选。

### 哲学 3：层层收敛、互相印证

不是单向流水线，而是：
- 正向聚合：LAC → Cell → BS → GPS（从大到小逐级确认）
- 反向纯化：BS → Cell（用聚合结果反向审视细节）
- 完整回归：纯化后重新处理全量数据（确保一致性）

### 哲学 4：基线驱动

冷启动时用多天数据构建可信基线，之后每天只做增量比对（新数据 vs 基线），不重跑全链路。

## 三、评审前必读：业务常识与已确认事项

> **请先阅读本节再看流程。** 以下是在之前评审中被错误质疑的点，它们要么不是问题，要么是已知局限（有专门的应对机制）。请评审者**不要在这些方向上发散**，将注意力集中在我们真正需要帮助的架构问题上。

### 3.1 不是问题的事项

| # | 曾被质疑的问题 | 为什么不是问题 |
|---|---|---|
| 1 | **Step 1 用 GPS 过滤 LAC 是否有循环依赖？**（GPS 在 Step 4 才修正，"GPS 漂移到北京外的 LAC 会被误杀"） | **不会。** LAC 是**非常大的粒度**——一个 LAC 下有大量 Cell 和 BS，对应的上报记录数以万计。纯 GPS 过滤是统计层面的（"大多数记录的 GPS 在北京范围内"），个别记录的 GPS 漂移不会影响 LAC 级别的统计判定。这和 Cell 级别（单条记录就可能决定去留）完全不同 |
| 2 | **Cell ID 的 BS 推算规则会不会变？** | **不会。** 4G 的 `cell_id ÷ 256` 和 5G 的 `cell_id ÷ 4096` 是行业标准编码规则，运营商几乎不可能调整。即使极小概率发生变化，后续的日常持续治理流程也会通过增量比对自然发现和处理 |
| 3 | **14 天窗口是否足够建立稳定基线？** | **当前阶段不需要讨论。** 我们正在跑 demo 验证流程，14 天足够。生产环境的窗口策略属于运营阶段的调优，不影响架构设计 |

### 3.2 已知局限（有应对机制，但需要架构上有位置放置处理逻辑）

| # | 已知局限 | 为什么存在 | 当前/计划应对 |
|---|---|---|---|
| 1 | **Step 3 碰撞检测使用未修正的 GPS** | 这是一个"先有鸡还是先有蛋"的经典问题——BS 中心点需要 GPS 来计算，而 GPS 修正又依赖 BS 中心点。但在**一般情况下**，GPS 漂移是偶发的、少量的，去掉极值后的中位数/中心点仍然准确。**绝大多数 BS 的首次计算结果就是可靠的** | 对于确实分成多簇的 BS，要么是其下 Cell 本身有问题（移动 Cell），要么是真碰撞。**这类特例需要引入独立的判断逻辑**——这是架构需要预留位置的地方 |
| 2 | **BS 聚合 GPS 不一定比单条记录准**（全室内 BS、碰撞 BS 等场景） | 当聚合的输入数据本身就系统性偏差时（例如某 BS 下所有 Cell 都在大型商场内，GPS 全部漂移），中位数也会偏。碰撞 BS 的中心点更是两个物理位置的"折中"，不代表任何真实位置 | **这类特征明显的异常 BS 需要独立的处理路径**，不能走通用的"钉成中心点"逻辑。架构上需要为这类特例预留旁路。长期看，引入外部基站坐标库（铁塔数据）做交叉验证是最好的解法 |

### 3.3 核心统计学前提

> 评审者在评估每一步时，请牢记以下前提：

1. **GPS 漂移是偶发的，不是系统性的。** 在正常室外场景下，大多数上报的 GPS 是准确的，只有少数因室内/遮挡/多径等因素漂移。因此，聚合统计量（中位数、去极值后的均值）天然就是对漂移的校正。
2. **粒度越大越不怕漂移。** LAC 级别的判断（万级样本）比 BS 级别（百级样本）更抗噪，BS 级别比 Cell 级别（十~百级样本）更抗噪。这是从 LAC 开始逐级收敛的核心原因。
3. **数据处理的目标是"足够可信的基线"，不是"100% 精确的真值"。** 基线用于日常比对，只要能识别"变化"和"异常"就够了。

## 四、当前处理流程（已在 PostgreSQL 上实现并验证）

### 4.1 冷启动：基线构建（8 步）

#### Step 1：确定可信 LAC
- **输入**：14 天全量原始数据
- **逻辑**：运营商过滤（五大 PLMN）→ LAC 编码格式过滤 → GPS 区域过滤（北京）→ 活跃天数+上报量过滤
- **产出**：可信 LAC 维表
- **原因**：LAC 是最大粒度的分区标识。先画出"可信的地盘边界"，后面所有分析限定在此范围内
- **⚠️ 不要质疑 GPS 过滤的循环依赖**：见 §3.1 #1

#### Step 2：用可信 LAC 确定可信 Cell
- **输入**：可信 LAC 范围内的明细数据
- **逻辑**：按 (运营商, 制式, cell_id, lac) 聚合 → 统计活跃天数/GPS 分布/上报量 → 剔除位置离散/偶发/低样本 Cell → 标记一个 Cell 出现在多个 LAC 的映射异常
- **产出**：Cell 统计维表
- **原因**：Cell 挂在 LAC 下，限定可信 LAC 范围内分析避免垃圾传播

#### Step 3：用 Cell 反推 BS
- **输入**：可信 Cell 数据
- **逻辑**：按 (制式, bs_id, lac) 聚合 → 计算 BS 中心点（中位数 GPS）→ 计算覆盖范围（P50/P90）→ 碰撞检测 → 标注风险等级
- **产出**：可信 BS 维表（含中心点/碰撞标记/风险级别）
- **原因**：BS 从 cell_id 推算而来（物理先有 Cell 数据，再聚合出 BS）。BS 中心点是多 Cell 多记录聚合的结果，比单条 GPS 更可靠
- **⚠️ 碰撞检测的已知局限**：见 §3.2 #1。分成多簇的 BS 属于特例，需要独立处理路径，**评审时请关注架构如何为这类特例预留位置**

#### Step 4：用 BS 修正 GPS
- **输入**：全量原始数据（Layer0）+ BS 维表
- **逻辑**：每条记录关联 BS → 计算原始 GPS 与 BS 中心点距离 → 在范围内则保留原 GPS，超出范围则钉成 BS 中心点 → 信号字段原封保留
- **产出**：GPS 已修正的全量明细
- **原因**：BS 的聚合 GPS 是更可靠的参照系。单条 GPS 可能漂但 cell 连接是真实的，修正 GPS 比丢弃记录更好
- **注意**：此步从全量原始数据出发（不是从 Step 2 过滤后的子集），因为"有效 cell_id = 有效上报，一条都不应该丢"
- **⚠️ BS 中心点不可靠的特例**：见 §3.2 #2。对碰撞 BS 和全室内 BS，"钉成中心点"是错误的，需要独立处理路径

#### Step 5：信号补齐
- **输入**：Step 4 产出
- **逻辑**：同 Cell 时间最近策略补齐 → 不足则退化到同 BS 下数据量最大 Cell 时间最近补齐 → 记录补齐来源
- **产出**：信号已补齐的全量明细（最终明细库）
- **原因**：信号缺失是上报问题不是连接问题，同场景同时段的信号应该相似，补齐比留空有价值

#### Step 6：BS 反向纯化 Cell
- **输入**：最终明细 + BS 维表
- **逻辑**：将 BS 的碰撞/异常标记传递到下属 Cell → 检查 Cell GPS 与所属 BS 画像的一致性 → 标记移动 Cell（位置在变化的 Cell，如高铁/公交场景）
- **产出**：Cell 级异常标记
- **原因**：这是闭环——正向聚合确认 BS 后，反向用 BS 审视 Cell

#### Step 7：完整回归
- **输入**：原始全量数据 + 最终的 BS/Cell/标记
- **逻辑**：用纯化后的所有维表和标记，从头再跑一遍处理逻辑
- **产出**：基于统一规则的最终明细库
- **原因**：避免"部分数据用旧逻辑、部分用新逻辑"的不一致

#### Step 8：构建三级基线画像
- **输入**：最终明细库
- **逻辑**：按 LAC/BS/Cell 三级聚合 → 计算均值、中位数、P50/P90 范围、活跃天数、信号覆盖率、碰撞标记、移动标记
- **产出**：三级基线画像表（LAC Profile / BS Profile / Cell Profile）
- **原因**：必须基于纯化后的数据才能得到靠谱的基线

### 4.2 日常运营：增量比对

每日新数据到达后：
1. 标准化（同 Step 1 的 PLMN/LAC/Cell 解析）
2. 按主键 JOIN 基线画像表
3. 分四种情况处理：
   - **已知 + 范围内** → 好数据，沉淀入库
   - **已知 + 指标漂移** → 标记变化（Cell 位移/信号变化），仍沉淀但打标签
   - **不存在**（新 Cell/BS）→ 进入观察池，连续稳定出现后升级
   - **超出范围**（严重异常）→ 标记噪点，不入正式库
4. 定期（周/月）用沉淀数据重算基线

## 五、当前实现的问题和架构反思

### 5.1 历史背景

当前系统是**从微观一点一点推导出来的**——先解决某个字段的某个问题，再解决下一个，逐步形成了 5 层（Layer_0 到 Layer_5）的结构。这导致：

### 5.2 已知工程问题

1. **层次过多，边界模糊**：当前 5 层（Layer_0~Layer_5），加上内部的 Step 编号（Step00~Step52），总共 20+ 个处理步骤。部分步骤之间的依赖关系不够显式，存在"隐含逻辑"（例如 Layer4 跳过 Layer2 直接从 Layer0 取数，原因是设计哲学决定的，但代码结构上不直观）。

2. **正向/反向逻辑交织**：Step 3（Cell→BS 正向聚合）和 Step 6（BS→Cell 反向纯化）分散在不同层（Layer_3 和 Layer_4/5），理解成本高。

3. **"回归"步骤的位置不明确**：完整回归是核心设计，但在当前实现中不是一个独立的、显式的步骤，而是隐含在 Layer4 "从 Layer0 出发"的设计中。

4. **冷启动 vs 日常运营未分离**：当前代码只实现了冷启动（一次性跑全流程），日常增量比对的逻辑还没有代码实现。

5. **PG SQL 的工程约束**：当前用了大量 PG 特有语法（DO 块、Session Setting、serial 类型等），迁移到 MaxCompute 需要系统性改写。

6. **辅助系统（UI/API/Obs Mart）与核心逻辑混合**：为了调试和可视化，构建了 API 服务和 UI 页面（dashboard/issues/patches 等），但这些是辅助工具，不是核心数据逻辑。迁移时需要明确区分。

7. **特例处理无独立位置**：碰撞 BS、全室内 BS、移动 Cell 等特例目前和主流程混在一起处理。架构上没有为这类需要独立判断逻辑的场景预留清晰的旁路。

### 5.3 希望评审解决的核心问题

> **在保持上述设计哲学不变的前提下，如何设计一套更简洁、更合理的分层架构？**

具体的评审问题：

1. **分层合理性**：当前 5 层 + 20 步是否可以精简？有没有更好的分法？特别是正向聚合和反向纯化是否应该在同一层内闭环？

2. **"回归"步骤的最佳位置**：完整回归应该作为显式的独立步骤，还是融入到某个层的设计中？

3. **冷启动与日常运营的架构分离**：如何让同一套表结构和 SQL 同时支持冷启动（批量）和日常（增量）两种模式？

4. **从 PG 单机到分布式数仓（MaxCompute）的架构适配**：有哪些设计模式需要调整（例如分区策略、JOIN 策略、无索引环境下的优化）？

5. **数据质量体系应该是内嵌在层中还是独立成层**？当前的 Obs Mart（质量门控表）是独立于 Layer 之外的，这合理吗？

6. **特例处理的架构位置**：碰撞 BS、全室内 BS、移动 Cell 等需要独立判断逻辑的场景，应该在架构的什么位置处理？作为主流程的分支？独立的旁路层？还是后置的修正步骤？

- 4G：`Cell_ID ÷ 256`（取整）
