# 本地治理闭环一期 UI 草图设计说明

> 版本：v1.0 | 日期：2026-03-22
> 依据：《最终整合意见文档》（已冻结）
> 定位：低保真草图级设计说明，不是高保真视觉稿

---

## 1. 全局导航与信息架构

### 1.1 导航结构

采用**左侧固定侧栏 + 顶部面包屑**的经典后台布局。

```
┌─────────────────────────────────────────────────────┐
│  WangYou 治理观察台           [全局过滤器] [版本抽屉] │
├────────┬────────────────────────────────────────────┤
│        │  面包屑：运行中心 > Run #0037              │
│ 侧栏   │────────────────────────────────────────────│
│        │                                            │
│ 运行中心│              主内容区                      │
│ 日更审查│                                            │
│ 对象中心│                                            │
│ 问题中心│                                            │
│ 字段治理│                                            │
│        │                                            │
│────────│                                            │
│ 规则实验│                                            │
│(折叠组) │                                            │
│        │                                            │
└────────┴────────────────────────────────────────────┘
```

### 1.2 导航分组逻辑

| 分组 | 页面 | 说明 |
|------|------|------|
| **核心观察** | 运行中心、日更审查、对象中心、问题中心、字段治理 | 一期主力页面，侧栏常驻 |
| **实验区** | 规则实验 | 一期最小版，侧栏折叠组 |
| **全局入口** | 版本/审计抽屉、基线摘要区块 | 不做独立页，以抽屉或区块形式嵌入 |

### 1.3 一期不做独立页的内容

| 内容 | 一期处理方式 | 理由 |
|------|-------------|------|
| 基线中心 | 作为"对象中心"内的区块 + "日更审查"页的参照面板 | 基线信息总是围绕对象和日更被查看，独立页跳转成本高 |
| 审计与版本 | 全局右侧抽屉，任何页面可呼出 | 版本信息是全局上下文，不是独立业务 |
| 数据流转/血缘 | 运行详情页内的"阶段流转"区块 | 一期血缘需求仅限于"某次 run 的数据怎么流转的" |
| 全局驾驶舱 | 不做 | 冻结文档明确后置 |

## 2. 一期页面清单

共 **6 个页面** + **2 个全局组件**：

| 序号 | 页面 | 类型 | 核心用途 |
|------|------|------|---------|
| P1 | 运行中心 | 列表 + 详情 | 看每次 run 跑了什么、到哪一步、版本绑定 |
| P2 | 日更审查 | 分流看板 + 钻取 | 看伪日更四分流结果，审查入库决策 |
| P3 | 对象中心 | 搜索 + 详情 | 按 LAC/BS/Cell 查看状态、画像、问题 |
| P4 | 问题中心 | 队列列表 + 详情 | 集中管理各类异常问题 |
| P5 | 字段治理 | 列表 + 详情 | 字段健康度、变更、影响范围 |
| P6 | 规则实验 | 列表 + 对比 | 规则假设、重跑、新旧对比（最小版） |
| G1 | 版本/审计抽屉 | 全局抽屉 | 查看当前版本体系、run 历史、变更记录 |
| G2 | 全局过滤器 | 顶栏组件 | 运营商、制式、LAC 范围筛选 |

## 4. 每页关键操作

| 页面 | 查看操作 | 动作操作 | 跳转操作 |
|------|---------|---------|---------|
| **运行中心** | 查看 run 列表、详情、阶段行数 | 发起新运行（冷启动/伪日更/局部重算） | → 问题中心（告警）、→ 对象中心（阶段产物） |

## 9. UI 设计原则总结

### 9.1 核心设计原则

1. **可解释性优先**：每个页面都要能回答"系统为什么这样判"。对象状态、日更分流、问题标记，都必须有可追溯的来源和理由。

2. **以对象/问题/基线/版本为中心**：页面命名和信息组织围绕业务语义，不以 Step 表、Layer 编号或内部 SQL 结构命名。用户看到的是"对象状态""问题队列""基线画像"，不是"Step3 产物""Layer4 表"。

3. **闭环跳转**：从"问题 → 对象 → 样本 → 规则实验 → 重跑结果 → 入库判定"必须形成完整可点击的链路，用户不需要手动拼 ID 到另一个页面查询。

4. **观察诊断为主，配置为辅**：一期页面以"看"和"判"为核心操作（查看状态、审查入库、确认问题），不做复杂的配置编辑界面。唯一的"写"操作集中在：发起运行、审查决策、创建实验、固化规则。

5. **渐进式信息展示**：主页面只展示摘要和关键指标；详情通过点击钻取或抽屉展开；样本数据通过二级页查看。避免一个页面塞太多信息。

### 9.2 交互原则

6. **轮询刷新**：一期不使用 WebSocket。运行中心和日更审查采用定时轮询（建议间隔 10-30 秒），其余页面按需刷新。页面顶部显示"最后刷新时间"和手动刷新按钮。

7. **全局过滤器保持**：运营商/制式/LAC 过滤器状态通过 URL 参数传递，跨页面保持一致。

8. **抽屉优于弹窗**：字段详情、版本信息、对象快速预览使用右侧抽屉，不使用模态弹窗（抽屉不阻断主页面操作）。

9. **列表页统一交互**：所有列表页使用一致的分页、排序、过滤模式。默认每页 20 条，支持按关键列排序。

### 9.3 约束原则

10. **不做的事情**：不做实时监控大屏、不做复杂权限控制、不做审批流、不做拖拽式配置、不做全量血缘图。

11. **最小操作面**：每个页面的"写操作"不超过 3-4 个。一期操作清单：
    - 运行中心：发起运行
    - 日更审查：审查入库、观察池晋升
    - 问题中心：确认问题、创建实验、关闭问题
    - 规则实验：创建实验、发起重跑、固化规则
    - 对象中心：人工标记问题
    - 字段治理：无（纯观察）

12. **后端驱动前端**：所有页面数据通过 API 获取，不直连数据库。API 返回的数据结构即为页面渲染契约，前后端以 payload 契约（第 7 节）为对齐依据。
