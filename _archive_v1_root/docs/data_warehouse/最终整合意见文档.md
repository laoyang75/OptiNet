# 最终整合意见文档

## 1. 整体结论

三份文档已经基本形成稳定共识：当前阶段应定义为“本地治理闭环一期”，而不是“正式数仓一期”。这一阶段的任务不是继续扩体系，而是把已经验证过的冷启动主链、伪日更闭环、规则补充与审查边界固定下来，形成后续研发和 UI 草图可直接使用的稳定骨架。

本轮结论是：

- 保留现有冷启动主逻辑，不推翻 `LAC -> Cell -> BS -> GPS -> 信号补齐 -> 完整回归 -> 基线画像` 的顺序。
- 保留 8 域方案作为本地一期的工程主结构，但明确它是“本地逻辑域骨架”，不是正式云端数仓分层。
- 当前阶段只需要补齐 3 份最小骨架短文档；除这 3 类内容外，没有新的结构性补充必要。
- UI 一期不建议按 9 页全部展开，应按“运行、审查、对象、问题、字段”五类核心能力分期落地。
- 所有容易把项目做重的建议，都应明确后置，避免本地一期滑向正式平台化建设。

## 2. 三份文档的共识部分

三份文档已经形成的明确共识如下：

1. 当前阶段定位明确：这是本地治理闭环一期，不是正式数仓一期，不应提前进入云端化、平台化、审批流化。
2. 现有治理哲学成立，应继续保留：`有效 cell_id = 有效上报`、`修正优于丢弃`、`先建可信对象再修正事实`、`基线驱动日更判定`。
3. 现有冷启动链路在业务上成立，问题主要不在逻辑顺序，而在结构表达和长期语义边界没有完全冻结。
4. 当前必须把“对象主档、治理后事实、规则判断、画像基线、日更决策”从探索式 Step 结构中拆开。
5. 本地一期必须补三块能力：字段/契约治理、异常研究与规则实验、UI 观察与日更审查。
6. 完整回归要显式保留，但它是控制步骤，不应继续被当成长期独立业务层。
7. 伪日更应基于历史数据回放完成，用于验证“范围内沉淀、漂移观察、新增观察、严重异常”的闭环。
8. run 和 version 体系必须前置，否则后续重跑、比较、回溯、审计都无法稳定。
9. 当前 Layer/Step 产物可继续保留在运行态，但不应再充当长期架构主语。
10. 需要防止过度工程化，能后置的内容必须后置。

## 3. 当前阶段真正还缺的内容

结论：除以下三类最小骨架外，没有新的结构性补充必要。

真正还缺的不是更多域、更多层、更多平台，而是 3 类会直接影响冻结的骨架定义：

1. 对象与日更治理骨架还需正式定稿。
内容应合并写清：对象状态机、日更判定矩阵、Cell 最低注册定义、特例路由最小规则。

2. 运行与版本骨架还需正式定稿。
内容应合并写清：`run_id`、`contract_version`、`rule_set_version`、`baseline_version` 以及“局部重算 / 完整回归”最小触发矩阵。

3. 数据起点与回放边界还需正式定稿。
内容应合并写清：raw_source 的临时起点、正式目标、Layer0 为空时的过渡口径、完整回归必须回到哪一层。

其中，Claude 评估提出的真正“当前必须补齐”内容，实质上都落在上述三类骨架里，不需要再单独扩成更多文档。

## 4. 建议冻结的最小补充文档集合

建议仅冻结以下 3 份补充短文档，不再继续拆分：

### 4.1 《对象与日更治理骨架说明》

建议合并以下内容：

- LAC / Cell / BS 的最小对象定义
- 对象状态枚举与最小状态机
- Cell 最低注册表定义
- 日更四分流判定矩阵
- 特例路由最小规则
- 观察池晋升、变化待审、异常拒收的最小动作口径

这份文档应成为研发建模和 UI 草图的第一依据。

### 4.2 《运行版本与重算策略说明》

建议合并以下内容：

- `run_id`
- `contract_version`
- `rule_set_version`
- `baseline_version`
- `run_type`
- 完整回归与局部重算触发矩阵
- 版本绑定关系和审计最小口径

这份文档应成为后续调度、重跑比较、版本页和审计页的共同依据。

### 4.3 《数据起点与回放边界说明》

建议合并以下内容：

- raw_source 的临时起点
- raw_source 的正式目标
- Layer0 为空时的过渡使用边界
- 完整回归必须回放到的最底层
- 历史数据伪日更的输入口径

这份文档应成为冷启动、伪日更和数据恢复工作的共同边界说明。

## 5. 明确后置的内容（避免过度工程化）

以下内容都属于“合理但后置”，不应纳入当前阶段必须项：

1. 正式云端数仓分层和物理建模，包括 ODS/DWD/DIM/DWS/ADS 的正式落库细节。
2. 复杂主数据平台化建设。Cell 只需要最小注册，不应在一期扩成复杂 MDM。
3. 复杂审批流、复杂权限体系、消息总线、事件驱动架构。
4. 企业级元数据平台、全量自动血缘、复杂依赖图引擎。
5. 区域化/场景化大规模参数体系，全国化阈值治理。
6. 外部真值源深度融合，例如大规模外部坐标库接入。
7. 面向下游的大规模资产服务产品化。
8. UI 全 9 页甚至第 10 页驾驶舱一次性全部展开。
9. 为了“像平台”而引入过重编排框架或过多抽象层。
10. 机器可读系统契约文件、schema 物理隔离等实现层建议，可做但不应作为当前冻结前提。

Claude 评估中应后置、不宜现在展开的补充，主要也集中在上述几点：完整字段依赖图、独立第 10 个总览页、强平台化元数据能力、实现细节级物理组织方式，都不是当前冻结必需项。

## 6. 本地一期的稳定结构边界

### 6.1 是否保留“8 域方案”作为本地一期的主结构

结论：保留。

但要加一句边界定义：8 域是本地一期的工程主结构，不是正式数仓主结构；评审文档提出的“对象/事实/判断/画像/决策”仍是上位语义，8 域只是本地落地组织方式。

### 6.2 Cell 是否需要一个“最小注册表 / 最低主档定义”

结论：需要，而且是当前必须。

但它只需要“最低注册”，不需要扩成复杂主数据平台。对 Cell 的冻结口径应是：

- Cell 需要被系统正式承认或拒绝承认；
- Cell 需要有最小状态；
- Cell 需要有当前归属关系和版本归属；
- Cell 的丰富统计信息仍应放在画像层，而不是把 Cell 主档做成大而全的对象中心。

换句话说，本地一期应有 `cell_registry` 的语义，但不应把它做成重平台。

### 6.3 伪日更的“好数据沉淀”是否写入 fact_governed

结论：写入。

但边界要保守：

- “范围内且允许沉淀”的数据，写入 `fact_governed`；
- “观察池晋升后允许沉淀”的数据，写入 `fact_governed`；
- “明显漂移待审查”和“严重异常”的数据，默认不直接写入 `fact_governed`，先留在 `decision_ops / issue_rule_lab`；
- `fact_governed` 应同时承接冷启动结果和伪日更沉淀结果，但必须带清晰的来源版本与来源 run。

这样可以保证 `fact_governed` 始终代表“已被治理系统接受的事实底盘”，而不是把所有待判定数据都混进去。

### 6.4 完整回归与局部重算，最小触发矩阵应该如何定义

建议冻结为以下最小矩阵：

| 触发类型 | 默认动作 |
| --- | --- |
| 原始字段解析规则、主键解析规则、关键契约口径变化 | 完整回归 |
| LAC / Cell / BS 可信判定主逻辑变化 | 完整回归 |
| BS 锚点计算、GPS 主修正规则变化 | 完整回归 |
| 仅新增或调整特例旁路规则，且影响范围可界定 | 局部重算 |
| 仅新增问题样本、局部对象复核、局部补数 | 局部重算 |
| 同规则同契约下的基线刷新 | 默认不做完整回归，只重算相关画像与日更判定 |
| 历史数据回放形成新的伪日更沉淀 | 不做完整回归，按增量沉淀 |

最小原则只有一句：会改变“全局解释口径”的，做完整回归；只影响“局部对象或局部规则命中”的，做局部重算。

### 6.5 本地一期 raw_source 的临时起点和正式目标分别是什么

结论如下：

- 临时起点：可允许使用当前 PG17 上已经存在、且足以支撑闭环验证的标准化/过滤后历史数据作为过渡输入，用于文档冻结、流程验证、UI 草图联调。
- 正式目标：必须回到真正的 raw batch + 标准化解析输入，也就是能够支撑完整回归的 Layer0 等价层。

需要明确写入文档的边界是：

- 截至 2026-03-22，PG17 上 Layer0 为空，这是“当前快照”，不是“正式方案目标”；
- 正式开发启动前，应恢复 raw_source 的真实起点，来源可以是 PG15 迁移或原始文件重导；
- 只有正式 raw_source 恢复后，完整回归的定义才真正成立。

因此，本地一期的 raw_source 应分为“过渡起点”和“正式目标”两层口径，不应混写。

### 6.6 字段治理在一期的最小边界是什么

结论：字段治理只保留最小必要能力，不向复杂元数据平台扩展。

一期最小能力建议保留：

1. 数据源 / 表 / 字段注册
2. 原始字段到标准字段的映射
3. 解析规则与输出契约版本
4. 批次级字段健康度快照
5. 字段变化记录
6. 最小下游影响说明

明确后置的能力：

1. 复杂字段依赖图
2. 自动血缘图引擎
3. 企业级语义目录
4. 大而全字段资产平台
5. 复杂字段审批流

### 6.7 Claude 评估中哪些补充是“当前必须”

应纳入当前冻结的，主要有 6 项：

1. `contract_version` 必须进入最小版本体系。
2. `run_id + contract_version + rule_set_version + baseline_version` 必须形成统一绑定。
3. Cell 需要最小注册表，而不是继续停留在“只有统计表”的状态。
4. 需要明确“伪日更沉淀是否进入 fact_governed”的口径。
5. 需要明确“完整回归 / 局部重算”的触发矩阵。
6. 需要明确 raw_source 的过渡起点与正式目标边界。

这些补充都应进入前述 3 份短文档，不再额外扩文档。

### 6.8 Claude 评估中哪些补充是“现在不要展开”

现在不建议展开的，主要有：

1. 第 10 个“全局驾驶舱”页面
2. 完整字段依赖图和自动推算引擎
3. 物理 schema 组织、具体存储方式等实现级建议
4. 机器可读契约文件作为冻结前置
5. 全量 9 页 UI 一次性齐备

这些建议可以保留为后续增强项，但不应成为当前冻结前提。

### 6.9 UI 一期是否真的要按 9 页全部展开，还是应分期

结论：应分期，不建议一期按 9 页全部展开。

UI 一期建议只做 5 类核心能力：

1. 运行中心
2. 日更模拟 / 差异审查
3. 对象观察页
4. 问题中心
5. 字段治理中心（最小版）

建议后置的页面或独立化能力：

1. 数据流转 / 血缘页
2. 规则实验室完整页
3. 基线中心独立页
4. 审计与版本独立页

其中“审计与版本”在一期可以先并入运行中心或对象页的附属视图，不必先做独立页面。

### 6.10 当前阶段最小可冻结的“版本体系”是什么

结论：冻结到 4 个核心标识即可，不再增加。

最小版本体系为：

- `run_id`：一次实际运行实例
- `contract_version`：字段解析与输出契约口径
- `rule_set_version`：正式治理规则集合口径
- `baseline_version`：对象画像与基线参照口径

补充边界：

- `rule_version` 可以留在规则实验域内部，不必上升为全局主版本；
- 所有对象、治理后事实、判定结果、画像结果，都应能回溯到上述 4 个标识；
- 这就是当前阶段最小可冻结版本体系，不建议继续扩。

## 7. UI 一期范围建议

UI 一期应服务于“看得见闭环、看得见差异、看得见对象、看得见问题”，而不是先做完整运营平台。

建议一期范围如下：

1. 能看每次 run 的输入窗口、阶段进度、行数变化和版本绑定关系。
2. 能看伪日更四分流结果，并支持按对象钻取差异原因。
3. 能按 LAC / BS / Cell 查看对象当前状态、画像摘要、最近问题和最近沉淀记录。
4. 能集中查看碰撞 BS、移动 Cell、字段变化等问题队列。
5. 能看到字段健康度、字段变更和最小影响范围。

建议后置的内容如下：

1. 独立的数据流转/血缘可视化
2. 完整规则实验室
3. 独立基线中心
4. 独立审计与版本页
5. 全局驾驶舱

一句话边界：UI 一期做“治理闭环可观察”，不做“全平台全视角”。

## 8. 待确认问题清单（只保留会阻塞冻结的关键问题）

当前没有新的结构性阻塞，真正需要在开发启动前确认的只剩 3 项：

1. raw_source 正式恢复路径最终采用“PG15 迁移”还是“原始文件重导”。
2. 对象状态机和日更判定矩阵的初版阈值参数由谁拍板、以什么配置形式维护。
3. UI 一期在资源有限时，五类核心页面是先做 5 页，还是再合并成 4 页更小启动集。

这 3 项都不改变本文冻结结构，只影响启动顺序和实施节奏。

## 9. 最终建议

本地治理闭环一期现在可以冻结骨架，不建议再继续扩结构。

最终建议是：

1. 以 8 域方案作为本地一期工程骨架继续推进。
2. 以“对象最低注册 + 治理后事实 + 基线画像 + 日更判定”作为核心稳定主线。
3. 只补 3 份短文档，不再拆更多文档。
4. 伪日更的好数据进入 `fact_governed`，但漂移待审和严重异常先不直接沉淀。
5. 用 `run_id + contract_version + rule_set_version + baseline_version` 冻结最小版本体系。
6. 用“全局口径变化=完整回归，局部规则变化=局部重算”冻结最小重算原则。
7. UI 一期严格分期，先保留核心观察与审查能力，后置完整平台页。

如果按这个边界推进，后续研发不会被迫在“探索式 Step 体系”和“正式平台化建设”之间反复摇摆；产品和 UI 也能在稳定骨架上展开，而不会被后续结构反复推翻。
