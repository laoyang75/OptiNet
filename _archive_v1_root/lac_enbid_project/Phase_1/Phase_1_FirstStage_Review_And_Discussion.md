# 第一阶段处理逻辑评估与优化讨论（Layer_0 → Layer_5）

> Version: 1.0  
> Date: 2025-12-29  
> Scope: 基于 `Phase_1_FirstStage_Data_Pipeline.md` 的独立评估；输出“逻辑合理性判断 + 待优化点 + 需要讨论的决策项”  
> 目标：为后续生成“项目合理流程（可长期复用）”做准备

---

## 0. 评估结论摘要（先给结论）

第一阶段的总体逻辑是合理的：以 Layer_3 的可信 BS 库为锚，将 Layer_0 明细纠偏/补齐并在 Layer_5 形成可消费画像，这是“从原始到可用资产”的正确拆解方式。  
但要把它固化成“长期可复跑的工程流程”，仍有几个必须明确/优化的点，主要集中在：

1) **口径边界的显式化**（城市阈值、LAC 缺失如何处理、严重碰撞桶处置）  
2) **异常标签的一致落点**（碰撞/动态/bs_id 异常需要在明细与画像均可见）  
3) **指标可解释性与可对比性**（避免被“天然缺失字段”主导的指标掩盖真实收益）  
4) **数据可穿透规范**（每层必须保留的字段契约 + 可审计元数据）

---

## 1. 逻辑合理性评估（Why it works）

### 1.1 分层拆解是正确的

- Layer_0 把解析复杂度与上层建库逻辑解耦，减少“源表变动/字段语义不一致”对上层的冲击。
- Layer_2 把可信集合（trusted lac / map unique / multi-lac 异常监测清单）显式化，为后续 `lac_dec_final` 提供可解释支撑。
- Layer_3 把“站级中心点”与“桶级质量（离散度/碰撞/共建）”落地，避免 Layer_4 回填凭空补数。
- Layer_4 在明细级提供“原始→纠偏→来源/策略”的可穿透字段，使得补齐收益可以被审计。
- Layer_5 把“扫明细”升级为“一行一对象”，是下游筛选与评估必需的工程化形态。

### 1.2 “可解释优先”的处置策略更适合工程交付

本阶段的关键取舍：严重碰撞桶仍回填但强标注。  
优点：最终表不会出现大量 NULL 导致“看不懂哪里坏”；下游可用标签过滤/降权；复核也更直观。  
风险：若下游忽略标签，可能把“错误中心”当作可信 GPS 使用。

结论：策略可接受，但必须在文档与画像层把标签“抬高到显眼位置”，并提供下游推荐用法（见 3.1）。

---

## 2. 当前流程的边界与风险点（Where it can fail）

### 2.1 城市阈值是硬编码口径（需要“城市/非城市”分流）

- 现行阈值（4G=1000m，5G=500m）是“城市研究”口径。
- 风险：非城市/高速场景会被误判为漂移，导致过度回填或过多 Drift 标签。

讨论点：下一阶段是否引入 `lac` 城市/非城市判定表（或规则）→ 阈值动态化。

### 2.2 `lac_dec_final` 缺失会导致 Layer_5 画像覆盖不足

Layer_5（BS/CELL 画像）当前默认过滤：`lac_dec_final > 0`。  
这符合“以 LAC 分桶”的一致性，但会导致：

- 明细里存在的动态/移动 cell（或异常桶）若 `lac_dec_final` 为 NULL，将无法进入画像表 → 下游只看画像会“漏看”。

讨论点：是否在 Layer_5 增加一类 “LAC 缺失桶”（例如 `LAC=-1` 或单独表/字段标记），保证画像覆盖与明细一致。

### 2.3 信号补齐指标容易被“天然缺失字段”主导

- `need_fill_row_cnt` 统计的是 8 个信号字段里“至少一个为空”的行数，若 `sig_ss` 等字段天然缺失率高，会导致该值接近总行数。
- 风险：补齐效果评估被噪声淹没，难以回答“补齐是否真的有用”。

讨论点：是否把“关键字段集合”（例如 RSRP/RSRQ/SINR/DBM）与“辅助字段”分开统计，并输出两套指标。

### 2.4 异常类型的闭环仍不完整（部分仅标注未进入画像）

已完成的标注链路：

- 碰撞/漂移：明细（Layer_4）→ 画像（Layer_5）已闭环  
- 动态 cell：明细已落表；画像落表但受 `lac_dec_final` 过滤影响

仍建议明确的点：

- bs_id 编码异常（`bs_id_final<256`、过短 hex 等）：目前多为“标注表”与“可选步骤”，应决定是否进入 Layer_5 画像的显式标签字段，避免下游忽略。

## 3. 优化建议（按优先级）

### P0（建议下一阶段必须做）

1) **明确下游使用约束（强推荐）**  
   - 下游若要使用 GPS：必须带过滤条件（例如 `is_severe_collision=false AND gps_status_final='Verified'`），或至少对 `Augmented_from_BS_SevereCollision` 降权。
   - 建议在 Layer_5 或项目级 README 给出“推荐 JOIN + 推荐过滤条件”模板。

2) **补齐 Layer_5 的覆盖边界**  
   - 方案 A：允许 `lac_dec_final IS NULL` 的对象进入画像（增加 `LAC缺失标记`），保证画像与明细可对齐。  
   - 方案 B：保持过滤不变，但输出一张“LAC 缺失对象清单/统计表”，作为下游必看补充。

3) **把 bs_id 异常显式化到画像**  
   - 将 Step44/Step36 的标注，整合为 Layer_4 明细字段（例如 `is_bs_id_lt_256` 已存在于 Step40 逻辑）并在 Layer_5 汇总成标签字段。

### P1（建议做，提升长期可维护性）

4) **统一“阶段级指标面板”**  
   建议新增一张（或一组）Phase_1 dashboard 表：记录每次全量跑的参数快照与关键指标（row_cnt、gps_source 分布、碰撞桶数量、动态 cell 数、bs_id 异常数、信号补齐收益），用于快速验收与回归对比。

5) **固化数据契约（Data Contract）与版本元数据**  
   - 把本阶段约定的核心字段集合写成“契约”，并在关键产物表 COMMENT 中注明：版本、阈值、运营商范围、时间窗口。  
   - 重要的参数（阈值、PLMN 列表、窗口天数）建议从“SQL 顶部常量”迁移为“参数表/会话参数”，避免多次修订后难以追溯。

### P2（性能/工程体验优化）

6) **增量/分区与分片策略固化**  
   - Step40/41 已支持按 BS 维度分片；建议把“分片跑 + 汇总指标”的流程写成固定 RUNBOOK（含失败重跑策略）。  
   - 若进入长期运行，建议按日期或 operator 做分区/增量物化，减少全量重扫成本。

## 4. 需要你拍板的讨论问题（用于生成“合理流程”）

1) 严重碰撞桶：是否永久采用“回填但强标注”的策略？下游是否必须默认过滤？  
2) Layer_5 是否必须覆盖 `lac_dec_final IS NULL` 的对象？若需要，用哪种规范表达（`LAC=-1` / 单独表 / 单独标签）？  
3) 城市/非城市阈值：下一阶段是否引入 lac 城市分类表，还是先维持城市阈值并仅做标注？  
4) bs_id 异常：是否进入 Layer_5 画像的显式标签字段（而不是仅保留 Step44/36 标注表）？  
5) 信号补齐评估：是否把“关键字段集合”先固定下来（RSRP/RSRQ/SINR/DBM），以便 Phase_2 做加权公式时有稳定口径？

---

## 5. 对“数据可穿透性”的最终建议（供后续流程固化）

建议把“可穿透性”作为工程硬要求写入项目流程：

- 每层主表必须能回答：输入是谁、输出是谁、关键字段如何变化、质量标记如何变化、收益如何量化。  
- 每层至少保留三类字段：
  1) 主键与分桶键（对象可定位）  
  2) before/after（例如 `lon_before_fix → lon_final`，`sig_* → sig_*_final`）  
  3) 来源/策略/原因（例如 `gps_source/gps_fix_strategy/collision_reason/signal_fill_source`）
