# L1-LAC：`原始lac` 筛选规则 v1（可直接按此操作）

> 目标：给出一份 **简短、可直接执行** 的 LAC 筛选说明，用于从  
> `public."网优cell项目_清洗补齐库_v1"` 中筛出“数值合规 + 制式合规（4G/5G）”的 LAC 明细，作为后续 CELL / ENBID / GPS 研究的统一起点。

本文件**独立可读**，不依赖其他文档。

---

## 1. 全局约束与直接过滤

**数据源：**

- 表：`public."网优cell项目_清洗补齐库_v1"`

**全局约束：**

- 运营商范围：只保留 5 个核心 PLMN：
  - `运营商id` ∈ {`46000`, `46001`, `46011`, `46015`, `46020`}
- LAC 必须是“数值合规”的：
  - `"原始lac"` 非空：`IS NOT NULL` 且 `btrim("原始lac") <> ''`
  - `"原始lac"` 为纯数字：`btrim("原始lac") ~ '^[0-9]+$'`
  - （在 5 个 PLMN 中，`0` / `-1` / 非数字几乎不存在，规则仍保留以防后续扩展）
- 制式范围：只保留 4G / 5G：
  - `lower(tech)` ∈ (`'4g'`, `'5g'`)
  - 过滤掉 2G/3G 和其他写法（视为“边缘/不参与本轮合规”的数据）。

**新增（必须）：明确剔除 LAC 溢出/占位值（用于“可回补但不可入库”的异常）**

以下 `lac_hex`（`upper(to_hex(lac_dec))`）在当前项目中视为**明确异常**，必须剔除：

- `0` / `NULL`：缺失/占位（应通过 `cell_id` 映射反哺补齐）
- `FFFF`、`FFFFFF`：典型 `-1` 溢出/占位（分别对应 16/24 bit）
- `FFFE`、`FFFFFE`：典型 `-2` 溢出/占位（分别对应 16/24 bit）
- `7FFFFFFF`：典型 32-bit 溢出/占位（十进制 2147483647）

> 说明：这些值即使“是正整数”，也不应进入“合规/可信”集合；它们只应该作为“待回补/待纠偏”的输入，被 Step06 之类的映射机制纠正。

## 2. 运营商分组与 LAC 特征（两大组）

为方便解释，按 LAC 之前的分析习惯，将 5 个运营商拆成两组：

- **第一组（移动系）**：
  - `运营商id` ∈ {`46000`, `46015`, `46020`}
- **第二组（联通 + 电信）**：
  - `运营商id` ∈ {`46001`, `46011`}

十六进制长度上的特征（仅作为认识，不在 v1 中做硬过滤）：

- 第一组（移动系）：
  - LAC 转成 16 进制后，绝大多数为 4 位或 6 位；
  - 少量 3/5 位可视为“特殊/可疑长度”，后续可以重点排查。
- 第二组（联通 + 电信）：
  - 3/4/5/6 位的 16 进制 LAC 都大量存在；
  - 不能用简单的“4 位=4G / 6 位=5G”规则判断制式，长度仅作为辅助特征。

本轮筛选 **不使用长度作为硬过滤条件**，只按“数值 + 制式”做统一 LAC 合规起点。

> 对北京明细（Layer_2）而言，已进一步把“十六进制位数”作为硬过滤（见 `Layer_2/sql/02_step2_compliance_mark.sql`），并且叠加了上面的“溢出/占位值剔除”。

---

## 3. 第一组（移动系）规则：有效规范与异常值

### 3.1 有效规范（保留条件）

对 `运营商id` ∈ {`46000`, `46015`, `46020`} 的记录，本轮认为“LAC 合规”的条件为：

- 制式必须是 4G 或 5G：
  - `lower(tech)` ∈ (`'4g'`, `'5g'`)
- LAC 数值必须合规：
  - `"原始lac" IS NOT NULL`
  - `btrim("原始lac") <> ''`
  - `btrim("原始lac") ~ '^[0-9]+$'`
  - （如在未来数据中出现 `0` / `-1` 等占位值，可酌情追加过滤）

当前数据中，满足上述条件的移动系 LAC：

- 占该组 LAC 的绝大多数；  
- 16 进制长度主要为 4 位和 6 位（与 4G/5G 的典型长度一致）。

### 3.2 异常值（本轮直接剔除）

对第一组，当前明确认为无效并直接过滤掉的 LAC 类型：

- 所有制式下：
  - `原始lac` 为空或只包含空格；
  - `原始lac` 含非数字字符；
  - `原始lac` 为 `0` 或 `-1`（虽然在当前 5 个 PLMN 中几乎没有，规则仍保留）。
  - `lac_hex` 命中溢出/占位值：`FFFF/FFFE/FFFFFE/FFFFFF/7FFFFFFF`。
- 2G/3G / OTHER 制式下的 LAC：
  - `lower(tech)` ∈ (`'2g'`, `'3g'`) 或其他值，统一视为“本轮不使用”，全部过滤。

---

## 4. 第二组（联通 + 电信）规则：有效规范与异常值

### 4.1 有效规范（保留条件）

对 `运营商id` ∈ {`46001`, `46011`} 的记录：

- 制式必须是 4G 或 5G：
  - `lower(tech)` ∈ (`'4g'`, `'5g'`)
- LAC 数值合规（与第一组相同）：
  - `"原始lac" IS NOT NULL`
  - `btrim("原始lac") <> ''`
  - `btrim("原始lac") ~ '^[0-9]+$'`

当前数据中，满足上述条件的联通/电信 LAC：

- 覆盖约 99.9% 的记录；  
- 十六进制长度分布在 3–6 位之间，5 位数量尤其多；  
- 本轮不对“长度是否为 5 位”等细节做硬限制，只统一视为 LAC 合规的起点集合。

### 4.2 异常值（本轮直接剔除）

对第二组，本轮明确剔除的 LAC 类型与第一组相同：

- 空值 / 空字符串；
- 含非数字字符；
- `0` / `-1`；
- `lac_hex` 命中溢出/占位值：`FFFF/FFFE/FFFFFE/FFFFFF/7FFFFFFF`；
- 制式为 2G/3G/OTHER 的全部记录。

## 5. 可直接使用的 LAC 筛选 SQL（统一规则）

下面是一条可以直接生成“本轮 L1-LAC 合规明细”的 SQL，已经包含上述所有过滤条件：

```sql
CREATE OR REPLACE VIEW public.v_lac_L1_stage1 AS
SELECT
  t.*
FROM public."网优cell项目_清洗补齐库_v1" t
WHERE
  -- 运营商范围：5 个核心 PLMN
  t."运营商id" IN ('46000','46001','46011','46015','46020')

  -- LAC 数值合规：非空 + 纯数字
  AND t."原始lac" IS NOT NULL
  AND btrim(t."原始lac") <> ''
  AND btrim(t."原始lac") ~ '^[0-9]+$'

  -- 明确剔除 LAC 溢出/占位值（FFFF/FFFE/FFFFFE/FFFFFF/7FFFFFFF）
  AND upper(to_hex(btrim(t."原始lac")::bigint)) NOT IN ('FFFF','FFFE','FFFFFE','FFFFFF','7FFFFFFF')

  -- 只保留 4G / 5G，统一过滤掉 2G/3G/其它制式
  AND lower(t.tech) IN ('4g','5g');
```

> 使用建议：
>
> - 后续所有关于 LAC 的统计、TOP 研究、与 cell / ENBID / GPS 的衔接分析，都可以直接以 `public.v_lac_L1_stage1` 作为输入；  
> - 若未来识别出明确无效的 LAC ID（例如极端高频、跨省/跨运营商出现且 GPS 显示为黑洞的 LAC），可在上述 SQL 的尾部追加：  
>   `AND t."原始lac"::bigint NOT IN (...)`，形成更严格的 LAC 合规视图。
