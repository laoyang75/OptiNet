## 1. `ss1` 字段解析的特别约定（避免缺少 PLMN / LAC 标签）

在现有数据中，部分 `cell_id` 来源于报文中的 `ss1` 字段，这类数据往往在后阶段才被解析和补齐，容易出现：

- `cell_id` 有值，但缺少配套的 `运营商id` 与 `原始lac` 标签；  
- 或者在后续全局补数时，仅按 `cell_id` 去外部库查，误把非中国运营商的数据补入。

为避免这类问题，L0 在解析 `ss1` 字段时增加如下规则（作为后续实现的设计约定）：

- 对每条原始报文，优先使用报文内已有的 `cell_info`（或同义字段）确定：  
  - 本条报文的 `运营商id`（PLMN）；  
  - 本条报文的 `原始lac`（若能解析）。  
- 在解析 `ss1` 字段时：  
  - 从 `ss1` 中取出 `cell_id`；  
  - 将该 `cell_id` 与同一报文中的 `cell_info` 所在 Cell 做比对 / 关联；  
  - 若匹配成功，则**优先继承本报文 `cell_info` 中的 `运营商id` 与 `原始lac`**，作为 `ss1` 派生记录的标签。  
- 只有在报文内部无法确定 `运营商id` / `原始lac` 时，才考虑使用外部补数或全局规则。

这样可以做到：

- 来自 `ss1` 字段的 `cell_id` 在进入 L1 之前，就已经尽量携带正确的运营商和 LAC 标签；  
- 后续在 L1-LAC / L1-CELL / L1-ENBID 阶段，始终可以使用 `(运营商id, 原始lac, cell_id)` 作为统一主键，减少因为后解析导致的标签缺失或错配。
