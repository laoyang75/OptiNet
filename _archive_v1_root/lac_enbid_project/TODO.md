# LAC / 基站项目 TODO（简版）

> 目标：一步一步验证从 GPS → LAC → cell → ENBID 的全流程是否合理，并在每一步都看清楚数据变化（输入/输出/被过滤掉的部分）。

1. 梳理表和数据流关系  
   - 画出（可纸上）并记录：  
     - `网优cell项目_清洗补齐库_v1` → `网优cell项目_lac统计库_v1`  
     - → `网优cell项目_lac可信库_v1` → `网优cell项目_cellid可信库_v1`  
     - → `网优cell项目_基站中心库_v1` → `网优cell项目_最终回填主库_v1`  
   - 写清楚每一层是“明细 / 按 LAC 聚合 / 按 cell 聚合 / 按 ENBID 聚合”。

2. 确认“北京主流 LAC 前缀”  
   - 在 `网优cell项目_lac统计库_v1` 中，按【制式(4G/5G) + 运营商】统计：  
     - 4G：LAC 16 进制前 2 位的分布；  
     - 5G：LAC 16 进制前 4 位的分布；  
   - 记录每个运营商+制式设备数/上报量最高的前几类前缀（例如移动 4G 的 `10/11`，移动 5G 的 `2000` 等），作为“北京主流前缀”参考。

3. 列出“前缀异常但量不小”的 LAC 清单  
   - 在同一张统计表中：  
     - 过滤掉属于主流前缀的 LAC；  
     - 在剩余 LAC 中按【设备数】或【总上报次数】排序；  
     - 取每个运营商+制式的 topN，形成“前缀异常 LAC 清单”（包含十进制和十六进制）。
   - 把特别关注的 LAC（例如 `58645, 58646, 10016, 12368, 405521, 405525, 81934, 73779, 2162683, 786464, 5570592`）单独标出来。

4. 针对每个重点 LAC 做 GPS 聚类检查  
   - 在 GPS 原始明细表中，针对清单里的每一个 LAC：  
     - 统计经纬度点的空间聚类（大致看是否集中在北京）；  
     - 统计城市 / 省份分布；  
     - 看 14 天内的时间稳定性（每天是否持续出现）。  
   - 对每个 LAC 给一个结论标签：  
     - 集中在北京；  
     - 集中在其他地区；  
     - 全国离散（疑似异常值）；  
     - GPS 算法疑似有问题（聚类逻辑偏差）。

5. 更新 `lac可信库_v1`（建立可信/异常 LAC 库）  
   - 根据上一步标签，在 `网优cell项目_lac可信库_v1` 中：  
     - 标记/写入“可信 LAC”（可以用于后续 cell/ENBID 推断）；  
     - 标记“异常 LAC”（需要在后续规则中屏蔽或特殊处理）；  
     - 对于确认 GPS 算法有问题的情况，记录案例用于后续算法调整。

6. 从可信 LAC 下筛选可信 cell_id  
   - 只在“可信 LAC”范围内统计 cell 的行为：  
     - 每个 cell 的活跃天数、经纬度分布、覆盖区域；  
   - 剔除：位置离散、跳跃严重或仅少量设备支撑的 cell；  
   - 将剩余的 cell 写入 `网优cell项目_cellid可信库_v1`。

7. 从可信 cell 聚合 ENBID，更新基站中心库  
   - 基于 `cellid可信库_v1`：  
     - 按 ENBID 聚合 cell，计算基站中心点和典型服务范围；  
     - 写入/更新 `网优cell项目_基站中心库_v1`。  
   - 确保一个 ENBID 不会跨省/跨大区域，否则需要重新检查对应的 cell / LAC。

8. 回填到最终主库，并做抽样验证  
   - 使用 LAC / cell / ENBID 可信库，把位置和基站信息回填到 `网优cell项目_最终回填主库_v1`；  
   - 抽样检查几类典型场景：  
     - 北京典型 LAC 前缀下的若干设备；  
     - 前缀异常但判定为正常的 LAC；  
     - 被标记为异常 LAC 的样本。  

9. 每一步都出一张简单报表  
   - 对每个环节，至少保留：  
     - 输入行数 / LAC 数 / cell 数 / ENBID 数；  
     - 输出行数；  
     - 被过滤掉的数量及比例；  
     - 前后前缀/区域分布的对比。  
   - 这样在调整过滤规则时，可以清楚看到每一步对数据的影响。
